**Связанные документы:**

---

## Назначение

Папка `llm_tasks/` содержит систему управления задачами для работы LLM с проектом.

**Важно:** Все задачи проекта должны вестись через `llm_tasks/`, а не через временные файлы типа PROJECT_IMPROVEMENTS.md.

---

## Структура

```
llm_tasks/
├── current_tasks.md      # Текущие задачи основного LLM
├── future_tasks.md       # Бэклог задач основного LLM
└── agents/               # Задачи для агентов
    └── amy-santiago/     # Задачи Documentation Manager
        ├── current_tasks.md
        └── future_tasks.md
```

**Разделение задач:**
- **Основной LLM** (`llm_tasks/current_tasks.md`) — общие задачи проекта (разработка, рефакторинг, интеграции)
- **Агенты** (`llm_tasks/agents/[имя]/`) — специализированные задачи агента (для Amy Santiago — аудит документации, создание документов, валидация)

---

## Задачи агентов

### Структура задач агента

Каждый агент имеет собственную папку в `llm_tasks/agents/[имя-агента]/`:

```
llm_tasks/agents/amy-santiago/
├── current_tasks.md      # Текущие задачи Amy Santiago
├── future_tasks.md       # Бэклог задач Amy Santiago
├── completed_tasks.md    # Архив завершённых задач
└── temp/                 # Временные файлы агента
    ├── report_2026-01-15.md
    ├── audit_results.json
    └── ...
```

**Временные файлы (`temp/`):**
- Все временные файлы, создаваемые агентом, хранятся в `temp/`
- Отчёты, промежуточные результаты, логи работы
- Имена файлов с датой: `report_YYYY-MM-DD.md`
- **Очистка:** При начале новой задачи агент очищает `temp/` и начинает работу с чистого листа

**Архив завершённых задач (`completed_tasks.md`):**
- Хранит историю выполненных задач агента
- Новые задачи добавляются **вверх** файла (обратная хронология)
- Содержит результаты, даты, ссылки на итоговые файлы

### Правила работы агентов с задачами

#### 1. При запуске агента:

1. **Прочитать** `current_tasks.md`
2. **Показать пользователю секцию "Что было сделано в прошлый раз":**
   - Список выполненных подзадач с ✅
   - Текущий прогресс задач
   - Статус незавершённых задач
3. **Предложить варианты:**
   - Продолжить текущие задачи
   - Начать новые задачи
   - Просмотреть бэклог (`future_tasks.md`)

#### 2. При начале новой задачи:

1. **Очистить папку** `temp/`:
   - Удалить все временные файлы из прошлой сессии
   - Начать работу с чистого листа
2. **Обновить статус** задачи: `planned` → `in_progress`
3. **Обновить дату** последнего запуска в `current_tasks.md`

#### 3. Во время работы:

1. **Обновлять статусы** в `current_tasks.md` в реальном времени:
   - Отмечать подзадачи: `- [ ]` → `- [x]`
2. **Сохранять временные файлы** в `temp/`:
   - Отчёты, промежуточные результаты, логи
   - Использовать имена с датой: `report_2026-01-15.md`
3. **Синхронизировать** `current_tasks.md` после каждого важного шага

#### 4. При завершении задачи:

1. **Переместить задачу** из `current_tasks.md` в `completed_tasks.md`:
   - Добавить **вверх** файла (обратная хронология)
   - Указать дату завершения, результат, ссылки на файлы
2. **Обновить секцию** "Что было сделано" в `current_tasks.md`
3. **Временные файлы** остаются в `temp/` до следующей сессии

#### 5. Формат задач агента:

- Аналогичен формату основного LLM (см. ниже)
- Специфичные задачи агента (для Amy — документация, для других — их область)
- Обязательная секция "Что было сделано в прошлый раз" в начале `current_tasks.md`

### Правила работы с current_tasks.md

1. **Начало новой сессии:**
   - Проверить current_tasks.md
   - Если есть незавершённые задачи — продолжить с ними
   - Добавить новые задачи, если получены от пользователя

2. **В процессе работы:**
   - Обновлять статус подзадач по мере выполнения
   - Перемещать выполненные задачи в раздел "Завершённые"
   - Добавлять новые задачи в "Ожидают выполнения"

3. **Завершение сессии:**
   - Переместить незавершённые задачи в future_tasks.md (если нужно)
   - Или оставить в current_tasks.md для следующей сессии
   - Обновить дату последнего обновления

4. **ID задач:**
   - Формат: `[ID-XXX]` где XXX — порядковый номер (001, 002, 003...)
   - ID уникален в рамках проекта
   - Не переиспользовать ID завершённых задач

### Статусы задач

- **`pending`** — задача создана, ожидает начала работы
- **`in_progress`** — задача в процессе выполнения
- **`blocked`** — задача заблокирована (указать причину)
- **`completed`** — задача завершена
- **`cancelled`** — задача отменена (указать причину)

### Приоритеты

- **`высокий`** — критичная задача, требует немедленного внимания
- **`средний`** — важная задача, выполнить в ближайшее время
- **`низкий`** — некритичная задача, можно отложить

## Workflow управления задачами

### 1. Начало новой сессии

```
1. Открыть current_tasks.md
2. Проверить незавершённые задачи
3. Спросить пользователя о приоритетах на сессию
4. Открыть future_tasks.md
5. Переместить выбранные задачи в current_tasks.md
6. Начать работу
```

### 2. Во время сессии

```
1. Работать над задачами из current_tasks.md
2. Обновлять статусы подзадач
3. Добавлять новые задачи при необходимости
4. Перемещать завершённые задачи в раздел "Завершённые"
```

### 3. Завершение сессии

```
1. Просмотреть current_tasks.md
2. Переместить незавершённые низкоприоритетные задачи в future_tasks.md
3. Оставить высокоприоритетные незавершённые задачи в current_tasks.md
4. Обновить дату последнего обновления
5. Сохранить файлы
```

### 4. Периодическое обслуживание

```
Еженедельно:
- Проверить актуальность задач в current_tasks.md
- Обновить приоритеты в future_tasks.md

Ежемесячно:
- Архивировать завершённые задачи
- Удалить неактуальные задачи из future_tasks.md
- Обновить оценки сложности
```

## Интеграция с документацией

### Связь задач с дискуссиями

Когда задача требует архитектурных решений:

1. Создать дискуссию в `general_docs/discuss/`
2. Указать ссылку на дискуссию в задаче
3. После утверждения дискуссии — обновить задачу

**Пример:**

```markdown
### [ID-050] Реализовать систему уведомлений

**Связанные документы:**
- [Дискуссия 005](../general_docs/discuss/005_notification_system.md)
- [Архитектура 010](../general_docs/architecture/010_notifications.md)
```

### Связь задач с планами реализации

Когда начинается реализация задачи:

1. Создать план реализации в `general_docs/imp_plans/`
2. Указать ссылку на план в задаче
3. Отслеживать прогресс в плане

**Пример:**

```markdown
### [ID-051] Реализовать Auth Service

**Связанные документы:**
- [План реализации 003](../general_docs/imp_plans/003_plan_auth_service.md)

**Прогресс:**
- [x] Настройка проекта
- [x] JWT интеграция
- [ ] OAuth провайдеры
- [ ] Тесты
```

## Интеграция с TodoWrite

**Важно:** TodoWrite (внутренний инструмент сессии) и llm_tasks/ служат разным целям:

- **TodoWrite:** Краткосрочное управление задачами в рамках одного сообщения/операции
- **llm_tasks/:** Долгосрочное управление задачами проекта между сессиями

### Правило использования

1. **Используй TodoWrite для:**
   - Отслеживания шагов в текущей операции
   - Визуализации прогресса для пользователя
   - Внутрисессионного управления

2. **Используй llm_tasks/ для:**
   - Хранения задач между сессиями
   - Планирования долгосрочных целей
   - Документирования выполненной работы

3. **Синхронизация:**
   - В конце сессии — обновить current_tasks.md на основе TodoWrite
   - В начале сессии — создать TodoWrite на основе current_tasks.md

## Виды документации

### Файл глоссария glossary.md
- **Назначение:** Глоссарий для всей терминологии проекта
- **Расположение:** general_docs/glossary.md
- **Аудитория:** Все участники
- **Структура:** Термины организованы по категориям (Документация, Claude Code, Управление задачами) с алфавитным указателем

Глоссарий должен содержать все термины, которые вводятся в проекте.
Термины должны быть оформлены в виде ссылок, на которые должны ссылаться все *.md файлы (например, 02_architecture/01_system_overview.md и все *_doc.md).

**Категории глоссария:**
- **Документация** (9 терминов) — термины системы документации
- **Claude Code** (3 термина) — агенты, скиллы, команды
- **Управление задачами** (2 термина) — система управления задачами LLM

**Текущий размер:** 14 терминов, ~180 строк

**Пороги масштабирования:** При достижении 150 терминов или 1000 строк — создать дискуссию о разделении глоссария на файлы по категориям.

**Пример описания термина:**
*   **Определение:** (Краткое, чёткое определение).
*   **Контекст:** (В каком модуле/архитектуре используется).
*   **Синонимы:** (Если есть).
*   **Антонимы:** (Если есть).
*   **Связанные термины:** (Ссылки на другие статьи глоссария).
*   **Источник:** (Ссылка на `discuss` или `architecture` файл, где термин был введён и утверждён).


### [📖 Общая документация](../general_docs/glossary.md#общая-документация)
- **Назначение:** Управление жизненным циклом функциональности — от идеи до [📖 плана реализации](../general_docs/glossary.md#план-реализации).
- **Расположение:** general_docs/ (в корне репозитория)
- **Содержание:** [📖 Дискуссии](../general_docs/glossary.md#дискуссия), архитектура, [📖 планы реализации](../general_docs/glossary.md#план-реализации), описания [📖 ресурсов](../general_docs/glossary.md#ресурс).
- **Аудитория:** Архитекторы, менеджеры, аналитики

### [📖 Документация папок](../general_docs/glossary.md#документация-папок)
- **Назначение:** Локальная документация модулей с ссылками на [📖 общую документацию](../general_docs/glossary.md#общая-документация).
- **Содержание:** Специфика модуля, связи с общей документацией, технические детали.
- **Аудитория:** Архитекторы, разработчики, тестировщики

## Шаблоны документов

**Расположение:** [templates/](templates/)

Все шаблоны для создания документации находятся в папке `llm_instructions/templates/`.

### Типы шаблонов

#### 1. Шаблоны для документов общей документации

**Назначение:** Создание новых документов в `general_docs/`

| Шаблон | Назначение | Использование |
|--------|------------|---------------|
| [discuss.md](templates/discuss.md) | Шаблон дискуссии | Создание файлов в `general_docs/01_discuss/` |
| [architecture.md](templates/architecture.md) | Шаблон архитектурного документа | Создание файлов в `general_docs/02_architecture/` |
| [decision_adr.md](templates/decision_adr.md) | Шаблон Architecture Decision Record | Создание файлов в `general_docs/04_decisions/` |
| [imp_plan.md](templates/imp_plan.md) | Шаблон плана реализации | Создание файлов в `general_docs/06_imp_plans/` |
| [resource.md](templates/resource.md) | Шаблон описания ресурса | Создание файлов в `general_docs/05_resources/` |
| [folder_doc.md](templates/folder_doc.md) | Шаблон документации папки | Создание README.md в папках модулей/сервисов |

#### 2. Шаблоны для индексных файлов и правил

**Назначение:** Создание служебных файлов в папках документации

| Шаблон | Назначение | Использование |
|--------|------------|---------------|
| [000_index.md](templates/000_index.md) | Шаблон индексного файла (ЧТО) | Создание 000_*.md файлов в папках `general_docs/` |
| [general_docs_README.md](templates/general_docs_README.md) | Шаблон README для основных папок (КАК) | Создание README.md в 01_discuss/, 02_architecture/, 03_diagrams/, 06_imp_plans/ |
| [resources_README.md](templates/resources_README.md) | Шаблон README для ресурсов (КАК) | Создание README.md в 05_resources/[подпапки]/ |

### Разделение ответственности: ЧТО vs КАК

Каждая папка в `general_docs/` содержит два служебных файла:

**000_[название_папки].md (ЧТО):**
- **Динамический индекс** всех документов в папке
- Таблица с метаданными (ID, название, описание, статус, даты)
- Статистика по статусам
- Автоматически обновляется скиллом `/doc-index`
- **Для LLM:** Быстрый обзор содержимого папки

**README.md (КАК):**
- **Статичные правила** работы с документами этого типа
- Процесс создания новых документов
- Workflow статусов и переходы
- Требования к структуре и разделам
- Примеры и частые ошибки
- **Для LLM:** Инструкции по созданию и поддержке

### Использование шаблонов

**Для LLM при создании нового документа:**

1. **Определить тип документа:**
   - Дискуссия → [discuss.md](templates/discuss.md)
   - Архитектура → [architecture.md](templates/architecture.md)
   - Decision (ADR) → [decision_adr.md](templates/decision_adr.md)
   - План → [imp_plan.md](templates/imp_plan.md)
   - Ресурс → [resource.md](templates/resource.md)

2. **Скопировать содержимое шаблона**

3. **Заполнить все секции:**
   - Заменить плейсхолдеры [НАЗВАНИЕ], [ID], [ОПИСАНИЕ]
   - Указать корректные метаданные
   - Добавить связи с другими документами

4. **Обновить индексный файл (000_*.md):**
   - Добавить запись в таблицу
   - Обновить статистику по статусам
   - Обновить дату последнего изменения

5. **Проверить документ:**
   - Все обязательные разделы заполнены
   - Ссылки рабочие
   - Статус корректный

**Автоматизация:** Скилл `/doc-index` автоматически обновляет индексные файлы (000_*.md).

### Обновление шаблонов

**Когда обновлять шаблоны:**
- При изменении требований к структуре документов
- При добавлении новых обязательных разделов
- При изменении workflow статусов
- При обнаружении часто повторяющихся ошибок

**Процесс обновления:**
1. Обновить шаблон в `llm_instructions/templates/`
2. Обновить этот документ (general_docs.md)
3. Создать дискуссию об изменении (если существенное)
4. Уведомить разработчиков через CLAUDE.md

### Дискуссии (Discuss)

[📖 Дискуссии](../general_docs/glossary.md#дискуссия) в папке 01_discuss/

**Назначение:** Фиксация обсуждений "пользователь ↔ LLM" о функциональности.

**Требования к файлам:**
- Правило наименования: ["ID_краткое_описание"].md (например: 001_user_auth_and_roles.md)
- Уникальный ID: Последовательная нумерация (001, 002, 003...)

**Обязательные разделы:**
- Исходный запрос пользователя
- Предложения по реализации (минимум 2 варианта)
- Критерии оценки с весами (Высокий/Средний/Низкий)
- Сравнительная таблица вариантов по критериям (⭐⭐⭐/⭐⭐/⭐/—, места 🏆🥈🥉)
- **Текстовое сравнение вариантов** — анализ отличий между вариантами
- **Рекомендуемые варианты** — 3-4 лучших варианта с кратким обоснованием
- Принятое решение (с аргументацией внутри раздела)
- Связанная архитектура — ссылки на создаваемые документы `02_architecture/`

#### Шаблон файла [📖 дискуссии](../general_docs/glossary.md#дискуссия)

**Шаблон:** [templates/discuss.md](templates/discuss.md)
**Скилл создания:** `/discussion` или `python scripts/discuss_new.py`

**Структура документа:**
1. **Метаданные:** Статус (с эмодзи), Дата
2. **Исходный запрос пользователя:** Дословная цитата
3. **Анализ цели и контекст:** Проблема, область обсуждения, что вне области
4. **Предложения:** Таблица вариантов с плюсами/минусами
5. **Критерии оценки:** Таблица критериев с весами
6. **Сравнение вариантов по критериям:** Таблица со звёздным рейтингом
7. **Текстовое сравнение вариантов:** Анализ отличий между вариантами
8. **Рекомендуемые варианты:** Таблица 3-4 лучших с обоснованием
9. **Принятое решение:** Выбор + аргументация (в одном разделе!)
10. **Вопросы и допущения:** Вопросы для уточнения с ответами, неявные допущения
11. **История обсуждения:** Хронология обсуждения
12. **Связанная архитектура:** Ссылки на `02_architecture/`
13. **Добавления к глоссарию:** Новые термины

#### Рабочий процесс с файлами [📖 дискуссий](../general_docs/glossary.md#дискуссия)

**Статусы файлов [📖 Дискуссий](../general_docs/glossary.md#дискуссия):**
- 🟡 draft — файл создан, варианты НЕ предложены
- 🔵 in_progress — варианты предложены, идёт обсуждение с пользователем
- 🟣 review — пользователь выбрал вариант, проводится ревью решения (`/discussion-review`)
- 🟢 approved — решение принято после ревью
- ⚪ final — созданы документы архитектуры (`02_architecture/`)

**[📖 Workflow](../general_docs/glossary.md#workflow-статусов):**
```
🟡 draft → 🔵 in_progress → 🟣 review → 🟢 approved → ⚪ final
                ↑                ↑             ↓
           обсуждение    /discussion-review   Создаются ТОЛЬКО файлы
                                              02_architecture/
```

**ВАЖНО:** При финализации дискуссии создаются **только файлы архитектуры**. Ресурсы и планы реализации создаются из [📖 Decision (ADR)](../general_docs/glossary.md#decision-adr), не напрямую из дискуссии или архитектуры.

**Обратная связь:** статус может измениться final → [📖 feedback](../general_docs/glossary.md#feedback) при уточнении документации

#### Правила обратной связи [📖 FeedBack](../general_docs/glossary.md#feedback)
**При изменении файлов архитектуры, которые имеют ссылки на файлы дискуссий:**
- Обновить ссылки в связанных файлах дискуссий
- Дополнить документы связанных дискуссий с фиксацией:
-   Какие части тел документов изменены
-   Причины изменений
-   Дата обновления

**Чек-лист проверки изменений:** 
- Обновлен файл дискуссий
- Проверены и обновлены все ссылки на ресурсы `02_architecture/`
- Обновлен статус документа


### Архитектура (Architecture)

Архитектурные документы в папке 02_architecture/

**Назначение:** Планирование архитектуры на основе дискуссий.

**Требования к файлам:**
- Именование: [ID]_[название_компонента].md (например: 001_system_overview.md)
- Уникальный ID: Последовательная нумерация (001, 002, 003...)
- Обязательная ссылка: на ВСЕ связанные дискуссии (все обсуждения, на которых основана архитектура)

**Обязательные действия в файле архитектуры (последовательность):** 
- Планирование архитектуры:
    - Диаграммы компонентов
    - Описание взаимодействий
    - Примеры использования
    - Границы ответственности
- Выделение состава ресурсов:
    - Перечень необходимых ресурсов
    - Классификация по типам ресурсов (см. [типы_ресурсов] на данной странице)
- Создание описаний ресурсов:
    - Если описание отсутствует → создать файл в 05_resources/[типы_ресурсов]/ (см. [типы_ресурсов] на данной странице)
    - Имя файла должно соответствовать имени ресурса
- Создание плана реализации:
    - Создать файл в 06_imp_plans/
    - Добавить ссылку на созданный план в файл архитектуры

#### Шаблон файла архитектуры
- **Цель и контекст компонента**
- **Ссылки:** На `01_discuss/`, `05_resources/`, `06_imp_plans/`.
- **Оглавление документа со ссылками на пункты документа**
- **Принципы и ограничения** (например, CAP-теорема, compliance).
- **Диаграммы:** C4-уровни (Context, Container, Component), UML-последовательности. Если нужна диаграмма, её можно создать в `03_diagrams/` в соответствие со следующими условиями:
    - Формат: Предпочтительно Mermaid (встроен в Markdown)
    - Инструменты: Draw.io (.drawio)
    - Расположение: В папке general_docs/diagrams
    - Требования: Должны быть самодостаточны (подписи, легенда)
    - Ссылку на файл диаграмы (если он был создан) необходимо добавить в файл архитектуры
- **Интерфейсы и контракты:** API (OpenAPI/Swagger), схемы сообщений, форматы событий.
- **Данные:** Модели, стратегии доступа, миграции.
- **Внешние зависимости:** Сервисы, библиотеки, API.
- **Требования к качеству:** (Нефункциональные требования): доступность, производительность, безопасность.
- **Риски и компромиссы.**
- **История изменений (только для файлов архитектуры)** Представляет из себя таблицу вида:
```## История изменений (Changelog)
| Версия | Дата | Автор | Описание изменений | Связанный Issue/PR |
|--------|------|-------|-------------------|-------------------|
| 1.0 | 2023-10-27 | AI Архитектор | Первоначальное создание | 01_discuss/001_* |
| 1.1 | 2023-10-28 | Петров И.И. | Уточнены требования к БД | feedback #2 |
```

#### Рабочий процесс с файлами архитектуры

**Статусы файлов Архитектуры:**
- draft (Планирование работы над архитектурой)
- in_progress (Работа над архитектурой) ИЛИ feedback (фиксация изменений из ресурсов)
- review (Ревью архитектуры)
- approved (Создание (или изменение) описаний ресурсов и Плана реализации)
- final (Файл реализован в соответствии с требованиями, переход к работе над ресурсами)

**[📖 Workflow](../general_docs/glossary.md#workflow-статусов):** draft → in_progress / feedback → review → approved → final
**Важно:** статус может измениться final → [📖 feedback](../general_docs/glossary.md#feedback) при работе с уточнением документации

#### Правила обратной связи [📖 FeedBack](../general_docs/glossary.md#feedback)
**При создании нового [📖 ресурса](../general_docs/glossary.md#ресурс) по некоторой архитектуре:**
- [📖 Ресурс](../general_docs/glossary.md#ресурс) должен быть добавлен в general_docs/05_resources/[типы_ресурсов]
- Файл архитектуры должен быть дополнен ссылкой на новый ресурс
- Запустить FeedBack для всех связанных с файлом архитектуры дискуссий

**Чек-лист проверки изменений:**
- Обновлен файл архитектуры
- Проверены и обновлены все ссылки на ресурсы `01_discuss/`, `05_resources/`, `06_imp_plans/`
- Запущен процесс Feedback: во ВСЕ связанные файлы 01_discuss/ добавлена запись об изменении
- Обновлен статус документа



### Диаграммы (Diagrams)

Диаграммы в папке 03_diagrams/

**Назначение:** Хранение диаграмм, если для файла архитектуры требуется диаграмма .drawio

**Требования к файлам:**
- Именование: [ID]_[название_компонента].drawio (например: 001_system_overview.drawio)
- Уникальный ID: Последовательная нумерация (001, 002, 003...)
- Обязательная ссылка: на ВСЕ связанные дискуссии (все обсуждения, на которых основана архитектура)
```
general_docs/
│
├── 02_architecture/                       # Архитектура
│   ├── 001_system_overview.md
│   ├── 002_authorization_module.md
│   └── 003_data_pipeline.md
│
└── 03_diagrams/                           # Диаграммы
    ├── 001_system_overview/            # Диаграммы архитектуры 001
    │   └── 001-diagram.drawio          # Диаграмма 001 для архитектуры 001
    └── 002_authorization_module/
```

**Обязательные действия при создании диаграммы:**
- Создать файл в папке `03_diagrams/[ID]_[название_архитектуры]/`
- Добавить легенду и подписи элементов
- Добавить ссылку на диаграмму в соответствующий файл архитектуры

#### Рабочий процесс с файлами диаграмм

**Статусы файлов диаграмм:**
- in_progress (Работа над файлом)
- final (Файл реализован в соответствии с требованиями, возврат к работе над архитектурой)

**Workflow:** in_progress → final
**Важно:** статус может измениться final → in_progress при работе с уточнением документации

#### Правила обратной связи FeedBack
**При создании нового ресурса по некоторой архитектуре:**
- Ресурс должен быть добавлен в general_docs/05_resources/[типы_ресурсов]
- Файл архитектуры должен быть дополнен ссылкой на новый ресурс
- Запустить FeedBack для всех связанных с файлом архитектуры дискуссий

**Требования к файлам:**
- Именование: [ID]_[plan]_[название_компонента].md (например: 001_plan_system_overview.md)
- Уникальный ID: Последовательная нумерация (001, 002, 003...)
- Обязательная ссылка: на ВСЕ связанные дискуссии (все обсуждения, на которых основана архитектура)

**Требования к файлам планов**:
- Ссылка на архитектуру: Обязательная ссылка на все связанные файлы архитектуры
- Декомпозиция: Разбиение на задачи с указанием:
    - Краткое название задачи
    - Описания задачи (что нужно сделать)
    - Статуса выполнения задачи (task_planned, task_in_progress, task_testing, task_done)
    - Зависимости от других задач/ресурсов

#### Рабочий процесс с файлами планов реализации

**Статусы файлов планов:**
- draft (Планирование задач)
- in_progress (Реализация задач)
- review (Ревью выполнения)
- test (Тестирование реализации)
- approved (Все задачи выполнены и протестированы)
- final (План завершён)

**Workflow:** draft → in_progress → review → test → approved → final
**Важно:** статус может вернуться на in_progress при обнаружении проблем на этапах review или test

## ПРАВИЛА ИНТЕГРАЦИИ И ОБНОВЛЕНИЯ

**Прямая [📖 цепочка зависимостей](../general_docs/glossary.md#цепочка-зависимостей):** [📖 Дискуссия](../general_docs/glossary.md#дискуссия) → Архитектура → [📖 Decision (ADR)](../general_docs/glossary.md#decision-adr) → [📖 Ресурсы](../general_docs/glossary.md#ресурс) → [📖 План реализации](../general_docs/glossary.md#план-реализации) → [📖 Документация папок](../general_docs/glossary.md#документация-папок)
**Обратная [📖 цепочка зависимостей](../general_docs/glossary.md#цепочка-зависимостей):** [📖 Документация папок](../general_docs/glossary.md#документация-папок) → [📖 План реализации](../general_docs/glossary.md#план-реализации) → [📖 Ресурсы](../general_docs/glossary.md#ресурс) → [📖 Decision (ADR)](../general_docs/glossary.md#decision-adr) → Архитектура → [📖 Дискуссия](../general_docs/glossary.md#дискуссия)

**Правила обновления для изменений в документах:**
- Изменение [📖 Decision (ADR)](../general_docs/glossary.md#decision-adr):
    - Обязательно обновить все связанные дискуссии и архитектуру
    - Проверить актуальность ссылок на ресурсы и планы
    - Если решение устарело — создать новую версию, старую переместить в archive/
- Изменение архитектуры:
    - Обновить связанные Decision (ADR)
    - Проверить актуальность ссылок на дискуссии
- Добавление/изменение/удаление ресурса:
    - Обязательно обновить связанный Decision (ADR) и архитектуру
    - При необходимости обновить документацию папок
- Изменение реализации:
    - Обновить документацию папки
    - При существенных изменениях — обновить описание ресурса и Decision (ADR)

**Правила обновления для изменений в коде (папке) и ресурсах:**
- **Критерий:** Если бы новый разработчик читал только общую документацию, был бы он введён в заблуждение текущим кодом?
- **Правила:**
    - Если изменение в коде (папке) затрагивает ответственность ресурса (например, изменился контракт API):
        - Обязательно обновить соответствующий файл в general_docs/05_resources/
    - Если изменение ресурса меняет архитектурные допущения (например, добавилась новая внешняя зависимость):
        - Запустить процесс feedback для соответствующего файла 02_architecture/
    
**Контроль целостности:**
- Проверка ссылок: Все ссылки должны быть относительными и рабочими
- Консистентность именования:
    - Единый стиль ID (001, 002...)
    - Соответствие имен файлов именам в кодовой базе
- Версионность:
    - Дата последнего обновления в каждом файле
    - История изменений для архитектурных решений
- Автоматизация:
    - При переименовании файлов обновлять все ссылки на них
    - Генерация диаграмм зависимостей

**Правила версионирования документов:** Формат версии: MAJOR.MINOR (например, 1.0, 1.1, 2.0)

- **Инкремент MAJOR (1.0 → 2.0):**
    - Существенное изменение архитектуры компонента
    - Изменение публичного API (breaking change)
    - Удаление или переименование ключевых сущностей
- **Инкремент MINOR (1.0 → 1.1):**
    - Добавление новой функциональности без изменения существующей
    - Уточнение описаний и диаграмм
    - Исправление ошибок в документации
    - Добавление примеров
- **Не требует изменения версии:**
    - Исправление опечаток
    - Форматирование


**Формат ссылок:**
- `[Термин](../../general_docs/glossary.md#термин)` — ссылка на термин в глоссарии
- `[См. раздел](../02_architecture/документ.md#раздел)` — ссылка на конкретный раздел документа
- Запретить абсолютные пути и URL (кроме внешних ссылок)

## Проверка документации

### Автоматическая проверка

Проект включает скрипты для автоматической проверки здоровья документации:

**Команды Makefile:**
```bash
make docs-health       # Полная проверка документации (ссылки, структура, статусы, метаданные)
make docs-links        # Проверка только ссылок
make docs-structure    # Проверка только структуры
make docs-status       # Проверка только статусов документов
make docs-metadata     # Проверка только метаданных
make docs-markdown     # Проверка форматирования markdown
make gloss-health      # Проверка глоссария
make gloss-unused      # Поиск неиспользуемых терминов в глоссарии
make docs-check        # Полная проверка (документация + глоссарий)
```

**Скиллы Claude Code:**
- `/doc-review` — смысловое ревью документа на качество и полноту (первый в workflow)
- `/doc-health` — техническая проверка документации (ссылки, структура, метаданные)
- `/doc-claude` — синхронизация CLAUDE.md и llm_instructions.md
- `/glossary-candidates` — найти термины-кандидаты для добавления в глоссарий
- `/glossary-review` — интерактивно обработать кандидатов в глоссарий
- `/glossary-link` — добавить ссылки на глоссарий во все .md файлы проекта

**Что проверяется:**

1. **Ссылки (docs-links):**
   - Ссылки на несуществующие файлы
   - Ссылки на несуществующие якоря (#заголовки)
   - Корректность относительных путей

2. **Структура (docs-structure):**
   - Наличие обязательных README.md в папках сервисов
   - Наличие обязательных разделов в документах
   - Соответствие шаблонам

3. **Статусы (docs-status):**
   - Корректность статусов документов (draft, in_progress, review, approved, final)
   - Соответствие workflow статусов

4. **Метаданные (docs-metadata):**
   - Наличие дат создания/обновления
   - Корректность формата версий
   - Наличие связанных документов

5. **Глоссарий (gloss-health):**
   - Все ли термины с [📖 иконкой] существуют в glossary.md
   - Корректность ссылок на глоссарий
   - Структура определений терминов
   - Размер глоссария (мониторинг порогов)

**Рекомендации:**
- Запускать `make docs-check` перед коммитом изменений в документации
- Использовать `/doc-health` для быстрой проверки и получения рекомендаций
- При добавлении новых терминов использовать `/glossary-candidates` и `/glossary-review`

## Критерии качества документации

- **Полнота:** Все значимые решения задокументированы
- **Актуальность:** Документация синхронизирована с кодом
- **Связность:** Все ссылки рабочие, цепочки зависимостей не разорваны
- **Читаемость:** Структурировано, ясно, достаточно для понимания LLM
- **Поддерживаемость:** Легко вносить изменения, не нарушая целостность
- **Проверяемость:** Регулярная автоматическая проверка через `make docs-check`

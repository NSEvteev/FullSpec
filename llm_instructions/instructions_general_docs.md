## Виды документации

### Файл глоссария glossary.md
- **Назначение:** Глоссарий для всей терминологии проекта
- **Расположение:** general_docs/glossary.md
- **Аудитория:** Все участники

Глоссарий должен содержать все термины, которые вводятся в проекте.
Термины должны быть оформлены в виде ссылок, на которые должны ссылаться все *.md файлы (например, architecture/01_system_overview.md и все *_doc.md).

**Пример описания термина:**
*   **Определение:** (Краткое, чёткое определение).
*   **Контекст:** (В каком модуле/архитектуре используется).
*   **Синонимы:** (Если есть).
*   **Антонимы:** (Если есть).
*   **Связанные термины:** (Ссылки на другие статьи глоссария).
*   **Источник:** (Ссылка на `discuss` или `architecture` файл, где термин был введён и утверждён).


### [📖 Общая документация](../general_docs/glossary.md#общая-документация)
- **Назначение:** Управление жизненным циклом функциональности — от идеи до [📖 плана реализации](../general_docs/glossary.md#план-реализации).
- **Расположение:** general_docs/ (в корне репозитория)
- **Содержание:** [📖 Дискуссии](../general_docs/glossary.md#дискуссия), архитектура, [📖 планы реализации](../general_docs/glossary.md#план-реализации), описания [📖 ресурсов](../general_docs/glossary.md#ресурс).
- **Аудитория:** Архитекторы, менеджеры, аналитики

### [📖 Документация папок](../general_docs/glossary.md#документация-папок)
- **Назначение:** Локальная документация модулей с ссылками на [📖 общую документацию](../general_docs/glossary.md#общая-документация).
- **Содержание:** Специфика модуля, связи с общей документацией, технические детали.
- **Аудитория:** Архитекторы, разработчики, тестировщики

### [📖 Дискуссии](../general_docs/glossary.md#дискуссия) (discuss/)

**Назначение:** Фиксация обсуждений "пользователь ↔ LLM" о функциональности.

**Требования к файлам:**
- Правило наименования: ["ID_краткое_описание"].md (например: 001_user_auth_and_roles.md)
- Уникальный ID: Последовательная нумерация (001, 002, 003...)

**Обязательные разделы:**
- Диалог с запросом пользователя
- Предложения по реализации от LLM
- Принятые решения с аргументацией принятого решения
- Ссылки на связанные документы архитектуры, создаваемые и/или изменяемые в рамках данной дискуссии

#### Шаблон файла [📖 дискуссии](../general_docs/glossary.md#дискуссия)
- **ID, название и метаданные:** `[ID]_[краткое_описание].md` и Статус
- **Исходный запрос пользователя:** Дословно.
- **Принятые архитектурные решения:** Список действий и ссылки на создаваемые файлы `architecture/`.
- **Анализ цели и контекст:** Какую проблему/возможность обсуждаем?
- **Принятое решение:** Четкая формулировка итогового выбора.
- **Аргументация:** Почему выбрали именно этот вариант?
- **Предложения LLM/архитектора:** Варианты решений с плюсами/минусами в формате таблицы со статусами ([ПРИНЯТ], [ОТКЛОНЕН]). Для отклонённых вариантов в разделе "Аргументация" кратко указывать причину отказа.
- **Дискуссия и анализ:** Критика, вопросы, уточнения.
- **Добавления к словарю глоссария:**.

#### Рабочий процесс с файлами [📖 дискуссий](../general_docs/glossary.md#дискуссия)

**Статусы файлов [📖 Дискуссий](../general_docs/glossary.md#дискуссия):**
- draft (Фиксация идеи)
- in_progress (Обсуждение) ИЛИ [📖 feedback](../general_docs/glossary.md#feedback) (фиксация изменений из архитектуры)
- review (Фиксация решения)
- approved (Создание файла архитектуры (если необходимо), создание ссылок на архитектуру)
- final (Файл реализован в соответствии c требованиями на момент завершения, переход к работе над архитектурой)

**[📖 Workflow](../general_docs/glossary.md#workflow-статусов):** draft → in_progress / feedback → review → approved → final
**Важно:** статус может измениться final → [📖 feedback](../general_docs/glossary.md#feedback) при работе с уточнением документации

#### Правила обратной связи [📖 FeedBack](../general_docs/glossary.md#feedback)
**При изменении файлов архитектуры, которые имеют ссылки на файлы дискуссий:**
- Обновить ссылки в связанных файлах дискуссий
- Дополнить документы связанных дискуссий с фиксацией:
-   Какие части тел документов изменены
-   Причины изменений
-   Дата обновления

**Чек-лист проверки изменений:** 
- Обновлен файл дискуссий
- Проверены и обновлены все ссылки на ресурсы `architecture/`
- Обновлен статус документа


### Архитектура (architecture/)

**Назначение:** Планирование архитектуры на основе дискуссий.

**Требования к файлам:**
- Именование: [ID]_[название_компонента].md (например: 001_system_overview.md)
- Уникальный ID: Последовательная нумерация (001, 002, 003...)
- Обязательная ссылка: на ВСЕ связанные дискуссии (все обсуждения, на которых основана архитектура)

**Обязательные действия в файле архитектуры (последовательность):** 
- Планирование архитектуры:
    - Диаграммы компонентов
    - Описание взаимодействий
    - Примеры использования
    - Границы ответственности
- Выделение состава ресурсов:
    - Перечень необходимых ресурсов
    - Классификация по типам ресурсов (см. [типы_ресурсов] на данной странице)
- Создание описаний ресурсов:
    - Если описание отсутствует → создать файл в resources/[типы_ресурсов]/ (см. [типы_ресурсов] на данной странице)
    - Имя файла должно соответствовать имени ресурса
- Создание плана реализации:
    - Создать файл в imp_plans/
    - Добавить ссылку на созданный план в файл архитектуры

#### Шаблон файла архитектуры
- **Цель и контекст компонента**
- **Ссылки:** На `discuss/`, `resources/`, `imp_plans/`.
- **Оглавление документа со ссылками на пункты документа**
- **Принципы и ограничения** (например, CAP-теорема, compliance).
- **Диаграммы:** C4-уровни (Context, Container, Component), UML-последовательности. Если нужна диаграмма, её можно создать в `diagrams/` в соответствие со следующими условиями:
    - Формат: Предпочтительно Mermaid (встроен в Markdown)
    - Инструменты: Draw.io (.drawio)
    - Расположение: В папке general_docs/diagrams
    - Требования: Должны быть самодостаточны (подписи, легенда)
    - Ссылку на файл диаграмы (если он был создан) необходимо добавить в файл архитектуры
- **Интерфейсы и контракты:** API (OpenAPI/Swagger), схемы сообщений, форматы событий.
- **Данные:** Модели, стратегии доступа, миграции.
- **Внешние зависимости:** Сервисы, библиотеки, API.
- **Требования к качеству:** (Нефункциональные требования): доступность, производительность, безопасность.
- **Риски и компромиссы.**
- **История изменений (только для файлов архитектуры)** Представляет из себя таблицу вида:
```## История изменений (Changelog)
| Версия | Дата | Автор | Описание изменений | Связанный Issue/PR |
|--------|------|-------|-------------------|-------------------|
| 1.0 | 2023-10-27 | AI Архитектор | Первоначальное создание | discuss/001_* |
| 1.1 | 2023-10-28 | Петров И.И. | Уточнены требования к БД | feedback #2 |
```

#### Рабочий процесс с файлами архитектуры

**Статусы файлов Архитектуры:**
- draft (Планирование работы над архитектурой)
- in_progress (Работа над архитектурой) ИЛИ feedback (фиксация изменений из ресурсов)
- review (Ревью архитектуры)
- approved (Создание (или изменение) описаний ресурсов и Плана реализации)
- final (Файл реализован в соответствии с требованиями, переход к работе над ресурсами)

**[📖 Workflow](../general_docs/glossary.md#workflow-статусов):** draft → in_progress / feedback → review → approved → final
**Важно:** статус может измениться final → [📖 feedback](../general_docs/glossary.md#feedback) при работе с уточнением документации

#### Правила обратной связи [📖 FeedBack](../general_docs/glossary.md#feedback)
**При создании нового [📖 ресурса](../general_docs/glossary.md#ресурс) по некоторой архитектуре:**
- [📖 Ресурс](../general_docs/glossary.md#ресурс) должен быть добавлен в general_docs/resources/[типы_ресурсов]
- Файл архитектуры должен быть дополнен ссылкой на новый ресурс
- Запустить FeedBack для всех связанных с файлом архитектуры дискуссий

**Чек-лист проверки изменений:**
- Обновлен файл архитектуры
- Проверены и обновлены все ссылки на ресурсы `discuss/`, `resources/`, `imp_plans/`
- Запущен процесс Feedback: во ВСЕ связанные файлы discuss/ добавлена запись об изменении
- Обновлен статус документа

- Обязательная ссылка: на ВСЕ связанные дискуссии (все обсуждения, на которых основана архитектура)

general_docs/
│
├── architecture/                       # Архитектура
│   ├── 001_system_overview.md
│   ├── 002_authorization_module.md
│   └── 003_data_pipeline.md
│
└── diagrams/                           # Диаграммы
    ├── 001_system_overview/            # Диаграммы архитектуры 001
    │   └── 001-diagram.drawio          # Диаграмма 001 для архитектуры 001

**Обязательные действия при создании диаграммы:**
- Создать файл в папке `diagrams/[ID]_[название_архитектуры]/`

- Добавить ссылку на диаграмму в соответствующий файл архитектуры

#### Рабочий процесс с файлами диаграмм

**Статусы файлов диаграмм:**

**Workflow:** in_progress → final
**Важно:** статус может измениться final → in_progress при работе с уточнением документации

- Файл архитектуры должен быть дополнен ссылкой на новый ресурс
- Запустить FeedBack для всех связанных с файлом архитектуры дискуссий

**Чек-лист проверки изменений:**
- Обновлен файл архитектуры
- Проверены и обновлены все ссылки на ресурсы `discuss/`, `resources/`, `imp_plans/`

- Обновлен статус документа



### [📖 План реализации](../general_docs/glossary.md#план-реализации) (imp_plans/)

**Назначение:** Декомпозиция и отслеживание выполнения работ.

**Требования к файлам:**
- Именование: [ID]_[plan]_[название_компонента].md (например: 001_plan_system_overview.md)
- Уникальный ID: Последовательная нумерация (001, 002, 003...)
- Обязательная ссылка: на ВСЕ связанные дискуссии (все обсуждения, на которых основана архитектура)

**Требования к файлам планов**:
- Ссылка на архитектуру: Обязательная ссылка на все связанные файлы архитектуры
- Декомпозиция: Разбиение на задачи с указанием:
    - Краткое название задачи
    - Описания задачи (что нужно сделать)
    - Статуса выполнения задачи (task_planned, task_in_progress, task_testing, task_done)
    - Зависимости от других задач/ресурсов

#### Рабочий процесс с файлами планов реализации

**Статусы файлов планов:**
- draft (Планирование задач)
- in_progress (Реализация задач)
- review (Ревью выполнения)
- test (Тестирование реализации)
- approved (Все задачи выполнены и протестированы)
- final (План завершён)

**Workflow:** draft → in_progress → review → test → approved → final
**Важно:** статус может вернуться на in_progress при обнаружении проблем на этапах review или test

### [📖 Ресурсы](../general_docs/glossary.md#ресурс) (resources/)

**Типы_ресурсов**:
- database
- backend
- frontend
- infra

**Управление типами ресурсов:** Если требуется отнести ресурс к типу, которого не существует, требуется:
- Спросить у пользователя, можно ли создать новый тип ресурсов
    - Если ответ "нет", то спросить у пользователя, к какому типу ресурсов отнести исследуемый ресурс
    - Если ответ "да", то нужно дополнить данный файл: 
        - Создать новый тип_ресурсов 
        - Обновить структуру
        - Добавить новый блок "Требования к полноте описаний"
        - Создать папку resources/[новый_тип]/

**Назначение:** Детальное описание всех ресурсов системы.

**Структура:**
```
resources/
├── resource_requirements.md        # Требования к полноте описания ВСЕХ ресурсов
│
├── database/                       # Ресурсы баз данных
│   ├── SQLite.md
│   ├── PostgreSQL.md
│   └── DuckDB.md
│
├── backend/                        # Бэкенд-ресурсы
│   ├── api_gateway.md
│   └── user_service.md
│
├── frontend/                       # Фронтенд-ресурсы
│   ├── web_app.md
│   └── admin_panel.md
│
└── infra/                          # Инфраструктурные ресурсы
    ├── kubernetes_cluster.md
    └── redis_cache.md
```

**Каждая категория содержит**: *_doc_requirements.md — требования к описанию, которые содержат требования к полноте описания. 

**Требования к файлам:**
- Все описания ресурсов должны содержать перечень включенных в ресурс модулей, функций и т.д. в виде, достаточным для понимания функционала для LLM
- Локальная специфика после начала разработки должна храниться в документах структуры "Документация_папок"

#### Требования к файлам полноты описаний типов ресурсов:

**Каждый файл в general_docs/* должен исполнять общие требования к документации:**
- Название файла должно отсылаться к кодовой базе 
- Полнота описания: Достаточная для понимания функционала LLM
- Перечень компонентов: Все модули, функции, классы, включенные в ресурс
- Локальная специфика: После начала разработки переносится в "Документацию папок"
- Соответствие требованиям: Каждый файл должен соответствовать *_doc_requirements.md своей категории

**Каждый файл в general_docs/* в зависимости от своего типа_ресурсов должен исполнять требования к документации своего типа:**

##### Требования к полноте описаний resources/database
- Схема данных: Таблицы, связи, индексы
- Миграции: Стратегия управления изменениями схемы
- Запросы: Критические запросы и их оптимизация
- Резервное копирование: Стратегия и процедуры
- Производительность: Бенчмарки, мониторинг

##### Требования к полноте описаний resources/backend
- API: Endpoints, методы, форматы запросов/ответов
- Бизнес-логика: Ключевые алгоритмы и процессы
- Интеграции: Внешние сервисы и протоколы
- Конфигурация: Настройки окружений
- Масштабирование: Стратегия горизонтального/вертикального

##### Требования к полноте описаний resources/frontend
- Компоненты: UI-компоненты и их свойства
- Состояние: Управление состоянием приложения
- Маршрутизация: Структура URL и переходы
- Стили: Дизайн-система, темы
- Производительность: Оптимизация загрузки, рендеринг

##### Требования к полноте описаний resources/infra
- Развертывание: Процедуры деплоя
- Мониторинг: Метрики, логи, алерты
- Безопасность: Доступы, сертификаты, политики
- Резервирование: Стратегия отказоустойчивости
- Масштабирование: Auto-scaling правила


## Структура "[📖 Документация папок](../general_docs/glossary.md#документация-папок)"

**Расположение:** В корне каждой значимой папки разработки:
- Папка представляет собой отдельный модуль/сервис (например, services/auth/, apps/web/).
- Папка содержит код с публичным API (интерфейсы, которые используются другими модулями).
- Папка имеет сложную внутреннюю структуру, требующую пояснения.
- Исключение: Папки кэша, папки внутри general_docs/, папки наподобие "__pycache__" — не значимые.
**Именование:** README.md
**Пример:** services/auth/README.md, apps/web/README.md, packages/shared/README.md

```
services/
├── auth/
│   ├── src/

## ПРАВИЛА ИНТЕГРАЦИИ И ОБНОВЛЕНИЯ

**Прямая [📖 цепочка зависимостей](../general_docs/glossary.md#цепочка-зависимостей):** [📖 Дискуссия](../general_docs/glossary.md#дискуссия) → Архитектура → [📖 Ресурсы](../general_docs/glossary.md#ресурс) → [📖 План реализации](../general_docs/glossary.md#план-реализации) → [📖 Документация папок](../general_docs/glossary.md#документация-папок)
**Обратная [📖 цепочка зависимостей](../general_docs/glossary.md#цепочка-зависимостей):** [📖 Документация папок](../general_docs/glossary.md#документация-папок) → [📖 План реализации](../general_docs/glossary.md#план-реализации) → [📖 Ресурсы](../general_docs/glossary.md#ресурс) → Архитектура → [📖 Дискуссия](../general_docs/glossary.md#дискуссия)

**Правила обновления для изменений в документах:**
- Изменение архитектуры: 
    - Обязательно обновить все связанные дискуссии
    - Проверить актуальность ссылок на ресурсы и планы
- Добавление/изменение/удаление ресурса: 
    - Обязательно обновить связанную архитектуру
    - При необходимости обновить документацию папок
- Изменение реализации: 
    - Обновить документацию папки
    - При существенных изменениях — обновить описание ресурса

**Правила обновления для изменений в коде (папке) и ресурсах:**
- **Критерий:** Если бы новый разработчик читал только общую документацию, был бы он введён в заблуждение текущим кодом?
- **Правила:**
    - Если изменение в коде (папке) затрагивает ответственность ресурса (например, изменился контракт API):
        - Обязательно обновить соответствующий файл в general_docs/resources/
    - Если изменение ресурса меняет архитектурные допущения (например, добавилась новая внешняя зависимость):
        - Запустить процесс feedback для соответствующего файла architecture/
    
**Контроль целостности:**
- Проверка ссылок: Все ссылки должны быть относительными и рабочими
- Консистентность именования:
    - Единый стиль ID (001, 002...)
    - Соответствие имен файлов именам в кодовой базе
- Версионность:
    - Дата последнего обновления в каждом файле
    - История изменений для архитектурных решений
- Автоматизация:
    - При переименовании файлов обновлять все ссылки на них
    - Генерация диаграмм зависимостей

**Правила версионирования документов:** Формат версии: MAJOR.MINOR (например, 1.0, 1.1, 2.0)

- **Инкремент MAJOR (1.0 → 2.0):**
    - Существенное изменение архитектуры компонента
    - Изменение публичного API (breaking change)
    - Удаление или переименование ключевых сущностей
- **Инкремент MINOR (1.0 → 1.1):**
    - Добавление новой функциональности без изменения существующей
    - Уточнение описаний и диаграмм
    - Исправление ошибок в документации
    - Добавление примеров
- **Не требует изменения версии:**
    - Исправление опечаток
    - Форматирование

## Критерии качества документации

- **Полнота:** Все значимые решения задокументированы
- **Актуальность:** Документация синхронизирована с кодом
- **Связность:** Все ссылки рабочие, цепочки зависимостей не разорваны
- **Читаемость:** Структурировано, ясно, достаточно для понимания LLM
- **Поддерживаемость:** Легко вносить изменения, не нарушая целостность

# Ревью решения дискуссии

Скилл для **критического анализа выбранного решения** — помогает пользователю понять последствия выбора.

## Связанная документация

| Документ | Назначение |
|----------|------------|
| [general_docs.md](../../../llm_instructions/general_docs.md#дискуссии-discuss) | Правила ведения дискуссий |
| [templates/discuss.md](../../../llm_instructions/templates/discuss.md) | Шаблон дискуссии |
| [discussion skill](../discussion/SKILL.md) | Скилл создания дискуссий |
| [doc-review skill](../doc-review/SKILL.md) | Скилл ревью документов |

## Когда использовать

**Триггеры:**
- Команда `/discussion-review`
- После выбора варианта в дискуссии (автоматически из doc-review)
- Явный запрос "ревью решения" или "что я смогу/не смогу сделать"

**Контекст:**
- Дискуссия находится в статусе `🔵 in_progress`
- Пользователь выбрал один из вариантов

## Workflow статусов

```
🔵 in_progress  → Пользователь выбирает вариант
      ↓
🟣 review       → Ревью решения (этот скилл!)
      ↓
🟢 approved     → Решение принято после ревью
      ↓
/summary-doc        → Обновление 000_SUMMARY.md (агрегация решений)
      ↓
/architect      → Создание архитектуры (читает SUMMARY + дискуссию)
      ↓
⚪ final        → Архитектура создана, дискуссия завершена
```

**ВАЖНО:** Этот скилл отвечает за переход `in_progress → review → approved`. После подтверждения вызывается `/summary-doc`, затем `/architect`.

## Инструкции

### Шаг 1: Прочитать дискуссию и выбранный вариант

Получить из контекста:
- Какой вариант выбрал пользователь
- Критерии оценки и их веса
- Сравнительную таблицу

### Шаг 2: Провести анализ последствий

**Анализировать по структуре:**

1. **Что вы СМОЖЕТЕ сделать:**
   - Перечислить возможности, которые даёт выбранный вариант
   - Связать с критериями, где вариант получил высокие оценки (⭐⭐⭐)
   - Указать конкретные примеры использования

2. **Что вы НЕ СМОЖЕТЕ сделать (или будет сложно):**
   - Перечислить ограничения выбранного варианта
   - Связать с критериями, где вариант получил низкие оценки (⭐ или —)
   - Указать, как это повлияет на проект

3. **Риски и как их минимизировать:**
   - Технические риски
   - Организационные риски
   - План mitigation для каждого риска

4. **Сравнение с альтернативами:**
   - Что вы потеряете по сравнению с топ-альтернативами
   - Что вы получите по сравнению с топ-альтернативами

### Шаг 3: Записать ревью в документ

**ВАЖНО:** Ревью записывается в секцию "Ревью решения" документа дискуссии!

1. Найти секцию `## Ревью решения`
2. Заменить `[Ожидает выбора варианта]` на структурированный анализ
3. Вывести пользователю краткое summary

**Формат записи в документ:**

```markdown
## Ревью решения: [Название варианта]

### Что вы СМОЖЕТЕ сделать

1. **[Возможность 1]** — [описание] (связано с критерием [X])
2. **[Возможность 2]** — [описание]
3. **[Возможность 3]** — [описание]

### Что вы НЕ СМОЖЕТЕ сделать (или будет сложно)

1. **[Ограничение 1]** — [описание] (связано с критерием [X])
2. **[Ограничение 2]** — [описание]

### Риски и mitigation

| Риск | Вероятность | Влияние | Как минимизировать |
|------|-------------|---------|-------------------|
| [Риск 1] | Высокая/Средняя/Низкая | Высокое/Среднее/Низкое | [План действий] |

### Сравнение с альтернативами

**Что вы теряете (по сравнению с [Альтернатива 1]):**
- [Что потеряете]

**Что вы получаете (по сравнению с [Альтернатива 2]):**
- [Что получите]

### Вывод

[Итоговая оценка: рекомендуется / с оговорками / не рекомендуется]
```

### Шаг 4: Спросить пользователя

После вывода отчёта спросить через `AskUserQuestion`:

```
Варианты:
1. Подтвердить решение → перейти в approved
2. Вернуться к обсуждению → остаться в in_progress
3. Выбрать другой вариант → провести ревью заново
```

### Шаг 5: Обновить документ по результатам

**Если пользователь подтвердил:**
1. Изменить статус: `🟣 review` → `🟢 approved`
2. Добавить запись в "История обсуждения": `[YYYY-MM-DD]: Решение подтверждено после ревью → approved`
3. Обновить индекс дискуссий (000_discuss.md)
4. **Вызвать `/summary-doc`** — обновить 000_SUMMARY.md с принятым решением
5. **Вызвать `/architect`** — создать архитектуру на основе SUMMARY и дискуссии

**Если пользователь вернулся к обсуждению:**
1. Изменить статус: `🟣 review` → `🔵 in_progress`
2. Добавить запись в "История обсуждения"
3. Очистить секцию "Ревью решения" (вернуть `[Ожидает выбора варианта]`)

## Пример

**Дискуссия:** Выбор UI-фреймворка
**Выбранный вариант:** Mantine

```markdown
## Ревью решения: Mantine

### Что вы СМОЖЕТЕ сделать

1. **Быстро создавать UI** — готовые компоненты с отличным DX (скорость разработки ⭐⭐⭐)
2. **Легко кастомизировать** — гибкая система стилей (кастомизация ⭐⭐⭐)
3. **Поддерживать dark/light mode** — темизация из коробки (темизация ⭐⭐⭐)
4. **Использовать с Next.js** — отличная поддержка SSR (SSR/SSG ⭐⭐⭐)
5. **Получить типобезопасность** — первоклассная поддержка TypeScript (TypeScript ⭐⭐⭐)

### Что вы НЕ СМОЖЕТЕ сделать (или будет сложно)

1. **Найти много готовых примеров** — сообщество меньше чем у MUI (сообщество ⭐⭐)
2. **Минимизировать bundle** — не самый лёгкий вариант (bundle size ⭐⭐)
3. **Использовать с Vue/Svelte** — только React

### Риски и mitigation

| Риск | Вероятность | Влияние | Как минимизировать |
|------|-------------|---------|-------------------|
| Меньше сообщество | Средняя | Среднее | Использовать официальную документацию, Discord канал |
| Bundle size | Низкая | Низкое | Tree-shaking, lazy loading компонентов |

### Сравнение с альтернативами

**Что вы теряете (по сравнению с MUI):**
- Меньше готовых компонентов
- Меньше примеров в интернете

**Что вы получаете (по сравнению с MUI):**
- Лучшая кастомизация
- Меньший bundle size
- Более современный API

### Вывод

Рекомендуется для проектов на React/Next.js, где важны скорость разработки и кастомизация. Ограничения (сообщество, bundle) некритичны.
```

## Чек-лист

При ревью всегда проверять:

- [ ] Связаны ли возможности/ограничения с конкретными критериями?
- [ ] Указаны ли реалистичные риски?
- [ ] Есть ли план mitigation для каждого риска?
- [ ] Понятно ли сравнение с альтернативами?
- [ ] Сделан ли однозначный вывод?

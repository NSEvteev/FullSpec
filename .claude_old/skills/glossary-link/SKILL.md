# Добавление ссылок на глоссарий

Команда проходит по всем .md файлам и добавляет ссылки на термины из глоссария.

## Формат ссылки на глоссарий

```markdown
[📖 Термин](../../general_docs/glossary.md#термин)
```

**Эмодзи `📖`** — визуально отличает ссылки на глоссарий от обычных ссылок.

## Инструкции

### Шаг 1: Прочитать глоссарий

Прочитать `general_docs/glossary.md` и собрать список всех терминов из раздела "## Термины".

Для каждого термина определить:
- Точное написание (с учётом регистра)
- Якорь для ссылки (lowercase, дефисы вместо пробелов)

### Шаг 2: Найти все .md файлы

Использовать Glob для поиска всех `.md` файлов:

```
**/*.md
```

**Исключить:**
- `general_docs/glossary.md` — сам глоссарий
- `README.md` — основной файл проекта (содержит отдельный раздел со ссылкой на глоссарий)
- `node_modules/`
- `.git/`

### Шаг 3: Обработать каждый файл

Для каждого .md файла:

#### 3.1: Найти вхождения терминов

Использовать Grep или Read для поиска терминов.

**Правила поиска:**
- Искать точные совпадения слов (не части слов)
- Учитывать различные формы: "Термин", "термин", "термины"
- Игнорировать термины в code blocks (``` ... ```)
- Игнорировать термины, которые уже являются ссылками

#### 3.2: Заменить на ссылки

**Заменять только первое вхождение** термина в каждом разделе (## или ###).

Формат замены:
```
Термин → [📖 Термин](путь/к/glossary.md#термин)
```

**Вычисление относительного пути:**
- Из `general_docs/02_architecture/doc.md` → `../glossary.md`
- Из `llm_instructions/file.md` → `../general_docs/glossary.md`

#### 3.3: Сохранить изменения

Использовать Edit tool для обновления файла.

### Шаг 4: Вывести результат

```
Обработано файлов: X
Добавлено ссылок: Y

Файлы с изменениями:
- general_docs/02_architecture/01_overview.md: 5 ссылок
- llm_instructions/general_docs.md: 3 ссылки

✓ Ссылки на глоссарий добавлены
```

## Важные правила

1. **Не заменять в заголовках** — термины в `#`, `##`, `###` не трогать
2. **Не заменять в code blocks** — внутри ``` ... ``` не трогать
3. **Не заменять уже существующие ссылки** — `[текст](url)` не трогать
4. **Первое вхождение в разделе** — повторные упоминания не делать ссылками
5. **Относительные пути** — всегда использовать относительные пути к glossary.md

## Откат изменений

Если нужно откатить:
```bash
git checkout -- **/*.md
```

## Примеры использования

### Пример 1: Добавление ссылок после обновления глоссария

**Сценарий:** В глоссарий добавлены 5 новых терминов из архитектурного документа (Decision (ADR), Ресурс, План реализации, Цепочка зависимостей, Документация папок). Нужно автоматически добавить ссылки на эти термины во всех существующих .md файлах.

**Команда:**
```bash
/glossary-link
```

**Ожидаемый результат:**
- Чтение всех терминов из `general_docs/glossary.md`
- Поиск всех .md файлов в проекте (кроме исключений)
- Автоматическая замена первых вхождений терминов на ссылки формата `[📖 Термин](путь/к/glossary.md#термин)`
- Отчёт о количестве добавленных ссылок

**Вывод (пример):**
```
Обработано файлов: 18
Добавлено ссылок: 47

Файлы с изменениями:
- general_docs/02_architecture/001_system_overview.md: 8 ссылок
- general_docs/04_decisions/DEC-001.md: 12 ссылок
- llm_instructions/general_docs.md: 15 ссылок
- llm_instructions/agents.md: 7 ссылок
- CLAUDE.md: 5 ссылок

Термины с наибольшим количеством ссылок:
- Decision (ADR): 12 ссылок
- Дискуссия: 10 ссылок
- Архитектура: 9 ссылок

✓ Ссылки на глоссарий добавлены
```

### Пример 2: Добавление ссылок в новый документ

**Сценарий:** Создан новый документ дискуссии `01_discuss/004_caching_strategy.md`. Документ содержит термины из глоссария, но ссылки не добавлены. Нужно добавить ссылки только в этот файл.

**Команда:**
```bash
/glossary-link
```
(После запуска скилла можно будет указать конкретный файл, если добавить параметр в SKILL.md)

**Ожидаемый результат:**
- Поиск терминов из глоссария в новом документе
- Замена первого вхождения каждого термина в каждом разделе на ссылку
- Относительный путь к glossary.md рассчитан корректно (`../glossary.md`)

**Вывод (пример):**
```
Обработано файлов: 1
Добавлено ссылок: 6

Файлы с изменениями:
- general_docs/01_discuss/004_caching_strategy.md: 6 ссылок
  - Архитектура (строка 15)
  - Decision (ADR) (строка 23)
  - Ресурс (строка 45)
  - План реализации (строка 67)
  - Документация папок (строка 89)
  - Агент (строка 102)

✓ Ссылки на глоссарий добавлены
```

### Пример 3: Проверка правильности путей после перемещения файлов

**Сценарий:** Файлы из `general_docs/resources/` перемещены в подпапки `05_resources/database/`, `05_resources/backend/` и т.д. Нужно убедиться, что ссылки на глоссарий обновились с правильными относительными путями.

**Команда:**
```bash
/glossary-link
```

**Ожидаемый результат:**
- Обработка всех файлов в новых подпапках
- Корректный расчёт относительных путей (из `05_resources/database/001_schema.md` → `../../glossary.md`)
- Проверка, что старые ссылки не задублировались

**Вывод (пример):**
```
Обработано файлов: 12
Добавлено ссылок: 0
Обновлено путей: 18

Файлы с обновлёнными путями:
- general_docs/05_resources/database/001_schema.md: 4 пути обновлены
- general_docs/05_resources/backend/002_api.md: 6 путей обновлены
- general_docs/05_resources/frontend/003_components.md: 5 путей обновлены

✓ Пути к глоссарию обновлены
✓ Новые ссылки не добавлены (термины уже были оформлены как ссылки)
```

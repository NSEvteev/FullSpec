# Проверка здоровья документации

Команда для комплексной проверки состояния документации проекта.

## Назначение

Скилл запускает скрипт `scripts/check_doc_health.py`, который проверяет:
1. Целостность ссылок (файлы и якоря)
2. Структуру документации (обязательные файлы и разделы)
3. Корректность статусов дискуссий/архитектуры
4. Наличие метаданных (даты, версии)
5. Markdown форматирование

## Инструкции

### Шаг 1: Запустить проверку

Выполни скрипт проверки:
```bash
python scripts/check_doc_health.py
```

**Опции:**
- `--verbose` или `-v` — подробный вывод процесса проверки
- `--check links` — только проверка ссылок
- `--check structure` — только проверка структуры
- `--check status` — только проверка статусов
- `--check metadata` — только проверка метаданных
- `--check markdown` — только проверка форматирования

### Шаг 2: Анализировать результаты

Скрипт выводит:
- **✓ Все проверки пройдены успешно!** — если проблем нет
- **✗ Найдены проблемы в документации:** — если есть ошибки

Для каждой проблемы указывается:
- Категория проблемы (BROKEN_LINKS, MISSING_SECTIONS, etc.)
- Файл и номер строки
- Описание проблемы

### Шаг 3: Предложить исправления

После вывода результатов:

1. **Если найдены проблемы:**
   - Сгруппировать проблемы по типу
   - Предложить план исправления
   - Спросить пользователя, исправить автоматически или вручную

2. **Если все в порядке:**
   - Сообщить об успешной проверке
   - Вывести краткую статистику (количество проверенных файлов)

### Шаг 4: Исправить проблемы (опционально)

Если пользователь согласен на автоматическое исправление:
- Исправить битые ссылки
- Добавить недостающие разделы
- Обновить статусы
- Добавить метаданные

**Важно:** Всегда спрашивать подтверждение перед изменением файлов!

## Примеры использования

### Пример 1: Полная проверка документации после создания новой дискуссии

**Сценарий:** Создан новый документ дискуссии `01_discuss/003_oauth_integration.md`. Нужно проверить, что документ соответствует стандартам, все ссылки работают, и метаданные заполнены корректно.

**Команда:**
```bash
/doc-health
```

**Ожидаемый результат:**
- Проверка всех .md файлов в `general_docs/`
- Валидация структуры нового документа
- Проверка ссылок на связанные документы
- Проверка метаданных (дата создания, статус, версия)

**Вывод (пример):**
```
Проверка документации проекта
Корневая директория: C:\00 Dev\project_template
============================================================

Проверено файлов: 23
Найдено ошибок: 0

✓ Все проверки пройдены успешно!
```

### Пример 2: Проверка только ссылок после рефакторинга структуры

**Сценарий:** Переименованы папки в `general_docs/` (добавлены префиксы `01_`, `02_` и т.д.). Нужно убедиться, что все ссылки между документами обновлены.

**Команда:**
```bash
/doc-health --check links
```

**Ожидаемый результат:**
- Проверка только целостности ссылок
- Быстрая проверка (пропускает остальные валидации)
- Список битых ссылок, если они есть

**Вывод (пример с проблемами):**
```

Корневая директория: C:\00 Dev\project_template
============================================================

✗ Найдены проблемы в документации:

============================================================

============================================================

  general_docs/02_architecture/001_auth_module.md:45
    Битая ссылка на файл: [Дискуссия](../discuss/001_auth.md)
    → Правильный путь: ../01_discuss/001_auth.md

  general_docs/04_decisions/DEC-001.md:23
    Битая ссылка на якорь: [Принятое решение](../01_discuss/001_auth.md#решение)
    → Якорь #решение не существует

============================================================
Всего проблем: 2
============================================================
```

### Пример 3: Проверка структуры нового типа документа

**Сценарий:** Добавлен новый раздел `general_docs/07_standards/` для хранения стандартов проекта. Нужно проверить, что структура папки соответствует правилам (есть `000_standards.md` индекс и `README.md`).

**Команда:**
```bash
/doc-health --check structure --verbose
```

**Ожидаемый результат:**
- Подробный вывод процесса проверки
- Проверка наличия обязательных файлов (`000_*.md`, `README.md`)
- Проверка обязательных разделов в документах

**Вывод (пример):**

## Связанные команды

- `/glossary-link` — добавление ссылок на глоссарий
- `/doc-project-structure` — обновление структуры проекта в документации
- `make docs-health` — проверка документации через Makefile

## Связанные файлы

- `scripts/check_doc_health.py` — основной скрипт проверки
- `scripts/check_gloss_health.py` — проверка глоссария (отдельный скрипт)
- `llm_instructions/general_docs.md` — правила ведения документации
- `llm_instructions/instructions_scripts.md` — описание скриптов

# Обновление SUMMARY дискуссий

Скилл для обновления файла `000_SUMMARY.md` дискуссий при переходе дискуссии в статус `🟢 approved`.

## Ключевая концепция: IT-сервисы

```
            ┌─────────────────────────────────────┐
            │          IT-СЕРВИС (SRV-XXX)        │
            │   general_docs/00_services/         │
            │   (бизнес-ценность для клиента)     │
            └─────────────────┬───────────────────┘
                              │
    ┌─────────────────────────┴─────────────────────────┐
    │           ТЕХНИЧЕСКАЯ РЕАЛИЗАЦИЯ                  │
    │                                                   │
    │  01_discuss → 02_architecture → 04_decisions →    │
    │  → 05_resources → 06_imp_plans                    │
    └───────────────────────────────────────────────────┘
```

**IT-сервис** — бизнес-сущность, которая:
- Описывает ЦЕННОСТЬ для клиента/бизнеса
- Объединяет связанные решения, компоненты, ресурсы
- Находится в `general_docs/00_services/[service-name]/`

**Этот скилл должен:**
1. Обновить стандартные секции SUMMARY (области, решения)
2. **Определить IT-сервис** — к какому сервису относится решение
3. **Обновить секцию IT-сервисы** в SUMMARY
4. **Обновить/создать сервис** в `00_services/`
5. **Добавить блок агрегации** в "Агрегация знаний по сервисам"

## Когда использовать

**Триггеры:**
- Команда `/summary-doc`
- Автоматически из `/discussion-review` при переходе `review → approved`

**Контекст:**
- Дискуссия переходит в статус `🟢 approved`
- Пользователь подтвердил решение после ревью

## Workflow

```
🟣 review → 🟢 approved → /summary-doc (этот скилл!) → /architect → ⚪ final
```

**ВАЖНО:** Этот скилл выполняется МЕЖДУ `approved` и `/architect`.

## Инструкции

### Шаг 1: Получить информацию о дискуссии

Из аргументов или контекста получить:
- Путь к файлу дискуссии (например: `general_docs/01_discuss/001_ui_staylgayd.md`)
- Или ID дискуссии (например: `001`)

### Шаг 2: Прочитать дискуссию и определить область

Извлечь из файла дискуссии:
1. **ID** — номер дискуссии (например: `001`)
2. **Тема** — заголовок из `## Исходный запрос пользователя` или H1
3. **Область** — определить по содержанию (см. правила ниже)
4. **Выбранный вариант** — из секции `## Принятое решение` → `**Выбран вариант:**`
5. **Ключевые аргументы** — из `**Аргументы за:**`
6. **Ограничения** — из секции `## Ревью решения` → `### Что вы НЕ СМОЖЕТЕ сделать`
7. **Влияние на другие области** — какие компоненты затрагивает решение

**Правила определения области:**

| Область | Ключевые слова и признаки |
|---------|--------------------------|
| **UI/Frontend** | UI, компоненты, стили, фреймворк (React, Vue), CSS, Tailwind, дизайн-система |
| **Backend/API** | API, сервис, endpoint, REST, GraphQL, сервер, бэкенд |
| **Auth/Security** | аутентификация, авторизация, JWT, OAuth, безопасность, пароли |
| **Database** | база данных, PostgreSQL, MongoDB, миграции, схема, ORM |
| **Infrastructure** | деплой, Docker, CI/CD, мониторинг, логирование, Kubernetes |
| **Architecture** | архитектура системы, микросервисы, паттерны, интеграция компонентов |

### Шаг 3: Обновить 000_SUMMARY.md

**Файл:** `general_docs/01_discuss/000_SUMMARY.md`

#### 3.1 Обновить таблицу "Быстрый обзор по областям"

Найти строку с соответствующей областью и увеличить счётчик:

```markdown
| UI/Frontend | 1 | [001] UI Styleguide |
```

Также обновить строку "Всего".

#### 3.2 Добавить строку в таблицу "Принятые решения"

Найти таблицу и добавить строку с указанием области:

```markdown
| [ID] | [Область] | [Тема] | [Выбранный вариант] | [Ключевые аргументы (кратко)] | [Ссылка] |
```

**Пример:**
```markdown
| 001 | UI/Frontend | UI Styleguide | Mantine | Скорость разработки, кастомизация, TypeScript | [001_ui_staylgayd.md](001_ui_staylgayd.md) |
```

#### 3.3 Добавить ссылку в секцию "Решения по областям"

Найти соответствующую секцию (например, `### UI/Frontend`) и добавить ссылку:

```markdown
### UI/Frontend

- **[001] UI Styleguide** — Mantine: скорость разработки и кастомизация. [Подробнее](001_ui_staylgayd.md)
```

Если там был текст "Решения в этой области отсутствуют." — удалить его.

#### 3.4 Добавить блок описания

В секцию "Краткое описание решений" добавить блок:

```markdown
### [ID] [Тема]

**Область:** [UI/Frontend | Backend/API | Auth/Security | Database | Infrastructure | Architecture]

**Решение:** [Название выбранного варианта]

**Суть:** [1-2 предложения из "Обоснование"]

**Ключевые аргументы:**
- [Аргумент 1]
- [Аргумент 2]

**Ограничения (что НЕ сможем):**
- [Ограничение 1]
- [Ограничение 2]

**Влияние на другие области:**
- [Какие области затрагивает это решение]

**Дискуссия:** [Ссылка на файл дискуссии]

---
```

#### 3.5 Удалить placeholder

Если в таблице есть строка `| - | - | - | - | - | - |`, удалить её.

Если есть текст "В настоящее время принятые решения отсутствуют.", удалить его.

### Шаг 4: Определить IT-сервис

Решение может:
1. **Относиться к существующему сервису** — добавить в его агрегацию
2. **Создавать новый сервис** — создать папку в `00_services/`
3. **Быть общим решением** — не привязывать к сервису

**Правила определения:**

| Признак | Действие |
|---------|----------|
| Упоминается существующий сервис (notification-service, auth-service, ...) | Добавить к существующему |
| Описывает новую бизнес-функцию (уведомления, платежи, аналитика, ...) | Создать новый сервис |
| Общее архитектурное решение (UI framework, код стандарты, CI/CD) | Не привязывать к сервису |

**Спросить пользователя если неясно:**
```
Решение [ID] описывает:
1. Существующий сервис: [список существующих]
2. Новый сервис: [предложенное имя]
3. Общее решение (не привязано к сервису)

К какому сервису относится это решение?
```

### Шаг 5: Обновить IT-сервисы (если применимо)

#### 5.1 Проверить существование сервиса

```bash
ls general_docs/00_services/[service-name]/
```

#### 5.2 Для НОВОГО сервиса

1. **Создать папку:**
   ```bash
   mkdir -p general_docs/00_services/[service-name]/
   ```

2. **Создать README.md:**
   ```markdown
   # [Service Name] (SRV-XXX)

   **Статус:** 🟡 planning

   ## Ценность

   [Что получает клиент/бизнес от этого сервиса]

   ## Документация

   | Тип | Документ | Статус |
   |-----|----------|--------|
   | Дискуссия | [ID] [Тема](../../01_discuss/XXX.md) | 🟢 approved |

   ## Компоненты

   <!-- Заполняется по мере создания ресурсов -->

   ## История изменений

   - **[YYYY-MM-DD]** — Создан сервис на основе дискуссии [ID]
   ```

3. **Обновить индекс** `00_services/000_services.md`

#### 5.3 Для СУЩЕСТВУЮЩЕГО сервиса

1. **Обновить README.md сервиса:**
   - Добавить строку в таблицу "Документация"
   - Обновить секцию "История изменений"

2. **Обновить индекс** если нужно

### Шаг 6: Обновить "Агрегация знаний по сервисам"

#### 6.1 Для НОВОГО сервиса

Добавить блок в секцию "Агрегация знаний по сервисам":

```markdown
### [service-name]

**Ценность:** [Что получает клиент/бизнес]

**Полное описание:** [00_services/[service-name]/README.md](../00_services/[service-name]/README.md)

#### Принятые решения

| ID | Решение | Суть |
|----|---------|------|
| [ID] | [Выбранный вариант] | [Краткое описание] |

#### Компоненты сервиса

| Область | Компонент | Назначение |
|---------|-----------|------------|
| [Backend/Frontend/Database/Infra] | [Название] | [Назначение] |

#### Ключевые ограничения

- [Из секции "Что НЕ сможем"]

#### Связанная документация

- **Дискуссия:** [ссылка]
- **Архитектура:** [создаётся позже]
- **ADR:** [создаётся позже]
- **План:** [создаётся позже]

---
```

#### 6.2 Для СУЩЕСТВУЮЩЕГО сервиса

1. **Добавить решение в таблицу** "Принятые решения"
2. **Дополнить компоненты** если появились новые
3. **Обновить ограничения** если появились новые
4. **Добавить ссылку** на дискуссию

### Шаг 7: Обновить таблицу "IT-сервисы"

Найти секцию "## IT-сервисы" и обновить таблицу:

```markdown
| Сервис | Ценность | Решения | Подробнее |
|--------|----------|---------|-----------|
| [service-name] | [Краткая ценность] | [ID1], [ID2] | [README](../00_services/[service-name]/README.md) |
```

Если сервис уже есть — добавить ID решения в колонку "Решения".

### Шаг 8: Вывести результат

**Если IT-сервис определён:**
```
SUMMARY обновлён для дискуссии [ID]:

📊 Решение:
- Область: [Область]
- Добавлено в таблицу "Принятые решения"
- Добавлена ссылка в секцию "[Область]"
- Обновлён счётчик в "Быстрый обзор по областям"

🏢 IT-сервис: [service-name]
- [Создан новый / Обновлён существующий] сервис
- Обновлена таблица "IT-сервисы"
- Обновлена секция "Агрегация знаний по сервисам"
- Файл сервиса: 00_services/[service-name]/README.md

Следующий шаг: /architect для создания архитектуры
```

**Если IT-сервис НЕ определён (общее решение):**
```

- Область: [Область]
- Добавлено в таблицу "Принятые решения"
- Добавлена ссылка в секцию "[Область]"
- Обновлён счётчик в "Быстрый обзор по областям"

ℹ️ Общее решение, не привязано к IT-сервису

Следующий шаг: /architect для создания архитектуры
```

## Пример

**Дискуссия:** `001_ui_staylgayd.md`

**Содержимое дискуссии (фрагмент):**
```markdown
## Принятое решение

**Выбран вариант:** Mantine

**Обоснование:**
Mantine — оптимальный выбор для проекта благодаря балансу между готовыми компонентами и гибкостью кастомизации.

**Аргументы за:**
- Скорость разработки: готовые компоненты ⭐⭐⭐
- Кастомизация: гибкая система стилей ⭐⭐⭐
- TypeScript: первоклассная поддержка ⭐⭐⭐

## Ревью решения

### Что вы НЕ СМОЖЕТЕ сделать (или будет сложно)

1. **Меньше готовых примеров** — сообщество меньше чем у MUI
2. **Bundle size** — не самый лёгкий вариант
```

**Определение области:** `UI/Frontend` (ключевые слова: UI, компоненты, стили, фреймворк)

**Результат в 000_SUMMARY.md:**

**Быстрый обзор по областям:**
```markdown
| Область | Количество | Ключевые решения |
|---------|------------|------------------|
| UI/Frontend | 1 | [001] UI Styleguide |
| Backend/API | 0 | — |
...
| **Всего** | **1** | — |
```

**Таблица "Принятые решения":**
```markdown
| ID | Область | Тема | Выбранный вариант | Ключевые аргументы | Ссылка |
|----|---------|------|-------------------|-------------------|--------|
| 001 | UI/Frontend | UI Styleguide | Mantine | Скорость разработки, кастомизация, TypeScript | [001_ui_staylgayd.md](001_ui_staylgayd.md) |
```

**Секция "UI/Frontend":**
```markdown
### UI/Frontend

- **[001] UI Styleguide** — Mantine: скорость разработки и кастомизация. [Подробнее](001_ui_staylgayd.md)
```

**Блок описания:**
```markdown
### 001 UI Styleguide

**Область:** UI/Frontend

**Решение:** Mantine

**Суть:** Mantine — оптимальный выбор для проекта благодаря балансу между готовыми компонентами и гибкостью кастомизации.

**Ключевые аргументы:**
- Скорость разработки: готовые компоненты
- Кастомизация: гибкая система стилей
- TypeScript: первоклассная поддержка

**Ограничения (что НЕ сможем):**
- Меньше готовых примеров — сообщество меньше чем у MUI
- Bundle size — не самый лёгкий вариант

**Влияние на другие области:**
- Backend: API должен возвращать данные в формате, удобном для Mantine компонентов

**Дискуссия:** [001_ui_staylgayd.md](001_ui_staylgayd.md)

---
```

## Чек-лист

При обновлении SUMMARY проверять:

**Основное:**
- [ ] Область определена корректно?
- [ ] Счётчик в "Быстрый обзор по областям" обновлён?
- [ ] Строка добавлена в таблицу "Принятые решения" с указанием области?
- [ ] Ссылка добавлена в соответствующую секцию "Решения по областям"?
- [ ] Placeholder удалён?
- [ ] Ссылка на дискуссию корректна?

**IT-сервисы:**
- [ ] IT-сервис определён (новый / существующий / общее решение)?
- [ ] Таблица "IT-сервисы" обновлена?
- [ ] Блок в "Агрегация знаний по сервисам" добавлен/обновлён?
- [ ] Папка сервиса в `00_services/` создана/обновлена?
- [ ] Индекс `000_services.md` актуален?
- [ ] README.md сервиса содержит ссылку на дискуссию?

**Содержание блока агрегации:**
- [ ] Ценность сервиса указана?
- [ ] Принятые решения перечислены?
- [ ] Компоненты по областям указаны?
- [ ] Ограничения из ревью включены?
- [ ] Ссылки на связанную документацию корректны?

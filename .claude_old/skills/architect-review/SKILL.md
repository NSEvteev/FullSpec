# Ревью архитектуры

Скилл для **критического анализа архитектурного документа** — проверка полноты, согласованности и готовности к созданию ADR.

## Связанная документация

## Когда использовать

**Триггеры:**
- Команда `/architect-review`
- После заполнения архитектуры (статус `🔵 in_progress`)
- Явный запрос "ревью архитектуры" или "проверить архитектуру"

**Контекст:**
- Архитектура находится в статусе `🔵 in_progress`
- Основные секции заполнены

## Workflow статусов

```
🟡 draft        → Архитектура создана, не заполнена
      ↓
🔵 in_progress  → Архитектура заполняется
      ↓
🟣 review       → Ревью архитектуры (этот скилл!)
      ↓
🟢 approved     → Архитектура одобрена
      ↓
/summary-arch   → Обновление 000_SUMMARY.md архитектуры
      ↓
/decision       → Создание ADR (читает ОБА SUMMARY)
      ↓
⚪ final        → ADR создан, архитектура завершена
```

**ВАЖНО:** Этот скилл отвечает за переход `in_progress → review → approved`. После одобрения вызывается `/summary-arch`, затем `/decision`.

## Инструкции

### Шаг 1: Прочитать архитектуру и связанную дискуссию

1. Прочитать архитектурный документ
2. Прочитать связанную дискуссию (для понимания контекста и выбранного решения)
3. Прочитать SUMMARY дискуссий (`01_discuss/000_SUMMARY.md`) для понимания других решений

### Шаг 2: Проверить структуру

**Обязательные секции:**

- [ ] Метаданные (статус, версия, дата)
- [ ] Цель и контекст (связь с дискуссией)
- [ ] Связанные документы (дискуссия, будущие ADR/ресурсы)
- [ ] Принципы и ограничения
- [ ] Компоненты (таблица с назначением)
- [ ] Диаграммы (или ссылки на 03_diagrams/)
- [ ] Интерфейсы и контракты (API)
- [ ] Данные (схема, хранение)
- [ ] Внешние зависимости
- [ ] Требования к качеству (SLA, производительность)
- [ ] Риски и компромиссы
- [ ] История изменений

### Шаг 3: Критический анализ

**Анализировать по структуре:**

1. **Полнота компонентов:**
   - Все ли необходимые компоненты описаны?
   - Есть ли упущенные компоненты (логирование, мониторинг, error handling)?
   - Соответствуют ли компоненты выбранному решению из дискуссии?

2. **Согласованность с IT-сервисом:**
   - Связана ли архитектура с конкретным сервисом в `00_services/`?
   - Покрывает ли архитектура все Outcomes сервиса?
   - Соответствует ли SLA сервиса требованиям архитектуры?

3. **Зависимости и интеграции:**
   - Описаны ли все внешние зависимости?
   - Есть ли план обработки failure внешних сервисов?
   - Описаны ли интерфейсы между компонентами?

4. **Данные и безопасность:**
   - Описана ли схема данных?
   - Учтены ли требования безопасности?
   - Есть ли план миграции данных (если применимо)?

5. **Масштабируемость и производительность:**
   - Есть ли требования к производительности?
   - Описан ли план масштабирования?
   - Учтены ли bottlenecks?

6. **Риски:**
   - Все ли риски идентифицированы?
   - Есть ли план mitigation для каждого риска?
   - Учтены ли технические долги?

### Шаг 4: Записать ревью в документ

**Добавить секцию "Ревью архитектуры" перед "История изменений":**

```markdown
## Ревью архитектуры

**Дата ревью:** [YYYY-MM-DD]
**Статус:** [Готов к ADR / Требует доработки]

### Проверка полноты

| Аспект | Статус | Комментарий |
|--------|--------|-------------|
| Компоненты | ✅/⚠️/❌ | [Комментарий] |
| Зависимости | ✅/⚠️/❌ | [Комментарий] |
| Интерфейсы | ✅/⚠️/❌ | [Комментарий] |
| Данные | ✅/⚠️/❌ | [Комментарий] |
| Безопасность | ✅/⚠️/❌ | [Комментарий] |
| Производительность | ✅/⚠️/❌ | [Комментарий] |
| Риски | ✅/⚠️/❌ | [Комментарий] |

### Связь с IT-сервисом

**Сервис:** [SRV-XXX название]
**Покрытие Outcomes:** [Все / Частично — какие не покрыты]

### Рекомендации

1. [Рекомендация 1]
2. [Рекомендация 2]

### Вывод

[Архитектура готова к созданию ADR / Требуется доработка: список]
```

### Шаг 5: Спросить пользователя

После вывода отчёта спросить через `AskUserQuestion`:

```
Варианты:
1. Одобрить архитектуру → перейти в approved, создать ADR
2. Доработать → остаться в in_progress
3. Отклонить → вернуться к дискуссии
```

### Шаг 6: Обновить документы по результатам

**Если пользователь одобрил:**
1. Изменить статус: `🟣 review` → `🟢 approved`
2. Добавить запись в "История изменений"
3. Обновить индекс архитектуры (000_architecture.md)
4. **Вызвать `/summary-arch`** — обновить 000_SUMMARY.md архитектуры
5. **Вызвать `/decision`** — создать ADR на основе ОБОИХ SUMMARY

**Если пользователь решил доработать:**
1. Изменить статус: `🟣 review` → `🔵 in_progress`
2. Добавить запись в "История изменений" с указанием, что требуется доработка
3. Перечислить конкретные пункты для доработки

**Если пользователь отклонил:**
1. Изменить статус: `🟣 review` → `🔴 rejected`
2. Добавить запись в "История изменений" с причиной отклонения
3. Возможно потребуется пересмотр дискуссии

## Пример

**Архитектура:** 001_sistema_email_uvedomleniy.md
**IT-сервис:** notification-service (SRV-001)

```markdown
## Ревью архитектуры

**Дата ревью:** 2026-01-16
**Статус:** Готов к ADR

### Проверка полноты

| Аспект | Статус | Комментарий |
|--------|--------|-------------|
| Компоненты | ✅ | Notification API, Email Worker, Redis Queue описаны |
| Зависимости | ✅ | Redis, SMTP, BullMQ указаны |
| Интерфейсы | ✅ | API endpoints описаны |
| Данные | ✅ | Таблицы email_queue, templates, logs описаны |
| Безопасность | ⚠️ | Нет описания защиты API ключей SMTP |
| Производительность | ✅ | SLA < 5 мин указан |
| Риски | ✅ | Redis failure, SMTP limits описаны |

### Связь с IT-сервисом

**Сервис:** SRV-001 notification-service
**Покрытие Outcomes:** Все — регистрация, восстановление пароля, изменения аккаунта, системные уведомления, настройки

### Рекомендации

1. Добавить описание защиты SMTP credentials (env vars, secrets manager)
2. Рассмотреть добавление rate limiting для API

### Вывод

Архитектура готова к созданию ADR. Рекомендации можно учесть при реализации.
```

## Чек-лист

При ревью архитектуры проверять:

- [ ] Все компоненты из выбранного решения описаны?
- [ ] Архитектура связана с IT-сервисом?
- [ ] Покрыты ли все Outcomes сервиса?
- [ ] Описаны ли внешние зависимости?
- [ ] Есть ли описание интерфейсов (API)?
- [ ] Описана ли схема данных?
- [ ] Учтены ли требования безопасности?
- [ ] Есть ли требования к производительности?
- [ ] Идентифицированы ли риски?
- [ ] Есть ли план mitigation для рисков?

- `/decision` — создание ADR (вызывается после /summary-arch)

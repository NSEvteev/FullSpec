# Обновление SUMMARY архитектуры

Скилл для обновления файла `000_SUMMARY.md` в папке архитектуры при переходе архитектуры в статус `🟢 approved`.

## Связанная документация

| Документ | Назначение |
|----------|------------|
| [general_docs.md](../../../llm_instructions/general_docs.md#архитектура-architecture) | Правила ведения архитектуры |
| [000_SUMMARY.md](../../../general_docs/02_architecture/000_SUMMARY.md) | SUMMARY архитектуры |
| [summary-doc skill](../summary-doc/SKILL.md) | Скилл обновления SUMMARY дискуссий (аналог) |
| [decision skill](../decision/SKILL.md) | Скилл создания ADR (читает этот SUMMARY) |

## Когда использовать

**Триггеры:**
- Команда `/summary-arch`
- Автоматически при переходе архитектуры `review → approved`
- После завершения ревью архитектуры

**Контекст:**
- Архитектура переходит в статус `🟢 approved`
- Ревью архитектуры завершено
- Готовность к созданию ADR

## Workflow

```
🟣 review (архитектура) → 🟢 approved → /summary-arch (этот скилл!)
                                                ↓
                                          /decision (создаёт ADR)
                                                ↓
                                          ⚪ final (архитектура)
```

**ВАЖНО:** Этот скилл выполняется ПЕРЕД `/decision` и ПОСЛЕ перехода архитектуры в `approved`.

## Инструкции

### Шаг 1: Получить информацию об архитектуре

Из аргументов или контекста получить:
- Путь к файлу архитектуры (например: `general_docs/02_architecture/001_sistema_email.md`)
- Или ID архитектуры (например: `001`)

### Шаг 2: Прочитать архитектуру и извлечь данные

Извлечь из файла архитектуры:
1. **ID** — номер архитектуры (например: `001`)
2. **Название компонента** — из H1 заголовка
3. **Область** — определить по содержанию (см. правила ниже)
4. **Технологии** — из секции "Внешние зависимости" или "Компоненты"
5. **Назначение** — из секции "Цель и контекст"
6. **Ключевые принципы** — из секции "Принципы и ограничения"
7. **Ограничения** — из секции "Риски и компромиссы"
8. **Зависимости** — какие другие компоненты использует
9. **Связанная дискуссия** — из секции "Связанные документы"

**Правила определения области:**

| Область | Ключевые слова и признаки |
|---------|--------------------------|
| **UI/Frontend** | UI, компоненты, стили, React, Vue, CSS, фронтенд |
| **Backend/API** | API, сервис, endpoint, REST, GraphQL, бэкенд |
| **Auth/Security** | аутентификация, авторизация, JWT, OAuth, безопасность |
| **Database** | база данных, PostgreSQL, MongoDB, миграции, схема |
| **Infrastructure** | деплой, Docker, Redis, очереди, воркеры, мониторинг |
| **Architecture** | интеграция, микросервисы, паттерны |

### Шаг 3: Обновить 000_SUMMARY.md

**Файл:** `general_docs/02_architecture/000_SUMMARY.md`

#### 3.1 Обновить таблицу "Быстрый обзор по областям"

Найти строку с соответствующей областью и увеличить счётчик:

```markdown
| Infrastructure | 1 | [001] Email-система |
```

Также обновить строку "Всего".

#### 3.2 Добавить/обновить таблицу "Архитектурные решения"

Если есть placeholder `| - | - | - | - | - | - |`, заменить его на первую запись.
Иначе добавить строку:

```markdown
| [ID] | [Область] | [Компонент] | [Технологии] | [Ключевые принципы] | [Ссылка] |
```

**Пример:**
```markdown
| 001 | Infrastructure | Система email-уведомлений | Redis, Node.js | Асинхронность, retry-логика | [001_sistema_email.md](001_sistema_email.md) |
```

#### 3.3 Добавить ссылку в секцию "Решения по областям"

Найти соответствующую секцию (например, `### Infrastructure`) и добавить ссылку:

```markdown
### Infrastructure

- **[001] Система email-уведомлений** — асинхронная очередь на Redis для надёжной отправки писем. [Подробнее](001_sistema_email.md)
```

Если там был текст "Архитектурные решения в этой области отсутствуют." — удалить его.

#### 3.4 Добавить блок описания

В секцию "Краткое описание решений" добавить блок:

```markdown
### [ID] [Название компонента]

**Область:** [UI/Frontend | Backend/API | Auth/Security | Database | Infrastructure | Architecture]

**Технологии:** [Список технологий]

**Назначение:** [1-2 предложения из "Цель и контекст"]

**Ключевые принципы:**
- [Принцип 1]
- [Принцип 2]

**Ограничения:**
- [Ограничение 1]

**Зависимости от других областей:**
- [Какие компоненты использует]

**Архитектура:** [Ссылка на файл архитектуры]
**Дискуссия:** [Ссылка на исходную дискуссию]

---
```

#### 3.5 Удалить placeholder

Если есть текст "В настоящее время архитектурные решения отсутствуют.", удалить его.

### Шаг 4: Вывести результат

```
SUMMARY архитектуры обновлён для [ID]:
- Область: [Область]
- Добавлена строка в таблицу "Архитектурные решения"
- Добавлена ссылка в секцию "[Область]"
- Обновлён счётчик в "Быстрый обзор по областям"
- Добавлено описание решения

Следующий шаг: /decision для создания ADR
```

## Пример

**Архитектура:** `001_sistema_email_uvedomleniy.md`

**Содержимое архитектуры (фрагменты):**
```markdown
# Система email-уведомлений

**Статус:** 🟢 approved

## Цель и контекст
Система для надёжной отправки email-уведомлений пользователям...

## Компоненты
- Notification Service — приём запросов на отправку
- Redis Queue — очередь сообщений
- Email Worker — обработка и отправка

## Внешние зависимости
- Redis 7.x
- Node.js 20+
- Nodemailer

## Связанные документы
- **Дискуссия:** [001_sistema_email.md](../01_discuss/001_sistema_email.md)
```

**Определение области:** `Infrastructure` (Redis, очереди, воркеры)

**Результат в 000_SUMMARY.md:**

**Быстрый обзор по областям:**
```markdown
| Область | Количество | Ключевые компоненты |
|---------|------------|---------------------|
| Infrastructure | 1 | [001] Email-система |
...
| **Всего** | **1** | — |
```

**Таблица "Архитектурные решения":**
```markdown
| ID | Область | Компонент | Технологии | Ключевые принципы | Ссылка |
|----|---------|-----------|------------|-------------------|--------|
| 001 | Infrastructure | Система email-уведомлений | Redis, Node.js | Асинхронность, retry | [001_sistema_email.md](001_sistema_email.md) |
```

**Секция "Infrastructure":**
```markdown
### Infrastructure

- **[001] Система email-уведомлений** — асинхронная очередь на Redis для надёжной отправки писем. [Подробнее](001_sistema_email.md)
```

**Блок описания:**
```markdown
### 001 Система email-уведомлений

**Область:** Infrastructure

**Технологии:** Redis 7.x, Node.js 20+, Nodemailer

**Назначение:** Система для надёжной асинхронной отправки email-уведомлений пользователям с поддержкой retry-логики и масштабирования.

**Ключевые принципы:**
- Асинхронная обработка через очередь
- Автоматические повторные попытки при сбоях
- Горизонтальное масштабирование воркеров

**Ограничения:**
- Требуется Redis в инфраструктуре
- Нет встроенной аналитики открытий/кликов

**Зависимости от других областей:**
- Backend/API: Notification Service интегрируется с API Gateway

**Архитектура:** [001_sistema_email.md](001_sistema_email.md)
**Дискуссия:** [001_sistema_email.md](../01_discuss/001_sistema_email.md)

---
```

## Чек-лист

При обновлении SUMMARY проверять:

- [ ] Область определена корректно?
- [ ] Счётчик в "Быстрый обзор по областям" обновлён?
- [ ] Строка добавлена в таблицу "Архитектурные решения"?
- [ ] Ссылка добавлена в соответствующую секцию области?
- [ ] Блок описания добавлен со всеми полями?
- [ ] Placeholder удалён?
- [ ] Ссылки на архитектуру и дискуссию корректны?

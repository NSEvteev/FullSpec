## 1. Назначение

**Зона ответственности:** КАК РАСПРЕДЕЛЯЕМ ОТВЕТСТВЕННОСТИ ([Стандарт SDD § 2.2](../standard-specs.md#22-таблица-объектов)).

Design отвечает на вопрос: **Как распределяем ответственности между сервисами? (РЕШАЕТ)**

**Роль — РЕШАТЕЛЬ:** Design **критически оценивает** все предложения Impact и **РЕШАЕТ** окончательное распределение. Impact ПРЕДЛАГАЕТ варианты затронутых сервисов, Design РЕШАЕТ: подтверждает, изменяет или отклоняет предложения, думает на масштабирование, распределяет нераспределённую функциональность, создаёт новые сервисы с архитектурными инкрементами ([Стандарт SDD § 3.2](../standard-specs.md#32-impact-design-предлагатель-решатель)).

**Источник контекста — Deep Scan:** Design обязан прочитать архитектурные документы **в деталях** до фазы Clarify. Порядок чтения:

| # | Что читать | Зачем |
|---|-----------|-------|
| 1 | **Parent Impact** (целиком) | Предложения: затронутые сервисы, компоненты, зависимости, риски |
| 2 | **Parent Discussion** (целиком) | Требования, user stories, критерии успеха |
| 3 | **`specs/architecture/services/README.md`** | Таблица всех сервисов, технологии |
| 4 | **`specs/architecture/system/overview.md`** | Общая картина, потоки данных |
| 5 | **`specs/architecture/services/{svc}.md`** для каждого затронутого сервиса | Детали: компоненты, API, Code Map, границы автономии |
| 6 | **`specs/technologies/README.md`** + per-tech стандарты | Реестр технологий, существующие стандарты кодирования |

Deep Scan — **6 источников** (vs 2 у Impact Quick Scan). Design читает architecture в деталях, включая per-service документы и технологический реестр — это основа для принятия решений о распределении ответственностей и выборе технологий. При отсутствии `specs/architecture/` (новый проект) — Design опирается на Impact + Discussion + Clarify.

Подробнее об архитектурных документах: [standard-service.md](../living-docs/service/standard-service.md).

**Содержит:**
- Окончательное распределение ответственностей между сервисами
- Компоненты каждого сервиса (подтверждённые/изменённые из Impact)
- Блоки взаимодействия между сервисами (контракты, протоколы, sequences)
- Решения о shared-компонентах (размещение в `/shared/` или внутри сервиса-владельца)
- Системные тест-сценарии (межсервисные e2e, integration, load)
- Решения о создании новых сервисов (подтверждение/отклонение предложений Impact)

**НЕ содержит:**
- Бизнес-обоснование и требования (→ Discussion)
- Предложения затронутых сервисов без решения (→ Impact ПРЕДЛАГАЕТ)
- Детали реализации конкретного сервиса — пакеты, классы, алгоритмы (→ ADR)
- Архитектурные решения — trade-offs, альтернативы, обоснование (→ ADR)
- Тестовые сценарии конкретного сервиса — unit, integration внутри сервиса (→ План тестов)
- Задачи на реализацию (→ План разработки)

**Граничные случаи (Design vs ADR):**
- Допустимо: "auth-сервис отвечает за генерацию и валидацию JWT" — распределение ответственности
- Допустимо: "POST /auth/token → {access_token, refresh_token, expires_in}" — контракт API в блоке взаимодействия
- Допустимо: "auth публикует UserCreatedEvent → notifications, billing потребляют" — блок взаимодействия
- Допустимо: "Shared validator размещается в /shared/auth/" — решение о shared
- Недопустимо: "Использовать RS256 вместо HS256 для JWT" — выбор алгоритма (→ ADR)
- Недопустимо: "Класс TokenGenerator в пакете auth.tokens" — детали реализации (→ ADR)
- Недопустимо: "Connection pool size = 20" — конфигурация (→ ADR)

**Когда создавать:**
- После одобрения Impact (статус WAITING)
- Всегда один Design на один Impact (1:1)

**Связи:** Parent — Impact (обязателен, 1:1). Дочерние объекты — ADR (1:N, по одному на каждый затронутый сервис). ([Стандарт SDD § 3.1](../standard-specs.md#31-frontmatter-и-навигация))

**Агентный режим ([решение #45](../standard-specs.md#12-решения)):** Design выполняется через скилл `/design-create`, который оркестрирует многоагентный процесс:
1. Скилл вызывает **design-agent** — агент выполняет Deep Scan, CLARIFY → GENERATE → VALIDATE (шаги 2-4 из [§ 2.3](../standard-specs.md#23-общий-паттерн-объекта)), создаёт Design-документ (DRAFT). Агент читает `specs/architecture/` в деталях, не засоряя контекст оркестратора.
2. LLM получает результат агента и вызывает **design-reviewer** — ревьюер проверяет распределение ответственностей, предлагает правки при необходимости (AGENT REVIEW, шаг 5).
3. **USER REVIEW (шаг 6) обязателен** — пользователь финально ревьюит результат и одобряет → WAITING.

**При WAITING:** 5 артефактов — Planned Changes в `services/{svc}.md` + `system/` + `domains/`, заглушки для новых сервисов, per-tech стандарты для новых технологий, создание ADR (1:N). Подробнее: [§ 4 — Артефакты Design → WAITING](#4-переходы-статусов).

**После DONE:** Design становится архивной записью проектных решений. При DONE → обновляются живые документы: `specs/architecture/system/`, `specs/architecture/domains/`, `specs/tests/system/` ([Стандарт SDD § 7.3](../standard-specs.md#73-обновление-при-реализации-to-done)). Изменения запрещены (только исправление опечаток). При изменении scope — CONFLICT через обратную связь от кода.

## 2. Расположение и именование

**Расположение:** `specs/design/`

**Формат имени:** `design-NNNN-topic.md`

| Элемент | Правило |
|---------|---------|
| `design` | Фиксированный префикс |
| `NNNN` | Четырёхзначный номер, **равен NNNN parent Discussion** (disc-0002 → design-0002) |
| `topic` | Kebab-case, латиница, описательное |

**Regex:** `^design-(\d{4})-(.+)\.md$`

**Нумерация:** NNNN Design **всегда равен** NNNN parent Discussion. Это обеспечивает 1:1 соответствие через всю цепочку: `disc-0002-topic.md` → `impact-0002-topic.md` → `design-0002-topic.md`. Пропуски в нумерации допустимы.

**Формат README-таблицы** (`specs/design/README.md`):

| ID | Документ | Статус | Parent Impact | ADR | Milestone | Описание |
|----|----------|--------|---------------|-----|-----------|----------|
| 0001 | design-0001-oauth2-service-design.md | RUNNING | impact-0001 | adr-0001 (auth, gateway, users) | v1.2.0 | Проектирование OAuth2 авторизации |

> **Примечание:** NNNN Design = NNNN parent Discussion (disc-0001 → design-0001).

## 3. Frontmatter

**SSOT frontmatter:** [standard-frontmatter.md](/.structure/.instructions/standard-frontmatter.md) ([§ 1 — базовые поля](/.structure/.instructions/standard-frontmatter.md#1-обязательные-поля) + [§ 4 — поля specs](/.structure/.instructions/standard-frontmatter.md#4-дополнительные-поля-для-спецификаций-specs))

| Поле | Значение для Design | Обязательность |
|------|---------------------|----------------|
| `description` | Описание проектирования (до 1024 символов) | Обязательно |
| `standard` | `specs/.instructions/design/standard-design.md` | Обязательно |
| `standard-version` | `v1.1` | Обязательно |
| `index` | `specs/design/README.md` | Обязательно |
| `parent` | Путь к Impact-документу (обязателен) | Обязательно |
| `children` | Пути к ADR-документам (1:N, по сервисам) | Обязательно (`children: []` при создании, заполняется после создания ADR) |
| `status` | Текущий статус ([Стандарт SDD § 5](../standard-specs.md#5-статусы)) | Обязательно |
| `milestone` | Наследуется от parent Discussion (через Impact) | Обязательно |

```yaml
---
description: Проектирование OAuth2 авторизации — распределение ответственностей auth/gateway/users, контракты API, системные тесты.
standard: specs/.instructions/design/standard-design.md
standard-version: v1.1
index: specs/design/README.md
parent: impact/impact-0001-oauth2-authorization.md
children:
  - services/auth/adr/adr-0001-oauth2-auth.md
  - services/gateway/adr/adr-0001-oauth2-gateway.md
  - services/users/adr/adr-0001-oauth2-users.md
status: DRAFT
milestone: v1.2.0
---
```

**Parent (обязателен):** Путь к Impact-документу. Design всегда имеет parent — Impact (1:1). Без parent документ невалиден.

**Children (1:N):** ADR создаются per-service. Один Design порождает N ADR — по одному на каждый сервис из секций SVC. Путь к ADR: `services/{svc}/adr/adr-NNNN-topic.md`.

**Milestone:** Наследуется от parent Discussion (через Impact). Значение milestone в Design **всегда совпадает** с milestone в parent Discussion. Вопросы о milestone на уровне Design не задаются.

## 4. Переходы статусов

Happy path — нормальный поток жизненного цикла Design ([Стандарт SDD § 6](../standard-specs.md#6-последовательность-статусов)).

**Зона ответственности Design:** только переход DRAFT → WAITING (создание и модификация документа). Все последующие переходы управляются на уровне цепочки — дочерними документами, каскадами или обратной связью от кода.

| Текущий статус | Условие перехода | Следующий статус | Кто управляет |
|---|---|---|---|
| **DRAFT** | Пользователь одобрил документ. Нет неразрешённых `[ТРЕБУЕТ УТОЧНЕНИЯ]`. Нет Dependency Barrier | **WAITING** | Design (этот документ) |
| **WAITING** | Все документы цепочки (Discussion → ... → Plan Dev) в WAITING. Пользователь подтвердил запуск | **RUNNING** | Цепочка ([§ 6.2](../standard-specs.md#62-waiting-to-running)) |
| **RUNNING** | Обратная связь от кода затронула уровень Design ([§ 6.3](../standard-specs.md#63-running-to-conflict)) | **CONFLICT** | Цепочка (tree-level) |
| **CONFLICT** | LLM исправил документ, пользователь одобрил исправление ([§ 6.4](../standard-specs.md#64-conflict-to-waiting)) | **WAITING** | Цепочка (top-down) |
| **RUNNING** | Все ADR (дочерние, 1:N) → DONE. Каскад снизу вверх ([§ 6.5](../standard-specs.md#65-running-to-done)) | **DONE** | Цепочка (bottom-up) |

Откат и отклонение (ROLLING_BACK, REJECTED): [Стандарт SDD § 6.6–6.7](../standard-specs.md#66-to-rolling_back).

**Артефакты Design → WAITING:**

**SSOT триггеров:** [standard-specs.md § 7.1](../standard-specs.md#71-обновление-при-планировании-to-waiting) — полная таблица файлов, создаваемых/обновляемых при Design → WAITING.

| # | Артефакт | Куда | Что создаётся | Формат |
|---|----------|------|---------------|--------|
| 1 | **Planned Changes** | `specs/architecture/services/{svc}.md` для каждого SVC | Навигационный указатель: ссылка на Discussion, Design, статус, что затрагивает | [standard-service.md § 5.7](../living-docs/service/standard-service.md#57-planned-changes) |
| 2 | **Planned Changes** | `system/overview.md`, `system/data-flows.md` (если INT-N), `system/infrastructure.md` (если затрагивает инфраструктуру), `domains/context-map.md`, `domains/{domain}.md` (если новый домен) | Навигационный указатель + дельты для системной/доменной архитектуры | [standard-architecture.md § 4](../living-docs/architecture/standard-architecture.md#4-обязательные-секции) |
| 3 | **Заглушка сервиса** (только для новых) | `specs/architecture/services/{svc}.md` | Файл-заглушка: Резюме + Planned Changes. Заполняется полным содержанием при ADR → DONE | [standard-service.md § 9.1](../living-docs/service/standard-service.md#шаблон-заглушки-design-waiting) |
| 4 | **Per-tech стандарты** (только для новых технологий) | `specs/technologies/standard-{tech}.md` | Полный per-tech стандарт — прескриптивный словарь правил | [standard-technology.md § 4](../technologies/standard-technology.md#4-триггер-создания) |
| 5 | **ADR-документы** (1:N) | `specs/design/services/{svc}/adr/adr-NNNN-topic.md` | По одному ADR на каждый сервис из секций SVC. ADR детализирует внутрисервисные решения | Следующий шаг цепочки |

**Побочные эффекты Design → DONE** ([standard-specs.md § 7.3](../standard-specs.md#73-обновление-при-реализации-to-done)):
- Обновляются `specs/architecture/system/`, `specs/architecture/domains/` (AS IS обновляется, Planned Changes → Changelog)
- Обновляются `specs/tests/system/` (системные тест-сценарии → AS IS)

## 5. Разделы документа

Design — **структурированный документ с тремя типами секций**: секции по сервисам (ответственности и компоненты), блоки взаимодействия (контракты между сервисами) и системные тест-сценарии.

| Раздел | Уровень | Нумерация | Обязательность | Заглушка |
|--------|---------|-----------|---------------|----------|
| **📋 Резюме** | h2 | — | Обязательный (контент обязателен) | — |
| **SVC-N: {Сервис}** | h2 | SVC-N | Обязательный (мин. 1 секция) | — |
| ├── 📋 Ответственность | h3 | — | Обязательный (контент обязателен) | — |
| ├── 📦 Компоненты | h3 | CMP-N | Обязательный | _Компоненты наследуются из Impact без изменений._ |
| └── 🔗 Зависимости | h3 | — | Обязательный | _Внешних зависимостей нет._ |
| **INT-N: {Взаимодействие}** | h2 | INT-N | Условный (мин. 1, если > 1 сервиса) | — |
| ├── Контракт | h3 | — | Обязательный (контент обязателен) | — |
| └── Sequence | h3 | — | Обязательный (контент обязателен) | — |
| **🧪 Системные тест-сценарии** | h2 | STS-N | Обязательный | _Системных тест-сценариев нет._ |

**Структура документа:** Резюме → секции SVC (по сервисам, каждая с 3 подсекциями) → блоки INT (взаимодействия, каждый с 2 подсекциями) → Системные тест-сценарии.

**Минимум 1 секция SVC обязательна.** Минимум 1 блок INT обязателен при > 1 сервиса. Секция "Резюме" всегда заполняется контентом. Заглушка в подсекциях SVC означает: подсекция рассмотрена, контент не требуется.

**Группировка секций SVC (опционально, при > 5 затронутых сервисов):** Если Design затрагивает 5+ сервисов с **идентичной ответственностью** (например, cross-cutting concern "добавить логирование"), допустима группировка в одну секцию SVC-N. Формат: заголовок `## SVC-1: {Группа сервисов} ({назначение})`, описание общей ответственности, таблица затронутых сервисов с индивидуальными компонентами. Группировка допустима только при идентичной ответственности — если ответственности различаются, каждый сервис получает отдельную секцию SVC-N.

**Правила нумерации (SVC-N, CMP-N, INT-N, STS-N):** Последовательная автоинкрементная, **сквозная по всему документу**. При добавлении — следующий номер = максимальный существующий + 1. При удалении — **не перенумеровывать** (пропуски допустимы, сохраняется ссылочная целостность).

**Нумерация Design — независимая от Impact.** SVC-1 в Design может описывать тот же сервис, что и SVC-1 в Impact, но это не обязательно. Design **автоинкрементирует SVC-N в порядке "Основной → Вторичный → Новый"** без привязки к нумерации Impact. LLM **не обязан** сохранять соответствие нумерации Impact — независимая нумерация позволяет Design менять порядок и группировку сервисов.

**Upward feedback:** При проектировании Design может обнаружить информацию, затрагивающую Impact или Discussion.

**Механика ([Стандарт SDD § 3.6](../standard-specs.md#36-upward-feedback)):**
1. Работа с Design **приостанавливается**
2. LLM обновляет Impact (статус остаётся **WAITING** — без перевода в DRAFT)
3. LLM **всегда проверяет Discussion** (правило каскада [§ 3.6](../standard-specs.md#36-upward-feedback)). Если обнаруженная информация затрагивает зону ответственности Discussion (ЗАЧЕМ, ЧТО) — обновляет. Критерий: попадает ли обнаруженное в колонку "Вопрос" Discussion ([§ 2.2](../standard-specs.md#22-таблица-объектов))
4. Пользователь подтверждает изменения
5. Работа с Design **возобновляется** (с учётом обновлённых уровней)

**Критерий upward feedback:** обнаруженная информация **изменяет характер** взаимодействия или состав сервисов, описанных в Impact. Детализация в рамках уже определённого блока — не триггер.

**Примеры:**
- **Требует:** обнаружен новый сервис, не упомянутый в Impact → дополняется Impact (новый SVC)
- **Требует:** обнаружена зависимость от инфраструктуры (Redis, Kafka), не учтённая в Impact → дополняется Impact
- **Требует:** изменение типа зависимости (sync → async), если в Impact указано sync → дополняется Impact (DEP обновляется)
- **НЕ требует:** детализация контракта API (добавление полей в JSON) внутри уже определённого блока взаимодействия — не меняет характер взаимодействия
- **НЕ требует:** изменение формата контракта (JSON → protobuf), **если** блок взаимодействия в Impact не специфицировал формат (это детализация Design). **Требует**, если Impact указывал REST (подразумевал JSON), а Design выявил необходимость gRPC — это изменение характера

### 📋 Резюме

Краткое описание проектных решений. Формат: свободный текст (1–3 абзаца).

**Обязательное содержимое:**
- Общий scope: количество сервисов подтверждённых/изменённых/добавленных/отклонённых
- Ключевые решения Design (распределение ответственностей, shared-компоненты)
- Количество блоков взаимодействия и системных тест-сценариев
- Ключевые изменения относительно Impact (если есть)

```markdown
## 📋 Резюме

Design подтверждает 3 сервиса из Impact: auth (основной), gateway (вторичный) и users
(вторичный). Отклонён notification-service — вынесен в отдельную Discussion. Token Validator
размещён как shared-компонент в /shared/auth/. 4 блока взаимодействия, 3 системных
тест-сценария.
```

### SVC-N: {Сервис}

Каждый сервис оформляется как **отдельная секция h2** с метаданными, описанием и 3 обязательными подсекциями.

**Заголовок секции:** `## SVC-N: {имя сервиса}`

**Метаданные сервиса** — сразу после заголовка, в одну строку:

```markdown
## SVC-1: auth

**Impact:** SVC-1 (Основной, Высокая) | **Решение Design:** Подтверждён

Auth отвечает за полный жизненный цикл OAuth2-токенов: генерацию JWT access + refresh
пары при логине, валидацию подписи, обновление по refresh-токену и отзыв. Владеет таблицей
refresh_tokens в PostgreSQL и blacklist в Redis. Gateway обращается к auth через INT-1
(Token Issuance) и INT-2 (Token Refresh).
```

**Описание ответственности (1–2 абзаца, после метаданных)** — человекочитаемая история флоу: что сервис делает, при каких действиях пользователя/системы, как данные проходят через сервис, с какими другими сервисами взаимодействует. Связывает компоненты, данные и API в единый нарратив. **НЕ дублирует** подсекцию "📋 Ответственность" — дополняет её контекстом и связями.

**Что включать:**
- Нарратив: при каком действии → что происходит → какие сервисы участвуют
- Ссылки на INT-N блоки (какие API предоставляет/потребляет, какие события)
- Решения — что изменено/добавлено/отклонено относительно Impact

**Чего избегать:** детали реализации (алгоритмы, пакеты, классы, библиотеки) — это уровень ADR. Не перечислять конкретные действия списком — для этого подсекция "📋 Ответственность".

**Метаданные "Impact → Design":**

| Impact предложил | Design решил | Описание |
|-----------------|-------------|----------|
| Основной | Подтверждён | Сервис подтверждён как основной |
| Вторичный | Подтверждён | Сервис подтверждён как вторичный |
| Новый (план создания) | Подтверждён (новый) | Создание нового сервиса подтверждено |
| Любой | Изменён: {старый} → {новый} | Тип влияния пересмотрен Design. Пример: "Изменён: Вторичный → Основной" |
| Любой | Отклонён | Сервис не нужен / вынесен в другую Discussion |
| — (не был в Impact) | Добавлен Design | Design обнаружил новый сервис при Deep Scan |

**Порядок секций SVC:** Первичный критерий — тип влияния (Основной → Вторичный → Новый). Вторичный критерий (внутри типа) — решение Design (Подтверждённые → Изменённые → Добавленные). **Пример:** Если есть Основной (Подтверждён), Основной (Изменён), Вторичный (Подтверждён) — порядок SVC-1: Основной (Подтверждён), SVC-2: Основной (Изменён), SVC-3: Вторичный (Подтверждён). Отклонённые сервисы **не получают секцию SVC h2** — упоминаются в Резюме с **обоснованием отклонения** (почему не подтверждён, куда вынесен, какой альтернативный подход). Пример: "Отклонён notification-service — вынесен в отдельную Discussion disc-0042."

**Косвенные сервисы (из Impact):** Сервис с типом влияния "Косвенный" **не получает секцию SVC-N** в Design. Упоминается только в Резюме или в блоках INT, если участвует во взаимодействии. Если при Deep Scan влияние оказалось прямым — Design меняет тип на "Вторичный" (upward feedback → Impact) и создаёт SVC-N.

#### 📋 Ответственность

Структурированный bullet list конкретных действий сервиса: **что** владеет, **что** предоставляет, **что** публикует/потребляет. **Формат: bullet list.** Контент обязателен — каждый сервис должен иметь явный список ответственностей. **Уровень детализации:** описание **что** сервис делает (действия, владение данными, события), **без** повторения контрактов API и схем данных (они в секциях INT и подсекциях Компоненты). Без флоу и связей — только список ответственностей.

```markdown
### 📋 Ответственность

- Генерация и валидация JWT access + refresh токенов
- Управление жизненным циклом refresh-токенов (выпуск, обновление, отзыв)
- Хранение refresh-токенов и blacklist отозванных
- Публикация событий аутентификации для аудита
```

#### 📦 Компоненты

Подтверждённые/изменённые компоненты сервиса. **Формат: таблица.** Нумерация: `CMP-N` (сквозная по документу).

Design подтверждает, изменяет или добавляет компоненты из Impact. Колонка "Решение" показывает, что Design изменил.

```markdown
### 📦 Компоненты

| ID | Компонент | Scope | Решение |
|----|-----------|-------|---------|
| CMP-1 | Token Generator | local | Подтверждён из Impact (CMP-1) |
| CMP-2 | Token Validator | shared (/shared/auth/) | Изменён: Impact (CMP-4, shared (gateway)) → размещение в /shared/auth/ |
| CMP-3 | Token Revoker | local | Добавлен Design — управление blacklist |
```

**Колонка "Решение" — что именно изменилось:**
- `Подтверждён из Impact (CMP-N)` — имя, scope и назначение без изменений
- `Изменён: Impact (CMP-N, scope: shared) → scope: local` — изменение scope
- `Изменён: Impact (CMP-N, "Token Validator") → объединён с CMP-M ("Token Generator")` — изменение структуры
- `Добавлен Design — описание` — новый компонент, не упомянутый в Impact

**Shared-компоненты:** Design **РЕШАЕТ** размещение shared-компонентов ([Стандарт SDD § 3.5](../standard-specs.md#35-shared-код-shared)):
- `shared (/shared/{область}/)` — компонент в `/shared/`, используется несколькими сервисами
- `shared (в {сервис})` — компонент в сервисе-владельце, предоставляется другим через API

#### 🔗 Зависимости

Ссылки на блоки взаимодействия (INT-N), в которых участвует данный сервис. **Формат: bullet list**, сгруппированный по направлению.

```markdown
### 🔗 Зависимости

- **Предоставляет:** INT-1 (Token Issuance → gateway), INT-2 (Token Refresh → gateway)
- **Потребляет:** INT-3 (User Roles ← users)
- **Публикует:** INT-4 (AuthTokenIssued → audit)
```

**Направления:**
- **Предоставляет** — сервис является provider в INT-N (отвечает на запрос)
- **Потребляет** — сервис является consumer в INT-N (отправляет запрос)
- **Публикует** — сервис является publisher в INT-N (отправляет событие)
- **Подписан** — сервис является subscriber в INT-N (получает событие)

### INT-N: Блоки взаимодействия

Блоки взаимодействия описывают **как сервисы общаются между собой**. Каждый блок — отдельная секция h2 с контрактом и sequence-диаграммой. Нумерация: `INT-N` (сквозная по документу).

**Заголовок блока:** `## INT-N: {Описание взаимодействия}`

**Метаданные** — сразу после заголовка:

```markdown
## INT-1: Token Issuance

**Участники:** auth (provider) ↔ gateway (consumer)
**Паттерн:** sync (REST)
**Источник Impact:** DEP-1 (gateway → auth, sync)
```

**Источник Impact:** ссылка на DEP-N из Impact, который описывал эту зависимость. `—` если блок добавлен Design.

**Паттерн — свободный текст.** Описывает характер взаимодействия:
- `sync (REST)`, `sync (gRPC)` — синхронные вызовы
- `async (events)`, `async (queue)` — асинхронные через message broker
- `async (scheduled)` — периодические задачи

#### Контракт

Спецификация **межсервисного** API или формата события. Контракт описывается в Design, потому что это **граница между двумя и более сервисами** — оба зависят от неё, и изменение контракта затрагивает оба. Design определяет **полный межсервисный контракт** — endpoint, метод, формат request/response (JSON-схемы), статус-коды, основные ошибки. Impact указывал только "POST /auth/token — новый", Design специфицирует формат. **Внутрисервисные решения** (алгоритмы, классы, пакеты, retry-политики, timeout-конфигурации, заголовки аутентификации) — уровень ADR.

**Для REST/gRPC:**

```markdown
### Контракт

**Endpoint:** `POST /api/v1/auth/token`

**Request:**
` ``json
{
  "grant_type": "password | refresh_token",
  "username": "string",
  "password": "string"
}
` ``

**Response (200):**
` ``json
{
  "access_token": "string (JWT)",
  "refresh_token": "string (opaque)",
  "token_type": "Bearer",
  "expires_in": 3600
}
` ``

**Ошибки:**

| Код | Описание |
|-----|----------|
| 400 | Невалидные credentials |
| 429 | Rate limit exceeded |
```

**Для событий (async):**

```markdown
### Контракт

**Событие:** `AuthTokenIssued`
**Канал:** `auth.events` (message broker)

**Формат:**
` ``json
{
  "event_type": "AuthTokenIssued",
  "user_id": "UUID",
  "issued_at": "ISO 8601",
  "ip_address": "string"
}
` ``
```

**Уровень детализации контракта:**
- Допустимо: endpoint, метод, формат request/response, статус-коды, формат событий
- Недопустимо: заголовки аутентификации, retry-политики, timeout-конфигурации (→ ADR)

**Breaking changes:** Если Impact указал breaking change для endpoint (колонка "Совместимость: Breaking"), Design **обязан** задокументировать это в контракте INT-N. Формат: примечание `⚠️ Breaking change` в секции "Контракт" с описанием, что именно ломает обратную совместимость.

**Версионирование контрактов:** При изменении контракта API (появление v2) создаётся **новый блок INT-N** для новой версии. Старый блок INT-N остаётся в документе с пометкой `[deprecated: vX.Y.Z]`. Пример: INT-1 (Token Issuance v1, deprecated: v1.3.0), INT-5 (Token Issuance v2).

**Async с несколькими consumers:** Если событие потребляется несколькими сервисами (например, UserCreatedEvent → notifications, billing, analytics), создаётся **один блок INT-N** с перечислением всех consumers в метаданных. Формат: `**Участники:** auth (publisher) → notifications, billing, analytics (consumers)`.

#### Sequence

Диаграмма последовательности в формате Mermaid. **Обязательна для каждого блока INT-N.** Показывает порядок вызовов между участниками. Даже для одношаговых взаимодействий (Client → Server → Response) диаграмма визуализирует поток и помогает при разработке тестов.

```markdown
### Sequence

` ``mermaid
sequenceDiagram
    participant C as Client
    participant G as Gateway
    participant A as Auth

    C->>G: POST /auth/token {credentials}
    G->>A: Forward request
    A->>A: Validate credentials
    A-->>G: 200 {access_token, refresh_token}
    G-->>C: 200 {access_token, refresh_token}
` ``
```

**Уровень детализации:** Sequence показывает межсервисные вызовы (сервис-к-сервису). Внутренние шаги сервиса (валидация, генерация) показываются как self-calls только если они критичны для понимания потока. Детальные внутренние шаги — уровень ADR.

### 🧪 Системные тест-сценарии

Межсервисные тест-сценарии, вытекающие из блоков взаимодействия. **Формат: таблица.** Нумерация: `STS-N`. Секция общая для всего документа (не внутри SVC). **Уровень детализации:** краткое описание флоу (без Given/When/Then) — что тестируется, участники, тип. Детальные шаги (preconditions, actions, assertions) — уровень Plan тестов. При Design → DONE сценарии переносятся в `specs/tests/system/` ([§ 7.3](../standard-specs.md#73-обновление-при-реализации-to-done)).

**Связь с Plan тестов:** Системные тест-сценарии (STS-N) из Design используются как **входные данные** для План тестов. План тестов **обязан** покрыть каждый STS-N конкретными тестами (unit/integration/e2e). Трассируемость: колонка "Источник" в Plan тестов ссылается на STS-N из Design.

```markdown
## 🧪 Системные тест-сценарии

| ID | Сценарий | Участники | Тип | Источник |
|----|----------|-----------|-----|----------|
| STS-1 | Полный цикл: логин → токен → запрос → успех | Client, Gateway, Auth, Users | e2e | INT-1, INT-3 |
| STS-2 | Refresh: access истёк → refresh → новый access → успех | Client, Gateway, Auth | e2e | INT-2 |
| STS-3 | Rate limiting: 100+ запросов/мин → 429 | Client, Gateway | integration | INT-1 |
```

**Колонка "Тип":**

| Тип | Описание |
|-----|----------|
| e2e | End-to-end: полный пользовательский сценарий через все сервисы |
| integration | Интеграция между конкретными сервисами (2-3 участника) |
| load | Нагрузочное тестирование межсервисного взаимодействия |

**Колонка "Источник":** ссылки на INT-N блоки, из которых вытекает сценарий.

## 6. Clarify

**SSOT:** [Стандарт SDD § 8](../standard-specs.md#8-clarify-и-блокирующие-правила) — полные правила Clarify, маркер `[ТРЕБУЕТ УТОЧНЕНИЯ]`, Dependency Barrier.

**Паттерн объекта:** [Стандарт SDD § 2.3 "Общий паттерн объекта"](../standard-specs.md#23-общий-паттерн-объекта) — цикл CLARIFY → GENERATE → VALIDATE → REVIEW.

**Принцип Clarify на уровне Design:** LLM **сам анализирует** Deep Scan (Impact + Discussion + Architecture) и **предлагает** проектные решения. Пользователь подтверждает или корректирует. LLM **НЕ спрашивает** пользователя "как распределить ответственности?" — это работа LLM.

| Что предлагает LLM | Пример предложения через AskUserQuestion |
|---------------------|------------------------------------------|
| Подтверждение/изменение Impact | "Impact предложил auth как основной. Подтверждаю: auth генерирует и валидирует JWT. Согласны?" |
| Shared-компоненты | "Token Validator используется auth и gateway. Предлагаю в /shared/auth/. Или оставить в auth?" |
| Отклонение сервиса | "Impact предложил notification-service (Предположительно). Не подтверждаю — вынести в отдельную Discussion?" |
| Контракты | "Для gateway → auth предлагаю REST /api/v1/auth/token. Или gRPC?" |
| Системные тесты | "Для INT-1 предлагаю 3 сценария: happy path, expired token, rate limit. Достаточно?" |

**Контекст из Impact:** **На шаге CLARIFY** LLM **обязан** прочитать parent Impact целиком: все секции SVC, зависимости, риски. Design строится на основе Impact.

**Контекст из Discussion:** LLM читает parent Discussion для понимания бизнес-требований и критериев успеха.

**Контекст из Architecture:** LLM выполняет Deep Scan ([§ 1](#1-назначение)): все 6 источников, включая `specs/architecture/services/{svc}.md` для каждого затронутого сервиса и `specs/technologies/README.md` + per-tech стандарты.

**Маркер `[ТРЕБУЕТ УТОЧНЕНИЯ]`:** LLM ставит маркеры на оставшиеся неясности **в обоих случаях** — после проведения Clarify и при `--auto-clarify`. Пока в документе есть хотя бы один неразрешённый маркер **или активен Dependency Barrier**, перевод в статус WAITING **невозможен**. Подробнее: [Стандарт SDD § 8](../standard-specs.md#8-clarify-и-блокирующие-правила).

Воркфлоу создания (`create-design.md`) завершается переводом в WAITING. Поэтому все маркеры должны быть разрешены до завершения создания — через AskUserQuestion, не откладывая на modify.

**Пропуск Clarify:** Пользователь может указать `--auto-clarify` — LLM пропускает Clarify, генерирует на основе своего понимания Impact + Architecture и ставит маркеры `[ТРЕБУЕТ УТОЧНЕНИЯ]` на все неясности. Шаг REVIEW остаётся обязательным.

**Примеры маркеров при --auto-clarify:**
- `[ТРЕБУЕТ УТОЧНЕНИЯ: Token Validator — разместить в /shared/auth/ или оставить внутри auth-сервиса?]`
- `[ТРЕБУЕТ УТОЧНЕНИЯ: gateway → auth — синхронная валидация через REST или кэширование публичных ключей?]`
- `[ТРЕБУЕТ УТОЧНЕНИЯ: Контракт POST /auth/token — какие grant_type поддерживаются?]`

## 9. Примеры по типам

### Пример 1: Новая функциональность

```markdown
# design-0001: OAuth2 авторизация

## 📋 Резюме

Design подтверждает 3 сервиса из Impact: auth (основной), gateway (вторичный), users
(вторичный). Отклонён notification-service — вынесен в отдельную Discussion. Token Validator
размещён как shared-компонент в /shared/auth/. 4 блока взаимодействия, 3 системных
тест-сценария.

## SVC-1: auth

**Impact:** SVC-1 (Основной, Высокая) | **Решение Design:** Подтверждён

Auth отвечает за полный жизненный цикл OAuth2-токенов: генерацию JWT access + refresh пары
при логине, валидацию подписи, обновление по refresh-токену и отзыв. Владеет таблицей
refresh_tokens в PostgreSQL и blacklist в Redis. Gateway обращается к auth через INT-1
и INT-2.

### 📋 Ответственность

- Генерация JWT access + refresh токенов при логине
- Обновление access-токена по refresh-токену
- Отзыв refresh-токенов
- Хранение refresh-токенов (PostgreSQL) и blacklist (Redis)
- Публикация события AuthTokenIssued

### 📦 Компоненты

| ID | Компонент | Scope | Решение |
|----|-----------|-------|---------|
| CMP-1 | Token Generator | local | Подтверждён из Impact (CMP-1) |
| CMP-2 | Auth Middleware | local | Подтверждён из Impact (CMP-2) |
| CMP-3 | Token Validator | shared (/shared/auth/) | Изменён: Impact (CMP-4, shared (gateway)) → /shared/auth/ |
| CMP-4 | Token Revoker | local | Добавлен Design — blacklist |

### 🔗 Зависимости

- **Предоставляет:** INT-1 (Token Issuance → gateway), INT-2 (Token Refresh → gateway)
- **Потребляет:** INT-3 (User Roles ← users)
- **Публикует:** INT-4 (AuthTokenIssued → audit)

## SVC-2: gateway

**Impact:** SVC-2 (Вторичный, Высокая) | **Решение Design:** Подтверждён

Gateway перехватывает все входящие запросы и выполняет JWT-валидацию через shared Token
Validator из /shared/auth/. Rate limiter переключается с session ID на OAuth2 client_id.

### 📋 Ответственность

- JWT-валидация входящих запросов (через Token Validator из /shared/auth/)
- Rate limiting по OAuth2 client_id
- Проксирование аутентифицированных запросов

### 📦 Компоненты

| ID | Компонент | Scope | Решение |
|----|-----------|-------|---------|
| CMP-5 | Rate Limiter | local | Подтверждён из Impact (CMP-3) — переключение на client_id |

### 🔗 Зависимости

- **Потребляет:** INT-1 (Token Issuance ← auth), INT-2 (Token Refresh ← auth)

## SVC-3: users

**Impact:** SVC-3 (Вторичный, Средняя) | **Решение Design:** Подтверждён

Users предоставляет роли пользователя по запросу auth при генерации JWT-claims. При первом
OAuth2-логине создаёт или обновляет профиль с привязкой к OAuth2-провайдеру.

### 📋 Ответственность

- Предоставление ролей пользователя для JWT-claims
- Создание/обновление профиля при первом OAuth2-логине

### 📦 Компоненты

_Компоненты наследуются из Impact без изменений._

### 🔗 Зависимости

- **Предоставляет:** INT-3 (User Roles → auth)

## INT-1: Token Issuance

**Участники:** auth (provider) ↔ gateway (consumer)
**Паттерн:** sync (REST)
**Источник Impact:** DEP-1 (gateway → auth, sync)

### Контракт

**Endpoint:** `POST /api/v1/auth/token`

**Request:**
` ``json
{
  "grant_type": "password",
  "username": "string",
  "password": "string"
}
` ``

**Response (200):**
` ``json
{
  "access_token": "string (JWT)",
  "refresh_token": "string (opaque)",
  "token_type": "Bearer",
  "expires_in": 3600
}
` ``

**Ошибки:**

| Код | Описание |
|-----|----------|
| 400 | Невалидные credentials |
| 429 | Rate limit exceeded |

### Sequence

` ``mermaid
sequenceDiagram
    participant C as Client
    participant G as Gateway
    participant A as Auth
    participant U as Users

    C->>G: POST /auth/token {credentials}
    G->>A: Forward request
    A->>U: GET /users/{id}/roles
    U-->>A: {roles}
    A->>A: Generate JWT + refresh token
    A-->>G: 200 {access_token, refresh_token}
    G-->>C: 200 {access_token, refresh_token}
` ``

## INT-2: Token Refresh

**Участники:** auth (provider) ↔ gateway (consumer)
**Паттерн:** sync (REST)
**Источник Impact:** DEP-1 (gateway → auth, sync)

### Контракт

**Endpoint:** `POST /api/v1/auth/token`

**Request:**
` ``json
{
  "grant_type": "refresh_token",
  "refresh_token": "string"
}
` ``

**Response (200):**
` ``json
{
  "access_token": "string (JWT)",
  "refresh_token": "string (opaque)",
  "token_type": "Bearer",
  "expires_in": 3600
}
` ``

**Ошибки:**

| Код | Описание |
|-----|----------|
| 400 | Невалидный или отозванный refresh_token |

### Sequence

` ``mermaid
sequenceDiagram

    participant G as Gateway
    participant A as Auth

    C->>G: POST /auth/token {refresh_token}
    G->>A: Forward request
    A->>A: Validate + revoke old + generate new
    A-->>G: 200 {access_token, refresh_token}
    G-->>C: 200 {access_token, refresh_token}
` ``

## INT-3: User Roles

**Участники:** users (provider) ↔ auth (consumer)
**Паттерн:** sync (REST)
**Источник Impact:** DEP-2 (users → auth, sync)

### Контракт

**Ошибки:**

` ``

## INT-4: Auth Events

**Участники:** auth (publisher) ↔ audit (consumer)
**Паттерн:** async (events, message broker)
**Источник Impact:** —

### Контракт

**Событие:** `AuthTokenIssued`
**Канал:** `auth.events`

**Формат:**
` ``json
{
  "event_type": "AuthTokenIssued",
  "user_id": "UUID",
  "issued_at": "ISO 8601",
  "ip_address": "string"
}
` ``

### Sequence

` ``

## 🧪 Системные тест-сценарии

| ID | Сценарий | Участники | Тип | Источник |
|----|----------|-----------|-----|----------|
| STS-1 | Полный цикл: логин → токен → запрос → успех | Client, Gateway, Auth, Users | e2e | INT-1, INT-3 |
| STS-2 | Refresh: access истёк → refresh → новый access → успех | Client, Gateway, Auth | e2e | INT-2 |
```

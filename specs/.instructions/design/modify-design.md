## Принципы

> **Структура по статусам.** Этот документ организован по текущему статусу объекта и переходам между статусами, а не по типам изменений. Статус определяет доступные операции.

> **Шаг 0 — точка входа.** Перед любым изменением определить статус документа (frontmatter → `status`), затем перейти к соответствующей секции.

> **Операции vs переходы.** Операции — изменения внутри текущего статуса. Переходы — смена статуса с условиями и шагами. Документ разделяет их явно.

> **SSOT — Стандарт SDD.** Каскады, условия переходов, уровни обратной связи — [Стандарт SDD § 6](../standard-specs.md#6-последовательность-статусов). Этот документ описывает операции на уровне Design, а не дублирует правила каскадов.

> **Роль — РЕШАТЕЛЬ.** Design РЕШАЕТ распределение ответственностей. При изменениях в Design не выходить за зону ответственности — нет деталей реализации (→ ADR), нет бизнес-обоснований (→ Discussion).

> **Артефакты при WAITING.** Design имеет артефакты (Planned Changes, заглушки, ADR). При переходах WAITING ↔ DRAFT — артефакты создаются/удаляются.

## Статус DRAFT — операции

Все операции ниже применимы **только к документам в статусе DRAFT**.

### Обновление контента

Изменение содержания разделов документа в статусе DRAFT.

#### Шаг 1: Прочитать документ

Прочитать весь документ проектирования.

**Проверить:** статус = DRAFT. Если статус ≠ DRAFT — **СТОП**, см. [Шаг 0](#шаг-0-определить-статус-документа).

#### Шаг 2: Внести изменения

**SSOT:** [standard-design.md § 5](./standard-design.md#5-разделы-документа)

Допустимые изменения:
- Добавление/изменение секций SVC-N (новый номер = max + 1) — включая подсекции (Ответственность, Компоненты, Зависимости)
- Добавление/изменение компонентов (CMP-N) внутри секции SVC
- Добавление/изменение блоков взаимодействия (INT-N) — включая подсекции (Контракт, Sequence)
- Добавление/изменение системных тест-сценариев (STS-N)
- Изменение решений Design (Подтверждён → Изменён, добавление Отклонённых)
- Обновление контрактов API (endpoint, request/response, ошибки)
- Обновление sequence-диаграмм (mermaid)
- Изменение shared-компонентов (scope, размещение)
- Изменение "📋 Резюме"
- Обновление frontmatter (description)

**Правила нумерации:** При добавлении — следующий номер. При удалении — **не перенумеровывать**.

**Зона ответственности ([standard-design.md § 1](./standard-design.md#1-назначение)):**

При изменении убедиться, что документ НЕ содержит:
- Бизнес-обоснование и требования (→ Discussion)
- Предложения затронутых сервисов без решения (→ Impact ПРЕДЛАГАЕТ)
- Детали реализации — алгоритмы, пакеты, классы, библиотеки (→ ADR)
- Архитектурные решения — trade-offs, альтернативы (→ ADR)
- Тестовые сценарии конкретного сервиса (→ План тестов)
- Задачи на реализацию (→ План разработки)

#### Шаг 3: Валидация

```bash
python specs/.instructions/.scripts/validate-design.py specs/design/design-NNNN-topic.md
```

**Если скрипт недоступен:** пройти чек-лист из [validation-design.md](./validation-design.md).

#### Шаг 4: Обновить README

Если изменился `description` или другие отображаемые поля — обновить запись в `specs/design/README.md`.

#### Шаг 5: Отчёт о выполнении

```
## 📋 Отчёт об изменении проектирования

✏️ **Изменён документ проектирования:** `specs/design/design-NNNN-topic.md`

🏷️ **Тип изменения:** Обновление

📝 **Что изменено:**
- {список изменений}

✅ **Валидация:** пройдена
```

### Разрешение маркеров

Замена `[ТРЕБУЕТ УТОЧНЕНИЯ]` маркеров на ответы пользователя.

#### Шаг 1: Собрать маркеры

Найти все `[ТРЕБУЕТ УТОЧНЕНИЯ: ...]` в документе.

```bash
python specs/.instructions/.scripts/validate-design.py specs/design/design-NNNN-topic.md
```

Скрипт покажет количество неразрешённых маркеров (если есть).

#### Шаг 2: Уточнить у пользователя

Показать маркеры пользователю через AskUserQuestion. Для каждого маркера предложить несколько решений, включая рекомендацию, и получить ответ от пользователя.

#### Шаг 3: Заменить маркеры

Заменить каждый `[ТРЕБУЕТ УТОЧНЕНИЯ: вопрос]` на ответ пользователя.

**Dependency Barrier** ([Стандарт SDD § 8](../standard-specs.md#dependency-barrier)): при создании документа LLM может остановить генерацию блоком `⛔ DEPENDENCY BARRIER`, если следующая секция зависит от неразрешённого маркера. Если в документе есть такой блок — после разрешения маркеров продолжить генерацию оставшихся секций.

#### Шаг 4: Валидация

```bash
python specs/.instructions/.scripts/validate-design.py specs/design/design-NNNN-topic.md
```

Проверить: маркеров не осталось.

## Переход: DRAFT → WAITING

**SSOT:** [standard-design.md § 4](./standard-design.md#4-переходы-статусов) | [Стандарт SDD § 6.1](../standard-specs.md#61-draft-to-waiting)

Единственный переход, управляемый на уровне Design. Все последующие переходы — на уровне цепочки.

### Условия (блокирующие)

Все условия должны быть выполнены:

| Условие | Проверка |
|---------|----------|
| Статус = DRAFT | frontmatter `status: DRAFT` |
| Нет `[ТРЕБУЕТ УТОЧНЕНИЯ]` | Ни одного маркера в документе |
| Нет [Dependency Barrier](../standard-specs.md#dependency-barrier) | Нет `⛔ DEPENDENCY BARRIER` |
| Валидация пройдена | Скрипт validate-design.py → 0 ошибок |

```bash
python specs/.instructions/.scripts/validate-design.py specs/design/design-NNNN-topic.md
```

**Если условия не выполнены:** сообщить пользователю какие — предложить разрешить маркеры или исправить ошибки.

### Шаг 1: Подтверждение пользователя

**БЛОКИРУЮЩЕЕ.** AskUserQuestion: "Документ проектирования готов к переводу в WAITING. Подтверждаете?"

| Ответ | Действие |
|-------|----------|
| Да | Перевести в WAITING |
| Нет | Оставить в DRAFT |

### Шаг 2: Обновить статус

В frontmatter документа: `status: DRAFT` → `status: WAITING`.

### Шаг 3: Обновить README

Обновить статус в `specs/design/README.md`: `DRAFT` → `WAITING`.

### Шаг 4: Создать артефакты

**SSOT:** [standard-design.md § 4 — Артефакты Design → WAITING](./standard-design.md#4-переходы-статусов)

| # | Артефакт | Действие |
|---|----------|----------|
| 1 | **Planned Changes** в `services/{svc}.md` | Для каждого SVC-N: добавить запись в секцию Planned Changes. Вызвать `/service-modify` или вручную |
| 2 | **Planned Changes** в `system/overview.md`, `domains/{domain}.md` | Если Design затрагивает системную/доменную архитектуру |
| 3 | **Заглушка сервиса** (только для новых) | Для сервисов "Подтверждён (новый)" / "Добавлен Design": вызвать `/service-create {svc}` |
| 4 | **ADR-документы** (1:N) | Для каждого SVC-N: создать ADR. Обновить `children` в frontmatter Design |

**Порядок:** артефакты 1-3 перед ADR (артефакт 4).

### Каскад DRAFT (возврат из WAITING)

**SSOT:** [Стандарт SDD § 6.1](../standard-specs.md#61-draft-to-waiting)

При возврате документа из WAITING → DRAFT (контекст родителя изменился) все его WAITING-дочерние тоже → DRAFT.

**На уровне Design:** если Design возвращается в DRAFT — его ADR (если в WAITING) тоже → DRAFT. Design снова открыт для операций из секции [Статус DRAFT — операции](#статус-draft-операции).

**Артефакты при возврате в DRAFT:** Planned Changes и заглушки, созданные при WAITING, **удаляются** или помечаются неактивными. ADR-документы возвращаются в DRAFT (не удаляются).

**Когда это происходит:**
- Пользователь решил внести изменения после одобрения
- Parent Impact вернулся в DRAFT → каскад: Design (если WAITING) → DRAFT
- Design вернулся из CONFLICT → при разрешении пользователь изменил контент (→ DRAFT вместо WAITING)

## Upward feedback при WAITING

**SSOT:** [Стандарт SDD § 3.6 — Upward feedback](../standard-specs.md#36-upward-feedback), [standard-design.md § 5](./standard-design.md#5-разделы-документа)

При работе на нижестоящих уровнях (ADR, Plan тестов и т.д.) LLM может обнаружить информацию, которая должна быть отражена в Design. **Статус остаётся WAITING** — без перевода в DRAFT.

**Отличие от "Каскад DRAFT":** Каскад DRAFT (WAITING → DRAFT) — пользователь решил переработать документ. Upward feedback — LLM дополняет документ точечно, пользователь подтверждает. Это разные сценарии.

#### Шаг 1: LLM предлагает дополнение

LLM формулирует конкретное дополнение (новый блок INT, уточнение контракта, новый STS) и предлагает пользователю через AskUserQuestion.

#### Шаг 2: Пользователь подтверждает

Пользователь подтверждает, корректирует или отклоняет предложение.

#### Шаг 3: Внести изменения

Дополнить затронутые секции Design. Нумерация элементов продолжается (следующий номер = max + 1).

#### Шаг 4: Валидация

```bash
python specs/.instructions/.scripts/validate-design.py specs/design/design-NNNN-topic.md
```

## Переход: WAITING → RUNNING

**SSOT:** [Стандарт SDD § 6.2](../standard-specs.md#62-waiting-to-running)

> **Tree-level.** Переход управляется на уровне цепочки, не Design.

**Триггер:** все документы цепочки (Discussion → Impact → ... → Plan Dev) в WAITING. LLM предлагает через AskUserQuestion: "Все спецификации готовы. Перейти в RUNNING?" Пользователь подтверждает.

**На уровне Design:** статус меняется `WAITING` → `RUNNING`. Операций нет — документ переходит в режим реализации.

---

## Статус RUNNING — ограничения

> **Прямые изменения запрещены.** Документ в RUNNING — согласованная спецификация. Изменения возможны только через CONFLICT ([Стандарт SDD § 6.3](../standard-specs.md#63-running-to-conflict)).

Если обнаружена необходимость изменить Design в RUNNING — это сигнал CONFLICT-уровня. См. [Переход: RUNNING → CONFLICT](#переход-running-conflict).

---

## Переход: RUNNING → CONFLICT

**SSOT:** [Стандарт SDD § 6.3](../standard-specs.md#63-running-to-conflict)

> **Tree-level каскад.** При обнаружении CONFLICT-уровня проблемы **все** документы цепочки → CONFLICT.

**Триггер:** обратная связь от кода выявила несовместимость со спецификациями на уровне Design или выше ([Стандарт SDD § 6.3 — уровни обратной связи](../standard-specs.md#63-running-to-conflict)).

**На уровне Design:** статус меняется `RUNNING` → `CONFLICT`. Переход инициируется на уровне цепочки — Design сам его не вызывает.

---

## Статус CONFLICT — операции

### Как Design попадает в CONFLICT

Design попадает в CONFLICT через tree-level каскад ([Стандарт SDD § 6.3](../standard-specs.md#63-running-to-conflict)): при обнаружении CONFLICT на любом уровне **все** документы цепочки → CONFLICT, включая Design.

LLM определяет самый высокий затронутый документ — снизу вверх, от Plan до Discussion: "Содержание этого документа стало неверным?"

### Операции при CONFLICT

**Если Design затронут** (распределение ответственностей оказалось неточным — новые сервисы, изменённые контракты, неверные блоки взаимодействия):

1. LLM читает **весь документ** целиком
2. Вносит **точечные правки** в затронутые секции (SVC, INT, STS, контракты), сохраняя остальной контент
3. Пользователь ревьюит изменения → документ → WAITING ([Переход: CONFLICT → WAITING](#переход-conflict-waiting))

**Если Design НЕ затронут** (конфликт локализован на уровне ADR или ниже):

1. LLM и пользователь **верифицируют** документ без изменений
2. Подтверждают, что распределение ответственностей остаётся верным
3. Документ → WAITING ([Переход: CONFLICT → WAITING](#переход-conflict-waiting))

## Переход: CONFLICT → WAITING

**SSOT:** [Стандарт SDD § 6.4](../standard-specs.md#64-conflict-to-waiting)

> **Per-document.** Каждый документ переходит в WAITING независимо после разрешения.

**Разрешение — сверху вниз:** начиная с самого высокого затронутого документа. Если Design — самый высокий затронутый, разрешение начинается с него.

**Шаги:**

1. LLM исправляет документ (или верифицирует без изменений)
2. Пользователь ревьюит → одобряет
3. Статус: `CONFLICT` → `WAITING`
4. Обновить README: `CONFLICT` → `WAITING`
5. Обновить артефакты (Planned Changes, если контракты изменились)
6. Когда **все** документы цепочки в WAITING → каскад RUNNING ([Стандарт SDD § 6.2](../standard-specs.md#62-waiting-to-running))

**Если пользователь отклоняет разрешение:**

| Исход | Переход |
|-------|---------|
| Конфликт разрешён | CONFLICT → WAITING |
| Конфликт неразрешим | → ROLLING_BACK ([Стандарт SDD § 6.6](../standard-specs.md#66-to-rolling_back)) |
| Пользователь отклоняет | → ROLLING_BACK ([Стандарт SDD § 6.6](../standard-specs.md#66-to-rolling_back)) |

## Переход: RUNNING → DONE

**SSOT:** [Стандарт SDD § 6.5](../standard-specs.md#65-running-to-done)

> **Bottom-up каскад.** Design → DONE когда все ADR (дочерние, 1:N) → DONE.

**На уровне Design:** статус меняется `RUNNING` → `DONE` автоматически. Переход инициируется снизу вверх.

После перехода в DONE:
- Обновить README: `RUNNING` → `DONE`
- Выполнить побочные эффекты (см. [Статус DONE — ограничения и побочные эффекты](#статус-done-ограничения-и-побочные-эффекты))

## Статус DONE — ограничения и побочные эффекты

**SSOT:** [standard-design.md § 1](./standard-design.md#1-назначение) → "После DONE", [Стандарт SDD § 7.3](../standard-specs.md#73-обновление-при-реализации-to-done)

> **Design — архивная запись проектных решений.** Прямые изменения запрещены.

**Допустимые изменения:**
- Исправление опечаток (typo corrections)
- Исправление битых ссылок

**Изменение scope:** при изменении scope — CONFLICT через обратную связь от кода.

**Побочные эффекты при DONE:**

| # | Что обновляется | Как |
|---|----------------|-----|
| 1 | `specs/architecture/system/overview.md` | AS IS обновляется, Planned Changes → Changelog |
| 2 | `specs/architecture/domains/{domain}.md` | AS IS обновляется, Planned Changes → Changelog |
| 3 | `specs/tests/system/` | Системные тест-сценарии (STS-N) → AS IS |

---

## Переход: → ROLLING_BACK

**SSOT:** [Стандарт SDD § 6.6](../standard-specs.md#66-to-rolling_back)

> **Tree-level.** Все документы цепочки → ROLLING_BACK.

**Триггеры:**
- Пользователь даёт команду на откат
- Конфликт неразрешим ([Стандарт SDD § 6.4](../standard-specs.md#64-conflict-to-waiting))
- Пользователь отклоняет разрешение конфликта

**На уровне Design:** Design **имеет артефакты** — откат **не no-op**.

**Шаги отката артефактов:**

| # | Артефакт | Действие |
|---|----------|----------|
| 1 | Planned Changes в `services/{svc}.md` | Удалить записи Planned Changes, добавленные этим Design |
| 2 | Planned Changes в `system/overview.md`, `domains/` | Удалить записи |
| 3 | Заглушки новых сервисов | Удалить файлы-заглушки (если сервис не создан другим Design) |
| 4 | ADR-документы | ADR → ROLLING_BACK (каскад вниз) |

- Обновить frontmatter: `status: ROLLING_BACK`
- Обновить README: статус → `ROLLING_BACK`

---

## Переход: ROLLING_BACK → REJECTED

**SSOT:** [Стандарт SDD § 6.7](../standard-specs.md#67-rolling_back-to-rejected)

> **REJECTED — финальный статус.** Изменения запрещены.

**Условие:** LLM проверяет, что все документы цепочки в ROLLING_BACK и артефакты каждого уровня откачены → вся цепочка → REJECTED.

**На уровне Design:** статус меняется `ROLLING_BACK` → `REJECTED`.

- Обновить frontmatter: `status: REJECTED`
- Обновить README: статус → `REJECTED`

**Перезапуск:** если бизнес-потребность остаётся актуальной, создать **новую** Discussion и **новую** цепочку.

## Обновление ссылок

Design содержит ссылки в frontmatter (`parent`, `children`). При изменении путей связанных документов:

1. Обновить `parent` — если Impact переименован
2. Обновить `children` — если ADR переименованы
3. Обновить запись в `specs/design/README.md` (колонки Parent Impact, ADR)

## Чек-лист

### Статус DRAFT — обновление
- [ ] Статус = DRAFT
- [ ] Изменения внесены
- [ ] Нумерация корректна (нет дублей, нет перенумерации)
- [ ] Зона ответственности не нарушена (нет деталей реализации → ADR)
- [ ] Валидация пройдена
- [ ] README обновлён (если нужно)

### Разрешение маркеров
- [ ] Все маркеры собраны
- [ ] Ответы получены от пользователя
- [ ] Маркеры заменены на ответы
- [ ] [Dependency Barrier](../standard-specs.md#dependency-barrier) разрешён (если был)
- [ ] Валидация пройдена — маркеров нет

### Upward feedback при WAITING
- [ ] Статус = WAITING
- [ ] LLM предложил дополнение
- [ ] Пользователь подтвердил
- [ ] Секции дополнены, нумерация продолжена
- [ ] Валидация пройдена
- [ ] Статус остаётся WAITING (без перевода в DRAFT)

### Переход DRAFT → WAITING
- [ ] Статус = DRAFT
- [ ] Нет `[ТРЕБУЕТ УТОЧНЕНИЯ]`
- [ ] Нет [Dependency Barrier](../standard-specs.md#dependency-barrier)
- [ ] Валидация пройдена — 0 ошибок
- [ ] Пользователь подтвердил перевод
- [ ] Статус обновлён в frontmatter
- [ ] Статус обновлён в README
- [ ] Артефакты созданы (Planned Changes, заглушки, ADR)

### Переход CONFLICT → WAITING
- [ ] Определено: Design затронут или нет
- [ ] Если затронут — точечные правки внесены
- [ ] Если не затронут — верификация пройдена
- [ ] Пользователь одобрил разрешение
- [ ] Статус обновлён в frontmatter
- [ ] Статус обновлён в README
- [ ] Артефакты обновлены (если контракты изменились)

### Статус DONE — побочные эффекты
- [ ] `specs/architecture/system/` обновлён (AS IS)
- [ ] `specs/architecture/domains/` обновлён (AS IS)
- [ ] `specs/tests/system/` обновлён (STS → AS IS)
- [ ] Только typo corrections или битые ссылки далее
- [ ] Изменение scope → CONFLICT через обратную связь

### Переход → ROLLING_BACK / REJECTED
- [ ] Артефакты Design откачены (Planned Changes удалены, заглушки удалены)
- [ ] ADR → ROLLING_BACK (каскад)
- [ ] Статус обновлён в frontmatter
- [ ] Статус обновлён в README

## Примеры

### Добавление блока взаимодействия (DRAFT)

```
Пользователь: "Добавь блок взаимодействия billing → payment-gateway"

1. Прочитать design-0001-oauth2-authorization.md
2. Шаг 0: статус = DRAFT → секция "Статус DRAFT — операции"
3. Определить следующий номер: INT-5
4. Добавить секцию: ## INT-5: Payment Processing
   с метаданными, подсекцией Контракт и Sequence
5. Обновить зависимости затронутых SVC
6. Добавить STS-4 для нового INT
7. Обновить "📋 Резюме" (количество INT/STS)
8. Валидация → ОК
```

### Разрешение маркеров перед WAITING (DRAFT)

```
Маркеры в design-0001:
- [ТРЕБУЕТ УТОЧНЕНИЯ: Token Validator — /shared/auth/ или /shared/jwt/?]
- [ТРЕБУЕТ УТОЧНЕНИЯ: gateway → auth — REST или gRPC?]

1. Шаг 0: статус = DRAFT → секция "Разрешение маркеров"
2. Показать маркеры пользователю
3. Ответы: "/shared/auth/", "REST"
4. Заменить маркеры на ответы, обновить контракт INT-1
5. Валидация → 0 маркеров → можно в WAITING
```

### Перевод в WAITING (DRAFT → WAITING)

```
1. Шаг 0: статус = DRAFT → секция "Переход: DRAFT → WAITING"
2. Проверить условия: маркеров нет, валидация ОК
3. AskUserQuestion: "Перевести design-0001 в WAITING?"
4. Пользователь: "Да"
5. status: DRAFT → status: WAITING
6. README обновлён
7. Артефакты: Planned Changes (3 сервиса), заглушки (0), ADR (3 документа)
```

### Upward feedback (WAITING)

```
Ситуация: при создании ADR для auth обнаружено, что Design не учёл зависимость
от кэширования публичных ключей (Redis).

1. Шаг 0: статус = WAITING → секция "Upward feedback при WAITING"
2. LLM предлагает: "Добавить INT-5: auth → Redis (кэш публичных ключей)"
3. AskUserQuestion: "Добавить блок взаимодействия с Redis?"
4. Пользователь: "Да"
5. Добавить INT-5, обновить SVC-1 (зависимости), добавить STS-4
6. Проверить Impact: нужен ли upward feedback → добавить DEP
7. Валидация → ОК
8. Статус остаётся WAITING
```

### Разрешение CONFLICT (CONFLICT → WAITING)

```
Ситуация: вся цепочка → CONFLICT из-за обратной связи от кода.
LLM определил самый высокий затронутый уровень: Design.
Design затронут — контракт INT-1 несовместим с реализацией.

1. Шаг 0: статус = CONFLICT → секция "Статус CONFLICT — операции"
2. Design затронут → точечные правки: обновить контракт INT-1 (новый формат response)
3. Обновить sequence INT-1
4. LLM и пользователь ревьюят изменения
5. status: CONFLICT → status: WAITING
6. README обновлён
7. Planned Changes обновлены
```

### Design → DONE (RUNNING → DONE)

```
Ситуация: все ADR (дочерние, 1:N) → DONE → bottom-up каскад.

1. Design автоматически → DONE
2. README обновлён: RUNNING → DONE
3. Побочные эффекты:
   - specs/architecture/system/overview.md → AS IS обновлён
   - specs/architecture/domains/ → AS IS обновлён
   - specs/tests/system/ → STS перенесены
4. Design — архивная запись. Изменение scope → CONFLICT
```

### Откат артефактов (ROLLING_BACK)

```
Ситуация: вся цепочка → ROLLING_BACK.

1. Design → ROLLING_BACK
2. Откат артефактов:
   - Planned Changes удалены из services/auth.md, services/gateway.md, services/users.md
   - Planned Changes удалены из system/overview.md
   - Заглушки новых сервисов удалены (если были)
   - ADR → ROLLING_BACK (каскад)
3. README обновлён: ROLLING_BACK
4. → REJECTED после проверки всей цепочки
```

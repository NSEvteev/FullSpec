## 1. Назначение

**Зона ответственности:** НА ЧТО ВЛИЯЕТ ([Стандарт SDD § 3](../standard-specs.md#3-зоны-ответственности)).

Импакт-анализ отвечает на вопросы: **Какие сервисы предположительно затронуты? Какие риски?**

**Роль — ПРЕДЛАГАТЕЛЬ:** Impact **ПРЕДЛАГАЕТ** варианты, а не принимает окончательных решений. LLM быстро сканирует `specs/architecture/` и предлагает варианты затронутых сервисов (1-3). Design критически оценивает предложения Impact и **РЕШАЕТ** окончательное распределение ([Стандарт SDD § 3](../standard-specs.md#3-зоны-ответственности)).

**Источник контекста — Quick Scan:** Impact обязан прочитать архитектурные документы **до** фазы Clarify. Порядок чтения:

| # | Что читать | Зачем |
|---|-----------|-------|
| 1 | **`specs/architecture/services/README.md`** — таблица сервисов | Список существующих сервисов, технологии, ключевые API |
| 2 | **`specs/architecture/system/overview.md`** | Общая картина, потоки между сервисами |

Quick Scan — **только 2 файла**. Достаточно для быстрого определения затронутых сервисов без загрузки контекста оркестратора. Детальное чтение `specs/architecture/services/{svc}.md` — задача Design-агента ([Стандарт SDD § 14, решение #45](../standard-specs.md#14-решения)).

**Нет сервисов:** Если `specs/architecture/services/README.md` не содержит сервисов (таблица пуста) — Quick Scan всё равно выполняется (overview.md может содержать контекст). Impact опирается на Discussion + Clarify и предлагает все сервисы как "Новый (план создания)" с уверенностью "Предположительно".

Подробнее об архитектурных документах: [standard-service.md](../living-docs/service/standard-service.md).

**Содержит:**
- Резюме влияния (краткое описание scope)
- Варианты затронутых сервисов с уровнем уверенности
- Затронутые компоненты внутри сервисов (shared-маркировка при необходимости)
- Влияние на данные и хранение
- Влияние на API (новые/изменённые эндпоинты)
- Зависимости между сервисами
- Планы создания новых сервисов (если нужного сервиса не существует)
- Риски и mitigation-стратегии

**НЕ содержит:**
- Бизнес-обоснование и требования (→ Discussion)
- Окончательное распределение ответственностей между сервисами (→ Design РЕШАЕТ)
- Контракты API с полными request/response схемами (→ Design)
- Архитектурные решения (→ ADR)
- Тестовые сценарии (→ План тестов)
- Задачи на реализацию (→ План разработки)

**Граничные случаи (Impact vs Design):**
- Допустимо: "auth-сервис предположительно затронут — нужна JWT-валидация" — предложение варианта
- Допустимо: "Новый эндпоинт POST /auth/token" — высокоуровневое описание API-изменений
- Допустимо: "Компонент может быть shared между auth и gateway" — маркировка shared
- Допустимо: "Нужен новый notification-service" — план создания нового сервиса
- Недопустимо: "auth-сервис отвечает за токены, gateway — за rate limiting" — окончательное распределение ответственностей (→ Design РЕШАЕТ)
- Недопустимо: Полная request/response спецификация эндпоинта (→ Design)

Impact **предлагает** варианты затронутых сервисов и описывает характер влияния. **Окончательное** распределение ответственностей — задача Design, который принимает сомнения и предположения Impact и РЕШАЕТ.

**Когда создавать:**
- После одобрения Discussion (статус WAITING)
- Всегда один Impact на одну Discussion (1:1)

**Связи:** Parent — Discussion (обязателен, 1:1). Дочерний объект — Design (1:1). ([Стандарт SDD § 4](../standard-specs.md#4-связи-и-frontmatter))

**После DONE:** Impact становится архивной записью анализа влияния. Изменения запрещены (только исправление опечаток). При изменении scope — CONFLICT через обратную связь от кода.

## 2. Расположение и именование

**Расположение:** `specs/impact/`

**Формат имени:** `impact-NNNN-topic.md`

| Элемент | Правило |
|---------|---------|
| `impact` | Фиксированный префикс |
| `NNNN` | Четырёхзначный автоинкремент (0001, 0002, ...) |
| `topic` | Kebab-case, латиница, описательное |

**Regex:** `^impact-(\d{4})-(.+)\.md$`

**Нумерация:** Независимая в папке `specs/impact/`. Следующий номер = максимальный существующий + 1.

**Формат README-таблицы** (`specs/impact/README.md`):

| ID | Документ | Статус | Parent Discussion | Design | Milestone | Описание |
|----|----------|--------|-------------------|--------|-----------|----------|
| 0001 | impact-0001-oauth2-authorization.md | RUNNING | disc-0001 | design-0001 | v1.2.0 | Импакт-анализ OAuth2 авторизации |

## 3. Frontmatter

**SSOT frontmatter:** [standard-frontmatter.md](/.structure/.instructions/standard-frontmatter.md) ([§ 1 — базовые поля](/.structure/.instructions/standard-frontmatter.md#1-обязательные-поля) + [§ 4 — поля specs](/.structure/.instructions/standard-frontmatter.md#4-дополнительные-поля-для-спецификаций-specs))

| Поле | Значение для Impact | Обязательность |
|------|---------------------|----------------|
| `description` | Описание импакт-анализа (до 1024 символов) | Обязательно |
| `standard` | `specs/.instructions/impact/standard-impact.md` | Обязательно |
| `standard-version` | `v1.0` | Обязательно |
| `index` | `specs/impact/README.md` | Обязательно |
| `parent` | Путь к Discussion-документу (обязателен) | Обязательно |
| `children` | Путь к Design-документу (1:1) | Обязательно (после создания Design) |
| `status` | Текущий статус ([Стандарт SDD § 7](../standard-specs.md#7-статусы)) | Обязательно |
| `milestone` | Наследуется от parent Discussion | Обязательно |

```yaml
---
description: Импакт-анализ OAuth2 авторизации — затронутые сервисы, компоненты, API, риски.
standard: specs/.instructions/impact/standard-impact.md
standard-version: v1.0
index: specs/impact/README.md
parent: discussion/disc-0001-oauth2-authorization.md
children:
  - design/design-0001-oauth2-service-design.md
status: DRAFT
milestone: v1.2.0
---
```

**Parent (обязателен):** Путь к Discussion-документу. Impact всегда имеет parent — Discussion (1:1). Без parent документ невалиден.

**Milestone:** Наследуется от parent Discussion. Значение milestone в Impact **всегда совпадает** с milestone в parent Discussion. При создании Impact LLM **обязан прочитать milestone из parent** и скопировать его в Impact без уточнения с пользователем. Вопросы о milestone на уровне Impact не задаются.

## 4. Переходы статусов

Happy path — нормальный поток жизненного цикла Impact ([Стандарт SDD § 8](../standard-specs.md#8-последовательность-статусов)).

**Зона ответственности Impact:** только переход DRAFT → WAITING (создание и модификация документа). Все последующие переходы управляются на уровне цепочки — дочерними документами, каскадами или обратной связью от кода.

| Текущий статус | Условие перехода | Следующий статус | Кто управляет |
|---|---|---|---|
| **DRAFT** | Пользователь одобрил документ. Нет неразрешённых `[ТРЕБУЕТ УТОЧНЕНИЯ]`. Нет Dependency Barrier | **WAITING** | Impact (этот документ) |
| **WAITING** | Все документы цепочки (Discussion → ... → Plan Dev) в WAITING. Пользователь подтвердил запуск | **RUNNING** | Цепочка ([§ 8.2](../standard-specs.md#82-waiting-to-running)) |
| **RUNNING** | Обратная связь от кода затронула уровень Impact ([§ 8.3](../standard-specs.md#83-running-to-conflict)) | **CONFLICT** | Цепочка (tree-level) |
| **CONFLICT** | LLM исправил документ, пользователь одобрил исправление ([§ 8.4](../standard-specs.md#84-conflict-to-waiting)) | **WAITING** | Цепочка (top-down) |
| **RUNNING** | Design (дочерний, 1:1) → DONE. Каскад снизу вверх ([§ 8.5](../standard-specs.md#85-running-to-done)) | **DONE** | Цепочка (bottom-up) |

Откат и отклонение (ROLLING_BACK, REJECTED): [Стандарт SDD § 8.6–8.7](../standard-specs.md#86-to-rolling_back).

## 5. Разделы документа

Импакт-анализ — **структурированный документ с фиксированными разделами** (см. таблицу ниже). Порядок разделов неизменен.

| Раздел | Нумерация | Обязательность | Заглушка (если нет контента) |
|--------|-----------|---------------|------------------------------|
| **📋 Резюме** | — | Обязательный (контент обязателен) | — |
| **🏗️ Затронутые сервисы** | SVC-N | Обязательный (мин. 1 элемент) | — |
| **📦 Компоненты** | CMP-N | Обязательный | _Компоненты не идентифицированы._ |
| **💾 Данные и хранение** | DATA-N | Обязательный | _Изменений в данных нет._ |
| **🔌 API** | API-N | Обязательный | _Изменений в API нет._ |
| **🔗 Зависимости** | DEP-N | Обязательный | _Зависимостей нет._ |
| **⚠️ Риски** | RISK-N | Обязательный (мин. 1 элемент) | — |

**Все 7 разделов присутствуют всегда.** Разделы "Резюме", "Затронутые сервисы" и "Риски" всегда заполняются контентом. Остальные разделы заполняются контентом или заглушкой (курсивом). Заглушка означает: раздел рассмотрен, контент не требуется.

**Правила нумерации (SVC-N, CMP-N, DATA-N, API-N, DEP-N, RISK-N):** Последовательная автоинкрементная. При добавлении — следующий номер = максимальный существующий + 1. При удалении — **не перенумеровывать** (пропуски допустимы, сохраняется ссылочная целостность).

**Заглушки vs маркеры `[ТРЕБУЕТ УТОЧНЕНИЯ]`:**
- **Заглушка** (например, _Изменений в данных нет._) — LLM **уверен**, что секция не применима. Решение основано на анализе Discussion и ответах пользователя при Clarify.
- **Маркер** `[ТРЕБУЕТ УТОЧНЕНИЯ: ...]` — LLM **не имеет достаточной информации** для решения. Ставится, когда невозможно определить, нужен ли контент в секции.
- При `--auto-clarify`: если Discussion не даёт ясности по секции — ставить маркер, **не заглушку**. Заглушка при `--auto-clarify` допустима только если из Discussion однозначно следует отсутствие влияния.

**Upward feedback:** При анализе влияния Impact может обнаружить информацию, затрагивающую Discussion (новые требования, неучтённые ограничения).

**Механика ([Стандарт SDD § 6](../standard-specs.md#upward-feedback)):**
1. Работа с Impact **приостанавливается**
2. LLM дополняет Discussion (статус остаётся **WAITING** — без перевода в DRAFT)
3. Пользователь подтверждает изменения
4. Работа с Impact **возобновляется** (с учётом обновлённой Discussion)

**Примеры:**
- **Требует upward feedback:** обнаружено требование к zero-downtime миграции, не указанное в критериях успеха → дополняется Discussion
- **Требует upward feedback:** обнаружена зависимость от внешнего сервиса, не упомянутого в scope → дополняется Discussion
- **НЕ требует:** уточнение компонентов внутри уже затронутого сервиса (детализация, не новая информация для Discussion)

### 📋 Резюме

Краткое описание влияния изменений. Формат: свободный текст (1–3 абзаца).

**Обязательное содержимое:**
- Общий scope влияния (количество затронутых сервисов)
- Характер изменений (новая функциональность / рефакторинг / миграция)
- Ключевые risk areas (если риски существуют — указать уровень, иначе: "критичных рисков не выявлено")

```markdown
## 📋 Резюме

Переход на OAuth2 затрагивает 3 сервиса: auth (основной — генерация и валидация токенов),
gateway (проксирование и rate limiting) и users (профили и роли). Основной риск —
миграция существующих session-based клиентов без даунтайма.
```

### 🏗️ Затронутые сервисы

Список сервисов, на которые предположительно влияют изменения. **Формат: таблица.** Нумерация: `SVC-N`. Минимум 1 элемент обязателен.

**Источник:** LLM определяет сервисы по `specs/architecture/` (README, перечень сервисов). Если `specs/architecture/` пуст или проект новый — LLM опирается на Discussion и задаёт уточняющие вопросы при Clarify.

**Несуществующие сервисы:** Если для реализации требований Discussion нужен сервис, которого ещё нет — Impact предлагает тип влияния "Новый (план создания)". Окончательное решение о создании принимает Design.

```markdown
## 🏗️ Затронутые сервисы

| ID | Сервис | Тип влияния | Уверенность | Описание |
|----|--------|-------------|-------------|----------|
| SVC-1 | auth | Основной | Высокая | Генерация JWT-токенов, валидация, refresh-механизм |
| SVC-2 | gateway | Вторичный | Высокая | Rate limiting, проксирование авторизованных запросов |
| SVC-3 | users | Вторичный | Средняя | Хранение ролей, привязка к OAuth2-профилю |
| SVC-4 | notification-service | Новый (план создания) | Предположительно | Уведомления о событиях авторизации (email, push) |
```

**Тип влияния:**

| Тип | Описание |
|-----|----------|
| Основной | Сервис — центр изменений, максимальный объём работ |
| Вторичный | Сервис затронут через зависимости или интеграции |
| Косвенный | Сервис не меняется, но может быть затронут (мониторинг, логи) |
| Новый (план создания) | Сервис не существует, Impact предлагает его создание. Design РЕШАЕТ |

**Уверенность:**

| Уровень | Описание |
|---------|----------|
| Высокая | Однозначно следует из Discussion и `specs/architecture/` |
| Средняя | Вероятно затронут, требуется подтверждение при Clarify или на уровне Design |
| Предположительно | Impact предполагает на основе косвенных данных. Design верифицирует |

### 📦 Компоненты

Затронутые компоненты внутри сервисов. **Формат: таблица.** Нумерация: `CMP-N`.

**Что считать компонентом:** Функциональная единица сервиса (middleware, service, generator), **без** указания пакетной структуры или классов. Примеры:
- Допустимо: "Auth Middleware", "Token Generator", "Rate Limiter"
- Недопустимо: "auth.tokens.JWTTokenService", "gateway.middleware.RateLimitMiddleware" (детали архитектуры → Design)

**Shared-компоненты:** Если компонент может использоваться несколькими сервисами — указать `shared` в колонке "Scope" и перечислить сервисы в колонке "Сервис" через запятую. Design принимает решение о размещении shared-компонента (в `/shared/` или внутри сервиса-владельца).

**Критерий полноты:** LLM обязан рассмотреть все слои сервиса:
- API-слой (endpoints, middleware, валидация)
- Бизнес-логика (services, handlers, processors)
- Слой данных (repositories, models, миграции)
- Инфраструктура (логирование, мониторинг, конфигурация)

Если слой не затронут — явно указать в Clarify или поставить заглушку.

```markdown
## 📦 Компоненты

| ID | Сервис | Компонент | Scope | Изменение |
|----|--------|-----------|-------|-----------|
| CMP-1 | auth | Token Generator | auth | Новый — генерация JWT access + refresh токенов |
| CMP-2 | auth | Auth Middleware | auth | Модификация — добавление JWT-валидации |
| CMP-3 | gateway | Rate Limiter | gateway | Модификация — учёт OAuth2-токенов |
| CMP-4 | auth, gateway | Token Validator | shared | Новый — валидация JWT-токенов (используется auth и gateway) |
```

### 💾 Данные и хранение

Влияние на схемы данных, миграции, хранилища. **Формат: таблица.** Нумерация: `DATA-N`.

```markdown
## 💾 Данные и хранение

| ID | Сервис | Хранилище | Изменение |
|----|--------|-----------|-----------|
| DATA-1 | auth | PostgreSQL | Новая таблица `refresh_tokens` |
| DATA-2 | auth | Redis | Хранение blacklist отозванных токенов |
```

### 🔌 API

Влияние на API — новые, изменённые или удалённые эндпоинты. **Формат: таблица.** Нумерация: `API-N`.

**Уровень детализации:** Метод + путь + краткое назначение. **Запрещено** указывать параметры запроса, формат ответа, статус-коды — это уровень Design. Impact показывает **что** меняется в API, Design описывает **как** (контракты).

**Колонка "Совместимость":**
- `N/A` — не применимо (новый эндпоинт)
- `Backward-compatible` — изменение не ломает существующих клиентов
- `Breaking (migration period: N)` — breaking change с указанием периода миграции
- `Deprecated (removal: vX.Y.Z)` — помечен устаревшим, срок удаления

```markdown
## 🔌 API

| ID | Сервис | Эндпоинт | Изменение | Совместимость |
|----|--------|----------|-----------|---------------|
```

## 6. Clarify

**SSOT:** [Стандарт SDD § 10](../standard-specs.md#10-clarify-и-блокирующие-правила) — полные правила Clarify, маркер `[ТРЕБУЕТ УТОЧНЕНИЯ]`, Dependency Barrier.

**Паттерн объекта:** [Стандарт SDD § 5 "Общий паттерн объекта"](../standard-specs.md#общий-паттерн-объекта) — цикл CLARIFY → GENERATE → VALIDATE → REVIEW.

**Принцип Clarify на уровне Impact:** LLM **сам анализирует** Quick Scan + Discussion и **предлагает** свои выводы. Пользователь подтверждает или корректирует предложения. LLM **НЕ спрашивает** пользователя "какие сервисы затронуты?" — это работа LLM.

| Что предлагает LLM | Пример предложения через AskUserQuestion |
|---------------------|------------------------------------------|
| Затронутые сервисы | "По Discussion и architecture, затронуты auth (основной) и gateway (вторичный). Верно? Есть ли другие?" |
| Компоненты | "В auth предположительно затронуты: Token Generator, Auth Middleware. Подтверждаете?" |
| Данные | "Потребуется новая таблица refresh_tokens в PostgreSQL. Есть ли другие изменения данных?" |
| API | "Предлагается новый эндпоинт POST /auth/token. Есть ли другие API-изменения?" |
| Зависимости | "Появляется зависимость gateway → auth (sync REST). Верно?" |
| Риски | "Основной риск — даунтайм при миграции session → OAuth2. Есть ли требования к zero-downtime?" |

**Контекст из Discussion:** **На шаге CLARIFY** (до генерации) LLM **обязан** прочитать parent Discussion целиком через frontmatter-поле `parent`: все разделы (проблема, фичи, требования, критерии успеха). Impact строится на основе Discussion. Чтение происходит **до** первого AskUserQuestion для формирования уточняющих вопросов с учётом контекста Discussion.

**Контекст из Architecture:** **На шаге CLARIFY** (до генерации) LLM **обязан** выполнить Quick Scan ([§ 1](#1-назначение)): `specs/architecture/services/README.md` + `specs/architecture/system/overview.md`. Если `specs/architecture/` пуст (новый проект) — LLM опирается только на Discussion и уточняет архитектурный контекст через AskUserQuestion.

**Маркер `[ТРЕБУЕТ УТОЧНЕНИЯ]`:** LLM ставит маркеры на оставшиеся неясности **в обоих случаях** — после проведения Clarify и при `--auto-clarify`. Пока в документе есть хотя бы один неразрешённый маркер, перевод в статус WAITING **невозможен**.

Воркфлоу создания (`create-impact.md`) завершается переводом в WAITING. Поэтому все маркеры должны быть разрешены до завершения создания — через AskUserQuestion, не откладывая на modify.

**Пропуск Clarify:** Пользователь может указать `--auto-clarify` — LLM пропускает Clarify, генерирует на основе своего понимания Discussion и ставит маркеры `[ТРЕБУЕТ УТОЧНЕНИЯ]` на все неясности. Шаг REVIEW остаётся обязательным.

**Примеры маркеров при --auto-clarify:**
- `[ТРЕБУЕТ УТОЧНЕНИЯ: Какие компоненты внутри auth-сервиса затронуты? Middleware, валидация, генерация токенов?]`
- `[ТРЕБУЕТ УТОЧНЕНИЯ: Нужна ли миграция данных для refresh_tokens? Создание новой таблицы или расширение существующей?]`
- `[ТРЕБУЕТ УТОЧНЕНИЯ: Зависимость gateway → auth — синхронная валидация через REST или кэширование ключей?]`

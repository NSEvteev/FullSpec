## Когда валидировать

После любого изменения в документе плана разработки:

- Создание → [create-plan-dev.md](./create-plan-dev.md)
- Изменение → [modify-plan-dev.md](./modify-plan-dev.md)
- Перед переводом DRAFT → WAITING — проверить: 0 маркеров
- При code review

## Шаги

### Шаг 0: Автоматическая валидация

```bash
python specs/.instructions/.scripts/validate-analysis-plan-dev.py {путь}
```

Скрипт проверяет все правила PD001-PD030. Если валидация пройдена — **готово**, шаги 1-12 не нужны.

**Если скрипт недоступен** — выполнить шаги 1-12 вручную.

### Шаг 1: Расположение и именование

**SSOT:** [standard-plan-dev.md § 2](./standard-plan-dev.md#2-расположение-и-именование)

- Файл расположен в `specs/analysis/NNNN-{topic}/plan-dev.md`
- Папка соответствует regex: `^\d{4}-.+$`
- `NNNN` — четырёхзначный номер (0001, 0002, ...)
- `{topic}` — kebab-case, латиница
- Имя файла — `plan-dev.md` (фиксированное)

### Шаг 2: Frontmatter

**SSOT:** [standard-plan-dev.md § 3](./standard-plan-dev.md#3-frontmatter)

**Проверить обязательные поля:**

| Поле | Проверка |
|------|----------|
| `description` | Присутствует, до 1024 символов |
| `standard` | = `specs/.instructions/analysis/plan-dev/standard-plan-dev.md` |
| `standard-version` | Присутствует, формат `vX.Y` |
| `index` | = `specs/analysis/README.md` |
| `parent` | Присутствует, путь к `plan-test.md` в той же папке |
| `status` | Валидный статус: DRAFT, WAITING, RUNNING, DONE, CONFLICT, ROLLING_BACK, REJECTED |
| `milestone` | Присутствует, формат `vX.Y.Z`, совпадает с parent Discussion |

**Проверить запрещённые поля:**
- `children` — **запрещено** (Plan Dev — терминальный объект)

**Проверить parent:**
- Файл `plan-test.md` существует по указанному пути
- Plan Tests в статусе WAITING (или выше)
- Milestone в Plan Dev совпадает с milestone в parent Discussion

### Шаг 3: Заголовок документа

**SSOT:** [standard-plan-dev.md § 2](./standard-plan-dev.md#2-расположение-и-именование)

- Формат: `# NNNN: {Тема} — Plan Dev`
- `NNNN` в заголовке совпадает с `NNNN` в имени папки
- `{Тема}` — до 80 символов, первое слово с заглавной буквы, без точки в конце
- Суффикс `— Plan Dev` обязателен

### Шаг 4: Обязательные разделы

**SSOT:** [standard-plan-dev.md § 5](./standard-plan-dev.md#5-разделы-документа)

| Раздел | Присутствие | Проверка |
|--------|------------|----------|
| `## Резюме` | Всегда (контент обязателен) | Присутствует и не пуст |
| `## {Сервис}` (per-service) | Минимум 1 | Хотя бы одна секция per-service |
| ├── `### Задачи` | Всегда | h3 внутри каждого per-service |
| └── `#### TASK-N: {Название}` | Минимум 1 на сервис | Хотя бы одна задача |
| `## Кросс-сервисные зависимости` | Всегда | Контент (таблица) или заглушка |
| `## Маппинг GitHub Issues` | Всегда | Описание процесса и таблица маппинга |

### Шаг 5: Per-service разделы и TASK-N

**SSOT:** [standard-plan-dev.md § 5](./standard-plan-dev.md#5-разделы-документа)

Для **каждой** задачи TASK-N:

1. **Заголовок** — формат `#### TASK-N: {Название}` (h4)

2. **5 обязательных полей:**

| Поле | Проверка |
|------|----------|
| Сложность | Формат `{1-10}/10`, целое число 1-10 |
| Приоритет | Одно из: `high`, `medium`, `low` (строчные) |
| Зависимости | `TASK-M` / `TASK-M, TASK-K` / `—` (em-dash) |
| TC | `TC-N` / `TC-N, TC-M` / `INFRA` — хотя бы одно значение |
| Источник | Формат `SVC-N § K` — ссылка на подсекцию Design |

3. **Значения полей:**
   - Сложность: целое число 1-10, формат `N/10`
   - Приоритет: ровно одно из трёх значений
   - Зависимости: если нет — символ `—` (em-dash, не дефис)
   - TC: хотя бы один TC-N из Plan Tests. Отсутствие TC-N — блокирующая ошибка (статус > DRAFT). Исключение: `TC: INFRA` для инфраструктурных задач
   - Источник: `SVC-N § K` — N = номер SVC-N секции, K = номер подсекции (1-9)

4. **Лимит INFRA:** Задачи с `TC: INFRA` — не более 20% от общего числа TASK-N

### Шаг 6: Подзадачи

**SSOT:** [standard-plan-dev.md § 5](./standard-plan-dev.md#5-разделы-документа) → Правила подзадач

Если блок `Подзадачи:` присутствует:

| Проверка | Описание |
|----------|----------|
| Точечная нотация | Формат `N.M` (N = номер задачи, M = номер подзадачи) |
| Чек-лист формат | `- [ ]` (при создании), `- [x]` (при выполнении) |
| Минимум 2 подзадачи | Если блок указан — минимум 2 пункта |
| Зависимости | `(deps: N.M)` — только на сиблингов внутри одной задачи |
| Нет кросс-задачных deps | Кросс-задачные зависимости — только в поле `Зависимости` TASK-N |

### Шаг 7: Кросс-сервисные зависимости

**SSOT:** [standard-plan-dev.md § 5 → Кросс-сервисные зависимости](./standard-plan-dev.md#кросс-сервисные-зависимости)

Если секция содержит таблицу (не заглушку):
- 3 колонки: Зависимость, Блокирует, Причина
- Только кросс-сервисные зависимости (между задачами разных сервисов)
- Каждая запись совпадает с полем `Зависимости` соответствующей TASK-N
- Нет внутрисервисных зависимостей в таблице

**Заглушка допустима:** `*Кросс-сервисных зависимостей нет.*`

### Шаг 8: Маппинг GitHub Issues

**SSOT:** [standard-plan-dev.md § 5 → Маппинг GitHub Issues](./standard-plan-dev.md#маппинг-github-issues)

- Секция присутствует
- Содержит описание процесса маппинга
- Содержит таблицу маппинга элементов (6 строк: TASK-N, Подзадачи, Приоритет, Зависимости, TC, Milestone)
- Конкретные номера Issues **не указаны** в плане

### Шаг 9: Нумерация и порядок TASK-N

**SSOT:** [standard-plan-dev.md § 5](./standard-plan-dev.md#5-разделы-документа)

- `TASK-N`: сквозная нумерация по всему документу (не по сервису)
- Порядок: per-service (в порядке Design: Основной → Вторичный → Новый)
- Пропуски допустимы, перенумерация запрещена
- Нет дублирующихся номеров
- Задачи с зависимостями расположены **после** задач, от которых зависят
- Нет циклических зависимостей между TASK-N

### Шаг 10: Зона ответственности

**SSOT:** [standard-plan-dev.md § 1](./standard-plan-dev.md#1-назначение)

Документ **НЕ должен** содержать:
- Тестовые сценарии (→ Plan Tests)
- Распределение ответственностей, контракты API (→ Design)
- Бизнес-обоснование и требования (→ Discussion)
- Конкретный код реализации (→ разработка)

### Шаг 11: Маркеры и статус

**SSOT:** [standard-plan-dev.md § 6](./standard-plan-dev.md#6-clarify), [standard-analysis.md § 8](../standard-analysis.md#8-clarify-и-блокирующие-правила)

Если `status` > DRAFT (WAITING, RUNNING, DONE, CONFLICT, ROLLING_BACK, REJECTED):
- Нет неразрешённых `[ТРЕБУЕТ УТОЧНЕНИЯ]` маркеров

### Шаг 12: Регистрация в README

**SSOT:** [standard-plan-dev.md § 2](./standard-plan-dev.md#2-расположение-и-именование)

- Запись существует в `specs/analysis/README.md`
- Статус в README совпадает со статусом в frontmatter документа

## Чек-лист

### Именование
- [ ] Файл в `specs/analysis/NNNN-{topic}/plan-dev.md`
- [ ] Папка: `NNNN-{topic}` (regex `^\d{4}-.+$`)
- [ ] Заголовок: `# NNNN: {Тема} — Plan Dev`
- [ ] NNNN в папке = NNNN в заголовке

### Frontmatter
- [ ] `description` — до 1024 символов
- [ ] `standard` = `specs/.instructions/analysis/plan-dev/standard-plan-dev.md`
- [ ] `standard-version` — формат `vX.Y`
- [ ] `index` = `specs/analysis/README.md`
- [ ] `parent` — путь к Plan Tests документу (обязателен)
- [ ] Нет поля `children` (терминальный объект)
- [ ] `status` — валидный статус
- [ ] `milestone` — совпадает с parent Discussion

### Структура
- [ ] Секция «Резюме» заполнена контентом
- [ ] Минимум 1 per-service раздел (h2)
- [ ] Каждый per-service раздел: Задачи (h3) с TASK-N (h4)
- [ ] Минимум 1 TASK-N в каждом per-service разделе
- [ ] Секция «Кросс-сервисные зависимости» — таблица или заглушка
- [ ] Секция «Маппинг GitHub Issues» — описание процесса и таблица маппинга

### Формат TASK-N
- [ ] Заголовок: `#### TASK-N: {Название}` (h4)
- [ ] Все 5 полей заполнены (Сложность, Приоритет, Зависимости, TC, Источник)
- [ ] Сложность: целое число 1-10, формат `N/10`
- [ ] Приоритет: одно из `high` / `medium` / `low`
- [ ] Зависимости: `TASK-M` или `—` (em-dash)
- [ ] TC: хотя бы один TC-N из Plan Tests или `INFRA`
- [ ] Источник: `SVC-N § K` из Design

### Подзадачи
- [ ] Точечная нотация: `N.M`
- [ ] Чек-лист формат: `- [ ]`
- [ ] Зависимости на сиблингов: `(deps: N.M)` — только внутри одной задачи
- [ ] Минимум 2 подзадачи, если блок указан
- [ ] Нет кросс-задачных зависимостей в подзадачах

### Нумерация и покрытие
- [ ] TASK-N — сквозная по всему документу (не по сервису)
- [ ] Пропуски допустимы, перенумерация запрещена
- [ ] Нет дублирующихся номеров
- [ ] Нет циклических зависимостей
- [ ] Задачи с зависимостями расположены после задач, от которых зависят
- [ ] Задачи с `TC: INFRA` — не более 20% от общего числа TASK-N

### Per-service разделы
- [ ] Каждый SVC-N из Design имеет per-service раздел
- [ ] Порядок разделов совпадает с Design (Основной → Вторичный → Новый)
- [ ] Каждый TC-N из Plan Tests покрыт хотя бы одной задачей

### Зона ответственности
- [ ] Нет тестовых сценариев (→ Plan Tests)
- [ ] Нет распределения ответственностей (→ Design)
- [ ] Нет бизнес-обоснований (→ Discussion)
- [ ] Нет конкретного кода реализации (→ разработка)

### Маркеры (если статус > DRAFT)
- [ ] Нет `[ТРЕБУЕТ УТОЧНЕНИЯ]`

### README
- [ ] Запись в `specs/analysis/README.md`
- [ ] Статус в README = статус в frontmatter

## Типичные ошибки

| Ошибка | Код | Причина | Решение |
|--------|-----|---------|---------|
| Неверное расположение файла | PD001 | Файл не в `specs/analysis/NNNN-{topic}/plan-dev.md` | Переместить файл |
| Отсутствует `description` | PD002 | Пустой или отсутствующий frontmatter | Добавить description |
| Неверный `standard` | PD003 | Путь не указывает на standard-plan-dev.md | Исправить путь |
| Невалидный `status` | PD004 | Статус не из допустимого списка | Использовать валидный статус |
| Отсутствует `parent` | PD005 | Plan Dev обязан иметь parent Plan Tests | Добавить поле parent |
| Parent Plan Tests не существует | PD006 | Файл по пути parent не найден | Исправить путь или создать Plan Tests |
| Milestone не совпадает с parent | PD007 | Milestone Plan Dev ≠ milestone Discussion | Синхронизировать milestone |
| Запрещённое поле `children` | PD008 | Plan Dev — терминальный, children запрещён | Удалить поле children |
| Отсутствует «Резюме» | PD009 | Обязательный раздел пропущен | Добавить раздел |
| Нет per-service раздела | PD010 | Минимум 1 per-service раздел | Добавить per-service раздел |
| Нет «Задачи» в per-service | PD011 | Подсекция h3 пропущена | Добавить подсекцию «Задачи» |
| Нет TASK-N в per-service | PD012 | Минимум 1 задача на сервис | Добавить задачу |
| TASK-N: невалидная Сложность | PD013 | Формат не `{1-10}/10` | Исправить формат |
| TASK-N: невалидный Приоритет | PD014 | Не high/medium/low | Использовать валидное значение |
| TASK-N: отсутствует поле | PD015 | Не все 5 полей заполнены | Заполнить все поля |
| TASK-N: нет TC-N | PD016 | Нет ссылки на тест-сценарий (и не INFRA) | Добавить TC-N или INFRA |
| TASK-N: невалидный Источник | PD017 | Формат не `SVC-N § K` | Исправить формат |
| Дублирование TASK-N | PD018 | Два TASK с одинаковым номером | Исправить нумерацию |
| Подзадача: 1 пункт в блоке | PD019 | Минимум 2 подзадачи, если блок указан | Добавить подзадачу или удалить блок |
| Подзадача: неверная нотация | PD020 | Формат не `N.M` | Исправить нотацию |
| Подзадача: кросс-задачная dep | PD021 | `(deps: ...)` ссылается на другую задачу | Перенести зависимость в поле Зависимости TASK-N |
| Нет «Кросс-сервисные зависимости» | PD022 | Обязательный раздел пропущен | Добавить раздел (таблицу или заглушку) |
| Нет «Маппинг GitHub Issues» | PD023 | Обязательный раздел пропущен | Добавить раздел |
| NNNN не совпадает | PD024 | Номер в папке ≠ номер в заголовке | Исправить заголовок или имя папки |
| Description слишком длинное | PD025 | Description > 1024 символов | Сократить description |
| Циклическая зависимость | PD026 | TASK-A → TASK-B → TASK-A | Устранить цикл |
| Задача перед зависимостью | PD027 | TASK-N с deps расположена до задачи, от которой зависит | Переупорядочить задачи |
| INFRA превышает 20% | PD028 | Слишком много задач с TC: INFRA | Добавить TC-N или убрать лишние INFRA |
| Маркер при статусе > DRAFT | PD029 | Неразрешённый `[ТРЕБУЕТ УТОЧНЕНИЯ]` | Разрешить маркер или вернуть в DRAFT |
| Кросс-зависимость не в таблице | PD030 | Кросс-сервисная зависимость не отражена в общей таблице | Добавить запись в таблицу |

## 1. Назначение

**Зона ответственности:** ЗАЧЕМ и ЧТО ([Навигатор SDD § 3](../standard-specs-workflow.md#3-зоны-ответственности)).

Дискуссия отвечает на вопросы: **Зачем это нужно? Какие требования?**

**Содержит:**
- Проблему, контекст (текущее состояние и что не работает)
- Требования (Given/When/Then)
- User Stories
- Варианты и предложения
- Критерии успеха
- Milestone

**НЕ содержит:**
- Технические детали реализации (→ ADR)
- Распределение по сервисам (→ Impact)
- Архитектурные решения (→ ADR)
- Контракты API (→ Design)
- Тестовые сценарии (→ План тестов)
- Задачи на реализацию (→ План разработки)

**Граничные случаи (Discussion vs Impact):**
- Допустимо: "Использовать OAuth2 вместо session-based" — упоминание технологии без привязки к сервисам
- Допустимо: "Рассмотреть варианты: OAuth2, SAML, JWT" — сравнение вариантов
- Недопустимо: "Реализовать OAuth2 в auth-сервисе" — привязка к сервису (→ Impact)
- Недопустимо: "auth-сервис отвечает за токены, gateway — за rate limiting" — распределение ответственностей (→ Impact)

Пользователь **не определяет** зоны ответственности по сервисам в Discussion. Это задача Impact-анализа.

**Когда создавать:**
- Новая функциональность (фича, интеграция, миграция)
- Группа связанных багфиксов ([решение #46](../standard-specs-workflow.md#8-решения))
- Cross-cutting concern (NFR: производительность, безопасность — [решение #21](../standard-specs-workflow.md#8-решения))

**Связи:** Discussion не имеет родителя. Дочерний объект — Impact (1:1). ([Справочник SDD § 1](../standard-specs-reference.md#1-связи-и-frontmatter))

**После DONE:** Discussion становится архивной записью реализованного решения. Изменения запрещены (только исправление опечаток). При появлении новых требований — создать новую Discussion со ссылкой на DONE-дискуссию в секции "Проблема / Контекст".

## 2. Расположение и именование

**Расположение:** `specs/discussion/`

**Формат имени:** `disc-NNNN-topic.md`

| Элемент | Правило |
|---------|---------|
| `disc` | Фиксированный префикс |
| `NNNN` | Четырёхзначный автоинкремент (0001, 0002, ...) |
| `topic` | Kebab-case, латиница, описательное |

**Regex:** `^disc-(\d{4})-(.+)\.md$`

**Нумерация:** Независимая в папке `specs/discussion/`. Следующий номер = максимальный существующий + 1.

**Формат README-таблицы** (`specs/discussion/README.md`):

| # | Документ | Статус | Impact | Milestone | Описание |
|---|----------|--------|--------|-----------|----------|
| 0001 | disc-0001-oauth2-authorization.md | RUNNING | impact-0001 | v1.2.0 | OAuth2 авторизация |

## 3. Frontmatter

**SSOT frontmatter:** [standard-frontmatter.md](/.structure/.instructions/standard-frontmatter.md) ([§ 1 — базовые поля](/.structure/.instructions/standard-frontmatter.md#1-обязательные-поля) + [§ 4 — поля specs](/.structure/.instructions/standard-frontmatter.md#4-дополнительные-поля-для-спецификаций-specs))

| Поле | Значение для Discussion | Обязательность |
|------|------------------------|----------------|
| `description` | Описание дискуссии (до 1024 символов) | Обязательно |
| `standard` | `specs/.instructions/discussion/standard-discussion.md` | Обязательно |
| `standard-version` | `v1.0` | Обязательно |
| `index` | `specs/discussion/README.md` | Обязательно |
| `parent` | **Нет** (Discussion — корневой объект цепочки) | Не указывается |
| `children` | Путь к Impact-документу (1:1) | Обязательно (после создания Impact) |
| `status` | Текущий статус ([Справочник SDD § 2](../standard-specs-reference.md#2-статусы)) | Обязательно |
| `milestone` | Целевой Milestone (определяется при Clarify) | Обязательно |

```yaml
---
description: OAuth2 авторизация — переход с session-based на OAuth2 для масштабирования до 10k RPS.
standard: specs/.instructions/discussion/standard-discussion.md
standard-version: v1.0
index: specs/discussion/README.md
children:
  - impact/impact-0001-oauth2-authorization.md
status: DRAFT
milestone: v1.2.0
---
```

**Milestone:** Целевая версия релиза (строка, например `v1.2.0`). Определяется при Clarify на уровне Discussion и сохраняется в frontmatter. Все дочерние документы цепочки наследуют milestone от Discussion. Один Milestone (например, `v1.2.0`) может быть указан в нескольких Discussions — это означает, что эти дискуссии реализуются в одном релизе.

**Параллельные дискуссии в одном Milestone:** [Навигатор SDD § 6](../standard-specs-workflow.md#6-параллельные-дискуссии) — механизм Planned Changes для обнаружения конфликтов между дискуссиями.

## 4. Переходы статусов

Happy path — нормальный поток жизненного цикла Discussion ([Справочник SDD § 3](../standard-specs-reference.md#3-последовательность-статусов)).

**Зона ответственности Discussion:** только переход DRAFT → WAITING (создание и модификация документа). Все последующие переходы управляются на уровне цепочки — дочерними документами, каскадами или обратной связью от кода.

| Текущий статус | Условие перехода | Следующий статус | Кто управляет |
|---|---|---|---|
| **DRAFT** | Пользователь одобрил документ. Нет неразрешённых `[ТРЕБУЕТ УТОЧНЕНИЯ]`. Нет Dependency Barrier | **WAITING** | Discussion (этот документ) |
| **WAITING** | Все документы цепочки (Discussion → ... → Plan Dev) в WAITING. Пользователь подтвердил запуск | **RUNNING** | Цепочка ([§ 3.2](../standard-specs-reference.md#32-waiting--running)) |
| **RUNNING** | Обратная связь от кода затронула уровень Discussion ([§ 3.3](../standard-specs-reference.md#33-running--conflict)) | **CONFLICT** | Цепочка (tree-level) |
| **CONFLICT** | LLM исправил документ, пользователь одобрил исправление ([§ 3.4](../standard-specs-reference.md#34-conflict--waiting)) | **WAITING** | Цепочка (top-down) |
| **RUNNING** | Impact (дочерний, 1:1) → DONE. Каскад снизу вверх ([§ 3.5](../standard-specs-reference.md#35-running--done)) | **DONE** | Цепочка (bottom-up) |

Откат и отклонение (ROLLING_BACK, REJECTED): [Справочник SDD § 3.6–3.7](../standard-specs-reference.md#36--rolling_back).

## 5. Разделы документа

Дискуссия — **структурированный документ с фиксированными разделами** (см. таблицу ниже). Порядок разделов неизменен. "Гибкость" означает: предложения могут менять фичи и user stories внутри документа до достижения консенсуса — итеративное уточнение контента, не структуры.

| Раздел | Обязательность | Когда добавлять | Когда НЕ добавлять |
|--------|---------------|-----------------|-------------------|
| **Проблема / Контекст** | Обязательный | Всегда | — |
| **Фичи** | Опциональный | Обсуждается конкретная функциональность | Багфиксы, NFR без новых фич |
| **User Stories** | Опциональный | Фича ориентирована на конечного пользователя | Технические рефакторинги, инфраструктурные изменения |
| **Требования** | Опциональный | Нужна формализация Given/When/Then | Простые изменения с очевидными критериями |
| **Предложения** | Опциональный | Есть альтернативные решения или изменения к фичам | Единственный очевидный путь решения |
| **Критерии успеха** | Обязательный | Всегда | — |

**Правила нумерации (F-N, US-N, REQ-N, PROP-N):** Последовательная автоинкрементная. При добавлении — следующий номер = максимальный существующий + 1. При удалении — **не перенумеровывать** (пропуски допустимы, сохраняется ссылочная целостность).

**Механика итеративного уточнения:** Предложения (PROP-N) описывают изменения в секции "Предложения" с указанием затронутых элементов (→ Влияет на: F-1, US-2). После одобрения пользователем LLM вносит изменения в секции "Фичи" и "User Stories" на месте (обновляется baseline). Старые версии не сохраняются в документе — история через git.

### Проблема и контекст

Описание текущей ситуации и причин для изменений. Единственный обязательный раздел помимо критериев успеха.

**Формат:** свободный текст. Рекомендуется описать:
- Текущее состояние (AS IS): что сейчас работает и как, какие метрики
- Бизнес-контекст или техническое ограничение
- Масштаб проблемы (количественные метрики если есть)

Если описание AS IS содержит привязку к конкретным сервисам — это будет проверено и формализовано на этапе Impact.

```markdown
## Проблема / Контекст

**Текущее состояние (AS IS):**
Авторизация реализована через session-based подход с хранением в Redis.
Время авторизации: p99 = 800ms при 5k RPS.

**Проблема:**
При масштабировании до 10k RPS Redis становится узким местом.
Кроме того, session не подходит для mobile-клиентов и third-party интеграций.
```

### Фичи

Конкретная функциональность, которую нужно реализовать. Нумерация: `F-N`.

```markdown
## Фичи

### F-1: OAuth2 авторизация для API
Авторизация через access + refresh токены вместо session.

### F-2: Управление ролями (RBAC)
Назначение ролей пользователям, проверка разрешений на уровне endpoint.

### F-3: Third-party интеграции
OAuth2 client credentials flow для внешних сервисов.
```

### User Stories

Описание функциональности с точки зрения пользователя. Нумерация: `US-N`. Формат: **Как {роль}, я хочу {действие}, чтобы {цель}**.

```markdown
## User Stories

### US-1: Авторизация через API
Как **API-клиент**, я хочу **получить OAuth2 токен через POST /auth/token**,
чтобы **авторизовывать запросы без хранения session на сервере**.

### US-2: Управление ролями
Как **администратор**, я хочу **назначать роли пользователям через UI**,
чтобы **контролировать доступ к функциям системы**.
```

### Требования

Формализованные требования в формате Given/When/Then. Нумерация: `REQ-N`. Требования могут ссылаться на фичи и user stories для трассируемости. На следующих этапах (Impact, Design, ADR) одно требование может разделиться на несколько задач по разным сервисам.

```markdown
## Требования

### REQ-1: Успешная авторизация (F-1, US-1)
- **GIVEN** пользователь с валидными credentials
- **WHEN** `POST /auth/token` с `grant_type=password`
- **THEN** ответ `200 OK` с `access_token` и `refresh_token`

### REQ-2: Refresh токена (F-1)
- **GIVEN** валидный `refresh_token`
- **WHEN** `POST /auth/token` с `grant_type=refresh_token`
- **THEN** новый `access_token`, прежний `refresh_token` инвалидируется

### REQ-3: Rate limiting для неавторизованных (F-1)
- **GIVEN** неавторизованный пользователь
- **WHEN** более 100 запросов за минуту
- **THEN** ответ `429 Too Many Requests`
```

### Предложения

Варианты решений, изменения к фичам и user stories. Нумерация: `PROP-N`. Предложения могут **менять** ранее описанные фичи и user stories — итеративное уточнение.

```markdown
## Предложения

### PROP-1: ES256 вместо RS256 для JWT
RS256 медленнее на целевом железе. ES256 даёт сопоставимую безопасность
при меньшем размере подписи и более быстрой верификации.
→ Влияет на: F-1 (алгоритм подписи)

### PROP-2: Убрать F-3 из текущего Milestone
Third-party интеграции требуют отдельного OAuth2-сервера.
Предлагаю вынести в v1.3.0.
→ Влияет на: F-3 (перенос), критерии успеха
```

### Критерии успеха

Измеримые критерии завершения. Обязательный раздел.

```markdown
## Критерии успеха

- Время авторизации < 100ms (p99) при 10k RPS
- Zero-downtime миграция с session-based на OAuth2
- Все существующие API-клиенты продолжают работать через compatibility layer
- 100% покрытие acceptance-тестами (REQ-1 — REQ-N)
```

## 6. Clarify

**SSOT:** [Справочник SDD § 5](../standard-specs-reference.md#5-clarify-и-блокирующие-правила) — полные правила Clarify, маркер `[ТРЕБУЕТ УТОЧНЕНИЯ]`, Dependency Barrier.

**Паттерн объекта:** [Навигатор SDD § 4 "Общий паттерн объекта"](../standard-specs-workflow.md#общий-паттерн-объекта) — цикл CLARIFY → GENERATE → VALIDATE → REVIEW.

На уровне Discussion LLM уточняет через AskUserQuestion:

| Что уточняется | Пример вопроса |
|----------------|---------------|
| Проблема и scope | "Какую конкретно проблему решаем? Это только авторизация или также аутентификация?" |
| Требования | "Какой SLA по времени авторизации? Есть ли требования к offline-режиму?" |
| Целевая аудитория | "Кто пользователи? Только внутренние сервисы или также external API consumers?" |
| Критерии успеха | "Как измерим успешность? Какие метрики?" |
| Milestone | "В каком Milestone планируем? Есть ли дедлайн?" |
| Границы (scope) | "Что явно НЕ входит в эту дискуссию?" |

**Маркер `[ТРЕБУЕТ УТОЧНЕНИЯ]`:** LLM ставит маркеры на оставшиеся неясности **в обоих случаях** — после проведения Clarify и при `--auto-clarify`. Пока в документе есть хотя бы один неразрешённый маркер, перевод в статус WAITING **невозможен**.

**Пропуск Clarify:** Пользователь может указать `--auto-clarify` — LLM пропускает Clarify, генерирует на основе своего понимания и ставит маркеры `[ТРЕБУЕТ УТОЧНЕНИЯ]` на все неясности. Шаг REVIEW остаётся обязательным.

## 8. Чек-лист качества

### Frontmatter
- [ ] `description` — до 1024 символов
- [ ] `standard` указывает на этот документ
- [ ] `children` — путь к Impact (или пустой список если Impact ещё не создан)
- [ ] `status` — валидный статус ([Справочник SDD § 2](../standard-specs-reference.md#2-статусы))
- [ ] `milestone` — указан целевой Milestone

### Содержание
- [ ] Раздел "Проблема / Контекст" заполнен (обязательный)
- [ ] Раздел "Критерии успеха" заполнен (обязательный)
- [ ] Фичи пронумерованы (F-N) если раздел присутствует
- [ ] User Stories пронумерованы (US-N) если раздел присутствует
- [ ] Требования в формате Given/When/Then и пронумерованы (REQ-N) если раздел присутствует
- [ ] Требования ссылаются на фичи/user stories (F-N, US-N в заголовке) если раздел присутствует
- [ ] Предложения пронумерованы (PROP-N) и указывают "Влияет на" если раздел присутствует
- [ ] Критерии успеха измеримы (числа, метрики, покрытие)
- [ ] Нумерация не перезаписана (пропуски допустимы)

### Зона ответственности
- [ ] Нет технических деталей реализации (→ ADR)
- [ ] Нет распределения по сервисам (→ Impact)
- [ ] Нет контрактов API (→ Design)
- [ ] Нет тестовых сценариев (→ План тестов)
- [ ] Нет задач на реализацию (→ План разработки)

### Маркеры
- [ ] Нет неразрешённых `[ТРЕБУЕТ УТОЧНЕНИЯ]` (если статус > DRAFT)
- [ ] Нет Dependency Barrier (если статус > DRAFT)

### Именование
- [ ] Имя файла соответствует формату `disc-NNNN-topic.md`
- [ ] Заголовок документа: `# disc-NNNN: {Тема}`
- [ ] Запись добавлена в `specs/discussion/README.md`

## 9. Примеры по типам

### Пример 1: Новая функциональность

```markdown
# disc-0001: OAuth2 авторизация

## Проблема / Контекст

**Текущее состояние (AS IS):**
Session-based авторизация, Redis для хранения сессий. p99 = 800ms при 5k RPS.

**Проблема:**
```

### Пример 2: Группа багфиксов

```markdown
# disc-0005: Исправление race conditions в кэше

## Проблема / Контекст

**Текущее состояние (AS IS):**
Кэш реализован через in-memory store с TTL. Несколько воркеров обновляют кэш параллельно.

**Проблема:**
3 бага связаны с race conditions:
- BUG-101: Stale data после обновления
- BUG-115: Двойная запись при concurrent update
- BUG-122: TTL сбрасывается при чтении

## Критерии успеха

- BUG-101, BUG-115, BUG-122 закрыты
- Нет race conditions при 100 concurrent writers
- p99 чтения кэша < 5ms
```

### Пример 3: Cross-cutting concern (NFR)

```markdown
# disc-0008: Снижение latency API до < 200ms

## Проблема / Контекст

**Текущее состояние (AS IS):**
Среднее время ответа API: p50 = 150ms, p99 = 1200ms. SLA: p99 < 500ms.

**Проблема:**
p99 превышает SLA в 2.4 раза. Основные причины: N+1 запросы к БД,
отсутствие кэширования, синхронные вызовы между сервисами.

## Требования

### REQ-1: API latency
- **GIVEN** авторизованный пользователь
- **WHEN** любой GET-запрос к API
- **THEN** время ответа p99 < 200ms

### REQ-2: SLA compliance
- **GIVEN** нагрузка 10k RPS
- **WHEN** мониторинг за 24 часа
- **THEN** 99.9% запросов < 500ms

## Критерии успеха

- p99 < 200ms (текущий: 1200ms)
- SLA compliance 99.9% (текущий: 97.2%)
- Без деградации функциональности
```

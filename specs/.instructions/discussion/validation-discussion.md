## Когда валидировать

После любого изменения в документе дискуссии:

- Создание → [create-discussion.md](./create-discussion.md)
- Изменение → [modify-discussion.md](./modify-discussion.md)
- Перед переводом DRAFT → WAITING
- При code review

## Шаги

### Шаг 0: Автоматическая валидация

```bash
python specs/.instructions/.scripts/validate-discussion.py {путь}
```

Скрипт проверяет все правила D001-D023. Если валидация пройдена — **готово**, шаги 1-9 не нужны.

**Если скрипт недоступен** — выполнить шаги 1-9 вручную.

### Шаг 1: Именование файла

**SSOT:** [standard-discussion.md § 2](./standard-discussion.md#2-расположение-и-именование)

- Файл расположен в `specs/discussion/`
- Имя соответствует regex: `^disc-(\d{4})-(.+)\.md$`
- `NNNN` — четырёхзначный номер (0001, 0002, ...)
- `topic` — kebab-case, латиница

### Шаг 2: Frontmatter

**SSOT:** [standard-discussion.md § 3](./standard-discussion.md#3-frontmatter)

**Проверить обязательные поля:**

| Поле | Проверка |
|------|----------|
| `description` | Присутствует, до 1024 символов |
| `standard` | = `specs/.instructions/discussion/standard-discussion.md` |
| `standard-version` | Присутствует, формат `vX.Y` |
| `index` | = `specs/discussion/README.md` |
| `status` | Валидный статус: DRAFT, WAITING, RUNNING, DONE, CONFLICT, ROLLING_BACK, REJECTED |
| `milestone` | Присутствует, формат `vX.Y.Z` |
| `children` | Список путей или пустой список `[]` |

**Проверить запрещённые поля:**

| Поле | Проверка |
|------|----------|
| `parent` | НЕ должен присутствовать (Discussion — корневой объект) |

### Шаг 3: Заголовок документа

**SSOT:** [standard-discussion.md § 2](./standard-discussion.md#2-расположение-и-именование)

- Формат: `# disc-NNNN: {Тема}`
- `NNNN` в заголовке совпадает с `NNNN` в имени файла

### Шаг 4: Обязательные разделы

**SSOT:** [standard-discussion.md § 5](./standard-discussion.md#5-разделы-документа)

| Раздел | Обязательность | Проверка |
|--------|---------------|----------|
| `## Проблема / Контекст` | Обязательный | Присутствует и не пуст |
| `## Критерии успеха` | Обязательный | Присутствует и не пуст |
| `## Фичи` | Опциональный | Если присутствует — содержит хотя бы один `### F-N:` |
| `## User Stories` | Опциональный | Если присутствует — содержит хотя бы один `### US-N:` |
| `## Требования` | Опциональный | Если присутствует — содержит хотя бы один `### REQ-N:` |
| `## Предложения` | Опциональный | Если присутствует — содержит хотя бы один `### PROP-N:` |

### Шаг 5: Нумерация элементов

**SSOT:** [standard-discussion.md § 5](./standard-discussion.md#5-разделы-документа)

- `F-N`: последовательная (пропуски допустимы, дублирование — нет)
- `US-N`: последовательная (пропуски допустимы, дублирование — нет)
- `REQ-N`: последовательная (пропуски допустимы, дублирование — нет)
- `PROP-N`: последовательная (пропуски допустимы, дублирование — нет)

**Проверить:** нет дублирующихся номеров в пределах каждого типа.

### Шаг 6: Формат требований

**SSOT:** [standard-discussion.md § 5 → Требования](./standard-discussion.md#требования)

Если раздел "Требования" присутствует:
- Каждое `REQ-N` содержит строки `**GIVEN**`, `**WHEN**`, `**THEN**`
- Заголовок `REQ-N` ссылается на фичи/user stories: `(F-N, US-N)` — рекомендация, не блокирующее

### Шаг 7: Зона ответственности

**SSOT:** [standard-discussion.md § 1](./standard-discussion.md#1-назначение)

Документ **НЕ должен** содержать:
- Привязку функциональности к конкретным сервисам ("реализовать в auth-сервисе")
- Распределение ответственностей между сервисами ("auth отвечает за..., gateway за...")
- Контракты API (эндпоинты, request/response схемы)
- Тестовые сценарии (кроме критериев успеха)
- Задачи на реализацию (Issues, таски)

**Допустимо:** упоминание технологий без привязки к сервисам ("использовать OAuth2", "рассмотреть варианты: OAuth2, SAML").

### Шаг 8: Маркеры и статус

**SSOT:** [standard-discussion.md § 4](./standard-discussion.md#4-переходы-статусов), [standard-discussion.md § 6](./standard-discussion.md#6-clarify)

Если `status` > DRAFT (WAITING, RUNNING, DONE, CONFLICT, ROLLING_BACK, REJECTED):
- Нет неразрешённых `[ТРЕБУЕТ УТОЧНЕНИЯ]` маркеров
- Нет `⛔ DEPENDENCY BARRIER`

### Шаг 9: Регистрация в README

**SSOT:** [standard-discussion.md § 2](./standard-discussion.md#2-расположение-и-именование)

- Запись существует в `specs/discussion/README.md`
- Колонки: `#`, `Документ`, `Статус`, `Impact`, `Milestone`, `Описание`
- Статус в README совпадает со статусом в frontmatter документа

## Чек-лист

### Именование
- [ ] Файл в `specs/discussion/`
- [ ] Имя: `disc-NNNN-topic.md`
- [ ] Заголовок: `# disc-NNNN: {Тема}`
- [ ] NNNN в имени = NNNN в заголовке

### Frontmatter
- [ ] `description` — до 1024 символов
- [ ] `standard` = `specs/.instructions/discussion/standard-discussion.md`
- [ ] `standard-version` — формат `vX.Y`
- [ ] `index` = `specs/discussion/README.md`
- [ ] `status` — валидный статус
- [ ] `milestone` — указан
- [ ] `children` — список или `[]`
- [ ] `parent` — отсутствует

### Содержание
- [ ] "Проблема / Контекст" — присутствует и не пуст
- [ ] "Критерии успеха" — присутствует и не пуст
- [ ] Критерии успеха измеримы (числа, метрики)
- [ ] Нумерация F-N, US-N, REQ-N, PROP-N без дублей
- [ ] REQ-N содержат Given/When/Then (если раздел есть)
- [ ] PROP-N содержат "Влияет на" (если раздел есть)

### Зона ответственности
- [ ] Нет привязки к конкретным сервисам
- [ ] Нет распределения ответственностей между сервисами
- [ ] Нет контрактов API
- [ ] Нет тестовых сценариев
- [ ] Нет задач на реализацию

### Маркеры (если статус > DRAFT)
- [ ] Нет `[ТРЕБУЕТ УТОЧНЕНИЯ]`
- [ ] Нет `⛔ DEPENDENCY BARRIER`

### README
- [ ] Запись в `specs/discussion/README.md`
- [ ] Статус в README = статус в frontmatter

## Типичные ошибки

| Ошибка | Код | Причина | Решение |
|--------|-----|---------|---------|
| Неверный формат имени файла | D001 | Имя не соответствует `disc-NNNN-topic.md` | Переименовать файл |
| Отсутствует `description` | D002 | Пустой или отсутствующий frontmatter | Добавить description |
| Неверный `standard` | D003 | Путь не указывает на standard-discussion.md | Исправить путь |
| Невалидный `status` | D004 | Статус не из допустимого списка | Использовать валидный статус |
| Присутствует `parent` | D005 | Discussion — корневой объект, parent запрещён | Удалить поле parent |
| Отсутствует "Проблема / Контекст" | D006 | Обязательный раздел пропущен | Добавить раздел |
| Отсутствует "Критерии успеха" | D007 | Обязательный раздел пропущен | Добавить раздел |
| Дублирование номера F-N | D008 | Два элемента с одинаковым номером | Исправить нумерацию |
| Дублирование номера US-N | D009 | Два элемента с одинаковым номером | Исправить нумерацию |
| Дублирование номера REQ-N | D010 | Два элемента с одинаковым номером | Исправить нумерацию |
| Дублирование номера PROP-N | D011 | Два элемента с одинаковым номером | Исправить нумерацию |
| REQ без Given/When/Then | D012 | Требование без формализованного формата | Добавить GIVEN/WHEN/THEN |
| Маркер при статусе > DRAFT | D013 | Неразрешённый `[ТРЕБУЕТ УТОЧНЕНИЯ]` | Разрешить маркер или вернуть в DRAFT |
| Dependency Barrier при статусе > DRAFT | D014 | Неразрешённый барьер | Разрешить барьер или вернуть в DRAFT |
| Привязка к сервису | D015 | Распределение по сервисам в Discussion | Перенести в Impact |
| Нет записи в README | D016 | Документ не зарегистрирован | Добавить запись в README |
| Рассинхрон статуса | D017 | Статус в README ≠ статус в frontmatter | Синхронизировать |
| NNNN не совпадает | D018 | Номер в имени файла ≠ номер в заголовке | Исправить заголовок или имя файла |
| Отсутствует `milestone` | D019 | Milestone не указан в frontmatter | Добавить milestone |
| Отсутствует или неверный `standard-version` | D020 | standard-version не указан или не в формате `vX.Y` | Добавить/исправить standard-version |
| Неверный `index` | D021 | index не указывает на `specs/discussion/README.md` | Исправить index |
| Опциональная секция без элементов | D022 | Секция (Фичи, User Stories и т.д.) есть, но пуста | Добавить элементы или убрать секцию |
| Description слишком длинное | D023 | Description > 1024 символов | Сократить description |

## spec-create

### Успешное создание

```
✅ Документ создан

Файл: /specs/{path}/{NNN}-{topic}.md
Тип: {type}
Статус: DRAFT
Родитель: {parent path}

Индекс обновлён: /specs/{path}/README.md
Backlink добавлен: {parent path}

Следующий шаг:
- Заполнить документ
- Перевести в REVIEW: /spec-status {short-path} review
```

### Dry-run режим

```
📋 Предварительный просмотр (--dry-run)

Будет создано:
- /specs/{path}/{NNN}-{topic}.md

Будет обновлено:
- /specs/{path}/README.md (добавлена строка)
- {parent-path} (backlink)

ℹ️ Изменения НЕ применены (--dry-run)
```

### Создание нового сервиса

```
⚠️ Сервис "{service}" не существует.

Создать структуру нового сервиса?
- /src/{service}/
- /tests/{service}/
- /specs/services/{service}/
- /doc/src/{service}/

[Y/n]
```

## spec-status

### Успешное изменение

```
✅ Статус изменён

Документ: /specs/{path}/{file}.md
Переход: {old-status} → {new-status}

Каскадные действия: {нет | список}

Следующий шаг:
- {рекомендация}
- /spec-status {short-path} {next-status}
```

### Проверка чек-листа

```
📋 Чек-лист перехода {type} {id}: {current} → {target}

- [x] Заполнена секция "Контекст"
- [x] Описаны варианты решения
- [ ] Указаны критерии оценки

❌ Чек-лист не пройден. Заполните недостающие пункты.
```

### Каскадные действия

```
✅ Статус изменён: Plan → DONE

📋 Каскадные действия:

1. Проверка: все планы ADR auth/002 завершены? ✅
   → Предлагаю: /spec-status auth/adr/002 done

Выполнить? [Y/n]
```

### REJECTED с причиной

```
⚠️ Для REJECTED необходимо указать причину.

Причина отклонения:
> {ввод пользователя}

✅ Статус изменён: REVIEW → REJECTED

Документ обновлён: добавлена причина отклонения

Варианты:
1. /spec-create {type} {parent} — создать новый документ
2. /spec-status {parent} rejected — отклонить родителя
```

## spec-update

### Контекст документа

```
📄 {Type} {id}-{topic}

Статус: {emoji} {STATUS}
Дата создания: {date}
Последнее изменение: {date}

Связанные документы:
- {Child type}: {path} ({status})
- {Child type}: {path} ({status})

Чек-лист текущего статуса:
- [x] {пункт 1}
- [x] {пункт 2}
- [ ] {пункт 3}

Что делаем?
```

### Режим валидации

```
📋 Валидация {Type} {id}-{topic}

Структура:
- [x] Метаданные корректны
- [x] Все обязательные секции присутствуют
- [x] Ссылки валидны

Чек-лист текущего статуса ({STATUS}):
- [x] {пункт 1}
- [ ] {пункт 2} ← не заполнено

Чек-лист следующего статуса ({NEXT-STATUS}):
- [ ] {пункт 1}
- [ ] {пункт 2}

Готовность к переходу: ❌ НЕ ГОТОВ
Недостающие пункты: {N}
```

## Диалоги подтверждения

### Формат подтверждения [Y/n]

```
{Вопрос}? [Y/n]
```

- `Y`, `y`, `да`, `yes`, пустой ввод → подтверждение
- `N`, `n`, `нет`, `no` → отмена

### Формат выбора

```
{Вопрос}? [все / критичные / выборочно / пропустить]
```

- `все`, `all`, `a` → выполнить все
- `критичные`, `critical`, `c` → только критичные
- `выборочно`, `select`, `s` → запросить по каждому
- `пропустить`, `skip`, `n` → ничего не делать

### Каскадные действия

```
📋 Каскадные действия:

{Список действий}

Выполнить? [Y/n]
```

При подтверждении — выполнить каждое действие последовательно.

## 1. Философия

**Спецификация первична, код вторичен.** Спецификации — SSOT проекта. Код — выражение спецификаций на конкретном языке. Обслуживание проекта = эволюция спецификаций. Пользователь описывает намерение, LLM собирает остальное.

**LLM не угадывает — уточняет.** На каждом уровне иерархии LLM задаёт уточняющие вопросы (Clarify) через AskUserQuestion, пока все неясности не закрыты. Если что-то осталось неясным — ставится блокирующий маркер `[ТРЕБУЕТ УТОЧНЕНИЯ]`.

**Тесты первичны, реализация вторична (ATDD).** Тестовые сценарии определяются до формирования плана реализации. Acceptance Test-Driven Development на уровне спецификаций.

**Инструкции распределены по объектам.** Нет единого файла "конституции". Принципы и правила живут в `.instructions/` каждого объекта — загружаются только при работе с ним. Шаблоны встроены в `standard-*.md`.

## 2. Объекты и иерархия

### Шесть уровней

| Уровень | Объект | Зона | Вопрос |
|---------|--------|------|--------|
| 1 | **Дискуссия** | ЗАЧЕМ и ЧТО | Что нужно? Какие требования? |
| 2 | **Импакт** | НА ЧТО ВЛИЯЕТ | Какие сервисы затронуты? Какие риски? |
| 3 | **Дизайн** | КАК ВСТРАИВАЕМ | Как распределяем ответственности? Какие контракты? |
| 4 | **ADR** | КАК КОНКРЕТНО | Какое техническое решение для сервиса? |
| 5 | **Тест-спек** | КАК ПРОВЕРЯЕМ | Как проверяем решение? Какие тестовые сценарии? |
| 6 | **План** | ЧТО ДЕЛАЕМ | Какие задачи? В каком порядке? |

Поток: Дискуссия → Импакт → Дизайн → ADR(ы) → Тест-спек(и) → План(ы) → GitHub Issues → Разработка.

### Таблица объектов

| Объект | Расположение | Родитель → Дети |
|--------|-------------|-----------------|
| **Дискуссия** | `specs/discussion/` | — → Импакт |
| **Импакт** | `specs/impact/` | Дискуссия → Дизайн |
| **Дизайн** | `specs/design/` | Импакт → ADR(ы) |
| **ADR** | `specs/services/{svc}/adr/` | Дизайн → Тест-спек(и) |
| **Тест-спек** | `specs/services/{svc}/test-specs/` | ADR → План (1:1) |
| **План** | `specs/services/{svc}/plans/` | Тест-спек → (терминальный) |

### Живые документы

Текущее состояние системы. Не имеют статусов — обновляются при каскаде DONE.

| Объект | Расположение | Когда обновляется |
|--------|-------------|-------------------|
| **Архитектура (системная)** | `specs/architecture/system/` | Design → DONE |
| **Архитектура (сервисная)** | `specs/architecture/services/{svc}.md` | ADR → DONE |
| **Архитектура (доменная)** | `specs/architecture/domains/` | Design → DONE |
| **Тесты (системные)** | `specs/tests/system/` | Design → DONE |
| **Тесты (сервисные)** | `specs/tests/services/{svc}/` | Test Spec → DONE |
| **Глоссарий** | `specs/glossary/` | На каждом уровне |

### Расширяемость

Добавление нового типа объекта = новая папка + новый `standard-*.md` в `.instructions/`. Существующие связи parent→children не меняются.

## 3. Связи и frontmatter

Каждый объект SDD содержит frontmatter со связями:

```yaml
---
parent: impact/001-oauth2-authorization.md
children:
  - services/auth/adr/001-jwt-to-oauth2.md
  - services/gateway/adr/001-oauth2-proxy.md
status: WAITING
milestone: v1.2.0   # только в Discussion, остальные наследуют
---
```

**Правила связей:**

| Объект | parent | children |
|--------|--------|----------|
| **Дискуссия** | нет | Импакт (1:1) |
| **Импакт** | Дискуссия | Дизайн (1:1) |
| **Дизайн** | Импакт | ADR(ы) |
| **ADR** | Дизайн | Тест-спек(и) |
| **Тест-спек** | ADR | План (1:1) |
| **План** | Тест-спек | нет (терминальный) |

**Milestone:** Определяется при Clarify на уровне Discussion и сохраняется в его frontmatter. Все дочерние документы наследуют milestone от Discussion.

## 4. Статусы

7 статусов жизненного цикла объекта:

| Статус | Значение |
|--------|----------|
| **DRAFT** | Документ создаётся, итерируется, ревьюится пользователем |
| **WAITING** | Пользователь согласовал. Ожидает готовности всей цепочки |
| **RUNNING** | Все уровни согласованы. Идёт реализация (код) |
| **DONE** | Реализация завершена, живые документы обновлены |
| **CONFLICT** | Обратная связь от кода требует пересмотра |
| **ROLLING_BACK** | Откат артефактов |
| **REJECTED** | Отклонён (финальный) |

**Допустимые переходы:**

| Из | В | Условие |
|----|---|---------|
| DRAFT | WAITING | Пользователь одобрил |
| DRAFT | REJECTED | Пользователь отклонил (работа не начиналась) |
| WAITING | DRAFT | Пользователь вернул на доработку |
| WAITING | REJECTED | Пользователь отклонил (работа не начиналась) |
| WAITING | RUNNING | Каскад — все планы в WAITING |
| RUNNING | DONE | Реализация завершена |
| RUNNING | CONFLICT | Обратная связь от кода |
| RUNNING | ROLLING_BACK | Откат (пользователь отменяет / каскад) |
| DONE | ROLLING_BACK | Каскад от сиблинга |
| DONE | CONFLICT | Каскад от родителя в CONFLICT |
| CONFLICT | RUNNING | Разрешение конфликта подтверждено |
| CONFLICT | ROLLING_BACK | Отклонение при разрешении |
| ROLLING_BACK | REJECTED | Откат завершён |

## 5. Каскады

### Прямой поток

Каждый документ проходит DRAFT → WAITING на своём уровне:

1. Discussion: DRAFT → [итерации с пользователем] → WAITING
2. Impact: DRAFT → [итерации] → WAITING
3. Design: DRAFT → [итерации] → WAITING
4. ADR (по каждому сервису): DRAFT → [итерации] → WAITING
5. Test Spec (по каждому сервису): DRAFT → [итерации] → WAITING
6. Plan (по каждому сервису): DRAFT → [итерации] → WAITING

### Каскад RUNNING

**Триггер:** LLM проверяет после каждого Plan → WAITING. Когда все Plans в WAITING — LLM предлагает переход через AskUserQuestion: "Все спецификации готовы. Перейти в RUNNING?"

**Переход:** При подтверждении **все** документы в цепочке (от Discussion до всех Plans) одновременно переходят в RUNNING. Это точка перехода к реализации — спецификации зафиксированы.

### Каскадное завершение (DONE)

Снизу вверх. Родитель → DONE когда **все дети DONE**.

| Триггер | Результат | Побочные эффекты |
|---------|-----------|-----------------|
| Plan → DONE | все задачи выполнены | — |
| Test Spec → DONE | все планы DONE | обновление `tests/services/{svc}/` |
| ADR → DONE | все тест-спеки DONE | обновление `architecture/services/{svc}.md`, удаление Planned Changes |
| Design → DONE | все ADR DONE | обновление `architecture/system/`, `domains/`, `tests/system/` |
| Impact → DONE | все дизайны DONE | — |
| Discussion → DONE | все импакты DONE | — |

Микс DONE/REJECTED невозможен — каскад через ROLLING_BACK гарантирует, что при отклонении одного ребёнка все сиблинги тоже откатываются.

**Глоссарий** не участвует в каскадном завершении. Обновляется непрерывно на каждом уровне при появлении новых терминов.

### Каскад REJECTED и ROLLING_BACK

Когда документ отклоняется:

1. **Каскад вниз:** все дочерние — RUNNING/DONE → ROLLING_BACK → REJECTED, DRAFT/WAITING → REJECTED напрямую (рекурсивно)
2. **Каскад по сиблингам:** все братья (дети того же родителя) — аналогично (+ их дети рекурсивно)
3. **Родитель → DRAFT:** общий контекст изменился — нужен пересмотр

Пользователь дорабатывает родителя → WAITING → новые дети генерируются заново.

**Пример:** Design → 3 ADR (auth, gateway, users). ADR auth отклонён:

```
ADR auth → ROLLING_BACK → REJECTED (откат architecture/services/auth.md)
  ├── вниз: Test Spec auth → ROLLING_BACK → REJECTED
  │         Plan auth → ROLLING_BACK → REJECTED (закрытие Issues)
  ├── сиблинги:
  │   ADR gateway (DONE) → ROLLING_BACK → REJECTED
  │     └── Test Spec, Plan gateway → ROLLING_BACK → REJECTED
  │   ADR users (WAITING) → REJECTED (артефактов нет)
  │     └── Test Spec users (DRAFT) → REJECTED
  └── вверх: Design → DRAFT
```

### Откат артефактов

При переходе в ROLLING_BACK LLM откатывает артефакты уровня:

| Уровень | Что откатывается |
|---------|-----------------|
| Discussion | нет артефактов (no-op) |
| Impact | нет артефактов (no-op) |
| Design | Откат `architecture/system/`, `architecture/domains/`, `tests/system/`. Удаление Planned Changes |
| ADR | Откат `architecture/services/{svc}.md` по дельта-блокам (ADDED удаляются, MODIFIED возвращаются, REMOVED восстанавливаются). Удаление Planned Changes |
| Test Spec | Откат `specs/tests/services/{svc}/` |
| Plan | Issues закрываются `--reason "not planned"`. Feature-ветка удаляется |

После отката → REJECTED (финальный).

## 6. Обратная связь Code-Specs

При разработке (RUNNING) код может выявить несовместимость со спецификациями. Проверку выполняет агент-разработчик непрерывно — при написании кода, запуске тестов, создании PR.

### Фаза 1. Обнаружение

Снизу вверх: LLM проверяет каждый уровень от Plan до Discussion — "Содержание этого документа стало неверным?"

**Документ затронут** если хотя бы одно утверждение фактически неверно из-за изменений в коде. **Не затронут** если утверждения остаются верными (рефакторинг внутри пакета, оптимизация).

**Критерий масштаба — границы автономии из Code Map:**

| Граница в Code Map | Уровень | Действие |
|---|---|---|
| **Свободно** (реализация внутри пакета) | не затронуты | нет обратной связи |
| **Флаг** (контракты между пакетами) | Plan / Test Spec | рабочие правки — LLM автономно обновляет, статус не меняется |
| **CONFLICT** (API сервиса, data model, пакеты) | ADR или выше | все затронутые + дочерние → CONFLICT. Работы останавливаются |

**Каскад CONFLICT на детей:**

| Статус ребёнка | Реакция |
|---|---|
| DRAFT | Перегенерация (остаётся DRAFT) |
| WAITING | → DRAFT |
| RUNNING | → CONFLICT |
| DONE | → CONFLICT (артефакты обновляются при разрешении) |
| CONFLICT | Учесть новые изменения |
| REJECTED | Не затрагивается |

CONFLICT каскадирует **только вниз**, не на сиблингов. Это отличие от REJECTED: REJECTED = "решение отклонено, весь контекст невалиден". CONFLICT = "нужен пересмотр конкретного документа".

### Фаза 2. Разрешение

Сверху вниз: начиная с самого высокого документа в CONFLICT.

1. Переписать самый высокий затронутый документ
2. На основе обновлённого — переписать дочерние (включая бывшие DONE)
3. Для DONE-документов: обновить документ **и** артефакты (живые документы, Issues)
4. Пользователь ревьюит → подтверждает → вся цепочка CONFLICT → RUNNING

Если пользователь отклоняет → ROLLING_BACK → REJECTED (каскад).

### Кросс-цепочечная обратная связь

Когда `architecture/` обновляется (каскад DONE или разрешение CONFLICT), проверить все другие цепочки, ссылающиеся на изменённые файлы.

Реакция затронутого документа зависит от его статуса:

| Статус | Реакция |
|---|---|
| DRAFT | Перегенерация с учётом нового `architecture/` |
| WAITING | → DRAFT |
| RUNNING | → CONFLICT |
| CONFLICT | Учесть новые изменения |
| DONE | LLM предлагает пользователю создать **новую Discussion** для приведения к общему знаменателю |
| REJECTED | Не обрабатывается |

**Определение "кого проверять":** Planned Changes в `architecture/` показывают, какие цепочки затрагивают какие файлы.

## 7. Зоны ответственности

| Зона | Папка | Вопрос | Содержит | НЕ содержит |
|------|-------|--------|----------|-------------|
| ЗАЧЕМ и ЧТО | `discussion/` | Зачем это нужно? | Проблему, требования, user stories, варианты, критерии | Технические детали |
| НА ЧТО ВЛИЯЕТ | `impact/` | Какие сервисы? | Список сервисов, риски, зависимости | Распределение ответственностей |
| КАК ВСТРАИВАЕМ | `design/` | Как распределяем? | Секции по сервисам, блоки взаимодействия, контракты, системные тест-сценарии | Детали реализации конкретного сервиса |
| КАК КОНКРЕТНО | `services/{svc}/adr/` | Какое решение? | Контекст, решение, trade-offs, последствия | Тестовые сценарии, задачи |
| КАК ПРОВЕРЯЕМ | `services/{svc}/test-specs/` | Как проверяем? | Тестовые сценарии, acceptance criteria, тестовые данные | Реализацию тестов |
| ЧТО ДЕЛАЕМ | `services/{svc}/plans/` | Какие задачи? | Задачи, сложность, зависимости | Бизнес-обоснование |
| АРХИТЕКТУРА | `architecture/` | Как устроена система? | Живое AS IS, Planned Changes | Исторические решения (в ADR) |
| ТЕСТЫ | `tests/` | Какие тесты существуют? | Живое AS IS. Зеркало кодовой базы | Сами тесты (в /tests/ и /src/) |
| ТЕРМИНЫ | `glossary/` | Что означает термин? | Определения по доменам | Решения и требования |
| ПРАВИЛА | `.instructions/` | Как создавать объекты? | Стандарты, чек-листы, шаблоны | Контент спецификаций |

**Границы:** specs/ отвечает за ЗАЧЕМ, ЧТО, КАК, ПРОВЕРКА. Остальной проект (`src/`, `tests/`, `.github/`, `config/`, `platform/`) — за РЕАЛИЗАЦИЮ.

## 8. Clarify и блокирующие правила

### Clarify на каждом уровне

Clarify — паттерн, повторяющийся на каждом уровне:

| Уровень | Что уточняется |
|---------|---------------|
| Дискуссия | Проблема, scope, требования, критерии успеха |
| Импакт | Какие сервисы затронуты, компоненты, скрытые зависимости |
| Дизайн | Распределение ответственностей, контракты API, порядок взаимодействия |
| ADR | Технический выбор, trade-offs, совместимость с архитектурой |
| Тест-спек | Типы тестов, покрытие, тестовые данные, граничные кейсы |
| План | Приоритеты задач, порядок реализации, ресурсы |

**Механизм:** LLM использует AskUserQuestion. Проходит по секциям шаблона из `standard-*.md` и для каждой определяет, достаточно ли контекста.

**Пропуск Clarify:** Пользователь может указать `--auto-clarify` — LLM пропускает Clarify и генерирует документ, ставя маркеры где необходимо. Шаг REVIEW остаётся обязательным.

### Маркер ТРЕБУЕТ УТОЧНЕНИЯ

**БЛОКИРУЮЩЕЕ. НЕПРИКАСАЕМОЕ.**

При создании или обновлении ЛЮБОГО объекта в specs/, если LLM не имеет достаточной информации:

1. **ОБЯЗАН** поставить маркер: `[ТРЕБУЕТ УТОЧНЕНИЯ: конкретный вопрос]`
2. **ЗАПРЕЩЕНО** угадывать, домысливать, делать допущения
3. **ЗАПРЕЩЕНО** продолжать генерацию зависимых секций
4. Документ **НЕ МОЖЕТ** покинуть статус DRAFT с неразрешёнными маркерами

**Разрешение:** LLM показывает маркеры пользователю → пользователь отвечает → LLM заменяет маркер на ответ.

### Dependency Barrier

**БЛОКИРУЮЩЕЕ.**

При генерации документа LLM может ставить независимые маркеры и продолжать генерацию. Но если для генерации следующей секции нужен ответ на ранее поставленный маркер — срабатывает Dependency Barrier.

**Режим 1 — Полная генерация** (маркеры независимы):

```markdown
## Секция А
Описание... [ТРЕБУЕТ УТОЧНЕНИЯ: какой протокол авторизации?]

## Секция Б
Описание... [ТРЕБУЕТ УТОЧНЕНИЯ: какой SLA требуется?]
```

**Режим 2 — Барьер** (обнаружена зависимость):

```markdown
## Секция А
Описание... [ТРЕБУЕТ УТОЧНЕНИЯ: какой протокол авторизации?]

## Секция Б
Описание... [ТРЕБУЕТ УТОЧНЕНИЯ: какой SLA требуется?]

---

### DEPENDENCY BARRIER

Дальнейшая генерация остановлена: секция В зависит от маркера в секции А.

**Требует дальнейшего описания:**

| Секция | Зависит от | Что будет описано |
|--------|-----------|-------------------|
| Секция В: Механизм обмена | протокол авторизации | Протокол обмена, формат |
| Секция Г: Ротация секретов | протокол авторизации | Алгоритм ротации |
```

LLM прекращает генерацию контента и переключается на перечисление оставшихся секций с зависимостями. Экономит токены, предотвращает переписывание.

**Взаимодействие:** Clarify происходит **до** генерации (снижает маркеры). Dependency Barrier срабатывает **во время** генерации.

## 9. Именование и формат README-таблиц

**Именование файлов спецификаций:**

```
NNN-topic.md
```

`NNN` — трёхзначный автоинкремент (001, 002, ...). Нумерация **независимая** в каждой папке.

**Формат README-таблицы** (в каждой папке `discussion/`, `impact/`, `design/`, `services/{svc}/adr/`, и т.д.):

| # | Документ | Статус | Описание |
|---|----------|--------|----------|
| 001 | 001-oauth2-authorization.md | RUNNING | OAuth2 авторизация |

**Стандарты объектов** определяют конкретные колонки для своих README-таблиц. Общие колонки: `#`, `Документ`, `Статус`.

## 10. Запреты

**Запрет миграции:** Объекты specs/ не мигрируют между уровнями. Discussion не становится ADR. Если нужен другой уровень — создаётся новый документ с ссылкой.

**Запрет архивирования:** Нет архива. REJECTED — финальный статус. Документ остаётся в папке для истории решений.

**Запрет версионирования файлов:** Документы не имеют файловых версий. Версионирование — через цепочку ADR: новый ADR видит AS IS из `architecture/` и определяет TO BE.

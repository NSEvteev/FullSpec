# disc-0001: Система уведомлений в реальном времени

## Проблема / Контекст

**Текущее состояние (AS IS):**

Платформа не имеет механизма уведомления пользователей о системных событиях. Информация о регистрации, смене пароля, действиях администратора и ошибках системы доступна только в логах и не доводится до конечного пользователя.

**Проблема:**

Пользователи не получают обратную связь о критических системных событиях — успешной регистрации, подтверждении смены пароля, действиях администратора (блокировка, изменение прав), сбоях сервисов. Это снижает прозрачность системы и вынуждает пользователей самостоятельно проверять статус операций. Необходима система in-app уведомлений в реальном времени через WebSocket, охватывающая frontend (UI-компоненты), backend (сервис уведомлений), database (хранение и статусы) и API Gateway (маршрутизация WebSocket-соединений).

## Фичи

| ID | Название | Описание |
|----|----------|----------|
| F-1 | WebSocket-соединение | Установка и поддержание постоянного WebSocket-соединения между клиентом и сервером через API Gateway с автоматическим reconnect при обрыве |
| F-2 | Notification Service | Backend-сервис, принимающий системные события от других сервисов и формирующий уведомления для целевых пользователей |
| F-3 | Хранение уведомлений | Персистентное хранение уведомлений в БД с поддержкой статусов (unread, read), пагинацией и TTL для автоочистки |
| F-4 | UI уведомлений | Компоненты фронтенда — bell-иконка с badge-счётчиком непрочитанных, dropdown-список, toast-нотификации для новых уведомлений |
| F-5 | Gateway WebSocket Proxy | Маршрутизация и проксирование WebSocket-соединений через API Gateway с аутентификацией и rate limiting |

## User Stories

| ID | Роль | Действие | Цель |
|----|------|----------|------|
| US-1 | Пользователь | Вижу toast-уведомление при успешной регистрации | Получить подтверждение что аккаунт создан |
| US-2 | Пользователь | Вижу badge на bell-иконке с числом непрочитанных | Знать сколько новых уведомлений требуют внимания |
| US-3 | Пользователь | Открываю dropdown и вижу список уведомлений | Просмотреть историю системных событий |
| US-4 | Пользователь | Нажимаю на уведомление и оно помечается прочитанным | Отслеживать что уже просмотрено |
| US-5 | Пользователь | Получаю уведомление о действиях администратора | Быть в курсе изменений моих прав или статуса аккаунта |
| US-6 | Администратор | Вижу уведомления об ошибках системы | Оперативно реагировать на сбои сервисов |

## Требования

| ID | Связь | Требование |
|----|-------|------------|
| REQ-1 | F-1, US-1 | При загрузке приложения клиент устанавливает WebSocket-соединение с аутентификацией по JWT-токену — сервер подтверждает подключение и начинает push уведомлений |
| REQ-2 | F-1 | При обрыве WebSocket-соединения клиент выполняет exponential backoff reconnect — максимум 5 попыток с интервалами 1s, 2s, 4s, 8s, 16s |
| REQ-3 | F-2, US-5, US-6 | Notification Service подписывается на системные события (регистрация, смена пароля, действия админа, ошибки) — формирует уведомление и доставляет целевому пользователю через WebSocket |
| REQ-4 | F-3, US-3 | Уведомления хранятся в БД с полями: id, user_id, type, title, body, status, created_at — API возвращает список с пагинацией (limit/offset) |
| REQ-5 | F-3 | Уведомления старше 90 дней автоматически удаляются фоновой задачей (TTL cleanup) |
| REQ-6 | F-4, US-2 | Bell-иконка отображает badge с количеством непрочитанных уведомлений — badge обновляется в реальном времени при получении нового уведомления или прочтении существующего |
| REQ-7 | F-4, US-1 | При получении нового уведомления отображается toast-нотификация в правом верхнем углу — автоматически скрывается через 5 секунд |
| REQ-8 | F-5 | API Gateway проксирует WebSocket upgrade-запросы на Notification Service — проверяет JWT-токен при handshake и применяет rate limiting (max 10 соединений на пользователя) |

## Критерии успеха

- Уведомление доставляется пользователю в течение 2 секунд после системного события
- WebSocket reconnect восстанавливает соединение без потери уведомлений (missed notifications доставляются при reconnect)
- API `/notifications` отвечает за < 200ms при выборке 50 уведомлений
- Bell badge точно отражает количество непрочитанных (рассинхронизация = 0)
- Система выдерживает 1000 одновременных WebSocket-соединений на один инстанс Notification Service

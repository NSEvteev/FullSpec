# Особенности реализации /.claude/instructions/specs/

> **Статус:** DRAFT
> **Дата:** 2025-01-21
> **Контекст:** Документ описывает паттерны и best practices, применённые при создании инструкций для `/specs/`

---

## Цель документа

Зафиксировать архитектурные решения и паттерны реализации инструкций `/specs/`, чтобы:
1. Использовать как эталон для рефакторинга других инструкций
2. Обеспечить консистентность при создании новых инструкций
3. Документировать "почему так сделано"

---

## Ключевые паттерны

### 1. Вынос шаблонов в отдельные файлы

**Было (в других инструкциях):**
```markdown
## Формат документа

```markdown
# Название

## Секция 1
...
```
```

**Стало (в specs):**
```markdown
## Формат документа

> **Шаблон:** [/.claude/templates/specs/discussion.md](/.claude/templates/specs/discussion.md)

Шаблон и пример заполнения находятся в файле шаблона.
```

**Преимущества:**
- Один источник истины
- Шаблон можно использовать программно (скиллами)
- Документация не расходится с реальным шаблоном
- Примеры заполнения в HTML-комментариях не загромождают инструкцию

**Расположение шаблонов:**
```
/.claude/templates/specs/
├── discussion.md
├── impact.md
├── adr.md
├── plan.md
└── architecture.md
```

### 2. Структура README.md как индекса

**Паттерн:**
- Оглавление в виде таблицы с секцией, инструкцией и описанием
- Нумерованные секции (1. Структура, 2. Статусы, ...)
- Краткое описание в README, детали в отдельных файлах
- Секции без отдельного файла помечены "—" в столбце инструкции

**Пример:**
```markdown
| Секция | Инструкция | Описание |
|--------|------------|----------|
| [1. Структура /specs/](#1-структура-specs) | — | Назначение и структура папки |
| [2. Статусы](#2-статусы) | [statuses.md](./statuses.md) | Унифицированная система статусов |
```

**Преимущества:**
- Быстрая навигация по всей системе
- Понятно, где искать информацию
- README не перегружен деталями

### 3. Чек-листы переходов статусов

**Паттерн:** Для каждого перехода статуса:
- Указан скилл для выполнения
- Условия перехода (когда)
- Чек-лист (что проверяется)
- Каскадные действия (что происходит автоматически)

**Пример:**
```markdown
### 📝 DRAFT → 🔍 REVIEW

| | |
|---|---|
| **Скилл** | `/spec-status discussions/ <id> review` |
| **Когда** | Документ готов к обсуждению |

**Чек-лист (проверяется скиллом):**
- [ ] Заполнена секция "Контекст"
- [ ] Описаны минимум 2 варианта решения

**Каскадные действия:** нет
```

**Преимущества:**
- Явные правила перехода
- Автоматизация через скиллы
- Чек-листы гарантируют качество

---

### 4. Единая система статусов

**Паттерн:** Все статусы описаны в одном файле `statuses.md`:
- Полный список статусов с emoji
- Схема переходов (mermaid диаграмма)
- Правила перехода по типам документов
- Каскадные проверки
- Правила для REJECTED

**Преимущества:**
- Один источник истины для статусов
- Легко понять полную картину workflow
- Нет дублирования в каждом типе документа

---

### 5. Связи между инструкциями

**Паттерн:** Frontmatter с `related`:
```yaml
---
type: standard
description: ...
related:
  - specs/statuses.md
  - specs/workflow.md
---
```

**Паттерн:** Секция "Связанные инструкции" в конце:
```markdown
## Связанные инструкции

- [statuses.md](./statuses.md) — система статусов документов
- [workflow.md](./workflow.md) — полный workflow от идеи до реализации
```

**Преимущества:**
- Навигация между связанными темами
- Понятен контекст инструкции
- Можно строить граф зависимостей

---

### 6. Секция "Скиллы автоматизации"

**Паттерн:** Таблица скиллов с описанием:
```markdown
| Скилл | Назначение |
|-------|------------|
| [/spec-create](/.claude/skills/spec-create/SKILL.md) | Создание документов |
| [/spec-status](/.claude/skills/spec-status/SKILL.md) | Изменение статуса |
```

**Преимущества:**
- Понятно, какие скиллы использовать
- Ссылки ведут на документацию скилла
- Автоматизация явно задокументирована

---

### 7. Запреты и правила

**Паттерн:** Явные секции для запретов:
- Запрет миграции
- Запрет удаления
- Запрет архивирования

Каждый запрет содержит:
- Обоснование (почему)
- Альтернатива (что делать вместо)
- Исключения (если есть)

**Пример:**
```markdown
## Запрет удаления

> **ПРАВИЛО:** Физическое удаление документов ЗАПРЕЩЕНО.

### Обоснование
- Git history сохраняет контекст

### Вместо удаления — статус
| Ситуация | Действие |
|----------|----------|
| Документ неактуален | REJECTED |
```

## Сравнение с другими инструкциями

| Аспект | specs | Другие |
|--------|-------|--------|
| Шаблоны | Вынесены в `/.claude/templates/` | Inline в документе |
| README | Нумерованные секции, таблица навигации | Простой список |
| Чек-листы | Для каждого перехода статуса | Редко |
| Статусы | Централизованы в `statuses.md` | Нет или разрозненно |
| Скиллы | Таблица с ссылками | Не всегда указаны |
| Запреты | Явные секции с обоснованием | Редко |
| Frontmatter | type + description + related | Не всегда полный |

## Рекомендации по применению

### Когда применять паттерны specs

1. **Папка с workflow** — если есть переходы между состояниями
2. **Несколько связанных типов документов** — если есть семейство сущностей
3. **Защищённая область** — если нужен контроль через скиллы
4. **Часто создаваемые документы** — если шаблоны используются регулярно

### Когда НЕ применять

1. **Простая инструкция** — если это одиночное правило без workflow
2. **Редко используемая** — если шаблоны не нужны
3. **Нет автоматизации** — если нет скиллов для области

## Чек-лист рефакторинга инструкций

При рефакторинге инструкций по паттернам specs:

### Фаза 1: Анализ
- [ ] Определить scope папки (какие сущности)
- [ ] Выявить шаблоны, которые можно вынести
- [ ] Определить связи между инструкциями
- [ ] Проверить наличие workflow/статусов

### Фаза 2: Шаблоны
- [ ] Вынести шаблоны в `/.claude/templates/{scope}/`
- [ ] Добавить примеры заполнения в HTML-комментариях
- [ ] Заменить inline-шаблоны на ссылки

### Фаза 3: Структура
- [ ] Создать/обновить README.md как индекс
- [ ] Добавить нумерованные секции
- [ ] Добавить таблицу навигации с описаниями

### Фаза 4: Связи
- [ ] Добавить frontmatter (type, description, related)
- [ ] Добавить секцию "Связанные инструкции"
- [ ] Добавить секцию "Скиллы" (если есть)

### Фаза 5: Качество
- [ ] Проверить все ссылки
- [ ] Убрать дублирование
- [ ] Добавить запреты/правила где нужно

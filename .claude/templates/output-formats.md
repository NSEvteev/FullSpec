# Output Formats — единый источник истины

> **SSOT:** Этот файл — единственный источник форматов вывода в скиллах.
> Все скиллы ДОЛЖНЫ следовать этим форматам для консистентности.

**Связанные шаблоны:**
- [error-handling.md](/.claude/templates/error-handling.md) — обработка ошибок (детальный формат ❌)

---

## Принцип

Вывод в скиллах должен быть:
- **Узнаваемым** — пользователь сразу понимает тип сообщения по иконке
- **Консистентным** — одинаковый формат во всех скиллах
- **Информативным** — содержит всё необходимое для понимания

---

## Иконки и их значение

| Иконка | Тип | Когда использовать |
|--------|-----|-------------------|
| ✅ | Успех | Операция завершена успешно |
| ❌ | Ошибка | Операция не выполнена |
| ⚠️ | Предупреждение | Требует внимания, но не блокирует |
| 📋 | Ревью/Список | Обзор, чек-лист, результаты проверки |
| ⏳ | Прогресс | Операция выполняется |
| ℹ️ | Информация | Нейтральное сообщение |
| 💡 | Предложение | Рекомендация, улучшение |
| ❓ | Вопрос | Требуется ввод пользователя |

---

## Форматы вывода

### Успех (✅)

**Когда:** Операция завершена полностью и успешно.

```
✅ {Объект} {действие}

{Детали результата}

Следующие шаги:
- {шаг 1}
- {шаг 2}
```

**Пример:**

```
✅ Скилл создан успешно

Файл: /.claude/skills/my-skill/SKILL.md
Категория: utility
Добавлен в индекс: /.claude/instructions/tools/skills.md

Следующие шаги:
- Проверить и доработать инструкции в SKILL.md
- Протестировать скилл
```

**Пример (короткий):**

```
✅ Issue #123 закрыт как выполненный
```

---

### Ошибка (❌)

**Когда:** Операция не может быть выполнена.

> **SSOT:** Детальный формат ошибок см. в [error-handling.md](/.claude/templates/error-handling.md)

**Краткий формат:**

```
❌ Ошибка: {краткое описание}

Причина: {детали}
Решение: {что сделать}
```

**Пример:**

```
❌ Ошибка: Файл уже существует

Причина: /.claude/skills/my-skill/SKILL.md уже существует
Решение: Используйте /skill-update для редактирования
```

---

### Предупреждение (⚠️)

**Когда:** Операция может продолжиться, но есть риск или особенность.

```
⚠️ {Краткое описание ситуации}

{Детали}

Варианты:
[1] {Вариант 1}
[2] {Вариант 2}
[3] Отменить
```

**Пример:**

```
⚠️ Issue #123 уже в работе

Исполнитель: @another-user
Взят: 2026-01-19

Варианты:
[1] Переназначить на себя
[2] Открыть без переназначения
[3] Отменить
```

**Пример (информационный):**

```
⚠️ Есть несохранённые изменения

Файлы:
- src/auth/login.ts (modified)
- src/auth/logout.ts (modified)

Stash и продолжить? [Y/n]
```

### Ревью / Список (📋)

**Когда:** Показ результатов проверки, обзор объекта, чек-лист.

**Формат обзора:**

```
📋 {Тип} #{номер или имя}

Заголовок: {название}
Статус: {статус}
{Другие поля}

{Дополнительная информация}
```

**Пример:**

```
📋 Issue #123

Заголовок: [AUTH] Добавить OAuth авторизацию
Статус: OPEN
Метки: service:auth, feature, in-progress
Исполнитель: @user

Критерии готовности:
- [x] Генерация QR-кода
- [x] Валидация TOTP
- [ ] Backup codes
```

**Формат ревью:**

```
📋 Ревью {объекта}

✅ Сильные стороны:
- {пункт 1}
- {пункт 2}

💡 Предложения по улучшению:
1. {предложение 1}
2. {предложение 2}

❓ Вопросы для уточнения:
- {вопрос}

Применить предложения? (все / выборочно / пропустить)
```

**Формат чек-листа выполнения:**

```
✅ Шаг 1: Получил номер Issue
✅ Шаг 2: Проверил статус
✅ Шаг 3: Добавил комментарий
⏳ Шаг 4: Закрываю Issue...
⬜ Шаг 5: Обновить документацию
```

---

### Прогресс (⏳)

**Когда:** Операция выполняется, промежуточный статус.

```
⏳ {Действие}...

{Детали, если есть}
```

**Пример:**

```
⏳ Создаю ветку feature/123-oauth-auth...
```

**Пример с прогрессом:**

```
⏳ Обновление документации (3/7)...

Обновлено:
- ✅ skills.md
- ✅ agents.md
- ✅ README.md
Осталось: 4 файла
```

---

### Информация (ℹ️)

**Когда:** Нейтральное сообщение, не требующее действий.

```
ℹ️ {Сообщение}

{Детали, если нужны}
```

**Пример:**

```
ℹ️ Изменения НЕ применены (--dry-run)

Для применения запустите команду без флага --dry-run
```

**Пример:**

```
ℹ️ Issue #123 уже закрыт

Статус: CLOSED
Причина: completed
Закрыт: 2026-01-15

Действий не требуется.
```

---

### Предложение (💡)

**Когда:** Рекомендация, не обязательная к выполнению.

```
💡 {Предложение}

{Обоснование}

Применить? [Y/n]
```

**Пример:**

```
💡 Рекомендую добавить тесты для нового скилла

Скилл my-skill не имеет tests.md.
Это нарушает стандарт 100% покрытия тестами.

Создать tests.md? [Y/n]
```

---

### Вопрос (❓)

**Когда:** Требуется выбор или ввод пользователя.

**Формат с вариантами:**

```
❓ {Вопрос}

[1] {Вариант 1} — {описание}
[2] {Вариант 2} — {описание}
[3] {Вариант 3} — {описание}

Что выбрать? [1/2/3]
```

**Формат да/нет:**

```
{Контекст}

{Вопрос}? [Y/n]
```

**Пример:**

```
❓ Причина закрытия?

[1] Неактуально (not_planned)
[2] Дубликат (duplicate)

Что выбрать? [1/2]
```

## Составные форматы

### Предварительный просмотр (--dry-run)

```
📋 Предварительный просмотр (--dry-run)

Будет создано:
- {путь 1}
- {путь 2}

Будет обновлено:
- {путь 3}
- {путь 4}

Будет вызвано:
- /{скилл-1}
- /{скилл-2}

ℹ️ Изменения НЕ применены (--dry-run)
```

### Частичный успех

```
⚠️ Частичный успех

Выполнено:
- ✅ {операция 1}
- ✅ {операция 2}

Не выполнено:
- ❌ {операция 3} — {причина}

Варианты:
[1] Откатить всё и повторить
[2] Оставить выполненное
[3] Продолжить вручную

Что выбрать?
```

### Результат с метриками

```
✅ {Операция} завершена

Статистика:
| Метрика | Значение |
|---------|----------|
| Обработано | {N} |
| Успешно | {M} |
| Пропущено | {K} |

Время: {X} сек
```

## Таблицы

### Формат таблицы статусов

```
| Объект | Статус | Примечание |
|--------|--------|------------|
| file1.md | ✅ Обновлён | — |
| file2.md | ⚠️ Пропущен | Нет изменений |
| file3.md | ❌ Ошибка | Нет доступа |
```

### Формат таблицы сравнения

```
| Метрика | Было | Стало | Δ |
|---------|------|-------|---|
| Файлов | 10 | 12 | +2 |
| Строк | 500 | 650 | +150 |
```

## FAQ

### Когда использовать ✅ vs ℹ️?

| Ситуация | Иконка |
|----------|--------|
| Операция выполнена успешно | ✅ |
| Операция не требовалась | ℹ️ |
| Режим --dry-run | ℹ️ |
| Объект уже в нужном состоянии | ℹ️ |

### Когда использовать ⚠️ vs ❌?

| Ситуация | Иконка |
|----------|--------|
| Можно продолжить с выбором | ⚠️ |
| Операция невозможна | ❌ |
| Есть риск, но не критичный | ⚠️ |
| Критическая ошибка | ❌ |

### Нужно ли дублировать "Следующие шаги"?

**Правило:** "Следующие шаги" добавляются только для операций создания или значимых изменений. Для простых операций (закрытие, удаление) — не нужны.

### Как форматировать длинные списки?

Если список > 5 элементов — использовать таблицу или сворачивать:

```
Обновлено файлов: 12
(используйте --verbose для полного списка)
```

## JSON формат (--json)

Для интеграции с CI/CD и скриптами скиллы поддерживают `--json` флаг.

### Структура JSON

```json
{
  "status": "success|error|warning",
  "message": "Краткое описание результата",
  "data": {
    // Специфичные данные операции
  },
  "meta": {
    "skill": "skill-name",
    "timestamp": "2026-01-20T14:30:00Z",
    "duration_ms": 1250
  }
}
```

### Примеры

**Успех:**
```json
{
  "status": "success",
  "message": "Скилл создан успешно",
  "data": {
    "path": "/.claude/skills/my-skill/SKILL.md",
    "category": "utility"
  },
  "meta": {
    "skill": "skill-create",
    "timestamp": "2026-01-20T14:30:00Z"
  }
}
```

**Ошибка:**
```json
{
  "status": "error",
  "message": "Файл уже существует",
  "error": {
    "code": "FILE_EXISTS",
    "details": "/.claude/skills/my-skill/SKILL.md"
  },
  "meta": {
    "skill": "skill-create"
  }
}
```

**С verbose:**
```json
{
  "status": "success",
  "message": "Проверка завершена",
  "data": {
    "files_checked": 87,
    "links_checked": 342,
    "valid": 340,
    "broken": 2
  },
  "details": [
    {"file": "doc/api.md", "line": 42, "link": "../src/auth", "reason": "not found"}
  ],
  "meta": {
    "skill": "links-validate",
    "verbose": true
  }
}
```

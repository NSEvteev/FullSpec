## 1. Назначение

Папка `/.claude/state/` — единое место для координации и логирования работы агентов.

**Решаемые проблемы:**

| Проблема | Решение |
|----------|---------|
| Агенты не знают друг о друге | `agents-status.json` |
| Конфликты при записи в файл | `locks.json` |
| Нет истории действий агента | `agent-{name}-operation.json` |
| Потеря контекста | Персистентное хранение |

**Когда использовать state:**

- Координация между параллельными агентами
- Логирование операций агента
- Блокировка файлов при записи
- Сохранение промежуточных результатов

**Когда НЕ использовать state:**

- Данные для версионирования в git
- Секретная информация (использовать `.env`)
- Результаты тестов (отдельная система)

## 2. Структура папки

```
/.claude/state/
├── agents-status.json              # Статус всех запущенных агентов
├── locks.json                      # Блокировки файлов
├── agent-{name}-operation.json     # Лог операций агента (создаётся при первом запуске)
└── {name}-*.json                   # Временные файлы агентов (опционально)
```

**Правила:**

| Тип файла | Формат имени | Создание |
|-----------|--------------|----------|
| Статус агентов | `agents-status.json` | Заранее создан |
| Блокировки | `locks.json` | Заранее создан |
| Лог агента | `agent-{name}-operation.json` | При первом запуске агента |
| Временные файлы | `{agent-name}-*.json` | По необходимости |

**Важно:**
- Файлы `agents-status.json` и `locks.json` должны существовать заранее
- Эти файлы **не требуют блокировки** — каждый агент пишет только свои данные (свой ключ в JSON)
- Агенты не создают эти файлы, только пишут в них

## 3. Формат файлов

### agents-status.json

> **Файл:** [/.claude/state/agents-status.json](/.claude/state/agents-status.json)

**Назначение:** Общий статус всех агентов для координации.

**Структура:**

```json
{
  "agents": {
    "refactoring-agent": {
      "status": "running",
      "started_at": "2026-01-31T10:30:00Z",
      "finished_at": null,
      "task": "Рефакторинг модуля auth",
      "working_on": ["src/auth/handlers.py", "src/auth/utils.py"]
    },
    "code-reviewer": {
      "status": "completed",
      "started_at": "2026-01-31T10:25:00Z",
      "finished_at": "2026-01-31T10:28:00Z",
      "task": "Ревью PR #42",
      "working_on": []
    }
  }
}
```

**Поля агента:**

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| `status` | enum | Да | `running`, `completed`, `failed` |
| `started_at` | string | Да | ISO 8601 — время начала |
| `finished_at` | string | Нет | ISO 8601 — время завершения (null если running) |
| `task` | string | Да | Краткое описание задачи |
| `working_on` | array | Да | Список файлов, с которыми работает |

**Алгоритм работы:**

1. При старте агент читает файл
2. Добавляет/обновляет **только свою** запись (`agents.{my-name}`) со статусом `running`
3. При завершении — обновляет статус на `completed`/`failed`

**Примечание:** Блокировка файла не нужна — агенты работают с разными ключами.

---

### locks.json

> **Файл:** [/.claude/state/locks.json](/.claude/state/locks.json)

**Назначение:** Единый файл блокировок для предотвращения конфликтов записи.

**Структура:**

```json
{
  "locks": {
    "/src/auth/handlers.py": {
      "agent": "refactoring-agent",
      "locked_at": "2026-01-31T10:30:00Z",
      "reason": "Рефакторинг функции validate_token"
    },
    "/src/api/routes.py": {
      "agent": "code-reviewer",
      "locked_at": "2026-01-31T10:31:00Z",
      "reason": "Добавление комментариев"
    }
  }
}
```

**Поля блокировки:**

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| `agent` | string | Да | Имя агента, создавшего блокировку |
| `locked_at` | string | Да | ISO 8601 — время блокировки |
| `reason` | string | Нет | Причина блокировки |

**Алгоритм работы:**

1. **Перед записью в файл проекта** — прочитать `locks.json`
2. **Проверить** — есть ли блокировка на нужный файл (от другого агента)
3. **Если заблокирован:**
   - Сделать 6 проверок с интервалом 5 секунд (суммарно 30 сек)
   - Если после 6 проверок блокировка не снята → вернуть ошибку: "Файл {путь} заблокирован агентом {имя} более 30 сек."
4. **Если свободен** — добавить свою запись о блокировке
5. **Выполнить операцию**
   - При любой ошибке (запись, валидация, внутренняя) — перейти к шагу 7
6. **Добавить запись в `operations[]`** с типом и путём
7. **Снять блокировку** — удалить **только свою** запись из `locks` (ВСЕГДА, даже при ошибке!)

**Примечание:** Сам `locks.json` не требует блокировки — каждый агент добавляет/удаляет только свои записи.

### agent-{name}-operation.json

**Назначение:** Детальный лог операций конкретного агента.

**Структура:**

```json
{
  "agent": "refactoring-agent",
  "description": "Рефакторинг модуля авторизации",
  "started_at": "2026-01-31T10:30:00Z",
  "finished_at": "2026-01-31T10:45:00Z",
  "context": "Задача от пользователя: улучшить читаемость кода auth",
  "operations": [
    {
      "timestamp": "2026-01-31T10:30:15Z",
      "action": "read",
      "file": "src/auth/handlers.py",
      "reason": "Анализ текущей структуры"
    },
    {
      "timestamp": "2026-01-31T10:32:00Z",
      "action": "edit",
      "file": "src/auth/handlers.py",
      "reason": "Выделение функции validate_token в отдельный метод"
    },
    {
      "timestamp": "2026-01-31T10:35:00Z",
      "action": "create",
      "file": "src/auth/validators.py",
      "reason": "Новый модуль для валидаторов"
    }
  ]
}
```

**Поля:**

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| `agent` | string | Да | Имя агента |
| `description` | string | Да | Описание задачи |
| `started_at` | string | Да | ISO 8601 — время начала |
| `finished_at` | string | Нет | ISO 8601 — время завершения |
| `context` | string | Да | Контекст/причина запуска |
| `operations` | array | Да | Список операций |

**Поля операции:**

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| `timestamp` | string | Да | ISO 8601 — время операции |
| `action` | enum | Да | `read`, `edit`, `create`, `delete` |
| `file` | string | Да | Путь к файлу |
| `reason` | string | Да | Зачем выполняется операция |

**Что логировать в operations[]:**

| Действие | action | Логировать? |
|----------|--------|-------------|
| Чтение файла проекта по задаче | `read` | Да |
| Запись/изменение файла | `write`/`edit` | Да |
| Создание нового файла | `create` | Да |
| Удаление файла | `delete` | Да |

**НЕ логировать:**

| Файл | Причина |
|------|---------|
| `standard-*.md`, `validation-*.md` | Это технический контекст агента |
| `locks.json`, `agents-status.json` | Это state-файлы координации |
| `/.claude/rules/*.md` | Это правила проекта (контекст) |
| `agent-*-operation.json` | Это сам лог, не нужно логировать лог |

**Обновление finished_at:**

- Обновить `finished_at` ТОЛЬКО когда агент вернул **финальный** отчёт пользователю
- Если пользователь запросил дополнительные правки → НЕ обновлять `finished_at` (это продолжение той же работы)
- Примеры:
  - Вернул отчёт "Валидация завершена" → обновить `finished_at`
  - Пользователь попросил "исправь ещё это" → НЕ обновлять, продолжить работу

## 4. Правила работы с state

### Правило блокировок

**Правило:** Перед записью в файл проекта — проверить и создать блокировку в `locks.json`.

**Примечание:** Сам `locks.json` не требует блокировки — каждый агент управляет только своими записями.

**Алгоритм (6 проверок × 5 сек = 30 сек):**

```
1. Прочитать locks.json
2. Проверить, заблокирован ли файл другим агентом
3. Если заблокирован:
   a. Подождать 5 секунд
   b. Повторить проверку (до 6 раз)
   c. Если после 6 проверок всё ещё занят → вернуть ошибку
4. Добавить свою блокировку
5. Выполнить операцию
6. Удалить свою блокировку из locks.json (ВСЕГДА, даже при ошибке!)
```

**Формат ошибки при timeout:**
```
"Файл {путь} заблокирован агентом {имя} более 30 сек. Операция отменена."
```

**Когда нужна блокировка:**

| Операция | Блокировка |
|----------|------------|
| Чтение файла | Не нужна |
| Запись/редактирование | Нужна |
| Создание нового файла | Нужна |
| Удаление файла | Нужна |

---

### Правило координации

**Правило:** При запуске агент обязан обновить `agents-status.json`.

**При старте:**
```json
{
  "status": "running",
  "started_at": "...",
  "task": "...",
  "working_on": [...]
}
```

**При завершении:**
```json
{
  "status": "completed",
  "finished_at": "..."
}
```

**Проверка перед работой:**

1. Прочитать `agents-status.json`
2. Проверить, работает ли другой агент с нужными файлами
3. Если файл заблокирован:
   - **Блокировка в текущем воркфлоу** (например, README.md при создании instruction) → ждать
   - **Есть другие задачи вне воркфлоу** → переключиться на другую задачу, вернуться позже
   - **Других задач нет** → сообщить "Ожидаю снятия блокировки..." и ждать

---

### Правило именования

**Правило:** Временные файлы агентов используют префикс `{agent-name}-`.

**Примеры:**

| Агент | Временный файл |
|-------|----------------|
| `refactoring-agent` | `refactoring-agent-cache.json` |
| `code-reviewer` | `code-reviewer-notes.json` |
| `todo-finder` | `todo-finder-results.json` |

## 5. Жизненный цикл агента

Часть жизненного цикла автоматизирована хуками `SubagentStart` и `SubagentStop`.

### Запуск агента

| Шаг | Кто выполняет | Действие |
|-----|---------------|----------|
| 1 | Хук `SubagentStart` | Добавить запись в `agents-status.json` (status: running) |
| 2 | Агент | Прочитать `locks.json` — проверить блокировки |
| 3 | Агент | Создать `agent-{name}-operation.json` с метаданными |

### Работа агента

| Шаг | Кто выполняет | Действие |
|-----|---------------|----------|
| 1 | Агент | Перед записью в файл — проверить `locks.json`, добавить блокировку |
| 2 | Агент | После каждой операции — записать в `agent-{name}-operation.json` |
| 3 | Агент | После записи — удалить блокировку из `locks.json` |

### Завершение агента

| Шаг | Кто выполняет | Действие |
|-----|---------------|----------|
| 1 | Агент | Обновить `agent-{name}-operation.json` (finished_at) |
| 2 | Хук `SubagentStop` | Обновить `agents-status.json` (status: completed/failed) |
| 3 | Хук `SubagentStop` | Удалить оставшиеся блокировки агента из `locks.json` (fallback) |

### Хуки

Хуки настраиваются в `.claude/settings.json`:

```json
{
  "hooks": {
    "SubagentStart": [{"matcher": "*", "hooks": [{"type": "command", "command": "python ./.claude/hooks/record-agent-start.py"}]}],
    "SubagentStop": [{"hooks": [{"type": "command", "command": "python ./.claude/hooks/cleanup-agent.py"}]}]
  }
}
```

**Входные данные хуков (stdin JSON):**
- `agent_id` — уникальный ID субагента
- `agent_type` — тип агента (Explore, Bash, кастомный)

**Преимущество:** Гарантированный cleanup — хук обновит статус и удалит "забытые" блокировки, даже если агент упал или прервался.

## 6. Безопасность

### Что НЕ хранить в state

| Запрещено | Альтернатива |
|-----------|--------------|
| API ключи | `.env` файл |
| Пароли | Secure vault |
| Токены доступа | Environment variables |
| Персональные данные | Обезличенные данные |

### Проверка целостности

Перед использованием state файла — проверить его структуру:

```python
def validate_state(data: dict, required_fields: list) -> bool:
    """Проверить наличие обязательных полей."""
    for field in required_fields:
        if field not in data:
            return False
    return True
```

### Восстановление после ошибок

При повреждённом state файле — пересоздать с начальной структурой:

```json
// agents-status.json
{"agents": {}}

// locks.json
{"locks": {}}
```

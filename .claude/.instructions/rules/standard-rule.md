## 1. Назначение

**Rule** — это markdown-файл с инструкциями, которые Claude Code автоматически загружает в контекст сессии.

**Принцип:** Rule = триггер для автоматической загрузки контекста, не дублирует инструкции.

### Когда создавать rule

| Критерий | Описание |
|----------|----------|
| Автоматический триггер | Нужно автоматически загружать инструкцию при открытии файлов определённого типа |
| Глобальное правило | Правило применяется ко всем файлам проекта всегда |
| Краткая ссылка | Нужна краткая ссылка на скилл или инструкцию |

**Примеры:**
- При работе с `.instructions/**` → автоматически загружать список скиллов `/instruction-*`
- Всегда проверять SSOT-ссылки перед выполнением → глобальный rule
- При работе с `.claude/rules/**` → автоматически загружать список скиллов `/rule-*`

### Когда НЕ создавать rule

| Ситуация | Решение |
|----------|---------|
| Полная инструкция | Создать инструкцию в `{область}/.instructions/` |
| Разовое правило | Добавить в существующий rule или инструкцию |
| Дублирование SSOT | Ссылаться на существующую инструкцию, не дублировать |

## 2. Расположение

**Путь:** `/.claude/rules/{name}.md`

**Структура папки:**
- Плоская структура — подпапки НЕ поддерживаются
- Все rules в одной директории `/.claude/rules/`

### Правила именования

| Правило | Пример ✅ | Пример ❌ |
|---------|----------|----------|
| Kebab-case | `core.md` | `CORE.md` |
| Латиница | `instructions.md` | `инструкции.md` |
| Нижний регистр | `instructions.md` | `Instructions.md` |
| Описательное | `skills.md` | `s.md` |

**Формат:** `{область}-{назначение}.md` или `{назначение}.md`

**Примеры:**
- `core.md` — глобальные критичные правила
- `instructions.md` — правила для работы с инструкциями, скриптами, скиллами, rules

## 3. Frontmatter

> **SSOT:** [standard-frontmatter.md](/.structure/.instructions/standard-frontmatter.md)

**Формат:**
```yaml
---
description: Краткое описание rule
standard: .claude/.instructions/rules/standard-rule.md
index: .claude/.instructions/rules/README.md
paths:          # Опционально — для условных rules
  - "паттерн/**"
---
```

### Поля frontmatter

| Поле | Обязательное | Использует Claude Code | Описание |
|------|:------------:|:----------------------:|----------|
| `description` | ✅ | ❌ | Краткое описание назначения rule |
| `standard` | ✅ | ❌ | Ссылка на стандарт: `standard-rule.md` |
| `index` | ✅ | ❌ | Ссылка на README области rules |
| `paths` | ❌ | ✅ | Glob-паттерны для условного применения |

**Важно:**
- Claude Code поддерживает **только поле `paths`** — остальные игнорируются
- Поля `description`, `standard`, `index` — расширение для согласованности с остальными инструкциями

## 4. Структура

**Минимальная структура rule:**

```markdown
---
description: {описание}
standard: .claude/.instructions/rules/standard-rule.md
index: .claude/.instructions/rules/README.md
paths:          # Только для условных rules
  - "{паттерн}"
---

{Содержимое}
```

**Требования:**
- Frontmatter в YAML-формате
- Краткое содержимое (максимум 50 строк)

## 5. Типы применения

### Глобальный rule

**Признак:** Frontmatter БЕЗ поля `paths`

**Поведение:** Загружается при **старте каждой сессии** Claude Code

**Применение:**
- Правила, применяемые всегда
- Проверки, выполняемые на каждом запросе
- Общие триггеры для скиллов

**Пример:**
```yaml
---
description: Обязательное чтение SSOT-файлов
standard: .claude/.instructions/rules/standard-rule.md
index: .claude/.instructions/rules/README.md
---
```

### Условный rule

**Признак:** Frontmatter С полем `paths`

**Поведение:** Загружается при работе с файлами, соответствующими glob-паттернам

**Применение:**
- Правила для конкретной области файлов
- Автоматические триггеры скиллов при открытии определённых файлов
- Специфичные инструкции для типов файлов

**Пример:**
```yaml
---
description: Правила работы с rules
standard: .claude/.instructions/rules/standard-rule.md
index: .claude/.instructions/rules/README.md
paths:
  - ".claude/rules/**"
---
```

### Стандартные paths по областям

| Область | Paths |
|---------|-------|
| Структура проекта | `.structure/**` |
| Скиллы | `.claude/skills/**` |
| Инструкции | `**/.instructions/**` |
| Скрипты | `**/.scripts/**` |
| Rules | `.claude/rules/**` |
| Глобальный | ❌ нет (без paths) |

## 6. Содержимое rule

**Максимальный размер:** 50 строк

**Причина:** Rules загружаются в контекст каждой сессии — чем короче, тем лучше.

**Структура:**
1. Краткое описание (1-2 предложения)
2. Ссылки на скиллы или инструкции

### Формат ссылок на скиллы

**Формат:** Список с описанием каждого скилла

**Пример:**
```markdown
При работе с rules ОБЯЗАТЕЛЬНО использовать скиллы (запрещено создавать вручную):
  - `/rule-create` - создание rule
  - `/rule-modify` - изменение rule
  - `/rule-validate` - валидация rule
```

### Best practices

| Принцип | Описание |
|---------|----------|
| Не дублировать инструкции | Ссылаться на SSOT, не копировать содержимое |
| Только триггеры | Rule = когда делать, инструкция = как делать |
| Краткость | До 50 строк, максимум информативности |
| Актуальность | Регулярно проверять и обновлять |

## 7. Конфликты paths

**Поведение Claude Code:** Применяет **все** rules, чьи `paths` совпали с текущим файлом.

**Это фича, не баг:**
- Позволяет комбинировать правила из разных областей
- Файл может одновременно соответствовать нескольким rules

**Проблема:** Если rules противоречат друг другу — ошибка проектирования.

**Решение:**
1. Проверить конфликт с помощью `list-rules.py` (показывает warnings при пересечении)
2. Изменить содержимое rules, чтобы они дополняли друг друга, а не противоречили
3. Если нужна эксклюзивность — уточнить `paths` паттерны

**Пример конфликта:**
```yaml
# core.md
# Глобальный - применяется всегда

# instructions.md
paths:
  - "**/.instructions/**"
  - ".claude/rules/**"
```

При работе с `.claude/.instructions/rules/` — применяются ОБА rules.

## 8. Примеры

### Пример глобального rule

**Файл:** `/.claude/rules/core.md`

```markdown
---
description: Критичные глобальные правила выполнения
standard: .claude/.instructions/rules/standard-rule.md
index: .claude/.instructions/rules/README.md
---

**SSOT:** Если в документе есть ссылка `**SSOT:** [файл](путь)` — прочитать файл перед выполнением. Нельзя работать по памяти или предположениям.

**Блокирующее подтверждение:** Если в воркфлоу скилла есть шаг с "подтверждением" — остановиться и ждать ответа пользователя.
```

### Пример условного rule

**Файл:** `/.claude/rules/instructions.md`

```markdown
---
description: Правила работы с инструкциями, скриптами, скиллами, rules, ссылками и структурой
standard: .claude/.instructions/rules/standard-rule.md
index: .claude/.instructions/rules/README.md
paths:
  - "**/.instructions/**"
  - "**/.scripts/**"
  - ".claude/rules/**"
  - ".claude/skills/**"
  - ".structure/**"
---

При работе с инструкциями ОБЯЗАТЕЛЬНО использовать скиллы (запрещено создавать вручную):
  - `/instruction-create` - создание инструкции
  - `/instruction-modify` - изменение инструкции
  - `/instruction-validate` - валидация инструкции

При работе со скриптами ОБЯЗАТЕЛЬНО использовать скиллы (запрещено создавать вручную):
  - `/script-create` - создание скрипта
  - `/script-modify` - изменение скрипта
  - `/script-validate` - валидация скрипта

При работе со скиллами ОБЯЗАТЕЛЬНО использовать скиллы (запрещено создавать вручную):
  - `/skill-create` - создание скилла
  - `/skill-modify` - изменение скилла
  - `/skill-validate` - валидация скилла

При работе с правилами ОБЯЗАТЕЛЬНО использовать скиллы (запрещено создавать вручную):
  - `/rule-create` - создание rule
  - `/rule-modify` - изменение rule
  - `/rule-validate` - валидация rule

При работе со ссылками ОБЯЗАТЕЛЬНО использовать скиллы:
  - `/links-validate` - валидация ссылок

При работе со структурой ОБЯЗАТЕЛЬНО использовать скиллы (запрещено создавать вручную):
  - `/structure-create` - создание папки
  - `/structure-modify` - изменение папки
  - `/structure-validate` - валидация структуры
```

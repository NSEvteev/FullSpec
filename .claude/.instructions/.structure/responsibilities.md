## Формат описания

Каждая папка описывается по шаблону:

```markdown
# /{папка}/ (папка)

Зона ответственности: <максимально подробное описание — что хранится, зачем, какие типы файлов, границы ответственности>

Регламентирует:
- {файл1.md} — <описание>
- {файл2.md} — <описание>
- {подпапка/} — <описание>

SSOT: [Инструкции по работе с папкой](ссылка на /.claude/.instructions/{папка}/)
```

### Элементы описания

| Элемент | Назначение |
|---------|------------|
| **Заголовок** | Путь к папке |
| **Зона ответственности** | Максимально подробное описание |
| **Регламентирует** | Список содержимого с описаниями |
| **SSOT** | Ссылка на инструкции (правила работы) |

### Принципы

1. **Зона ответственности** — чем подробнее, тем лучше. Описывает: что хранится, зачем, какие типы файлов подходят, границы (что НЕ должно быть здесь)
2. **Регламентирует** — перечисляет ключевые файлы/папки с описаниями
3. **SSOT** — всегда ссылается на зеркальную папку в инструкциях

## Алгоритм размещения файла

```
1. Новый файл X нужно разместить

2. Читаю /.structure/responsibilities.md

3. Ищу папку, чья "Зона ответственности" подходит для X

4. Если нашёл → размещаю там

5. Если не нашёл:
   → Создаю новую папку
   → Описываю её по формату
   → Добавляю в /.structure/responsibilities.md
   → Создаю зеркальную папку в /.claude/.instructions/
```

## Примеры

### Пример: /.structure/

```markdown
# /.structure/ (папка)

Зона ответственности: фактическая структура проекта — единый источник правды (SSOT).
Хранит декларативное описание того, ЧТО есть в проекте: дерево папок, маппинг между
папками и инструкциями, зоны ответственности каждой папки. Только факты, без правил
"как делать". Изменяется при добавлении/удалении папок проекта. НЕ содержит: правила
работы (они в инструкциях), код, документацию.

Регламентирует:
- project.md — полное дерево папок проекта с комментариями
- mapping.md — таблица соответствия: папка проекта ↔ папка инструкций
- responsibilities.md — зоны ответственности всех папок проекта

SSOT: [Инструкции](/.claude/.instructions/.structure/)
```

### Пример: /src/

```markdown
# /src/ (папка)

Зона ответственности: исходный код всех сервисов проекта. Каждый сервис — отдельная
папка с полной структурой: backend (handlers, routes, services, models), frontend
(если есть), database (схемы, миграции), tests (unit, integration), docs (API, guides).
Сервис автономен и может быть развёрнут независимо. НЕ содержит: общий код между
сервисами (→ /shared/), инфраструктуру (→ /platform/), спецификации (→ /specs/).

Регламентирует:
- {service}/ — папка сервиса со всем необходимым для его работы
- {service}/README.md — описание сервиса, как запустить
- {service}/Makefile — команды: dev, test, build
- {service}/backend/ — серверный код
- {service}/tests/ — тесты сервиса (unit, integration)

SSOT: [Инструкции](/.claude/.instructions/src/)
```

### Пример: /specs/

```markdown
# /specs/ (папка)

Зона ответственности: спецификации, архитектурные решения, планы развития проекта.
Всё, что описывает "почему" и "как будет" в отличие от кода, который описывает
"как есть". Включает: обсуждения фич до реализации (discussions), анализ влияния
изменений (impact), архитектурные решения по сервисам (ADR), планы реализации (plans),
глоссарий терминов проекта. НЕ содержит: документацию кода (→ */docs/), README
файлы (→ в соответствующих папках).

Регламентирует:
- discussions/ — обсуждения фич до принятия решения (DISC-*.md)
- impact/ — анализ влияния изменений (IMPACT-*.md)
- services/{service}/adr/ — архитектурные решения сервиса (ADR-*.md)
- services/{service}/plans/ — планы реализации (PLAN-*.md)
- glossary.md — глоссарий терминов проекта

SSOT: [Инструкции](/.claude/.instructions/specs/)
```

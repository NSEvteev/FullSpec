# Примеры использования

---

## Куда положить файл?

### Пример 1: Новый handler для API

**Задача:** Создать handler для регистрации пользователей в сервисе auth.

**Анализ:**
- Относится к одному сервису? ДА → `/src/{service}/`
- Это код бэкенда? ДА → `backend/`
- Это версионированный API? ДА → `v*/`

**Решение:** `/src/auth/backend/v1/handlers/register.ts`

---

### Пример 2: Общая библиотека валидации

**Задача:** Создать библиотеку валидации email, которая используется несколькими сервисами.

**Анализ:**
- Относится к одному сервису? НЕТ
- Относится к нескольким сервисам? ДА → `/shared/`
- Это библиотека? ДА → `libs/`

**Решение:** `/shared/libs/validation/email.ts`

---

### Пример 3: E2E тест регистрации

**Задача:** Создать end-to-end тест полного флоу регистрации пользователя.

**Анализ:**
- Это тест сервиса? НЕТ (тестирует весь флоу)
- Это тест всей системы? ДА → `/tests/`
- Это E2E? ДА → `e2e/`

**Решение:** `/tests/e2e/registration-flow.test.ts`

---

### Пример 4: Docker-compose для разработки

**Задача:** Создать docker-compose файл для локальной разработки.

**Анализ:**
- Относится к инфраструктуре? ДА → `/platform/`
- Это Docker? ДА → `docker/`

**Решение:** `/platform/docker/docker-compose.dev.yml`

---

### Пример 5: ADR для выбора базы данных

**Задача:** Задокументировать решение использовать PostgreSQL для сервиса users.

**Анализ:**
- Это спецификация/архитектура? ДА → `/specs/`
- Это ADR? ДА → `services/{service}/adr/`

**Решение:** `/specs/services/users/adr/001-postgresql.md`

---

### Пример 6: OpenAPI контракт

**Задача:** Создать OpenAPI спецификацию для сервиса auth.

**Анализ:**
- Относится к нескольким сервисам? ДА (контракт = соглашение между сервисами)
- Это API контракт? ДА → `contracts/`
- Это REST? ДА → `openapi/`

**Решение:** `/shared/contracts/openapi/auth.yaml`

---

## Где написать инструкцию?

### Пример 1: Правила версионирования API

**Задача:** Написать инструкцию о том, как версионировать API в проекте.

**Анализ:**
- Про разработку кода сервиса? ДА → `src/`
- Про API дизайн? ДА → `api/`

**Решение:** `/.claude/.instructions/src/api/versioning.md`

---

- Про Git? ДА → `git/`

**Решение:** `/.claude/.instructions/.github/git/commits.md`

---

## Быстрая шпаргалка

### Файлы проекта

| Тип файла | Куда |
|-----------|------|
| Код сервиса | `/src/{service}/backend/` |
| Тесты сервиса | `/src/{service}/tests/` |
| Документация сервиса | `/src/{service}/docs/` |
| Общая библиотека | `/shared/libs/` |
| API контракт | `/shared/contracts/` |
| E2E тест | `/tests/e2e/` |
| Нагрузочный тест | `/tests/load/` |
| Docker конфиг | `/platform/docker/` |
| Runbook | `/platform/runbooks/` |
| ADR | `/specs/services/{service}/adr/` |
| GitHub workflow | `/.github/workflows/` |
| Скилл | `/.claude/skills/{name}/SKILL.md` |

### Инструкции

| Тема | Куда |
|------|------|
| API дизайн | `src/api/` |
| База данных | `src/database/` |
| Безопасность | `src/security/` |
| Unit тесты | `src/testing/unit/` |
| E2E тесты | `tests/e2e/` |
| Docker | `platform/docker/` |
| Мониторинг | `platform/observability/` |
| Git правила | `.github/git/` |
| Правила скиллов | `.claude/skills/` |
| Правила инструкций | `.claude/.instructions/` |

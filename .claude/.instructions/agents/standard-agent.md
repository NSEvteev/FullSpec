## 1. Назначение агентов

Агенты — автономные помощники для выполнения специализированных задач через инструмент `Task`.

**Когда создавать агента:**

| Критерий | Порог |
|----------|-------|
| Повторяемость задачи | Выполнялась или будет выполняться > 3 раз |
| Специализация | Узкая область (explore, validate, build) |
| Автономность | Задача выполнима без участия пользователя после запуска (может запросить подтверждение 1-2 раза для критичных операций) |

**Когда НЕ создавать агента:**

- Разовая задача → выполнить напрямую
- Требует постоянного диалога → использовать основной контекст
- Простая команда → использовать Bash

## 2. Типы агентов

| Тип | Назначение | Инструменты | Пример |
|-----|------------|-------------|--------|
| **Explore** | Исследование кодовой базы | Glob, Grep, Read | Поиск паттернов, анализ структуры |
| **Bash** | Выполнение команд | Bash | Git операции, сборка, тесты |
| **Plan** | Планирование реализации | Read, Glob, Grep | Архитектурные решения |
| **general-purpose** | Универсальные задачи | Все инструменты | Сложные многошаговые задачи |

**Встроенные агенты Claude Code:**

```
Explore      — быстрый поиск по кодовой базе
Bash         — выполнение shell-команд
Plan         — планирование реализации
general-purpose — универсальный агент
```

**Кастомные агенты:**

Создаются в `/.claude/agents/` для специфичных задач проекта.

## 3. Формат конфигурации

> **Официальный формат Claude Code:** Markdown с YAML frontmatter.
>
> **SSOT базовых полей frontmatter:** [standard-frontmatter.md](/.structure/.instructions/standard-frontmatter.md)

### Структура файла

```
/.claude/agents/{agent-name}/
└── AGENT.md          # Конфигурация агента
```

**Файл AGENT.md:**

```markdown
---
name: agent-name
description: Краткое описание — когда Claude должен использовать агента
standard: .claude/.instructions/agents/standard-agent.md
standard-version: v1.1
index: .claude/.instructions/agents/README.md
model: sonnet
tools: Read, Grep, Glob, Bash, Edit, Write, AskUserQuestion
disallowedTools: WebSearch, WebFetch
permissionMode: default
skills:
  - instruction-create
  - instruction-validate
---

## Роль
{Кто агент и какова его специализация}

## Задача
{Что агент должен делать}

## Ограничения
{Чего агент НЕ должен делать}

## Формат вывода
{Как агент должен оформлять результат}
```

### Обязательные поля

| Поле | Тип | Описание |
|------|-----|----------|
| `name` | string | Уникальное имя агента (kebab-case, латиница) |
| `description` | string | Когда Claude должен использовать агента |
| `standard` | string | Путь к стандарту агентов (`.claude/.instructions/agents/standard-agent.md`) |
| `standard-version` | string | Версия стандарта агентов (`v1.2`) |
| `index` | string | Путь к индексу агентов (`.claude/.instructions/agents/README.md`) |

### Официальные опциональные поля

| Поле | Тип | По умолчанию | Описание |
|------|-----|--------------|----------|
| `model` | enum | `inherit` | Модель: `haiku`, `sonnet`, `opus`, `inherit` |
| `tools` | string/list | Все | Разрешённые инструменты (через запятую или список) |
| `disallowedTools` | string/list | — | Запрещённые инструменты |
| `permissionMode` | enum | `default` | Режим разрешений (см. ниже) |
| `skills` | list | — | Скиллы для предзагрузки в контекст |
| `hooks` | object | — | Lifecycle hooks (см. ниже) |

### Кастомные поля проекта

| Поле | Тип | Описание |
|------|-----|----------|
| `type` | enum | Категория: `explore`, `bash`, `plan`, `general-purpose` |
| `max_turns` | int | Максимум API-вызовов (предохранитель от зацикливания) |
| `version` | string | Версия агента (формат: `vX.Y`, например `v1.0`) |

### Поле `model`

| Значение | Описание |
|----------|----------|
| `haiku` | Минимальная модель для простых задач (поиск, сканирование) — быстрая и дешёвая |
| `sonnet` | Сбалансированная — анализ, код-ревью |
| `opus` | Максимум возможностей — сложные задачи |
| `inherit` | Использует модель основной сессии (по умолчанию) |

### Поле `tools`

Доступные инструменты Claude Code:

```
Read              # Чтение файлов
Write             # Запись файлов
Edit              # Редактирование файлов
Bash              # Выполнение Bash команд
Glob              # Поиск файлов по паттернам
Grep              # Поиск в файлах
WebSearch         # Веб-поиск
WebFetch          # Получение содержимого URL
AskUserQuestion   # Вопросы к пользователю
```

**Примеры:**

```yaml
# Строка (через запятую)
tools: Read, Grep, Glob, Bash

# Список
tools:
  - Read
  - Grep
  - Glob
```

### Поле `permissionMode`

| Режим | Поведение |
|-------|-----------|
| `default` | Стандартные проверки с запросами к пользователю |
| `acceptEdits` | Автоматически принимать редактирование файлов в области работы агента (не затрагивает критичные файлы: .env, package.json, конфиги) |
| `dontAsk` | Автоматически отклонять запросы разрешений |
| `bypassPermissions` | Пропускать все проверки (⚠️ опасно) |
| `plan` | Режим планирования (только чтение) |

### Поле `skills`

Предзагрузка скиллов в контекст агента:

```yaml
skills:
  - instruction-create
  - instruction-validate
  - principles-validate
```

> **Важно:** Агенты не наследуют скиллы автоматически — нужно указать явно в поле `skills`. Правила из `/.claude/rules/` наследуются всегда.

### Поле `hooks`

Lifecycle hooks — команды, выполняемые на этапах жизненного цикла:

```yaml
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "echo 'Executing Bash command'"
  PostToolUse:
    - matcher: "Edit|Write"
      hooks:
        - type: command
          command: "python .instructions/.scripts/validate-instruction.py"
  Stop:
    - type: command
      command: "echo 'Agent finished'"
```

| Событие | Когда срабатывает |
|---------|-------------------|
| `PreToolUse` | Перед использованием инструмента |
| `PostToolUse` | После использования инструмента |
| `Stop` | Когда агент завершает работу |

### Поле `max_turns` (кастомное)

Ограничивает количество итераций агента (вызовов API). Предохранитель от зацикливания.

| Тип задачи | Рекомендуемое значение |
|------------|------------------------|
| Простой поиск | 5-10 |
| Анализ папки | 10-20 |
| Код-ревью | 15-30 |
| Сложная задача | 30-50 |
| Без ограничения | не указывать (или 100+) |

> **Принцип:** Лучше указать с запасом. Агент остановится раньше, если закончит задачу.

## 4. Правила написания промптов

### Процесс формирования промпта

> **Промпт формируется итеративно с подтверждением пользователя.**

**Шаги (выполняются при создании агента):**

1. **Сбор контекста** — LLM автоматически:
   - Получает список инструкций: `python .instructions/.scripts/list-instructions.py`
   - Анализирует релевантные инструкции для задачи агента
   - Определяет, какие скиллы могут понадобиться

2. **Генерация черновика промпта** — на основе:
   - Запроса пользователя
   - Найденных инструкций и стандартов
   - Рекомендуемых скиллов

3. **Подтверждение пользователем** — AskUserQuestion:
   - "Промпт корректен?" → Да / Нужны правки
   - При правках: уточнить что исправить, вернуться к шагу 2

4. **Выбор скиллов** — если агент general-purpose:
   - LLM предлагает список скиллов на основе задачи
   - Пользователь выбирает, какие разрешить

> **Детали процесса:** см. [create-agent.md](./create-agent.md)

### Структура промпта

```markdown
## Роль
{Кто агент и какова его специализация}

## Задача
{Что агент должен делать}

## Инструкции и SSOT
{Ссылки на инструкции, которым следует агент}

## Скиллы
Используй скиллы из frontmatter вместо ручных операций.

## Ограничения
{Чего агент НЕ должен делать}

## Формат вывода
{Как агент должен оформлять результат}
```

### Принципы

> **Специфичность.** Промпт описывает конкретную задачу, не общие инструкции.

> **Ограничения явные.** Что агент НЕ должен делать — важнее того, что должен.

> **Формат вывода.** Всегда указывать ожидаемый формат результата.

**Хороший промпт:**

```yaml
prompt: |
  ## Роль
  Ты — агент для поиска TODO-комментариев в коде.

  ## Задача
  Найти все TODO, FIXME, HACK комментарии в указанной папке.

  ## Ограничения
  - НЕ модифицировать файлы
  - НЕ предлагать исправления
  - Только поиск и отчёт

  ## Формат вывода
  Таблица: | Файл | Строка | Комментарий |
```

### Антипаттерны

| Антипаттерн | Пример ❌ | Исправление ✅ |
|-------------|----------|----------------|
| Слишком общий | "Помоги с кодом" | "Найди неиспользуемые импорты в src/" |
| Без ограничений | "Исправь баги" | "Найди баги, НЕ исправляй автоматически" |
| Без формата | "Проанализируй" | "Выведи список в формате markdown" |

## 5. Контекст для агента

### Обязательные данные

Для качественной работы агенту необходим полный контекст задачи.

**Чек-лист данных для передачи агенту:**

| Данные | Описание | Пример |
|--------|----------|--------|
| **Цель** | Что именно нужно сделать | "Найти все неиспользуемые функции" |
| **Область** | Где искать/работать | "src/services/", "*.py файлы" |
| **Критерии** | Как определить успех | "Функция не вызывается нигде" |
| **Исключения** | Что игнорировать | "tests/, __init__.py" |
| **Формат результата** | Как оформить вывод | "Таблица markdown" |
| **Ограничения** | Чего НЕ делать | "НЕ удалять, только отчёт" |

**Полный шаблон промпта для агента:**

```markdown
## Роль
{Кто агент: специализация, область экспертизы}

Пример: "Ты — агент для поиска технического долга в Python-коде."

## Задача
{Что конкретно должен сделать агент}

Пример: "Найти все TODO, FIXME, HACK комментарии и сгруппировать по приоритету."

## Область работы
{Где искать/работать, какие файлы/папки}

- Путь: {src/, tests/, конкретная папка}
- Типы файлов: {*.py, *.ts, *.md}
- Исключения: {node_modules/, .git/, __pycache__/}

## Инструкции и SSOT
{Ссылки на инструкции проекта, которым агент должен следовать}

Релевантные инструкции:
- /.instructions/standard-principles.md — принципы кода
- /.structure/.instructions/standard-readme.md — формат README

## Скиллы
Используй скиллы из frontmatter вместо ручных операций.

## Правила проекта
{Агент следует тем же правилам, что и основной LLM}

Правила из /.claude/rules/:
- core.md — приоритет скиллов, SSOT, шаблоны

## Критерии успеха
{Как определить, что задача выполнена правильно}

- {критерий 1}
- {критерий 2}

## Ограничения
{Чего агент НЕ должен делать — критически важно}

- НЕ {ограничение 1}
- НЕ {ограничение 2}
- ВСЕГДА {обязательное поведение}

## Формат вывода
{Точный формат результата — таблица, список, markdown}

Пример структуры вывода:
| Колонка 1 | Колонка 2 | Колонка 3 |
|-----------|-----------|-----------|
| ...       | ...       | ...       |

Итого: {что должно быть в конце}
```

### Работа с файлами состояния

Агенты могут создавать файлы для сохранения промежуточных и итоговых результатов:

| Папка | Назначение | Пример |
|-------|------------|--------|
| `/.claude/state/` | Временные файлы между сессиями | `analysis-cache.json` |
| `/.claude/drafts/` | Черновики для больших задач | `refactoring-plan.md` |

**Когда создавать драфт:**

- Задача требует > 10 итераций
- Результат нужен основному LLM после завершения
- Анализ слишком большой для вывода в чат

**Пример использования:**

```yaml
prompt: |
  ## Сохранение результатов

  Для больших результатов:
  1. Создать драфт в /.claude/drafts/{task-name}.md
  2. Записать полный анализ в драфт
  3. Вернуть краткое резюме + путь к драфту

  Данные между сессиями (кеш, промежуточные результаты) сохраняй в /.claude/state/
```

> **Принцип:** Драфт остаётся после завершения работы агента — основной LLM может его использовать.

### Координация между агентами

> **SSOT:** [standard-state.md](../state/standard-state.md) — формат файлов состояния
> **Rule:** [core.md](../../rules/core.md) — секция "State"

#### Автоматизация через хуки

Часть координации автоматизирована хуками Claude Code:

| Хук | Действие |
|-----|----------|
| `SubagentStart` | Регистрирует агента в `agents-status.json` (status: running) |
| `SubagentStop` | Срабатывает при любом завершении агента (нормальное, ошибка, прерывание) — обновляет статус и выполняет cleanup блокировок |

**Что остаётся на агенте:**
- Проверка и создание блокировок в `locks.json`
- Снятие блокировок **сразу после операции** (не в конце работы!)
- Ведение лога в `agent-{name}-operation.json`

#### Файлы состояния

| Файл | Назначение |
|------|------------|
| `agents-status.json` | Статус всех агентов |
| `locks.json` | Блокировки файлов |
| `agent-{name}-operation.json` | Лог операций агента |

**Формат agents-status.json:**

```json
{
  "agents": {
    "todo-finder": {
      "status": "running",
      "agent_type": "explore",
      "started_at": "2026-01-15T10:30:00Z",
      "finished_at": null,
      "task": "Поиск TODO в src/",
      "working_on": ["src/"]
    }
  }
}
```

**Формат locks.json:**

```json
{
  "locks": {
    "/src/auth/handlers.py": {
      "agent": "refactoring-agent",
      "locked_at": "2026-01-15T10:30:00Z",
      "reason": "Рефакторинг"
    }
  }
}
```

#### Цикл блокировки файла

```
1. Прочитать locks.json
2. Если файл заблокирован другим агентом:
   - Подождать 5 сек, повторить чтение (до 6 раз)
   - Если после 6 проверок всё ещё занят → вернуть ошибку
3. Если файл свободен → добавить свою блокировку в locks.json
4. Выполнить операцию (edit/write/delete)
5. Снять блокировку (ВСЕГДА, даже при ошибке!)
```

> **Атомарность операций:** В рамках одного агента операция "read-modify-write" для state-файлов считается атомарной (Claude Code гарантирует последовательное выполнение).

**Алгоритм ожидания блокировки:**

| Шаг | Действие |
|-----|----------|
| 1 | Проверить `locks.json` |
| 2 | Если файл свободен → добавить блокировку, продолжить |
| 3 | Если занят → подождать 5 сек, повторить (до 6 раз) |
| 4 | Если после 6 проверок занят → вернуть ошибку: "Файл {путь} заблокирован агентом {имя} более 30 сек" |

#### Правила координации

| Правило | Описание |
|---------|----------|
| Блокировать перед записью | Проверить и создать блокировку в locks.json |
| Снимать сразу | Удалить блокировку сразу после операции (даже при ошибке) |
| Вести лог | Записывать операции в agent-{name}-operation.json |
| Избегать конфликтов | Ждать если файл заблокирован (6 проверок × 5 сек = 30 сек) |

#### Что логировать в operations[]

| Действие | action | Что логировать |
|----------|--------|----------------|
| Чтение | `read` | Файлы в области работы агента (анализируемые, редактируемые) |
| Запись | `write` | Созданные или перезаписанные файлы |
| Редактирование | `edit` | Изменённые файлы |
| Создание | `create` | Новые файлы |

**Не логировать:** технические чтения (стандарты, state-файлы, rules, SSOT-ссылки для контекста) — это внутренний контекст агента.

#### Обновление finished_at

- Обновить `finished_at` когда агент **завершает работу** (возвращает управление основному LLM с отчётом о выполнении)
- Если пользователь запросил дополнительные правки В ТОЙ ЖЕ сессии агента → НЕ обновлять `finished_at`, продолжить работу

### Скиллы и правила

> **Агенты наследуют правила основного LLM.** Все rules из `/.claude/rules/` применяются к агентам автоматически.

**Автоматический сбор контекста при создании:**

При создании агента LLM ОБЯЗАН:

1. Выполнить поиск инструкций:
   ```bash
   python .instructions/.scripts/list-instructions.py
   ```

2. Проанализировать какие инструкции релевантны задаче агента

3. Включить ссылки на SSOT в промпт агента

4. Предложить список скиллов для разрешения (AskUserQuestion с multiSelect)

**Доступ к скиллам:**

Агенты типа `general-purpose` могут использовать скиллы. Указывайте их во **frontmatter** (поле `skills`), а не в промпте:

```yaml
---
skills:
  - structure-validate
  - links-validate
  - principles-validate
---

## Скиллы

**Если skills указаны во frontmatter:**
> Используй скиллы из frontmatter вместо ручных операций.

**Если skills НЕ указаны:**
> *Скиллы не используются этим агентом.*
```

**Когда назначать скиллы:**

| Задача агента | Рекомендуемые скиллы |
|---------------|---------------------|
| Валидация проекта | /structure-validate, /links-validate |
| Рефакторинг | /principles-validate |
| Создание файлов | /instruction-create, /skill-create, /structure-create |
| Изменение структуры | /structure-modify, /instruction-modify |

## 8. Обработка ошибок

Агент должен явно сообщать об ошибках и проблемах.

### Типы ошибок

| Тип | Действие агента | Пример |
|-----|-----------------|--------|
| **Блокирующая** | Остановиться, вернуть ошибку | Файл не найден, нет доступа |
| **Восстановимая** | Пропустить, продолжить | Один файл повреждён из 100 |
| **Предупреждение** | Записать, продолжить | Подозрительный паттерн |

### Формат сообщения об ошибке

```markdown
## Ошибка выполнения

**Тип:** {блокирующая/восстановимая}
**Шаг:** {на каком шаге произошла}
**Причина:** {описание проблемы}

**Контекст:**
- Файл: {путь}
- Операция: {что пытался сделать}

**Рекомендации:**
1. {что можно сделать}
```

### В промпте агента

```yaml
prompt: |
  ## Обработка ошибок

  При ошибках:
  1. Записать в state/{agent-name}-errors.json
  2. Продолжить если ошибка восстановимая
  3. Остановиться если блокирующая
  4. Вернуть отчёт с секцией "Ошибки"

  Формат записи:
  {
    "timestamp": "...",
    "type": "blocking|recoverable|warning",
    "message": "...",
    "context": {...}
  }
```

## 9. Отмена и прерывание

### Когда агент должен остановиться

| Ситуация | Действие |
|----------|----------|
| Критическая ошибка | Немедленная остановка + отчёт |
| Превышен max_turns | Остановка + промежуточный результат |
| Конфликт с другим агентом | Ожидание + уведомление (см. ниже) |
| Задача невыполнима | Остановка + объяснение |

### Конфликт с другим агентом

При обнаружении конфликта (файл занят другим агентом):

1. **Записать конфликт** в `state/conflicts.json`
2. **Уведомить основной LLM** через возврат сообщения
3. **Ожидать освобождения** — проверять `state/agents-status.json` каждые 5 секунд
4. **Проверить изменения** — прочитать файл после разблокировки, сравнить с планом своих изменений
5. **Проверить совместимость** — убедиться что свои изменения не перезатрут чужие
6. **При конфликте изменений** — спросить основной LLM (он спросит пользователя)
7. **Продолжить** только после разрешения конфликта

```yaml
prompt: |
  ## При конфликте с другим агентом

  1. НЕ останавливаться сразу
  2. Записать в state/conflicts.json:
     {
       "agent": "{my-name}",
       "blocked_by": "{other-agent}",
       "file": "{путь}",
       "status": "waiting",
       "waiting_since": "timestamp",
       "resumed_at": null
     }
  3. Вернуть сообщение: "Ожидаю освобождения {файл} от {agent}"
  4. Проверять state/agents-status.json каждые 5 секунд
  5. Когда other-agent завершил:
     a. Прочитать изменения другого агента
     b. Сравнить с запланированными своими изменениями
     c. Если конфликт:
        - Вернуть сообщение основному LLM с описанием конфликта
        - Ждать решения пользователя
     d. Если нет конфликта — продолжить работу
  6. При продолжении работы обновить state/conflicts.json:
     - status: "resumed"
     - resumed_at: "timestamp"
  7. Вернуть сообщение: "Продолжаю работу после ожидания {время}"
  8. Таймаут ожидания: 2 минуты → остановка с ошибкой
```

**Формат сообщения о конфликте изменений:**

```markdown
## Конфликт изменений

**Файл:** {путь}

**Изменения агента {other-agent}:**
- {что изменил}

**Мои запланированные изменения:**
- {что хотел изменить}

**Конфликт:**
- {описание конфликта}

**Варианты:**
1. Применить мои изменения (перезаписать чужие)
2. Отменить мои изменения (оставить чужие)
3. Объединить вручную
```

### Graceful shutdown

```yaml
prompt: |
  ## При прерывании

  Если задача прервана:
  1. Сохранить текущий прогресс в state/{agent-name}-checkpoint.json
  2. Записать что уже сделано
  3. Записать что осталось
  4. Вернуть частичный результат

  Формат checkpoint:
  {
    "completed": ["шаг1", "шаг2"],
    "pending": ["шаг3"],
    "partial_result": {...}
  }
```

### Триггеры graceful shutdown

| Триггер | Описание |
|---------|----------|
| Превышен `max_turns` | Агент достиг лимита итераций |
| Критическая ошибка | Файл не найден, нет доступа, недопустимая операция |
| Задача невыполнима | Невозможно достичь цели с доступными инструментами |

## 10. Безопасность

### Использование официальных полей

> **Рекомендация:** Используйте `tools`, `disallowedTools` и `permissionMode` для контроля доступа.

**Пример безопасной конфигурации:**

```yaml
---
name: code-analyzer
tools: Read, Grep, Glob
disallowedTools: Write, Edit, Bash
permissionMode: plan
---
```

### Запрещённые операции

| Операция | Почему запрещено |
|----------|------------------|
| `rm -rf /` | Уничтожение системы |
| Удаление .git | Потеря истории |
| Модификация .env | Утечка секретов |
| Push без подтверждения | Необратимое действие |
| Удаление node_modules в продакшене | Поломка сервиса |

### Рекомендации по `disallowedTools`

| Тип агента | Рекомендуемые disallowedTools |
|------------|-------------------------------|
| Анализ (explore, plan) | `Write, Edit, Bash` |
| Код-ревью | `Write, Edit` |
| Создание контента | `Bash, WebFetch` |

### Рекомендации по `permissionMode`

| Режим | Когда использовать |
|-------|-------------------|
| `plan` | Агенты только для анализа (explore, plan) |
| `default` | Агенты с записью — требуют подтверждения |
| `acceptEdits` | Доверенные агенты для автоматических правок |

> **⚠️ Не используйте `bypassPermissions`** для кастомных агентов.

### Обязательные ограничения в промпте

Даже с правильной конфигурацией, добавьте ограничения в промпт:

```markdown
## Ограничения безопасности

ЗАПРЕЩЕНО без явного разрешения пользователя:
- Удалять файлы (только предлагать к удалению)
- Модифицировать .env, credentials, secrets
- Выполнять git push, git reset --hard
- Устанавливать зависимости

ОБЯЗАТЕЛЬНО:
- Спрашивать подтверждение перед деструктивными операциями
- Логировать все изменения
```

### Уровни доступа (концептуально)

| Уровень | tools | disallowedTools | permissionMode |
|---------|-------|-----------------|----------------|
| **read-only** | `Read, Grep, Glob` | `Write, Edit, Bash` | `plan` |
| **safe-write** | `Read, Grep, Glob, Edit, Write` | `Bash` | `default` |
| **full** | Все | — | `default` |

> **Агенты НЕ должны иметь full-доступа.** Деструктивные операции — только основной LLM.

## 11. Версионирование агентов

### Поле `version`

Каждый агент должен иметь версию в формате `vX.Y`:

```yaml
---
name: todo-finder
version: v1.0
---
```

**Правила:**

| Действие | Изменение версии |
|----------|------------------|
| Мелкие исправления (опечатки, уточнения) | `v1.0` → `v1.1` |
| Новая секция или значительное изменение | `v1.1` → `v1.2` |
| Кардинальное переосмысление (breaking changes) | `v1.x` → `v2.0` |

### CHANGELOG.md

Каждый агент должен иметь файл истории изменений:

```
/.claude/agents/{agent-name}/
├── AGENT.md
└── CHANGELOG.md
```

**Формат CHANGELOG.md:**

```markdown
# CHANGELOG — {Agent Name}

## vX.Y (YYYY-MM-DD)

### Добавлено
- {что добавлено}

### Изменено
- {что изменено}

### Исправлено
- {что исправлено}
```

**Пример:**

```markdown
# CHANGELOG — Todo Finder

## v1.1 (2026-02-01)

### Добавлено
- Секция "Обработка ошибок"

### Изменено
- Уточнён формат вывода

---

## v1.0 (2026-01-30)

### Добавлено
- Первая версия агента
```

## 12. Удаление файлов агентом

Агенты НЕ ДОЛЖНЫ удалять файлы напрямую. Вместо удаления:

1. **Пометить файл** — переименовать с префиксом `_old_`
2. **Записать в лог** — action `mark_for_deletion` в operations[]
3. **Записать в отчёт** — список файлов для удаления
4. **Основной LLM** — удалит после подтверждения пользователя

```markdown
## Удаление файлов

ЗАПРЕЩЕНО: rm, удаление файлов напрямую

Если нужно удалить файл:
1. Переименовать: file.py → _old_file.py
2. Записать в лог операций: action "mark_for_deletion"
3. В отчёте указать: "Файлы помечены на удаление: ..."

Основной LLM после ревью:
- Удалит _old_* файлы
- Или восстановит их
```

## Правила

### Правило fail-fast

Проверки выполняются ДО начала создания. При ошибке — немедленная остановка.

### Правило валидации структуры

Все обязательные элементы SKILL.md должны присутствовать. При отклонении — предупреждение, не блокировка.

### Правило ревью

Ревью проводится ПЕРЕД завершением создания. Пользователь выбирает, какие улучшения применить.

### Правило приоритетов

- 🔴 Критично — блокирует качество, применяется всегда
- 🟡 Рекомендуется — улучшает качество, применяется в --auto
- 🟢 По желанию — косметика, по выбору пользователя

## Fail-fast проверки

> **КРИТИЧНО:** Эти проверки выполняются ДО начала создания скилла. При любой ошибке — немедленная остановка.

### Список проверок

| Проверка | Команда | При ошибке |
|----------|---------|------------|
| Папка `.claude/skills/` существует | `ls .claude/skills/` | Создать или сообщить об ошибке |
| Git status чистый в `.claude/` | `git status .claude/ --porcelain` | Предупредить о несохранённых изменениях |
| Скилл с таким именем не существует | `ls .claude/skills/{название}/` | Спросить: перезаписать / отменить |
| Формат названия корректен | Regex: `^[a-z]+(-[a-z]+)+$` | Показать примеры правильного формата |

### Формат вывода при ошибке

```
❌ Fail-fast проверка не пройдена

Проверка: {название проверки}
Проблема: {описание}

Решение: {что сделать}

Продолжить невозможно. Исправьте проблему и попробуйте снова.
```

### Формат вывода при успехе

```
✅ Все предварительные проверки пройдены

- Папка .claude/skills/ существует
- Git status чистый
- Скилл {название} не существует
- Формат названия корректен

Продолжаю создание...
```

## Валидация структуры SKILL.md

### Обязательные элементы

**Frontmatter:**
- [ ] `name` — имя скилла
- [ ] `description` — описание (1 строка)
- [ ] `allowed-tools` — список инструментов
- [ ] `category` — категория из [rules.md](./rules.md#категории-скиллов)
- [ ] `triggers.commands` — команды для вызова
- [ ] `triggers.phrases` — фразы (ru/en)

**Структура:**
- [ ] Заголовок `# {Название}`
- [ ] Связанные скиллы (после описания)
- [ ] Оглавление со ссылками на 4 раздела: Формат вызова, Воркфлоу, Чек-лист, Примеры
- [ ] Раздел "Формат вызова"
- [ ] Раздел "Воркфлоу" с шагами
- [ ] Раздел "Чек-лист"
- [ ] Раздел "Примеры" со ссылкой на SSOT

**Воркфлоу (правила шагов):**
- [ ] Каждый шаг содержит ТОЛЬКО название и `> **SSOT:** [ссылка]`
- [ ] ⛔ ЗАПРЕЩЕНО добавлять текст после SSOT-ссылки (списки, комментарии, пояснения)
- [ ] Вся логика, алгоритмы, примеры — в SSOT-инструкциях, не в скилле
- [ ] Если шаг вызывает другой скилл: `### Шаг N: {Название} → /{skill-name}`

**Вёрстка:**
- [ ] Frontmatter — корректный YAML
- [ ] Ссылки в оглавлении соответствуют заголовкам
- [ ] Нет вложенных блоков кода (``` внутри ```)
- [ ] Таблицы форматированы корректно
- [ ] Все блоки кода закрыты
- [ ] Все файлы и папки репозитория оформлены как ссылки

### При отклонениях

- Предупредить пользователя
- Не блокировать создание
- Предложить исправить

## Ревью скилла

### Что проверять

1. **Полнота воркфлоу:**
   - Все ли шаги описаны достаточно подробно?
   - Нет ли пропущенных шагов между действиями?
   - Понятно ли, что делать на каждом шаге?

2. **Качество правил:**
   - Достаточно ли правил для корректной работы?
   - Нет ли противоречий между правилами?
   - Покрыты ли граничные случаи?

3. **Примеры:**
   - Достаточно ли примеров для понимания?
   - Покрывают ли примеры типичные сценарии?
   - Есть ли примеры обработки ошибок?

4. **Интеграция:**
   - Корректно ли описаны связи с другими скиллами?
   - Нужны ли дополнительные зависимости?

### Формат вывода ревью

```
📋 Ревью скилла {название}

✅ Сильные стороны:
- {что хорошо}

💡 Предложения по улучшению (по приоритету):

🔴 Критично (блокирует качество):
1. {проблема функциональности или безопасности}
2. {отсутствует обязательный раздел}

🟡 Рекомендуется (улучшает качество):
3. {улучшение UX}
4. {добавить пример}

🟢 По желанию (nice to have):
5. {косметическое улучшение}

❓ Вопросы для уточнения:
- {вопрос, если есть}

Применить предложения? (все / критичные / выборочно / пропустить)
```

### Приоритеты предложений

| Приоритет | Когда использовать | Пример |
|-----------|-------------------|--------|
| 🔴 Критично | Влияет на работоспособность, безопасность, нарушает стандарты | Отсутствует обработка ошибок |
| 🟡 Рекомендуется | Улучшает качество, UX, поддерживаемость | Добавить пример обработки ошибки |
| 🟢 По желанию | Косметические улучшения, стиль | Переформулировать описание |

**В режиме --auto:** применяются только 🔴 Критично и 🟡 Рекомендуется.

## Чек-лист создания скилла

Полный чек-лист для `/skill-create`:

### Подготовка
- [ ] Прочитал SSOT инструкции (rules.md, workflow.md, parameters.md)
- [ ] Проверил пересечения с существующими скиллами
- [ ] Fail-fast проверки пройдены

### Создание
- [ ] Получил название, проверил формат `{объект}-{действие}`
- [ ] Определил категорию
- [ ] Сгенерировал метаданные, получил подтверждение
- [ ] Создал папку `.claude/skills/{название}/`
- [ ] Создал `SKILL.md` по [шаблону](/.claude/.instructions/.claude/skills/template-skill.md)
- [ ] Заполнил воркфлоу с конкретными шагами
- [ ] Описал правила скилла
- [ ] Добавил примеры использования (минимум 1)
- [ ] Создал чек-лист

### Валидация
- [ ] Провёл ревью, вывел предложения по улучшению
- [ ] Применил выбранные пользователем улучшения
- [ ] Проверил структуру SKILL.md
- [ ] Проверил вёрстку (YAML, ссылки, блоки кода)
- [ ] Все файлы и папки репозитория оформлены как ссылки

### Интеграция
- [ ] Прочитал [agents.md](/.claude/agents/README.md), предложил назначить агентам
- [ ] Вызвал [/links-update](/.claude/skills/links-update/SKILL.md) для синхронизации
- [ ] Вызвал [/skill-update](/.claude/skills/skill-update/SKILL.md) для интеграции

### Завершение
- [ ] Проверил выполнение всех пунктов чек-листа
- [ ] Вывел полный отчёт пользователю

## Примеры

### Пример 1: Fail-fast успех

```
✅ Все предварительные проверки пройдены

- Папка .claude/skills/ существует
- Git status чистый
- Скилл my-skill не существует
- Формат названия корректен

Продолжаю создание...
```

### Пример 2: Fail-fast ошибка

```
❌ Fail-fast проверка не пройдена

Проверка: Формат названия
Проблема: "mySkill" содержит заглавные буквы

Решение: Используйте kebab-case: my-skill

Продолжить невозможно.
```

### Пример 3: Ревью скилла

```
📋 Ревью скилла agent-create

✅ Сильные стороны:
- Чёткий воркфлоу

🟡 Рекомендуется:
1. Добавить пример ошибки

🟢 По желанию:
2. Расширить описание

Применить? (все / выборочно / пропустить)
```

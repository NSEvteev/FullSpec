## 1. Типы черновиков

## 2. Именование

**Формат:**
```
YYYY-MM-DD-<topic>.md
```

**Правила:**

| Правило | Пример ✅ | Пример ❌ |
|---------|----------|----------|
| Дата в начале | `2024-01-23-auth.md` | `auth-2024-01-23.md` |
| Kebab-case | `api-versioning.md` | `api_versioning.md` |
| Латиница | `refactoring.md` | `рефакторинг.md` |
| Описательное имя | `auth-refactor.md` | `draft1.md` |

### Конфликты имён

Если черновик с таким именем существует:
- Добавить суффикс `-2`, `-3` и т.д.
- Пример: `2024-01-23-auth-refactor-2.md`

**Проверка перед созданием:** если файл существует — использовать суффикс.

## 3. Формат файла

**Frontmatter:**

Frontmatter опционален для черновиков (в отличие от инструкций). Если используется — следовать [standard-frontmatter.md](/.structure/.instructions/standard-frontmatter.md).

**Минимальная структура:**

```markdown
# {Тема}

{Краткое описание — 1-2 предложения}

## Оглавление

- [Контекст](#контекст)
- [Содержание](#содержание)

---

## Контекст

{Почему создан черновик, какая задача решается}

## Содержание

{Основное содержание: план, заметки, результаты исследования}
```

**Дополнительные секции (опционально):**

| Секция | Когда добавлять |
|--------|-----------------|
| `## Решения` | Если есть принятые решения |
| `## Открытые вопросы` | Если есть нерешённые вопросы |
| `## Ссылки` | Если есть связанные документы |

## 4. Жизненный цикл

**Характеристики:**

| Аспект | Значение |
|--------|----------|
| В git | Да (история сохраняется) |
| Привязка | К рабочим процессам |
| Жизненный цикл | Временные (не являются постоянной документацией, но могут храниться как история) |

**После завершения работы:**

| Действие | Когда |
|----------|-------|
| Оставить | Для истории решений |
| Удалить | Информация перенесена в финальные документы |

> **Важно:** "Временные" означает, что черновики НЕ являются SSOT и НЕ заменяют постоянную документацию. Однако они МОГУТ храниться в git как история принятия решений.

### Когда удалять черновик

- Информация полностью перенесена в финальные документы (README, инструкции, specs)
- **ТОЛЬКО пользователь** принимает решение об удалении
- ЗАПРЕЩЕНО удалять черновик автоматически (LLM/агент)

## 5. Правила работы

### Когда создавать черновик

**Создавать:**
- Планирование задачи > 5 шагов
- Сравнение 2+ альтернативных решений
- Исследование архитектуры/API перед реализацией
- Анализ технических ограничений
- Сложные размышления, требующие структурированной фиксации

**НЕ создавать:**
- Простые задачи (1-4 шага)
- Рутинные операции (создание файла, правка README)
- Ответы на вопросы пользователя

### Создание

- Черновики создаются Claude по инициативе (без явного запроса пользователя) согласно критериям выше
- Агенты могут создавать собственные черновики (например, дискуссии между LLM и агентом)
- **Не требуется** формальный процесс создания
- Должны следовать стандарту форматирования

### Self-check после заполнения

После первичного заполнения черновика контентом LLM ОБЯЗАН провести анализ самодостаточности:

> **Вопрос:** "Могу ли я, читая ТОЛЬКО черновик, реализовать изменения по нему, используя ТОЛЬКО написанное в черновике? Если нет — что нужно дописать и как обновить Tasklist, чтобы это стало возможно?"

**Процедура:**
1. Перечитать черновик целиком как "новый читатель"
2. Для каждого TASK в Tasklist проверить: достаточно ли description + ссылка на секцию для автономного выполнения?
3. Если нет — дописать недостающий контекст:
   - Точные цитаты "БЫЛО → СТАЛО" для каждого изменения
   - Имена файлов, секций, строк для поиска
   - Конкретные значения (не "обновить", а "заменить X на Y")
4. Обновить Tasklist если появились новые задачи или изменились зависимости

### Ограничения

| Что МОЖНО | Что НЕЛЬЗЯ |
|-----------|-----------|
| Неформальная структура | Хранить постоянную документацию |
| Исследования и планы | Заменять спецификации |
| Временные записи | Использовать как SSOT |

### Привязка к задачам

Каждый черновик ДОЛЖЕН содержать в секции "Контекст" явную привязку:

```markdown
## Контекст

**Задача:** {описание задачи или ссылка на issue/ticket}
**Почему создан:** {причина создания — 1-2 предложения}
**Связанные файлы:** {пути к файлам, если есть} — опционально
```

Пример:
```markdown
## Контекст

**Задача:** Рефакторинг аутентификации (issue #123)
**Почему создан:** Анализ миграции JWT → OAuth2 перед реализацией
**Связанные файлы:** `/specs/ADR-005-auth.md`
```

## 6. Отличия от specs

| Аспект | drafts | specs |
|--------|--------|-------|
| Назначение | Думать о том, что разрабатывается | Документация разработки проекта |
| Формальность | Минимальная | Строгий стандарт |
| Жизненный цикл | Временные | Постоянные |
| Ценность | История принятия решений | SSOT проекта |

**drafts** — хранилище для размышлений, истории принятия решений и временных заметок. На основе черновиков реализуется задуманное.

**specs** — формальная документация разработки проекта (ADR, планы, архитектура).

## 8. Шаблон черновика

### Пример: План рефакторинга

**Файл:** `2024-01-23-auth-refactor.md`

```markdown
# Рефакторинг аутентификации

Переход с JWT на OAuth2 для унификации авторизации.

## Оглавление

- [Контекст](#контекст)
- [Содержание](#содержание)
- [Решения](#решения)
- [Открытые вопросы](#открытые-вопросы)

---

## Контекст

**Задача:** Унификация авторизации в микросервисах (issue #123)
**Почему создан:** Анализ альтернативных подходов перед реализацией
**Связанные файлы:** `/specs/ADR-005-auth.md`

## Содержание

### Текущая проблема

- JWT токены не поддерживают централизованный отзыв
- Разные микросервисы используют разные библиотеки

### Альтернативы

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| OAuth2 | Централизованный контроль | Сложность настройки |
| Session-based | Простота | Не подходит для API |

## Решения

- Выбран OAuth2 с Keycloak
- Миграция поэтапная (сначала новые сервисы)

## Открытые вопросы

- Как мигрировать существующие токены?
- Нужен ли fallback на JWT?

## Tasklist

Задачи для исполнения через TaskCreate. Порядок строгий — зависимости указаны в blockedBy.

TASK 1: Создать скрипт миграции auth
  description: >
    Драфт: .claude/drafts/2024-01-23-auth-refactor.md (секция "Содержание")
    Скрипт миграции JWT → OAuth2 для существующих пользователей.
  activeForm: Создаю скрипт миграции

TASK 2: Обновить конфигурацию сервисов
  blockedBy: [1]
  description: >
    Драфт: .claude/drafts/2024-01-23-auth-refactor.md (секция "Содержание")
    Переключение auth-провайдера на Keycloak в конфигах dev/staging/prod.
  activeForm: Обновляю конфигурацию сервисов
```

## 9. Tasklist

Секция `## Tasklist` — **обязательная** финальная секция черновика типа "План". Содержит готовые к исполнению задачи для `TaskCreate`.

### Назначение

Tasklist — мост между анализом (содержание черновика) и исполнением (TaskCreate в Claude Code). Каждая задача ссылается на конкретную секцию черновика и содержит достаточно контекста для автономного выполнения.

### Формат

```
## Tasklist

Задачи для исполнения через TaskCreate. Порядок строгий — зависимости указаны в blockedBy.

TASK 1: {title — императив, глагол + объект}
  description: >
    Драфт: {путь к черновику} (секция "{название секции}")
    {Подробное описание: что создать/изменить, какой скилл использовать,
    какие конкретные изменения внести. 2-5 строк.}
  activeForm: {Present Continuous — что делается}

TASK 2: {title}
  blockedBy: [1]
  description: >
    Драфт: {путь к черновику} (секция "{название секции}")
    {Подробное описание.}
  activeForm: {Present Continuous}
```

### Правила

| Правило | Описание |
|---------|----------|
| Нумерация | Последовательная: TASK 1, TASK 2, ... TASK N |
| blockedBy | Список номеров задач-зависимостей. Отсутствует если нет зависимостей |
| description | Обязательная ссылка на секцию черновика + конкретные инструкции |
| activeForm | Present Continuous для отображения в спиннере TaskCreate |
| Скиллы | Если задача выполняется скиллом — указать имя скилла в description |
| Гранулярность | Одна задача = один артефакт или одно логическое изменение |

### Когда обязателен

| Тип черновика | Tasklist |
|---------------|----------|
| План | **Обязательно** |
| Исследование | Опционально (если есть план действий по результатам) |
| Заметка | Не требуется |

## 2. Именование

**Формат:**
```
YYYY-MM-DD-<topic>.md
```

**Правила:**

| Правило | Пример ✅ | Пример ❌ |
|---------|----------|----------|
| Дата в начале | `2024-01-23-auth.md` | `auth-2024-01-23.md` |
| Kebab-case | `api-versioning.md` | `api_versioning.md` |
| Латиница | `refactoring.md` | `рефакторинг.md` |
| Описательное имя | `auth-refactor.md` | `draft1.md` |

## 3. Формат файла

**Минимальная структура:**

```markdown
# {Тема}

{Краткое описание — 1-2 предложения}

## Оглавление

- [Контекст](#контекст)
- [Содержание](#содержание)

---

## Контекст

{Почему создан черновик, какая задача решается}

## Содержание

{Основное содержание: план, заметки, результаты исследования}
```

**Дополнительные секции (опционально):**

| Секция | Когда добавлять |
|--------|-----------------|
| `## Решения` | Если есть принятые решения |
| `## Открытые вопросы` | Если есть нерешённые вопросы |
| `## Ссылки` | Если есть связанные документы |

## 4. Жизненный цикл

**Характеристики:**

| Аспект | Значение |
|--------|----------|
| В git | Да (история сохраняется) |
| Привязка | К рабочим процессам |
| Жизненный цикл | Временные |

**После завершения работы:**

| Действие | Когда |
|----------|-------|
| Оставить | Для истории решений |
| Удалить | Информация перенесена в финальные документы |
| Перенести в дискуссию | Материал готов для `/specs/discussions/` |

> **Важно:** Черновик НЕ становится спецификацией напрямую. Spec создаётся отдельно, черновик может быть источником материала для дискуссии.

**Обновление индекса:**

При добавлении/удалении черновика — обновить таблицу в [drafts/README.md](/.claude/drafts/README.md).

## 5. Правила работы

### Создание

- Черновики создаются Claude **спонтанно** в процессе работы
- Агенты могут создавать собственные драфты (например, дискуссии между LLM и агентом)
- **Не требуется** формальный процесс создания
- Должны следовать стандарту форматирования

### Ограничения

| Что МОЖНО | Что НЕЛЬЗЯ |
|-----------|-----------|
| Неформальная структура | Хранить постоянную документацию |
| Исследования и планы | Заменять спецификации |
| Временные записи | Использовать как SSOT |

### Привязка к задачам

Каждый черновик ДОЛЖЕН быть связан с конкретной задачей или текущей сессией работы.

## 6. Отличия от specs

| Аспект | drafts | specs |
|--------|--------|-------|
| Назначение | Думать о том, что разрабатывается | Документация разработки проекта |
| Формальность | Минимальная | Строгий стандарт |
| Жизненный цикл | Временные | Постоянные |
| Ценность | История принятия решений | SSOT проекта |

**drafts** — хранилище для размышлений, истории принятия решений и временных заметок. На основе drafts реализуется задуманное.

**specs** — формальная документация разработки проекта (ADR, планы, архитектура).

### Связь drafts → specs/discussions

Черновик может быть **источником материала** для дискуссии:
1. В драфте прорабатывается идея или решение
2. Готовый материал переносится в `/specs/discussions/DISC-*.md`
3. Дискуссия проходит формальный процесс принятия решения
4. Драфт остаётся как история размышлений

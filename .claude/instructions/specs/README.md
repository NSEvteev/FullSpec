# 1. Структура /specs/

`/specs/` — единая папка для проектных спецификаций.

**Содержание:** Discussions (исследование идей), Impact (анализ влияния), ADR (архитектурные решения), Plans (планы реализации), Architecture (текущее состояние), Glossary (термины).

```
/specs/
├── discussions/                  # Дискуссии (идеи, исследования)
│   ├── README.md                 # Индекс дискуссий
│   └── 001-{topic}.md
├── impact/                       # Импакт-анализ (какие сервисы затронуты)
│   ├── README.md                 # Индекс импактов
│   └── 001-{topic}.md
├── services/                     # Сервисы
│   └── {service}/
│       ├── README.md             # Описание сервиса
│       ├── architecture.md       # Архитектура сервиса
│       ├── adr/                  # Архитектурные решения
│       │   ├── README.md         # Индекс ADR
│       │   └── 001-{topic}.md
│       └── plans/                # Планы реализации
│           ├── README.md         # Индекс планов
│           └── {topic}-plan.md
└── glossary.md                   # Глоссарий терминов
```

# 2. Статусы

Унифицированная система статусов для всех типов документов в `/specs/`: Discussion, Impact, ADR, Plan.

**Содержание:** все статусы, схема переходов, правила по типам, каскадные проверки, правила REJECTED.

| Статус | Emoji | Значение |
|--------|-------|----------|
| DRAFT | 📝 | Черновик |
| REVIEW | 🔍 | На проверке |
| APPROVED | 🆗 | Одобрено |
| RUNNING | ⏳ | В работе |
| DONE | ✅ | Завершено |
| REJECTED | ❌ | Отклонено |
| SUPERSEDED | 🚫 | Заменено |

**Инструкция:** [statuses.md](./statuses.md)

---

# 3. Workflow

Полный workflow от идеи до реализации: Discussion → Impact → ADR (параллельно) → Plan → Реализация + Документация.

**Содержание:** 6 фаз, точка синхронизации, полная диаграмма переходов.

```
Discussion → Impact → ADR → Plan → Implementation + Docs
    │           │        │       │
    ▼           ▼        ▼       ▼
 исследование  анализ  решение  задачи
```

**Инструкция:** [workflow.md](./workflow.md)

---

# 4. Discussions

Точка входа для новых идей и изменений. Исследование проблемы, сбор информации, принятие решения.

**Содержание:** расположение, формат, статусы, чек-листы переходов (DRAFT→REVIEW→APPROVED→RUNNING→DONE).

**Инструкция:** [discussions.md](./discussions.md)

---

# 5. Impact

Импакт-анализ определяет какие сервисы затронуты изменением. Один Impact может создать несколько ADR.

**Содержание:** расположение, формат, статусы, создание нового сервиса, связь Impact↔ADR, чек-листы переходов.

**Инструкция:** [impact.md](./impact.md)

---

# 6. ADR

Architecture Decision Records — фиксация архитектурных решений для конкретного сервиса. Каждый ADR ссылается на родительский Impact.

**Содержание:** расположение, формат, статусы, проверка бизнес-логики, проверка конфликтов, Breaking Changes, чек-листы переходов.

**Инструкция:** [adr.md](./adr.md)

---

# 7. Plans

Декомпозиция ADR на конкретные задачи. План требует согласования с пользователем перед реализацией.

**Содержание:** статусы планов, согласование с пользователем, автоматическое создание GitHub Issues, версионирование планов (v2, v3), критерий готовности, чек-листы переходов.

**Инструкция:** [plans.md](./plans.md)

---

# 8. Architecture

Живой документ — описание текущего состояния сервиса. Каждый раздел ссылается на ADR, который его ввёл.

**Содержание:** расположение, формат документа, правила ссылок на ADR, workflow обновления при ADR → DONE.

**Особенность:** Без статусов (DRAFT/REVIEW/DONE). Обновляется автоматически при завершении ADR.

**Инструкция:** [architecture.md](./architecture.md)

---

# 9. Glossary

Единое место для терминов проекта. Обеспечивает консистентность терминологии во всех документах.

**Содержание:** структура термина, категории, пороги масштабирования, правила ведения, скиллы (`/glossary-candidates`, `/glossary-review`, `/glossary-link`).

**Особенность:** Без статусов. Управляется отдельными скиллами, не входящими в систему `/spec-*`.

**Инструкция:** [glossary.md](./glossary.md)

# 11. Принятые решения

Ключевые решения по организации `/specs/`.

| Вопрос | Решение |
|--------|---------|
| Название папки спецификаций | `/specs` |
| Где хранить глоссарий | `/specs/glossary.md` |

| Где хранить импакт-анализ | `/specs/impact/` (только общий уровень) |
| Структура | Двухуровневая: общее + по сервисам |
| ADR и архитектура | Только по сервисам |
| Формат названий | Порядковые номера: `001-topic.md` |
| README.md в папках | Индексные файлы с навигацией и статусами |
| ADR → Impact | Обязательная ссылка на родительский Impact |
| Impact → ADR | Один Impact может создать несколько ADR |

# 12. Связи между документами

Граф зависимостей и обязательные ссылки между документами.

**Содержание:** цепочка документов, граф зависимостей, обязательные ссылки, обратные ссылки (backlinks).

### 12.1. Цепочка документов

```
Discussion → Impact → ADR → Plan → Implementation
```

### 12.2. Граф зависимостей

```
┌─────────────────┐
│   Discussion    │
│ 001-auth-flow   │
│ status: APPROVED│
└────────┬────────┘
         │ создаёт
         ▼
┌─────────────────┐
│     Impact      │
│ 001-auth-flow   │
│ status: APPROVED│
└────────┬────────┘
         │ создаёт несколько ADR
         ├──────────────────────────────┐
         ▼                              ▼
┌─────────────────────┐    ┌─────────────────────┐
│ ADR: auth/001-jwt   │    │ ADR: gateway/001-.. │
│ impact: 001-auth    │    │ impact: 001-auth    │
│ status: APPROVED    │    │ status: APPROVED    │
└─────────────────────┘    └─────────────────────┘
```

### 12.3. Обязательные ссылки

| Документ | Обязательно ссылается на |
|----------|--------------------------|
| Impact | Discussion (родительская) |
| ADR | Impact (родительский) |
| Plan | ADR (родительский) |
| Architecture раздел | ADR (который его ввёл) |

### 12.4. Обратные ссылки (Backlinks)

При создании дочернего документа **родительский документ обновляется** со ссылкой на дочерний.

| Действие | Родитель | Что обновить |
|----------|----------|--------------|
| `/spec-create impact <discussion>` | Discussion | Секция "Связанные документы" |
| `/spec-create adr <impact> <service>` | Impact | Таблица "Затронутые сервисы" |
| `/spec-create plan <adr>` | ADR | Секция "План реализации" |

# 13. Связь /specs/ ↔ /doc/

Разделение ответственности между проектной документацией и документацией кода.

**Содержание:** назначение папок, принцип разделения, ссылки между папками, синхронизация.

### 13.1. Назначение папок

| Папка | Назначение | Что хранит | Когда обновляется |
|-------|------------|------------|-------------------|
| `/specs/` | **Проектная документация** | Решения, планы, архитектура | При принятии решений |
| `/doc/` | **Документация кода** | API, компоненты, runbooks | При изменении кода |

### 13.2. Принцип разделения

```
/specs/ — "ПОЧЕМУ и ЧТО решили делать"
  └── Discussion → Impact → ADR → Plan

/doc/ — "КАК это сделано сейчас"
  └── API docs, компоненты, runbooks
```

### 13.3. Структура /doc/

```
/doc/
├── README.md                         # ИНДЕКС: точка входа
├── src/                              # Зеркало /src/
│   └── {service}/
│       ├── README.md                 # ИНДЕКС: ссылки на specs
│       ├── api.md
│       └── runbooks/
├── shared/
├── platform/
└── runbooks/
```

### 13.4. Синхронизация

| Событие | /specs/ | /doc/ |
|---------|---------|-------|
| ADR → DONE | architecture.md обновлён | README/api.md обновлены |
| Рефакторинг кода | Не меняется | Обновляется |
| Новое ADR | Новый документ | Не меняется (пока ADR не DONE) |

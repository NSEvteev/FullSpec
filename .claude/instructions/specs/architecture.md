# Архитектура /specs/services/{service}/architecture.md

Документ архитектуры — описание текущего состояния сервиса. Каждый раздел ссылается на ADR, который его ввёл.

> **⚠️ Без статусов:** Architecture — единый живой документ. Он не имеет статусов DRAFT/REVIEW/DONE. Обновляется автоматически при переходе ADR → DONE. Чек-листы для Architecture не требуются.

## Правила ссылок на ADR

### Каждый раздел ссылается на ADR

```markdown
#### {Компонент}
> Введён в [XXX-...](adr/XXX-....md)
> Обновлён в [YYY-...](adr/YYY-....md)  # если были изменения
```

### Пример раздела архитектуры

```markdown
### Аутентификация

#### JWT токены
> Введён в [002-jwt-tokens](adr/002-jwt-tokens.md)

- **Access Token:** 1 час, подписывается HS256
- **Refresh Token:** 7 дней, хранится в Redis
- **Ротация:** При каждом refresh выдаётся новый refresh token

#### Rate Limiting
> Введён в [003-rate-limiting](adr/003-rate-limiting.md)

- 100 запросов/минуту для авторизованных
- 20 запросов/минуту для анонимных
```

## Обновление architecture.md

При завершении ADR (→ DONE):
1. Обновить соответствующий раздел
2. Добавить ссылку на ADR
3. Добавить запись в "История изменений"

### Workflow обновления

```
ADR → ✅ DONE
    │
    ▼
Обновить architecture.md:
    │
    ├── Найти затронутый раздел
    │   └── Если раздела нет → создать
    │
    ├── Добавить/обновить описание
    │   └── Добавить ссылку "> Введён в [ADR-...]" или "> Обновлён в [ADR-...]"
    │
    └── Добавить запись в "История изменений"
        └── | YYYY-MM-DD | [ADR-...](adr/...) | Краткое описание |
```

### Формат истории изменений

```markdown
## История изменений

| Дата | ADR | Изменение |
|------|-----|-----------|
| 2025-01-21 | [002-jwt-tokens](adr/002-jwt-tokens.md) | Добавлена JWT аутентификация |
| 2025-01-25 | [003-rate-limiting](adr/003-rate-limiting.md) | Добавлен rate limiting |
```

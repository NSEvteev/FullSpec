# Формат ошибок

Единый формат ответов об ошибках для всех API сервисов.

## Структура ответа

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Ошибка валидации входных данных",
    "details": {
      "fields": {
        "email": "Некорректный формат email",
        "age": "Значение должно быть положительным числом"
      }
    },
    "request_id": "req_abc123def456"
  }
}
```

## Поля ошибки

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| `code` | string | да | Машиночитаемый код ошибки (UPPER_SNAKE_CASE) |
| `message` | string | да | Человекочитаемое описание ошибки |
| `details` | object | нет | Дополнительная информация об ошибке |
| `request_id` | string | да | Уникальный идентификатор запроса для трейсинга |

## Коды ошибок

### Клиентские ошибки (4xx)

| HTTP | Код | Описание |
|------|-----|----------|
| 400 | `VALIDATION_ERROR` | Ошибка валидации входных данных |
| 400 | `INVALID_REQUEST` | Некорректный формат запроса |
| 400 | `MISSING_PARAMETER` | Отсутствует обязательный параметр |
| 401 | `UNAUTHORIZED` | Требуется аутентификация |
| 401 | `INVALID_TOKEN` | Невалидный или истёкший токен |
| 403 | `FORBIDDEN` | Нет прав на выполнение операции |
| 403 | `RATE_LIMIT_EXCEEDED` | Превышен лимит запросов |
| 404 | `NOT_FOUND` | Ресурс не найден |
| 409 | `CONFLICT` | Конфликт с текущим состоянием ресурса |
| 409 | `ALREADY_EXISTS` | Ресурс уже существует |
| 422 | `UNPROCESSABLE_ENTITY` | Семантическая ошибка в данных |

### Серверные ошибки (5xx)

| HTTP | Код | Описание |
|------|-----|----------|
| 500 | `INTERNAL_ERROR` | Внутренняя ошибка сервера |
| 502 | `BAD_GATEWAY` | Ошибка upstream-сервиса |
| 503 | `SERVICE_UNAVAILABLE` | Сервис временно недоступен |
| 504 | `GATEWAY_TIMEOUT` | Таймаут upstream-сервиса |

## Примеры

### Ошибка валидации

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Ошибка валидации входных данных",
    "details": {
      "fields": {
        "email": "Некорректный формат email",
        "password": "Минимальная длина — 8 символов"
      }
    },
    "request_id": "req_7f3a9b2c"
  }
}
```

### Ресурс не найден

```json
{
  "error": {
    "code": "NOT_FOUND",
    "message": "Пользователь не найден",
    "details": {
      "resource": "user",
      "id": "usr_12345"
    },
    "request_id": "req_8d4e1f6a"
  }
}
```

### Ошибка аутентификации

```json
{
  "error": {
    "code": "INVALID_TOKEN",
    "message": "Токен доступа истёк",
    "details": {
      "expired_at": "2026-01-20T12:00:00Z"
    },
    "request_id": "req_2c5b8a3d"
  }
}
```

### Внутренняя ошибка

```json
{
  "error": {
    "code": "INTERNAL_ERROR",
    "message": "Произошла внутренняя ошибка",
    "request_id": "req_9e7f4c1b"
  }
}
```

## Правила формирования

### Код ошибки (code)

- Формат: `UPPER_SNAKE_CASE`
- Уникальный в рамках проекта
- Не содержит HTTP-статус в названии
- Описывает причину, а не следствие

### Сообщение (message)

- На русском языке (или локализуемое)
- Понятно конечному пользователю
- Не содержит технических деталей
- Не раскрывает внутреннюю реализацию

### Детали (details)

- Содержит машиночитаемую информацию
- Для валидации — объект `fields` с ошибками по полям
- Для ресурсов — `resource` и `id`
- Для лимитов — `retry_after` в секундах

### Request ID (request_id)

- Формат: `req_` + 8-12 символов hex
- Генерируется на входе в систему
- Передаётся через все сервисы
- Логируется в каждом сервисе

## Антипаттерны

```json
// ❌ Неправильно: массив вместо объекта
{
  "errors": ["email invalid", "password too short"]
}

// ❌ Неправильно: HTTP статус в коде
{
  "error": {
    "code": "404_NOT_FOUND"
  }
}

// ❌ Неправильно: технические детали в message
{
  "error": {
    "message": "SQLException: connection refused to db-master-01"
  }
}

// ❌ Неправильно: отсутствует request_id
{
  "error": {
    "code": "INTERNAL_ERROR",
    "message": "Ошибка"
  }
}
```

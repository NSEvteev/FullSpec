### Проблема 1.1: Неопределённое "на будущее" (P1)

**Где:** Секция "2. Принципы программирования" (строка 115)

**Цитата:**
> | **YAGNI** | Не добавлять функционал "на будущее" |

**Как может понять LLM:**
- Интерпретация A: Не добавлять код для функционала, который не требуется в текущей задаче
- Интерпретация B: Не добавлять функционал, который будет использоваться через месяц
- Интерпретация C: Не добавлять расширяемость через аргументы (например, `--format json` для будущего API)

**Рекомендация:**
> YAGNI — реализовывать только то, что нужно для текущей задачи. Не добавлять опциональные аргументы, абстракции или функции без конкретного применения прямо сейчас.

**Критичность:** Высокая (P1) — влияет на решения при создании скриптов.

## 2. Риск неправильной интерпретации LLM

### Проблема 2.1: Противоречие в использовании ERROR_CODES (P1)

**Где:** Секция "6. Структура кода" → "ERROR_CODES для валидаторов" (строка 306-357)

**Суть проблемы:** Документ утверждает, что ERROR_CODES — только для `validate-*.py`, но таблица на строке 323 содержит коды для **всех** типов проверок:

```
| Префикс | Область | Пример |
|---------|---------|--------|
| `S0xx` | Структура скриптов | `S001: Отсутствует shebang` |
| `P0xx` | Принципы программирования | `P001: Переусложнённый код` |
| `I0xx` | Инструкции | `I001: Неверный frontmatter` |
| `K0xx` | Скиллы | `K001: Отсутствует SSOT-ссылка` |
```

**Как может понять LLM:**
- Интерпретация A: Префиксы `I0xx`, `K0xx`, `T0xx` используются в скриптах `validate-instruction.py`, `validate-skill.py`, `validate-structure.py` (корректно)
- Интерпретация B: Префиксы универсальны для ВСЕХ скриптов проекта, даже не-валидаторов (ошибка)

**Рекомендация:**
> Добавить после таблицы префиксов:
>
> **Важно:** Таблица префиксов показывает области проверки для **всех** валидаторов проекта. Префикс выбирается в зависимости от того, ЧТО проверяет скрипт `validate-*.py`:
> - `S0xx` — используется в `validate-script.py`
> - `P0xx` — используется в `validate-principles.py`
> - `I0xx` — используется в `validate-instruction.py`
> - И так далее.

**Критичность:** Высокая (P1) — риск создания ERROR_CODES в неподходящих скриптах.

## 4. Полнота описания объекта

**Почему важно:** LLM может не понимать, что создание скрипта — это не только написание кода, но и:
1. Добавление в инструкцию (секция "Скрипты")
2. Создание связанного скилла
3. Обновление README.md области
4. Добавление в validation-script.md для проверки

**Стоит ли добавлять:** Да (P1)

**Что даст:** Комплексное понимание создания скрипта как части экосистемы, а не изолированного файла.

**Рекомендация:**
> Добавить секцию "10. Жизненный цикл скрипта":
>
> ```markdown
> ## 10. Жизненный цикл скрипта
>
> | Этап | Действия |

> | **Изменение** | 1. Обновить код → 2. Обновить docstring → 3. Обновить инструкции → 4. Проверить совместимость → 5. Валидация |
> | **Удаление** | 1. Найти зависимости → 2. Обновить инструкции → 3. Удалить из секций "Скрипты" → 4. Удалить файл |

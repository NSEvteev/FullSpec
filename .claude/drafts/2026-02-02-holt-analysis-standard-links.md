## 1. Двойственность формулировок

### Проблема 1.1: Неопределённый критерий выбора пути

**Где:** Секция "2. Абсолютные vs относительные" (строки 71-96)

**Цитата:**
> ```
> Та же папка         → ./файл.md
> Соседняя папка      → ../папка/файл.md
> Далёкая папка       → /абсолютный/путь.md
> ```

**Как может понять LLM:**
- Интерпретация A: "Далёкая" = не соседняя (через 2+ уровня)
- Интерпретация B: "Далёкая" = в другой ветке структуры (например, из `/src` в `/.claude`)
- Интерпретация C: "Далёкая" = субъективная оценка расстояния

**Рекомендация:**
> Заменить формулировку на конкретный критерий:
> ```
> Та же папка         → ./файл.md
> Родитель (../папка) → ../папка/файл.md
> 2+ уровня вверх     → /абсолютный/путь.md
> Другая ветка        → /абсолютный/путь.md
> ```

**Приоритет:** P1 (критично)

**Что даст:** Устранит субъективность при выборе типа пути. LLM получит чёткие численные критерии.

### Проблема 1.2: Неясность формата "допустим trailing slash"

**Где:** Секция "3. Ссылки на папки" (строки 100-124)

**Цитата:**
> **Предпочтительно:** ссылка на README.md папки.
>
> ### Альтернативный формат
>
> ```markdown
> [Название](путь/к/папке/)
> ```
>
> Допустим trailing slash `/` в конце пути.

**Как может понять LLM:**
- Интерпретация A: "Допустим" = можно использовать, если README.md отсутствует
- Интерпретация B: "Допустим" = разрешён, но всегда предпочитать README.md
- Интерпретация C: "Допустим" = валиден в любом случае (равнозначен README.md)

**Рекомендация:**
> **Предпочтительно:** ссылка на README.md папки. Если README.md отсутствует или недоступен — использовать trailing slash `/`.

**Приоритет:** P2 (важно)

**Что даст:** Устранит неопределённость условия применения альтернативного формата.

### Проблема 1.3: Размытость термина "без trailing slash"

**Где:** Секция "1. Типы ссылок" → Правила для внешних ссылок (строка 66)

**Цитата:**
> **Правила:**
> - Предпочитать HTTPS
> - Без trailing slash
> - Ссылаться на конкретную страницу

**Как может понять LLM:**
- Интерпретация A: Без trailing slash = `https://example.com` (корень)
- Интерпретация B: Без trailing slash = `https://example.com/docs` (путь без `/` в конце)
- Интерпретация C: Без trailing slash = правило применяется только к URL без пути

**Рекомендация:**
> **Правила:**
> - Предпочитать HTTPS
> - Без trailing slash в конце URL (пример: `https://example.com/docs`, НЕ `https://example.com/docs/`)
> - Ссылаться на конкретную страницу (не корень домена)

**Приоритет:** P2 (важно)

**Что даст:** Устранит неопределённость правила для внешних ссылок.

## 2. Риск неправильной интерпретации

### Проблема 2.1: Неявная инструкция по выбору абсолютных путей

**Где:** Секция "2. Абсолютные vs относительные" → Правила выбора (строки 78-84)

**Цитата:**
> ```
> Та же папка         → ./файл.md
> Соседняя папка      → ../папка/файл.md
> Далёкая папка       → /абсолютный/путь.md
> ```

**Как может понять LLM неправильно:**
LLM может создать относительный путь с длинной цепочкой `../../../`, игнорируя правило "далёкая папка".

**Рекомендация:**
> Добавить явный пример ошибки в секцию:
> ```markdown
> ### Неправильно (цепочка > 2 уровней):
> [файл](../../../другая-папка/файл.md)
>
> ### Правильно:
> [файл](/другая-папка/файл.md)
> ```

**Приоритет:** P1 (критично)

**Что даст:** Покажет границу применения правила. Предотвратит создание сложных относительных путей.

### Проблема 2.2: Противоречие в формате ссылок на папки

**Где:** Секция "3. Ссылки на папки" (строки 100-124)

**Сейчас:**
- Строка 105-108: "**Предпочтительно:** ссылка на README.md папки"
- Строка 110-116: "Допустим trailing slash `/` в конце пути"
- Таблица примеров (строки 119-124): оба формата показаны как равнозначные

**Как может понять LLM неправильно:**
Таблица примеров не отражает приоритет. LLM может использовать trailing slash как основной формат.

**Рекомендация:**
> Изменить таблицу примеров:
> ```markdown
> ### Примеры
>
> ```markdown
> <!-- Предпочтительно -->
> [skills/README.md](./skills/README.md)
> [/src/](/src/README.md)
>
> <!-- Допустимо (если README.md недоступен) -->
> [skills/](./skills/)
> [/src/](/src/)
> ```
> ```

**Приоритет:** P1 (критично)

**Что даст:** Устранит противоречие между текстом и примерами. Закрепит приоритет форматов.

## 4. Полнота описания объекта

### 4.1. Не хватает: Правила для URL-encoded символов в якорях

**Почему важно:** Якоря с кириллицей или спецсимволами могут требовать URL-encoding в зависимости от рендерера Markdown (GitHub, GitLab, локальный парсер).

**Стоит ли добавлять:** Опционально

**Что даст:** Устранит неопределённость для граничных случаев (например, заголовок "API (v2)" → якорь `api-v2` или `api-%28v2%29`?).

**Пример добавления:**
> ### URL-encoding якорей
>
> По умолчанию якоря используются без encoding. Если рендерер требует encoding — замена выполняется автоматически парсером Markdown.
>
> **Пример:**
> - Заголовок: `Шаги работы`
> - Якорь в ссылке: `#шаги-работы` (без encoding)
> - Реальный URL: `#%D1%88%D0%B0%D0%B3%D0%B8-%D1%80%D0%B0%D0%B1%D0%BE%D1%82%D1%8B` (encoding выполняется браузером)

---

### 4.2. Не хватает: Правила для ссылок на конкретные строки кода

**Почему важно:** В GitHub/GitLab можно ссылаться на файл с якорем на строку (например, `file.py#L42-L45`). Непонятно, допустимо ли это в проекте.

**Стоит ли добавлять:** Да

**Что даст:** Покроет сценарий ссылок на код. Устранит неопределённость.

**Пример добавления:**
> ### Ссылки на строки кода
>
> Для файлов с кодом допустимы якоря на строки (GitHub/GitLab формат):
>
> ```markdown
> [Функция validate](./validator.py#L42)
> [Блок обработки ошибок](./validator.py#L100-L115)
> ```
>
> **Правила:**
> - Якорь формата `#LN` (одна строка) или `#LN-LM` (диапазон)
> - Применимо только для файлов кода (не для .md)

---

### 4.3. Не хватает: Правила для ссылок в таблицах и блоках кода

**Почему важно:** Ссылки в блоках кода (например, в примерах) не должны валидироваться, но ссылки в таблицах — должны. Непонятно, где граница.

**Стоит ли добавлять:** Да

**Что даст:** Закроет граничный случай для валидации. Предотвратит ложные срабатывания.

**Пример добавления:**
> ### Ссылки в блоках кода
>
> Ссылки внутри блоков кода (` ``` `) НЕ валидируются — это примеры.
>
> Ссылки в таблицах, инлайн-коде (`` ` ``) и цитатах — валидируются как обычные ссылки.

## 5. Слепые зоны автора

### Зона 5.1: Нет правил для ссылок между разными репозиториями (monorepo)

**Суть:** Если проект — monorepo или имеет зависимости от других репозиториев, непонятно:
- Как ссылаться на файлы в другом репозитории?
- Какой формат использовать (GitHub URL? Относительный путь из корня monorepo?)

**Почему важно:** При росте проекта может появиться необходимость ссылок между репозиториями. Отсутствие правил приведёт к хаотичным форматам.

**Расширение качества:** Умеренное

**Рекомендация:** Добавить сейчас (предотвратит будущие проблемы)

**Пример добавления:**
> ### Ссылки между репозиториями
>
> Если проект — monorepo или имеет зависимости:
>
> **В одном monorepo:**
> - Использовать абсолютные пути от корня monorepo: `/packages/api/README.md`
>
> **Между репозиториями:**
> - Использовать GitHub URL с привязкой к версии:
>   ```markdown
>   [API Docs](https://github.com/org/repo/blob/v1.2.3/docs/api.md)
>   ```
> - НЕ использовать `main`/`master` (ссылка может стать битой при изменениях)

### Зона 5.2: Нет обработки устаревших/битых ссылок (lifecycle)

**Суть:** Документ описывает формат ссылок, но не отвечает на вопросы:
- Как помечать устаревшие ссылки?
- Как обрабатывать ссылки на удалённые/переименованные файлы?
- Нужен ли changelog для миграции ссылок?

**Почему важно:** При рефакторинге структуры могут возникнуть сотни битых ссылок. Без правил lifecycle ссылки станут техдолгом.

**Расширение качества:** Значительное

**Рекомендация:** Добавить сейчас

**Пример добавления:**
> ### Lifecycle ссылок
>
> #### Переименование/перемещение файлов
>
> При переименовании файла:
> 1. Обновить ВСЕ ссылки на файл (использовать `validation-links.md` → скрипт `update-skill-refs.py`)
> 2. Если файл — SSOT для скилла → обновить frontmatter скилла
>
> #### Удаление файлов
>
> Перед удалением файла:
> 1. Проверить входящие ссылки: `grep -r "путь/к/файлу.md" .`
> 2. Удалить ИЛИ перенаправить ссылки на альтернативный документ
> 3. Если перенаправление невозможно — указать в commit message список битых ссылок
>
> #### Устаревшие ссылки
>
> Если документ устарел, но не удалён:
> - Добавить в начало документа:
>   ```markdown
>   > ⚠️ **DEPRECATED:** Документ устарел. См. [новый документ](путь).
>   ```

### Зона 5.3: Нет правил для ссылок в автоматически генерируемых файлах

**Суть:** Если документация генерируется автоматически (например, из кода или из шаблонов), непонятно:
- Применяются ли к ней те же правила?
- Как валидировать автогенерируемые ссылки?
- Нужно ли коммитить автогенерируемые файлы?

**Почему важно:** Автогенерируемые файлы могут создавать ссылки в неправильном формате, но их нельзя исправить вручную (перезапишется при следующей генерации).

**Расширение качества:** Умеренное

**Рекомендация:** Отложить (добавить при появлении первого генератора)

**Пример добавления:**
> ### Ссылки в автогенерируемых файлах
>
> Файлы с пометкой в frontmatter:
> ```yaml
> autogenerated: true
> generator: {script/tool}
> ```
>
> **Правила:**
> - Применяются ВСЕ правила стандарта ссылок
> - Генератор ДОЛЖЕН создавать корректные ссылки
> - Валидация выполняется после генерации
> - Если ссылки битые — исправить генератор, не файл

### Зона 5.4: Нет интеграции со схемой структуры проекта

**Суть:** Стандарт ссылок не отвечает на вопросы:
- Какие папки/файлы ДОЛЖНЫ быть связаны ссылками (граф зависимостей)?
- Есть ли "обязательные" ссылки (например, каждая инструкция ДОЛЖНА ссылаться на standard)?
- Как проверить полноту графа ссылок?

**Почему важно:** Отсутствие интеграции со структурой может привести к "островкам" документов без входящих ссылок (dead docs).

**Расширение качества:** Значительное

**Рекомендация:** Добавить сейчас (критично для масштабирования)

**Пример добавления:**
> ### Граф зависимостей ссылок
>
> #### Обязательные ссылки
>
> | Тип документа | Обязательные ссылки |
> |---------------|---------------------|
> | Инструкция (standard-) | → standard-instruction.md (frontmatter) |
> | Инструкция (create-/modify-/validation-) | → связанный standard-*.md (frontmatter) |
> | Скилл | → SSOT-инструкция (frontmatter и текст) |
> | README папки | → README родителя (секция "Полезные ссылки") |
>
> #### Проверка графа
>
> Скрипт `check-link-graph.py` проверяет:
> - Каждый документ имеет минимум 1 входящую ссылку (кроме корневых)
> - Нет циклических зависимостей в SSOT-цепочках
> - Все обязательные ссылки присутствуют

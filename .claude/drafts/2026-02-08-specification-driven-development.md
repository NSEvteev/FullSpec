## Контекст

**Задача:** Переход к Specification-Driven Development — пользователь указывает желания, LLM собирает архитектуру, ADR, проводит аналитику, планирует Issues, Milestones, пишет требования к тестам и задачи на разработку.

**Почему создан:** Сравнение 4 внешних подходов с текущей системой specs/ для выбора лучших решений.

**Связанные файлы:** `/specs/`, `/specs/.instructions/`

## Содержание

### 1. Обзор изученных подходов

#### 1.1. Наш текущий подход (specs/)

**Воркфлоу:** Discussion → Impact → ADR → Plan → Implementation

**Артефакты:**

| Тип | Расположение | Назначение |
|-----|-------------|------------|
| Discussion | `specs/discussions/disc-NNNN-topic.md` | Исследование проблемы |
| Impact | `specs/impact/impact-NNNN-topic.md` | Анализ влияния на сервисы |
| ADR | `specs/services/{svc}/adr/adr-NNNN-topic.md` | Архитектурное решение |
| Plan | `specs/services/{svc}/plan-dev/plan-dev-NNNN-topic.md` | Декомпозиция на задачи |
| Architecture | `specs/services/{svc}/architecture.md` | Живой документ архитектуры |
| Glossary | `specs/glossary.md` | Терминология |

**Особенности:**
- 7-статусная система (DRAFT → REVIEW → APPROVED → RUNNING → DONE / REJECTED / SUPERSEDED)
- Каскадное завершение (Plan → ADR → Impact → Discussion)
- Обязательные backlinks (родитель знает детей, ребёнок знает родителя)
- Запрет физического удаления (только статусы REJECTED/SUPERSEDED)
- Привязка к конкретным сервисам (ADR и Plan живут внутри `services/{svc}/`)
- Скиллы: `/spec-create`, `/spec-status`, `/spec-update`, `/specs-health`

**Сильные стороны:** глубокая иерархия, каскадные статусы, привязка к сервисам, health checks.

**Слабые стороны:** тяжеловесность (4 типа документов до начала работы), нет поддержки brownfield, нет автоматической декомпозиции через LLM.

#### 1.2. Habr: мультиагентная разработка (Венделев)

**Источник:** https://habr.com/ru/articles/993470/

**Воркфлоу:** Epic → Feature (Spec) → TaskSummary → TaskForDev + TaskForTest → Implementation → Documentation

**Артефакты:**

| Тип | Расположение | Назначение |
|-----|-------------|------------|
| EpicSummary | `Doc/Backlog/YYYY/TZ{N}/EpicSummary.md` | Бизнес-требования + стадии |
| Spec | `Doc/FeatureList/YYYY/MM/TZ{N}_{NN}/Spec.md` | Функциональные требования |
| TaskSummary | `.../TaskSummary.md` | Общий технический план |
| TaskForDev | `.../TaskList/Task{N}_TaskForDev.md` | Задача для разработки |
| TaskForTest | `.../TaskList/Task{N}_TaskForTest.md` | Задача для тестирования |

**Особенности:**
- 12 специализированных агентов (epic-writer, feature-writer, dev-plan-writer, php-developer...)
- Каждый агент = одна ответственность + чистый контекст
- Оркестратор координирует последовательность
- Разделение `.ai/` (бизнес-логика) и `.claude/` (инструмент)
- Разные модели для разных агентов (haiku для планирования, sonnet для кодирования)

**Сильные стороны:** специализация агентов, изоляция контекста, explicit dev+test разделение задач.

**Слабые стороны:** привязка к PHP-стеку, жёсткие стадии (нет итеративности), сложная структура папок `Doc/Backlog/YYYY/TZ...`.

#### 1.3. GitHub Spec Kit

**Источник:** https://github.com/github/spec-kit

**Воркфлоу:** Constitution → Specify → Clarify → Plan → Tasks → Implement

**Артефакты:**

| Тип | Расположение | Назначение |
|-----|-------------|------------|
| Constitution | `.specify/memory/constitution.md` | Неизменяемые принципы проекта |
| Spec | `specs/NNN-feature/spec.md` | ЧТО и ЗАЧЕМ (user stories) |
| Plan | `specs/NNN-feature/plan.md` | КАК (технический план) |
| Research | `specs/NNN-feature/research.md` | Исследование технологий |
| Data Model | `specs/NNN-feature/data-model.md` | Модель данных |
| Contracts | `specs/NNN-feature/contracts/` | API-контракты |
| Tasks | `specs/NNN-feature/tasks.md` | Декомпозиция на задачи |
| Checklists | `specs/NNN-feature/checklists/` | Чек-листы качества |

**Особенности:**
- Constitution — "конституция" проекта с обязательными принципами
- Constitution Gates — проверка плана по принципам
- `/speckit.clarify` — структурированный опрос для уточнения спецификации
- Организация по фичам (одна папка = одна фича со всеми артефактами)
- CLI инструмент (Python) для инициализации
- Поддержка 17+ AI-агентов

**Сильные стороны:** constitution как архитектурный "якорь", clarify-шаг, всё по фичам (co-location), чек-листы качества.

**Слабые стороны:** нет поддержки brownfield, нет статусов жизненного цикла, нет каскадных зависимостей.

#### 1.4. OpenSpec (Fission-AI)

**Источник:** https://github.com/Fission-AI/OpenSpec

**Воркфлоу:** [Explore →] New → Continue/FastForward → Apply → Verify → Sync → Archive

**Артефакты:**

| Тип | Расположение | Назначение |
|-----|-------------|------------|
| Project | `openspec/project.md` | Описание проекта |
| Specs | `openspec/specs/{domain}/spec.md` | Текущее поведение (SSOT) |
| Proposal | `openspec/changes/{name}/proposal.md` | ЗАЧЕМ менять |
| Delta Specs | `openspec/changes/{name}/specs/` | ЧТО меняется (ADDED/MODIFIED/REMOVED) |
| Design | `openspec/changes/{name}/design.md` | КАК реализовать |
| Tasks | `openspec/changes/{name}/tasks.md` | Чеклист задач |

**Особенности:**
- **Дельта-спеки** — описание изменений как ADDED/MODIFIED/REMOVED вместо переписывания всего
- **Brownfield-first** — работает с существующими кодовыми базами
- Нет жёстких фаз — артефакты обновляемы в любой момент
- Archive сливает дельты обратно в specs/ (SSOT обновляется)
- Кастомизируемые YAML-схемы workflow
- RFC 2119 формат требований (SHALL/SHOULD/MAY + Given/When/Then)
- 20+ AI-инструментов
- CLI на TypeScript

**Сильные стороны:** дельта-спеки для brownfield, итеративность, specs как живой SSOT, кастомные схемы.

**Слабые стороны:** нет аналога constitution, нет каскадных статусов, плоская организация (без привязки к сервисам).

#### 1.5. Claude Task Master

**Источник:** https://github.com/eyaltoledano/claude-task-master

**Воркфлоу:** PRD → Parse → Analyze Complexity → Expand → Find Next → Implement → Update

**Артефакты:**

| Тип | Расположение | Назначение |
|-----|-------------|------------|
| PRD | `.taskmaster/docs/prd.txt` | Исходные требования |
| Tasks | `.taskmaster/tasks/tasks.json` | SSOT задач (JSON) |
| Complexity Report | `.taskmaster/reports/*.json` | Анализ сложности |
| Research | `.taskmaster/docs/research/` | Результаты исследований |

**Особенности:**
- **PRD → задачи за один шаг** — LLM парсит документ требований в структурированные задачи
- Анализ сложности (1-10) с рекомендацией количества подзадач
- Автоматическая декомпозиция (expand) на основе анализа сложности
- **find-next-task** — алгоритм выбора следующей задачи по зависимостям и приоритетам
- Tagged architecture — параллельные рабочие потоки
- MCP-интеграция (AI вызывает TaskMaster как tool)
- 17+ AI-провайдеров
- Zod-схемы валидации
- 7 статусов (pending, in-progress, done, review, deferred, cancelled, blocked)

**Сильные стороны:** автоматическая декомпозиция через LLM, анализ сложности, find-next-task алгоритм, MCP, мульти-провайдерность.

**Слабые стороны:** нет спецификаций (только PRD → задачи), нет ADR, нет архитектурных решений, JSON вместо Markdown (трудно читать).

### 3. Что лучше у каждого подхода

#### Взять у себя (specs/)

1. **Привязка к сервисам** — ADR и планы живут внутри `services/{svc}/`, что соответствует микросервисной архитектуре
2. **Каскадные статусы** — автоматическое завершение родителя при завершении всех детей
3. **Impact Analysis** — уникальный артефакт; ни один другой подход не анализирует влияние изменения на все сервисы
4. **Глоссарий** — единая терминология проекта
5. **Запрет удаления** — история решений сохраняется навсегда

#### Взять у Habr (Венделев)

6. **Разделение Dev и Test задач** — каждая задача порождает `TaskForDev` + `TaskForTest`. Это гарантирует, что тесты не забудут написать
7. **Специализированные агенты** — epic-writer, feature-writer, dev-plan-writer, test-plan-writer. Каждый работает в чистом контексте

#### Взять у Spec Kit

8. **Constitution** — неизменяемые архитектурные принципы проекта, через которые проверяется каждый Plan. Аналог "конституции" для кода
9. **Clarify** — структурированный опрос для уточнения спецификации (до 5 вопросов, приоритизация по Impact × Uncertainty)
10. **Checklists** — валидация качества спецификации до начала реализации

#### Взять у OpenSpec

11. **Дельта-спеки** — для brownfield: описание изменений как ADDED/MODIFIED/REMOVED вместо переписывания всей спецификации
12. **Specs как живой SSOT** — `openspec/specs/` описывает текущее поведение; после archive дельты сливаются обратно
13. **Кастомизируемые схемы** — YAML-конфигурация workflow (какие артефакты, в каком порядке)
14. **Given/When/Then формат** — RFC 2119 требования, которые читаемы и тестируемы

#### Взять у Task Master

15. **Автоматическая декомпозиция (LLM)** — PRD → задачи → подзадачи за один шаг
16. **Анализ сложности** — оценка 1-10 с рекомендацией количества подзадач
17. **find-next-task** — алгоритм выбора следующей задачи по зависимостям, приоритетам и статусам
18. **testStrategy на каждой задаче** — каждая задача содержит стратегию тестирования

#### 4.2. Изменить (существующее)

| # | Что | Как | Зачем |
|---|-----|-----|-------|
| M1 | **Workflow** | Discussion → **Clarify** → Impact → ADR → Plan → **Auto-tasks** → Implementation | Добавить уточнение и автоматизацию |

| M3 | **Discussion** | Добавить Given/When/Then для требований | Тестируемые требования |
| M4 | **Скиллы specs** | Добавить: `/spec-clarify`, `/spec-decompose`, `/spec-next` | Покрыть новые шаги workflow |

#### 4.3. Оставить без изменений

| Что | Почему |
|-----|--------|
| Привязка к сервисам | Соответствует нашей микросервисной архитектуре |
| 7 статусов + каскад | Работает, уникальное преимущество |
| Impact Analysis | Уникальный артефакт |
| Глоссарий | Нужен для терминологической согласованности |
| Markdown формат | Читаем, версионируется в git, привычен |

#### 4.4. НЕ брать

| Что | Откуда | Почему |
|-----|--------|--------|
| JSON для задач | Task Master | Markdown читаемее, git diff понятнее |
| Папки `Doc/Backlog/YYYY/MM/` | Habr | Слишком глубокая иерархия по датам — у нас привязка к сервисам |
| Отдельная `openspec/` папка | OpenSpec | У нас уже есть `specs/`, нет смысла менять |
| 12 специализированных агентов | Habr | Слишком сложно для начала; лучше один оркестратор с подзадачами |

| Кастомные YAML-схемы | OpenSpec | Overkill для одного проекта |

### 5. Целевой воркфлоу

#### Пользовательская перспектива

```
Пользователь: "Хочу добавить авторизацию через OAuth2"
        ↓
  LLM автоматически:
    1. Уточняет (Clarify) — задаёт 3-5 ключевых вопросов
    2. Создаёт Discussion — фиксирует намерение + требования (Given/When/Then)
    3. Проводит Impact Analysis — какие сервисы затронуты
    4. Создаёт ADR для каждого затронутого сервиса
    5. Проверяет ADR по Constitution (principles.md)
    6. Создаёт Plan с автоматической декомпозицией
    7. Оценивает сложность задач
    8. Генерирует GitHub Issues + Milestone
    9. Указывает стратегию тестирования для каждой задачи
        ↓
  Пользователь: ревьюит, корректирует, одобряет
        ↓
  LLM: реализует задачу за задачей (find-next-task)
```

#### Детальный workflow

```
                        Пользователь
                            │
                    "Хочу X для Y"
                            │
                            ▼
                    ┌───────────────┐
                    │   1. CLARIFY  │  LLM задаёт уточняющие вопросы
                    │  (3-5 вопр.)  │  Приоритет: Impact × Uncertainty
                    └───────┬───────┘
                            │
                            ▼
                    ┌───────────────┐
                    │ 2. DISCUSSION │  Фиксирует проблему + требования
                    │  (auto-gen)   │  Given/When/Then, варианты решений
                    └───────┬───────┘
                            │
                            ▼
                    ┌───────────────┐
                    │  3. IMPACT    │  Анализ: какие сервисы затронуты
                    │  (auto-gen)   │  Что меняется в каждом
                    └───────┬───────┘
                            │
                    ┌───────┴───────┐  (по одному ADR на сервис)
                    ▼               ▼
             ┌─────────────┐ ┌─────────────┐
             │  4. ADR #1  │ │  4. ADR #2  │  Архитектурные решения
             │  (auth-svc) │ │  (api-gw)   │  + Constitution Check
             └──────┬──────┘ └──────┬──────┘
                    │               │
                    ▼               ▼
             ┌─────────────┐ ┌─────────────┐
             │  5. PLAN #1 │ │  5. PLAN #2 │  Авто-декомпозиция (LLM)
             │  + tasks    │ │  + tasks    │  + анализ сложности
             │  + tests    │ │  + tests    │  + testStrategy
             └──────┬──────┘ └──────┬──────┘
                    │               │
                    ▼               ▼
             ┌─────────────────────────────┐
             │  6. GitHub Issues + Mile-   │  Автоматическое создание
             │     stone                   │  Issues из задач Plan
             └──────────────┬──────────────┘
                            │
                            ▼
             ┌─────────────────────────────┐
             │  7. IMPLEMENT               │  find-next-task →
             │  (task by task)             │  реализация → тесты →
             │                             │  status update → каскад
             └─────────────────────────────┘
```

#### Скиллы для нового workflow

| Скилл | Что делает | Шаг |
|-------|-----------|-----|
| `/spec-init` | Полный цикл: вопросы → Discussion → Impact → ADR → Plan | 1-5 |
| `/spec-clarify` | Только уточняющие вопросы | 1 |
| `/spec-create` | Создание одного артефакта (как сейчас) | 2-5 |
| `/spec-decompose` | LLM-декомпозиция Plan на задачи | 5 |
| `/spec-complexity` | Анализ сложности задач в Plan | 5 |
| `/spec-issues` | Генерация GitHub Issues из Plan | 6 |
| `/spec-next` | Выбор следующей задачи для реализации | 7 |
| `/spec-status` | Изменение статуса (как сейчас) | * |
| `/spec-update` | Обновление документа (как сейчас) | * |
| `/specs-health` | Health check (как сейчас) | * |

#### Ключевой скилл: `/spec-init`

Это "оркестратор", который запускает весь цикл одной командой:

```
/spec-init "Добавить OAuth2 авторизацию для микросервисов"
```

LLM:
1. Анализирует запрос
2. Задаёт уточняющие вопросы (AskUserQuestion)
3. На основе ответов генерирует Discussion
4. Проводит Impact Analysis (читает существующие specs и код)
5. Создаёт ADR для каждого затронутого сервиса
6. Проверяет ADR по principles.md
7. Декомпозирует каждый ADR в Plan с задачами
8. Оценивает сложность каждой задачи

## Открытые вопросы

1. **principles.md vs отдельные файлы?** — Один файл (как Constitution в Spec Kit) или набор файлов по категориям (Architecture, CodeStyle, Testing)?

2. **Дельта-спеки — нужны ли сейчас?** — Пока проект greenfield, дельта-спеки не нужны. Добавить позже, когда появится устоявшийся код?

3. **Автоматическое создание Issues — когда?** — При создании Plan (сразу) или при переходе Plan в RUNNING?

4. **specs/{svc}/specs/ — не запутает ли вложенность?** — `specs/services/auth/specs/` выглядит рекурсивно. Альтернатива: `specs/services/auth/current/` или `specs/services/auth/behavior/`.

5. **Степень автоматизации Clarify** — Фиксированный набор вопросов по категориям (как Spec Kit) или свободная форма?

6. **Как интегрировать с существующими скиллами?** — Существующие `/spec-create`, `/spec-status`, `/spec-update` сохраняются. Новые добавляются рядом?

7. **Приоритет реализации** — В каком порядке внедрять? Предложение:
   - Фаза 1: principles.md + Clarify + Given/When/Then формат
   - Фаза 2: авто-декомпозиция + анализ сложности + testStrategy
   - Фаза 3: spec-init оркестратор + spec-issues + spec-next
   - Фаза 4: дельта-спеки + живой SSOT (когда brownfield)

## 1. Двойственность формулировок

### Проблема 1.1: Неопределённое правило "первый уровень вложенности"

**Где:** Секция "Раздел 'Содержание'" (строка 130)

**Цитата:**
> Описание содержимого папки: файлы, вложенные папки и их назначение. Только первый уровень вложенности.

**Как может понять LLM:**
- Интерпретация A: Описывать только прямых потомков (папки и файлы непосредственно внутри текущей папки)
- Интерпретация B: Описывать первый уровень подпапок, но все файлы (включая файлы в подпапках)
- Интерпретация C: Указывать только первый уровень в дереве, но описывать все элементы в секциях

**Рекомендация:**
> Описание содержимого папки: файлы и вложенные папки, находящиеся непосредственно в текущей папке (прямые потомки). Файлы и папки в подпапках НЕ описываются в этом разделе — они описываются в README соответствующих подпапок.

**Критичность:** Высокая (P1)

**Где:** Секция "Секция 'Расширенное описание'" (строка 137)

**Цитата:**
> 1-2 предложения связного текста, объединяющего контекст содержимого папки (первый уровень вложенности). Упоминать подпапки и файлы в формате `имя/` и `имя.ext`.

- Интерпретация C: Обобщить функционал, упомянув только наиболее важные элементы

**Рекомендация:**
> 1-2 предложения связного текста, описывающего назначение папки через её содержимое. Упомянуть основные подпапки и файлы в формате `имя/` и `имя.ext`, показывая их роль в общей структуре. Не перечислять все элементы — выбрать наиболее значимые для понимания назначения папки.

## 2. Риск неправильной интерпретации LLM

### Проблема 2.1: Отсутствие явных границ "заполнить плейсхолдеры"

**Где:** Секция "Раздел 'Заголовок'" (строка 102)

**Цитата:**
> - заполняется в соответствии с форматом: `# /{папка}/ — {Назначение}`

**Как может понять LLM:**
- Интерпретация A: `{папка}` — это имя папки (без пути), `{Назначение}` — краткое описание
- Интерпретация B: `{папка}` — это полный путь от корня, `{Назначение}` — полное описание из frontmatter
- Интерпретация C: `{папка}` — может быть с префиксом `.` для скрытых папок

**Рекомендация:**
> - заполняется в соответствии с форматом: `# /{имя_папки}/ — {Назначение}`
> - `{имя_папки}` — имя папки БЕЗ пути (для `.claude/` писать `.claude`, а не полный путь)
> - `{Назначение}` — краткое описание роли папки (одно предложение без точки)

**Критичность:** Высокая (P1)

---

### Проблема 2.2: Противоречие между "Оглавление" и "Дерево"

**Где:** Секция "Раздел 'Оглавление'" (строки 117-119)

**Цитата:**
> - содержит ссылки на секции раздела "Содержание"
> - **структура двухуровневая:** секции верхнего уровня + вложенные элементы
> - вложенные элементы — папки/файлы внутри секции (отступ 2 пробела)

**Как может понять LLM:**
- Проблема: В секции "Дерево" (строка 153) написано "Если есть подпапки — добавить их тоже", но в оглавлении вложенные элементы — это элементы секции, а не подпапки элементов.

**Рекомендация:**
> Уточнить: "вложенные элементы — папки/файлы, перечисленные в секции 'Содержание' (первый уровень вложенности текущей папки). НЕ подпапки этих папок."

**Критичность:** Высокая (P1)

## 4. Полнота описания объекта

### 4.1. Не хватает: Правила версионирования README

**Суть:** Документ описывает как создавать README, но не описывает когда и как обновлять при изменении стандарта.

**Почему важно:** При обновлении `standard-readme.md` до версии 2.0 непонятно:
- Как мигрировать существующие README?
- Какие README обязательно мигрировать, какие опционально?
- Как отслеживать, какие README созданы по старой версии?

**Стоит ли добавлять:** Да (P1)

**Что даст:** Предотвратит рассинхронизацию стандарта и реальных файлов.

**Пример добавления:**
> ## 6. Миграция при обновлении стандарта
>
> При изменении `standard-readme.md`:
> - Увеличить `Версия стандарта: X.Y` в теле документа
> - Проверить все README с `standard-version < X.Y` через `check-version-drift.py`
> - Мигрировать критичные изменения в приоритетном порядке (корневые README → подпапки)

---

### 4.2. Не хватает: Обработка ошибок и граничных случаев

**Суть:** Стандарт описывает "правильный путь", но не описывает:
- Что делать, если папка содержит 50+ файлов (дерево станет огромным)?
- Что делать, если имя папки содержит спецсимволы (`@`, `#`, пробелы)?
- Что делать, если README уже существует при создании папки?

**Почему важно:** LLM может встретить граничный случай и не знать, как поступить.

**Стоит ли добавлять:** Да (P2)

**Что даст:** Уменьшит количество вопросов к пользователю в нестандартных ситуациях.

**Пример добавления:**
> ## 7. Граничные случаи
>
> | Ситуация | Действие |
> |----------|----------|
> | Папка содержит >30 файлов | Группировать файлы по категориям в дереве, не перечислять все в секции "Файлы" |
> | Имя папки со спецсимволами | Экранировать в markdown: `\@папка\@/` |
> | README уже существует | Прочитать, извлечь описания, обновить структуру по стандарту |

## 5. Слепые зоны автора

### Зона 5.1: Связь README с другими форматами документации

**Суть:** Стандарт описывает README как единственный формат документации папки. Но проект может использовать:
- JSDoc/TypeDoc для кодовой базы
- Swagger/OpenAPI для API
- Changelog для истории изменений

**Почему важно:** README может быть не единственным источником правды. Возникает вопрос:
- Дублировать ли информацию из других форматов в README?
- Как ссылаться на внешние форматы документации?
- Как синхронизировать README с автогенерируемой документацией?

**Расширение качества:** Умеренное

**Рекомендация:** Добавить сейчас (P1)

**Пример добавления:**
> ## 6. Связь с другими форматами документации
>
> README папки — SSOT для структуры, но НЕ для детальной документации кода.
>
> | Формат | Когда использовать | Как связать с README |
> |--------|-------------------|---------------------|
> | JSDoc/TypeDoc | Документация API кода | Ссылка в секции "Файлы": `Документация: [TSDoc](./docs/)` |
> | Swagger/OpenAPI | REST API эндпоинты | Ссылка в README: `API спецификация: [OpenAPI](./api.yaml)` |

### Зона 5.2: Автоматизация синхронизации README с кодовой базой

**Суть:** Стандарт предполагает ручное обновление README при изменении структуры. Но в крупных проектах:
- Файлы добавляются/удаляются часто
- README быстро устаревает
- Ручная синхронизация становится узким местом

**Почему важно:** Автоматизация могла бы:
- Генерировать секцию "Дерево" из файловой системы
- Обнаруживать новые файлы без описания
- Помечать устаревшие секции

**Расширение качества:** Значительное (для крупных проектов)

**Рекомендация:** Отложить (это инфраструктурная задача, выходит за рамки стандарта)

**Пример (если добавлять):**
> ## 9. Автоматизация синхронизации
>
> **Pre-commit hook:**
> ```bash
> python .structure/.instructions/.scripts/sync-readme.py --check
> ```
>
> Скрипт проверяет:
> - Все файлы папки описаны в README?
> - Дерево соответствует файловой системе?
> - Нет битых ссылок?

### Зона 5.3: Интернационализация README

**Суть:** Стандарт написан на русском языке и предполагает русскоязычные README. Но проект может быть международным.

**Почему важно:** Для open-source проектов или международных команд нужны:
- README на английском (или других языках)
- Правила именования файлов (`README.md` vs `README.ru.md`)
- Синхронизация переводов

**Расширение качества:** Минимальное (для текущего проекта)

**Рекомендация:** Не нужно (over-engineering)

**Пример (если добавлять):**
> ## 10. Интернационализация
>
> По умолчанию: README на русском (`README.md`)
>
> Для многоязычных проектов:
> - `README.md` — основной язык проекта
> - `README.en.md` — английский
> - `README.{lang}.md` — другие языки
>
> Синхронизация: вручную или через i18n-инструменты.

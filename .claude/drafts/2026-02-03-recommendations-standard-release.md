# Рекомендации по дополнению standard-release.md

**Дата:** 2026-02-03
**Статус:** draft
**Проанализированный документ:** `.github/.instructions/releases/standard-release.md`

---

## Краткая оценка

Документ **очень детальный** — один из самых полных в наборе. Охватывает SemVer, Git-теги, Changelog, связь с Milestones.

**Сильные стороны:**
- Полное описание SemVer с примерами breaking changes
- Таблицы правил инкремента версий
- Два способа создания changelog (автогенерация и вручную)
- Процедуры Yanking, Pre-release → Stable, Draft Release

## Рекомендации по улучшению

### 1. Дублирование: правила создания Release

**Проблема:** Секция "§10. Правила создания Release" частично дублирует standard-release-workflow.md (секции "Подготовка релиза" и "Создание релиза").

**Рекомендация:** Уточнить разделение ответственности:
- **standard-release.md:** ЧТО такое Release (свойства, версионирование, changelog формат)
- **standard-release-workflow.md:** КАК создавать Release (процесс, проверки)

Перенести "Правила создания Release" в standard-release-workflow.md или сделать явную ссылку.

### 2. Не описано: автоматическое определение версии

**Проблема:** Описано ручное определение версии по типу изменений, но нет автоматизации.

**Рекомендация:** Добавить подсекцию "Автоматическое определение версии":
- Использование conventional commits для автоопределения
- `feat:` → MINOR, `fix:` → PATCH, `BREAKING CHANGE:` → MAJOR
- Инструменты: semantic-release, standard-version

### 3. Неясность: формат CHANGELOG.md header

**Проблема:** Показан пример CHANGELOG.md, но не описано:
- Обязательность header-ссылок внизу файла
- Формат даты (ISO 8601 или другой)

**Рекомендация:** Уточнить:
- Header: `## [X.Y.Z] - YYYY-MM-DD` (ISO 8601)
- Ссылки внизу файла — рекомендуются, но опциональны
- Порядок секций: Added, Changed, Deprecated, Removed, Fixed, Security

### 4. Отсутствует: Release Notes vs CHANGELOG.md

**Проблема:** Не ясна разница между Release Notes (в GitHub Release) и CHANGELOG.md (в репозитории).

**Рекомендация:** Добавить сравнительную таблицу:
| Аспект | Release Notes | CHANGELOG.md |
|--------|---------------|--------------|
| Расположение | GitHub UI | Файл в репо |
| Формат | Markdown | Keep a Changelog |
| Генерация | `--generate-notes` | Ручная или через скрипт |
| Аудитория | Пользователи GitHub | Разработчики, пакетные менеджеры |

### 5. Не описано: релиз с пустым changelog

**Проблема:** Что делать если `--generate-notes` не находит PR (например, первый релиз или все изменения через direct commits)?

**Рекомендация:** Добавить граничный случай:
- Если PR нет — написать changelog вручную
- Для первого релиза: "Initial release"
- Direct commits не попадают в автогенерацию — использовать ручной changelog

### 6. Неясность: кастомизация release.yml

**Проблема:** Показан пример `.github/release.yml`, но не ясно:
- Является ли он обязательным
- Как соотносятся категории с метками проекта

**Рекомендация:** Уточнить:
- `.github/release.yml` — ОПЦИОНАЛЕН
- Без него GitHub генерирует плоский список PR
- Категории должны соответствовать меткам из standard-labels.md

### 7. Мелкое: нет примера Release Notes с Milestone ссылкой

**Проблема:** В секции "§7. Связь с Milestones" показана структура, но URL в примере содержит `{owner}/{repo}` placeholders.

**Рекомендация:** Добавить примечание или скрипт для генерации полного URL:
```bash
MILESTONE_URL=$(gh api repos/$OWNER/$REPO/milestones/$NUMBER -q '.html_url')
```

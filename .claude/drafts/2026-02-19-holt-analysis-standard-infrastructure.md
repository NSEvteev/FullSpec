## 1. Двойственность формулировок

### Проблема 1.1: Неопределённость "если важен" для порядка запуска

**Где:** Секция "2. Секции" → строка таблицы #1 (Локальный запуск), а также секция "Локальный запуск"

**Цитата:**
> Порядок запуска (если важен — зависимости при старте)

**Как может понять LLM:**
- Интерпретация A: Описывать порядок запуска только если один сервис зависит от другого технически (healthcheck, wait-for-it)
- Интерпретация B: Описывать порядок запуска всегда, если есть несколько контейнеров (даже без явных зависимостей)
- Интерпретация C: Пропустить этот элемент и написать "порядок не важен"

Критерий "если важен" не определён. LLM не знает, кто и по каким критериям определяет важность.

**Рекомендация:**
> Порядок запуска — описать явно, даже если он произвольный. Если зависимостей нет — написать: "Порядок запуска не имеет значения, сервисы запускаются параллельно."

**Критичность:** Высокая

---

### Проблема 1.2: Неопределённость формата "Формат сообщений" в Message Broker

**Где:** Секция "Message Broker" (описание), строка 3 обязательных элементов

**Цитата:**
> Формат сообщений — code-блок JSON с обязательными полями

**Как может понять LLM:**
- Интерпретация A: Показать один универсальный JSON-пример для всего брокера
- Интерпретация B: Показать отдельный JSON для каждого канала из таблицы каналов
- Интерпретация C: Показать схему (поля + типы), а не конкретный пример

Из примера в секции 5 видно, что используется один общий JSON — но это неявно следует из примера, а не из правила.

**Рекомендация:**
> Формат сообщений: один code-блок JSON на каждый канал из таблицы. Если все каналы используют одинаковую обёртку — один блок с пометкой "общий формат для всех каналов".

**Критичность:** Средняя

---

### Проблема 1.3: Неопределённость "stub-текст в курсиве" для пустых секций

**Где:** Секция "Message Broker" и секция "3. Принципы" → "Пустые секции"

**Цитата:**
> stub-текст в курсиве: `*Брокер сообщений не используется.*`. Удалять секцию запрещено.

**Как может понять LLM:**
- Интерпретация A: Написать ровно эту фразу для Message Broker, и придумать аналогичную для других необязательных секций
- Интерпретация B: Написать любой текст в курсиве — конкретная формулировка не важна
- Интерпретация C: Правило применяется только к Message Broker (единственный пример)

Правило сформулировано на примере одной секции. Не ясно, применяется ли оно к дополнительным секциям (Rate Limiting, CDN, Feature Flags) если они добавлены но не используются.

**Рекомендация:**
> Пустые секции — stub-текст в курсиве по шаблону: `*{Название раздела} не используется.*`. Правило применяется ко всем секциям документа, включая дополнительные.

**Критичность:** Средняя

---

### Проблема 1.4: "Дополнительные секции допустимы после Секретов" — без критериев

**Где:** Секция "2. Секции", первый абзац; секция "3. Принципы" → "Расширяемость"

**Цитата:**
> 8 обязательных секций в фиксированном порядке. Дополнительные секции допустимы после Секретов.

**Как может понять LLM:**
- Интерпретация A: Любые секции можно добавить — документ полностью расширяем
- Интерпретация B: Только секции из списка примеров (Rate Limiting, CDN, Feature Flags) — список исчерпывающий
- Интерпретация C: Добавить секцию нужно только если содержание явно не попадает ни в одну из 8 обязательных

**Рекомендация:**
> Дополнительные секции добавляются, если содержание не покрыто ни одной из 8 обязательных секций. Примеры: Rate Limiting, CDN, Feature Flags. Новую секцию не создавать, если содержание можно разместить в существующей.

**Критичность:** Низкая

## 2. Риск неправильной интерпретации

### Проблема 2.1: Отсутствие явного указания что делать при отсутствии хранилищ

**Где:** Секция "Хранилища" (описание раздела)

**Цитата:**
> Для каждого хранилища — h3-подсекция с именем типа и назначением.

**Как может понять LLM:**
- Интерпретация A: Если хранилищ нет — секцию не писать
- Интерпретация B: Если хранилищ нет — написать stub-текст в курсиве (по аналогии с Message Broker)
- Интерпретация C: Секция "Хранилища" обязательна — значит хранилище есть всегда

Секция "Хранилища" помечена как "Обязательна" в таблице (строка 3). Но проект без хранилищ (pure stateless) — реальный случай. Правило для этого случая не описано, а правило stub-текста явно указано только для Message Broker.

**Рекомендация:**
> Добавить в описание секции "Хранилища": если хранилищ нет — написать stub-текст: `*Внешние хранилища не используются.*`

**Критичность:** Высокая

---

### Проблема 2.2: Таблица Service Discovery — неясно что делать если сервисов один

**Где:** Секция "Service Discovery" (описание таблицы)

**Цитата:**
> Обязательная таблица: Сервис-источник | Сервис-цель | Механизм | Значение (dev) | Env-переменная

**Как может понять LLM:**
- Интерпретация A: Таблица обязательна даже при 1 сервисе — написать пустую таблицу с заголовками
- Интерпретация B: При 1 сервисе написать: "Service discovery не применяется — сервис работает автономно"
- Интерпретация C: Секция обязательна, но её содержание зависит от архитектуры — написать вводный абзац без таблицы

Раздел "2. Секции" помечает Service Discovery как "Обязательна" без оговорок. Правило для монолита или single-service проекта не определено.

**Рекомендация:**
> Добавить в описание секции: если сервис один (нет межсервисных вызовов) — написать: `*Межсервисное взаимодействие отсутствует.*`

**Критичность:** Высокая

### Проблема 2.3: Connection string в шаблоне использует несуществующий синтаксис

**Где:** Секция "4. Шаблон" → блок "Хранилища"

**Цитата:**
> **Connection string:** `{protocol}://${PREFIX}_USER:${PREFIX}_PASSWORD@${PREFIX}_HOST:${PREFIX}_PORT/${PREFIX}_DB`

**Как может понять LLM:**
- Интерпретация A: Это буквальный шаблон — переменные `${PREFIX}_USER` нужно заменить на `${POSTGRES_USER}` подставив конкретный PREFIX
- Интерпретация B: Это псевдокод — в реальном документе нужно написать конкретные имена переменных: `postgresql://${POSTGRES_USER}:${POSTGRES_PASSWORD}@...`
- Интерпретация C: Шаблон нужно воспроизвести дословно в документе

Пример в секции 5 использует другой (конкретный) формат: `postgresql://${POSTGRES_USER}:...`. Шаблон в секции 4 и пример в секции 5 расходятся — шаблон содержит метапеременную `{PREFIX}`, а пример — реальные имена переменных.

**Рекомендация:**
> В шаблоне заменить `${PREFIX}_USER` на `${VAR_NAME}` с пояснением: "VAR_NAME = фактическое имя переменной, например POSTGRES_USER". Или убрать псевдосинтаксис и написать: "Connection string: заменить переменные на фактические имена из таблицы параметров".

**Критичность:** Высокая

---

### Проблема 2.4: "Минимальные строки" таблицы Окружений — статус не определён

**Где:** Секция "Окружения" (описание)

**Цитата:**
> **Минимальные строки:** Реплики сервисов, БД, Секреты, Домен, Логирование.

**Как может понять LLM:**
- Интерпретация A: Эти 5 строк обязательны, остальные — по усмотрению автора
- Интерпретация B: Список минимально достаточный — нужно добавить только то, чем окружения реально отличаются
- Интерпретация C: Список рекомендательный — можно не добавлять строки если окружения не отличаются по этому параметру

Слово "минимальные" не уточняет: они обязательны или это нижняя граница.

**Рекомендация:**
> Заменить "минимальные строки" на "обязательные строки" или добавить пояснение: "Эти строки присутствуют всегда, даже если значения одинаковы для всех окружений — явная одинаковость лучше неявной."

**Критичность:** Средняя

## 3. Ручные операции вместо автоматизации

### Проблема 3.1: Нет ссылки на скрипты или скиллы создания/валидации

**Где:** Секция "Связанные документы" (таблица в шапке документа)

**Цитата:**
> | Валидация | *Не создано* |
> | Создание | *Не требуется* |
> | Модификация | *Не создано* |

Для `docs/.system/infrastructure.md` отсутствует validation-инструкция. LLM, создавая документ, не сможет проверить его корректность автоматически — будет выполнять проверку вручную или пропустит её.

**Существует:** Нет. Однако по стандарту `standard-instruction.md` секция "Скиллы" обязательна. Отсутствие валидации — известный пробел (помечено "Не создано"), но последствие не описано.

**Рекомендация:** Добавить явное указание: "Валидация выполняется вручную по чек-листу секции [N]. Скрипт валидации планируется." Это устраняет ситуацию, когда LLM пропускает валидацию совсем.

**Критичность:** Низкая (пробел известен, риск минимален пока нет скрипта)

## 4. Полнота описания объекта

### 4.1. Не хватает: правило для проектов без staging-окружения

**Почему важно:** Таблица "Окружения" жёстко трёхколоночная (dev / staging / prod). Проект без staging-окружения вынужден писать пустую колонку или придумывать правило самостоятельно. LLM может либо добавить фиктивный staging, либо поломать структуру таблицы.

**Стоит ли добавлять:** Да.

**Что даст:** Устраняет неопределённость — вместо трёх вариантов поведения LLM получает одно правило.

**Рекомендация:** Добавить в описание секции "Окружения": "Если staging отсутствует — колонку включить со значением `—` или убрать с пояснением в заголовке таблицы."

---

### 4.2. Не хватает: описание что делать при нескольких брокерах

**Почему важно:** Секция Message Broker описывает шаблон для одного брокера через h3-подсекцию. Если в проекте два брокера (RabbitMQ + Kafka) — правило предполагает две h3-подсекции, но явно не подтверждает это. LLM может создать плоскую структуру или объединить в одну таблицу.

**Стоит ли добавлять:** Да.

**Что даст:** Явное подтверждение паттерна "одна h3 на брокер".

---

### 4.3. Не хватает: правило для Management UI брокера в prod-окружении

**Почему важно:** Таблица параметров брокера содержит "Management UI" с dev-значением. В prod Management UI часто закрыт или недоступен. Шаблон не указывает что писать для prod — и LLM может написать несуществующий URL или оставить dev-значение.

**Стоит ли добавлять:** Да.

**Что даст:** Снимает риск указания недоступного URL в продакшн-документе.

**Рекомендация:** Добавить к описанию таблицы брокера: "Если Management UI доступен только в dev — указать `*недоступно в staging/prod*`."

---

### 4.4. Не хватает: явная фиксация что Секции 1–8 НЕ переименовываются

**Почему важно:** Стандарт описывает 8 секций с конкретными именами. Но нет явного запрета переименовывать их. LLM может назвать секцию "Базы данных" вместо "Хранилища" или "Discovery-механизм" вместо "Service Discovery" — и документ формально будет соответствовать описанию, но сломает навигацию и потенциальные скрипты валидации.

**Стоит ли добавлять:** Да.

**Что даст:** Исключает семантически эквивалентные, но структурно сломанные варианты.

**Рекомендация:** В начале секции 2 добавить: "Имена секций фиксированы — использовать точные названия из таблицы."

## 5. Слепые зоны автора

### Зона 5.1: Нет правила для документа, написанного ДО появления нового сервиса

**Суть:** `infrastructure.md` содержит таблицу "Сервисы и порты" — список всех сервисов. При добавлении нового сервиса в проект этот документ устаревает. Стандарт не описывает триггеры и процесс обновления документа. Это не покрыто и соседними стандартами.

**Почему важно:** LLM, добавляя новый сервис, не знает что нужно обновить `infrastructure.md`. Документ будет актуальным только при создании, потом неизбежно устареет без явного процесса обновления.

**Расширение качества:** Умеренное.

**Рекомендация:** Добавить сейчас — один абзац или NOTE: "При добавлении нового сервиса обновить секции: Сервисы и порты, Хранилища (если у сервиса своя БД), Service Discovery (если сервис вызывает другие)."

**Пример добавления:**
> **Триггеры обновления:** добавление сервиса → Сервисы и порты + Хранилища + Service Discovery; добавление хранилища → Хранилища; смена окружения → Окружения.

---

### Зона 5.2: Нет описания поведения при конфликте между infrastructure.md и реальной конфигурацией

**Суть:** Документ содержит значения портов, имена баз, URL сервисов. Эти значения могут расходиться с реальными `.env.example` или `docker-compose.yml`. Стандарт не определяет что является SSOT при конфликте — документ или конфигурационный файл.

**Почему важно:** LLM при генерации кода или конфигурации будет использовать значения из `infrastructure.md`, не проверяя их соответствие реальным файлам.

**Расширение качества:** Значительное.

**Рекомендация:** Добавить сейчас — один принцип в секцию "3. Принципы": "SSOT значений — `.env.example` и `docker-compose.yml`. `infrastructure.md` синхронизируется с ними, не является источником конфигурации."

---

### Зона 5.3: Отсутствует граничный случай — монорепозиторий с несколькими независимыми системами

**Суть:** Стандарт предполагает один `infrastructure.md` на весь проект ("один файл — вся платформа"). Но если в монорепо живёт несколько независимых систем — один файл становится нечитаемым (15+ сервисов, 5+ баз).

**Почему важно:** Архитектурный выбор "один файл на всё" может оказаться практически неудобным при масштабировании.

**Расширение качества:** Умеренное.

**Рекомендация:** Отложить — это over-engineering для большинства проектов. Достаточно добавить NOTE: "Если сервисов более 10 — рассмотреть разбивку по доменам."

## 6. Другие проблемы

### Проблема 6.1: Шаблон содержит синтаксис placeholders двух несовместимых форматов

**Где:** Секция "4. Шаблон" — вся секция

**Суть:** В шаблоне смешаны два формата placeholders:
- `{svc}` — фигурные скобки без доллара (текстовая замена)
- `${PREFIX}_USER` — bash-переменная с долларом (env var)

**Как может понять LLM:**
- Интерпретация A: `{svc}` — заменить, `${PREFIX}_USER` — оставить как есть (это env var, не placeholder)
- Интерпретация B: Оба формата — placeholders, оба нужно заменить
- Интерпретация C: `${PREFIX}_USER` — буквально включить в connection string (shell-подстановка при runtime)

Пример в секции 5 показывает вариант A (env vars оставлены как `${POSTGRES_USER}`), но шаблон в секции 4 использует `${PREFIX}_USER` — псевдо-env-var с метапеременной PREFIX.

**Рекомендация:** Разделить в шаблоне: placeholders для заполнения автором документа — в фигурных скобках `{значение}`, env vars в connection string — в реальных именах `${POSTGRES_USER}` (как в примере секции 5). Привести шаблон в соответствие с примером.

**Приоритет:** P1 — расхождение шаблона и примера блокирует корректное использование шаблона.

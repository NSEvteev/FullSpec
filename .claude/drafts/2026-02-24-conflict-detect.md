## Контекст

**Задача:** G10 из standard-process.md — определение уровня CONFLICT не автоматизировано
**Почему создан:** Определить возможность и целесообразность автоматизации
**Связанные файлы:**
- `specs/.instructions/analysis/standard-analysis.md` — § 6.3 RUNNING to CONFLICT
- `specs/.instructions/.scripts/chain_status.py` — classify_feedback()
- `specs/.instructions/standard-process.md` — §6 Путь B, шаг B.4

## Содержание

### Текущее состояние

**Шаг B.4 (обнаружение уровня)** — полностью на LLM:
1. LLM получает описание проблемы (feedback от кода/тестов/ревью)
2. LLM проверяет снизу вверх: Plan Dev → Plan Tests → Design → Discussion
3. LLM определяет самый высокий затронутый уровень
4. С этого уровня начинается разрешение (B.5)

**classify_feedback(level)** уже существует, но принимает level как входной параметр — LLM сам определяет "free"/"flag"/"conflict".

### Что можно автоматизировать

| Подход | Описание | Сложность | Точность |
|--------|----------|-----------|----------|
| **Эвристики по diff** | Анализ git diff: изменились API endpoints → Design, изменились тесты → Plan Tests | Средняя | Низкая — diff не показывает "почему" |
| **Маппинг файл → уровень** | Файл в `src/{svc}/api/` → Design, файл в `src/{svc}/tests/` → Plan Tests | Низкая | Низкая — расположение ≠ причина |
| **LLM с контекстом** | LLM + docs/{svc}.md Code Map → автоматическое определение границы | Низкая | Высокая — LLM уже это делает |

### Почему оставить на LLM

1. **LLM справляется** — определение уровня требует понимания семантики изменения, не синтаксиса
2. **Code Map в docs/{svc}.md** — уже содержит границы (свободно/флаг/CONFLICT)
3. **Формализовано** — standard-analysis.md § 6.3 описывает критерии, LLM следует им
4. **Автоматизация не даст выигрыша** — LLM всё равно должен прочитать feedback и документы

### Если всё-таки автоматизировать

Расширить `chain_status.py`:

```python
def detect_conflict_level(self, feedback: str, diff: str) -> dict:
    """
    Анализ feedback + diff → определение затронутого уровня.
    Returns: {"level": "plan-dev"|"plan-test"|"design"|"discussion",
              "confidence": float, "reason": str}
    """
```

Но это по сути дублирование того, что LLM делает лучше.

## Решения

- Приоритет низкий — LLM справляется
- Формализовано в стандарте — критерии чёткие
- Автоматизация не даёт значимого выигрыша

## Что уже описано в проекте

### 1. Классификация обратной связи: `classify_feedback()` (chain_status.py, строки 826-887)

Функция `ChainManager.classify_feedback(level: FeedbackLevel)` уже реализована как **guard** (не классификатор). Принимает готовый уровень (`"free"` / `"flag"` / `"conflict"`) и проверяет, допустим ли этот уровень для текущего статуса цепочки:

- `"free"` -- всегда `allowed=True`, `action="no_action"`
- `"flag"` -- `allowed=True` только при `RUNNING`, `action="log_only"`
- `"conflict"` -- `allowed=True` при `RUNNING` или `REVIEW`, `action="transition_to_conflict"`

**Вывод:** Функция не определяет уровень, а валидирует решение LLM. Определение уровня (шаг B.4) полностью за LLM.

### 2. Три уровня границ автономии в docs/{svc}.md (standard-service.md, секция "Границы автономии LLM")

Каждый `docs/{svc}.md` содержит обязательную секцию 8 "Границы автономии LLM" с тремя уровнями:

```
- **Свободно:** реализация внутри пакета (алгоритмы, рефакторинг, оптимизация)
- **Флаг:** изменение контрактов между пакетами (может затронуть тесты)
- **CONFLICT:** изменение API (/notifications/*), data model (колонки, индексы), WebSocket-протокол
```

Правила пересечения (standard-service.md): при пересечении Флаг и CONFLICT -- применять CONFLICT. Любое изменение, затрагивающее публичные контракты или схему данных -- CONFLICT, даже если оно также затрагивает тесты.

**Путь к секции:** `specs/.instructions/docs/service/standard-service.md` -- полный стандарт секции, строки 376-395.

### 3. Критерии затронутости документа (standard-analysis.md, § 6.3 "Обнаружение затронутого уровня")

LLM определяет самый высокий затронутый документ снизу вверх. Критерий: "Содержание этого документа стало неверным?" Документ затронут, если хотя бы одно его утверждение стало фактически неверным из-за изменений в коде.

**Конкретные примеры из стандарта (строки 518-524):**

| Тип изменения | Затронут? |
|---------------|-----------|
| Изменён алгоритм (bcrypt -> argon2) в SVC-N "Решения по реализации" | Да -- Design |
| Изменено имя метода внутри пакета (без смены API) | Нет |
| Изменена схема БД (добавлена колонка) | Да -- Design (SVC-N data model) |
| Изменён retry-механизм без смены контракта | Нет |

### 4. Граница "рабочие правки" vs CONFLICT (standard-analysis.md, § 6.3, строки 486-487)

Рабочие правки (flag) -- изменения, НЕ затрагивающие:
- Ответственность сервиса (SVC-N)
- API контракт сервиса (INT-N)
- Data model

Если правка затрагивает хотя бы одно из трёх -- это Design и выше, следовательно CONFLICT.

### 5. Процесс CONFLICT в standard-process.md (§ 6, Путь B, шаги B.1-B.7)

Полная последовательность из `specs/.instructions/standard-process.md`, строки 306-331:

| Шаг | Описание | Инструменты |
|-----|----------|-------------|
| B.1 | Обнаружение несовместимости | chain_status.py (classify_feedback) |
| B.2 | Классификация: free/flag/CONFLICT | standard-analysis.md § 6.3 |
| B.3 | Каскад: вся цепочка -> CONFLICT | chain_status.py (T4/T8) |
| B.4 | Обнаружение уровня (снизу вверх) | **LLM (не автоматизировано)** -- это G10 |
| B.5 | Разрешение сверху вниз | -modify скиллы |
| B.6 | Per-doc -> WAITING | chain_status.py (T5) |
| B.7 | Повторный запуск | chain_status.py (T3), /dev-create --resume |

### 6. Design SVC-N секции как маппинг (standard-analysis.md, § 2.3, строки 156-178)

SVC-N секции Design-документа имеют подсекции §§ 1-8, **идентичные** секциям §§ 1-8 в `docs/{svc}.md`:

- § 1 Назначение
- § 2 API контракты
- § 3 Data Model
- § 4 Потоки
- § 5 Code Map
- § 6 Зависимости
- § 7 Доменная модель
- § 8 Границы автономии LLM

Это создаёт прямой маппинг: если diff затрагивает файл, описанный в Code Map пакета X, и Code Map относит пакет X к уровню "CONFLICT" -- автоматический вывод возможен.

### 7. Upward feedback (standard-analysis.md, § 3.5, строки 285-309)

При работе на уровне N может обнаружиться информация, затрагивающая уровень N-1 и выше. LLM проверяет каждый факт: попадает ли он в зону ответственности вышестоящего уровня и отсутствует ли в документе этого уровня? Правило каскада: LLM всегда проверяет вверх до Discussion включительно.

### 8. Кросс-цепочечная обратная связь (standard-analysis.md, § 7.2 + chain_status.py строки 732-822)

`check_cross_chain()` сканирует `<!-- chain: NNNN-{topic} -->` маркеры в docs/, находит пересечения, возвращает `CrossChainAlert` с severity (info/warning/critical). Матрица реакции по статусу другой цепочки: DRAFT=info, WAITING=warning, RUNNING=critical, DONE=info.

### 9. TRANSITION_MATRIX и tree-level переходы (chain_status.py, строки 63-85)

T4 (RUNNING->CONFLICT) и T8 (REVIEW->CONFLICT) -- оба tree-level. Все документы цепочки переходят одновременно. DONE-документы не затрагиваются (строки 700-704).

## Best practices

### 1. Fitness Functions для архитектурной целостности (рекомендация: применить к Code Map)

**Концепция:** Fitness function -- автоматический тест, проверяющий архитектурную характеристику системы. Эквивалент unit-теста, но для архитектуры. Термин из книги *Building Evolutionary Architectures* (Ford, Parsons, Kua).

**Релевантность для G10:** Секция "Границы автономии LLM" в `docs/{svc}.md` -- по сути уже является fitness function boundary specification. Можно усилить:

- **Статическая fitness function:** скрипт, который анализирует `git diff` и сопоставляет изменённые файлы с таблицей пакетов из Code Map (§ 5 `docs/{svc}.md`). Если файл принадлежит пакету, помеченному как "CONFLICT" -- предупреждение.
- **Пример реализации:** парсить `docs/{svc}.md` Code Map -> Пакеты, извлечь таблицу `| Пакет | Назначение |`, построить маппинг `file_path -> package -> autonomy_level`.

**Ключевой факт (из исследования 2025):** В анализе 2400 PR за 12 месяцев code reviewers обнаружили только 17% архитектурных нарушений. Автоматические fitness functions снижают drift с 34 нарушений/год до 3.

**Ограничение для G10:** Fitness function определяет "что изменилось" (синтаксис), но не "почему" (семантику). Для G10 нужна семантика -- fitness function может быть только подсказкой, не заменой LLM.

### 2. AST-based diff analysis (рекомендация: не применять)

**Концепция:** Анализ Abstract Syntax Tree diff вместо текстового diff. Позволяет определить, что изменился не просто файл, а конкретный API endpoint, метод, модель данных.

**Исследования 2025:**
- Datalog-based language-agnostic change impact analysis для микросервисов (ICSE 2025) -- AST-анализ с фактами в формате Datalog.
- Автоматическое извлечение паттернов изменений из AST (arXiv 1309.3730).

**Почему не применять для G10:**
- Высокая сложность реализации (парсеры для каждого языка).
- Проект мультиязычный (Python, возможно другие) -- нужны отдельные парсеры.
- LLM уже выполняет семантический анализ diff лучше любого AST-парсера.
- Code Map в `docs/{svc}.md` уже содержит структурную информацию, которую AST-анализ пытается извлечь из кода.

### 3. LLM-assisted change classification с обогащённым контекстом (рекомендация: текущий подход оптимален)

**Исследования 2025:**
- **ContextCRBench** (ByteDance, Nov 2025): обогащение LLM-промптов контекстом (issue descriptions, PR descriptions) значительно улучшает качество code review. Текстовый контекст (описания задач) даёт больший прирост, чем код окружения.
- **Rethinking Code Review with LLMs** (WirelessCar, May 2025): RAG-pipeline для сборки релевантного контекста из проекта перед ревью. LLM дополняет, а не заменяет ревьюера.
- **MetaMateCR** (Meta, Jul 2025): LLM как классификатор "actionability" комментариев ревью. Few-shot prompting для бинарной классификации.

**Как это подтверждает текущий подход G10:**
- LLM уже получает обогащённый контекст: `docs/{svc}.md` (Code Map + Границы автономии) + `design.md` (SVC-N секции) + feedback от кода/тестов.
- Исследование ByteDance: текстовый контекст (описание задачи, requirements из Discussion) важнее кода окружения.
- Текущий SDD-процесс обеспечивает именно этот обогащённый контекст: Discussion (WHY) + Design (HOW) + docs/ (AS IS).

### 4. Semantic diff и specification drift detection (рекомендация: мониторинг, не автоматизация B.4)

**Концепция:** Semantic diff анализирует не текстовые различия, а смысловые -- "изменился API контракт" vs "изменился комментарий". Specification drift -- расхождение между спецификацией и реализацией.

**Как можно применить (не для B.4, а для мониторинга):**
- Периодическая проверка: код все ещё соответствует `docs/{svc}.md`? API endpoints в коде совпадают с описанными в API контракты? Data Model в коде совпадает с описанной?
- Это **отдельная задача** от G10 -- ближе к CI/CD fitness function.

### 5. Pull request-based change impact analysis (рекомендация: применимо как дополнение к B.2)

**Исследование (Empirical Software Engineering, May 2025):** Enhanced code reviews using pull request based change impact analysis. Автоматический анализ PR показывает, какие компоненты затронуты, ускоряя ревью.

**Потенциальное улучшение для SDD:** При шаге B.2 (Классификация) LLM может получать автоматически сгенерированный отчёт о затронутых файлах с маппингом на пакеты из Code Map. Это не заменяет LLM, но ускоряет его работу:

```
Изменённые файлы:
- src/notification/api/routes.py -> пакет notification.api -> Границы: CONFLICT (API контракты)
- src/notification/storage/models.py -> пакет notification.storage -> Границы: CONFLICT (Data Model)
- src/notification/events/handlers.py -> пакет notification.events -> Границы: Свободно (внутри пакета)

Вывод: затронуты CONFLICT-области -> рекомендация: CONFLICT level
```

Такой отчёт можно генерировать скриптом, парсящим Code Map из `docs/{svc}.md` и сопоставляющим с `git diff --name-only`. Это helper-функция (открытый вопрос 1), не замена LLM.

### 6. Hallucination risks в LLM-классификации (рекомендация: учитывать)

**Исследование (arxiv 2401.16310, 2025):** LLM могут генерировать правдоподобные, но фактически неверные оценки при security code review. Hallucination может привести к неверной классификации уровня (flag вместо CONFLICT).

**Митигация в текущем SDD:**
- Пользователь ревьюит каждый документ при разрешении CONFLICT (шаг B.6 -- human-in-the-loop).
- Маркер `[ТРЕБУЕТ УТОЧНЕНИЯ]` блокирует переход DRAFT -> WAITING (standard-analysis.md § 8.2).
- Но: если LLM ошибочно классифицирует CONFLICT как flag, рабочие правки вносятся автономно (без ревью пользователя) -- потенциальная точка отказа.

**Рекомендация:** Добавить логирование решений LLM о классификации (открытый вопрос 2) для post-hoc анализа точности.

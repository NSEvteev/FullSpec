## 1. Двойственность формулировок

### Проблема 1.1: Неопределённое условие "По требованию провайдера"

**Где:** Секция "4. Категории секретов" (строка 181)

**Цитата:**
> | **API** | `API_` | Ключи внешних сервисов | По требованию провайдера |

**Как может понять LLM:**
- Интерпретация A: Ротация только когда провайдер явно уведомит об обязательной ротации (например, после инцидента безопасности)
- Интерпретация B: Ротация согласно документации провайдера (например, каждые 90 дней для Stripe API keys)
- Интерпретация C: Ротация когда пользователь решит, что "требование" существует (субъективно)

**Рекомендация:**
> **API** | `API_` | Ключи внешних сервисов | Согласно политике безопасности провайдера (проверить документацию). Если не указано — 180 дней.

**Критичность:** Средняя

### Проблема 1.2: Размытое условие "При смене интеграции"

**Где:** Секция "4. Категории секретов" (строка 184)

**Цитата:**
> | **Notification** | `NOTIFY_` | Slack, email webhooks | При смене интеграции |

**Как может понять LLM:**
- Интерпретация A: Только при физическом переносе webhook URL (изменение канала, смена Slack workspace)
- Интерпретация B: При каждом изменении конфигурации интеграции (добавление нового канала, изменение фильтров)
- Интерпретация C: При подозрении на компрометацию webhook URL

**Рекомендация:**
> **Notification** | `NOTIFY_` | Slack, email webhooks | При изменении канала/сервиса уведомлений или подозрении на компрометацию. Регулярная ротация — 365 дней.

**Критичность:** Низкая

### Проблема 1.3: Отсутствие критериев выбора уровня хранения

**Где:** Секция "3. Уровни хранения" (строки 109-113)

**Цитата:**
> | **Repository** | Один репозиторий | Секреты, уникальные для проекта |
> | **Environment** | Один репозиторий + конкретная среда | Разные значения для dev/staging/prod |
> | **Organization** | Все (или выбранные) репозитории организации | Общие секреты (Docker Hub, npm, Slack) |

**Как может понять LLM:**
- Интерпретация A: Если секрет используется в 2+ репозиториях → Organization level
- Интерпретация B: Если секрет МОЖЕТ быть использован в других проектах (даже если сейчас используется в одном) → Organization level
- Интерпретация C: Если секрет имеет разные значения для сред → Environment level (даже если он общий для нескольких репозиториев)

**Рекомендация:**
Добавить подсекцию "Алгоритм выбора уровня":

> **Алгоритм выбора уровня хранения:**
> 1. Секрет используется в 2+ репозиториях? → Organization
> 2. Секрет имеет разные значения для dev/staging/prod? → Environment
> 3. Иначе → Repository

**Критичность:** Высокая (блокирует корректное создание секретов)

## 2. Риск неправильной интерпретации

### Проблема 2.1: Неявное поведение при дублировании секретов

**Где:** Секция "3. Уровни хранения" (не покрыто в документе)

**Что отсутствует:**
Документ не описывает приоритет при наличии секрета с одинаковым именем на разных уровнях (Repository + Environment + Organization).

**Как может понять LLM:**
- Интерпретация A: Environment секреты перезаписывают Repository секреты
- Интерпретация B: Repository секреты имеют приоритет над Organization
- Интерпретация C: GitHub вернёт ошибку при дублировании имён

**Рекомендация:**
Добавить подсекцию "Приоритет секретов при дублировании имён":

> **Приоритет (от высшего к низшему):**
> 1. Environment secrets
> 2. Repository secrets
> 3. Organization secrets
>
> Если секрет `DATABASE_URL` существует на всех трёх уровнях, workflow получит значение из **Environment** (если указан `environment:`), иначе — из **Repository**.

**Критичность:** Высокая (блокирует понимание поведения при конфликтах)

### Проблема 2.2: Неясная граница "автоматической маскировки" секретов

**Где:** Секция "7. Secrets и Variables" в standard-action.md (строки 701-703)

**Цитата (из анализируемого документа, секция 1):**
> | Использование секретов в workflows (`${{ secrets.X }}`) | [standard-action.md § 7](../standard-action.md#7-secrets-и-variables) |

**Проблема:**
Документ `standard-secrets.md` утверждает, что использование секретов описано в другом стандарте, но не предупреждает о граничных случаях маскировки.

**Как может понять LLM:**
- Интерпретация A: GitHub маскирует ТОЛЬКО точные совпадения значения секрета в логах
- Интерпретация B: GitHub маскирует секрет во ВСЕХ контекстах (включая base64, URL-encoded)
- Интерпретация C: Если секрет передан через env и модифицирован (trim, uppercase), GitHub НЕ замаскирует результат

**Рекомендация:**
Добавить в секцию "7. CLI команды" подсекцию "Примечание о маскировке в логах":

> **Маскировка секретов в логах:**
> - GitHub маскирует только точное значение секрета из `secrets.*`
> - Если секрет модифицирован (trim, base64, uppercase) — маскировка НЕ сработает
> - Для debug используйте `echo "Value: ***"` вместо вывода секрета напрямую

**Критичность:** Средняя

### Проблема 2.3: Неявная зависимость между Environment secrets и Protection Rules

**Где:** Секция "3. Уровни хранения → Environment Secrets" (строки 149-155)

**Цитата:**
> **Правила Protection Rules** (настраиваются в Settings → Environments):
>
> | Правило | Описание |
> |---------|----------|
> | Required reviewers | Ручное подтверждение перед деплоем |

**Проблема:**
Документ описывает Protection Rules КАК ЕСЛИ они опциональны, но не объясняет, что эти правила БЛОКИРУЮТ доступ к Environment secrets до выполнения условий.

**Как может понять LLM:**
- Интерпретация A: Protection Rules — опциональная функция для "дополнительной безопасности"
- Интерпретация B: Если Required reviewers настроен, workflow ОСТАНОВИТСЯ и будет ждать approval (блокирующее поведение)
- Интерпретация C: Protection Rules применяются только к deployment, но НЕ к чтению секретов

**Рекомендация:**
Переформулировать параграф:

> **Правила Protection Rules (БЛОКИРУЮЩИЕ):**
>
> Если настроены для Environment — workflow **ОСТАНОВИТСЯ** до выполнения условий:
>
> | Правило | Поведение |
> |---------|----------|
> | Required reviewers | Workflow ожидает N approvals. Job НЕ запустится до подтверждения. |
> | Wait timer | Задержка X минут перед началом job. |
> | Deployment branches | Job запустится ТОЛЬКО для указанных веток (например, `main`). Остальные — fail. |

**Критичность:** Высокая (влияет на понимание блокирующего поведения)

### Проблема 2.4: Неопределённость "по умолчанию" для `open-pull-requests-limit`

**Где:** Секция "3. Dependabot" в standard-security.md (строка 147, ссылка из standard-secrets.md)

**Связь с анализируемым документом:**
`standard-secrets.md` НЕ описывает Dependabot, но упоминает ротацию секретов, которая может конфликтовать с Dependabot PR.

**Цитата (из примера в standard-security.md):**
> open-pull-requests-limit: 5

**Проблема:**
Параметр указан как опциональный ("по умолчанию 5"), но неясно: если НЕ указать — будет ли Dependabot создавать бесконечные PR или остановится на 5?

**Как может понять LLM:**
- Интерпретация A: Если не указать — Dependabot создаст ВСЕ PR сразу (для 50 зависимостей → 50 PR)
- Интерпретация B: Если не указать — Dependabot использует дефолт 5 (как указано в документации)

**Рекомендация:**
Это проблема НЕ анализируемого документа, но влияет на операции ротации. В `standard-secrets.md` добавить примечание в секцию "5. Ротация":

> **Примечание:** Если ротация секрета запускается через Dependabot PR (например, обновление npm token), учитывайте лимит `open-pull-requests-limit` в `.github/dependabot.yml`.

**Критичность:** Низкая

### Проблема 3.1: Процедура ротации выполняется вручную (5 шагов)

**Где:** Секция "5. Ротация → Процедура ротации" (строки 200-206)

**Сейчас:**
> 1. **Создать новый секрет** в провайдере (новый ключ/пароль)
> 2. **Обновить секрет в GitHub** (`gh secret set ...`)
> 3. **Проверить** — запустить workflow с новым секретом
> 4. **Отозвать старый** секрет в провайдере
> 5. **Документировать** дату ротации

**Существует:**
CLI команды для автоматизации шагов 2, 3:
- `gh secret set {NAME} --body "{value}"` (шаг 2)
- `gh workflow run {workflow}.yml` (шаг 3)

**Рекомендация:**
Создать скрипт `.github/.instructions/actions/security/.scripts/rotate-secret.py` для автоматизации шагов 2-5:

```python
# Использование:
# python rotate-secret.py --secret-name DB_POSTGRES_PASSWORD --new-value "..."

1. Обновить секрет через gh CLI
2. Запустить smoke-test workflow
3. Ждать результата (timeout 5 минут)
4. Если успех — вывести сообщение "Ротация завершена. Отзовите старый секрет в провайдере."
5. Добавить запись в changelog: .github/secrets-rotation-log.json
```

Добавить в секцию "Скрипты":

> | Скрипт | Назначение |
> |--------|------------|
> | [rotate-secret.py](./.scripts/rotate-secret.py) | Автоматизация ротации секрета (шаги 2-5) |

**Критичность:** Средняя (ручная ротация подвержена ошибкам, но не блокирует работу)

## 4. Полнота описания объекта

### 4.1. Не хватает: Обработка конфликтов имён секретов

**Что отсутствует:**
Описание поведения при дублировании имён секретов на разных уровнях (Repository + Environment + Organization).

**Почему важно:**
При создании секрета `DATABASE_URL` на уровне Repository и Environment workflow может получить неожиданное значение. LLM может создать дубликат, не понимая приоритета.

**Стоит ли добавлять:** Да (критично для корректного использования)

**Что даст:**
Устранит риск неправильной интерпретации приоритета секретов (см. Проблему 2.1).

**Где добавить:**
Секция "3. Уровни хранения" → новая подсекция "Приоритет секретов".

### 4.2. Не хватает: Процедура удаления секрета

**Что отсутствует:**
Документ описывает создание, обновление, ротацию, но НЕ описывает удаление секрета.

**Почему важно:**
Секреты могут устаревать (сервис больше не используется, секрет заменён на другой). LLM может оставить "мёртвые" секреты в репозитории.

**Стоит ли добавлять:** Да

**Что даст:**
Полный жизненный цикл секрета (создание → использование → ротация → удаление).

**Рекомендация:**
Добавить подсекцию "Удаление секрета" в секцию "5. Ротация":

> **Удаление секрета:**
> 1. Проверить, используется ли секрет в workflows: `grep -r "secrets.{NAME}" .github/workflows/`
> 2. Если используется — удалить ссылки из workflows
> 3. Удалить секрет: `gh secret delete {NAME}`
> 4. Отозвать секрет в провайдере (если применимо)

**Критичность:** Средняя

### 4.3. Не хватает: Примеры для Environment Secrets (разные среды)

**Что отсутствует:**
Секция "3. Уровни хранения → Environment Secrets" описывает СИНТАКСИС, но не приводит полный пример workflow с несколькими средами.

**Почему важно:**
LLM может не понять, как workflow выбирает секрет из нужной среды (production vs staging).

**Стоит ли добавлять:** Да

**Что даст:**
Понимание использования Environment secrets в реальном workflow.

**Рекомендация:**

```yaml

  deploy-staging:

    environment: staging

      - run: echo "DB=${{ secrets.DATABASE_URL }}"  # Значение из staging

  deploy-production:
    runs-on: ubuntu-latest
    environment: production
    steps:
      - run: echo "DB=${{ secrets.DATABASE_URL }}"  # Значение из production
```

## 5. Слепые зоны автора

### Зона 5.1: Ротация секретов через автоматизированный workflow (scheduled)

**Суть:**
Документ описывает ручную ротацию секретов (5 шагов), но не рассматривает автоматизацию через GitHub Actions scheduled workflow.

**Почему важно:**
При 50+ секретах с разными периодами ротации (90/180/365 дней) ручная ротация становится неуправляемой. Можно создать workflow, который:
- Проверяет дату последнего обновления секрета (через GitHub API)
- Создаёт Issue с напоминанием о ротации (за 7 дней до истечения)
- Опционально: интеграция с Vault/1Password для автоматической генерации новых секретов

**Расширение качества:** Значительное (для проектов с большим количеством секретов)

**Рекомендация:** Добавить сейчас (опционально, но полезно для enterprise-проектов)

**Пример добавления:**

```markdown
### Автоматизация ротации (опционально)

Для проектов с 10+ секретами рекомендуется создать scheduled workflow:

**Файл:** `.github/workflows/secret-rotation-reminder.yml`

```yaml
name: Secret Rotation Reminder

on:
  schedule:
    - cron: "0 9 * * 1"  # Каждый понедельник 09:00 UTC

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - run: |
          # Проверить даты обновления секретов через API
          # Создать Issue для секретов, которые нужно ротировать
          gh api repos/:owner/:repo/actions/secrets --jq '.secrets[] | select(.updated_at < "2025-11-01") | .name'
```

**Альтернатива:** Интеграция с Vault/1Password API для полностью автоматической ротации.
```

**Критичность:** Опционально (полезно, но не критично для базовых сценариев)

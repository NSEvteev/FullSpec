## Контекст

Этот драфт — детализация **Фазы 5** из `align-structure-instructions.md`, но с расширенным скоупом: не только скрипты, а весь объект "Инструкции".

### Иерархия компонентов (из align-structure-instructions.md)

```
┌─────────────────────────────────────────────────────────┐
│                    ИНСТРУКЦИИ (SSOT)                    │
│                                                         │
│  ┌──────────┐  ┌───────────┐  ┌─────────┐  ┌─────────┐ │
│  │ Стандарт │  │ Валидация │  │Создание │  │Изменение│ │
│  │(справка) │  │(воркфлоу) │  │(воркфлоу)│  │(воркфлоу)│ │
│  └────┬─────┘  └─────┬─────┘  └────┬────┘  └────┬────┘ │
└───────┼──────────────┼─────────────┼───────────┼───────┘
        │              │             │           │
        │              ▼             ▼           ▼
        │         ┌──────────────────────────────────────┐
        │         │              СКРИПТЫ                 │
        │         │      (автоматизация рутины)          │
        │         └──────────────────────────────────────┘
        │              │             │           │
        │              ▼             ▼           ▼
        │         ┌──────────────────────────────────────┐
        │         │              СКИЛЛЫ                  │
        │         │   (триггер + параметры + → SSOT)     │
        │         └──────────────────────────────────────┘
        │              │             │           │
        ▼              ▼             ▼           ▼
┌─────────────────────────────────────────────────────────┐
│                       RULES                             │
│              (автозагрузка по paths)                    │
│                                                         │
│  - Ссылаются на скиллы (для воркфлоу)                  │
│  - Могут ссылаться на стандарты (справочно)            │
└─────────────────────────────────────────────────────────┘
```

### Принцип связей

| Тип инструкции | Скрипт | Скилл | Rule ссылается на |
|----------------|--------|-------|-------------------|
| Стандарт | — | — | Стандарт напрямую |
| Валидация | ✅ | ✅ | Скилл |
| Создание | ✅ | ✅ | Скилл |
| Изменение | ✅ | ✅ | Скилл |

**Принцип:** Каждый уровень ссылается на предыдущий, не дублирует.

### Облегчённый формат скиллов

Скилл **НЕ дублирует** шаги из инструкции. Скилл содержит только:
- Frontmatter с триггерами
- Ссылка на SSOT-инструкцию
- Формат вызова
- "→ Выполнить шаги из SSOT"
- Примеры

## 1. Проблема

Текущая папка `/.instructions/` не соответствует целевой концепции документирования объектов.

### Текущее состояние (устаревшее)

```
/.instructions/
├── README.md               # 11 секций (нестандартно)
├── structure.md            # Расположение
├── types.md                # Типы (standard/project)
├── validation.md           # Валидация
├── statuses.md             # Статусы в README
├── workflow.md             # Жизненный цикл (обзор)
├── workflow-create.md      # CREATE
├── workflow-update.md      # UPDATE
├── workflow-deactivate.md  # DEACTIVATE
├── patterns.md             # Паттерны поиска
├── examples.md             # Правила примеров
└── standard-instruction.md # Стандарт формата
```

**Проблемы:**
1. **Имена файлов** — нет типизированных префиксов (standard-, validation-, create-, modify-)
2. **README** — 11 секций вместо 5 фиксированных
3. **Смешение** — стандарты и воркфлоу на одном уровне без структуры
4. **Дублирование** — workflow.md + отдельные workflow-*.md
5. **Устаревшие концепции** — statuses.md (система статусов в README не используется)

## 6. Воркфлоу создания папки инструкций

При создании **любой** папки инструкций (не только `/.instructions/`) используется следующий воркфлоу:

### Шаги

```
1. Определить объекты
   └── Какие объекты документируются в этой папке инструкций?

2. Создать драфт
   └── Обсудить объекты и их свойства
   └── Определить какие файлы нужны (4 или меньше на объект)

3. Создать дерево документов
   └── Для каждого объекта создать все его файлы одним скопом
   └── (standard-, validation-, create-, modify-)

4. Ревью решения
   └── Все ли вопросы учтены?
   └── Нужны ли скрипты для процессов?

5. Создать скрипты
   └── Для валидации, создания, модификации
   └── Только если нужны

6. Повторить ревью (шаг 4)
   └── Пока не нужны новые скрипты

7. Создать скиллы
   └── Облегчённый формат (→ SSOT)

8. Создать rules
   └── Для автозагрузки по paths
```

### Диаграмма

```
┌──────────────────┐
│ 1. Определить    │
│    объекты       │
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 2. Создать драфт │
│    (обсуждение)  │
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 3. Создать       │
│    документы     │
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 4. Ревью         │◄────────┐
└────────┬─────────┘         │
         ▼                   │
    ┌─────────┐              │
    │ Нужны   │── Да ──►┌────┴───────────┐
    │скрипты? │         │ 5. Создать     │
    └────┬────┘         │    скрипты     │
         │ Нет          └────────────────┘
         ▼
┌──────────────────┐
│ 7. Создать       │
│    скиллы        │
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 8. Создать       │
│    rules         │
└──────────────────┘
```

#### Скрипт с субкомандами

```python
#!/usr/bin/env python3
"""
crud.py — Управление объектами.

Подкоманды:
    add     Добавить объект
    delete  Удалить объект

Использование:
    python crud.py add <name> --description "Desc"
    python crud.py delete <name>

Примеры:
    python crud.py add item --description "New item"
    python crud.py delete old-item

Возвращает:
    0 — успех
    1 — ошибка
"""

import argparse
import sys


def cmd_add(name: str, description: str) -> bool:
    print(f"✅ Добавлен: {name}")
    return True


def cmd_delete(name: str) -> bool:
    print(f"✅ Удалён: {name}")
    return True


def main():
    if sys.platform == "win32":
        sys.stdout.reconfigure(encoding="utf-8")

    parser = argparse.ArgumentParser(description="Управление объектами")
    subparsers = parser.add_subparsers(dest="command", required=True)

    # add
    add_p = subparsers.add_parser("add", help="Добавить объект")
    add_p.add_argument("name", help="Имя объекта")
    add_p.add_argument("--description", "-d", required=True)

    # delete
    del_p = subparsers.add_parser("delete", help="Удалить объект")
    del_p.add_argument("name", help="Имя объекта")

    args = parser.parse_args()

    if args.command == "add":
        success = cmd_add(args.name, args.description)
    elif args.command == "delete":
        success = cmd_delete(args.name)

    sys.exit(0 if success else 1)


if __name__ == "__main__":
    main()
```

```python
        help="Корень репозитория (по умолчанию: текущая папка)"
    )

    args = parser.parse_args()

    repo_root = find_repo_root(Path(args.repo))
    input_path = Path(args.input)

    success = process(input_path)
    sys.exit(0 if success else 1)


if __name__ == "__main__":
    main()
```

#### standard — стандарты

**Характеристики:**
- Описывают КАК делать
- Универсальны для разных проектов
- Можно переиспользовать без изменений
- Содержат правила и паттерны

#### project — специфика проекта

**Характеристики:**
- Описывают ЧТО есть в проекте
- Специфичны для данного проекта
- Требуют заполнения при инициализации
- Содержат конфигурации и структуры

**Примеры:** окружения, деплой, контракты

#### Как определить тип

**Вопрос:** Можно ли использовать эту инструкцию в другом проекте без изменений?

| Ответ | Тип |
|-------|-----|
| Да, это универсальные правила | `standard` |
| Нет, это специфика этого проекта | `project` |

### 19.3. Секция 2: Деактивация инструкции

#### Принцип

**Инструкции НЕ удаляются. Они деактивируются.**

- Сохраняется история
- Можно реактивировать
- Ссылки остаются валидными (с пометкой)

#### Когда деактивировать

- Инструкция больше не актуальна
- Технология/практика не используется
- Инструкция заменена другой

#### Шаги деактивации

```
1. Найти инструкцию
   └── {область}/.instructions/{name}.md

2. Найти все ссылки
   └── Использовать паттерны поиска (секция 4)

3. Добавить предупреждение
   └── После frontmatter

4. Закомментировать содержимое
   └── HTML-комментарий <!-- ... -->

5. Обновить ссылающиеся документы
   ├── Удалить из related
   ├── Пометить в README
   └── Обновить скиллы

6. Сохранить файл
   └── Frontmatter остаётся незакомментированным
```

#### Формат деактивированной инструкции

```markdown

>
> Содержимое закомментировано. Для реактивации — раскомментировать.

<!--

-->
```

### 19.5. Секция 4: Паттерны поиска ссылок

#### Где искать

| Место | Glob-паттерн |
|-------|--------------|
| Инструкции | `**/.instructions/**/*.md` |
| Скиллы | `/.claude/skills/*/SKILL.md` |
| README | `**/README.md` |
| Агенты | `/.claude/agents/*.md` |
| Драфты | `/.claude/drafts/*.md` |

#### Типы ссылок

**Markdown-ссылка:**
```
[текст](путь/к/инструкции.md)
```

**Frontmatter related:**
```yaml
related:
  - путь/к/инструкции.md
```

**Таблица README:**
```
| [имя.md](./путь/имя.md) | описание |
```

#### Команды поиска

```bash
# Поиск в markdown
grep -r "instruction-name.md" --include="*.md"

# Поиск в скиллах
grep -r "instruction-name" .claude/skills/

# Поиск в frontmatter
grep -A 5 "related:" **/*.md | grep "instruction-name"
```

#### Действия при обнаружении

| Тип ссылки | При деактивации | При миграции |
|------------|-----------------|--------------|
| Markdown | Удалить/пометить | Обновить путь |
| related | Удалить из массива | Обновить путь |
| README таблица | Пометить неактивной | Обновить путь |
| Скилл | Добавить пометку | Обновить путь |

- Разовое действие

#### 19.5.2. Секция 1: Когда валидировать

**Обязательно:**
- Перед коммитом нового скилла
- После изменения скилла
- При code review

**Опционально:**
- При рефакторинге
- В CI/CD pipeline

#### 19.5.3. Секция 2: Что проверяется

**Frontmatter:**

| Проверка | Обязательно | Описание |
|----------|:-----------:|----------|
| `name` | ✅ | Присутствует, kebab-case |
| `description` | ✅ | Присутствует, не пустое |
| `allowed-tools` | ✅ | Присутствует |
| `category` | ✅ | Одна из: documentation, code, testing, deployment |
| `triggers.commands` | ✅ | Минимум 1 команда |
| `triggers.phrases` | ✅ | ru и en присутствуют |

**Структура:**

| Проверка | Обязательно | Описание |
|----------|:-----------:|----------|
| Заголовок H1 | ✅ | Единственный `#` |
| SSOT-ссылка | ✅ | `**SSOT:**` после заголовка |
| Формат вызова | ✅ | Секция с таблицей параметров |
| Воркфлоу | ✅ | Секция с "→ Выполнить из SSOT" |
| Чек-лист | ✅ | Ссылка на SSOT |
| Примеры | ✅ | Минимум 1 пример |

| Проверка | Обязательно | Описание |

| Нет логики | Скилл НЕ содержит алгоритмов и условий |
| Размер | < 80 строк |

#### 19.6.1. Структура документа

```markdown
# Создание скилла

Пошаговый процесс создания нового скилла в сокращённом формате.

## Оглавление

- [5. Чек-лист](#5-чек-лист)
```

#### 19.6.2. Секция 1: Когда создавать

**Создавать скилл когда:**

| Критерий | Условие |
|----------|---------|
| SSOT-инструкция | ✅ Существует |
| Воркфлоу | ✅ Есть в инструкции |
| Повторяемость | > 2 раз |

**НЕ создавать скилл когда:**

- ❌ Нет SSOT-инструкции → сначала создать инструкцию
- ❌ Разовое действие
- ❌ Простое действие без воркфлоу

**Порядок создания:**

```
1. Инструкция (SSOT)     ← Сначала
2. Скилл (триггер)       ← Потом
```

#### 19.6.3. Секция 2: Подготовка

**2.1. Проверить SSOT-инструкцию**

```
Вопросы:
1. Инструкция существует?
2. Есть секция "Воркфлоу" или "Шаги"?
3. Есть секция "Чек-лист"?
```

Если нет → создать инструкцию через `/instruction-create`.

**2.2. Определить имя скилла**

Формат: `{object}-{action}`

| Объект | Действие | Имя скилла |
|--------|----------|------------|
| structure | create | `structure-create` |
| instruction | update | `instruction-update` |
| links | validate | `links-validate` |

**2.3. Определить категорию**

| Категория | Когда использовать |
|-----------|-------------------|
| `documentation` | Работа с документацией |
| `code` | Работа с кодом |
| `testing` | Тестирование |
| `deployment` | Деплой |

**2.4. Определить триггеры**

- Команда: `/{skill-name}`
- Фразы ru: 2-3 варианта
- Фразы en: 1-2 варианта

#### 19.6.4. Секция 3: Шаги создания

```
1. Проверить SSOT-инструкцию
   └── Существует, есть воркфлоу и чек-лист

2. Проверить уникальность
   └── Скилл с таким именем не существует

3. Создать папку
   └── /.claude/skills/{skill-name}/

4. Создать SKILL.md
   └── По шаблону (секция 4)

5. Заполнить frontmatter
   ├── name, description
   ├── allowed-tools, category
   └── triggers (commands, phrases)

6. Добавить SSOT-ссылку
   └── **SSOT:** [инструкция](путь)

7. Заполнить "Формат вызова"
   └── Синтаксис + таблица параметров

8. Заполнить "Воркфлоу"
   └── Предупреждение + "→ Выполнить из SSOT"

9. Заполнить "Чек-лист"
   └── Ссылка на SSOT#чек-лист

10. Добавить примеры
    └── 2-3 примера вызова

11. Валидировать
    └── python validate-skill.py {name}

12. Обновить /.claude/skills/README.md
    └── Добавить скилл в таблицу
```

#### 19.6.5. Секция 4: Шаблон

```markdown
      - {фраза 2}
    en:

**SSOT:** [{инструкция}.md]({путь к SSOT-инструкции})

## Формат вызова

```

```

| Параметр | Описание | Обязательный |
|----------|----------|--------------|
| `<параметр>` | {описание} | Да |
| `--опция` | {описание} | Нет |

## Воркфлоу

> ⚠️ **Перед выполнением** прочитать [{инструкция}.md]({путь})

→ Выполнить шаги из SSOT-инструкции.

## Чек-лист

→ См. [{инструкция}.md#чек-лист]({путь}#чек-лист)

## Примеры

```


#### 19.7.2. Секция 1: Обновление скилла

**Когда обновлять:**
- Изменился путь к SSOT-инструкции
- Изменились параметры вызова
- Добавляются новые триггеры

**Шаги обновления:**

```
1. Найти скилл
   └── /.claude/skills/{name}/SKILL.md

2. Внести изменения
   ├── Frontmatter (если нужно)
   ├── SSOT-ссылка (если изменился путь)
   ├── Параметры (если изменились)
   └── Примеры (если изменились)

3. Валидировать
   └── python validate-skill.py {name}

4. Обновить README
   └── Если изменилось описание
```

**Что НЕ менять в скилле:**
- Логику (она в SSOT-инструкции)
- Шаги воркфлоу (они в SSOT)
- Чек-лист (он в SSOT)

#### 19.7.3. Секция 2: Удаление скилла

**Когда удалять:**
- SSOT-инструкция удалена
- Скилл заменён другим
- Функционал больше не нужен

**Шаги удаления:**

```
1. Найти все ссылки на скилл
   └── Паттерны поиска (секция 4)

2. Обновить ссылающиеся документы
   ├── Удалить упоминания
   └── Заменить на альтернативу

3. Удалить из README
   └── /.claude/skills/README.md

4. Удалить папку скилла
   └── rm -rf /.claude/skills/{name}/

5. Закоммитить
   └── "chore: удалён скилл {name} — {причина}"
```

#### 19.7.4. Секция 3: Миграция скилла

**Когда мигрировать:**
- Переименование скилла
- Изменение структуры (object-action)

**Шаги миграции:**

```
1. Найти все ссылки на старое имя
   └── Паттерны поиска (секция 4)

2. Создать новый скилл
   └── /skill-create {new-name}

3. Обновить все ссылки
   └── Заменить старое имя на новое

4. Удалить старый скилл
   └── rm -rf /.claude/skills/{old-name}/

5. Обновить README
   └── Удалить старый, проверить новый
```

#### 19.7.5. Секция 4: Паттерны поиска ссылок

**Где искать:**

| Место | Glob-паттерн |
|-------|--------------|
| Скиллы | `/.claude/skills/*/SKILL.md` |
| Инструкции | `**/.instructions/**/*.md` |
| README | `**/README.md` |
| CLAUDE.md | `/CLAUDE.md` |
| Rules | `/.claude/rules/*.md` |

**Типы ссылок:**

```markdown
# Markdown-ссылка
[/skill-name](/.claude/skills/skill-name/SKILL.md)

# Упоминание команды
`/skill-name`

# В таблице
| [/skill-name](path) | описание |
```

**Команды поиска:**

```bash
# Все упоминания
grep -r "skill-name" --include="*.md"

# Ссылки на SKILL.md
grep -r "skills/skill-name" --include="*.md"

# Команды
grep -r "/skill-name" --include="*.md"
```

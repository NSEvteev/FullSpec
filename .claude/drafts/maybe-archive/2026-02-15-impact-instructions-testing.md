### Итерация 1: Находки

**Тема:** API Rate Limiting (disc-0002 → impact-0001)

**Процесс:** Discussion создана вручную → WAITING, Impact создан через скрипт+заполнение → WAITING. Валидации прошли.

#### Находки по файлам

**specs/impact/README.md:**
- README Impact имел формат таблицы, не совпадающий со стандартом § 2. Стандарт: `ID | Документ | Статус | Parent Discussion | Design | Milestone | Описание`. Был: `# | Тема | Discussion | Статус | Сервисы`. **Рекомендация:** Привести формат README к стандарту или обновить стандарт (если формат README должен быть другим).

**create-impact.md:**
- Шаг 3-4: Инструкция ссылается на скрипт `create-impact-file.py`, но не описывает как заполнить `parent` из шаблона — скрипт создаёт файл с placeholder `discussion/disc-NNNN-topic.md`. LLM должен сам заменить это значение. **Рекомендация:** Добавить в шаг 5 явное указание: "Заменить placeholder `discussion/disc-NNNN-topic.md` на реальный путь parent Discussion из шага 1".
- Шаг 2 (Quick Scan): Не описано что делать, если overview.md — пустой шаблон (как в нашем случае). Стандарт говорит "Quick Scan всё равно выполняется", но create-impact.md не уточняет поведение при пустом overview. **Рекомендация:** Добавить в Шаг 2: "Если overview.md — пустой шаблон (заглушки), Quick Scan считается выполненным. LLM опирается на Discussion + Clarify."
- Нет шага по обновлению `children` в parent Discussion после создания Impact. **Рекомендация:** Добавить шаг (после Шаг 8 или в Шаг 8): "Обновить frontmatter parent Discussion — добавить Impact в `children`."

**create-impact-file.py:**
- Скрипт не принимает `--parent` параметр — нет возможности указать parent Discussion при создании. **Рекомендация:** Добавить опциональный `--parent` параметр, который заменит placeholder в шаблоне.

**validate-impact.py:**
- Скрипт не проверяет формат `parent` (regex `discussion/disc-NNNN-*.md`). Поле просто проверяется на непустоту и существование файла. **Рекомендация:** Добавить проверку формата parent path.
- Скрипт не проверяет что milestone в Impact совпадает с milestone в parent Discussion. **Рекомендация:** Добавить сверку milestone (прочитать parent, сравнить milestone).

**standard-impact.md:**
- § 5 "Разделы документа": Не указано что делать если Redis (DEP-2) не является "сервисом" — это инфраструктурный компонент. Зависимость `gateway → Redis` не является сервисной зависимостью в смысле SDD. **Рекомендация:** Уточнить в стандарте: зависимости от инфраструктуры (Redis, PostgreSQL, Kafka) включаются или нет? Если да — какой формат (тип = "infra"?).

**validation-impact.md:**
- Чек-лист § "Содержание": Нет пункта проверки "Резюме содержит scope, характер и risk areas" (не проверяется минимальное содержание Резюме). **Рекомендация:** Добавить проверку минимального содержания Резюме (не просто "не пуст", а содержит ключевые элементы).

**modify-impact.md:**
- Нет описания "Upward feedback при создании Impact → Discussion". Секция "Upward feedback при WAITING" описывает обратную связь от Design, но upward feedback при создании (Шаг 7 create-impact.md) — отдельный сценарий. **Рекомендация:** Verify, что upward feedback описан в create-impact.md (уже есть) и не дублируется в modify.

### Итерация 2: Находки

**Тема:** Email Notification System (disc-0003 → impact-0002)

**Процесс:** Discussion и Impact созданы, заполнены, валидация пройдена. Сценарий с async-зависимостями (queue) и внешним провайдером.

#### Находки по файлам

**create-impact.md (подтверждение итерации 1):**
- Подтверждено: parent placeholder не заполняется скриптом. LLM каждый раз должен вручную заменять `discussion/disc-NNNN-topic.md`. Рекомендация из итерации 1 актуальна.
- **Новая находка:** Шаг 5 (frontmatter) не указывает что нужно обновить `children` в parent Discussion. Discussion disc-0003 имеет `children: []`, а должна иметь ссылку на impact-0002. **Рекомендация:** Добавить в создание Impact шаг: обновить `children` в parent Discussion.

**standard-impact.md (подтверждение итерации 1):**
- Подтверждено: § 5 "Зависимости" — нет ясности с внешними зависимостями. В итерации 2: `DEP-1: notification-service → Email Provider (SendGrid/SES)` — это не SDD-сервис, а внешний провайдер. Формат "К сервису" не предусматривает внешние зависимости. **Рекомендация:** Добавить в стандарт примечание: зависимости от внешних сервисов (SMTP, cloud APIs) включаются в секцию "Зависимости" с пометкой "external" в колонке "Тип".
- **Новая находка:** Колонка "Тип" в зависимостях: `sync/async` — слишком узко. В итерации 2 используются `async (HTTP)` и `async (queue)`. Стандарт показывает только `sync (REST)` в примере. **Рекомендация:** Расширить допустимые значения типа: `sync (REST)`, `sync (gRPC)`, `async (queue)`, `async (HTTP)`, `async (event)` или сделать поле свободным текстом.

**validate-impact.py (подтверждение итерации 1):**
- Подтверждено: нет проверки milestone соответствия. Impact-0002 имеет milestone=v1.2.0, parent disc-0003 тоже v1.2.0, но скрипт это не верифицирует.
- **Новая находка:** Нет проверки "Резюме не пустое". Скрипт не проверяет минимальное содержание секции "Резюме". Если оставить Резюме с placeholder — валидация пройдёт. **Рекомендация:** Добавить проверку I033: "Резюме пустое или содержит только placeholder".

**create-impact-file.py:**
- Подтверждено: нет параметра `--parent`. LLM каждый раз вручную заменяет placeholder.

**create-impact.md (новые находки):**
- Нет шага "Обновить Discussion README" — колонка Impact в `specs/discussion/README.md` не обновляется при создании Impact. **Рекомендация:** Добавить шаг: "Обновить колонку Impact в `specs/discussion/README.md`".
- Шаг 11 (Отчёт): шаблон отчёта не включает "Parent Discussion". **Рекомендация:** Добавить поле "📄 **Parent Discussion:** `disc-NNNN-topic.md`".

### Итерация 3: Находки

**Тема:** User Activity Audit Logging (disc-0004 → impact-0003)

**Процесс:** Discussion и Impact созданы, заполнены, валидация пройдена. Сценарий с синхронными зависимостями (REST), scheduled-задачами, cross-cutting middleware и 4 сервисами.

#### Находки по файлам

**create-impact-file.py (подтверждение итераций 1-2):**
- Подтверждено (3/3): нет параметра `--parent`. Каждую итерацию LLM вручную заменяет placeholder `discussion/disc-NNNN-topic.md`. Это friction при каждом создании Impact.

**create-impact.md (подтверждение итераций 1-2):**
- Подтверждено (3/3): parent placeholder не заполняется скриптом.
- Подтверждено (3/3): нет шага обновления `children` в parent Discussion. disc-0004 имела `children: []` до ручного обновления.
- Подтверждено (2/3): Discussion README (колонка Impact) не обновляется инструкцией. disc-0002, disc-0003 имели `—` в колонке Impact до ручной правки.
- Подтверждено (2/3): шаблон отчёта (Шаг 11) не включает "Parent Discussion".

**standard-impact.md (подтверждение + новые находки):**
- Подтверждено (3/3): § 5 "Зависимости" — нет ясности с не-сервисными зависимостями. Итерация 1: Redis (инфра). Итерация 2: Email Provider (внешний). Итерация 3: S3 Cold Storage (инфра/cloud). Три разных паттерна — стандарт не покрывает ни один.
- Подтверждено (2/3): колонка "Тип" в зависимостях — де-факто свободный текст. Итерация 1: `sync (REST)`. Итерация 2: `async (HTTP)`, `async (queue)`. Итерация 3: `sync (REST)`, `async (scheduled)`. Стандарт показывает только `sync (REST)` в примере.
- **Новая находка:** § 5 "Компоненты" — критерий полноты описан по слоям (API-слой, бизнес-логика, слой данных, инфраструктура), но реальные компоненты именуются по функции (Audit Event Receiver, Audit Storage Manager и т.д.). Не противоречие, но нечёткость: критерий полноты — это чек-лист для проверки coverage, а не шаблон именования. **Рекомендация:** Уточнить в стандарте, что слои — чек-лист полноты ("рассмотрены все слои?"), а не шаблон именования компонентов.

**validate-impact.py (подтверждение итераций 1-2):**
- Подтверждено (3/3): нет проверки milestone соответствия с parent Discussion.
- Подтверждено (2/3): нет проверки содержательности "Резюме" (не пуст, не placeholder).
- **Новая находка:** `check_zone_responsibility()` использует узкие regex-паттерны: `(\\S+-сервис|сервис\\s+\\S+)\\s+отвечает\\s+за` и `(\\S+)\\s+будет\\s+отвечать\\s+за`. Проблема: (1) не ловит англоязычные конструкции ("is responsible for"); (2) не ловит конструкции без слова "сервис" ("gateway отвечает за routing"). **Рекомендация:** Расширить паттерны: добавить `\w+\s+отвечает\s+за` (без привязки к "сервис"), добавить англоязычные: `is responsible for`, `will handle`, `owns`.
- **Новая находка:** Скрипт не проверяет формат колонки "Тип" в зависимостях. Любое значение (включая пустое или бессмысленное) пройдёт валидацию. **Рекомендация:** Если стандарт определит допустимые значения "Тип" — добавить проверку. Если поле свободное — как минимум проверять непустоту.

**validation-impact.md:**
- **Новая находка:** Шаг 2 (Frontmatter) говорит "milestone — совпадает с milestone parent Discussion", но validate-impact.py эту проверку не реализует. Расхождение между manual и automated validation. **Рекомендация:** Синхронизировать — либо добавить проверку в скрипт, либо пометить в validation-impact.md что это manual-only проверка.

**modify-impact.md:**
- Подтверждено (из итерации 1): upward feedback при создании Impact → Discussion описан в create-impact.md (Шаг 7) и не дублируется в modify. Это корректно.

## Содержание

## A. Двойственные формулировки

LLM может интерпретировать неоднозначно — нужно уточнить формулировки.

### A1. Правило "СТОП" при проверке Code → Specs

**Где:** раздел 3, "Обратная связь Code → Specs"

**Цитата:**
> "Проверка продолжается вверх до уровня, где содержание не затронуто → СТОП"

**Проблема:** Правило предполагает, что если уровень N не затронут, то N-1 тоже не затронут. Это не всегда так.

**Пример:** код меняет алгоритм JWT с RS256 на ES256, но через библиотеку с совместимым API:
- Plan затронут (другая библиотека в зависимостях)
- План тестов **НЕ** затронут (тест-сценарии те же — "проверить что токен валиден")
- ADR **затронут** (алгоритм изменился: RS256 → ES256)

По текущему правилу LLM остановится на План тестов и **пропустит** ADR.

**Решение:** СТОП на первом незатронутом сохраняется, но с исключением: незатронутый План тестов не означает, что ADR не затронут — ADR проверять всегда. Выше ADR иерархия монотонно абстрактна, СТОП работает корректно.

**Статус:** ✅ решён — обновлены правило + решение #22. Сценарии сохранены с оригинальными СТОП

---

### A2. "Переписывается" — перегенерация или правки?

**Где:** раздел 3, "Фаза 2 — Разрешение"

**Цитата:**
> "каждый уровень последовательно переписывается с учётом новых реалий"

**Проблема:** LLM может интерпретировать как:
- (a) Полная перегенерация документа с нуля
- (b) Точечные правки затронутых секций

Разница принципиальна — (a) теряет ценный контент, (b) может пропустить связанные изменения.

**Решение:** "обновляется" — LLM читает **весь документ целиком** и вносит точечные правки в затронутые секции, сохраняя остальной контент. Чтение всего документа обязательно, чтобы не пропустить косвенно затронутые места в длинных документах.

**Статус:** ✅ решён — "переписывается" → "обновляется" + уточнение "читает весь документ" в Фазе 2 + 3 сценария

---

### A3. "Рабочие правки" — граница нечёткая

**Где:** раздел 3, "Обратная связь Code → Specs"

**Цитата:**
> "Если затронуты только Plan / План тестов — рабочие правки: LLM автономно обновляет"

**Проблема:** Кто решает, что затронут "только" Plan/План тестов? Сам LLM. Он же и проверяет. Ситуация "судья в своём деле" — LLM мотивирован не эскалировать до CONFLICT (это останавливает работу) и может занизить масштаб.

**Решение:** Связать с Code Map. Границы автономии LLM из `architecture/services/{svc}.md` используются как формальный критерий: Свободно → нет обратной связи, Флаг → рабочие правки (Plan/План тестов), CONFLICT → ADR+. LLM не "решает сам" — он сверяется с Code Map.

**Статус:** ✅ решён — в Фазе 1 добавлена таблица маппинга Code Map → уровень обратной связи

---

### A4. "Все неясности не закрыты" — нет критерия

**Где:** раздел 1, "LLM не угадывает — уточняет"

**Цитата:**
> "LLM задаёт уточняющие вопросы, пока все неясности не закрыты"

**Проблема:** LLM не может определить, что **все** неясности закрыты — он не знает того, чего не знает. Нет ни лимита вопросов, ни эвристики "когда достаточно". LLM может остановиться после 2 вопросов или задать 20.

**Решение:** LLM проходит по секциям шаблона из `standard-*.md` и для каждой определяет, достаточно ли контекста. Шаблон — неявный чек-лист для Clarify.

**Статус:** ✅ решён — уточнение добавлено в механизм Clarify

---

### A5. Кросс-цепочка: "ту же реакцию" для детей с разным статусом

**Где:** раздел 3, "Кросс-цепочечная обратная связь"

**Цитата:**
> "все его дочерние получают ту же реакцию (WAITING-дети → DRAFT, RUNNING-дети → CONFLICT и т.д.)"

**Проблема:** Дети одного документа могут быть в **разных** статусах (один WAITING, другой DRAFT, третий RUNNING). "Ту же реакцию" неоднозначно — реакцию **родителя** или реакцию, соответствующую **своему** текущему статусу?

**Решение:** Каждый ребёнок реагирует **по своему текущему статусу** согласно таблице реакций. WAITING-ребёнок → DRAFT (не CONFLICT), DRAFT-ребёнок → перегенерация.

**Статус:** ✅ решён — переформулировано в кросс-цепочечной обратной связи

---

### A6. "Информирует пользователя" — как?

**Где:** раздел 3, "Обратная связь Code → Specs"

**Цитата:**
> "LLM автономно обновляет документы, продолжает работу и информирует пользователя"

**Проблема:** Каким способом? Сообщение в чат? Комментарий в документе? Changelog? Не определено.

**Решение:** Сообщение в чат — краткое резюме изменений, не дожидаясь ответа. Естественный канал для Claude Code.

**Статус:** ✅ решён — уточнено в таблице Code Map → обратная связь

## B. Логические противоречия

Места, где правила документа противоречат друг другу.

### B1. RUNNING → REJECTED — переход отсутствует в диаграмме

**Где:** раздел 3, диаграмма статусов vs каскад REJECTED

**Проблема:** Диаграмма статусов показывает:
- `RUNNING → DONE`
- `RUNNING → CONFLICT`
- `CONFLICT → REJECTED`

Но **нет** `RUNNING → REJECTED`. При каскаде REJECTED от сиблинга — документ в RUNNING должен стать REJECTED, но этот переход не определён. LLM увидит, что такой переход "нелегален" по диаграмме.

Также: пользователь не может напрямую отклонить RUNNING-документ ("передумал") — только через CONFLICT → REJECTED. Это намеренно?

**Решение:** Добавлен статус ROLLING_BACK. RUNNING → ROLLING_BACK → REJECTED. Пользователь может инициировать ROLLING_BACK напрямую. Откат артефактов описан по уровням в новой секции "Откат артефактов (ROLLING_BACK)".

**Статус:** ✅ решён — добавлен ROLLING_BACK, диаграмма обновлена (7 статусов), решение #49

---

### B2. WAITING → DRAFT — переход отсутствует в диаграмме

**Где:** раздел 3, кросс-цепочечная обратная связь vs диаграмма статусов

**Проблема:** Кросс-цепочечная обратная связь определяет: `WAITING → DRAFT`. Но в диаграмме статусов есть только `WAITING → REJECTED`. LLM увидит противоречие.

**Решение:** Переход WAITING → DRAFT добавлен в диаграмму статусов при обновлении для ROLLING_BACK.

**Статус:** ✅ решён — диаграмма обновлена

---

### B3. DONE + REJECTED сиблинги — невозможно по правилам, возможно на практике

**Где:** раздел 3, каскад REJECTED + каскадное завершение

**Проблема:**
1. ADR auth → План тестов auth → Plan auth → DONE (вся цепочка завершена, architecture/ обновлён)
2. ADR gateway → План тестов gateway в RUNNING → CONFLICT → REJECTED
3. Каскад REJECTED: сиблинги ADR gateway → ADR auth должен стать REJECTED. Но он DONE!

`DONE → REJECTED` перехода нет. Живые документы уже обновлены от ADR auth. Что делать?

**Решение:** DONE → ROLLING_BACK → REJECTED. ROLLING_BACK откатывает артефакты (architecture/ возвращается к состоянию до ADR auth). Пример обновлён в каскаде REJECTED.

**Статус:** ✅ решён — DONE → ROLLING_BACK → REJECTED, откат артефактов по уровням

### B5. "ADR никогда не удаляется" vs "Очистка REJECTED"

**Где:** раздел 7, правило версионирования + очистка REJECTED

**Проблема:**
- "ADR никогда не удаляется — это история решений"
- "Раз в неделю LLM предлагает список REJECTED-документов для удаления"

REJECTED ADR — тоже ADR. Удаляем или нет?

**Решение:** DONE-ADR никогда не удаляется (история принятых решений). REJECTED-документы (включая REJECTED-ADR) подлежат очистке — отклонённые решения не часть истории.

**Статус:** ✅ решён

## C. Недостаточно описанные области

Области, где описание слишком кратко или отсутствует.

### C1. Greenfield / Cold Start

**Проблема:** Весь документ предполагает, что `architecture/`, `specs/tests/` уже существуют. Но при первой дискуссии в пустом проекте:
- Code Map ссылается на пакеты в `src/{svc}/` — а кода нет
- ADR делает дельту к `architecture/services/X.md` — а файла нет
- LLM "читает AS IS" — а читать нечего

**Нужно:** описать bootstrap-сценарий — как выглядит первый проход.

**Решение:** Добавлено уточнение "Greenfield" к правилу чтения для ADR: если файл не существует, вся дельта = ADDED, Code Map создаётся при первом ADR → DONE.

**Статус:** ✅ решён

---

### C2. Shared код

**Проблема:** CLAUDE.md упоминает `/shared/` (контракты API, события, общие библиотеки). Изменения в shared/ затрагивают несколько сервисов. Через чей ADR это проходит? Или нужен отдельный механизм?

**Решение:** shared/ — **не сервис**, `specs/services/shared/` не создаётся. Контракты описываются в Design (блоки взаимодействия). Создание файлов в shared/ — через ADR сервиса-провайдера (дельта ADDED/MODIFIED). Использование — через ADR сервисов-потребителей (внешняя зависимость). Code Map каждого сервиса указывает зависимости от shared/. Изменение контракта = CONFLICT уровня Design.

**Статус:** ✅ решён — новая секция "Shared код (/shared/)" в разделе 4, обновлён пример Code Map, решение #50

---

### C3. Mono-сервис проект

**Проблема:** Архитектура заточена под микросервисы. Что если сервис один? Impact всегда "один сервис", Design имеет одну секцию, ADR один. Уровни 2-3 (Impact, Design) вырождаются.

**Решение:** Не требует изменений. Impact уже анализирует компоненты, данные, API, риски — не только сервисы. Design уже описывает внутреннюю структуру сервиса (компоненты, зависимости). Mono-сервис — частный случай, где сервис один. Уровни не вырождаются, архитектура работает одинаково.

**Статус:** ✅ решён — не требует изменений (решение #18 покрывает)

---

### C4. Что триггерит проверку Code → Specs?

**Проблема:** "код может выявить несовместимость". Но **когда** и **кто** запускает проверку?
- LLM во время кодирования?
- При PR review?
- При мерже?
- При запуске тестов?

Не определён триггер.

**Решение:** Триггер — агент-разработчик непрерывно в процессе работы: при написании кода, запуске тестов, создании PR. Не отдельный шаг, а часть процесса разработки.

**Статус:** ✅ решён — уточнение добавлено в начало секции "Обратная связь Code → Specs"

---

### C5. CONFLICT и уже созданные GitHub Issues

**Проблема:** "Соответствующие Issues обновляются". Но **как**?
- Закрыть и создать новые?
- Отредактировать существующие?
- Что с Issues, у которых уже есть PR, ветки, код?

**Решение:** Специального механизма не нужно — существующие стандарты Issues покрывают все случаи. CONFLICT: неизменённые Issues остаются, изменённые обновляются (`gh issue edit`), удалённые закрываются (`not planned`), новые создаются. ROLLING_BACK: все Issues закрываются (`not planned`), feature-ветка удаляется, revert не нужен — код в main отсутствует (merge только после завершения всех задач дискуссии). SSOT-ссылки: [modify-issue.md](/.github/.instructions/issues/modify-issue.md), [standard-issue.md § 6](/.github/.instructions/issues/standard-issue.md#6-закрытие-issue), [create-issue.md](/.github/.instructions/issues/create-issue.md), [standard-github-workflow.md](/.github/.instructions/standard-github-workflow.md).

**Статус:** ✅ решён — таблица действий с SSOT-ссылками в "CONFLICT и задачи", ROLLING_BACK для Plan уточнён

### C6. Milestone во frontmatter

**Проблема:** Решение #44: "Milestone определяется при создании Discussion". Но пример frontmatter не содержит поля `milestone`. Где хранится?

**Решение:** `milestone` добавлен во frontmatter Discussion (только Discussion — остальные уровни наследуют). Раздел "Milestone" добавлен в таблицу разделов Discussion. При создании Issues Milestone берётся из frontmatter Discussion. Если объём задач не укладывается в дату Milestone — сдвигается дата того же Milestone (не создаётся новый).

**Статус:** ✅ решён — frontmatter обновлён, раздел Milestone в Discussion, корректировка даты в правиле создания Issues

### C7. Когда у плана тестов несколько Plans?

**Проблема:** Правило связей: "План тестов → children: План(ы)" — множественное число. Когда у одного плана тестов несколько планов? Не приведён пример.

**Решение:** План тестов → План (1:1). Множественное число убрано. Декомпозиция покрывается задачами внутри Plan. Если в будущем понадобится несколько планов — расширяемость сохраняется (решение #2).

**Статус:** ✅ решён — "План(ы)" → "План (1:1)" в таблице объектов и правилах связей

### C8. Planned Changes — не обогащаются по мере спуска

**Проблема:** Planned Changes создаются при Design → WAITING. Но когда ADR детализирует изменения (дельта-блоки ADDED/MODIFIED/REMOVED), Planned Changes не обновляются. Параллельная дискуссия Б видит только общее описание из Design, но не детали из ADR.

**Решение:** Обогащение не нужно. Planned Changes — навигационный указатель (ссылка на Design). LLM при необходимости переходит по ссылке → Design → дочерние ADR (через frontmatter children) для получения конкретных дельт. Обогащение не производится: дельты могут измениться при CONFLICT, дублирование нарушает SSOT. Из 6 уровней только ADR параллельной дискуссии нуждается в деталях — и получает их через 2 перехода по ссылкам.

**Статус:** ✅ решён — уточнение навигации добавлено в описание Planned Changes

## Содержание

### 1. Эволюция с момента исследования

Исследование (08.02) анализировало нашу систему с 7 статусами, 4 уровнями (Discussion → Impact → ADR → Plan), без Clarify, без CONFLICT-механизма. С тех пор архитектура прошла через **44 решения** и существенно изменилась:

| Аспект | Было (исследование) | Стало (архитектура) |
|--------|---------------------|---------------------|
| Уровни | 4 (Discussion → Impact → ADR → Plan) | **6** (+Design, +Test Spec) |
| Статусы | 7 (DRAFT → REVIEW → APPROVED → RUNNING → DONE/REJECTED/SUPERSEDED) | **6** (DRAFT → WAITING → RUNNING → DONE/CONFLICT/REJECTED) |
| Clarify | Нет | **На каждом уровне** + Dependency Barrier |
| Обратная связь Code → Specs | Нет | **Двухфазная** (обнаружение ↑, разрешение ↓) |
| Параллельные цепочки | Нет | **Planned Changes** + кросс-цепочечная обратная связь |
| Навигация по коду | Нет | **Code Map** в architecture/services/ |
| Технологические стандарты | Нет | **standard-{tech}.md** + автозагрузка через rules |

### 2. Сравнение с каждым фреймворком

#### 2.1. vs Habr (Венделев) — мультиагентная разработка

| Аспект | Habr | Наш подход | Преимущество |
|--------|------|-----------|--------------|
| **Агенты** | 12 специализированных (epic-writer, dev-plan-writer...) | 1 оркестратор + скиллы | **Наш** — проще поддерживать, единый контекст проекта |
| **Dev/Test разделение** | TaskForDev + TaskForTest | Test Spec (уровень 5) + Plan (уровень 6) | **Наш** — тест-сценарии определяются ДО плана (ATDD), а не параллельно |
| **Организация** | По датам `Doc/Backlog/YYYY/MM/` | По сервисам `services/{svc}/` | **Наш** — соответствует микросервисной архитектуре |
| **Итеративность** | Жёсткие стадии, нет возврата | CONFLICT-механизм, upward feedback | **Наш** — код может влиять на спецификации |
| **Статусы** | Нет | 6 + каскады (DONE, REJECTED, CONFLICT) | **Наш** — полный lifecycle management |
| **Что мы НЕ взяли** | Разделение `.ai/` и `.claude/`, привязка к PHP-стеку | — | Избыточно для нашего контекста |

**Итого:** Habr дал идею разделения задач на код и тесты — мы реализовали это глубже через отдельный уровень Test Spec (ATDD). Мультиагентность не взяли — один оркестратор с инструкциями для каждого объекта масштабируется лучше.

---

#### 2.2. vs GitHub Spec Kit

| Аспект | Spec Kit | Наш подход | Преимущество |
|--------|----------|-----------|--------------|
| **Constitution** | Единый `constitution.md` | Принципы распределены по `.instructions/` объектов | **Наш** — загружается только релевантное, не засоряет контекст |
| **Clarify** | `/speckit.clarify` (фиксированный набор) | Clarify на **каждом** уровне + `--auto-clarify` + Dependency Barrier | **Наш** — уточнение не разовое, а сквозное |
| **Checklists** | Отдельная папка `checklists/` | Встроены в `standard-*.md` каждого объекта | **Наш** — чек-лист = часть стандарта, не теряется |
| **Организация** | По фичам `specs/NNN-feature/` | По сервисам + по типам объектов | **Наш** — для микросервисов; Spec Kit лучше для монолитов |
| **Статусы** | Нет | 6 + каскады | **Наш** — Spec Kit не знает, в каком состоянии спецификация |
| **Каскады** | Нет | DONE, REJECTED, CONFLICT с чёткими правилами | **Наш** — автоматическое управление зависимостями |
| **Brownfield** | Нет | Planned Changes + живые документы architecture/ | **Наш** — AS IS/TO BE паттерн |
| **Constitution Gates** | Проверка Plan по Constitution | Каждый ADR проверяется по своему `standard-adr.md` | **Паритет** — разные механизмы, одна цель |

**Итого:** Spec Kit дал Clarify и идею Constitution. Мы взяли Clarify и усилили: на каждом уровне + Dependency Barrier. Constitution заменили распределёнными принципами — не загружаем 1000 строк принципов при работе с одним объектом.

---

#### 2.3. vs OpenSpec (Fission-AI)

| Аспект | OpenSpec | Наш подход | Преимущество |
|--------|---------|-----------|--------------|
| **Дельта-спеки** | ADDED/MODIFIED/REMOVED | Planned Changes в architecture/ | **OpenSpec** для brownfield лучше; **наш** для greenfield проще |
| **Specs как SSOT** | `openspec/specs/` обновляется через archive | `architecture/` обновляется при DONE | **Паритет** — оба поддерживают живой SSOT |
| **Given/When/Then** | RFC 2119 (SHALL/SHOULD/MAY) | G/W/T в Discussion | **Паритет** — оба используют структурированные требования |
| **Итеративность** | Артефакты обновляемы в любой момент | Чёткие переходы статусов + CONFLICT | **Наш** — предсказуемее; OpenSpec гибче, но хаотичнее |
| **Каскады** | Нет (Archive/Active) | 3 типа каскадов + кросс-цепочечная обратная связь | **Наш** — автоматическое управление зависимостями |
| **Impact Analysis** | Нет | Отдельный уровень 2 | **Наш** — явный анализ влияния |
| **Привязка к сервисам** | Плоская (по доменам) | По сервисам | **Наш** — для микросервисов |
| **YAML-схемы** | Кастомизируемые | Фиксированные 6 уровней | **OpenSpec** гибче; **наш** проще |
| **Что мы НЕ взяли** | Дельта-спеки, кастомные YAML | — | Overkill для greenfield |

**Итого:** OpenSpec — лучший для brownfield. Мы взяли идею живого SSOT (architecture/) и G/W/T формат. Дельта-спеки отложили (решение #5) — когда проект станет brownfield, можно добавить.

#### 2.4. vs Claude Task Master

| Аспект | Task Master | Наш подход | Преимущество |
|--------|------------|-----------|--------------|
| **Декомпозиция** | PRD → задачи (LLM авто) | Plan с авто-декомпозицией | **Паритет** — оба используют LLM |
| **Анализ сложности** | 1-10 с рекомендацией подзадач | Предусмотрен в Plan | **Task Master** зрелее — у нас ещё не реализовано |
| **find-next-task** | Алгоритм по зависимостям/приоритетам | Предусмотрен (скилл `/spec-next`) | **Task Master** зрелее |
| **Спецификации** | Нет (только PRD → задачи) | 6 уровней спецификаций | **Наш** — Task Master пропускает проектирование целиком |
| **ADR** | Нет | Уровень 4, привязан к сервисам | **Наш** — история архитектурных решений |
| **Обратная связь** | Нет | CONFLICT + двухфазная Code → Specs | **Наш** — код может влиять на спецификации |
| **Формат** | JSON (tasks.json) | Markdown | **Наш** — читаемее, git diff понятнее |
| **MCP** | Интеграция как tool | Скиллы Claude Code | **Паритет** — разные механизмы |
| **Статусы** | 7 (pending, in-progress, done, review, deferred, cancelled, blocked) | 6 (DRAFT, WAITING, RUNNING, DONE, CONFLICT, REJECTED) + каскады | **Наш** — каскады автоматизируют зависимости |

**Итого:** Task Master — лучший для task management (декомпозиция, сложность, find-next-task). Но он полностью пропускает **проектирование** — PRD сразу → задачи без Impact Analysis, Design, ADR. Мы взяли идеи авто-декомпозиции и testStrategy, но добавили 6 уровней спецификаций до задач.

### 3. Уникальные преимущества нашего подхода

Механизмы, которых **нет ни в одном** из 4 фреймворков:

| Механизм | Что делает | Почему уникально |
|----------|-----------|------------------|
| **CONFLICT-каскад** | Код выявляет проблему → двухфазное разрешение (↑↓) | Ни один фреймворк не обрабатывает обратную связь от кода |
| **Кросс-цепочечная обратная связь** | Обновление architecture/ проверяет ВСЕ другие цепочки | Параллельные спецификации координируются автоматически |
| **Dependency Barrier** | LLM останавливает генерацию при зависимости от неразрешённого маркера | Экономит токены, предотвращает переписывание |
| **Code Map + границы автономии** | LLM знает: свободно / флаг / CONFLICT | Явные правила "что можно менять без согласования" |
| **Planned Changes** | Параллельные дискуссии видят планируемые изменения | Предотвращает конфликты до начала разработки |
| **Каскад REJECTED с сайблингами** | REJECTED одного ADR → все сайблинги тоже REJECTED + родитель → DRAFT | Гарантирует целостность: частичный контекст невозможен |
| **6-уровневая иерархия с ATDD** | Test Spec определяется ДО Plan | Acceptance tests первичны на уровне спецификаций |

### 5. Слабые стороны нашего подхода — анализ и ответы

| # | Заявленная слабость | Наш ответ | Статус |
|---|---------------------|-----------|--------|
| 1 | **Task Master** зрелее в task management | **Проработано.** Анализ сложности (1-10) встроен в Plan, порядок задач = порядок выполнения, scope-adjustment покрыт CONFLICT-механизмом, интеграция с существующим standard-issue.md. Детали: [task-master-analysis.md](./2026-02-09-task-master-analysis.md) § 6 | Снято (решение #48) |
| 2 | **OpenSpec** лучше для brownfield (дельта-спеки) | **Не слабость + улучшено.** ADR теперь содержит формальные дельта-блоки ADDED/MODIFIED/REMOVED (решение #47). ADR = дельта-спека с обоснованием (контекст, trade-offs) + формальной разметкой. Лучше OpenSpec: у них дельта без обоснования, у нас — с обоснованием | Снято (решение #47) |
| 3 | **Spec Kit** проще для малых проектов | **Не слабость.** 6 уровней масштабируются: для малого проекта Discussion = 10 строк, Impact = "1 сервис", ADR = "X потому что Y". Структура одинаковая — объём масштабируется. Spec Kit с 3 уровнями потребует миграции при росте. Наша архитектура готова к росту изначально | Снято |
| 4 | **Habr** — изоляция контекста агентов | **Запланировано.** Специализированные агенты на этапах после Discussion: агент-импакт, агент-дизайнер, агент-ADR, агент-тестировщик, агент-планировщик, агент-разработчик. Каждый читает свой уровень + входные данные от родителя | Снято (решение #45) |
| 5 | **Тяжеловесность** — 6 уровней для баг-фикса | **Намеренное решение.** Управляемость > скорость. Мелкие баги группируются в одну дискуссию ("Баг-фиксы спринта 12"), что убирает overhead. Полнота документации гарантирована | Снято (решение #46) |

### 6. Task management: наше решение

По результатам анализа Task Master ([task-master-analysis.md](./2026-02-09-task-master-analysis.md)) определён подход к управлению задачами в Plan.

**SSOT:** [specs-architecture.md § 7 — Формат задач в Plan, Маппинг Plan → GitHub Issues](../examples/2026-02-08-specs-architecture.md#формат-задач-в-plan)

#### 6.1. Что взяли из Task Master

| Механизм Task Master | Наша адаптация |
|----------------------|----------------|
| **Анализ сложности** (отдельный шаг, AI, report.json) | Встроен в генерацию Plan. Отдельный шаг не нужен — контекст из 5 уровней |
| **Авто-декомпозиция** (expand-task, expansionPrompt) | LLM декомпозирует при генерации Plan. expansionPrompt не нужен — есть ADR, Test Spec, Design |
| **find-next-task** (детерминистический алгоритм) | Порядок задач в Plan = порядок выполнения |
| **scope-adjustment** (up/down с сохранением done) | Покрыт CONFLICT-механизмом |
| **testStrategy на задаче** | Ссылка на конкретные сценарии из Test Spec |
| **Зависимости < ID** | Задача зависит только от задач с меньшим номером |

#### 6.2. Почему наш подход лучше Task Master для наших целей

| Аспект | Task Master | Наш подход |
|--------|------------|-----------|
| **Контекст при декомпозиции** | PRD → задачи (минимальный контекст) | 5 уровней спецификаций → Plan (полный контекст) |
| **Отдельный шаг анализа** | Нужен (нет контекста) | Не нужен (контекст есть) |
| **Отдельный report.json** | Нужен (связь между шагами) | Не нужен (всё в одном Plan) |
| **scope-adjustment** | Отдельный механизм | Покрыт CONFLICT |
| **Связь с тестами** | testStrategy (текст) | Ссылка на Test Spec (формальный уровень) |
| **Связь с архитектурой** | Нет | Дельта-блоки из ADR |
| **Issues** | Нет интеграции | Полная интеграция с standard-issue.md |

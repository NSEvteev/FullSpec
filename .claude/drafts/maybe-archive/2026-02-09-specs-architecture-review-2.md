## A. Двойственные формулировки

LLM может интерпретировать неоднозначно — нужно уточнить формулировки.

### A1. DONE-ребёнок при CONFLICT родителя

**Где:** раздел 3, каскад CONFLICT (строка 372) + диаграмма статусов (строки 194-222)

**Проблема:** Каскад REJECTED подробно описывает поведение по статусу каждого ребёнка (строки 319-321):
- RUNNING/DONE → ROLLING_BACK → REJECTED
- DRAFT/WAITING → REJECTED напрямую

Но каскад CONFLICT (строка 372) говорит только:
> "CONFLICT каскадирует **только вниз** (на детей)"

**Без разбивки по текущему статусу детей.** При этом диаграмма статусов допускает `DONE → ROLLING_BACK`, но **не** `DONE → CONFLICT`.

**Пример:** Design (RUNNING) → CONFLICT из-за Code → Specs. ADR auth уже DONE (architecture/ обновлён), ADR gateway ещё RUNNING.

| Ребёнок | Текущий статус | Что делать? |
|---------|---------------|-------------|
| ADR gateway | RUNNING | → CONFLICT (по диаграмме ✅) |
| ADR auth | DONE | → ??? (DONE → CONFLICT нет в диаграмме ❌) |

LLM не знает: DONE-ребёнок → ROLLING_BACK? → CONFLICT (нужен новый переход)? Или не затрагивается?

**Решение:** Добавлен переход DONE → CONFLICT в диаграмму. DONE-документ переходит в CONFLICT при каскаде от родителя. Артефакты уже применены — обновляются на месте при разрешении (Фаза 2), без отката. Добавлена таблица каскада CONFLICT по статусу ребёнка (аналогично REJECTED): DRAFT → перегенерация, WAITING → DRAFT, RUNNING → CONFLICT, DONE → CONFLICT, REJECTED → игнор. После разрешения: CONFLICT → RUNNING → DONE.

**Статус:** ✅ решён — диаграмма, таблица каскада, Фаза 2, решение #51

### A2. Момент создания Issues — два противоречивых описания

**Где:** раздел 3, прямой поток (строка 264) vs раздел 7 (строка 977)

**Цитата 1 (прямой поток, шаг 7):**
> "Пользователь подтверждает создание GitHub Issues + Milestone"
— как часть перехода в RUNNING

**Цитата 2 (раздел 7, правило создания Issues):**
> "GitHub Issues и Milestones создаются **только отдельной командой**, **только после подтверждения плана**"

**Проблема:** Первое описание — Issues создаются одновременно с переходом в RUNNING. Второе — Issues создаются отдельной командой после подтверждения. LLM получит противоречивые инструкции о том, когда именно создавать Issues.

**Решение:** Шаг 7 прямого потока уточнён: "Пользователь **отдельной командой** запускает создание GitHub Issues + Milestone" со ссылкой на раздел 7. Issues создаются после перехода в RUNNING, не одновременно.

**Статус:** ✅ решён — прямой поток шаг 7 приведён в соответствие с разделом 7

### A3. Upward feedback: механика возврата родителя в DRAFT

**Где:** раздел 4, "Обновление вышестоящих уровней" (строки 788-802)

**Цитата:**
> "При работе на уровне N может обнаружиться информация, затрагивающая уровень N-1 или выше. Обновление **обязательно**"

**Проблема:** Таблица описывает **что** обновить (ADR → Дизайн, Дизайн → Импакт), но не **механику**:

1. ADR (DRAFT) обнаруживает проблему в Design (WAITING)
2. Design WAITING → DRAFT? (переход есть в диаграмме)
3. ADR приостанавливается до повторного WAITING родителя?
4. Пользователь ревьюит обновлённый Design заново?
5. Или LLM правит Design "тихо" (как рабочие правки)?

Без описания механики LLM может: (a) молча поправить WAITING-документ, (b) вернуть его в DRAFT и ждать ревью, (c) продолжить ADR, проигнорировав проблему.

**Решение:** Формальный возврат. N-1 (WAITING) → DRAFT, работа с N приостанавливается. LLM обновляет N-1, пользователь ревьюит → N-1 → WAITING, работа с N возобновляется. Если каскад выше — аналогично до точки остановки.

**Статус:** ✅ решён — механика добавлена в "Обновление вышестоящих уровней"

### A4. Задачи: последовательно или параллельно?

**Где:** раздел 7, "Формат задач в Plan" (строки 1030-1032)

**Цитата:**
> "Порядок задач = порядок выполнения. Агент-кодер получает ссылку на Plan и идёт сверху вниз, проверяя зависимости."

**Проблема:** Если задача 1 и задача 2 независимы (нет зависимости) — должен ли агент выполнить 1, **потом** 2 строго последовательно? Или может параллельно?

- "Сверху вниз" → строго последовательно
- "Проверяя зависимости" → если зависимостей нет, можно параллельно

LLM-агент может интерпретировать как разрешение на параллелизм (если зависимостей нет) или как запрет (строго сверху вниз).

**Решение:** Строго последовательно сверху вниз. Если задача заблокирована незавершённой зависимостью — пропустить, взять следующую незаблокированную. Параллельное выполнение не предусмотрено.

**Статус:** ✅ решён — уточнено в "Формат задач в Plan"

### A5. Триггер перехода в RUNNING

**Где:** раздел 3, прямой поток (строки 262-265)

**Цитата:**
> "Когда ВСЕ планы в WAITING: ВСЕ документы в цепочке переходят в RUNNING"

**Проблема:** Кто проверяет условие "все планы в WAITING"?
- Автоматически после каждого Plan → WAITING?
- Пользователь явно даёт команду?
- Оркестратор?

Без явного триггера LLM может не проверить (ждать команды) или проверять слишком часто (после каждого шага).

**Решение:** LLM проверяет после каждого Plan → WAITING. Когда все Plans в WAITING — предлагает переход через AskUserQuestion. Пользователь может подтвердить (→ RUNNING) или отложить (остаётся WAITING).

**Статус:** ✅ решён — триггер и AskUserQuestion добавлены в "Каскад RUNNING"

### B2. Очистка REJECTED: триггер "понедельник"

**Где:** раздел 3, кросс-цепочечная обратная связь (строка 489)

**Цитата:**

### B7. Cross-plan зависимости задач при создании

**Где:** раздел 4, "Shared код" (строка 782) + раздел 7

**Цитата:**
> "Задача 'создать схему в shared/' (из Plan провайдера) блокирует задачи 'реализовать обработчик' (из Plan потребителей). Зависимость через `**Зависит от:** #N` в GitHub Issues"

**Проблема:** Plans создаются последовательно (по сервисам, сверху вниз). При создании Plan auth — Plan notifications ещё не существует. Как LLM при создании Plan notifications узнает номер задачи из Plan auth, чтобы проставить `Зависит от: #N`?

Варианты:
- Plan notifications ссылается на задачу Plan auth по имени (а не по Issue #)?
- При создании Issues (после всех Plans) LLM проставляет кросс-ссылки?
- Plans создаются параллельно и кросс-ссылки ставятся post-hoc?

Механизм не описан.

**Решение:** В документе Plan зависимость указывается по имени задачи из другого Plan: `Plan auth → Задача 1 "Создать схему UserCreatedEvent"`. При создании Issues (после RUNNING) LLM превращает именные ссылки в `#N` — к этому моменту все Plans существуют и Issues создаются за один проход.

**Статус:** ✅ решён — уточнение добавлено в поле "Зависимости" таблицы полей задачи

## D. Не описано

Области, которые в документе отсутствуют полностью.

### D1. Обратная связь от тестов к Test Spec

**Проблема:** Test Spec определяет "что тестировать". План определяет "как реализовать". Агент пишет код и тесты. Часть тестов упала. Далее:

1. Как LLM связывает **результаты тестов** с конкретными сценариями Test Spec?
2. Обновляет ли Test Spec (новые edge cases, уточнение покрытия)?
3. Является ли это Code → Specs обратной связью или отдельный механизм?
4. Что если тесты выявили недостающие сценарии — это "рабочие правки" или CONFLICT?

**Решение:** Отдельный механизм не нужен — покрыто Code → Specs. Результаты тестов = код. Упавший тест из-за реализации → починить код. Неточный сценарий → рабочие правки Test Spec (Флаг). Архитектурная проблема → CONFLICT. Уточнение добавлено: "Код включает и результаты тестов".

**Статус:** ✅ решён — уточнение в начале секции "Обратная связь Code → Specs"

---

### D2. Обработка ошибок в воркфлоу

**Проблема:** Описаны все "счастливые" и "конфликтные" пути. Но не описан **тупиковый** путь:

1. LLM не может сгенерировать документ — контекста недостаточно даже после Clarify, пользователь не может ответить
2. LLM сгенерировал документ, но пользователь возвращает на доработку 5+ раз — нет прогресса
3. Внешняя зависимость недоступна (API, сервис, документация)

Есть ли explicit failure state? Таймаут? Эскалация?

**Решение:** Отдельный failure state не нужен — покрыто существующими механизмами. Недостаточно контекста → маркер `[ТРЕБУЕТ УТОЧНЕНИЯ]` → Dependency Barrier → документ остаётся в DRAFT. Пользователь решает: продолжить, отклонить (→ REJECTED), или вернуться позже. DRAFT с неразрешёнными маркерами и есть "заблокированное" состояние.

**Статус:** ✅ решён — не требует изменений, покрыто `[ТРЕБУЕТ УТОЧНЕНИЯ]` + Dependency Barrier

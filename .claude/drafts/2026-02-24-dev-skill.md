## Контекст

**Задача:** Встроить процесс запуска разработки (переход analysis chain из WAITING в RUNNING) в `standard-development.md`. Создать скилл `/dev {NNNN}` как команду входа. Обеспечить навигацию "план разработки → разработка" из plan-dev инструкций. Добавить `/analysis-status` для отслеживания статусов цепочек.

**Почему создан:** Сейчас переход WAITING → RUNNING описан в `standard-analysis.md § 6.2` одним абзацем. Шаги (Issues, Milestone, Branch, перевод статусов) разбросаны по 5 стандартам. Ни `standard-development.md`, ни plan-dev инструкции не указывают на этот шаг как "следующее действие". Отслеживать статусы цепочек по frontmatter неудобно.

**Связанные файлы:**
- `.github/.instructions/development/standard-development.md` — основной документ для расширения
- `specs/.instructions/analysis/plan-dev/standard-plan-dev.md` — ссылка на "следующий шаг"
- `specs/.instructions/analysis/plan-dev/create-plan-dev.md` — AskUserQuestion "начать разработку?"
- `specs/.instructions/analysis/plan-dev/modify-plan-dev.md` — AskUserQuestion после CONFLICT → WAITING
- `specs/.instructions/analysis/standard-analysis.md` — § 4.1, § 6.2
- `.github/.instructions/standard-github-workflow.md` — стадии 1-3
- `specs/analysis/README.md` — dashboard статусов

## Содержание

### Проблема

1. **`standard-development.md` начинается с "Взятие задачи"** — предполагает, что Issues уже созданы. Нет § 0 "Запуск разработки" для перехода от analysis chain к разработке
2. **plan-dev инструкции не указывают следующий шаг.** `create-plan-dev.md` заканчивается отчётом, `modify-plan-dev.md` заканчивается сменой статуса. Ни один не говорит "теперь вызови `/dev`"
3. **Нет AskUserQuestion.** При завершении создания Plan Dev или при возврате CONFLICT → WAITING (когда все 4 документа снова в WAITING) LLM не предлагает начать разработку
4. **Отслеживание статусов неудобно.** Нужно вручную читать frontmatter 4 файлов. Нет сводки
5. **Нет dashboard.** `specs/analysis/README.md` не показывает текущий статус цепочек

### Решение: два изменения

**A.** Расширить `standard-development.md` новой секцией § 0 "Запуск разработки" — полный воркфлоу перехода WAITING → RUNNING. Скилл `/dev` — обёртка над этой секцией.

**B.** В plan-dev инструкциях (create, modify, standard) добавить указание на `/dev` как следующий шаг. В create-plan-dev.md и modify-plan-dev.md добавить AskUserQuestion при достижении состояния "все 4 в WAITING".

**C.** Создать `/analysis-status {NNNN}` и `/analysis-status --all` — скрипт + скилл для отслеживания статусов. Автоматически обновляемая таблица в `specs/analysis/README.md`.

### Изменение A: standard-development.md — новая § 0

Добавить секцию **до** текущей § 1 "Взятие задачи". Перенумерация: § 1→2, § 2→3, ..., § 9→10.

```markdown
## 0. Запуск разработки

> Эта секция применяется при работе с analysis chain (specs/analysis/).
> Если Issues созданы вручную — перейти к [§ 1 Взятие задачи](#1-взятие-задачи).

### Предусловия

- Все 4 документа цепочки (Discussion, Design, Plan Tests, Plan Dev) существуют
- Все 4 документа в статусе `WAITING`
- Маркеров `[ТРЕБУЕТ УТОЧНЕНИЯ]` = 0 во всех документах

### Воркфлоу запуска

1. **Проверить готовность цепочки**
   - Прочитать frontmatter всех 4 документов → `status: WAITING`
   - Если не все в WAITING → СТОП: "Цепочка не готова. {документ} в статусе {status}"

2. **Подтверждение пользователя**
   - AskUserQuestion: "Цепочка NNNN-{topic} готова к разработке.
     {N} TASK-N, Milestone {vX.Y.Z}. Начать?"
   - Если "Нет" → СТОП

3. **Создать GitHub Issues**
   - Для каждого TASK-N из Plan Dev → `/issue-create`
   - Sub-issues для подзадач (N.M)
   - Обновить таблицу маппинга в plan-dev.md

4. **Создать/привязать Milestone**
   - Проверить: Milestone {vX.Y.Z} существует?
   - Если нет → `/milestone-create`
   - Привязать все Issues к Milestone

5. **Создать ветку**
   - `/branch-create {NNNN}`

6. **Перевести цепочку в RUNNING**
   - Обновить `status: WAITING` → `status: RUNNING` во всех 4 документах
   - Обновить `specs/analysis/README.md`

7. **Отчёт**
   - Вывести: Issues (#N), Milestone, Branch, статус цепочки → RUNNING

### Выход из секции

Issues созданы → Milestone назначен → ветка создана → цепочка в RUNNING →
переход к [§ 1 Взятие задачи](#1-взятие-задачи).
```

## Решения

### Q1: Куда поместить основную логику?

**В `standard-development.md` § 0.** `/dev` — это запуск разработки, `standard-development.md` — стандарт разработки. `create-dev.md` — короткий делегат на § 0.

### Q2: Как связать plan-dev → /dev?

**AskUserQuestion.** При завершении `create-plan-dev.md` (шаг 11) и при возврате CONFLICT → WAITING в `modify-plan-dev.md` — LLM проверяет, все ли 4 документа в WAITING, и предлагает `/dev`. В `standard-plan-dev.md` § 4 — текстовое указание.

### Q3: Взаимодействие с review-status-integration?

**`/review-create` при Plan Dev → WAITING** (по решению из черновика review-status-integration). `/dev` не трогает review.md — к моменту `/dev` шаблон review.md уже создан.

### Q4: /analysis-status — скрипт или скилл?

**Оба.** Скрипт `analysis-status.py` — логика. Скилл `/analysis-status` — точка входа для LLM. Dashboard в README — автоматически обновляемая таблица (маркеры BEGIN/END).

### Q5: Перенумерация §§ в standard-development.md?

**Да.** §§ 1-9 сдвигаются → §§ 1-10. Все внутренние ссылки (#1-взятие-задачи → #1-взятие-задачи остаётся, т.к. якоря по заголовку, не по номеру). Но номера в тексте и TOC обновляются. Ссылки из других файлов на standard-development.md по якорям — не ломаются (якоря = текст заголовка).

### Проблема 2.1: Неявное условие "пропуск невозможен"

**Где:** Секция "Порядок создания инструкций" (строка 893)

**Цитата:**
> > **ЗАПРЕЩЕНО:** Параллельное создание инструкций разных типов. Пропуск шагов невозможен.

**Как может понять LLM:**
- Интерпретация A: Нельзя создать create-api.md, если нет validation-api.md → ДОЛЖЕН ОСТАНОВИТЬСЯ и создать validation сначала
- Интерпретация B: Нельзя создать create-api.md, если нет validation-api.md → СООБЩИТЬ ПОЛЬЗОВАТЕЛЮ об ошибке, не выполнять
- Интерпретация C: Нельзя пропустить validation, но можно создать все 3 (standard, validation, create) за одну сессию последовательно

**Рекомендация:**
> **ЗАПРЕЩЕНО:** Параллельное создание инструкций разных типов. Пропуск шагов невозможен.
>
> **Действие при нарушении порядка:**
> 1. LLM обнаруживает отсутствие предыдущего типа (например, validation отсутствует при создании create)
> 2. ОСТАНОВИТЬ выполнение текущей задачи
> 3. СООБЩИТЬ пользователю: "Невозможно создать create-{object}.md без validation-{object}.md. Сначала создайте validation."
> 4. ПРЕДЛОЖИТЬ создать недостающий тип СЕЙЧАС (если пользователь согласен — создать последовательно)

**Критичность:** Высокая (P1)

## 5. Слепые зоны автора

### Зона 5.1: Отсутствие правил для инструкций БЕЗ связанного скилла (технические стандарты)

**Суть:** Документ утверждает "каждая create/modify/validation ДОЛЖНА иметь связанный скилл" (§ 10, строка 1011). Но что если инструкция описывает РУЧНОЙ процесс без автоматизации? Например, "standard-naming-conventions.md" — описывает правила именования, но скилл для этого не нужен.

**Почему важно:** Правило "ДОЛЖНА иметь скилл" блокирует создание технических стандартов (описательных, не исполняемых)

**Расширение качества:** Значительное (устраняет противоречие между "всё должно иметь скилл" и реальностью)

**Рекомендация:** Добавить сейчас

**Пример добавления:**
> ### Связь со скиллом
>
> > **Каждая инструкция create/modify/validation ДОЛЖНА иметь связанный скилл.**
>
> **Исключения:** Инструкции, описывающие РУЧНЫЕ процессы (принципы, naming conventions, архитектурные решения), НЕ требуют скилла. В секции "Скиллы" указать: `*Скилл не требуется (ручной процесс).*`

**Критичность:** Высокая (P1)

### Зона 5.2: Отсутствие правил миграции инструкций при обновлении стандарта

**Суть:** Документ описывает как СИНХРОНИЗИРОВАТЬ версии (скрипт `sync-standard-version.py`), но НЕ описывает что делать с СОДЕРЖИМЫМ инструкции при breaking change в стандарте. Например, если `standard-instruction.md` v2.0 добавляет новую ОБЯЗАТЕЛЬНУЮ секцию — как мигрировать все существующие инструкции?

**Почему важно:** Без правил миграции зависимые инструкции останутся на старой версии формата (техдолг)

**Расширение качества:** Значительное (обеспечивает эволюцию стандартов без поломки зависимых файлов)

**Рекомендация:** Добавить сейчас

**Пример добавления:**
> ### Миграция при breaking changes
>
> При изменении стандарта с breaking changes (v1.x → v2.0):
> 1. Создать миграционную инструкцию `migrate-instruction-v1-to-v2.md`
> 2. Описать изменения формата (что добавилось, что удалилось)
> 3. Создать скрипт миграции (если возможно автоматизировать)
> 4. Уведомить авторов зависимых инструкций (через issue/PR)
> 5. Установить дедлайн миграции (например, 1 месяц)

**Критичность:** Средняя (P2)

---

### Зона 5.3: Отсутствие правил ВЕРСИОНИРОВАНИЯ самих инструкций (не стандартов)

**Суть:** Документ описывает версионирование СТАНДАРТОВ (поле `standard-version`), но НЕ описывает версионирование ИНСТРУКЦИЙ. Например, если `create-structure.md` значительно изменяется — как отследить версию инструкции? Агенты имеют поле `version: v1.0` (§ 12 в standard-agent.md), но инструкции — нет.

**Почему важно:** Без версионирования инструкций невозможно отследить изменения между редакциями (какие шаги добавились/удалились)

**Расширение качества:** Умеренное (улучшает прозрачность изменений, но не критично для работы)

**Рекомендация:** Отложить (over-engineering для инструкций, достаточно git history)

**Пример добавления (если решить добавить):**
> ### Версионирование инструкций (опционально)
>
> Инструкции МОГУТ иметь поле `version: vX.Y` во frontmatter (по аналогии с агентами). Обновлять версию при значительных изменениях. Если поле отсутствует — версионирование через git history.

**Критичность:** Низкая (P3)

---

### Зона 5.4: Отсутствие правил для КОМПОЗИЦИИ инструкций (переиспользование секций)

**Суть:** Документ НЕ покрывает кейс, когда несколько инструкций используют ОДИНАКОВУЮ секцию. Например, секция "Обработка ошибок" может быть общей для всех create-инструкций. Сейчас это приводит к дублированию (нарушение DRY).

**Почему важно:** Дублирование секций между инструкциями создаёт техдолг (изменение в одной инструкции требует правок во всех копиях)

**Расширение качества:** Умеренное (улучшает поддерживаемость, но усложняет структуру)

**Рекомендация:** Отложить (YAGNI — проблема возникнет, когда инструкций станет > 50)

**Пример добавления (если решить добавить):**
> ### Переиспользование секций
>
> Если секция повторяется в > 3 инструкциях — вынести в общий файл:
> ```markdown
> <!-- В инструкции -->
> ## Обработка ошибок
>
> > **SSOT:** [common/error-handling.md](./common/error-handling.md)
> ```

**Критичность:** Низкая (P3)

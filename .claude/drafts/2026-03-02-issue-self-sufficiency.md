## Контекст

**Задача:** Issue #56 (и все остальные #42-#61) содержат минимальный body — указатель на документы, а не самодостаточную задачу. LLM-агент не может разработать по такому Issue без навигации по 5+ файлам.
**Почему создан:** Каждый Issue будет выполняться отдельным dev-agent (субагентом). Субагенты НЕ имеют общего контекста с основной сессией — каждый Issue должен быть полностью самодостаточным "промтом": все пути, все инструкции, все зависимости.
**Связанные файлы:**
- `.github/.instructions/issues/standard-issue.md` — стандарт Issues (§ 4 Body)
- `.github/.instructions/issues/create-issue.md` — воркфлоу создания Issue
- `.github/.instructions/issues/validation-issue.md` — валидация Issue
- `.github/.instructions/issues/modify-issue.md` — воркфлоу изменения Issue
- `.github/.instructions/issues/issue-templates/standard-issue-template.md` — стандарт YAML-шаблонов
- `.github/.instructions/issues/issue-templates/validation-type-templates.md` — валидация шаблонов
- `.github/ISSUE_TEMPLATE/*.yml` — YAML-шаблоны Issues (7 файлов)
- `.github/.instructions/.scripts/validate-issue.py` — скрипт валидации Issues
- `.github/.instructions/development/create-development.md` — воркфлоу /dev-create
- `specs/.instructions/analysis/plan-dev/standard-plan-dev.md` — стандарт Plan Dev
- `specs/.instructions/docs/technology/standard-technology.md` — стандарт per-tech
- `specs/.instructions/docs/system/standard-system.md` — стандарт .system/ файлов
- `specs/docs/.technologies/` — per-tech стандарты кодирования (9 файлов)
- `specs/docs/.system/` — кросс-сервисные документы (4 файла)
- `specs/docs/{svc}.md` — per-service документы

**Почему самодостаточность критична:** Субагенты (dev-agent) запускаются через Agent tool и работают в ИЗОЛИРОВАННОМ контексте. Они НЕ имеют доступа к TaskList основной сессии, НЕ видят историю разговора. Единственный вход — текст промта (Issue body). Если в Issue нет ссылки на документ — агент его не прочитает.

**Пересечение:** Этот черновик меняет те же файлы, что `2026-03-02-expand-type-labels.md` (standard-issue.md, create-issue.md, create-development.md, validate-issue.py). TYPE-драфт уже применён — self-sufficiency идёт вторым.

### 2. Issue как промт для LLM-агента

Ключевая идея: Issue = **полный промт** для dev-agent. Агент получает Issue и должен выполнить задачу БЕЗ дополнительного контекста.

**Единая структура body (5 секций) для ВСЕХ Issues:**

| # | Секция | Назначение | Аналогия в промтинге |
|---|--------|-----------|---------------------|
| 1 | Описание задачи | ЧТО, ЗАЧЕМ, зависимости | System prompt — контекст задачи |
| 2 | Документы для изучения | ВСЕ документы: спеки, .system/, per-tech | Tool calls — чтение конкретных файлов |
| 3 | Задание | ЧТО ИМЕННО сделать (пошагово) | User prompt — конкретная инструкция |
| 4 | Критерии готовности | КАК ПРОВЕРИТЬ результат | Acceptance criteria — чек-лист / ссылка на Plan Test |
| 5 | Практический контекст | Существующий код, зависимости, запуск | Environment — где и как работать |

Каждая секция обязывает агента ПРОЧИТАТЬ конкретные файлы. Issue — не справочник, а МАРШРУТ.

**Единая структура, а не двухтрековая:** Старая структура (3 секции: Описание, Связанная документация, Критерии готовности) заменяется 5-секционной для ВСЕХ Issues. Нет разделения "ручной Issue" vs "Issue из Plan Dev" — одна структура, разная глубина наполнения. Для ручного баг-репорта "Документы для изучения" может содержать одну строку, для TASK-N из Plan Dev — 10 строк. Структура единая.

**Почему 5, а не 8:** Критический анализ показал, что 8 секций содержат дублирование ("Связанная документация" = копия "Документов"), пустые по существу секции ("Зависимости" = одна строка) и искусственное разделение ("Системные документы" и "Технологии" — это такие же документы для изучения). 5 секций покрывают всё без дублирования.

**Почему единая, а не двухтрековая:** Две параллельные структуры (3 секции для ручных, 5 для Plan Dev) создают путаницу: валидация должна проверять обе, шаблоны должны поддерживать обе, modify-issue должен знать обе. Единая структура — одна валидация, одни шаблоны, один стандарт.

### 4. Секция "Документы для изучения"

**Единая таблица.** ВСЕ документы, которые агент должен прочитать, в одном месте: .system/ файлы, Design секции, per-tech стандарты, сервисная документация, Plan Test.

**Формат:** Таблица с 4 колонками: #, Документ, Путь, Что искать.

**Источники данных (при `/dev-create`):** Полный алгоритм — § 8 (шаги 6-10). Кратко: .system/ (§ 7), Design SVC-N §§, specs/docs/{svc}.md, per-tech стандарты из Tech Stack, Plan Test TC-N.

**Не копировать данные:** Issue содержит ПУТИ и ОПИСАНИЯ — что искать. LLM сам прочитает через Read tool. Документы могут обновиться при CONFLICT, а Issue — нет. Ссылки важнее snapshot-ов.

**Per-tech стандарты в таблице:** Технологии — это такие же документы для изучения, как Design или conventions.md. Мы специально пишем per-tech стандарты, чтобы агент им следовал. Отдельная секция — искусственное разделение. В колонке "Что искать" указать: "ОБЯЗАТЕЛЬНО следуй при написании кода".

**Алгоритм определения per-tech (при `/dev-create`):** см. § 8 шаги 8-9. Кратко: Источник → SVC-N → `specs/docs/{svc}.md` → Tech Stack → `standard-{tech}.md` → фильтр по задаче → в таблицу.

### 5. Секция "Задание"

Ядро промта — явная пошаговая инструкция человеческим языком.

**Формат вступления:** См. пример секции "Задание" в § 3. Шаблон: "В соответствии с описанием {сервис}-сервиса в specs/docs/{svc}.md, на основании проектирования в Design SVC-N (...) и критериев приёмки из Plan Test TC-N..TC-M (...), реализуй {описание задачи}:"

**Принципы каждого пункта:**
- Начало: **действие** + **путь к файлу** (создай `src/.../file.ts`)
- Контекст: что должен делать файл/модуль
- Ссылки: "Формат ответа: см. Design SVC-N § K", "Формат ошибок: см. conventions.md"
- Acceptance: "(TC-29)" — привязка к конкретному тест-кейсу
- Зависимости: "через API-слой из #55"
- Инструкции: "Следуй паттернам из standard-react.md"

**Тесты — обязательный пункт:** Последний пункт "Задания" ВСЕГДА — "Напиши тесты". Поле `TC: TC-N` в TASK-N указывает тест-кейсы из Plan Test — это сценарии, по которым агент ДОЛЖЕН написать тесты. Если TC не указаны — пункт про тесты опускается.

**Откуда генерировать:** Подзадачи TASK-N (N.1, N.2, ...) + контекст из Design SVC-N §§. Каждая подзадача разворачивается в полную инструкцию. Последний пункт — тесты из TC.

### 6. Секция "Практический контекст"

**Новая секция.** Информация, которой нет в спецификациях, но без которой агент не может начать работу.

**Три блока:**

| Блок | Откуда берётся | Зачем |
|------|---------------|-------|
| Существующий код | `ls src/{svc}/` при `/dev-create` | Агент знает, что уже есть, не создаёт конфликтующие файлы |
| Зависимость #N экспортирует | Из Issue зависимости → его body/подзадачи | Агент знает конкретный импорт, а не абстрактный "API-слой" |
| Как запустить/проверить | Из Makefile / {svc}.md § Dev | Агент может запустить и проверить результат |

**Снимок существующего кода:** При `/dev-create` выполнить `ls src/{svc}/` и включить результат в Issue. Это snapshot — при CONFLICT может устареть, но для начала работы достаточно. Агент прочитает актуальную структуру через Glob/Read.

**Зависимости: что КОНКРЕТНО экспортирует #N:** Не "API-слой из #55", а "src/frontend/api/auth.ts → authApi.login(email, password): Promise\<LoginResponse\>". Агент должен знать ИНТЕРФЕЙС зависимости: файл, функция, сигнатура.

### 7. Фильтрация документов по релевантности

**Проблема:** Не все .system/ файлы нужны для каждой задачи. infrastructure.md (порты, БД, Docker) нерелевантен для фронтенд-стора. Кроме того, infrastructure.md содержит визуализацию (пример из стандарта, не реальные данные проекта) — включение может ввести агента в заблуждение.

**testing.md:** Включается ВСЕГДА, если TASK-N имеет поле `TC:` (тест-кейсы). Агент должен знать стратегию тестирования, мокирование и структуру тест-файлов. Если `TC:` отсутствует — testing.md опускается.

**Фильтр по типу сервиса из `Источник: SVC-N`:**

| Тип задачи | overview.md | conventions.md | infrastructure.md | testing.md |
|------------|:-----------:|:--------------:|:-----------------:|:----------:|
| frontend | + | + (JWT, ошибки) | — | + (если есть TC) |
| backend | + | + | + (порты, БД) | + (если есть TC) |
| infra (без SVC) | + | — | + (PRIMARY) | + (если есть TC) |
| test | + | + | + | + (PRIMARY) |

**Определение типа:** Из `Источник: SVC-N` → `specs/docs/{svc}.md` → § Code Map → Tech Stack. Если React/Vue → frontend. Если Express/Fastify/NestJS → backend. Если нет SVC → infra.

**Дополнительный фильтр по подзадачам:**
- Подзадача содержит "тест" / "test" → добавить testing.md
- Подзадача содержит "порт" / "Docker" / "env" → добавить infrastructure.md
- Подзадача содержит "API" / "endpoint" / "ошибк" → добавить conventions.md

**Per-tech стандарты:** Фильтровать по Tech Stack сервиса. Из 9 доступных стандартов для frontend-задачи нужны только React + TypeScript (2 из 9).

### 8. Алгоритм сбора контекста при /dev-create

Для каждого TASK-N из Plan Dev:

```
1. Прочитать TASK-N: Источник (SVC-N §§), TC (TC-N), Зависимости, подзадачи
2. Определить сервис: SVC-N → {svc} → specs/docs/{svc}.md
```

## Решения

- **Issue = полный промт** — агент получает Issue и работает автономно. Никакого "общего контекста" — субагенты изолированы.
- **Навигация, а не копирование** — Issue содержит ПУТИ к документам + описание "что искать". LLM читает через Read tool. Документы — SSOT, Issue — маршрут.
- **Единая таблица документов** — .system/, Design, per-tech стандарты, Plan Test — в ОДНОЙ таблице "Документы для изучения". Нет искусственного разделения: per-tech стандарты — такие же документы, как и Design.
- **Фильтрация по релевантности** — НЕ все 4 .system/ файла всегда. Фильтр по типу сервиса (frontend/backend/infra) + дополнительный фильтр по подзадачам. Исключает нерелевантные документы (infrastructure.md для фронтенд-стора).
- **Практический контекст** — то, чего нет в спецификациях: существующий код, конкретные экспорты зависимостей, команды запуска. Без этого агент не может начать работу.
- **Задание-промт** — пошаговая инструкция с ссылками на Design/TC/conventions. Формат: "создай X, который делает Y (см. Design § Z, conventions.md)".
- **Критерии через Plan Test** — ссылка на SSOT (Plan Test) + конкретные TC-N, а не дублирование.
- **5 секций вместо 8** — убраны дубли ("Связанная документация" = копия "Документов"), пустые секции ("Зависимости" = одна строка → в "Описание"), искусственные разделения ("Системные документы" и "Технологии" → в единую таблицу).
- **Единая структура для ВСЕХ Issues** — одна 5-секционная структура вместо двух параллельных (3 для ручных, 5 для Plan Dev). Разница — в глубине наполнения, не в структуре. Одна валидация, одни шаблоны, один стандарт.
- **Issue templates обновляются** — YAML-шаблоны генерируют body с 5 секциями. Каждый шаблон адаптирован под тип Issue (bug vs feature vs infra), но структура единая.

## 1. Двойственность формулировок

### Проблема 1.1: Неопределённость в описании стандарта для README

**Где:** Секция "standard" (строки 79-87)

**Цитата:**
> | Тип документа | standard |
> |---------------|----------|
> | README папки проекта | `.structure/.instructions/standard-readme.md` |
> | README папки инструкций | `.structure/.instructions/standard-readme.md` |

**Как может понять LLM:**
- Интерпретация A: Оба типа README используют один и тот же стандарт
- Интерпретация B: Для README есть единый стандарт, но внутри него разные секции для разных типов
- Интерпретация C: Это ошибка дублирования — должны быть разные стандарты

**Рекомендация:**
> Добавить пояснение:
> ```markdown
> | Тип документа | standard |
> |---------------|----------|
> | README папки проекта | `.structure/.instructions/standard-readme.md` (секция 2) |
> | README папки инструкций | `.structure/.instructions/standard-readme.md` (секция 3) |
> ```
>
> Или добавить примечание: "Оба типа README используют один стандарт, но разные секции внутри него."

**Критичность:** Средняя (P2)

### Проблема 1.3: Неясность с обновлением standard-version

**Где:** Секция "standard-version" (строки 98-100)

**Цитата:**
> - Указывать версию из строки `Версия стандарта: X.Y` в теле стандарта
> - При обновлении документа — проверить актуальность версии стандарта
> - Если стандарт обновился — мигрировать документ и обновить `standard-version`

**Как может понять LLM:**
- Интерпретация A: При любом обновлении документа нужно проверять версию стандарта
- Интерпретация B: Только при структурных изменениях проверять версию
- Интерпретация C: "Обновление документа" = обновление стандарта, а не зависимого файла

**Рекомендация:**
> Уточнить формулировки:
> ```markdown
> - Указывать версию из заголовка `Версия стандарта: X.Y` в файле-стандарте
> - При редактировании зависимого документа — сверить `standard-version` с актуальной версией стандарта
> - Если версия стандарта изменилась — мигрировать документ по новому стандарту, обновить `standard-version`
> ```

**Критичность:** Средняя (P2)

## 2. Риск неправильной интерпретации LLM

### Проблема 2.1: Неявное требование к синхронизации версий

**Где:** Секция "Синхронизация версий" (строки 102-120)

**Цитата:**
> **Синхронизация версий:**
>
> | Скрипт | Назначение |
> |--------|------------|
> | `bump-standard-version.py` | Увеличить версию стандарта |
> | `sync-standard-version.py` | Синхронизировать зависимые файлы |
> | `check-version-drift.py` | Проверить расхождения версий |

**Риск интерпретации:**
LLM может понять, что синхронизация версий — опциональная (есть скрипты "если нужно"), но не понимать:
- Когда ОБЯЗАТЕЛЬНО синхронизировать
- Что произойдёт, если не синхронизировать
- Кто запускает эти скрипты (автор стандарта? пользователь? CI/CD?)

**Рекомендация:**
> Добавить секцию "Обязательные действия при обновлении стандарта":
> ```markdown
> **Когда обновлять standard-version:**
>
> | Действие | Кто обновляет |
> |----------|---------------|
> | Изменение стандарта → bump версии в стандарте | Автор стандарта (вручную или через скрипт) |
> | Миграция зависимого файла | Автор файла (вручную обновить frontmatter) |
> | CI/CD проверка | Автоматически (check-version-drift.py в pipeline) |
> ```

**Критичность:** Высокая (P1) — влияет на процесс обновления документов

> **Граничные случаи:**

### Проблема 2.3: Неопределённость формата allowed-tools

**Где:** Секция "allowed-tools" (строки 161-174)

**Цитата:**
> **Доступные инструменты:**
>
> | Инструмент | Описание |
> |------------|----------|
> | `Bash` | Выполнение команд в терминале |
> | `Read` | Чтение файлов |
> | `Write` | Создание файлов |
> | `Edit` | Редактирование файлов |
> | `Glob` | Поиск файлов по паттерну |
> | `Grep` | Поиск в содержимом файлов |

**Риск интерпретации:**
- Формат значения поля: строка через запятую? Массив YAML?
- Нужны ли кавычки?
- Регистр важен (Bash vs bash)?
- Можно ли указать "все инструменты" через `*` или оставить поле пустым?

**Рекомендация:**
> Добавить примеры форматов:
> ```markdown
> **Формат значения:**
>
> ```yaml
> # Вариант 1: строка (через запятую, без пробелов после запятой опционально)
> allowed-tools: Read, Bash, Glob, Grep, Write, Edit

> # Вариант 2: YAML-массив
> allowed-tools:
>   - Read
>   - Bash
>   - Glob
> ```
>
> **Правила:**
> - Регистр: первая буква заглавная (`Read`, `Bash`, НЕ `read`, `bash`)
> - Все инструменты: не указывать поле (по умолчанию все доступны)
> ```

**Критичность:** Высокая (P1) — влияет на корректность frontmatter скиллов

## 4. Полнота описания объекта

### 4.1. Не хватает: валидация frontmatter

**Почему важно:**
Документ описывает формат frontmatter, но не объясняет:
- Как проверить корректность frontmatter?
- Какие ошибки возможны?
- Есть ли автоматическая валидация?

Риск: LLM создаст файл с невалидным frontmatter, и ошибка будет обнаружена только при попытке использования.

**Стоит ли добавлять:** Да — критично для корректности документов.

**Что даст:** Явная ссылка на `validation-frontmatter.md` или встроенный чек-лист валидации.

**Рекомендация:**
> Добавить секцию в конце:
> ```markdown
> ## Валидация frontmatter
>
> **Автоматическая проверка:**
> - Выполняется через [validation-links.md](./validation-links.md) (проверяет ссылки в полях `standard`, `index`)
>
> **Ручная проверка (если автоматика недоступна):**
> - [ ] Все обязательные поля присутствуют
> - [ ] Значения полей соответствуют типу документа
> - [ ] Файлы, указанные в `standard` и `index`, существуют
> - [ ] Версия `standard-version` соответствует версии в файле-стандарте
> ```

**Критичность:** Высокая (P1)

**Рекомендация:**

> ```markdown

> |----------|----------|

## 5. Слепые зоны автора

### Зона 5.1: Отсутствие примеров невалидного frontmatter

**Суть:**
Документ показывает только корректные примеры, но не показывает типичные ошибки и их последствия.

**Почему важно:**
LLM лучше учится на контрастных примерах ("что правильно" + "что неправильно и почему"). Отсутствие антипаттернов приводит к повторению одних и тех же ошибок.

**Расширение качества:** Значительное — снижает количество ошибок при создании файлов.

**Рекомендация:** Добавить сейчас.

**Пример добавления:**
> Добавить секцию в конце документа:
> ```markdown
> ## Типичные ошибки
>
> | Ошибка | Почему неправильно | Как исправить |
> |--------|-------------------|---------------|
> | `standard: /standard-instruction.md` | Ведущий `/` запрещён | `standard: .instructions/standard-instruction.md` |
> | `standard_version: v1.0` | Подчёркивание вместо дефиса | `standard-version: v1.0` |
> | `index: .instructions/` | Ссылка на папку без файла | `index: .instructions/README.md` |
> | `allowed-tools: read, bash` | Строчные буквы | `allowed-tools: Read, Bash` |
> | Отсутствие `description` | Обязательное поле | Добавить `description: ...` |
> ```

**Критичность:** Высокая (P1)

### Зона 5.2: Нет правил для файлов вне стандартной структуры

**Суть:**
Документ предполагает, что все файлы находятся в стандартных папках (`.instructions/`, `.claude/`, `.structure/`), но не описывает:
- Как оформлять frontmatter для файлов в корне проекта? (CLAUDE.md, CONTRIBUTING.md)
- Как оформлять для файлов в `docs/`, `specs/` (если они есть)?
- Нужен ли frontmatter для технических файлов (.gitignore, package.json)?

**Почему важно:**
Проекты эволюционируют — появляются новые папки и типы документов. Без явных правил LLM будет придумывать свои, что нарушит консистентность.

**Расширение качества:** Умеренное — покрывает граничные случаи.

**Рекомендация:** Добавить сейчас (как подсекцию "Особые случаи").

**Пример добавления:**
> ```markdown
> ## Особые случаи
>
> ### Файлы в корне проекта
>
> | Файл | standard | index |
> |------|----------|-------|
> | `CLAUDE.md` | `.instructions/standard-claude.md` (если есть) | `README.md` |
> | `CONTRIBUTING.md` | `.instructions/standard-contributing.md` (если есть) | `README.md` |
> | `README.md` | `.structure/.instructions/standard-readme.md` | `README.md` (ссылка на себя) |
>
> ### Файлы вне стандартных папок
>
> - Если папка имеет `.instructions/` → использовать стандарты из неё
> - Если нет `.instructions/` → ссылаться на ближайший родительский стандарт
> - Технические файлы (`.gitignore`, `package.json`) → frontmatter не требуется
> ```

### Зона 5.3: Отсутствие миграционной стратегии

**Суть:**

- Можно ли иметь файлы с разными версиями одновременно?

**Почему важно:**
Стандарты эволюционируют. Без чёткой миграционной стратегии проект станет зоопарком форматов.

**Расширение качества:** Значительное — обеспечивает долгосрочную поддержку стандартов.

**Рекомендация:** Добавить (ссылка на `modify-frontmatter.md` или встроенная секция).

**Пример добавления:**
> ```markdown
> ## Миграция при обновлении стандарта
>
> **Шаги:**
>
> 1. **Автор стандарта обновляет версию:**
>    - Изменить `Версия стандарта: X.Y` в теле стандарта
>    - Увеличить версию на +0.1 (минорные изменения) или +1.0 (мажорные)
>
> 2. **Автор стандарта создаёт инструкцию миграции (опционально):**
>    - Если изменения breaking → создать `migration-v{X}-to-v{Y}.md`
>
> 3. **Зависимые файлы обновляются:**
>    - Автоматически (через `sync-standard-version.py`) ИЛИ
>    - Вручную при следующем редактировании
>
> 4. **CI/CD проверяет версии:**
>    - `check-version-drift.py` предупреждает о расхождениях
> ```

**Критичность:** Высокая (P1) — влияет на управление жизненным циклом стандартов

## Рекомендации по улучшению

### 1. Циклическая зависимость с standard-release.md

**Проблема:** Milestone ссылается на Release, Release ссылается на Milestone. Это создаёт путаницу при чтении.

**Рекомендация:** Уточнить направление зависимости:
- Milestone — контейнер для группировки Issues
- Release — результат закрытия Release Milestone
- Последовательность: создать Milestone → добавить Issues → закрыть Milestone → создать Release

### 2. Не описано: перепланирование при срыве сроков

**Проблема:** Описано продление `due_on`, но нет процедуры для систематического срыва сроков.

**Рекомендация:** Добавить подсекцию "Ретроспектива Milestone":
- После закрытия — анализ что было запланировано vs выполнено
- Если >30% Issues перенесено — пересмотреть подход к планированию
- Корректировка scope следующего Milestone

### 3. Неясность: когда создавать Sprint vs Release Milestone

**Проблема:** Оба типа описаны, но не ясно можно ли использовать оба одновременно.

**Рекомендация:** Добавить примеры комбинаций:
```
# Вариант 1: Только Release Milestones
v1.0.0 → v1.1.0 → v2.0.0

# Вариант 2: Только Sprint Milestones
Sprint 1 → Sprint 2 → Sprint 3

# Вариант 3: Комбинация
Sprint 2025-W05 (текущая работа)
v1.0.0 (целевой релиз, включает Issues из нескольких спринтов)
```

И рекомендацию: для команды <5 человек — один тип (Release или Sprint).

### 4. Не описано: что делать с "вечными" Milestones

**Проблема:** Нет процедуры для Milestones типа "Tech Debt" или "Nice to have", которые никогда не закрываются.

**Рекомендация:** Добавить подсекцию:
- Избегать "бессрочных" Milestones
- Если нужен backlog — использовать метки или Project, не Milestone
- Milestone ВСЕГДА должен иметь `due_on`

### 5. Отсутствует: автоматизация создания Sprint Milestones

**Проблема:** Каждый Sprint создаётся вручную через API.

**Рекомендация:** Добавить пример скрипта или GitHub Action:
```bash
# Скрипт создания Sprint Milestone на следующую неделю
WEEK=$(date -d "next monday" +%Y-W%V)
gh api POST /repos/{owner}/{repo}/milestones \
  -f title="Sprint $WEEK" \
  -f due_on="$(date -d "next friday" +%Y-%m-%dT23:59:59Z)"
```

### 6. Не описано: связь Milestone с GitHub Projects

**Проблема:** Milestones упоминаются в контексте Projects (фильтрация), но не описана интеграция.

**Рекомендация:** Добавить ссылку на standard-project.md с примером:
- Issue в Milestone автоматически попадает в Project (если настроена автоматизация)
- View "Current Sprint" фильтрует по Milestone

### 7. Мелкое: формат due_on в примерах

**Проблема:** Примеры используют `2025-02-09T23:59:59Z`, но не объяснено почему 23:59:59.

**Рекомендация:** Добавить пояснение:
- Время 23:59:59 = конец дня (чтобы весь день был включён)
- Альтернатива: 00:00:00 следующего дня
- GitHub отображает только дату, время для сортировки

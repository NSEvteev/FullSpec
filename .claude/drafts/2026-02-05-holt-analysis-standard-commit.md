## 1. Двойственность формулировок

### Проблема 1.1: Неопределённое "опустить scope"

**Где:** § 3 "Scope" (строка 89)

**Цитата:**
> **Когда опустить scope:** Если изменение затрагивает весь проект или не привязано к конкретному модулю: `chore: update dependencies`.

**Как может понять LLM:**
- Интерпретация A: Опустить scope, если изменение касается всех модулей одновременно (глобальная зависимость)
- Интерпретация B: Опустить scope, если невозможно определить конкретный модуль (размытая область)
- Интерпретация C: Опустить scope, если изменение инфраструктурное (не относится к коду)

**Рекомендация:**
> **Когда опустить scope:** Если изменение соответствует одному из критериев:
> - Затрагивает зависимости на уровне всего проекта (пример: `chore: update dependencies`)
> - Относится к инфраструктуре (docker, CI/CD) без привязки к конкретному сервису
> - Изменяет глобальные настройки проекта (пример: `.editorconfig`, `.gitignore`)

**Приоритет:** P1

### Проблема 1.2: Размытое понятие "один логический блок"

**Где:** § 5 "Правила оформления" (строка 128)

**Цитата:**
> | Один логический блок = один коммит | Отдельный коммит на фичу | Все изменения за день в одном коммите |

**Как может понять LLM:**
- Интерпретация A: "логический блок" = одна завершённая фича (может включать файлы из разных модулей)
- Интерпретация B: "логический блок" = изменения в одном файле
- Интерпретация C: "логический блок" = изменения одного типа (например, все fix или все refactor)

**Рекомендация:**
> **Один логический блок = один коммит:**
> - Коммит завершает одну цель (одна фича, один баг, один рефакторинг)
> - Атомарность: откат коммита не ломает проект
> - Коммит не должен содержать несвязанные изменения (нельзя: fix + feat в одном коммите)
>
> **Примеры:**
> - ✅ Корректно: `feat(auth): add JWT validation` (все файлы относятся к одной фиче)
> - ❌ Некорректно: `feat(auth): add login + fix(api): fix typo` (две несвязанные цели)

**Приоритет:** P1

### Проблема 1.3: Двусмысленность "imperative"

**Где:** § 5 "Правила оформления" (строка 125)

**Цитата:**
> | Начинать с глагола (imperative) | `docs: update readme` | `docs: readme updated` |

**Как может понять LLM:**
- Интерпретация A: Использовать инфинитив без частицы "to" (английский императив)
- Интерпретация B: Использовать глагол в любой форме, но не причастие
- Интерпретация C: Использовать команду (форма приказа)

**Рекомендация:**
> | Начинать с глагола (imperative) | Использовать форму "сделай это" | `docs: update readme`, `fix: resolve error` | `docs: readme updated`, `fix: error resolved` |

**Приоритет:** P2

### Проблема 1.4: Неявное правило про scope и общие слова

**Где:** § 3 "Scope" (строки 85-87)

**Цитата:**
> **Не использовать:**
> - Действия: ~~`fix`~~, ~~`add`~~ — это type, не scope
> - Общие слова: ~~`common`~~, ~~`utils`~~ — уточнить конкретный модуль

**Как может понять LLM:**
- Интерпретация A: Запрещено использовать `common` и `utils` как scope ВСЕГДА (даже если модуль так называется)
- Интерпретация B: Использовать конкретное имя модуля, даже если модуль называется `utils` (например, `utils/formatter.js` → `formatter`)
- Интерпретация C: Если модуль действительно называется `utils` — допустимо использовать как scope

**Рекомендация:**
> **Не использовать:**
> - Действия: ~~`fix`~~, ~~`add`~~ — это type, не scope
> - Общие слова: ~~`common`~~, ~~`utils`~~ — уточнить до конкретного компонента внутри модуля
>
> **Пример:** Вместо `refactor(utils): improve formatter` использовать `refactor(formatter): improve validation logic`

**Приоритет:** P2

### Проблема 1.5: Двусмысленность "footer через пустую строку"

**Где:** § 4 "Body и Footer" (строка 105)

**Цитата:**
> Метаинформация после body. Отделяется от body **пустой строкой**. Каждый элемент — с новой строки.

**Как может понять LLM:**
- Интерпретация A: Между body и footer — одна пустая строка (стандарт Conventional Commits)
- Интерпретация B: Footer отделяется пустой строкой от subject, если body отсутствует
- Интерпретация C: Между каждым элементом footer — пустая строка

**Рекомендация:**
> Footer отделяется от body **одной пустой строкой**. Если body отсутствует — footer отделяется от subject одной пустой строкой. Элементы footer разделяются переносом строки без пустых строк между ними.
>
> **Пример:**
> ```
> feat(auth): add JWT validation
>
> Closes #123
> Reviewed-by: @username
> ```

**Приоритет:** P1

> 1. Линтинг кода
> 2. Форматирование
> 3. Валидация сообщения коммита (если настроен commitlint)

**Как может понять LLM:**

> 3. Валидация сообщения коммита (commitlint)

## 2. Риск неправильной интерпретации LLM

### Проблема 2.1: Отсутствие границ для body

**Где:** § 4 "Body и Footer" (строки 95-101)

**Цитата:**
> Детальное описание изменения. Отделяется от subject **пустой строкой**.
> - Отвечает на вопрос "почему" или "что изменилось"
> - Рекомендуемая длина строки: до 72 символов
> - Может содержать несколько абзацев

**Как LLM может понять неправильно:**
- LLM может создать body на несколько экранов (формально "несколько абзацев" не ограничено)
- Непонятно, сколько абзацев допустимо: 2, 5, 10?

**Рекомендация:**
> Body содержит 1-3 абзаца (более длинное описание — признак слишком большого коммита, нужно разделить на несколько). Каждый абзац до 72 символов на строку.

**Приоритет:** P1

### Проблема 2.2: Неявная связь между type и scope

**Где:** § 2 "Типы коммитов", § 3 "Scope"

**Проблема:**
Документ описывает type и scope отдельно, но не объясняет, как они взаимодействуют. LLM может создать некорректные комбинации:
- `docs(api)` — документация для API (корректно)
- `docs(button)` — документация для кнопки (некорректно, кнопка — UI, не область документации)
- `test(chore)` — тест для chore (бессмысленно)

**Рекомендация:**
> **Связь type и scope:**
> - Scope уточняет область применения type
> - Не все комбинации type+scope допустимы
>
> **Примеры:**
> - ✅ `docs(api): update endpoint description` — область API имеет документацию
> - ❌ `test(chore): add test for chore` — chore не тестируется (это действие, а не код)
> - ✅ `refactor(auth): extract validation` — рефакторинг модуля auth

**Приоритет:** P2

### Проблема 2.3: Отсутствие порядка применения pre-commit hooks

**Где:** § 6 "Процесс коммита" (строки 146-149)

**Цитата:**
> При выполнении `git commit` автоматически запускаются hooks:
> 1. Линтинг кода
> 2. Форматирование
> 3. Валидация сообщения коммита (если настроен commitlint)

**Как LLM может понять неправильно:**
- LLM может предположить, что hooks выполняются параллельно
- Непонятно, прерывается ли выполнение, если hook #1 провалился

**Рекомендация:**
> Hooks выполняются последовательно. Если любой hook провалился — следующие не выполняются, коммит блокируется.
>
> **Порядок выполнения:**
> 1. Линтинг кода → провал = СТОП
> 2. Форматирование → провал = СТОП
> 3. Валидация сообщения → провал = СТОП
> 4. Если ВСЕ hooks прошли → коммит создаётся

**Приоритет:** P1

### Проблема 2.4: Двусмысленность "исправить message"

**Где:** § 6 "Процесс коммита" (строка 156)

**Цитата:**
> - Ошибка в формате сообщения → исправить message по правилам [§ 5](#5-правила-оформления)

**Как LLM может понять неправильно:**
- LLM может попытаться выполнить `git commit --amend -m "новое сообщение"` (но коммит не создан)
- Непонятно, как "исправить message": повторить `git commit -m "..."` или редактировать через редактор

**Рекомендация:**
> - Ошибка в формате сообщения → повторить `git commit -m "исправленное сообщение"` (коммит НЕ создан, поэтому используется обычный commit, не amend)

**Приоритет:** P1

### Проблема 2.5: Неявная зависимость между hooks и --amend

**Где:** § 6 "Процесс коммита" (строка 159)

**Цитата:**
> **Важно:** Не использовать `git commit --amend` после провала hooks — коммит ещё не создан, нечего amend'ить.

**Как LLM может понять неправильно:**
- LLM может подумать, что после создания коммита hooks НЕ запускаются при `--amend`
- Непонятно, что происходит при `--amend`: hooks запускаются повторно или нет?

**Рекомендация:**
> **Важно:**
> - Не использовать `git commit --amend` после провала hooks — коммит ещё не создан, нечего amend'ить.
> - При использовании `git commit --amend` на существующем коммите — hooks НЕ запускаются повторно (изменяется только message или добавляются файлы).

**Приоритет:** P2

### Проблема 2.6: Неясность "коммит уже запушен"

**Где:** § 7 "Исправление коммитов" (строка 178)

**Цитата:**
> **Ограничение:** `--amend` изменяет историю. Не использовать, если коммит уже запушен в remote. В этом случае — создать новый коммит с исправлением.

**Как LLM может понять неправильно:**
- LLM может интерпретировать "уже запушен" как "уже запушен и кто-то уже сделал pull"
- Непонятно, что делать, если коммит запушен, но работа ведётся в feature-ветке (только я работаю)

**Рекомендация:**
> **Ограничение:** `--amend` изменяет историю.
> - Запрещено использовать, если коммит уже запушен в общую ветку (main, develop) или другой разработчик сделал pull
> - Допустимо в feature-ветке, если никто кроме вас не работает с веткой (после amend потребуется `git push --force-with-lease`)
> - Безопасная альтернатива: создать новый коммит с исправлением (`fix: correct previous commit`)

**Приоритет:** P1

## 5. Слепые зоны автора

### Зона 5.1: Локализация сообщений коммитов

**Суть:** Документ не упоминает язык сообщений (английский или русский).

**Почему важно:**
- В командах с международными участниками требуется английский
- В локальных проектах допустим русский
- Отсутствие правила приводит к смешанным коммитам (часть на русском, часть на английском)

**Расширение качества:** Умеренное

**Рекомендация:** Добавить сейчас

**Пример добавления:**

> ## 8. Язык сообщений
>

### Зона 5.3: Массовые изменения (bulk commits)

**Суть:** Документ не описывает, как оформлять коммиты, затрагивающие множество файлов (например, обновление зависимостей во всех package.json).

**Почему важно:**
- LLM может создать один коммит на 100 файлов без структурирования
- Отсутствие правила приводит к неудобной истории

**Расширение качества:** Умеренное

**Рекомендация:** Добавить сейчас

**Пример добавления:**

> **Массовые изменения:**
> - Если изменение затрагивает > 10 файлов одного типа — использовать общий scope
> - Указать в body количество затронутых файлов
>
> **Пример:**
> ```
> chore: update dependencies across all services
>
> Updated 15 package.json files to use latest versions.
> All tests passed after update.
> ```

**Приоритет:** P3

# plantest-reviewer — запись PROP-N в plan-test.md

plantest-reviewer должен записывать свои замечания как PROP-N предложения прямо в plan-test.md (аналогично design-reviewer → design.md). Сейчас ревьюер read-only — возвращает вердикт в чат, но не оставляет следов в документе. Требуются изменения в агенте, стандарте, шаблоне, скрипте создания файла, валидаторе и инструкциях.

## Контекст

**Задача:** При тестовом прогоне `/chain --resume 0001` (цепочка `0001-task-dashboard`) пользователь заметил, что plantest-reviewer не записывает свои замечания в plan-test.md, хотя design-reviewer записывает PROP-N предложения прямо в design.md (секции "Предложения" и "Отвергнутые предложения").

**Почему:** plantest-reviewer (v1.0) создавался как read-only агент — без Edit/Write tools. Паттерн design-reviewer (v2.0), который пишет PROP-N прямо в документ, не был перенесён на plantest-reviewer при его создании.

**Связанные файлы:**

Эталон (design-reviewer):
- [design-reviewer AGENT.md](/.claude/agents/design-reviewer/AGENT.md) — v2.0, tools: Read/Grep/Glob/**Edit**, пишет PROP-N

Текущий (plantest-reviewer):
- [plantest-reviewer AGENT.md](/.claude/agents/plantest-reviewer/AGENT.md) — v1.0, tools: Read/Grep/Glob (без Edit!), read-only

Стандарт, шаблон, инструкции:
- [standard-plan-test.md](/specs/.instructions/analysis/plan-test/standard-plan-test.md) — § 5, § 7, § 8 — нет секции "Предложения"
- [create-plan-test.md](/specs/.instructions/analysis/plan-test/create-plan-test.md) — шаг 6 возвращает вердикт в чат
- [create-analysis-plan-test-file.py](/specs/.instructions/.scripts/create-analysis-plan-test-file.py) — не генерирует "Предложения"
- [validate-analysis-plan-test.py](/specs/.instructions/.scripts/validate-analysis-plan-test.py) — "Предложения" не в special set
- [validation-plan-test.md](/specs/.instructions/analysis/plan-test/validation-plan-test.md) — чек-лист без "Предложения"

### 3. Эталон: design-reviewer

Ключевые механики design-reviewer (v2.0) для переноса:

**3.1 Секция "Предложения":** после "Блоки тестирования", группировка по `###` аспектам, каждый PROP-N: ID + Приоритет + Предложение + Влияет на. Заглушка: "_Предложений нет._"

**3.2 Секция "Отвергнутые предложения":** после "Предложения", формат: ID + Приоритет + Предложение + Причина отклонения.

**3.3 Tools:** `Read, Grep, Glob, Edit` (Edit для записи PROP-N), `permissionMode: default`.

**3.4 Алгоритм:** определить следующий PROP-N → создать/обновить секцию через Edit → не дублировать при повторном запуске.

**Стало:** "Ты читаешь Plan Tests-документ и parent Design/Discussion, определяешь пробелы в покрытии и записываешь рекомендации как PROP-N прямо в секцию 'Предложения'. Если секции нет — создаёшь через Edit после 'Блоки тестирования'."

#### 4.3 Ограничения (строки 93-97)

**Было:** 5 пунктов, ключевой — "НЕ редактировать plan-test.md", "ТОЛЬКО читать и выносить вердикт".

```
- НЕ изменять frontmatter / статус документа
- ТОЛЬКО добавлять PROP-N записи в секцию "Предложения"

```

#### 4.4 Область работы (строки 76-77)

**Было:** "Запись: нет (read-only ревьюер)"

**Стало:** "Запись: только секция 'Предложения' в plan-test.md (через Edit)"

#### 4.5 Новый алгоритм PROP-N

Добавить "Фаза 4: Запись PROP-N в документ" после "Фаза 3: Формирование вердикта":

1. Определить следующий свободный PROP-N
2. Сгруппировать расхождения по аспектам (### подзаголовки)
3. Записать PROP-N через Edit
4. Вернуть резюме в чат

**Категории предложений:**

| Категория | Формат описания |
|-----------|----------------|
| Покрытие REQ-N | "REQ-N не покрыт ни одним TC" |
| Покрытие STS-N | "STS-N не покрыт ни одним TC" |
| Формат TC-N | "TC-N: Given/When/Then вместо естественного предложения" |
| Fixtures | "TC-N: fixture `{name}` отсутствует в Тестовые данные {SVC-N}" |
| BLOCK-N | "TC-N не принадлежит ни одному BLOCK" |
| Межсекционные fixtures | "Fixture `{name}` определён в SVC-X, используется в SVC-Y" |
| Антигаллюцинация | "TC-N: INVENTED — тестирует {что}, отсутствующее в Design" |
| Зона ответственности | "TC-N содержит {что} — это зона Plan Dev/Design" |

**Группировка по аспектам:** Покрытие требований, Формат и данные, Блоки тестирования, Антигаллюцинации, Зона ответственности.

#### 4.6 Формат вывода (строки 99-121)

**Было:** Таблица расхождений с колонками `#, Приоритет, Критерий, Описание`.

**Стало:**
```markdown
## Ревью плана тестов: NNNN-{topic}

**Документ:** specs/analysis/NNNN-{topic}/plan-test.md
**Покрытие:** REQ-N {covered}/{total}, STS-N {covered}/{total}

### {Аспект}

|------|-----------|-----------|----------|
| PROP-1 | P1 | {категория} | {описание} |

**Записано в:** plan-test.md → секция "Предложения"
```

### 9. Workflow before/after

**Было (v1.0 — read-only):**

```
Оркестратор ──► plantest-reviewer ──► Read plan-test.md
                                  ──► Read design.md, discussion.md
              ◄── Вердикт в чат (ACCEPT/REVISE + расхождения)
                  (расхождения НЕ в документе, теряются)
              ──► if REVISE: plantest-agent (из памяти чата)
```

**Стало (v2.0 — запись PROP-N):**

```
Оркестратор ──► plantest-reviewer ──► Read plan-test.md
                                  ──► Read design.md, discussion.md
                                  ──► Edit plan-test.md (PROP-N → "Предложения")
              ◄── Резюме в чат (кол-во PROP, аспекты, вердикт)
              ──► if REVISE: plantest-agent (читает PROP-N из документа)
              ──► after ACCEPT: AskUserQuestion (показать PROP-N пользователю)
                  Отклонённые → "Отвергнутые предложения"
```

**Ключевые отличия:**
- plantest-reviewer записывает PROP-N через Edit (а не возвращает в чат)
- plantest-agent при REVISE читает PROP-N из документа (персистентный источник)
- После ACCEPT пользователь видит PROP-N и может принять/отклонить
- Аудитный след: все замечания сохранены в plan-test.md

## Решения

| ID | Решение | Обоснование |
|----|---------|-------------|
| D-1 | plantest-reviewer записывает PROP-N в plan-test.md (как design-reviewer) | Единообразие chain-документов, аудитный след, переиспользуемость замечаний |
| D-2 | Секции "Предложения" + "Отвергнутые предложения" в standard-plan-test.md | Паттерн из standard-design.md, проверен на практике |
| D-3 | Вердикт определяется по PROP-N: нет P1 и ≤ 2 P2 → ACCEPT | Тот же критерий что у plantest-reviewer сейчас, но PROP-N записываются в документ |

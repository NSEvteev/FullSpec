## 1. Двойственность формулировок

### Проблема 1.1: Неопределённое условие "может быть"

**Где:** Секция "4. Жизненный цикл" → подсекция "После завершения работы" (строка 118)

**Цитата:**
> Черновик может быть **источником материала** для дискуссии

**Как может понять LLM:**
- Интерпретация A: Черновик иногда становится источником (зависит от содержания)
- Интерпретация B: Черновик всегда может стать источником (это одна из опций)
- Интерпретация C: Процесс переноса опционален (не обязателен)

**Рекомендация:**
> Черновик МОЖЕТ быть источником материала для дискуссии. Критерии переноса:
> - Черновик содержит проработанное решение или анализ альтернатив
> - Материал требует формального обсуждения в команде
> - Пользователь инициирует создание дискуссии на основе черновика

**Критичность:** Средняя

---

### Проблема 1.2: Неопределённое "спонтанно"

**Где:** Секция "5. Правила работы" → подсекция "Создание" (строка 131)

**Цитата:**
> Черновики создаются Claude **спонтанно** в процессе работы

**Как может понять LLM:**
- Интерпретация A: Без явного запроса пользователя (автоматически)
- Интерпретация B: По инициативе Claude при необходимости
- Интерпретация C: В любой момент, без правил

**Рекомендация:**
> Черновики создаются Claude по инициативе (без явного запроса пользователя) в следующих случаях:
> - Планирование задачи требует более 3 шагов
> - Анализ альтернативных решений (сравнение подходов)
> - Исследование архитектуры перед реализацией
> - Сложные размышления, требующие структурированной фиксации
>
> ЗАПРЕЩЕНО создавать черновик для простых задач (1-2 шага).

**Критичность:** Высокая

---

### Проблема 1.3: Противоречие "временные" vs "оставить для истории"

**Где:** Секция "4. Жизненный цикл" (строки 109, 115)

**Цитата 1:**
> | Жизненный цикл | Временные |

**Цитата 2:**
> | Оставить | Для истории решений |

**Как может понять LLM:**
- Интерпретация A: Временные = удалить после использования
- Интерпретация B: Временные = не постоянная документация, но хранятся как история

**Рекомендация:**
> | Жизненный цикл | Временные (не являются постоянной документацией, но могут храниться как история) |

**И добавить пояснение:**
> **Важно:** "Временные" означает, что черновики НЕ являются SSOT и НЕ заменяют постоянную документацию. Однако они МОГУТ храниться в git как история принятия решений.

**Критичность:** Средняя

## 2. Риск неправильной интерпретации

### Проблема 2.1: Отсутствие границ "спонтанного создания"

**Где:** Секция "5. Правила работы" → подсекция "Создание" (строка 131)

**Цитата:**
> Черновики создаются Claude **спонтанно** в процессе работы

**Риск:** LLM может создавать черновики слишком часто (при каждом размышлении) или наоборот — никогда не создавать, считая это инициативой пользователя.

**Рекомендация:**
> Добавить секцию "5.1. Когда создавать черновик":
>
> ```markdown
> ## 5.1. Когда создавать черновик
>
> **Создавать:**
> - Планирование задачи > 3 шагов
> - Сравнение 2+ альтернативных решений
> - Исследование архитектуры/API перед реализацией
> - Анализ технических ограничений
>
> **НЕ создавать:**
> - Простые задачи (1-2 шага)
> - Рутинные операции (создание файла, правка README)
> - Ответы на вопросы пользователя
> ```

**Критичность:** Высокая

### Проблема 2.2: Неявная зависимость README.md

**Где:** Секция "4. Жизненный цикл" → "Обновление индекса" (строка 123)

**Цитата:**
> При добавлении/удалении черновика — обновить таблицу в [drafts/README.md](/.claude/drafts/README.md).

**Риск:** LLM может:
- Создать черновик, но забыть обновить README.md
- Не понять, что обновление README.md — часть процесса создания, а не отдельный шаг

**Рекомендация:**
> **Обязательный шаг после создания/удаления:**
>
> ПОСЛЕ создания черновика:
> 1. Добавить запись в `/.claude/drafts/README.md` → таблица "1. Файлы"
> 2. Формат записи: `| [имя-файла.md](./имя-файла.md) | Описание |`
>
> ПОСЛЕ удаления черновика:
> 1. Удалить запись из `/.claude/drafts/README.md` → таблица "1. Файлы"

**Критичность:** Высокая

### Проблема 2.3: Размытая граница "привязка к задачам"

**Где:** Секция "5. Правила работы" → "Привязка к задачам" (строка 146)

**Цитата:**
> Каждый черновик ДОЛЖЕН быть связан с конкретной задачей или текущей сессией работы.

**Риск:** LLM не понимает, как именно связать черновик с задачей:
- Добавить ссылку на задачу в черновик?
- Упомянуть задачу в контексте?
- Именовать файл с ID задачи?

**Рекомендация:**
> Каждый черновик ДОЛЖЕН содержать в секции "Контекст" явную привязку:
>
> **Задача:** `{описание задачи или ссылка на issue/ticket}`
> **Или**
> **Текущая сессия:** `{дата} — {тема работы}`
>
> Пример:
> ```markdown
> ## Контекст
>
> **Задача:** Рефакторинг аутентификации (issue #123)
> **Почему создан:** Анализ миграции JWT → OAuth2
> ```

**Критичность:** Средняя

## 4. Полнота описания объекта

### 4.1. Не хватает: Критерии удаления черновика

**Где:** Секция "4. Жизненный цикл" → "После завершения работы" (строка 116)

**Цитата:**
> | Удалить | Информация перенесена в финальные документы |

**Почему важно:** Не указано, КТО принимает решение (пользователь или LLM) и КАК проверить "перенесена ли информация".

**Стоит ли добавлять:** Да

**Что даст:** Устранит риск преждевременного удаления важных черновиков.

**Рекомендация:** Добавить подсекцию:

```markdown
### Когда удалять черновик

- Информация полностью перенесена в финальные документы (README, инструкции, specs)
- Пользователь подтвердил удаление через AskUserQuestion
- ЗАПРЕЩЕНО удалять черновик автоматически без подтверждения

**Проверка перед удалением:**
1. Спросить пользователя: "Черновик `{имя}` — вся информация перенесена?"
2. Если "Да" → удалить файл + запись из README.md
3. Если "Нет" → оставить черновик
```

**Приоритет:** P2

## 5. Слепые зоны автора

### Зона 5.1: Поиск по черновикам

**Суть:** Документ не рассматривает, как искать черновики по содержимому, дате или связанным задачам.

**Почему важно:** При накоплении 50+ черновиков становится критично их находить.

**Расширение качества:** Умеренное

**Рекомендация:** Добавить сейчас (опционально)

**Пример добавления:**

```markdown
## 8. Поиск черновиков

### По дате

```bash
ls -1 .claude/drafts/ | grep "^2024-01-"
```

### По теме

```bash
grep -r "auth" .claude/drafts/
```

### По задаче

Указывать ID задачи в секции "Контекст" для быстрого поиска:

```bash
grep -r "issue #123" .claude/drafts/
```

### Все черновики за период

```bash
ls -1 .claude/drafts/ | grep "^2024-01-2[0-9]"
```
```

**Приоритет:** P3

**Почему важно:** Без явной поддержки связей LLM не будет их создавать.

**Расширение качества:** Минимальное

**Рекомендация:** Не нужно (over-engineering). Черновики — временные документы, сложные связи избыточны.

**Приоритет:** —

### Зона 5.3: Автоматическая архивация старых черновиков

**Суть:** Через год в `/.claude/drafts/` может накопиться 200+ файлов. Стандарт не упоминает архивацию.

**Почему важно:** Замедление git, сложность навигации.

**Расширение качества:** Умеренное

**Рекомендация:** Отложить. Добавить правило при возникновении проблемы:

```markdown
## 9. Архивация

Черновики старше 6 месяцев перемещаются в `/.claude/drafts/archive/YYYY/`.

**Автоматизация:**

```bash

**Приоритет:** P3 (отложить до накопления > 100 черновиков)

**Рекомендация:** Не нужно. Черновики в git — достаточно для истории изменений. Явный workflow избыточен.

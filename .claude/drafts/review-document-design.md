# Дизайн: Документ ревью кода в analysis chain (review.md)

Драфт для проработки нового типа документа в `specs/analysis/`.

---

## Контекст

Ревью кода — этап разработки, когда код уже написан, тесты проходят, и нужно проверить качество реализации перед merge. Сейчас результат ревью существует только в чате или в комментариях к PR — нет трассируемого документа.

**Что уже есть (ревью документов):** discussion-reviewer, design-reviewer — проверяют документы analysis chain на полноту. Это НЕ про код.

**Что нужно (ревью кода):** Агент code-reviewer проверяет написанный код против постановки и стандартов. Результат фиксируется в `review.md`.

## Место в процессе разработки

Review.md НЕ является уровнем analysis chain (Discussion → Design → Plan Tests → Plan Dev). Это **артефакт этапа разработки** — создаётся при RUNNING, когда код уже написан.

```
specs/analysis/NNNN-{topic}/
├── discussion.md      # Ур. 1: WHY + WHAT
├── design.md          # Ур. 2: HOW
├── plan-test.md       # Ур. 3: HOW TO VERIFY
├── plan-dev.md        # Ур. 4: WHAT TASKS
└── review.md          # Артефакт ревью кода (создаётся при RUNNING)
```

**В process-map (шаги 10, 12):**
- Шаг 10: Branch Review — `/review` → code-reviewer → review.md (до создания PR)
- Шаг 12: PR Review — `/review {N}` → code-reviewer → обновление review.md + gh pr comment

## Входные данные code-reviewer

Агент читает данные из трёх контуров: код, постановка, стандарты.

### Контур 1: Код (git diff)

| Источник | Что извлекает |
|----------|---------------|
| `git diff {base}...HEAD` / `gh pr diff {N}` | Изменённые файлы, добавленные/удалённые строки |
| Каждый изменённый файл (Read целиком) | Полный контекст: структура, API, зависимости |
| Импорты и вызывающий код (Grep) | Совместимость изменений с потребителями |
| Тесты для изменённого кода | Покрытие новой логики |

### Контур 2: Постановка (specs/analysis/{branch-name}/)

Имя ветки = имя папки analysis chain.

| Файл | Что извлекает | Для чего |
|------|---------------|----------|
| `discussion.md` | F-N (фичи), REQ-N (требования), критерии успеха, "Вне scope" | Сверка: все ли REQ-N реализованы? Нет ли выхода за scope? |
| `design.md` | SVC-N §§ 1-9 (ответственность, API, Data Model, решения), INT-N (контракты), STS-N (системные тесты) | Сверка: API-сигнатуры, data model, контракты между сервисами |
| `plan-test.md` | TC-N (тестовые сценарии), матрица покрытия | Сверка: реализованы ли ожидаемые тесты? |
| `plan-dev.md` | TASK-N (задачи), подзадачи, зависимости | Сверка: выполнены ли задачи? Соблюдён ли порядок? |

### Контур 3: Стандарты и документация (docs/ + .instructions/)

| Источник | Что извлекает | Для чего |
|----------|---------------|----------|
| `docs/{svc}.md` §§ 1-8 | AS IS: API контракты (§ 2), Data Model (§ 3), потоки (§ 4), Code Map (§ 5), зависимости (§ 6), доменная модель (§ 7) | Сверка: не нарушены ли существующие контракты? Совместимы ли изменения с AS IS? |
| `docs/{svc}.md` § 8 | **Границы автономии LLM** (Свободно / Флаг / CONFLICT) | **Ключевое для P1:** определяет, является ли изменение рабочей правкой или расхождением (→ § 6.3) |
| `docs/{svc}.md` § 9 | Planned Changes (дельты ADDED/MODIFIED/REMOVED из Design) | **Ключевое для P1:** сравнить запланированные изменения с фактическим кодом. Расхождение между Planned Changes и diff — это P1 |
| `docs/{svc}.md` § 1 | Criticality level сервиса (critical-high / critical-medium / critical-low) | Определяет глубину ревью: пороги покрытия тестами, количество reviewers |
| `docs/.system/overview.md` | Архитектура, карта сервисов, сквозные потоки (§ 4), контекстная карта доменов (§ 5) | Проверка: не ломает ли код архитектурные потоки между сервисами? |
| `docs/.system/conventions.md` | Формат ответов (§ 1), формат ошибок (§ 2), пагинация (§ 3), аутентификация (§ 4), версионирование API (§ 5), shared-пакеты (§ 6), логирование (§ 7) | Проверка: соответствуют ли API-ответы, ошибки, пагинация единым конвенциям? |
| `docs/.system/testing.md` | Типы тестов (§ 1), структура файлов (§ 2), стратегия мокирования (§ 3), межсервисные тесты (§ 4), тестовые данные (§ 5) | Проверка: соответствуют ли тесты стратегии? Правильно ли расположены? |
| `docs/.system/infrastructure.md` | Порты (§ 2), хранилища данных (§ 3), брокер сообщений (§ 4), окружения (§ 6) | Проверка при инфраструктурных изменениях: порты, конфигурации, топики |
| `docs/.technologies/standard-{tech}.md` | Версия и настройка (§ 1), конвенции именования (§ 2), паттерны кода (§ 3), **антипаттерны** (§ 4), структура файлов (§ 5), тестирование (§ 7), логирование (§ 8) | Проверка кода по per-tech стандартам. § 4 антипаттерны — прямой источник P3 замечаний |
| `.instructions/standard-principles.md` | Принципы кодирования проекта | Проверка качества кода |

### Роль Planned Changes и границ автономии

**Planned Changes (§ 9) как эталон для P1:**

Design при переходе в WAITING создаёт Planned Changes в `docs/{svc}.md` § 9 — структурированные дельты (ADDED/MODIFIED/REMOVED) для каждой секции. Reviewer сравнивает:

```
Planned Changes (§ 9)    vs    Фактический код (diff)
──────────────────────         ─────────────────────
ADDED: POST /api/users  →     Есть в коде? Сигнатура совпадает?
MODIFIED: таблица users  →     Миграция есть? Колонки совпадают?
REMOVED: GET /api/legacy →     Удалён из кода?
```

Если код реализует что-то, чего нет в Planned Changes, или не реализует то, что есть — это P1.

**Границы автономии (§ 8) как критерий P1 vs рабочая правка:**

| Граница в § 8 | Изменение в коде | Приоритет |
|---------------|-----------------|-----------|
| **Свободно** (внутри пакета) | Рефакторинг, оптимизация | Не замечание (допустимо) |
| **Флаг** (контракты между пакетами) | Изменение внутреннего API пакета | **P2** (дефект — нужен флаг) |
| **CONFLICT** (API сервиса, data model, архитектура) | Изменение, не предусмотренное в Design | **P1** (расхождение) |

### Criticality level и глубина ревью

Criticality из `docs/{svc}.md` § 1 влияет на ожидания reviewer:

| Criticality | Unit-test coverage | Integration tests | E2E tests |
|-------------|-------------------|-------------------|-----------|
| critical-high | >= 80% | Обязательны | Обязательны |
| critical-medium | >= 60% | Обязательны | Обязательны |
| critical-low | >= 40% | Рекомендуются | Опционально |

Если код для critical-high сервиса не имеет 80% coverage — это P2 (дефект — недостаточное покрытие).

## Классификация замечаний

Каждое замечание (RV-N) классифицируется по приоритету. Приоритет определяет **действие**, а не субъективную "серьёзность".

| Приоритет | Название | Суть | Действие |
|-----------|----------|------|----------|
| **P1** | **Расхождение** | Реализация расходится с постановкой (REQ-N, SVC-N, INT-N) | → Автоматически CONFLICT (§ 6.3). Top-down ревью всех документов начиная с Discussion |
| **P2** | **Дефект** | Реализация неполная, что-то забыто, пропущен edge case, реальный баг | → AskUserQuestion: исправлять или принять как есть |
| **P3** | **Нарушение** | Несоответствие стандартам (per-tech, conventions, principles), стилистика | → AskUserQuestion: исправлять или принять как есть |

### Два исхода P1

P1 (расхождение) означает, что код и постановка **говорят разное**. Причина может быть любой — неважно кто "прав". Оба исхода проходят через один и тот же механизм CONFLICT (§ 6.3-6.4):

| Исход | Что правится | Пример |
|-------|-------------|--------|
| **Принять код** | Постановка обновляется под реализацию. Top-down: Discussion → Design → Plan Tests → Plan Dev | Разработчик нашёл лучший алгоритм, чем предусмотрен в Design |
| **Принять постановку** | Код переписывается под спецификации. Specs не меняются, но проходят верификацию | Разработчик неправильно понял требование REQ-N |

В обоих случаях: вся цепочка → CONFLICT → top-down разрешение (§ 6.4) → каждый документ ревьюится → WAITING → RUNNING.

## Что именно проверяет ревью

### Сверка с постановкой (контур 2) → P1 или P2

| Проверка | Приоритет |
|----------|-----------|
| Реализация ПРОТИВОРЕЧИТ REQ-N из discussion.md | **P1** (расхождение) |
| API-сигнатуры НЕ соответствуют SVC-N § 2 из design.md | **P1** (расхождение) |
| Data Model НЕ соответствует SVC-N § 3 из design.md | **P1** (расхождение) |
| Контракты INT-N из design.md НЕ соблюдены | **P1** (расхождение) |
| REQ-N не реализован (но не противоречит) | **P2** (дефект — неполная реализация) |
| TC-N из plan-test.md не реализован | **P2** (дефект — забыт тест) |
| TASK-N из plan-dev.md не выполнен | **P2** (дефект — забыта задача) |
| Реализация выходит за "Вне scope" из discussion.md | **P2** (дефект — лишний код) |

### Анализ кода (контур 1) → P2

| Проверка | Приоритет |
|----------|-----------|
| Некорректная логика, edge cases, off-by-one | **P2** (дефект — баг) |
| Несовместимость с вызывающим кодом | **P2** (дефект — сломает потребителей) |
| Race conditions, утечки ресурсов | **P2** (дефект) |
| SQL-инъекции, XSS, секреты в коде | **P2** (дефект — уязвимость) |
| Пропущенные тесты для новой логики | **P2** (дефект — нет покрытия) |

### Проверка по стандартам (контур 3) → P3

| Проверка | Приоритет |
|----------|-----------|
| Код не соответствует per-tech стандарту (standard-{tech}.md) | **P3** (нарушение) |
| Антипаттерны из standard-{tech}.md § 4 | **P3** (нарушение) |
| API-ответы не соответствуют conventions.md | **P3** (нарушение) |
| Тесты не соответствуют testing.md (типы, мокирование) | **P3** (нарушение) |
| Нарушение принципов из standard-principles.md | **P3** (нарушение) |
| Debug-код, закомментированный код | **P3** (нарушение) |
| TODO без ссылки на Issue | **P3** (нарушение) |

### Граница P1 vs P2

**P1 — реализация расходится с постановкой:** код делает НЕ ТО, что описано в specs. API-путь другой, data model другая, контракт другой, поведение противоречит требованию.

**P2 — реализация неполная или содержит баг:** код делает ТО, что описано, но не всё или с ошибкой. Забыт edge case, пропущен тест, не реализована задача, баг в логике.

**Правило:** если замечание можно сформулировать как "код говорит X, а постановка говорит Y" — это P1. Если "код делает X, но не делает Y" — это P2.

## Связь с механизмом CONFLICT (standard-analysis.md § 6.3)

Review.md формализует механизм обратной связи Code → Specs:

| Приоритет | Действие | Механизм § 6.3 |
|-----------|----------|----------------|
| **P1** (расхождение) | → Автоматически CONFLICT | Вся цепочка → CONFLICT. Top-down разрешение (§ 6.4): Discussion → Design → Plan Tests → Plan Dev → WAITING → RUNNING |
| **P2** (дефект) | → AskUserQuestion | Пользователь решает: исправить код (рабочие правки) или принять как есть (wontfix) |
| **P3** (нарушение) | → AskUserQuestion | Пользователь решает: исправить код или принять как есть |

**P1 — единственный приоритет, который автоматически инициирует CONFLICT.** P2 и P3 не затрагивают specs — это вопросы качества кода, которые решает пользователь.

## Формат review.md (предварительный)

```markdown
---
description: Ревью кода для {NNNN}-{topic} — {N} замечаний (P1:{N}, P2:{N}, P3:{N}).
standard: specs/.instructions/analysis/review/standard-review.md
standard-version: v1.0
index: specs/analysis/README.md
status: OPEN
iteration: 1
---

# review: {NNNN} {Тема}

**Ветка:** {branch-name}
**Base:** {base-branch}
**Файлов изменено:** {N}
**Дата:** {YYYY-MM-DD}

## Сверка с постановкой

- REQ-N покрытие: {X}/{Y} требований реализовано
- Расхождения: {есть (RV-N) / нет}
- Вне scope: {есть (RV-N) / нет}
- TASK-N выполнение: {X}/{Y}
- TC-N покрытие: {X}/{Y}

## Замечания

### RV-1: {краткое описание}

- **Файл:** `{path}:{line}`
- **Приоритет:** {P1 расхождение / P2 дефект / P3 нарушение}
- **Ссылка:** {REQ-N / TASK-N / TC-N / SVC-N / INT-N / standard-{tech}.md / —}
- **Контекст:** {почему это проблема — что говорит код vs что говорит постановка/стандарт}
- **Рекомендация:** {что исправить}
- **Статус:** {open / resolved / wontfix}

### RV-2: ...

## Итого

| Приоритет | Название | Количество | Open | Resolved | Wontfix |
|-----------|----------|-----------|------|----------|---------|
| P1 | Расхождение | {N} | {N} | {N} | — |
| P2 | Дефект | {N} | {N} | {N} | {N} |
| P3 | Нарушение | {N} | {N} | {N} | {N} |

**Вердикт:** {READY / NOT READY / CONFLICT}

- **CONFLICT** — есть хотя бы 1 open P1 → вся цепочка → CONFLICT (§ 6.3)
- **NOT READY** — нет P1, но есть open P2 → пользователь решает по каждому
- **READY** — нет open P1/P2, P3 resolved или wontfix
```

### Отличия frontmatter от 4 уровней chain

| Поле | 4 уровня chain | review.md |
|------|----------------|-----------|
| `parent` | Путь к родительскому документу | **Нет** (не участвует в иерархии parent/children) |
| `children` | Путь к дочернему документу | **Нет** |
| `status` | DRAFT/WAITING/RUNNING/DONE/CONFLICT/ROLLING_BACK/REJECTED | **OPEN/RESOLVED** (свой цикл) |
| `milestone` | Наследуется от Discussion | **Нет** (определяется контекстом цепочки) |
| `iteration` | **Нет** | Номер итерации ревью (1, 2, 3...) |

### Lifecycle review.md

```
OPEN → P1 найден → CONFLICT цепочки → specs/код правятся → re-review → OPEN (iteration +1)
OPEN → P2/P3 найдены → пользователь решает → код правится или wontfix → re-review → OPEN (iteration +1)
OPEN → всё resolved/wontfix → RESOLVED
```

- **OPEN:** есть open замечания (P1, P2 или P3)
- **RESOLVED:** все P1 resolved, все P2/P3 resolved или wontfix. Можно merge.

Review.md **не участвует** в каскадах DONE, CONFLICT, ROLLING_BACK. Он может быть **причиной** CONFLICT (через P1), но сам переходит только OPEN → RESOLVED.

**P1 и wontfix:** P1 не может быть wontfix. Расхождение между кодом и постановкой ОБЯЗАТЕЛЬНО разрешается — либо правится код, либо правится постановка.

## Открытые вопросы

### Существующие

1. Как обрабатывать повторные запуски? (Обновление review.md с iteration +1 vs перезапись)
2. Как фиксировать resolved? (Статус `resolved` в RV-N или перенос в отдельную секцию)
3. Нужен ли отдельный скилл `/review-create` или `/review` сам создаёт review.md?
4. Нужна ли валидация review.md через pre-commit hook?
5. Как review.md учитывается при каскаде DONE? (Должен ли быть RESOLVED перед Plan Dev → DONE?)

### Новые (выявлены при проверке полноты)

6. **Ветки без analysis chain.** Hotfix, chore, documentation-ветки не имеют `specs/analysis/{branch-name}/`. Варианты:
   - (a) review.md создаётся только при наличии analysis chain, иначе — лёгкое ревью без документа
   - (b) review.md создаётся всегда, но секция "Сверка с постановкой" помечается N/A
   - (c) review.md создаётся всегда, контуры 2-3 проверяются только при наличии specs

7. **Определение scope ревью.** Как reviewer определяет, какие `docs/{svc}.md` и `standard-{tech}.md` читать? Варианты:
   - (a) По путям файлов в diff: `src/{service}/` → `docs/{service}.md`
   - (b) По расширениям в diff: `.py` → `standard-python.md`, `.ts` → `standard-typescript.md`
   - (c) Из plan-dev.md: TASK-N содержит service и tech-stack
   - (d) Комбинация: plan-dev.md если есть, иначе вывод из путей и расширений

8. **Мультисервисные изменения.** Один diff затрагивает несколько сервисов. Варианты:
   - (a) Одна review.md с RV-N для всех сервисов (текущий дизайн)
   - (b) Отдельная review.md на сервис (review-auth.md, review-gateway.md)

## 2. Риск неправильной интерпретации

### Проблема 2.1: Цикл работы с файлами — может зациклиться

**Где:** Секция "Работа со state", шаги 1-7, строки 138-148

**Цитата:**
> 3. **Если заблокирован:**
>    - Проверять каждые 5 секунд
>    - Таймаут: 30 секунд от момента первой проверки

**Как может понять LLM:**
LLM может попытаться реализовать цикл ожидания прямо в коде агента, что невозможно в среде Claude Code (нет sleep/wait механизма).

**Риск:**
Агент может зациклиться, пытаясь читать locks.json в бесконечном цикле, или остановиться без объяснения.

**Рекомендация:**
> 3. **Если заблокирован:**
>    - Прочитать locks.json один раз
>    - Если файл заблокирован — вернуть сообщение пользователю: "Файл {путь} заблокирован агентом {имя}, ожидаю освобождения"
>    - Остановить выполнение, ждать повторного вызова пользователем
>    - При повторном вызове — снова проверить locks.json
>    - Если блокировка держится более 30 секунд от первой попытки — вернуть ошибку

**Критичность:** Высокая

### Проблема 2.2: "Использование скиллов" — противоречие с Edit/Write

**Где:** Секция "Использование скиллов", строка 109

**Цитата:**

**Риск:**
Агент может решить, что Edit/Write запрещены полностью, или наоборот — использовать их вместо скиллов.

> 3. Если скилл не сработал — остановиться и вернуть ошибку, НЕ использовать Edit/Write как fallback
> 4. Исключение: скилл явно указывает на использование Edit/Write внутри воркфлоу (тогда это часть скилла)

### Проблема 2.3: "После завершения работы" — когда это?

**Где:** Секция "Работа со state", строка 151

**Цитата:**
> **После завершения работы:**
> - Обновить `finished_at` в `agent-amy-santiago-operation.json`

**Как может понять LLM:**
- Интерпретация A: После выполнения всех шагов задачи
- Интерпретация B: После каждого вызова агента пользователем
- Интерпретация C: Только после финального возврата результата

**Риск:**
Если агент обновляет `finished_at` после каждого вызова, а пользователь вызывает его повторно в той же сессии — `finished_at` будет перезаписан некорректно.

**Рекомендация:**
> **Обновление finished_at:**
> - Обновить `finished_at` ТОЛЬКО когда агент вернул финальный отчёт пользователю
> - Если пользователь запросил дополнительные правки — НЕ обновлять `finished_at`, это продолжение той же работы
> - `finished_at` означает: задача полностью завершена, агент больше не будет вызван в рамках этой задачи

**Критичность:** Средняя

Пользователю может быть полезна статистика: сколько инструкций создано, сколько валидаций пройдено.

**Стоит ли добавлять:** Нет (over-engineering для текущей версии)

**Что даст:**
Дополнительная информация для пользователя, но не критична для работы агента.

**Рекомендация:**
Оставить как есть. Можно добавить в будущей версии (v1.1+).

## 5. Слепые зоны автора

### Зона 5.1: Работа в многопользовательской среде

**Суть:**
Документ не рассматривает сценарий: два пользователя одновременно работают с проектом (например, через git).

**Почему важно:**
Блокировки в `locks.json` локальны — они не синхронизируются между машинами. Если два разработчика запустят Amy Santiago одновременно на разных машинах, произойдёт конфликт.

**Расширение качества:** Значительное

**Рекомендация:** Добавить сейчас

**Пример добавления:**
> ## Ограничения
>
> ...
> - НЕ поддерживает одновременную работу нескольких пользователей (locks.json локальны)
> - При работе в команде — координация через git (один пользователь — один агент)

### Зона 5.3: Dry-run режим

**Суть:**
Возможность запустить агента в режиме "только показать, что будет сделано", без реальных изменений.

**Почему важно:**
Пользователь может проверить, что агент правильно понял задачу, перед применением изменений.

**Расширение качества:** Умеренное

**Рекомендация:** Добавить сейчас

**Пример добавления:**
> ## Dry-run режим
>
> Для предварительного просмотра без изменений:
> 1. Попросить пользователя: "Выполнить dry-run (только показать изменения)?"
> 2. Если "Да":
>    - Выполнить все шаги, НО вместо Edit/Write — показать diff в чате
>    - В отчёте указать: "Режим: dry-run (изменения не применены)"

### Зона 5.4: Batch-операции

**Суть:**
Создание/изменение нескольких инструкций одновременно (например, создать standard + validation + create + modify за один вызов).

**Почему важно:**
Сокращает количество вызовов агента при создании полного набора инструкций для объекта.

**Расширение качества:** Значительное

**Рекомендация:** Добавить сейчас

**Пример добавления:**
> ## Batch-операции
>
> При запросе "Создать полный набор инструкций для {object}":
> 1. Уточнить: "Создать standard, validation, create, modify для {object}?"
> 2. Создать standard-{object}.md
> 3. Валидировать
> 4. Создать validation-{object}.md
> 5. Создать create-{object}.md
> 6. Создать modify-{object}.md
> 7. Вернуть отчёт со списком созданных файлов

---

### Зона 5.5: Зависимости между инструкциями

**Суть:**
Документ не описывает, как агент должен обрабатывать ситуацию: пользователь хочет создать `create-api.md`, но `standard-api.md` ещё не существует.

**Почему важно:**
Нарушение порядка создания (standard → validation → create → modify) приведёт к некорректным инструкциям.

**Расширение качества:** Значительное

**Рекомендация:** Добавить сейчас (КРИТИЧНО)

**Пример добавления:**
> ## Зависимости между инструкциями
>
> **Правило:** Инструкции создаются в строгом порядке: standard → validation → create → modify.
>
> **Проверка перед созданием:**
> 1. Определить тип создаваемой инструкции (по префиксу)
> 2. Проверить наличие предыдущих типов:
>    - Для validation → требуется standard
>    - Для create → требуется validation (и соответственно standard)
>    - Для modify → требуется create (и все предыдущие)
> 3. Если предыдущий тип отсутствует:
>    - Вернуть ошибку: "Невозможно создать {type}-{object}.md без {prev-type}-{object}.md"
>    - Предложить: "Создать {prev-type}-{object}.md сначала?"

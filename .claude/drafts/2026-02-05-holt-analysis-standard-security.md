**Где:** Секция "1. Назначение", строка 56

**Цитата:**

**Как может понять LLM:**

- Интерпретация B: Включить желательно, но опционально (рекомендация)
- Интерпретация C: Проверить наличие языка, затем решить по ситуации (условие без действия)

**Рекомендация:**
> Code Scanning — включить ВСЕГДА, если в проекте используется один из поддерживаемых языков (Python, JavaScript/TypeScript, Go, Java, C/C++, C#, Ruby, Swift, Kotlin). Если языка нет в списке — пропустить настройку Code Scanning.

**Критичность:** Средняя (P2)

### Проблема 1.2: Неопределённость "при обнаружении CVE"

**Где:** Секция "3. Dependabot", строка 87

**Цитата:**
> GitHub сканирует lock-файлы и манифесты зависимостей, создаёт Security Alerts при обнаружении CVE.

**Как может понять LLM:**
- Интерпретация A: GitHub автоматически сканирует сразу после включения Dependabot Alerts
- Интерпретация B: GitHub сканирует по расписанию (неясна частота: раз в день? раз в неделю?)
- Интерпретация C: Сканирование запускается только при push новых коммитов

**Рекомендация:**
> GitHub сканирует lock-файлы и манифесты зависимостей автоматически:
> - При включении Dependabot Alerts — немедленное первое сканирование
> - Далее — при каждом push в default branch
> - Дополнительно — при обновлении GitHub Advisory Database (несколько раз в день)
>
> Security Alerts создаются при обнаружении CVE в используемых зависимостях.

**Критичность:** Низкая (P3)

### Проблема 1.4: Неявное условие "при срабатывании"

**Где:** Секция "5. Secret Scanning", строка 273-278

**Цитата:**
> ### Действия при срабатывании
> 1. **НЕ игнорировать алерт** — даже если секрет "тестовый"

**Как может понять LLM:**
- Интерпретация A: Выполнять все 4 шага всегда при любом алерте Secret Scanning
- Интерпретация B: Сначала проверить, является ли алерт false positive, затем решить выполнять или закрыть
- Интерпретация C: Выполнять только если секрет действительно был скомпрометирован (не тестовый)

**Рекомендация:**
> ### Действия при срабатывании Secret Scanning
>
> **Для ЛЮБОГО алерта (даже если кажется false positive):**
>
> 1. **Оценить риск:**
>    - Это настоящий секрет? (production API key, токен, пароль)
>    - Это тестовый/фиктивный секрет? (example.com, placeholder)
>    - Это false positive? (строка, похожая на секрет, но не являющаяся им)
>
> 2. **Если настоящий секрет:**
>    - Ротировать НЕМЕДЛЕННО (отозвать старый, выпустить новый)
>    - Очистить историю Git (если секрет в коммитах)
>    - Обновить использование секрета (GitHub Secrets, Environment)
>    - Закрыть алерт с причиной `revoked`
>
> 3. **Если тестовый/false positive:**
>    - Закрыть алерт с причиной `false positive` или `used in tests`
>    - Добавить комментарий с обоснованием

**Критичность:** Высокая (P1) — влияет на безопасность

## 2. Риск неправильной интерпретации

### Проблема 2.1: Противоречие между "Default Setup" и "Advanced Setup"

**Где:** Секция "4. Code Scanning (CodeQL)", строки 179-186

**Цитата:**
> **Вариант 1: Default Setup** (рекомендуется для начала)
> Settings → Code security and analysis → Code scanning → Enable → Default setup.
>
> **Вариант 2: Advanced Setup** (кастомный workflow)

**Риск:** LLM может создать ОБА варианта (включить Default Setup через UI И создать файл codeql.yml), что приведёт к:
- Дублированию сканирования (двойной расход CI-минут)
- Конфликту между автоматической и кастомной конфигурацией
- Непонятному поведению (какой workflow запускается?)

Нет явного указания "ЛИБО default, ЛИБО advanced — НЕЛЬЗЯ использовать оба".

**Рекомендация:**
Добавить предупреждение перед описанием вариантов:

> **ВАЖНО:** Default Setup и Advanced Setup — взаимоисключающие варианты. Выбрать ОДИН.
>
> - **Default Setup:** GitHub автоматически создаёт и управляет workflow. НЕ создавать `.github/workflows/codeql.yml` вручную.
> - **Advanced Setup:** Создать `.github/workflows/codeql.yml` вручную. НЕ включать Default Setup через Settings.
>
> **Критерий выбора:**
> - Default Setup → для стандартных проектов (один язык, без кастомизации)
> - Advanced Setup → для сложных проектов (несколько языков, кастомные queries, матричная стратегия)

**Критичность:** Высокая (P1) — риск дублирования сканирования

### Проблема 2.2: Неявная связь между Dependabot Version Updates и файлом dependabot.yml

**Где:** Секция "3. Dependabot", строки 97-101

**Цитата:**

> Автоматические PR для поддержания актуальных версий зависимостей.

**Риск:** LLM может подумать, что:
- Файл dependabot.yml нужен ТОЛЬКО для Version Updates
- Dependabot Alerts и Security Updates работают БЕЗ файла dependabot.yml

Это ЧАСТИЧНО верно (Alerts работают без файла), но LLM может не создавать dependabot.yml вообще.

**Рекомендация:**
Добавить в начало секции "3. Dependabot":

> **Структура Dependabot:**
>
> | Функция | Требуется ли dependabot.yml | Как включить |
> |---------|----------------------------|--------------|
> | **Dependabot Alerts** | Нет (работает без файла) | Settings → Dependabot alerts → Enable |
> | **Dependabot Security Updates** | Нет (работает без файла) | Settings → Dependabot security updates → Enable |
> | **Dependabot Version Updates** | **Да (файл обязателен)** | Создать `.github/dependabot.yml` |
>
> **Рекомендация:** Создавать dependabot.yml ВСЕГДА (даже если используются только Alerts), чтобы настроить расписание и префиксы коммитов.

**Критичность:** Средняя (P2)

### 4.3. Не хватает: Обработка ошибок при включении инструментов

**Почему важно:** LLM может столкнуться с ошибками:
- Dependabot не может распарсить manifest (некорректный формат package.json, requirements.txt)
- CodeQL не поддерживает используемый язык или версию
- Secret Scanning недоступен (private repo без GHAS)

Нет секции "Типичные ошибки" (как в validation-инструкциях).

**Что даст:** Возможность диагностировать проблемы и исправить их.

**Стоит ли добавлять:** Да (P2)

**Пример добавления:**
> ## 10. Типичные ошибки
>
> | Ошибка | Причина | Решение |
> |--------|---------|---------|
> | `Dependabot cannot parse manifest` | Некорректный синтаксис package.json, requirements.txt | Проверить файл через JSON/YAML linter |
> | `CodeQL: language not supported` | Язык не входит в список поддерживаемых | Пропустить Code Scanning для этого языка |
> | `Secret Scanning: 403 Forbidden` | Private repo без GHAS лицензии | Связаться с администратором организации или использовать альтернативу (pre-commit hook) |
> | `Push Protection blocked commit` | Коммит содержит обнаруженный секрет | Удалить секрет из файлов, использовать GitHub Secrets |

### Зона 5.1: Lifecycle секретов после ротации

**Суть:** Документ описывает "ротировать секрет" (строка 276), но НЕ описывает полный цикл:
- Где хранить новый секрет (GitHub Secrets? Environment variables? Vault?)
- Как обновить все references на секрет в коде и конфигурации
- Как уведомить команду о ротации (чтобы никто не использовал старый локально)
- Как проверить, что старый секрет нигде не используется

**Почему важно:** Ротация секрета — это НЕ только "отозвать старый и выпустить новый". Без описания полного процесса:
- LLM может ротировать секрет, но НЕ обновить его в GitHub Secrets → deploy сломается
- Разработчики продолжат использовать старый секрет локально → утечка продолжится

**Расширение качества:** Значительное (ротация — критичная операция безопасности)

**Рекомендация:** В стандарте допустимо: описать ЧТО делать (оценить → ротировать → отозвать → очистить → закрыть). Детальный процесс с командами → SSOT: `modify-security.md`.

⚠️ **Зона ответственности:** Пошаговый процесс ротации (5 шагов, CLI-команды, уведомление команды) — зона `modify-security.md`. Стандарт описывает ПРАВИЛА (что считается секретом, когда ротировать), modify описывает КАК ротировать.

### Зона 5.2: Интеграция с CI/CD pipeline

**Суть:** Документ описывает инструменты безопасности изолированно, НЕ описывает:
- Как блокировать merge PR при наличии Dependabot alerts или CodeQL findings
- Как интегрировать security checks в required status checks
- Как автоматизировать approve и merge Dependabot PR (для minor/patch updates)

**Почему важно:** Инструменты безопасности должны быть частью process enforcement, а не просто "nice to have". Без интеграции с branch protection:
- Разработчики могут игнорировать алерты
- Vulnerable код попадёт в production
- Security сканирование становится бессмысленным (результаты не влияют на процесс)

**Расширение качества:** Значительное (без интеграции security инструменты малоэффективны)

**Рекомендация:** В стандарте security допустимо: summary "рекомендуется добавить CodeQL и Dependency Review в required checks" + SSOT-ссылки. Детали — в зонах-владельцах.

⚠️ **Зона ответственности:**
- **Branch Protection Rules** (required checks, блокировка merge) — зона [standard-review.md](/.github/.instructions/review/standard-review.md)
- **Auto-merge workflow YAML** — зона [standard-action.md](/.github/.instructions/actions/standard-action.md)
- **Dependabot auto-merge процесс** — зона `create-security.md` (при настройке) или `modify-security.md` (при изменении)

### Зона 5.3: Политика обработки алертов

**Суть:** Документ описывает КАК закрыть алерт (CLI команды, строки 340-372), НЕ описывает КОГДА можно закрыть как:
- `tolerable_risk` — кто принимает решение? какие критерии?
- `false positive` — как определить, что это действительно false positive?
- `won't fix` — допустимые сценарии (legacy код, несовместимость)

**Почему важно:** Без policy LLM (или разработчик) может:
- Закрывать все алерты как "false positive" без проверки
- Игнорировать критичные уязвимости как "tolerable risk"
- Не документировать решение (почему алерт закрыт)

**Расширение качества:** Умеренное (влияет на security posture)

**Рекомендация:** Добавить опционально (P2)

**Пример добавления:**
> ## 13. Политика обработки Security Alerts
>
> ### Критерии закрытия алертов
>
> | Причина закрытия | Когда использовать | Кто принимает решение | Требуется ли документация |
> |------------------|-------------------|----------------------|--------------------------|
> | `revoked` | Секрет отозван, уязвимость исправлена | Любой разработчик | Да (комментарий в алерте) |
> | `false positive` | Алерт некорректен (не является уязвимостью/секретом) | Любой разработчик | Да (объяснение, почему false positive) |
> | `used in tests` | Секрет используется только в тестах (не production) | Любой разработчик | Да (указать файл теста) |
> | `tolerable_risk` | Уязвимость известна, но риск принят | Tech Lead / Security Team | Да (risk assessment документ) |
> | `won't fix` | Уязвимость в legacy коде, исправление невозможно | Tech Lead / Security Team | Да (причина + compensating controls) |
>
> ### Процесс для `tolerable_risk` и `won't fix`
>
> 1. Создать Issue: "Security Alert #{number}: Risk Assessment"
> 2. Описать:
>    - Почему уязвимость не может быть исправлена
>    - Какие compensating controls применяются (WAF, rate limiting)
>    - Срок пересмотра решения (например, ежеквартально)
> 3. Получить approve от Tech Lead / Security Team
> 4. Закрыть алерт с причиной и ссылкой на Issue

### Зона 5.4: Масштабирование на монорепо

**Суть:** Пример dependabot.yml (строки 103-137) показывает конфигурацию для простого проекта. Не описано:
- Как настроить для монорепо (10+ сервисов с разными package ecosystems)
- Как группировать PR по директориям (чтобы не создавать 50+ PR в неделю)
- Как настроить разных reviewers для разных областей кода
- Как использовать `groups` для объединения обновлений

**Почему важно:** В монорепо с 10 микросервисами Dependabot без группировки создаст:
- 5-10 PR на каждый сервис в неделю (50-100 PR total)
- Noise в Pull Requests (невозможно review)
- Конфликты между PR (merge одного сломает другой)

**Расширение качества:** Умеренное (критично для сложных проектов, но не для базового использования)

**Рекомендация:** Добавить опционально (P3)

**Пример добавления:**
> ## 14. Dependabot для монорепо
>
> ### Группировка обновлений
>
> ```yaml
> version: 2
> updates:
>   # Frontend
>   - package-ecosystem: "npm"
>     directory: "/frontend"
>     schedule:
>       interval: "weekly"
>     groups:
>       frontend-minor-patch:
>         patterns:
>           - "*"
>         update-types:
>           - "minor"
>           - "patch"
>     labels:
>       - "frontend"
>       - "dependencies"
>
>   # Backend service 1
>   - package-ecosystem: "pip"
>     directory: "/services/auth"
>     schedule:
>       interval: "weekly"
>     groups:
>       auth-dependencies:
>         patterns:
>           - "*"
>     reviewers:
>       - "auth-team"
>     labels:
>       - "auth-service"
>       - "dependencies"
> ```
>
> **Преимущества:**
> - Один PR для всех minor/patch обновлений (вместо 10+)
> - Разные reviewers для разных сервисов
> - Разные метки для фильтрации

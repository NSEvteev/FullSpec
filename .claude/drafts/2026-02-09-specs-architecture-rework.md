### 2. Двойственные формулировки

#### 2.1. "Финальный статус" — скрытое противоречие

**Где:** Раздел 8, каскадное завершение (DONE)

**Формулировка:**
> Родитель → DONE когда ВСЕ дети в финальном статусе И хотя бы ОДИН ребёнок DONE.

**Проблема:** "Финальный статус" = DONE или REJECTED. Подразумевается микс (2 DONE + 1 REJECTED → родитель DONE). Но каскад REJECTED гарантирует, что при REJECTED одного ребёнка ВСЕ сайблинги тоже REJECTED. Микс невозможен.

**Риск для LLM:** Может решить, что микс DONE/REJECTED допустим.

**Решение:** Упрощено до "все дети DONE → родитель DONE" + пояснение почему микс невозможен. Все "финальный статус" в диаграмме DONE заменены на явное "DONE".

**Статус:** ✅ Решено

---

#### 2.2. "Рабочие правки" — нет определения границ

**Где:** Раздел 8, обратная связь Code → Specs

**Формулировка:**
> Если затронуты только Plan / Test Spec — рабочие правки, статус не меняется, пользователь информируется.

**Проблема:** Что такое "рабочие правки"? LLM может интерпретировать как:
- (a) мелкие косметические правки
- (b) полная переписка Plan/Test Spec без согласования

Не сказано, нужно ли одобрение пользователя.

**Решение:** Вариант (a) — рабочие правки = LLM автономно обновляет Plan/Test Spec, продолжает работу, информирует пользователя после. Подтверждение не требуется.

**Статус:** ✅ Решено

---

#### 2.3. CONFLICT-каскад на сайблингов — не описан

**Где:** Раздел 8, обратная связь Code → Specs

**Формулировка:**
> все затронутые документы + все их дочерние → CONFLICT

**Проблема:** Если ADR-A → CONFLICT, что с ADR-B и ADR-C (сайблингами)? Документ молчит. Текущий механизм:
- Если Design затронут → все его дети (B, C) → CONFLICT (через каскад детей)
- Если Design НЕ затронут → только A → CONFLICT, B и C остаются RUNNING

Но это нигде не объяснено явно. LLM может по аналогии с REJECTED каскадом поставить CONFLICT и сайблингам.

**Решение:** Вариант (a) — CONFLICT каскадирует только вниз, НЕ на сайблингов. Сайблинги затрагиваются через родителя. Добавлены 3 конкретных сценария (ADR, Design, Impact) с пояснениями.

**Статус:** ✅ Решено

---

#### 2.4. "Пользователь ревьюит → WAITING" — два прочтения

**Где:** Раздел 3, диаграмма полного воркфлоу

**Формулировка:**
> 3. Пользователь ревьюит → WAITING

**Проблема:** LLM может понять как:
- (a) Ревью — часть перехода в WAITING (правильно, раскрыто в разделе 8)
- (b) Ревью — отдельный шаг, после которого отдельно ставится WAITING

**Решение:** Заменено на "Пользователь одобряет → WAITING" во всех диаграммах.

**Статус:** ✅ Решено

---

#### 2.5. Planned Changes — неполный жизненный цикл

**Где:** Раздел 3 (параллельные дискуссии) + Раздел 8 (каскад REJECTED)

**Проблема:** Planned Changes добавляются при Design → WAITING, удаляются при ADR → DONE. Но:
- Что если Design → REJECTED до создания ADR? Planned Changes уже в architecture/ — кто их удаляет?
- Строка 931 ("Planned Changes удаляются при REJECTED") — в контексте каскада REJECTED, но не явно для самого Design

**Решение:** Уточнено: "удаляется при любом переходе связанного документа в REJECTED — включая сам Design".

**Статус:** ✅ Решено

---

#### 2.6. "Содержание затронуто" — субъективный критерий

**Где:** Раздел 8, обратная связь Code → Specs, фаза 1

**Формулировка:**
> Проверка продолжается вверх до уровня, где содержание не затронуто → СТОП

**Проблема:** Кто определяет "затронуто"? LLM принимает решение самостоятельно. Критерий субъективен — разные LLM могут остановиться на разных уровнях.

**Решение:** Формальный критерий + примеры: "затронут = утверждение стало неверным (контракт, компонент, алгоритм). Не затронут = утверждения верны (рефакторинг, оптимизация)".

**Статус:** ✅ Решено

### 4. Что ещё нужно описать

#### 4.1. Восстановление после сбоя

**Решение:** Неактуально. Документ в DRAFT, контент частично написан. Следующая сессия LLM читает файл и продолжает штатный цикл. Специальный механизм не нужен.

**Статус:** ✅ Снят

---

#### 4.2. Параллельные цепочки — кросс-цепочечная обратная связь

**Решение:** При обновлении architecture/ (DONE или CONFLICT-разрешение) проверить все другие цепочки, ссылающиеся на изменённые файлы. Реакция зависит от статуса: DRAFT → перегенерация, WAITING → DRAFT, RUNNING → CONFLICT, CONFLICT → учесть, DONE → новая Discussion, REJECTED → игнорировать. Добавлена секция "Кросс-цепочечная обратная связь" в раздел 3.

**Статус:** ✅ Решено

---

#### 4.3. Архивация REJECTED документов

**Решение:** Еженедельная очистка (понедельник) — LLM предлагает пользователю список REJECTED-документов для удаления.

**Статус:** ✅ Решено

---

#### 4.4. Точка входа — как LLM знает, что нужен SDD

**Решение:** Rule в `.claude/rules/` — при любом вводе функциональности в проект начинать с Discussion. Детали при создании оркестратора.

**Статус:** ✅ Решено

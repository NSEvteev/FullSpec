# Процесс обратной связи для Captain Holt

## Проблема

Агент анализирует документы, но нет способа узнать:
- Применил ли пользователь рекомендации?
- Были ли рекомендации полезны или избыточны?
- Повторяются ли одни и те же проблемы в разных документах?

Без обратной связи агент не может улучшать качество анализа.

## Предлагаемое решение

### 1. Структура хранения

```
/.claude/state/holt/
├── feedback/                    # Архив фидбэков
│   ├── 2026-02-01-standard-instruction.json
│   └── 2026-02-01-standard-agent.json
├── analytics/                   # Агрегированная статистика
│   └── common-issues.json
└── README.md
```

### 2. Формат файла фидбэка

```json
{
  "analysis_file": "/.claude/drafts/2026-02-01-holt-analysis-standard-instruction.md",
  "analyzed_document": "/.instructions/standard-instruction.md",
  "date": "2026-02-01",
  "version": "v1.0",
  "recommendations_total": 10,
  "recommendations_applied": 8,
  "recommendations_rejected": 2,
  "rejected_details": [
    {
      "id": "3.1",
      "reason": "Over-engineering — скилл не нужен для этого случая"
    },
    {
      "id": "5.2",
      "reason": "Уже исправлено в другом PR"
    }
  ],
  "user_comment": "Анализ полезен, но слишком детальный"
}
```

### 3. Процесс сбора фидбэка

**Когда собирать:** После применения рекомендаций из анализа.

**Кто собирает:** Основной LLM (не агент).

**Шаги:**
1. Пользователь применяет рекомендации из анализа
2. Основной LLM спрашивает: "Какие рекомендации применили? Какие отклонили?"
3. LLM записывает фидбэк в `/.claude/state/holt/feedback/`
4. Раз в месяц — агрегация в `common-issues.json`

### 4. Формат агрегированной статистики

```json
{
  "period": "2026-02",
  "total_analyses": 15,
  "common_issues": {
    "Неопределённые местоимения": {
      "found": 23,
      "applied": 21,
      "rejected": 2
    },
    "Отсутствие примеров": {
      "found": 18,
      "applied": 15,
      "rejected": 3
    }
  },
  "most_rejected_reasons": [
    "Over-engineering",
    "Уже исправлено в другом месте",
    "Не применимо к контексту"
  ]
}
```

### 5. Использование статистики

**Автоматические выводы:**
- Если категория проблем отклоняется >50% — пересмотреть критерии
- Если одна проблема повторяется >10 раз — предложить обновить стандарт
- Если пользователь пишет "over-engineering" — снизить детальность

**Обновление агента:**
1. Раз в месяц LLM анализирует `common-issues.json`
2. Предлагает изменения в AGENT.md
3. Документирует в CHANGELOG.md

## Открытые вопросы

1. **Автоматизация сбора?** Можно ли автоматически определять, какие рекомендации применены (diff анализируемого документа)?

2. **Хранение истории?** Нужен ли ротатор для старых фидбэков (>3 месяцев)?

3. **Интеграция с CI?** Собирать фидбэк автоматически при мерже PR?

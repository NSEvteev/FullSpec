## 1. Двойственность формулировок

### Проблема 1.1: Неопределённое условие "при необходимости"

**Где:** Секция "Когда создавать rule" (строка 60)

**Цитата:**
> **Принцип:** Rule = триггер для автоматической загрузки контекста, не дублирует инструкции.

**Проблема:** Формулировка "не дублирует инструкции" неоднозначна — что именно запрещено дублировать?

**Как может понять LLM:**
- Интерпретация A: Нельзя копировать содержимое существующих инструкций (ссылаться на них)
- Интерпретация B: Нельзя создавать rule с тем же функционалом, что у существующей инструкции
- Интерпретация C: Нельзя писать полные инструкции внутри rule (только краткие триггеры)

**Рекомендация:**
> **Принцип:** Rule = триггер для автоматической загрузки контекста. Rule содержит только краткие ссылки на инструкции или скиллы, не копирует их содержимое.

**Критичность:** P2 — Средняя

---

### Проблема 1.2: Размытый критерий "краткая ссылка"

**Где:** Секция "Когда создавать rule" (строка 66, таблица)

**Цитата:**
> | Краткая ссылка | Нужна краткая ссылка на скилл или инструкцию |

**Проблема:** Термин "краткая ссылка" не определён — чем она отличается от обычной ссылки?

**Как может понять LLM:**
- Интерпретация A: Ссылка без дополнительного описания (просто путь)
- Интерпретация B: Ссылка с кратким описанием (1-2 строки)
- Интерпретация C: Любая ссылка, которая загружается автоматически (без явного чтения)

**Рекомендация:**
> | Автоматический контекст | Нужно автоматически загружать ссылки на скиллы или инструкции при работе с файлами |

**Критичность:** P2 — Средняя

---

### Проблема 1.3: Неопределённое понятие "разовое правило"

**Где:** Секция "Когда НЕ создавать rule" (строка 78, таблица)

**Цитата:**
> | Разовое правило | Добавить в существующий rule или инструкцию |

**Проблема:** Что такое "разовое правило"? Критерий разовости не определён.

**Как может понять LLM:**
- Интерпретация A: Правило, которое применяется только к одному файлу
- Интерпретация B: Правило, которое будет использовано один раз и забыто
- Интерпретация C: Правило, применимое только к одной узкой ситуации (не универсальное)

**Рекомендация:**
> | Узкоспециализированное правило | Если правило применяется только к одной операции или файлу — добавить в существующий rule |

**Критичность:** P1 — Высокая (влияет на решение создавать ли rule)

### Проблема 1.4: Неопределённый критерий "плоская структура"

**Где:** Секция "2. Расположение" (строка 88)

**Цитата:**
> - Плоская структура — подпапки НЕ поддерживаются

**Проблема:** Что означает "не поддерживаются"? Технически невозможно или запрещено стандартом?

**Как может понять LLM:**
- Интерпретация A: Claude Code технически не загружает rules из подпапок
- Интерпретация B: Стандарт запрещает подпапки (но технически они работают)
- Интерпретация C: Можно создавать подпапки, но они будут игнорироваться

**Рекомендация:**
> - Плоская структура — все rules размещаются в корне `/.claude/rules/`. Claude Code не загружает rules из подпапок.

**Критичность:** P1 — Высокая (влияет на создание структуры)

## 2. Риск неправильной интерпретации LLM

### Проблема 2.1: Неявная зависимость между "содержимым" и "максимальным размером"

**Где:** Секция "6. Содержимое rule" (строки 219-221)

**Цитата:**
> **Максимальный размер:** 50 строк
>
> **Причина:** Rules загружаются в контекст каждой сессии — чем короче, тем лучше.

**Проблема:** LLM может не связать "50 строк" с "содержимым rule" и применить ограничение ко всему файлу (включая frontmatter).

**Рекомендация:**
> **Максимальный размер содержимого:** 50 строк (без учёта frontmatter)
>
> **Причина:** Rules загружаются в контекст каждой сессии — чем короче, тем лучше.

**Критичность:** P2 — Средняя

---

### Проблема 2.2: Противоречие "фича, не баг" и "ошибка проектирования"

**Где:** Секция "7. Конфликты paths" (строки 252-258)

**Цитата:**
> **Это фича, не баг:**
> - Позволяет комбинировать правила из разных областей
> - Файл может одновременно соответствовать нескольким rules
>
> **Проблема:** Если rules противоречат друг другу — ошибка проектирования.

**Проблема:** LLM может запутаться: пересечение paths — это хорошо (фича) или плохо (ошибка проектирования)?

**Как может понять LLM:**
- Интерпретация A: Пересечение paths всегда допустимо
- Интерпретация B: Пересечение paths допустимо только если rules дополняют друг друга
- Интерпретация C: Пересечение paths — ошибка, нужно избегать

**Рекомендация:**
> **Поведение Claude Code:** Применяет **все** rules, чьи `paths` совпали с текущим файлом.
>
> **Допустимо:** Пересечение paths разрешено, если rules дополняют друг друга (разные аспекты одной области).
>
> **Ошибка проектирования:** Rules противоречат друг другу (один требует действие A, другой запрещает A).

**Критичность:** P1 — Высокая (влияет на проектирование rules)

### Проблема 2.3: Неявное условие "если скрипт существует"

**Где:** Примеры в секциях "Шаг 1" create-rule.md и modify-rule.md (ссылки на скрипты)

**Контекст:** В стандарте упоминаются скрипты `list-rules.py` и `validate-rule.py`, но не сказано, что делать, если скрипт отсутствует.

**Критичность:** P1 — Высокая (влияет на выполнимость воркфлоу)

### Проблема 2.4: Неопределённый порядок применения rules при пересечении

**Где:** Секция "7. Конфликты paths" (строка 252)

**Цитата:**
> **Поведение Claude Code:** Применяет **все** rules, чьи `paths` совпали с текущим файлом.

**Проблема:** Не указан порядок применения — это может быть важно, если rules содержат взаимозависимые инструкции.

**Как может понять LLM:**
- Интерпретация A: Rules применяются в алфавитном порядке имён файлов
- Интерпретация B: Rules применяются в случайном порядке (порядок не гарантирован)
- Интерпретация C: Глобальные rules применяются первыми, затем условные

**Рекомендация:**
> **Поведение Claude Code:** Применяет **все** rules, чьи `paths` совпали с текущим файлом. Порядок применения не гарантирован — не полагайтесь на последовательность загрузки rules.

**Критичность:** P2 — Средняя

### Проблема 2.5: Неявная зависимость frontmatter от типа rule

**Где:** Секция "3. Frontmatter" (строка 118)

**Цитата:**
```yaml
paths:          # Опционально — для условных rules
  - "паттерн/**"
```

**Проблема:** LLM может не понять, что для глобальных rules поле `paths` нужно **полностью удалить**, а не оставить пустым.

**Рекомендация:** Явно указать:

```yaml
---
description: Краткое описание rule
standard: .claude/.instructions/rules/standard-rule.md
index: .claude/.instructions/rules/README.md
# paths: Только для условных rules — для глобальных rules это поле удалить
---
```

**Критичность:** P1 — Высокая (влияет на корректность frontmatter)

## 4. Полнота описания объекта

### 4.1. Не хватает: Примеры ошибочных rules

**Почему важно:** Разработчики могут создавать некорректные rules, не понимая, что именно нарушает стандарт. Примеры антипаттернов (с объяснением почему это неправильно) предотвращают типичные ошибки.

**Стоит ли добавлять:** Да (P2)

**Что даст:** Снижение количества ошибок при создании rules, быстрее обучение новых разработчиков.

**Пример добавления:**
```markdown
### Антипаттерны

#### ❌ Rule дублирует инструкцию

**Плохо:**
```yaml
---
description: Создание папки
---

## Шаги

1. Проверить существование
2. Создать директорию
3. Создать README.md
```

**Почему плохо:** Rule дублирует содержимое инструкции. Это увеличивает контекст сессии и создаёт два источника правды.

**Правильно:**
```markdown
При создании папки → `/structure-create`
```
```

---

### 4.2. Не хватает: Связь между rule и скиллом

**Почему важно:** В документе не явно сказано, что rule часто создаётся ПОСЛЕ создания скилла (как триггер для автоматической загрузки контекста скилла).

**Стоит ли добавлять:** Да (P2)

**Что даст:** Понимание life cycle: создали скилл → создали rule для автозагрузки контекста.

**Пример добавления:**
```markdown
## Life cycle rule

1. **Создали скилл** (например, `/api-validate`)
2. **Создали rule** для автоматической загрузки контекста скилла при работе с файлами области
3. **Обновили rule** при добавлении новых скиллов (через `/rule-modify`)
```

---

### 4.3. Не хватает: Миграция rule при изменении структуры проекта

**Почему важно:** Если меняется структура проекта (переименование папок, перемещение файлов), paths в rules могут устареть.

**Стоит ли добавлять:** Опционально (P3)

**Что даст:** Понимание процесса обновления rules при рефакторинге структуры проекта.

**Пример добавления:**
```markdown
## Миграция при рефакторинге структуры

**Сценарий:** Папка `src/api/` переименована в `src/services/api/`

**Действия:**
1. Найти rules с paths, содержащими старый путь:
   ```bash
   grep -r "src/api/" .claude/rules/
   ```
2. Обновить paths в найденных rules
3. Валидация → `/rule-validate`
```

## 5. Слепые зоны автора

### Зона 5.1: Версионирование rules

**Суть:** В документе нет упоминания версионирования rules. Если стандарт rule изменится (например, добавится обязательное поле в frontmatter), как определить, какие rules нужно мигрировать?

**Почему важно:** Без версионирования невозможно отследить, какие rules созданы по старому стандарту и требуют обновления.

**Расширение качества:** Значительное

**Рекомендация:** Добавить сейчас (P2)

**Пример добавления:**
```markdown
## Версионирование rules

### standard-version в frontmatter

```yaml
---
description: Критичные глобальные правила
standard: .claude/.instructions/rules/standard-rule.md
standard-version: v1.0
index: .claude/.instructions/rules/README.md
---
```

**Назначение:** Отслеживание актуальности rules при обновлении стандарта.

**Миграция rules:** При обновлении стандарта (например, до v1.1) — выполнить:
```bash
python .claude/.instructions/rules/.scripts/migrate-rules-to-v1.1.py
```
```

### Зона 5.2: Тестирование rules в изолированной сессии

**Суть:** В документе упоминается, что rules применяются только после рестарта сессии, но нет рекомендаций по тестированию нового rule перед коммитом в основную ветку.

**Почему важно:** Разработчик может закоммитить некорректный rule, который сломает работу всей команды (все будут загружать неправильный контекст).

**Расширение качества:** Умеренное

**Рекомендация:** Добавить сейчас (P3)

**Пример добавления:**
```markdown
## Тестирование rule перед коммитом

### Шаг 1: Создать тестовую ветку
```bash
git checkout -b test/rule-api
```

### Шаг 2: Создать rule
```bash
/rule-create
```

### Шаг 3: Рестарт сессии
1. Завершить текущую сессию
2. Начать новую сессию

### Шаг 4: Проверить загрузку
```
/memory
# Проверить: rule api.md в списке загруженных
```

### Шаг 5: Проверить триггеры
1. Открыть файл, соответствующий paths (например, `src/api/users.ts`)
2. Попросить выполнить действие из rule
3. Убедиться, что Claude следует правилам rule

### Шаг 6: Если тесты пройдены — merge в main
```bash
git checkout main
git merge test/rule-api
```
```

### Зона 5.3: Метрики использования rules

**Суть:** В документе нет упоминания о том, как отслеживать, насколько часто rule загружается или используется. Это важно для оптимизации — неиспользуемые rules можно деактивировать.

**Почему важно:** Rules загружаются в контекст каждой сессии — чем меньше rules, тем меньше нагрузка на контекст. Если rule создан, но никогда не триггерится — он бесполезен.

**Расширение качества:** Минимальное

**Рекомендация:** Не нужно (over-engineering) — на данном этапе проекта метрики избыточны.

### Зона 5.4: Интеграция с CI/CD

**Суть:** В документе нет упоминания о проверке rules в CI/CD пайплайне (например, запуск `validate-rule.py` для всех rules перед merge).

**Почему важно:** Автоматическая валидация rules перед коммитом предотвращает попадание невалидных rules в main-ветку.

**Расширение качества:** Умеренное

**Рекомендация:** Добавить сейчас (P3)

**Пример добавления:**
```markdown
## CI/CD проверка

### Pre-commit hook

Добавить в `.git/hooks/pre-commit`:

```bash
#!/bin/bash
# Валидация всех rules перед коммитом

for rule in .claude/rules/*.md; do
  if [[ $rule != *"_old-"* ]]; then
    name=$(basename "$rule" .md)
    python .claude/.instructions/rules/.scripts/validate-rule.py "$name" || exit 1
  fi
done

echo "✅ Все rules валидны"
```

### GitHub Actions

Добавить в `.github/workflows/validate-rules.yml`:

```yaml
name: Validate Rules
on: [push, pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: |
          for rule in .claude/rules/*.md; do
            name=$(basename "$rule" .md)
            python .claude/.instructions/rules/.scripts/validate-rule.py "$name"
          done
```
```

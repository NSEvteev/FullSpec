## 2. Риск неправильной интерпретации

### Проблема 2.1: Противоречие в работе со state

**Где:** Секции "Работа со state" (строка 104-112) vs standard-state.md

**Сейчас в AGENT.md:**
> При каждой записи файла:
> 1. Добавить запись в `operations[]` с типом `write` и путём к файлу
> 2. После завершения работы — обновить `finished_at`

**Противоречие:**
В `standard-state.md` (строка 181-191, 254-273) чётко описан алгоритм:
1. Проверить `locks.json` ПЕРЕД записью
2. Добавить блокировку
3. Выполнить операцию
4. Снять блокировку СРАЗУ (не в конце работы!)

В AGENT.md этот алгоритм полностью опущен.

**Риск:** Агент создаст race condition — запишет в файл, не проверив блокировку.

**Рекомендация:**
```markdown
## Работа со state

> **SSOT:** [standard-state.md](/.claude/.instructions/state/standard-state.md)

**Лог операций:** `/.claude/state/agent-captain-holt-operation.json`

**Цикл работы с файлами:**
1. **ПЕРЕД записью** в `/.claude/drafts/` — проверить `locks.json` (блокирован ли файл?)
2. Добавить свою блокировку в `locks.json`
3. Записать файл
4. Добавить запись в `operations[]` с типом `write` и путём
5. Снять блокировку из `locks.json` (СРАЗУ, не ждать завершения работы!)

**После завершения анализа:**
6. Обновить `finished_at` в `agent-captain-holt-operation.json`
```

**Критичность:** Высокая

### Проблема 2.2: Неявная граница анализа

**Где:** Секция "Формат вывода" (строка 141-233)

**Сейчас:**
Формат вывода описан детально, но нет условия остановки.

**Риск интерпретации:**
- LLM может анализировать документ частично (найти 2-3 проблемы и остановиться)
- LLM может анализировать бесконечно (углубляться в каждую формулировку)
- Нет критерия "достаточно" vs "недостаточно"

**Рекомендация:**
Добавить в секцию "Формат вывода" перед шаблоном:

> **Критерий полноты анализа:**
> - Документ прочитан ПОЛНОСТЬЮ (все секции)
> - Каждая секция проверена по всем 5 критериям
> - Если проблем не найдено — указать "Проблем не найдено" в категории
> - Если категория пуста — не пропускать, явно написать "Не найдено"

**Критичность:** Высокая

### Проблема 2.3: Неопределённость приоритизации

**Где:** Секция "Итоговые рекомендации" (строка 227-232)

**Цитата:**
> | # | Действие | Приоритет | Эффект |
> | 1 | {действие} | {P1/P2/P3} | {что даст} |

**Риск:**
LLM не знает:
- Сколько рекомендаций включать (все проблемы или топ-N?)
- Как определить P1/P2/P3 (нет легенды)
- Как сортировать (по приоритету? по категории?)

**Рекомендация:**
Добавить перед таблицей:

> **Легенда приоритетов:**
> - P1 — Критично (риск неправильного выполнения, блокирующая проблема)
> - P2 — Важно (снижает качество, но не блокирует)
> - P3 — Желательно (улучшение, но можно отложить)
>
> **Правила:** Включить ВСЕ найденные проблемы. Сортировать по приоритету (P1 → P2 → P3).

**Критичность:** Средняя

> 2. Документы, упомянутые в тексте (даже без SSOT-метки)

### Проблема 2.5: Неявное поведение при обнаружении конфликта

**Где:** Отсутствует секция "Конфликты и ошибки"

**Риск:**
Что должен делать агент, если:
- Файл для анализа не существует?
- Файл для анализа повреждён (невалидный markdown)?
- Файл drafts уже существует (перезаписать? спросить?)?
- Блокировка на файл drafts не снята (другой агент упал)?

**Рекомендация:**
Добавить секцию после "Работа со state":

```markdown
## Обработка ошибок

**При ошибках:**

| Ситуация | Действие |
|----------|----------|
| Анализируемый файл не существует | Вернуть ошибку: "Файл {путь} не найден" |

## 5. Слепые зоны автора

### Зона 5.1: Отсутствие механизма обратной связи

**Суть:** Агент анализирует документы, но нет способа узнать:
- Применил ли пользователь рекомендации?
- Были ли рекомендации полезны или избыточны?
- Повторяются ли одни и те же проблемы в разных документах?

**Почему важно:** Без обратной связи агент не может улучшать качество анализа. Если в 10 анализах всегда находится одна и та же проблема — возможно, нужно обновить стандарт, а не повторять рекомендацию.

**Расширение качества:** Значительное (позволит выявить системные проблемы в документации)

**Рекомендация:** Добавить сейчас (базовый механизм)

**Пример добавления:**

```markdown
## Обратная связь

**После применения рекомендаций пользователем:**

Основной LLM может обновить `/.claude/state/holt-feedback.json`:

```json
{
  "analyses": [
    {
      "file": "standard-instruction.md",
      "date": "2026-02-01",
      "recommendations_applied": 8,
      "recommendations_rejected": 2,
      "rejected_reasons": ["Over-engineering", "Уже исправлено в другом PR"]
    }
  ],
  "common_issues": {
    "Неопределённые местоимения": 15,
    "Отсутствие примеров": 12
  }
}
```

**Использование:** Раз в месяц основной LLM анализирует feedback и предлагает обновления стандартов.
```

### Зона 5.2: Нет поддержки инкрементального анализа

**Суть:** Документ может меняться частями (добавляется 1 секция). Агент анализирует весь документ заново, вместо того чтобы анализировать только изменения.

**Почему важно:** Экономия времени и токенов. Особенно критично для больших документов (>2000 строк).

**Расширение качества:** Умеренное (оптимизация, не критично для работы)

**Рекомендация:** Отложить (сначала довести базовый функционал до стабильности)

**Пример добавления (при реализации):**

```markdown
## Инкрементальный анализ

**Если передан параметр `--incremental` и существует предыдущий анализ:**

1. Прочитать предыдущий анализ из `/.claude/state/holt-cache/{file-hash}.json`
2. Сравнить с текущей версией документа (diff)
3. Анализировать только изменённые секции
4. Объединить с предыдущими результатами
5. Сохранить новый кэш

**Формат кэша:**
```json
{
  "file": "standard-instruction.md",
  "analyzed_at": "2026-02-01T10:00:00Z",
  "sections": {
    "Frontmatter": {"last_modified": "2026-01-15", "issues": []},
    "Шаги": {"last_modified": "2026-02-01", "issues": [...]}
  }
}
```
```

### Зона 5.3: Нет интеграции с CI/CD

**Суть:** Анализ запускается вручную. Нет автоматической проверки при PR или коммите.

**Почему важно:** Предотвращает регрессии — новые документы/изменения автоматически проверяются на ясность.

**Расширение качества:** Значительное (превращает агента из инструмента в систему контроля качества)

**Рекомендация:** Добавить сейчас (базовая интеграция через GitHub Actions)

**Пример добавления:**

```markdown
## Интеграция с CI/CD

**GitHub Actions workflow:**

```yaml
name: Documentation Clarity Check
on: [pull_request]
jobs:
  analyze:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Run captain-holt
        run: |
          # Найти изменённые .md файлы
          git diff --name-only origin/main...HEAD | grep '\.md$' > changed.txt
          # Анализировать каждый
          while read file; do
            claude --agent captain-holt "Проанализируй $file"
          done < changed.txt
      - name: Upload results
        uses: actions/upload-artifact@v2
        with:
          name: holt-analysis
          path: .claude/drafts/
```

**Блокировка PR:** Если найдены проблемы P1 (критичные) — PR блокируется до исправления.
```

### Зона 5.4: Нет поддержки мультиязычности

**Суть:** Агент работает только с русскоязычными документами. Если в проекте есть английская документация — анализ будет неточным.

**Почему важно:** Многие проекты имеют документацию на нескольких языках. Формулировки "неопределённое местоимение" могут иметь разный смысл в русском и английском.

**Расширение качества:** Минимальное (специфично для мультиязычных проектов)

**Рекомендация:** Не нужно (over-engineering для текущей стадии проекта)

**Пример добавления (если потребуется):**

```markdown
## Мультиязычность

**При анализе:**

1. Определить язык документа (по frontmatter `lang: en` или автоопределение)
2. Использовать языкоспецифичные правила:
   - EN: Проверять пассивный залог ("is being processed" → "processes")
   - RU: Проверять неопределённые местоимения ("это", "они")
   - Общие: Двусмысленность, отсутствие примеров

**Формат вывода:** На языке анализируемого документа.
```

### Зона 5.5: Нет версионирования стандартов анализа

**Суть:** Агент использует критерии анализа из промпта, но эти критерии могут меняться. Нет способа понять, по какой версии критериев был выполнен анализ.

**Почему важно:** Если критерии анализа обновляются (например, добавляется новая категория "6. Доступность"), старые анализы становятся неполными. Нет способа переанализировать по новым критериям.

**Расширение качества:** Значительное (обеспечивает согласованность анализа во времени)

**Рекомендация:** Добавить сейчас (простое решение)

**Пример добавления:**

```markdown
## Версионирование

**В начале каждого анализа добавить:**

```markdown
**Версия стандарта анализа:** v1.0 (2026-02-01)
**Критерии:** 5 категорий (Двойственность, Риск интерпретации, Ручные операции, Полнота, Слепые зоны)
```

**При обновлении критериев:**
1. Обновить версию в AGENT.md (например, v1.1)
2. Документировать изменения в `/.claude/agents/captain-holt/CHANGELOG.md`
3. Опционально: переанализировать критичные документы по новой версии

**Формат CHANGELOG.md:**
```markdown
## v1.1 (2026-03-01)
- Добавлена категория "6. Доступность"
- Удалены неактуальные примеры из категории "Полнота"
```
```

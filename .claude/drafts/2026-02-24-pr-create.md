### Проблема

При создании PR пользователь вручную:
1. Собирает номера Issues из milestone'а или plan-dev.md
2. Формирует body с `Closes #N` для каждого Issue
3. Определяет type label из характера изменений
4. Вспоминает формат title
5. Вызывает `gh pr create` с множеством флагов

### Скрипт: collect-pr-issues.py

**Назначение:** Быстрый сбор данных для PR из analysis chain.

**Расположение:** `.github/.instructions/.scripts/collect-pr-issues.py`

**Входные данные:**
- `{NNNN}` — номер chain
- Путь к `specs/analysis/NNNN-{topic}/plan-dev.md`

**Что делает:**
1. Парсит таблицу "Маппинг GitHub Issues" из plan-dev.md
2. Извлекает номера Issues (#N)
3. Парсит frontmatter discussion.md для milestone и description
4. Проверяет статус Issues через `gh issue view --json state`

**Выходные данные (JSON):**

```json
{
  "chain": "0001",
  "topic": "oauth2-authorization",
  "milestone": "v1.2.0",
  "branch": "0001-oauth2-authorization",
  "description": "OAuth2 авторизация для микросервисов",
  "issues": [
    {"number": 42, "title": "Добавить JWT токены", "state": "open", "task": "TASK-1"},
    {"number": 43, "title": "Middleware авторизации", "state": "open", "task": "TASK-2"},
    {"number": 44, "title": "Refresh токены", "state": "open", "task": "TASK-3"}
  ],
  "suggested_type": "feature",
  "suggested_title": "feature: add OAuth2 authorization"
}
```

**Преимущество скрипта:** LLM не тратит контекст на парсинг — скрипт возвращает готовые данные.

### Нужен ли агент?

**Нет.** Процесс полностью формульный:
1. Скрипт собирает данные (быстро)
2. LLM формирует body из шаблона (простая подстановка)
3. `gh pr create` выполняется

Агент нужен когда есть нетривиальные решения, исследование или многораундовый диалог. PR-создание — это конвейер данные → шаблон → команда.

### Пример использования

```
Пользователь: /pr-create

1. Ветка: 0001-oauth2-authorization → chain 0001
2. Prerequisites: chain RUNNING, есть коммиты ✓
3. Скрипт: collect-pr-issues.py 0001 → 3 Issues (#42, #43, #44)
4. Title: "feature: add OAuth2 authorization"
5. Body:
   ## Summary
   OAuth2 авторизация для микросервисов — JWT токены, middleware, refresh.

   ## Changes
   - JWT token generation and validation (TASK-1)
   - Authorization middleware (TASK-2)
   - Refresh token flow (TASK-3)

   ## Test plan
   - [ ] Unit tests passing
   - [ ] Integration tests passing
   - [ ] Manual testing completed

   ## Related Issues
   Closes #42
   Closes #43
   Closes #44

6. Labels: feature, high
7. Push: git push -u origin 0001-oauth2-authorization
8. Preview: [показать пользователю] → "Создать PR?" → Да
9. gh pr create --title "feature: add OAuth2 authorization" --body "..." --label feature --label high
10. Отчёт: PR #15 создан, 3 Issues привязаны
```

## Решения

- **Инструкция → скилл:** create-pull-request.md (SSOT) → /pr-create (обёртка), по архитектуре проекта
- Основной источник Issues: plan-dev.md маппинг (не `gh issue list --milestone`), потому что он точнее — содержит только Issues этого chain'а
- Скрипт `collect-pr-issues.py` — для быстрого сбора без LLM парсинга
- Агент не нужен — процесс формульный
- Preview перед созданием — обязательный (AskUserQuestion), чтобы пользователь мог скорректировать title/body
- `--dry-run` для безопасного предпросмотра

## Что уже описано в проекте

### standard-pull-request.md (полный стандарт PR)

- **Title формат:** `{type}: {краткое описание}`, до 70 символов, нижний регистр, без точки. Опциональный scope: `{type}({scope}): {описание}`. Тип ДОЛЖЕН совпадать с одной из меток TYPE из labels.yml.
- **Допустимые TYPE-метки:** `bug`, `task`, `docs`, `refactor` (в labels.yml НЕТ `feature` — это важно! В драфте примеры используют `feature`, но такой метки не существует).
- **Body:** заполняется по шаблону `PULL_REQUEST_TEMPLATE.md`. Секции: Summary, Changes, Breaking changes (опц.), Test plan (чек-лист), Related issues, Notes (опц.).
- **Обязательные метки при создании PR:** ровно 1 TYPE + ровно 1 PRIORITY.
- **Связь с Issues:** `Closes #N` на отдельных строках в секции `## Related issues`. Issues закрываются автоматически ТОЛЬКО после merge в целевую ветку.
- **Группировка:** одна фича/область = группа Issues = один PR. Запрещено создавать отдельный PR на каждый Issue. Разделять на несколько PR: если Issues из разных фич, если >500 строк изменений.
- **Draft PR:** для early feedback, WIP. Merge запрещён до Ready. Summary обязательно с `[WIP]`/`[Draft]`. Related issues обязательно для трекинга. Перевод: `gh pr ready 123`.
- **Жизненный цикл:** СОЗДАНИЕ → АВТОПРОВЕРКИ (pre-commit, CI) → CODE REVIEW (approve/request changes) → MERGE (squash) → АВТОЗАКРЫТИЕ ISSUES.
- **CLI:** `gh pr create --title "..." --body "..." --label ...`. Также: `--fill` (из коммитов), `--draft`, `--base develop`, `--reviewer @user`.
- **Граничные случаи:** PR без Issue запрещён (исключение: минорные правки). Hotfix: Issue с `critical`, ветка `{NNNN}-hotfix-{desc}`, ускоренное ревью. Длительная разработка: синхронизировать с main минимум 1 раз в 2 дня.
- **В стандарте НЕТ `create-pull-request.md`** — именно этот документ и является целью данного драфта.

### standard-pr-template.md (стандарт шаблона)

- **Расположение:** `.github/PULL_REQUEST_TEMPLATE.md` — единственный шаблон, подставляется автоматически.
- **Обязательные секции:** Summary (2-5 предложений), Changes (bullet list), Test plan (чек-лист), Related issues (Closes #N).
- **Опциональные секции:** Breaking changes, Notes. Если не заполнены — удалить (пустые с placeholder недопустимы в готовом PR).
- **Placeholder-ы:** квадратные скобки `[...]`, императив. Для Related issues — HTML-комментарий `<!-- -->` чтобы GitHub не парсил `#N`.
- **Строка `Generated with Claude Code`** добавляется автоматически, не включать в шаблон.

### PULL_REQUEST_TEMPLATE.md (сам шаблон)

```markdown
## Summary
[Опишите изменения в 2-5 предложениях]

## Changes
- [Изменение 1]
- [Изменение 2]
- [Изменение 3]

## Breaking changes
[Перечислите ломающие изменения, если есть. Если нет — удалите эту секцию.]

## Test plan
- [ ] Локально протестировано (`make dev`)
- [ ] Pre-commit проверки пройдены
- [ ] Unit-тесты пройдены
- [ ] Интеграционные тесты пройдены (если применимо)
- [ ] Документация обновлена (если нужно)

## Related issues
<!-- Укажите связанные Issues: Fixes #номер, Closes #номер, Related to #номер -->

## Notes
[Дополнительная информация для ревьюера. Если нет — удалите эту секцию.]
```

### standard-issue.md (стандарт Issues)

- **Жизненный цикл Issue:** СОЗДАНИЕ → ОТКРЫТ → В РАБОТЕ (ветка создана) → PR СОЗДАН → REVIEW → ЗАКРЫТ (автоматически при merge).
- **Закрытие Issue:** автоматически через `Closes #N` в PR. Ручное закрытие с `completed` ЗАПРЕЩЕНО. Только `not planned` для дубликатов/неактуальных.
- **Связь с Branch и PR:** Issue → ветка `{NNNN}-{topic}` → PR с `Fixes #N` → merge → Issue закрывается автоматически.
- **Milestone:** Issue ДОЛЖЕН быть добавлен в Milestone. Milestone берётся из frontmatter Discussion.

### standard-github-workflow.md (полный цикл)

- **Стадия 6 (PR):** Процесс: 1) `/review` (локальное ревью до PR), 2) `git push -u origin {branch}`, 3) `gh pr create`. Все Issues группы закрываются одним PR через `Closes #42, #43, #44`.
- **Принцип:** одна фича/область → группа Issues → одна ветка → один PR.

### standard-process.md (шаг 4.2 — PR Create)

- **Фаза 4, шаг 4.2:** PR Create = `git push` + `gh pr create` с привязкой Issues. Скилла нет, инструкции нет — помечен как пробел G2.
- **Покрытие:** в сводной таблице инструментов шаг 4.2 имеет SSOT `standard-pull-request.md` + `standard-pr-template.md`, но **НЕТ скилла и НЕТ скрипта**.
- **Pre-commit хук:** `pr-template-validate` выполняется на шаге 4.2.

### Маппинг TASK-N → Issues (plan-dev.md)

- **Формат таблицы маппинга** в plan-dev.md (создаётся `/dev-create` на шаге 3):
  ```
  | Элемент Plan Dev | GitHub Issue |
  |------------------|-------------|
  | TASK-N | Отдельный Issue через `/issue-create` |
  | Подзадачи N.M | Чек-лист в body Issue |
  | Приоритет | Label приоритета |
  | Зависимости | `**Зависит от:** #N` в body |
  | TC | Ссылка в секции "Связанная документация" |
  | Milestone | Из frontmatter Discussion |
  ```
- **Конкретные номера Issues** создаются `/dev-create` и записываются в план. Скрипт `collect-pr-issues.py` должен парсить эту обновлённую таблицу.

### Discussion frontmatter (источник metadata)

- **Поля:** `description` (до 1024 символов), `milestone` (vX.Y.Z), `status`, `children`.
- Milestone проекта = Milestone из Discussion frontmatter.
- Description проекта = `description` из Discussion frontmatter — кандидат для PR Summary.

### create-development.md (dev-create workflow)

- **Шаг 3:** Создаёт Issues из TASK-N, обновляет таблицу маппинга в plan-dev.md.
- **Шаг 4:** Создаёт/привязывает Milestone, привязывает все Issues.
- **Шаг 5:** Создаёт ветку через `/branch-create {NNNN}`.
- **Результат:** После `/dev-create` в plan-dev.md есть актуальная таблица маппинга с конкретными номерами Issues.

### validation-development.md (валидация перед push)

- **Когда:** перед push, перед созданием PR, при code review.
- **Шаги:** тесты (`make test`), линтер (`make lint`), сборка (`make build`), E2E тесты, зависимости, полнота реализации.
- **Связь с PR:** validation-development должна быть выполнена ДО вызова `/pr-create`.

### validation-review.md (ревью до PR)

- **Два этапа:** локальное ревью ветки (ДО создания PR) и ревью PR (ПОСЛЕ создания).
- **Шаги branch review:** получить diff, анализировать изменения, вывести замечания.
- **Связь с PR:** `/review` (branch review) выполняется ПЕРЕД созданием PR (шаг 4.1 в standard-process.md).

### labels.yml (справочник меток)

- **TYPE (обязательно, ровно 1):** `bug`, `task`, `docs`, `refactor`. НЕТ `feature` — в драфте и стандарте PR примеры используют `feature`, но такой метки нет.
- **PRIORITY (обязательно, ровно 1):** `critical`, `high`, `medium`, `low`.
- **Опциональные:** STATUS (`ready`, `wip`, `in-review`, `blocked`), AREA (`backend`, `frontend`, `database`, `platform`, `api`, `tests`, `specs`), EFFORT (`xs`, `s`, `m`, `l`, `xl`), ENV (`production`, `staging`, `local`), SVC (динамические `svc:{name}`).

### chain_status.py (модуль статусов)

- chain_status.py НЕ содержит PR-функциональности — работает только со статусами analysis chain.
- `ChainManager.transition()` для переходов (T1-T10).
- `classify_feedback(level)` для определения уровня обратной связи.
- Скрипт `collect-pr-issues.py` будет НОВЫМ, дополнительным к chain_status.py.

## Best practices

### PR Size и Scope

- **Идеальный размер PR:** до 200 строк изменений. 50 строк — оптимум (мержатся на 40% быстрее). >400 строк — слишком много для одного review. >500 строк — стандарт проекта рекомендует разделить.
- **Правило 1 PR = 1 фича/область** уже закреплено в стандарте — это совпадает с индустриальным best practice.
- **Один analysis chain = один PR** — но если chain слишком велик, стоит проверять diff size перед созданием и предупреждать пользователя.

### Автоматическая привязка Issues

- **`gh issue develop N -c --base main`** — создаёт ветку автоматически привязанную к Issue. При последующем `gh pr create` PR автоматически линкуется. Но в проекте ветки создаются через `/branch-create` из chain'а, поэтому этот подход не применим напрямую.
- **`Closes #N` в body** — основной механизм привязки. Каждый Issue на отдельной строке (GitHub парсит корректно). Проект уже использует этот подход.
- **Нет нативного `gh pr edit --add-issue`** — feature request открыт (cli/cli#11405), пока не реализован. Привязка только через keywords в body.
- **Предотвращение дубликатов:** Перед `gh pr create` проверять `gh pr list --head "$BRANCH" --json number --jq length`. Если PR уже существует — предупредить пользователя.

### Draft PR Workflow

- **Когда использовать Draft:** большая фича, нужен early feedback, PR в процессе разработки. Стандарт проекта уже описывает Draft PR подробно.
- **`/pr-create --draft`** — передаёт `--draft` в `gh pr create`. Summary должен содержать `[WIP]`/`[Draft]` в начале.
- **Перевод в Ready:** `gh pr ready {N}` — после заполнения всех секций и прохождения CI.

### Генерация PR Description из коммитов

- **`gh pr create --fill`** — заполняет title и body из коммитов. Для squash: title = первый коммит, body = остальные коммиты.
- **Лучший подход для проекта:** НЕ использовать `--fill`, а генерировать body из analysis chain (discussion.md description + plan-dev.md TASK-N). Коммиты — слишком гранулярные, discussion — слишком высокоуровневая. Оптимально: Summary из discussion.md `description`, Changes из TASK-N с привязкой к Issue#.
- **Conventional Commits в проекте** уже дают структурированные сообщения — их можно использовать как фоллбэк для Changes если chain-данных нет.

### Автоматизация через скрипт

- **Скрипт `collect-pr-issues.py`** — правильный подход: LLM не тратит контекст на парсинг markdown, скрипт возвращает JSON.
- **Данные для скрипта:** plan-dev.md (маппинг TASK-N → Issue#), discussion.md frontmatter (milestone, description), текущая ветка (имя chain'а).
- **Выходной JSON** должен включать: chain, topic, milestone, branch, description, issues[], suggested_type, suggested_title.
- **Определение suggested_type:** Из TYPE-метки Issues (все Issues в chain'е должны иметь одинаковый TYPE). Если метки разные — взять наиболее частую. Если chain содержит `bug` — тип PR = `bug`. Если только `task` — тип PR = `task`.
- **Валидация скриптом:** проверить что все Issues в статусе `open` (если все закрыты — PR не нужен), проверить что ветка существует, проверить наличие непушенных коммитов.

### Prerequisites перед PR

- **Порядок** (из standard-process.md фаза 4): Branch Review (`/review`) → PR Create → PR Review (`/review {N}`) → Merge.
- **Рекомендация:** `/pr-create` должен проверять: 1) текущая ветка не main, 2) есть коммиты для push, 3) chain в RUNNING, 4) validation-development пройдена (make test, make lint). Если нет — предупреждать.
- **validation-development.md** рекомендует полную проверку (тесты, линтер, сборка, E2E) ПЕРЕД созданием PR. `/pr-create` может запустить `make test && make lint` автоматически или проверить что они были запущены.

### Безопасность и UX

- **Preview обязателен** — AskUserQuestion перед `gh pr create`. Показать: title, body (полный), labels, Issues, milestone. Пользователь может скорректировать.
- **`--dry-run`** — показать что будет создано без выполнения. Хорошая практика для CI/CD automation.
- **Идемпотентность:** Если PR для ветки уже существует, не создавать второй. Проверить через `gh pr list --head {branch}`.
- **HEREDOC для body** — использовать heredoc `$(cat <<'EOF' ... EOF)` для передачи multiline body в `gh pr create`, чтобы избежать проблем с кавычками и спецсимволами.

### Что подсмотреть у индустрии

- **GitHub Agentic Workflows (2026 preview):** natural language markdown в `.github/workflows/` → GitHub Actions. `gh aw compile` конвертирует. Пока в preview, но направление: описываешь что хочешь → автоматизация создаётся.
- **Graphite AI PR descriptions:** автоматическая генерация описания из diff + коммитов. В проекте аналог — скрипт `collect-pr-issues.py` + LLM подстановка в шаблон.
- **PR Size Labels (GitHub Marketplace):** автоматическая метка размера PR (XS/S/M/L/XL). Можно добавить в CI или в `/pr-create` как предупреждение если diff > 500 строк.

### Ответы на открытые вопросы (предложения)

1. **`suggested_type`:** Определять из TYPE-меток Issues chain'а через `gh issue view --json labels`. Все Issues создаются `/dev-create` с одинаковым TYPE из Discussion. Если Discussion про фичу — все Issues будут `task` (в labels.yml нет `feature`). Тип PR = TYPE метка Issues. Фоллбэк: определить по Discussion description ключевым словам.
2. **PR без chain'а (C.4 doc-only):** Да, поддерживать. Если ветка не содержит chain NNNN — скрипт возвращает пустой issues[], и `/pr-create` работает в "ручном" режиме: пользователь сам указывает title, body, labels. Или автоматически определяет type=`docs` если изменены только .md файлы.
3. **Summary из discussion.md или коммитов:** Discussion description — для Summary (2-5 предложений о ЧТО). Коммиты — для Changes (bullet list конкретных изменений). Гибридный подход: Summary из discussion.md `description`, Changes из TASK-N title + Issue number.

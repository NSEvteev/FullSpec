## Содержание

### Артефакты

По архитектуре проекта: **инструкция (SSOT) → скилл (обёртка)**. Скилл без SSOT-инструкции запрещён.

| # | Артефакт | Путь | Назначение |
|---|---------|------|------------|
| 1 | **Воркфлоу-инструкция** (SSOT) | `/.github/.instructions/pull-requests/create-pull-request.md` | Пошаговый процесс создания PR — шаги, чек-лист, примеры |
| 2 | **Скилл** (обёртка) | `/.claude/skills/pr-create/SKILL.md` | Ссылка на SSOT, формат вызова `/pr-create` |
| 3 | **Скрипт** | `/.github/.instructions/.scripts/collect-pr-issues.py` | Быстрый сбор Issues/metadata из chain без LLM парсинга |

Инструкция регистрируется в `/.github/.instructions/pull-requests/README.md` и в таблице "Связанные документы" `standard-pull-request.md`.

### Порядок создания

1. `/instruction-create create-pull-request --path .github/.instructions/pull-requests/` — инструкция
2. `/script-create collect-pr-issues --area .github/.instructions/` — скрипт
3. `/skill-create pr-create` — скилл, SSOT → create-pull-request.md

### Проблема

При создании PR пользователь вручную:
1. Собирает номера Issues из milestone'а или plan-dev.md
2. Формирует body с `Closes #N` для каждого Issue
3. Определяет type label из характера изменений
4. Вспоминает формат title
5. Вызывает `gh pr create` с множеством флагов

### Источники данных для автосбора

| Источник | Что даёт | Как получить |
|----------|---------|--------------|
| `plan-dev.md` таблица маппинга | TASK-N → Issue #N (точный список) | Парсинг markdown таблицы |
| `gh issue list --milestone {vX.Y.Z}` | Все Issues milestone'а (включая не из chain'а) | GitHub API / CLI |
| `discussion.md` frontmatter | milestone, описание | Парсинг frontmatter |
| Текущая ветка | Имя ветки `{NNNN}-{topic}` | `git branch --show-current` |
| `labels.yml` | Допустимые type labels | Парсинг YAML |

**Рекомендуемый источник:** `plan-dev.md` таблица маппинга — точный список Issues, созданных `/dev-create`.

### Предлагаемая связка

**Инструкция:** `create-pull-request.md` (SSOT — шаги, чек-лист, примеры)
**Скилл:** `/pr-create` (обёртка — ссылка на SSOT, формат вызова)
**Скрипт:** `collect-pr-issues.py` (сбор данных)

### Формат вызова

```
/pr-create [--draft] [--dry-run]
```

| Параметр | Описание | Обязательный |
|----------|----------|--------------|
| `--draft` | Создать Draft PR | Нет |
| `--dry-run` | Показать что будет создано, не выполнять | Нет |

### Шаги инструкции (create-pull-request.md)

| Шаг | Действие | Инструмент |
|-----|---------|------------|
| 1 | Определить chain | Из имени ветки: `{NNNN}-{topic}` → chain NNNN | git |
| 2 | Проверить prerequisites | Ветка не main, есть unpushed commits, chain в RUNNING | git, chain_status.py |
| 3 | Собрать Issues | Скрипт: парсинг plan-dev.md маппинга → список Issue #N | `collect-pr-issues.py` |
| 4 | Определить title | `{type}: {description}` из Discussion description | Парсинг discussion.md |
| 5 | Сформировать body | Шаблон PR + `Closes #N` для каждого Issue | Шаблон + данные |
| 6 | Определить labels | type + priority из Discussion/Issues | `gh issue view` |
| 7 | Push ветки | `git push -u origin {branch}` (если не pushed) | git |
| 8 | Показать preview | Вывод: title, body, labels, Issues | AskUserQuestion подтверждение |
| 9 | Создать PR | `gh pr create --title "..." --body "..." --label ...` | gh CLI |
| 10 | Отчёт | PR URL, привязанные Issues, labels | Вывод |

### Скрипт: collect-pr-issues.py

**Назначение:** Быстрый сбор данных для PR из analysis chain.

**Расположение:** `.github/.instructions/.scripts/collect-pr-issues.py`

**Входные данные:**
- `{NNNN}` — номер chain
- Путь к `specs/analysis/NNNN-{topic}/plan-dev.md`

**Что делает:**
1. Парсит таблицу "Маппинг GitHub Issues" из plan-dev.md
2. Извлекает номера Issues (#N)
3. Парсит frontmatter discussion.md для milestone и description
4. Проверяет статус Issues через `gh issue view --json state`

**Выходные данные (JSON):**

```json
{
  "chain": "0001",
  "topic": "oauth2-authorization",
  "milestone": "v1.2.0",
  "branch": "0001-oauth2-authorization",
  "description": "OAuth2 авторизация для микросервисов",
  "issues": [
    {"number": 42, "title": "Добавить JWT токены", "state": "open", "task": "TASK-1"},
    {"number": 43, "title": "Middleware авторизации", "state": "open", "task": "TASK-2"},
    {"number": 44, "title": "Refresh токены", "state": "open", "task": "TASK-3"}
  ],
  "suggested_type": "feature",
  "suggested_title": "feature: add OAuth2 authorization"
}
```

**Преимущество скрипта:** LLM не тратит контекст на парсинг — скрипт возвращает готовые данные.

### Нужен ли агент?

**Нет.** Процесс полностью формульный:
1. Скрипт собирает данные (быстро)
2. LLM формирует body из шаблона (простая подстановка)
3. `gh pr create` выполняется

Агент нужен когда есть нетривиальные решения, исследование или многораундовый диалог. PR-создание — это конвейер данные → шаблон → команда.

### Пример использования

```
Пользователь: /pr-create

1. Ветка: 0001-oauth2-authorization → chain 0001
2. Prerequisites: chain RUNNING, есть коммиты ✓
3. Скрипт: collect-pr-issues.py 0001 → 3 Issues (#42, #43, #44)
4. Title: "feature: add OAuth2 authorization"
5. Body:
   ## Summary
   OAuth2 авторизация для микросервисов — JWT токены, middleware, refresh.

   ## Changes
   - JWT token generation and validation (TASK-1)
   - Authorization middleware (TASK-2)
   - Refresh token flow (TASK-3)

   ## Test plan
   - [ ] Unit tests passing
   - [ ] Integration tests passing
   - [ ] Manual testing completed

   ## Related Issues
   Closes #42
   Closes #43
   Closes #44

6. Labels: feature, high
7. Push: git push -u origin 0001-oauth2-authorization
8. Preview: [показать пользователю] → "Создать PR?" → Да
9. gh pr create --title "feature: add OAuth2 authorization" --body "..." --label feature --label high
10. Отчёт: PR #15 создан, 3 Issues привязаны
```

## Решения

- **Инструкция → скилл:** create-pull-request.md (SSOT) → /pr-create (обёртка), по архитектуре проекта
- Основной источник Issues: plan-dev.md маппинг (не `gh issue list --milestone`), потому что он точнее — содержит только Issues этого chain'а
- Скрипт `collect-pr-issues.py` — для быстрого сбора без LLM парсинга
- Агент не нужен — процесс формульный
- Preview перед созданием — обязательный (AskUserQuestion), чтобы пользователь мог скорректировать title/body
- `--dry-run` для безопасного предпросмотра

## Открытые вопросы

- Откуда брать `suggested_type`? Из Discussion (может быть "feature"/"bug"/"refactor") или определять автоматически?
- Нужно ли поддерживать PR без chain'а (для C.4 doc-only changes)?
- Формировать summary из discussion.md description или из коммитов (`--fill`)?

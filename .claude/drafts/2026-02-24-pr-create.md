## Контекст

**Задача:** G2 из standard-process.md — автоматизация создания PR с привязкой Issues chain'а
**Почему создан:** PR создаётся одной командой, но сбор Issues из plan-dev.md, формирование body по шаблону и определение labels — рутинная работа. Скрипт парсит данные без LLM, агент выполняет создание в subprocess.
**Связанные файлы:**
- `/.github/.instructions/pull-requests/standard-pull-request.md` — стандарт PR (SSOT правил)
- `/.github/.instructions/pull-requests/pr-template/standard-pr-template.md` — шаблон body
- `/.github/.instructions/branches/standard-branching.md` — формат имени ветки `{NNNN}-{topic}` (парсинг для определения chain)
- `/.github/.instructions/development/create-development.md` — Шаг 3 (маппинг TASK-N → Issue)
- `specs/.instructions/analysis/plan-dev/standard-plan-dev.md` — таблица маппинга

## Содержание

### Почему не скилл

| Критерий | Скилл `/pr-create` | Агент `pr-create-agent` |
|----------|---------------------|-------------------------|
| Контекст основного LLM | Тратит (скилл = промпт в основном контексте) | Не тратит (subprocess) |
| Чтение стандарта + парсинг | Каждый раз в основном контексте | Один раз в subprocess |
| Сбор данных | LLM парсит markdown | Скрипт возвращает JSON |
| Вызов | Явный `/pr-create` | Автоматический через Task tool |

Агент + скрипт решают две проблемы: **экономия контекста** (subprocess) и **точность сбора данных** (скрипт парсит без LLM).

### Архитектура

```
standard-pull-request.md       (правила PR — что)
standard-pr-template.md        (шаблон body — что)
       ↓ ссылаются
create-pull-request.md         (инструкция процесса — как)
       ↓ SSOT для
collect-pr-issues.py           (сбор данных — дёшево, без LLM)
pr-create-agent (AGENT.md)     (исполнитель — вызывает скрипт, заполняет шаблон, создаёт PR)
```

### Обогащение стандартов: не требуется

- **standard-pull-request.md** — уже покрывает PR исчерпывающе: title формат, body структура, labels, lifecycle, CLI, граничные случаи.
- **standard-pr-template.md** — шаблон body с обязательными/опциональными секциями.
- Единственное обновление: таблица "Связанные документы" — заполнить `Создание: create-pull-request.md` (сейчас `*Будет создан*`).

### Артефакт 1: Скрипт `collect-pr-issues.py`

**Путь:** `/.github/.instructions/.scripts/collect-pr-issues.py`
**Действие:** Создать через `/script-create`.

**Назначение:** Быстрый сбор данных для PR из analysis chain без LLM парсинга.

**Входные данные:**
- `{NNNN}` — номер chain (из имени ветки или аргумент)
- Автоматический поиск `specs/analysis/NNNN-*/plan-dev.md`

**Что делает:**
1. Находит папку chain по номеру NNNN в `specs/analysis/`
2. Парсит таблицу "Маппинг GitHub Issues" из plan-dev.md → извлекает номера Issues (#N) и TASK-N
3. Парсит frontmatter discussion.md → milestone, description
4. Определяет suggested_type из TYPE-меток Issues через `gh issue view --json labels`
5. Проверяет статус Issues через `gh issue view --json state`
6. Проверяет нет ли существующего PR для ветки: `gh pr list --head {branch} --json number`

**Выходные данные (JSON на stdout):**

```json
{
  "chain": "0001",
  "topic": "oauth2-authorization",
  "milestone": "v1.2.0",
  "branch": "0001-oauth2-authorization",
  "description": "OAuth2 авторизация для микросервисов",
  "issues": [
    {"number": 42, "title": "Добавить JWT токены", "state": "open", "task": "TASK-1"},
    {"number": 43, "title": "Middleware авторизации", "state": "open", "task": "TASK-2"},
    {"number": 44, "title": "Refresh токены", "state": "open", "task": "TASK-3"}
  ],
  "suggested_type": "task",
  "suggested_priority": "high",
  "suggested_title": "task: add OAuth2 authorization",
  "existing_pr": null,
  "errors": []
}
```

**Обработка ошибок (поле `errors`):**
- `"no_plan_dev"` — plan-dev.md не найден для chain NNNN
- `"no_issues"` — таблица маппинга пуста (Issues не созданы)
- `"all_closed"` — все Issues закрыты (PR не нужен)
- `"pr_exists"` — PR для ветки уже существует (номер в `existing_pr`)

**Важно:** `suggested_type` берётся из TYPE-меток Issues, НЕ из слова "feature". В labels.yml нет метки `feature` — есть `task`, `bug`, `docs`, `refactor`.

### Артефакт 2: Инструкция `create-pull-request.md`

**Путь:** `/.github/.instructions/pull-requests/create-pull-request.md`
**Действие:** Создать через `/instruction-create`.
**Тип:** воркфлоу (create-*)

Содержание инструкции — операционные детали процесса создания PR:

#### 2.1. Определение chain

1. `git branch --show-current` → `{NNNN}-{topic}`
2. Парсинг NNNN из имени ветки (формат: standard-branching.md)
3. Если ветка = `main` → ошибка: "Нельзя создать PR из main"
4. Если имя не содержит NNNN → fallback: ручной режим (без chain)

#### 2.2. Prerequisites

1. Ветка не main ✓
2. Есть коммиты для push: `git log origin/{branch}..HEAD --oneline` (если remote не существует — все коммиты новые)
3. Chain в RUNNING: `python specs/.instructions/.scripts/chain_status.py --chain {NNNN} --status`
4. Нет существующего PR: проверяется скриптом (поле `existing_pr`)

#### 2.3. Сбор данных (скрипт)

```bash
python .github/.instructions/.scripts/collect-pr-issues.py {NNNN}
```

Агент парсит JSON-ответ. Если `errors` не пуст — обработать:
- `no_plan_dev` → сообщить пользователю, предложить ручной режим
- `all_closed` → "Все Issues закрыты, PR не нужен"
- `pr_exists` → "PR #{N} уже существует для этой ветки"

#### 2.4. Формирование title

Формат: `{type}: {description}` (до 70 символов, нижний регистр, без точки).

- `type` — из `suggested_type` скрипта (метка TYPE из Issues)
- `description` — из `description` скрипта (discussion.md frontmatter), сокращённое до 70 символов минус тип

**SSOT типов:** labels.yml секция TYPE. Допустимые: `bug`, `task`, `docs`, `refactor`. НЕТ `feature`.

#### 2.5. Формирование body

По шаблону PULL_REQUEST_TEMPLATE.md:

```markdown
## Summary
{description из discussion.md — 2-5 предложений}

## Changes
- {TASK-1 title} (#{issue_number})
- {TASK-2 title} (#{issue_number})
- {TASK-3 title} (#{issue_number})

## Test plan
- [ ] Локально протестировано (`make dev`)
- [ ] Pre-commit проверки пройдены
- [ ] Unit-тесты пройдены
- [ ] Интеграционные тесты пройдены (если применимо)
- [ ] Документация обновлена (если нужно)

## Related issues
Closes #{issue_1}
Closes #{issue_2}
Closes #{issue_3}
```

- Секцию **Breaking changes** включить только если есть (иначе удалить)
- Секцию **Notes** включить только если есть (иначе удалить)
- `Closes #N` — каждый Issue на отдельной строке

#### 2.6. Определение labels

- TYPE: из `suggested_type` (ровно 1)
- PRIORITY: из `suggested_priority` (ровно 1)
- Формат флагов: `--label {type} --label {priority}`

#### 2.7. Push ветки

```bash
git push -u origin {branch}
```

Если remote branch уже существует и up-to-date — skip.

#### 2.8. Preview и подтверждение

Показать пользователю через AskUserQuestion:
- Title
- Body (полный текст)
- Labels (type + priority)
- Issues (список Closes #N)
- Milestone

Пользователь может скорректировать любое поле. Подтверждение обязательно.

Если `--draft` — показать пометку "[DRAFT]".
Если `--dry-run` — показать preview и остановиться (не создавать PR).

#### 2.9. Создание PR

```bash
gh pr create --title "{title}" --body "$(cat <<'EOF'
{body}
EOF
)" --label {type} --label {priority}
```

Дополнительные флаги:
- `--draft` если запрошен draft PR
- `--milestone {milestone}` если milestone определён

#### 2.10. Отчёт

Вывести:
- PR URL
- PR номер
- Привязанные Issues (количество и номера)
- Labels
- Milestone
- Следующий шаг: "Запустите `/review {PR-N}` для code review"

**Путь:** `/.claude/agents/pr-create-agent/AGENT.md`
**Действие:** Создать через `/agent-create`.

**Конфигурация:**

### Fallback: PR без chain (C.4 doc-only)

Если ветка не содержит `{NNNN}-{topic}`:
1. Скрипт возвращает `errors: ["no_chain"]`, пустой `issues[]`
2. Агент переходит в ручной режим:
   - type определяется из diff (`git diff --stat` — только .md файлы → `docs`)
   - Issues не привязываются (или пользователь указывает вручную)
   - Summary из коммитов (`git log --oneline`)
3. Preview и подтверждение — как обычно

## Решения

- **Скилл `/pr-create` не создаём** — агент решает задачу экономии контекста лучше
- **Скрипт `collect-pr-issues.py`** — парсинг данных без LLM (дёшево, точно, быстро)
- **Агент `pr-create-agent`** — subprocess, вызывает скрипт, заполняет шаблон, создаёт PR
- **Модель агента: haiku** — формульный процесс (скрипт → шаблон → команда)
- **Стандарты не обогащаем** — standard-pull-request.md и standard-pr-template.md уже полные
- **Основной источник Issues:** plan-dev.md маппинг (точнее чем `gh issue list --milestone`)
- **Preview обязателен** — AskUserQuestion перед `gh pr create`
- **`--dry-run`** — показать preview без создания PR
- **Fallback без chain** — ручной режим для C.4 doc-only changes
- **`suggested_type` из TYPE-меток Issues** — НЕ из слова "feature" (в labels.yml нет `feature`)
- **Summary из discussion.md** — description для Summary, TASK-N для Changes (гибридный подход)
- **HEREDOC для body** — `$(cat <<'EOF' ... EOF)` для multiline без проблем с кавычками

## Что уже описано в проекте

### standard-pull-request.md (полный стандарт PR)

- **Title формат:** `{type}: {краткое описание}`, до 70 символов, нижний регистр, без точки. Опциональный scope: `{type}({scope}): {описание}`. Тип ДОЛЖЕН совпадать с одной из меток TYPE из labels.yml.
- **Допустимые TYPE-метки:** `bug`, `task`, `docs`, `refactor` (в labels.yml НЕТ `feature` — это важно!).
- **Body:** заполняется по шаблону `PULL_REQUEST_TEMPLATE.md`. Секции: Summary, Changes, Breaking changes (опц.), Test plan (чек-лист), Related issues, Notes (опц.).
- **Обязательные метки при создании PR:** ровно 1 TYPE + ровно 1 PRIORITY.
- **Связь с Issues:** `Closes #N` на отдельных строках в секции `## Related issues`. Issues закрываются автоматически ТОЛЬКО после merge в целевую ветку.
- **Группировка:** одна фича/область = группа Issues = один PR. Запрещено создавать отдельный PR на каждый Issue. Разделять на несколько PR: если Issues из разных фич, если >500 строк изменений.
- **Draft PR:** для early feedback, WIP. Merge запрещён до Ready. Summary обязательно с `[WIP]`/`[Draft]`. Related issues обязательно для трекинга. Перевод: `gh pr ready 123`.
- **Жизненный цикл:** СОЗДАНИЕ → АВТОПРОВЕРКИ (pre-commit, CI) → CODE REVIEW (approve/request changes) → MERGE (squash) → АВТОЗАКРЫТИЕ ISSUES.
- **CLI:** `gh pr create --title "..." --body "..." --label ...`. Также: `--fill` (из коммитов), `--draft`, `--base develop`, `--reviewer @user`.
- **Граничные случаи:** PR без Issue запрещён (исключение: минорные правки). Hotfix: Issue с `critical`, ветка `{NNNN}-hotfix-{desc}`, ускоренное ревью. Длительная разработка: синхронизировать с main минимум 1 раз в 2 дня.
- **В стандарте `Создание: *Будет создан*`** — именно этот документ и является целью данного драфта.

### standard-pr-template.md (стандарт шаблона)

- **Расположение:** `.github/PULL_REQUEST_TEMPLATE.md` — единственный шаблон, подставляется автоматически.
- **Обязательные секции:** Summary (2-5 предложений), Changes (bullet list), Test plan (чек-лист), Related issues (Closes #N).
- **Опциональные секции:** Breaking changes, Notes. Если не заполнены — удалить (пустые с placeholder недопустимы в готовом PR).
- **Placeholder-ы:** квадратные скобки `[...]`, императив. Для Related issues — HTML-комментарий `<!-- -->` чтобы GitHub не парсил `#N`.
- **Строка `Generated with Claude Code`** добавляется автоматически, не включать в шаблон.

### PULL_REQUEST_TEMPLATE.md (сам шаблон)

```markdown
## Summary
[Опишите изменения в 2-5 предложениях]

## Changes
- [Изменение 1]
- [Изменение 2]
- [Изменение 3]

## Breaking changes
[Перечислите ломающие изменения, если есть. Если нет — удалите эту секцию.]

## Test plan
- [ ] Локально протестировано (`make dev`)
- [ ] Pre-commit проверки пройдены
- [ ] Unit-тесты пройдены
- [ ] Интеграционные тесты пройдены (если применимо)
- [ ] Документация обновлена (если нужно)

## Related issues
<!-- Укажите связанные Issues: Fixes #номер, Closes #номер, Related to #номер -->

## Notes
[Дополнительная информация для ревьюера. Если нет — удалите эту секцию.]
```

### standard-issue.md (стандарт Issues)

- **Жизненный цикл Issue:** СОЗДАНИЕ → ОТКРЫТ → В РАБОТЕ (ветка создана) → PR СОЗДАН → REVIEW → ЗАКРЫТ (автоматически при merge).
- **Закрытие Issue:** автоматически через `Closes #N` в PR. Ручное закрытие с `completed` ЗАПРЕЩЕНО. Только `not planned` для дубликатов/неактуальных.
- **Связь с Branch и PR:** Issue → ветка `{NNNN}-{topic}` → PR с `Fixes #N` → merge → Issue закрывается автоматически.
- **Milestone:** Issue ДОЛЖЕН быть добавлен в Milestone. Milestone берётся из frontmatter Discussion.

### standard-github-workflow.md (полный цикл)

- **Стадия 6 (PR):** Процесс: 1) `/review` (локальное ревью до PR), 2) `git push -u origin {branch}`, 3) `gh pr create`. Все Issues группы закрываются одним PR через `Closes #42, #43, #44`.
- **Принцип:** одна фича/область → группа Issues → одна ветка → один PR.

### standard-process.md (шаг 4.2 — PR Create)

- **Фаза 4, шаг 4.2:** PR Create = `git push` + `gh pr create` с привязкой Issues. Скилла нет, инструкции нет — помечен как пробел G2.
- **Покрытие:** в сводной таблице инструментов шаг 4.2 имеет SSOT `standard-pull-request.md` + `standard-pr-template.md`, но **НЕТ скилла и НЕТ скрипта**.
- **Pre-commit хук:** `pr-template-validate` выполняется на шаге 4.2.

### Маппинг TASK-N → Issues (plan-dev.md)

- **Формат таблицы маппинга** в plan-dev.md (создаётся `/dev-create` на шаге 3):
  ```
  | Элемент Plan Dev | GitHub Issue |
  |------------------|-------------|
  | TASK-N | Отдельный Issue через `/issue-create` |
  | Подзадачи N.M | Чек-лист в body Issue |
  | Приоритет | Label приоритета |
  | Зависимости | `**Зависит от:** #N` в body |
  | TC | Ссылка в секции "Связанная документация" |
  | Milestone | Из frontmatter Discussion |
  ```
- **Конкретные номера Issues** создаются `/dev-create` и записываются в план. Скрипт `collect-pr-issues.py` парсит эту обновлённую таблицу.

### Discussion frontmatter (источник metadata)

- **Поля:** `description` (до 1024 символов), `milestone` (vX.Y.Z), `status`, `children`.
- Milestone проекта = Milestone из Discussion frontmatter.
- Description проекта = `description` из Discussion frontmatter — используется для PR Summary.

### create-development.md (dev-create workflow)

- **Шаг 3:** Создаёт Issues из TASK-N, обновляет таблицу маппинга в plan-dev.md.
- **Шаг 4:** Создаёт/привязывает Milestone, привязывает все Issues.
- **Шаг 5:** Создаёт ветку через `/branch-create {NNNN}`.
- **Результат:** После `/dev-create` в plan-dev.md есть актуальная таблица маппинга с конкретными номерами Issues.

### validation-development.md (валидация перед push)

- **Когда:** перед push, перед созданием PR, при code review.
- **Шаги:** тесты (`make test`), линтер (`make lint`), сборка (`make build`), E2E тесты, зависимости, полнота реализации.
- **Связь с PR:** validation-development должна быть выполнена ДО вызова pr-create-agent.

### validation-review.md (ревью до PR)

- **Два этапа:** локальное ревью ветки (ДО создания PR) и ревью PR (ПОСЛЕ создания).
- **Шаги branch review:** получить diff, анализировать изменения, вывести замечания.
- **Связь с PR:** `/review` (branch review) выполняется ПЕРЕД созданием PR (шаг 4.1 в standard-process.md).

### labels.yml (справочник меток)

- **TYPE (обязательно, ровно 1):** `bug`, `task`, `docs`, `refactor`. **НЕТ `feature`** — скрипт и агент используют `task` для фич.
- **PRIORITY (обязательно, ровно 1):** `critical`, `high`, `medium`, `low`.
- **Опциональные:** STATUS (`ready`, `wip`, `in-review`, `blocked`), AREA (`backend`, `frontend`, `database`, `platform`, `api`, `tests`, `specs`), EFFORT (`xs`, `s`, `m`, `l`, `xl`), ENV (`production`, `staging`, `local`), SVC (динамические `svc:{name}`).

### chain_status.py (модуль статусов)

- chain_status.py НЕ содержит PR-функциональности — работает только со статусами analysis chain.
- `ChainManager.transition()` для переходов (T1-T10).
- Скрипт `collect-pr-issues.py` будет НОВЫМ, дополнительным к chain_status.py.

## Best practices

### PR Size и Scope

- **Идеальный размер PR:** до 200 строк изменений. 50 строк — оптимум (мержатся на 40% быстрее). >400 строк — слишком много для одного review. >500 строк — стандарт проекта рекомендует разделить.
- **Правило 1 PR = 1 фича/область** уже закреплено в стандарте — совпадает с индустриальным best practice.
- **Один analysis chain = один PR** — но если chain слишком велик, стоит проверять diff size перед созданием и предупреждать пользователя.

### Автоматическая привязка Issues

- **`Closes #N` в body** — основной механизм привязки. Каждый Issue на отдельной строке (GitHub парсит корректно).
- **Нет нативного `gh pr edit --add-issue`** — привязка только через keywords в body.
- **Предотвращение дубликатов:** скрипт проверяет `gh pr list --head {branch}` и возвращает `existing_pr`.

### Draft PR Workflow

- **Когда использовать Draft:** большая фича, early feedback, PR в процессе разработки.
- **`--draft` параметр** — передаётся в `gh pr create --draft`. Summary должен содержать `[WIP]`/`[Draft]`.
- **Перевод в Ready:** `gh pr ready {N}` — после заполнения всех секций.

### Генерация PR Description

- **НЕ использовать `--fill`** — коммиты слишком гранулярные.
- **Гибридный подход:** Summary из discussion.md `description`, Changes из TASK-N title + Issue#.
- **Conventional Commits** — фоллбэк для Changes если chain-данных нет.

### Безопасность и UX

- **Preview обязателен** — AskUserQuestion перед `gh pr create`.
- **`--dry-run`** — показать что будет создано без выполнения.
- **Идемпотентность:** скрипт проверяет существующий PR, агент не создаёт дубликат.
- **HEREDOC для body** — `$(cat <<'EOF' ... EOF)` для multiline без проблем с кавычками.

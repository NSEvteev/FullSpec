## 1. Двойственность формулировок

### Проблема 1.1: Неопределённое местоимение "вы"

**Где:** Секция "Approve и Merge" (строки 93-101)

**Цитата:**
> Агент работает под вашей учёткой GitHub. Формального "Approved by @reviewer" не будет — вы сами делаете approve после проверки комментариев.

**Как может понять LLM:**
- Интерпретация A: Пользователь должен вручную запустить `gh pr review --approve` после чтения комментариев агента
- Интерпретация B: Агент сам выполняет approve от имени пользователя после написания комментариев
- Интерпретация C: Approve опционален, можно сразу мержить

**Рекомендация:**
> После прочтения комментариев агента пользователь ДОЛЖЕН вручную выполнить approve через `gh pr review {PR} --approve`. Агент НЕ выполняет approve автоматически — он только пишет комментарии.

**Критичность:** Высокая (P1)

---

### Проблема 1.2: Размытое условие "Если всё хорошо"

**Где:** Секция "Approve и Merge" (строки 96-101)

**Цитата:**
> ```bash
> # Если всё хорошо — мержим
> gh pr merge 123 --squash
> ```

**Как может понять LLM:**
- Интерпретация A: "Всё хорошо" = нет комментариев от агента
- Интерпретация B: "Всё хорошо" = комментарии есть, но некритичные
- Интерпретация C: "Всё хорошо" = пользователь субъективно оценил качество кода

**Рекомендация:**
> ```bash
> # Если комментарии агента проверены и не требуют исправлений — мержим
> gh pr review 123 --approve
> gh pr merge 123 --squash
>
> # Если комментарии требуют исправлений — внести правки, затем повторить ревью
> ```

**Критичность:** Средняя (P2)

### Проблема 1.3: Неявное поведение при конфликтах

**Где:** Секция "Разрешение конфликтов" (строки 208-223)

**Цитата:**
> 3. Если конфликты сложные — попросить автора PR их разрешить

**Как может понять LLM:**
- Интерпретация A: LLM сам решает, "сложные" конфликты или нет, и действует соответственно
- Интерпретация B: LLM всегда пытается разрешить конфликты сам, и только при неудаче сообщает пользователю
- Интерпретация C: Критерий "сложные" не определён — LLM не знает, когда применять пункт 3

**Рекомендация:**
> 3. Если конфликты затрагивают более 3 файлов ИЛИ более 50 строк ИЛИ автоматическое слияние невозможно — сообщить пользователю: "Конфликты требуют ручного разрешения. Выполните merge локально."

**Критичность:** Высокая (P1)

### Проблема 1.4: Двусмысленная формулировка "По желанию"

**Где:** Секция "Branch Protection Rules" (таблица рекомендаций, строки 236-246)

**Цитата:**
> | **Require conversation resolution** | Все комментарии resolved | По желанию |
> | **Require signed commits** | Подписанные коммиты | По желанию |

**Как может понять LLM:**
- Интерпретация A: Пользователь решает, включать или нет (рекомендация слабая)
- Интерпретация B: Включать, если команда большая; не включать для соло-разработчика
- Интерпретация C: Включать всегда, но не блокирующее требование

**Рекомендация:**
> | **Require conversation resolution** | Все комментарии resolved | Для команд 2+ человек |
> | **Require signed commits** | Подписанные коммиты | Для публичных проектов |

**Критичность:** Средняя (P2)

### Проблема 1.5: Неявный порядок действий при revert

**Где:** Секция "Откат после merge (revert)" (строки 352-371)

**Цитата:**
> **Процесс:**
> 1. Создать revert-коммит в main:
>    ```bash
>    git checkout main
>    git pull origin main
>    git revert {commit-hash-of-merge}
>    git push origin main
>    ```
> 2. Альтернатива через GitHub UI:
>    - Открыть PR: `gh pr view {номер} --web`
>    - В UI GitHub: нажать "Revert" (создаст новый PR с откатом)
>    - Смержить revert-PR: `gh pr merge {revert-PR-number} --squash`

**Как может понять LLM:**
- Интерпретация A: Выполнить пункт 1 ИЛИ пункт 2 (на выбор LLM или пользователя)
- Интерпретация B: Сначала пункт 1, если не получилось — пункт 2
- Интерпретация C: Пункт 1 нарушает правило "запрет прямых коммитов в main" — LLM не должен его использовать

**Рекомендация:**
> **Процесс:**
>
> **Вариант 1: Через CLI (для срочных hotfix):**
> ```bash
> git checkout main
> git pull origin main
> git revert {commit-hash-of-merge}
> git push origin main
> # ⚠️ ВНИМАНИЕ: Прямой коммит в main. Использовать ТОЛЬКО для критичных откатов.
> ```
>
> **Вариант 2: Через PR (рекомендуется для плановых откатов):**
> 1. Открыть PR: `gh pr view {номер} --web`
> 2. В UI GitHub: нажать "Revert" (создаст новый PR с откатом)
> 3. Смержить revert-PR: `gh pr merge {revert-PR-number} --squash`

**Критичность:** Средняя (P2)

## 2. Риск неправильной интерпретации LLM

### Проблема 2.1: Пропущен шаг между локальным ревью и push

**Где:** Секция "Этап 1: Локальное ревью" (строки 55-73)

**Цитата:**
> **Результат:** Агент выводит замечания локально. Исправить **до** создания PR.
>
> **Цель:** PR создаётся уже проверенным, без тривиальных проблем.

**Риск:**
LLM может подумать, что после вывода замечаний агента нужно сразу создавать PR, пропустив шаг исправления замечаний и коммита исправлений.

**Рекомендация:**
> **Результат:** Агент выводит замечания локально.
>
> **Процесс исправления:**
> 1. Прочитать замечания агента
> 2. Внести исправления в код
> 3. Закоммитить изменения: `git add . && git commit -m "fix: address review comments"`
> 4. Повторить `/review-branch` (опционально, если исправления значительные)
> 5. Только после устранения всех замечаний — создавать PR
>
> **Цель:** PR создаётся уже проверенным, без тривиальных проблем.

**Критичность:** Высокая (P1)

---

### Проблема 2.2: Неявная зависимость между "Approve" и "Merge"

**Где:** Секция "Approve и Merge" (строки 93-101)

**Цитата:**
> ```bash
> # Если всё хорошо — мержим
> gh pr merge 123 --squash
> ```

**Риск:**
LLM может выполнить `gh pr merge` без предварительного `gh pr review --approve`, если Branch Protection Rules не настроены (минимум 1 approval).

**Рекомендация:**
> ```bash
> # Шаг 1: Approve (ОБЯЗАТЕЛЬНО, даже если вы автор PR)
> gh pr review 123 --approve
>
> # Шаг 2: Merge (выполняется ПОСЛЕ approve)
> gh pr merge 123 --squash
>
> # Если нужны правки — исправляем, пушим, повторяем ревью
> ```

**Критичность:** Высокая (P1)

### Проблема 2.3: Инструкция содержит прямой коммит в main (противоречие с § 5)

**Где:** Секция "Откат после merge (revert)" (строки 356-362)

**Цитата:**
> ```bash
> git checkout main
> git pull origin main
> git revert {commit-hash-of-merge}
> git push origin main
> ```

**Риск:**
LLM может использовать этот пример как разрешение на прямые коммиты в main в других ситуациях, противореча секции "5. Блокирующие условия" (строки 278-286): "Прямые коммиты в main запрещены."

**Рекомендация:**
Добавить явное предупреждение после команды:
> ```bash
> git revert {commit-hash-of-merge}
> git push origin main
> # ⚠️ ИСКЛЮЧЕНИЕ: Это единственный случай прямого коммита в main.
> # Использовать ТОЛЬКО для критичных откатов production.
> # В остальных случаях — создавать revert-PR (см. Вариант 2).
> ```

**Критичность:** Высокая (P1)

### Проблема 2.4: Отсутствие явного алгоритма "когда применять auto-merge"

**Где:** Секция "Auto-merge" (строки 188-206)

**Цитата:**
> **Условия срабатывания auto-merge:**
> 1. Все CI workflows завершились успешно (status: success)
> 2. Получены необходимые approvals (если настроены Branch Protection Rules)
> 3. Нет активных "Requested changes" от reviewers
> 4. Нет merge conflicts

**Риск:**
LLM не знает, **когда** пользователю стоит использовать `--auto`, а когда мержить вручную сразу. Отсутствуют критерии выбора между `gh pr merge --squash` и `gh pr merge --auto --squash`.

**Рекомендация:**
Добавить подсекцию "Когда использовать auto-merge":
> **Когда использовать:**
> - CI workflows запущены, но ещё не завершены (ожидание 5+ минут)
> - Нужно мержить вне рабочего времени (например, настроить auto-merge перед уходом)
> - PR требует approval от других ревьюеров, которые ещё не ответили
>
> **Когда НЕ использовать:**
> - Все условия уже выполнены (CI пройден, approvals получены) → мержить вручную сразу
> - Срочный hotfix → мержить вручную

**Критичность:** Средняя (P2)

### Проблема 2.5: Пропущен шаг "удаление локальной ветки" после merge

**Где:** Секция "После merge" (строки 224-229)

**Цитата:**
> - Feature-ветка автоматически удаляется
> - Все Issues закрываются (если указаны `Closes #42`, `Closes #43`, `Closes #44`)
> - PR переходит в статус "Merged"

**Риск:**
LLM может подумать, что локальная ветка удаляется автоматически. На самом деле автоматически удаляется только **remote-ветка** на GitHub. Локальная ветка остаётся.

**Рекомендация:**
> **После merge:**
>
> **Автоматически (на GitHub):**
> - Remote feature-ветка удаляется
> - Все Issues закрываются (если указаны `Closes #42`, `Closes #43`)
> - PR переходит в статус "Merged"
>
> **Локальная очистка:** → [standard-branching.md § 3](../branches/standard-branching.md#3-жизненный-цикл-ветки) (SSOT жизненного цикла ветки)

**Критичность:** Средняя (P2)
**Зона:** Детальные команды очистки — зона `standard-branching.md`. В review добавить только ссылку.

**Стоит ли добавлять:** Да (критично для LLM)

LLM сможет корректно обработать ситуации: "insufficient permissions", "conflict detected", "PR already merged".

**Предложение:**

> ### Обработка ошибок
>
> | Ошибка | Причина | Решение |
> |--------|---------|---------|
> | `HTTP 403: Resource not accessible` | Нет прав на репозиторий | Проверить `gh auth status`, выполнить `gh auth login` |
> | `pull request is in clean status` | PR уже смержен или закрыт | Проверить статус: `gh pr view {номер}` |
> | `merge conflict` | Конфликты с целевой веткой | См. [§ 3. Разрешение конфликтов](#разрешение-конфликтов) |
> | `required status checks must pass` | CI не пройден | Дождаться успешного прохождения CI или исправить ошибки |

## 5. Слепые зоны автора

### Зона 5.1: Метрики качества ревью

**Суть:**
Документ описывает процесс ревью, но не содержит метрик: сколько комментариев агента — норма? Какой процент PR требует Request changes? Какое среднее время ревью?

**Почему важно:**
Метрики позволяют оценить качество кода и эффективность процесса ревью. LLM может использовать метрики для рекомендаций: "Агент нашёл 15 замечаний — это выше среднего, рекомендуется глубокий рефакторинг."

**Расширение качества:** Умеренное

**Рекомендация:** Добавить опционально (не блокирует работу)

**Пример добавления:**
> ### Метрики качества ревью (опционально)
>
> | Метрика | Норма | Пример трекинга |
> |---------|-------|-----------------|
> | Среднее количество комментариев агента на PR | 3-7 | `/.claude/metrics/review-stats.json` |
> | Процент PR с Request changes | 20-30% | GitHub API: `state:changes_requested` |
> | Среднее время от создания PR до merge | 1-3 дня | Вычислить по `created_at` и `merged_at` |
>
> **Использование:**
> - Если метрика выше нормы → рассмотреть улучшение процесса разработки
> - Если метрика ниже нормы → возможно, ревью недостаточно строгое

**Критичность:** Низкая (P3) — улучшение, но не блокирует

---

### Зона 5.2: Интеграция с уведомлениями (notifications)

**Суть:**
Документ не описывает, как пользователь узнаёт о событиях: "PR готов к ревью", "Request changes от ревьюера", "CI завершён". GitHub отправляет уведомления, но как LLM может интегрироваться с этим процессом?

**Почему важно:**
LLM-агент может автоматически реагировать на уведомления: при получении "CI failed" → запустить диагностику, при "Request changes" → предложить исправления.

**Расширение качества:** Значительное

**Рекомендация:** Отложить (требует агентов с доступом к GitHub API events)

**Рекомендация:**
> **Зона другого стандарта.** Webhook-триггеры и CI-интеграция — зона [standard-workflow-file.md](../workflows-files/standard-workflow-file.md). Отложить.

**Критичность:** Низкая (P3) — over-engineering для текущей стадии

## 1. Двойственность формулировок

### Проблема 1.1: Неопределённый критерий "при проблемах с окружением"

**Где:** Секция "2. Make-команды" → таблица "Команды разработки" (строка 93)

**Цитата:**
> `make clean` | Полная очистка (docker down -v) при проблемах с окружением

**Как может понять LLM:**
- Интерпретация A: Использовать `make clean` при любом подозрении на проблему (слишком широко)
- Интерпретация B: Использовать только когда `make dev` не запускается или сервисы падают (конкретные симптомы)
- Интерпретация C: Использовать если `make stop` не помог (есть промежуточный шаг)

**Рекомендация:**
> `make clean` | Полная очистка (docker down -v). Использовать, если: сервисы не запускаются после `make dev`, зависшие контейнеры после `make stop`, ошибки "port already in use" или "volume not found"

**Критичность:** P1 (Высокая — неверное применение `make clean` может удалить данные)

### Проблема 1.2: Двойственность "после каждого логического блока изменений"

**Где:** Секция "2. Make-команды" → таблица "Команды проверки" (строка 99)

**Цитата:**
> `make test` | После каждого логического блока изменений

**Как может понять LLM:**
- Интерпретация A: После каждой сохранённой функции (слишком часто, замедляет разработку)
- Интерпретация B: Перед каждым коммитом (совпадает с требованием из § 4, дублирование)
- Интерпретация C: После реализации каждой задачи из Issues (может быть несколько коммитов на задачу)

**Рекомендация:**
> `make test` | После реализации функциональности перед коммитом. Минимум: при готовности закоммитить изменения. Опционально: чаще при итеративной разработке

**Критичность:** P2 (Средняя — влияет на эффективность процесса)

### Проблема 1.3: Противоречие "перед коммитом" и "перед push"

**Где:** Секция "2. Make-команды" (строки 100-101) и "4. Проверки качества" (строки 138-139)

**Цитата:**
```
make lint | Перед коммитом — проверка стиля кода
make build | Перед push — проверка, что проект собирается
```

**vs.**

```
1. **Тесты:** `make test` — все тесты проходят
2. **Линтинг:** `make lint` — нет ошибок стиля
```

**Как может понять LLM:**
- Интерпретация A: `make lint` запускать ПЕРЕД коммитом (указано в § 2), `make build` ПЕРЕД push (указано в § 2)
- Интерпретация B: `make lint` запускается АВТОМАТИЧЕСКИ при коммите (pre-commit hooks из § 4), вручную не нужен
- Интерпретация C: И вручную перед коммитом, и через pre-commit hooks (двойная проверка)

**Рекомендация:**
Уточнить в § 2:
> `make lint` | Опционально вручную перед коммитом. Автоматически запускается через pre-commit hooks при `git commit`
> `make build` | Перед push — проверка, что проект собирается для production

Уточнить в § 4:
> 1. **Тесты:** `make test` — все тесты проходят (вручную перед коммитом)
> 2. **Линтинг:** Автоматически через pre-commit hooks. Вручную: `make lint` (если нужна проверка до коммита)

> 🔀 **ЗОНА [D1]:** Вторгается в Commits (standard-commit.md §6). Pre-commit hooks уже покрыты в commit §6. При применении: НЕ дублировать описание hooks — SSOT-ссылка → standard-commit.md §6.

**Критичность:** P1 (Высокая — непонятно, нужно ли запускать вручную или hooks достаточно)

### Проблема 1.4: Неопределённость "если затронуты интеграционные сценарии"

**Где:** Секция "3. Тестирование" → "Требования перед созданием PR" (строка 122)

**Цитата:**
> `make test-e2e` проходит без ошибок (если затронуты интеграционные сценарии)

**Как может понять LLM:**
- Интерпретация A: Запускать всегда — любое изменение может затронуть интеграции (перестраховка)
- Интерпретация B: Запускать, если изменения в `/src/{service}/` затрагивают API endpoints или database (конкретно)
- Интерпретация C: Запускать, если автор PR считает, что затронуты — субъективная оценка (риск пропуска)

**Рекомендация:**
> `make test-e2e` проходит без ошибок. Обязательно при изменениях: API endpoints, database schema, inter-service communication, gateway routes. Опционально: изменения только внутри одного сервиса без внешних зависимостей

**Критичность:** P1 (Высокая — пропуск E2E тестов может привести к поломке интеграций в production)

**Где:** Секция "1. Процесс разработки" → "Цикл работы в feature-ветке" → "Принципы" (строка 76)

**Цитата:**
> **Итеративность** — повторять шаги 2-4 до достижения результата

**vs. ссылка в standard-branching.md (строка 131):**
> При длительной разработке (>2 дней) — синхронизировать feature-ветку с main

**Как может понять LLM:**

- Интерпретация B: "Итеративность" включает синхронизацию с main при >2 дней (нигде не упомянуто в standard-development.md)
- Интерпретация C: Синхронизация — отдельный процесс, не часть "Итеративности" (неясно, когда вызывать)

**Рекомендация:**
Добавить в "Принципы":
> **Синхронизация с main** — при разработке >2 дней синхронизировать feature-ветку с main (→ [standard-sync.md § 3](../sync/standard-sync.md#3-процесс-синхронизации)) для минимизации конфликтов при merge

> 🔀 **ЗОНА [D2]:** Вторгается в Sync (standard-sync.md). Триггер синхронизации уже покрыт в sync §2. При применении: добавить SSOT-ссылку → standard-sync.md, не детали.

**Критичность:** P2 (Средняя — пропуск синхронизации усложняет merge, но не блокирует разработку)

### Проблема 1.6: Неопределённость "исправить failing тесты"

**Где:** Секция "1. Процесс разработки" → "Цикл работы в feature-ветке" (строка 64)

**Цитата:**
> 3. ЛОКАЛЬНОЕ ТЕСТИРОВАНИЕ
>    └─ make test
>    └─ Исправить failing тесты

**Как может понять LLM:**
- Интерпретация A: Исправить failing тесты сейчас (блокирующее действие, нельзя продолжать пока не пройдут)
- Интерпретация B: Отметить failing тесты и вернуться к ним позже (можно продолжить шаг 2)
- Интерпретация C: Исправить только если это тесты текущих изменений, пропустить если упали другие тесты (выборочное исправление)

**Рекомендация:**
> 3. ЛОКАЛЬНОЕ ТЕСТИРОВАНИЕ
>    └─ make test
>    └─ Все тесты должны пройти. При failing тестах:
>       - Если упали тесты текущих изменений → исправить код, повторить шаг 2-3
>       - Если упали существующие тесты → проверить актуальность main (синхронизация), исправить breaking changes
>       **БЛОКИРОВКА:** Переход к шагу 4 запрещён при failing тестах

**Критичность:** P2 (Средняя — неясно, блокирует ли failing тест переход к коммиту)

## 2. Риск неправильной интерпретации

### Проблема 2.2: Неявное требование "все изменения"

**Где:** Секция "4. Проверки качества" → "Checklist перед push" (строка 147)

**Цитата:**
> - [ ] Код реализует задачи из Issues группы

**Как может понять LLM:**
- Интерпретация A: Все задачи из Issues группы должны быть реализованы ПОЛНОСТЬЮ перед push (блокирующее требование)
- Интерпретация B: Можно пушить частичную реализацию, если хотя бы одна задача завершена (итеративная разработка)
- Интерпретация C: Чек-лист проверяет соответствие КОММИТОВ задачам, не требует полной реализации всех Issues (проверка корректности, не полноты)

**Рекомендация:**
> - [ ] Код реализует ВСЕ задачи из Issues группы ИЛИ Issues частично реализованных задач отмечены как "In Progress" (→ [standard-issue.md](../issues/standard-issue.md))

Или, если разрешены частичные реализации:
> - [ ] Код реализует минимум одну ЗАВЕРШЁННУЮ задачу из Issues группы. Незавершённые задачи отмечены "In Progress"

**Критичность:** P2 (Средняя — влияет на процесс итеративной разработки, но не блокирует критично)

### Проблема 2.3: Отсутствие критериев "проект собирается"

**Где:** Секция "4. Проверки качества" → "Checklist перед push" (строка 150)

**Цитата:**
> - [ ] Проект собирается (`make build`)

**Как может понять LLM:**
- Интерпретация A: `make build` завершился с exit code 0 (формальная проверка)
- Интерпретация B: `make build` создал валидные артефакты сборки И запускается (функциональная проверка)
- Интерпретация C: `make build` прошёл без ошибок компиляции, warnings допустимы (частичная проверка)

**Рекомендация:**
> - [ ] Проект собирается (`make build` завершается успешно — exit code 0, нет ошибок компиляции)

Или добавить в § 2 описание ожидаемого результата `make build`:
```markdown
**Ожидаемый результат:** exit code 0, артефакты созданы в `/build/` или `/dist/`, нет ошибок компиляции. Warnings допустимы.
```

**Критичность:** P3 (Низкая — большинство разработчиков понимают "сборка прошла", но уточнение повысит ясность)

### Проблема 2.5: Неопределённость "ошибки линтера"

**Где:** Секция "1. Процесс разработки" → "Цикл работы в feature-ветке" (строка 68)

**Цитата:**
> 4. ПРОВЕРКИ КАЧЕСТВА
>    └─ make lint
>    └─ Исправить ошибки линтера

**Как может понять LLM:**
- Интерпретация A: Исправить ВСЕ ошибки И warnings из вывода `make lint` (максимально строго)
- Интерпретация B: Исправить только ERRORS, warnings можно игнорировать (частичное исправление)

> 4. ПРОВЕРКИ КАЧЕСТВА
>    └─ make lint
>    └─ Исправить все ERRORS в выводе линтера. Warnings: исправить в изменённых файлах, существующие warnings в неизменённом коде — допустимы
>    **БЛОКИРОВКА:** Переход к шагу 5 запрещён при наличии ERRORS

## 4. Полнота описания объекта

### 4.1. Не хватает: Процесс отладки при failing тестах

**Что отсутствует:**
Секция или подсекция "Отладка и диагностика" с описанием:
- Как читать вывод `make test` при ошибках
- Как запустить отдельный тест для отладки
- Как использовать debugger в локальном окружении
- Куда смотреть при ошибках "test timeout" или "connection refused"

**Почему важно:**
Шаг "Исправить failing тесты" (строка 64) предполагает, что разработчик умеет диагностировать причину. Новички или LLM без контекста могут не понимать, КАК исправить failing тест.

**Стоит ли добавлять:** Да (опционально — можно вынести в отдельный документ `troubleshooting-development.md`)

**Что даст:** Снижение времени на отладку, ясные шаги для LLM при провале тестов

**Критичность:** P2 (Средняя)

### 4.2. Не хватает: Процесс работы с зависимостями

**Что отсутствует:**
- Как добавить новую зависимость (npm install, pip install, go get)
- Как обновить зависимость
- Нужно ли коммитить lockfile (package-lock.json, Pipfile.lock, go.sum)
- Когда запускать `make setup` повторно после добавления зависимости

**Почему важно:**
§ 2 описывает make-команды, но не покрывает управление зависимостями — критичный аспект разработки. LLM может добавить зависимость, но не знать, нужно ли коммитить lockfile или обновлять окружение.

**Стоит ли добавлять:** Да

**Что даст:** Полнота описания процесса разработки, предотвращение ошибок "dependency not found" у других разработчиков

**Критичность:** P1 (Высокая — отсутствие правил работы с зависимостями может сломать окружение команды)

### 4.4. Не хватает: Граничные случаи для "длительной разработки"

**Что отсутствует:**
- Как часто синхронизировать feature-ветку с main при разработке >2 дней
- Что делать при конфликтах во время синхронизации
- Нужно ли force-push после rebase (если синхронизация через rebase)

**Почему важно:**
Документ упоминает синхронизацию при >2 дней (Проблема 1.5), но детали отсутствуют. standard-sync.md может содержать процесс, но standard-development.md должен явно указать КОГДА вызывать синхронизацию.

**Стоит ли добавлять:** Да (добавить SSOT-ссылку + краткое описание "когда")

**Что даст:** Ясность для LLM при разработке feature-веток дольше 2 дней, предотвращение накопления конфликтов

**Критичность:** P2 (Средняя — влияет на качество merge, но не блокирует разработку)

## 5. Слепые зоны автора

### Зона 5.2: Метрики качества процесса разработки

**Суть:**
Документ описывает ЧТО делать, но не предлагает метрик для оценки процесса:
- Сколько времени в среднем занимает цикл "написание → тесты → линтинг → коммит"
- Сколько failing тестов допустимо на итерацию (в идеале 0, но на практике?)
- Как измерить покрытие тестами (coverage threshold)

**Почему важно:**
Метрики помогают LLM и разработчикам понять, когда процесс "здоровый", а когда нужна оптимизация. Например, если цикл занимает >30 минут — возможно, тесты слишком медленные и нужен рефакторинг.

**Расширение качества:** Минимальное (метрики — улучшение процесса, не критичны для базовой разработки)

**Рекомендация:** Не добавлять сейчас (over-engineering для стандарта разработки, можно вынести в отдельный документ "development-metrics.md" при необходимости)

**Критичность:** P3 (Низкая)

## 1. Двойственность формулировок

### Проблема 1.1: Неопределённое условие "спецификации зафиксированы"

**Где:** § 5 "Каскады" → "Каскад RUNNING" (строка 197)

**Цитата:**
> При подтверждении **все** документы в цепочке (от Discussion до всех Plans) одновременно переходят в RUNNING. Это точка перехода к реализации — спецификации зафиксированы.

**Как может понять LLM:**
- Интерпретация A: "Зафиксированы" = файлы не изменяются технически (read-only mode)
- Интерпретация B: "Зафиксированы" = концептуально согласованы, но файлы остаются редактируемыми
- Интерпретация C: "Зафиксированы" = commit в Git (версионирование)

**Рекомендация:**
> При подтверждении **все** документы в цепочке (от Discussion до всех Plans) одновременно переходят в RUNNING. Спецификации согласованы и переходят в режим реализации: изменения возможны только через CONFLICT-каскад (§ 6).

**Критичность:** P1 (Высокая — влияет на понимание блокировки изменений)

### Проблема 1.2: Расплывчатое "обновляются непрерывно"

**Где:** § 5 "Каскады" → "Каскадное завершение (DONE)" (строка 214)

**Цитата:**
> **Глоссарий** не участвует в каскадном завершении. Обновляется непрерывно на каждом уровне при появлении новых терминов.

**Как может понять LLM:**
- Интерпретация A: LLM автоматически обновляет глоссарий при создании любого документа
- Интерпретация B: Только при explicit-шагах воркфлоу (например, после DRAFT → WAITING)
- Интерпретация C: При каждом обнаружении нового термина (но кто обнаруживает — LLM или пользователь?)

**Рекомендация:**
> **Глоссарий** не участвует в каскадном завершении. Обновляется LLM автоматически при создании/обновлении любого документа (DRAFT/WAITING/RUNNING), если в тексте появляется термин, не определённый в `specs/glossary/`.

**Критичность:** P2 (Средняя — риск пропуска обновлений глоссария)

### Проблема 1.3: Неясная граница "документ затронут"

**Где:** § 6 "Обратная связь Code → Specs" → "Фаза 1 — Обнаружение" (строка 264)

**Цитата:**
> **Документ затронут** если хотя бы одно утверждение фактически неверно из-за изменений в коде. **Не затронут** если утверждения остаются верными (рефакторинг внутри пакета, оптимизация).

**Как может понять LLM:**
- Интерпретация A: "Утверждение фактически неверно" = любое несоответствие (даже если код работает)
- Интерпретация B: Только если утверждение описывает конкретное поведение, которое изменилось
- Интерпретация C: Проверять каждое предложение документа на соответствие коду

**Рекомендация:**
> **Документ затронут** если изменение кода нарушает описанное поведение: контракты API, data model, публичные интерфейсы, порядок взаимодействия. **Не затронут** если изменение внутреннее (реализация функции, оптимизация алгоритма) и не меняет внешнее поведение.

**Критичность:** P1 (Высокая — ключевой критерий для CONFLICT-каскада)

## 2. Риск неправильной интерпретации

### Проблема 2.1: Пропущен механизм "LLM предлагает переход"

**Где:** § 5 "Каскады" → "Каскад RUNNING" (строка 195)

**Цитата:**
> **Триггер:** LLM проверяет после каждого Plan → WAITING. Когда все Plans в WAITING — LLM предлагает переход через AskUserQuestion: "Все спецификации готовы. Перейти в RUNNING?"

**Риск:**
- LLM может проверять только при явном запросе пользователя
- Непонятно, КАК LLM проверяет (читает все Plans? есть индекс?)
- Не указано, что делать при отказе пользователя

**Рекомендация:**
> **Триггер:** LLM проверяет после каждого Plan → WAITING:
> 1. Прочитать frontmatter всех Plans в цепочке (parent Discussion → Impact → Design → ADRs → Test Specs → Plans)
> 2. Если все Plans имеют `status: WAITING` → AskUserQuestion: "Все спецификации готовы. Перейти в RUNNING?"
> 3. При подтверждении → обновить status во frontmatter всех документов цепочки (Discussion → Plans)
> 4. При отказе → остаться в WAITING (пользователь может вернуть на DRAFT любой документ)

**Критичность:** P1 (Высокая — отсутствие алгоритма проверки)

### Проблема 2.2: Не указан порядок разрешения CONFLICT

**Где:** § 6 "Обратная связь Code → Specs" → "Фаза 2 — Разрешение" (строка 290)

**Цитата:**
> Сверху вниз: начиная с самого высокого документа в CONFLICT.
>
> 1. Переписать самый высокий затронутый документ
> 2. На основе обновлённого — переписать дочерние (включая бывшие DONE)
> 3. Для DONE-документов: обновить документ **и** артефакты (живые документы, Issues)
> 4. Пользователь ревьюит → подтверждает → вся цепочка CONFLICT → RUNNING

**Риск:**
- Непонятно, как LLM определяет "самый высокий" (нет явного списка приоритетов)
- Нет инструкции для параллельных веток (например, 2 ADR в CONFLICT одновременно)
- "Переписать" — полностью заменить или частично обновить?

**Рекомендация:**
> Сверху вниз: начиная с самого высокого документа в CONFLICT.
>
> **Приоритет уровней:** Discussion > Impact > Design > ADR > Test Spec > Plan.
>
> **Шаги разрешения:**
> 1. LLM находит все документы в статусе CONFLICT
> 2. Выбирает самый высокий по приоритету (если несколько — обрабатывает по одному)
> 3. Переписывает документ: заменяет секции, затронутые CONFLICT-маркерами
> 4. Регенерирует всех дочерних (рекурсивно): новый документ + существующий код → обновлённые дочерние
> 5. Для DONE-документов: обновляет документ + `architecture/`, `tests/`, Issues
> 6. Пользователь ревьюит каждый уровень → подтверждает → уровень CONFLICT → RUNNING
> 7. Когда все CONFLICT решены → вся цепочка RUNNING

**Критичность:** P1 (Высокая — ключевой механизм разрешения конфликтов)

### Проблема 2.3: Непонятно, как проверять "все другие цепочки"

**Где:** § 6 "Обратная связь Code → Specs" → "Кросс-цепочечная обратная связь" (строка 301)

**Цитата:**
> Когда `architecture/` обновляется (каскад DONE или разрешение CONFLICT), проверить все другие цепочки, ссылающиеся на изменённые файлы.

**Риск:**
- Нет алгоритма "проверить все другие цепочки" (как найти? grep? индекс?)
- Не указано, какие файлы проверять (все в `architecture/` или только те, что изменились?)
- Непонятно, что считается "ссылкой" (frontmatter? текстовая ссылка? import?)

**Рекомендация:**
> Когда `architecture/` обновляется (каскад DONE или разрешение CONFLICT):
> 1. LLM определяет список изменённых файлов в `architecture/` (через Planned Changes или Git diff)
> 2. Для каждого изменённого файла:
>    - Grep по `specs/` на путь файла (формат: `architecture/system/{file}.md`, `architecture/services/{svc}.md`)

### 4.1. Не хватает: Механизм синхронизации frontmatter при каскадах

**Где:** § 5 "Каскады" — все подсекции

**Почему важно:** При каскадах (RUNNING, DONE, REJECTED, CONFLICT) меняется поле `status` во frontmatter. Непонятно:
- Кто обновляет frontmatter (LLM? скрипт? пользователь?)
- Какой инструмент использовать (Edit? Write? скрипт?)
- Как обрабатывать ошибки при обновлении множества файлов

**Стоит ли добавлять:** Да (P1)

**Что даст:** Устранит риск рассинхронизации статусов между документами

**Пример добавления:**
```markdown
### Обновление frontmatter при каскадах

**Инструмент:** LLM использует Edit для обновления поля `status` в frontmatter.

**Порядок обновления:**
1. Составить список всех затронутых документов
2. Для каждого документа: Edit → изменить строку `status: OLD` на `status: NEW`
3. При ошибке (файл не существует / битый frontmatter) → прервать каскад, сообщить пользователю

**Пакетное обновление:** Если затронуто > 5 файлов — использовать скрипт `.instructions/.scripts/update-specs-status.py {Discussion-path} {new-status}`.
```

## 5. Слепые зоны автора

### Зона 5.1: Версионирование и эволюция спецификаций

**Суть:** Документ описывает жизненный цикл спецификаций (DRAFT → DONE), но не покрывает:
- Как обрабатывать ситуацию "новая версия фичи" (Discussion v2)?
- Как связать старую и новую Discussion (референс? deprecated-маркер?)
- Как архивировать завершённые цепочки (DONE навсегда или можно reopened?)

**Почему важно:** Проект живёт годами. Спецификации эволюционируют. Без механизма версионирования:
- Риск создания дубликатов (Discussion 001 и Discussion 015 — одна фича)
- Невозможно отследить историю решений
- Нет связи между итерациями фичи

**Расширение качества:** Значительное

**Рекомендация:** Добавить секцию "Версионирование спецификаций" СЕЙЧАС (не over-engineering, т.к. это базовый сценарий эксплуатации)

**Пример добавления:**
```markdown
### Версионирование спецификаций

**Ситуация:** Фича реализована (цепочка Discussion → Plan → DONE). Требуется новая версия фичи.

**Решение:**
1. Создать новую Discussion с суффиксом версии: `002-oauth2-authorization-v2.md`
2. В новой Discussion добавить ссылку на предыдущую: `**Предыдущая версия:** [001-oauth2-authorization.md](./001-oauth2-authorization.md)`
3. Старая Discussion получает маркер: `**Статус:** DEPRECATED (заменена [002](./002-oauth2-authorization-v2.md))`
4. Новая цепочка создаётся с нуля (новые Impact/Design/ADR/Test Spec/Plan)

**Запрет reopened:** Документы в статусе DONE не переоткрываются. Изменения — через новую версию.
```

### Зона 5.2: Метрики и аналитика SDD

**Суть:** Документ описывает процессы, но не покрывает:
- Как измерять эффективность SDD (время от Discussion до DONE? количество CONFLICT?)
- Какие метрики собирать для улучшения процесса
- Как анализировать узкие места (например, слишком много CONFLICT на уровне ADR → проблема в Design?)

**Почему важно:** Без метрик невозможно:
- Оценить реальную пользу SDD (ускорение разработки? снижение багов?)
- Выявить проблемы процесса (частые CONFLICT = плохое Design? долгий Clarify = неясные требования?)
- Обосновать инвестиции в SDD перед командой

**Расширение качества:** Умеренное

**Рекомендация:** Отложить (не критично для MVP SDD, но добавить в Roadmap)

**Пример добавления (для будущего):**
```markdown
### Метрики SDD

**Метрики процесса:**
- Время от Discussion DRAFT до DONE (target: < 2 недели)
- Количество CONFLICT на уровне (ADR > Design — сигнал о проблемах в распределении ответственностей)
- Процент цепочек REJECTED (> 20% — сигнал о низком качестве Clarify)

**Метрики качества:**
- Количество багов после DONE (целевой критерий: < 5% от задач в Plan)
- Процент Issues с изменениями после RUNNING (< 10% — хороший показатель стабильности спецификаций)

**Сбор метрик:** Автоматический скрипт `specs/.instructions/.scripts/metrics-sdd.py` (запускается при каждом изменении статуса).
```

### Зона 5.3: Интеграция SDD с CI/CD и Git workflow

**Суть:** Документ описывает внутреннюю логику specs/, но не покрывает:
- Как интегрировать каскады с Git branches (Discussion → feature-branch?)
- Должны ли спецификации коммититься отдельно от кода?
- Как обрабатывать merge conflicts в specs/ (если 2 разработчика одновременно обновили Design)
- Как связать PR с Discussion (метаданные? frontmatter?)

**Почему важно:** В реальной разработке:
- Несколько разработчиков работают параллельно
- Спецификации и код должны быть синхронизированы в Git
- Без правил интеграции — риск конфликтов и потери данных

**Расширение качества:** Значительное

**Рекомендация:** Добавить секцию "Интеграция с Git" СЕЙЧАС (это базовый сценарий командной работы)

**Пример добавления:**
```markdown
### Интеграция SDD с Git workflow

**Branch strategy:**
- Каждая Discussion → отдельная feature-ветка: `feat/001-oauth2-authorization`
- Спецификации (specs/) и код (src/, tests/) коммитятся в одну ветку
- Переход Discussion → RUNNING = создание feature-ветки (скилл `/branch-create` использует Discussion как контекст)

**Коммиты:**
- Спецификации коммитятся отдельно от кода:
  - `docs(specs): add Discussion 001 — OAuth2 authorization`
  - `feat(auth): implement OAuth2 flow (Discussion 001)`

**Merge conflicts:**
- Спецификации конфликтуют редко (каждая Discussion = отдельные файлы)
- При конфликте: преимущество у более свежей ветки (более поздний RUNNING)
- Если конфликт критичен — создать новую Discussion для разрешения

**PR → Discussion:**
- В frontmatter PR добавить: `discussion: specs/discussion/001-oauth2-authorization.md`
- При мерже PR → автоматически обновить статусы через скрипт
```

## 1. Двойственность формулировок

### Проблема 1.1: Неопределённость границ термина "связанные документы"

**Где:** Секция "## 2. Два уровня обновления" → подсекция "Уровень 1: Связанные документы" (строки 66-76)

**Цитата:**
> Файлы, которые РЕГЛАМЕНТИРУЮТ работу с объектом:
> - `validation-{object}.md`
> - `create-{object}.md`
> - `modify-{object}.md`

**Как может понять LLM:**
- Интерпретация A: Только эти 3 типа файлов являются "связанными документами"
- Интерпретация B: Эти файлы — примеры, но могут быть и другие (например, `standard-{object}.md` сам тоже связан)
- Интерпретация C: Только файлы с такими префиксами, но для разных объектов (например, `validation-api.md`, `create-user.md` — тоже связанные)

**Рекомендация:**
> Связанные документы Уровня 1 — это три типа инструкций для ИЗМЕНЁННОГО стандарта (для того же объекта):
> - `validation-{object}.md` — валидация экземпляров этого объекта
> - `create-{object}.md` — создание экземпляров этого объекта
> - `modify-{object}.md` — модификация экземпляров этого объекта
>
> Где `{object}` — имя объекта из `standard-{object}.md`.

**Критичность:** Средняя

### Проблема 1.2: Размытое условие "должны обрабатывать новую версию"

**Где:** Секция "Каскад изменений" (строки 43-60)

**Цитата:**
> │ validation-{object}.md             │
> │ create-{object}.md                 │  ← должны обрабатывать новую версию
> │ modify-{object}.md                 │

**Как может понять LLM:**
- Интерпретация A: Обновить frontmatter `standard-version: vX.Z`
- Интерпретация B: Обновить шаги, чек-листы, примеры — чтобы соответствовали новым требованиям стандарта
- Интерпретация C: И то, и другое

**Рекомендация:**
> │ validation-{object}.md             │
> │ create-{object}.md                 │  ← обновить frontmatter (standard-version)
> │ modify-{object}.md                 │  ← обновить шаги/чек-листы/примеры

**Критичность:** Высокая (P1 — блокирует корректное выполнение миграции)

### Проблема 1.3: Неопределённое местоимение "сама"

**Где:** Секция "## 3. Порядок миграции" (строка 94)

**Цитата:**
> **Почему:** Нельзя обновлять экземпляры через modify-инструкцию, которая сама устарела.

**Как может понять LLM:**
- Интерпретация A: "сама" = modify-инструкция (та, через которую обновляем экземпляры)
- Интерпретация B: "сама" = modify-инструкция объекта, которая тоже нуждается в обновлении
- Интерпретация C: "сама" = стандарт, который устарел

**Рекомендация:**
> **Почему:** Нельзя обновлять экземпляры через `modify-{object}.md`, если эта инструкция ещё не обновлена под новую версию стандарта.

**Критичность:** Низкая

## 2. Риск неправильной интерпретации

### Проблема 2.1: Отсутствие чётких границ обновления Уровня 1

**Где:** Секция "### Уровень 1: Связанные документы" (строки 66-76)

**Цитата:**
> **Что обновлять:** Шаги, чек-листы, примеры — чтобы соответствовали новой версии стандарта.

**Риск:** LLM может обновить только шаги, но забыть про frontmatter (`standard-version`). Или обновить frontmatter, но не проверить содержание.

**Рекомендация:**
> **Что обновлять:**
> 1. Frontmatter: `standard-version: vX.Y` → `vX.Z`
> 2. Содержание: шаги, чек-листы, примеры — чтобы соответствовали новым требованиям стандарта
> 3. Валидация: каждая инструкция должна пройти проверку после обновления

**Критичность:** Высокая (P1 — риск неполной миграции)

**Рекомендация:**
> **Критерии устаревания экземпляра:**
> - Файл имеет `standard: .instructions/migration/standard-migration.md` в frontmatter
> - `standard-version` в файле < `standard-version` в стандарте
>
> **Проверка:**
> ```bash
> python .scripts/check-version-drift.py --standard standard-migration.md
> ```

**Критичность:** Высокая (P1 — LLM может пропустить часть файлов)

>
> **Критерий успеха:**
> - Все файлы Уровня 1 обновлены: ✅
> - Все критичные экземпляры Уровня 2 обновлены: ✅
> - Валидация пройдена: ✅

## 5. Слепые зоны автора

### Зона 5.1: Зависимости между стандартами (каскадная миграция)

**Суть:** Что если `standard-migration.md` ссылается на `standard-instruction.md`, и `standard-instruction.md` обновился? Нужна ли миграция `standard-migration.md`?

**Почему важно:** Стандарты не изолированы. Изменение базового стандарта (например, `standard-instruction.md`) может потребовать каскадной миграции всех зависимых стандартов.

**Расширение качества:** Значительное (выводит описание миграции на уровень всей системы, а не одного стандарта)

**Рекомендация:** Добавить сейчас

**Пример добавления:**
> ## 8. Каскадная миграция
>
> Если обновлённый стандарт является **зависимостью** для других стандартов:
>
> **Пример:**
> - `standard-instruction.md` v1.2 → v1.3 (изменилась структура frontmatter)
> - `standard-migration.md` ссылается на `standard-instruction.md` в frontmatter
>
> **Действия:**
> 1. Обновить `standard-migration.md` (Уровень 0 — зависимые стандарты)
> 2. Затем обновить Уровень 1 (`validation-migration.md`, `create-migration.md`, `modify-migration.md`)
> 3. Затем обновить Уровень 2 (экземпляры)
>
> **Автоматизация:**
> ```bash
> python .scripts/check-standard-dependencies.py --standard standard-instruction.md
> # Выводит список зависимых стандартов, требующих миграции
> ```

**Почему важно:** Реальные проекты требуют гибкости. Нельзя блокировать работу из-за миграции.

**Расширение качества:** Умеренное (покрывает граничные случаи)

**Рекомендация:** Добавить сейчас

**Пример добавления:**
> ## 9. Частичная миграция
>
> Если часть экземпляров нельзя обновить сейчас:
>
> 1. **Использовать feature flag в frontmatter:**
>    ```yaml
>    standard-version: v1.0
>    migration-planned: v1.1  # будет обновлён после слияния ветки feature-X
>    ```
> 2. **Отметить в отчёте:**
>    > ⏭️ 3 файла отложены (причина: ожидают слияния feature-X, план: 2026-02-10)

>    - Добавить TODO в issue tracker
>    - Добавить ссылку в отчёт миграции

### Зона 5.3: Валидация корректности миграции (не только version drift)

**Суть:** `check-version-drift.py` проверяет только версии в frontmatter. Но миграция может обновить версию, но не обновить СОДЕРЖАНИЕ (забыть добавить секцию).

**Почему важно:** Формальное соответствие (версия обновлена) не гарантирует семантическое соответствие (содержание соответствует стандарту).

**Расширение качества:** Значительное (КРИТИЧНО — P1)

**Рекомендация:** Добавить сейчас

**Пример добавления:**
> ## 10. Валидация корректности миграции
>
> После обновления версий:
>
> 1. **Проверить структуру:**
>    ```bash
>    python .scripts/validate-structure.py --file {экземпляр} --standard {стандарт}
>    # Проверяет наличие обязательных секций из стандарта
>    ```
> 2. **Проверить семантическую корректность:**
>    - Вызвать captain-holt для анализа миграции (если стандарт изменил критичные секции)
> 3. **Если обнаружены расхождения:**
>    - Исправить содержание экземпляра
>    - Повторить валидацию

### Зона 5.4: Миграция скриптов и скиллов при изменении стандарта

**Суть:** Стандарт описывает объект. Но для работы с объектом есть скрипты и скиллы. Если стандарт изменился — скрипты и скиллы тоже могут требовать обновления.

**Почему важно:** Документ описывает миграцию документов, но не упоминает код (скрипты, скиллы). Это критичный пробел.

**Расширение качества:** Значительное (КРИТИЧНО — P1)

**Рекомендация:** Добавить сейчас

**Пример добавления:**
> ## 11. Миграция скриптов и скиллов
>
> При изменении стандарта проверить:
>
> **1. Скрипты в `.instructions/.scripts/`:**
> - Если скрипт парсит структуру объекта (например, `validate-api.py` проверяет секции API) — обновить логику парсинга
> - Если скрипт создаёт объект (например, `create-api.py` генерирует файл) — обновить шаблон
>
> **2. Скиллы в `/.claude/skills/`:**
> - Если скилл ссылается на изменённую инструкцию в SSOT — обновить промпт скилла
> - Если скилл использует старые термины/секции — обновить формулировки
>
> **Воркфлоу:**
> 1. Найти скрипты: `grep -r "standard-{object}.md" .instructions/.scripts/`
> 2. Найти скиллы: `grep -r "standard-{object}.md" .claude/skills/`
> 3. Обновить через `/script-modify` и `/skill-modify`

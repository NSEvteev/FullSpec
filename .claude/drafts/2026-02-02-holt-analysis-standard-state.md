## 2. Риск неправильной интерпретации LLM

### Проблема 2.1: Неявная граница "6 проверок" и timeout

**Где:** Секция "locks.json" (строка 189)

**Цитата:**
> Если после 6 проверок блокировка не снята → вернуть ошибку

**Как может понять LLM:**
- Интерпретация A: Ошибка возвращается после 6-й проверки (время: 25 секунд)
- Интерпретация B: Ошибка возвращается через 30 секунд от первой проверки (независимо от количества проверок)
- Интерпретация C: Агент должен проверять до 6 раз, затем прекратить и вернуть ошибку

**Риск:**
LLM может реализовать проверку без учёта времени (только счётчик попыток) или без учёта количества попыток (только таймер).

**Рекомендация:**
> Алгоритм: сделать 6 проверок с интервалом 5 секунд (0s, 5s, 10s, 15s, 20s, 25s). Если после 6-й проверки (через 25 секунд от первой) файл всё ещё заблокирован → вернуть ошибку. Суммарное время ожидания: до 25 секунд.

**Критичность:** Высокая

---

### Проблема 2.2: Противоречие в требовании блокировки для `locks.json`

**Где:** Секция "locks.json" (строки 196, 290)

**Цитата (строка 196):**
> Примечание: Сам `locks.json` не требует блокировки — каждый агент добавляет/удаляет только свои записи.

**Цитата (строка 290):**
> **Примечание:** Сам `locks.json` не требует блокировки — каждый агент управляет только своими записями.

**Как может понять LLM:**
- Интерпретация A: Блокировка не нужна вообще (запись напрямую без проверок)
- Интерпретация B: Блокировка нужна, но на уровне ОС (file locking)
- Интерпретация C: Агент должен использовать атомарную запись (read-modify-write как одна операция)

**Риск:**
LLM может реализовать прямую запись в файл без чтения текущего состояния, что приведёт к потере блокировок других агентов.

**Рекомендация:**
> Сам `locks.json` не требует блокировки через свой механизм (рекурсия невозможна), но агент ДОЛЖЕН использовать атомарную операцию: прочитать весь `locks.json`, добавить/удалить свою запись в структуре, записать весь JSON обратно. Это гарантирует сохранение блокировок других агентов.

**Критичность:** Высокая

### Проблема 2.3: Неявное условие создания `agent-{name}-operation.json`

**Где:** Секция "Структура папки" (строка 89)

**Цитата:**
> `agent-{name}-operation.json` — Лог операций агента (создаётся при первом запуске)

**Как может понять LLM:**
- Интерпретация A: Создаётся при первом запуске агента ВООБЩЕ (один раз, потом переиспользуется)
- Интерпретация B: Создаётся при каждом запуске агента (перезаписывает предыдущий)
- Интерпретация C: Создаётся только если не существует (логи накапливаются между запусками)

**Риск:**
LLM может накапливать логи разных запусков в одном файле или перезаписывать логи при каждом запуске.

**Рекомендация:**
> Файл `agent-{name}-operation.json` создаётся при каждом запуске агента с новым содержимым (перезапись). Если требуется сохранить историю прошлых запусков — агент должен переименовать старый файл в `agent-{name}-operation-{timestamp}.json` перед созданием нового.

**Критичность:** Средняя

### Проблема 2.4: Отсутствие обработки ошибок записи в state

**Где:** Секция "Правило блокировок" (строки 292-303)

**Цитата:**
> 5. Выполнить операцию
>    - При любой ошибке (запись, валидация, внутренняя) — перейти к шагу 7

**Как может понять LLM:**
- Интерпретация A: Если запись в файл проекта упала — снять блокировку и вернуть ошибку
- Интерпретация B: Если запись в state-файл (`operations[]`) упала — снять блокировку и продолжить (лог не критичен)
- Интерпретация C: Если запись в `locks.json` упала — попробовать снова (retry)

**Риск:**
LLM может не обработать ошибку записи в state или заблокировать файл навсегда при падении.

**Рекомендация:**
> При ошибке записи в файл проекта: снять блокировку из `locks.json` (ОБЯЗАТЕЛЬНО) → вернуть ошибку пользователю. При ошибке записи в `agent-{name}-operation.json` (лог): продолжить выполнение (лог не блокирует работу). При ошибке снятия блокировки: сообщить ошибку, но не блокировать продолжение — хук `SubagentStop` очистит блокировку автоматически.

**Критичность:** Высокая

ЗАПРЕЩЕНО создавать эти файлы автоматически — это может скрыть проблему неправильной инициализации проекта.

### Проблема 2.6: Неопределённость в обновлении `working_on`

**Где:** Секция "agents-status.json" (строка 138)

**Цитата:**
> `working_on` — array — Да — Список файлов, с которыми работает

## 4. Полнота описания объекта

### 4.1. Не хватает: Обработка race conditions

**Почему важно:** Два агента могут одновременно читать `locks.json`, оба увидят файл свободным, оба запишут блокировку. Один перезапишет блокировку другого.

**Стоит ли добавлять:** Да

**Что даст:** Гарантия атомарности операций с state

**Рекомендуемое добавление:**
> **Секция "Race conditions"**: При работе с `locks.json` и `agents-status.json` используйте паттерн read-modify-write как атомарную операцию. В Python: используйте `fcntl.flock()` для file locking. В Node.js: используйте `proper-lockfile`. Если file locking недоступен — добавьте retry-логику с exponential backoff при конфликтах записи (detect через проверку mtime файла перед записью).

---

### 4.2. Не хватает: Примеры кода для работы с state

**Почему важно:** LLM может неправильно реализовать атомарную запись или обработку блокировок

**Стоит ли добавлять:** Да

**Что даст:** Единообразие реализации, снижение риска ошибок

**Рекомендуемое добавление:**
> **Секция "Примеры кода"**: Добавить примеры на Python для:
> 1. Чтение и обновление `agents-status.json`
> 2. Добавление блокировки в `locks.json`
> 3. Снятие блокировки с retry при ошибке
> 4. Добавление записи в `agent-{name}-operation.json`

---

### 4.3. Не хватает: Валидация структуры state при чтении

**Почему важно:** Повреждённый state может привести к падению агента или некорректной работе

**Стоит ли добавлять:** Да

**Что даст:** Раннее обнаружение проблем, graceful degradation

**Рекомендуемое добавление:**
> **Секция "Валидация state"**: Перед использованием `locks.json` или `agents-status.json` проверить:
> 1. Файл существует
> 2. JSON валиден
> 3. Структура соответствует ожидаемой (наличие ключей `locks`/`agents`)
> 4. При ошибке — использовать восстановление (см. "Восстановление после ошибок")

---

### 4.4. Не хватает: Сценарий cleanup при аварийном завершении

**Почему важно:** Хук `SubagentStop` может не отработать при kill -9 или crash

**Стоит ли добавлять:** Да

**Что даст:** Защита от "забытых" блокировок при аварийном завершении

**Рекомендуемое добавление:**
> **Секция "Аварийное завершение"**: Если агент завершился аварийно (crash, kill -9) и хук `SubagentStop` не отработал — блокировки останутся в `locks.json`. Решения:
> 1. Ручной cleanup: команда `python ./.claude/scripts/cleanup-locks.py {agent-name}`
> 2. Автоматический cleanup при старте нового агента: проверить `locked_at` — если блокировка старше 5 минут → удалить (считается "забытой")
> 3. Периодический cron-скрипт для очистки старых блокировок

## 5. Слепые зоны автора

### Зона 5.1: Масштабирование при большом количестве агентов

**Суть:** При 100+ параллельных агентах файлы `agents-status.json` и `locks.json` могут стать узким местом (частые чтения/записи, конфликты)

**Почему важно:** Проект может расти, добавляться параллельные воркфлоу

**Расширение качества:** Умеренное (актуально для крупных проектов)

**Рекомендация:** Отложить (текущий подход достаточен для <20 агентов)

**Пример добавления:**
> **Секция "Масштабирование"**: При >20 параллельных агентах рекомендуется:
> 1. Разделить `locks.json` по областям: `locks-instructions.json`, `locks-scripts.json`
> 2. Использовать внешнюю БД для state (Redis, SQLite)
> 3. Добавить механизм heartbeat для обнаружения "мёртвых" агентов

---

### Зона 5.2: Версионирование state при изменении структуры

**Суть:** Если структура state изменится (добавятся новые поля), старые агенты могут сломаться

**Расширение качества:** Значительное

**Рекомендация:** Добавить сейчас

**Пример добавления:**
> **Секция "Версионирование state"**: Каждый state-файл ДОЛЖЕН содержать поле `version`:
> ```json
> {
>   "version": "1.0",
>   "agents": {...}
> }
> ```
> При чтении state агент проверяет версию. Если версия не совпадает с ожидаемой → вернуть ошибку: "State version mismatch. Expected: 1.0, got: {version}. Обновите агента или выполните миграцию state."

## 6. Другие проблемы

### Проблема 6.1: Неконсистентность в именовании полей

**Где:** Секция "agents-status.json" (строка 138) vs Хуки (строка 412)

**Цитата (строка 138):**
> `working_on` — array — Да — Список файлов, с которыми работает

**Цитата (строка 412):**
> - `agent_id` — уникальный ID субагента
> - `agent_type` — тип агента

**Проблема:**
В `agents-status.json` используется snake_case (`started_at`, `finished_at`, `working_on`), но в хуках используется `agent_id` (snake_case) и `agent_type` (snake_case). Нет единого стиля именования.

**Рекомендация:**
> Использовать snake_case для всех полей JSON во всех state-файлах: `started_at`, `finished_at`, `working_on`, `locked_at`, `agent_id`, `agent_type`.

**Критичность:** Низкая

---

### Проблема 6.2: Отсутствие явного требования ISO 8601 формата для timestamp

**Где:** Секция "agents-status.json" (строки 135-136)

**Цитата:**
> `started_at` — string — Да — ISO 8601 — время начала
> `finished_at` — string — Нет — ISO 8601 — время завершения (null если running)

**Проблема:**
Нет примера формата ISO 8601 (с timezone или без? microseconds или нет?). LLM может использовать разные форматы в разных местах.

**Рекомендация:**
> Формат timestamp: ISO 8601 с timezone UTC и microseconds. Пример: `"2026-01-31T10:30:15.123456Z"`. Используйте `datetime.datetime.now(datetime.timezone.utc).isoformat()` в Python или `new Date().toISOString()` в JavaScript.

**Критичность:** Высокая

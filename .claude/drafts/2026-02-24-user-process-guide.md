Руководство для разработчика: как правильно взаимодействовать с Claude Code на каждом этапе поставки ценности. Перевод standard-process.md из "что делает система" в "что делает ты".

## Контекст

**Задача:** Создать user-facing процесс из standard-process.md, чтобы разработчик знал, когда и как взаимодействовать с Claude
**Почему создан:** `standard-process.md` описывает **систему** (что происходит), но не описывает **действия пользователя** (что делать). Разработчику нужен чёткий гайд: "на этом шаге ты делаешь X, потом ждёшь Y, потом подтверждаешь Z"
**Связанные файлы:**
- `specs/.instructions/standard-process.md` — оркестратор процесса (системный)
- `CLAUDE.md` — точка входа для Claude
- `.claude/onboarding.md` — onboarding для новых участников

## Содержание

### Проблема

`standard-process.md` написан для **Claude** — он описывает фазы, статусы, скиллы, SSOT-ссылки. Разработчик-человек читает его и видит:
- Фаза 1: discussion-create, design-create... — "а что мне-то делать?"
- CONFLICT → WAITING → RUNNING — "а когда мне писать код?"
- `/dev-create`, `/review` — "в каком порядке? когда подтверждать?"

Нужен документ "от лица пользователя": пошаговая инструкция с точками взаимодействия.

### Формат

Для каждой фазы:
1. **Что ты делаешь** — конкретные действия пользователя
2. **Что делает Claude** — что происходит автоматически
3. **Точки решения** — где Claude спросит, и что отвечать
4. **Результат** — что получишь на выходе

### Черновик структуры

```
## Шаг 1: У тебя есть идея
  → Расскажи Claude что хочешь сделать
  → Claude создаст Discussion (спросит уточняющие вопросы)
  → Ты отвечаешь на вопросы
  → Claude покажет документ — ты подтверждаешь или просишь изменить

## Шаг 2: Claude проектирует решение
  → Claude создаст Design, Plan Tests, Plan Dev
  → На каждом шаге — покажет документ, ты подтверждаешь
  → Это может занять несколько итераций

## Шаг 3: Запуск разработки
  → Claude создаст Issues на GitHub и ветку
  → Ты проверяешь на GitHub что всё ок

## Шаг 4: Разработка
  → Claude пишет код по задачам
  → Ты можешь следить за прогрессом
  → Если что-то не так — говоришь Claude, он откатит спецификации (CONFLICT)

## Шаг 5: Доставка
  → Claude сделает ревью, создаст PR
  → Ты проверяешь PR на GitHub
  → Claude мержит

## Шаг 6: Релиз
  → Claude создаст GitHub Release
  → Ты проверяешь
```

### Где размещать

| Вариант | Путь | Плюсы | Минусы |
|---------|------|-------|--------|
| A: В CLAUDE.md | `CLAUDE.md` | Видно сразу | CLAUDE.md станет большим |
| B: В onboarding.md | `.claude/onboarding.md` | Логичное место для новичков | Не виден при каждом запуске |
| C: Отдельный файл | `.claude/user-guide.md` или `docs/.system/workflow.md` | Чистое разделение | Ещё один файл |
| D: В standard-process.md | Секция "Для разработчика" | Всё в одном месте | Смешение системного и user-facing |

### Выводы по текущему состоянию

1. **Разрыв аудиторий:** Документы для человека (README.md, initialization.md) описывают setup, но не процесс. Документы процесса (standard-process.md, standard-github-workflow.md) написаны для Claude.

2. **Нет единого пути:** Человек, впервые работающий с этим проектом через Claude Code, должен сам собирать картину из 8+ документов. Нет документа "читай только это, и ты поймёшь весь процесс".

3. **Quick Reference (§9) — ближайший кандидат:** Секция §9 из standard-process.md по формату близка к тому, что нужно, но:
   - Написана для Claude (список команд без объяснений)
   - Не содержит точек решения ("когда Claude спросит — ответь X")
   - Не объясняет, что пользователь увидит на каждом шаге
   - Не описывает "что если что-то пошло не так"

4. **initialization.md — эталон формата:** Этот документ хорошо написан для человека: пошагово, с командами, с решением проблем, с проверкой. Но покрывает только Фазу 0.

5. **Onboarding — философия, не workflow:** Описывает принципы (SSOT, скиллы вместо команд, миграция), но не описывает последовательность действий при реальной работе.

## Best practices

### 1. Progressive Disclosure — от простого к сложному

**Принцип:** Не показывать всё сразу. Начинать с минимального набора информации, раскрывать детали по мере необходимости (Jakob Nielsen, 1995).

**Применение к User Process Guide:**

| Уровень | Что показывать | Формат |
|---------|---------------|--------|
| **Level 0: Cheat Sheet** | 6 шагов × 1 предложение. Одна страница. Для тех, кто уже знает процесс | Односторонняя памятка (карточка) |
| **Level 1: Happy Path** | 6 шагов с точками решения и ожидаемыми результатами. "Что ты делаешь → Что делает Claude → Что ты видишь" | Основной документ — user-guide.md |
| **Level 2: Troubleshooting** | Что делать когда: CONFLICT, ошибка pre-commit, CI не прошёл, нужен откат | Секция "Если что-то пошло не так" внутри каждого шага или отдельная секция |
| **Level 3: Deep Dive** | Ссылки на SSOT-документы для каждого шага — standard-process.md, standard-development.md, и т.д. | Ссылки "Подробнее:" в конце каждого шага |

**Рекомендация:** Не более 2 уровней раскрытия. Исследования показывают, что 3+ уровня снижают usability — пользователь теряется между уровнями. Лучше: Level 0 (памятка) + Level 1 (основной гайд) со ссылками на Level 3 (SSOT).

**Антипаттерн:** standard-process.md пытается быть одновременно оркестратором для Claude (Level 3) и Quick Reference для человека (Level 0 в §9). Это нарушает progressive disclosure — пользователь видит §9, но для контекста вынужден читать §§1-8.

### 2. Runbook-формат для каждого шага

**Принцип:** Операционная процедура должна быть самодостаточной — содержать всё необходимое для выполнения без обращения к другим документам (AWS Well-Architected, SkeltonThatcher).

**Шаблон шага User Process Guide:**

```markdown
## Шаг N: Название (глагол — что ты делаешь)

**Ты:** [конкретное действие пользователя — 1 предложение]
**Claude:** [что происходит автоматически — 1-2 предложения]

### Что ты увидишь
[Описание или пример вывода Claude — что появится в чате]

### Точки решения
- Claude спросит: "..." → Отвечай: [варианты]
- Если не устраивает результат → Скажи: "..."

### Результат
[Что получишь на выходе — файл, GitHub Issue, PR, и т.д.]

### Если что-то пошло не так
- [Проблема] → [Решение]

> **Подробнее:** [ссылка на SSOT]
```

**Почему runbook-формат:** initialization.md в проекте уже использует этот паттерн (пошагово, с командами, с решением проблем). User Process Guide — расширение этого подхода на весь процесс.

### 3. Human-AI Collaboration: когда решаешь ты, когда делегируешь

**Принцип:** Лучшие команды используют прозрачное планирование с человеческими контрольными точками — AI выполняет, человек контролирует направление, качество и архитектурные решения.

**Матрица ролей для User Process Guide:**

| Тип действия | Кто делает | Примеры в процессе |
|--------------|-----------|-------------------|
| **Подтверждение** (approve/reject) | Человек | Одобрение Discussion, Design, Plan Tests, Plan Dev; одобрение PR; решение о релизе |
| **Выбор** (decide) | Человек | Выбор направления при CONFLICT; приоритизация задач; решение "мержить или нет" |
| **Инициация** (trigger) | Человек | "Хочу добавить OAuth2"; "Сделай релиз"; "Откати изменения" |
| **Генерация** (create/write) | Claude | Создание документов, код, тесты, PR body, review comments |
| **Валидация** (check) | Claude + pre-commit | Запуск тестов, линтинг, проверка формата, pre-commit hooks |
| **Оркестрация** (orchestrate) | Claude | Выбор следующего шага, вызов скиллов, управление статусами |

**Ключевой паттерн:** Claude НИКОГДА не принимает решение за человека. В каждой фазе есть "блокирующее подтверждение" — Claude останавливается и ждёт ответа. Это уже заложено в architecture (rule core.md: "Если в воркфлоу скилла есть шаг с подтверждением — остановиться и ждать").

**Рекомендация для User Guide:** Явно маркировать каждую точку решения иконкой или меткой:
- `[ПОДТВЕРДИ]` — Claude ждёт "ок" или правок
- `[ВЫБЕРИ]` — Claude предложит варианты, ты выбираешь
- `[СКАЖИ]` — Claude ждёт от тебя инициативу (описание задачи, команду)

### 4. First-Time User Experience (FTUE)

**Принцип:** Первый опыт определяет восприятие всей системы. Time to First Success (TTFS) — метрика, показывающая сколько времени нужно новому пользователю до первого результата. Цель — минимизировать TTFS.

**Применение:**

| Метрика | Текущее состояние | Цель |
|---------|------------------|------|
| **Time to First Hello World** | Неизвестно — нет единого пути | Один документ → первый `make dev` за 10 минут |
| **Time to First Contribution** | Человек должен прочитать 5+ документов | Один документ → первая идея → Discussion за 20 минут |
| **Time to First Full Cycle** | Не измеряется | Один документ → полный цикл Idea → Merge за первый день |

**Рекомендация:** Начать User Guide с секции "Быстрый старт — первые 5 минут", включающей:
1. Установка: `make setup` (ссылка на initialization.md)
2. Первая идея: "Расскажи Claude что хочешь сделать"
3. Что произойдёт дальше: краткое описание 6 шагов

### 5. Consistency и Predictability

**Принцип:** Непоследовательность — убийца cognitive load. Разработчик должен знать: "на каждом шаге процесс одинаковый: Claude делает → показывает → я проверяю → подтверждаю или прошу изменить".

**Применение:** Каждый из 4 документов аналитической цепочки (Discussion, Design, Plan Tests, Plan Dev) проходит один и тот же паттерн из 7 шагов (PREPARE → CLARIFY → GENERATE → VALIDATE → AGENT REVIEW → USER REVIEW → REPORT). С точки зрения пользователя, это выглядит так:

```
1. Claude задаёт уточняющие вопросы → ты отвечаешь
2. Claude генерирует документ → ты читаешь
3. Claude показывает финальную версию → ты подтверждаешь или просишь изменить
4. Повтор шагов 1-3 до удовлетворения
```

**Рекомендация:** В User Guide описать этот паттерн один раз в начале, а потом для каждого из 4 документов указывать только специфику (какие вопросы задаёт, какие секции содержит документ, на что обращать внимание при ревью).

### 6. Error Recovery и "Что если"

**Принцип:** Runbook без секции troubleshooting — неполный runbook. AWS рекомендует: "Include checks before and after the procedure to ensure everything is working correctly".

**Типичные ситуации для User Guide:**

| Ситуация | Что видит пользователь | Что делать |
|----------|----------------------|------------|
| Pre-commit hooks упали | Красные ошибки в терминале при `git commit` | Прочитать ошибку, исправить, повторить коммит |
| Claude показал документ с ошибками | Документ выглядит неверно | Сказать Claude что не так, он перегенерирует |
| CONFLICT при разработке | Claude сообщает о несовместимости с проектированием | Claude откатит статусы, исправит документы, попросит подтвердить каждый |
| CI не прошёл | GitHub показывает красный X на PR | Прочитать логи CI, исправить, запушить |
| Claude "завис" или не понимает | Нет ответа или ответ не по теме | Сформулировать задачу конкретнее, указать на нужный скилл |
| Нужно откатить всё | Решили отменить фичу | Сказать Claude: "Откати цепочку {NNNN}" → Claude переведёт в ROLLING_BACK → REJECTED |

### 7. Примеры диалогов как обучающий инструмент

**Принцип:** Interactive tutorials и guided workflows — самый эффективный способ обучения для инструментов, требующих взаимодействия (GitHub Copilot research, 2025-2026). Пример реального диалога лучше абстрактного описания.

**Рекомендация:** Включить в User Guide хотя бы 1 сквозной пример:

```markdown
### Пример: добавляем OAuth2 авторизацию

**Ты:** Хочу добавить OAuth2 авторизацию через Google для пользователей

**Claude:** Создаю Discussion... [задаёт 3 уточняющих вопроса]
1. Какие OAuth2 провайдеры нужны кроме Google?
2. Нужна ли авторизация через email/пароль тоже?
3. Какие данные пользователя хранить?

**Ты:** Только Google. Email/пароль тоже. Хранить: email, имя, аватар.

**Claude:** Создал Discussion → показывает документ → [ПОДТВЕРДИ]

**Ты:** Ок, всё верно.

[... и так далее через все 6 шагов]
```

### 8. Формат размещения — рекомендация

**Анализ вариантов:**

| Вариант | Оценка | Обоснование |
|---------|--------|-------------|
| A: В CLAUDE.md | Плохо | CLAUDE.md — точка входа для Claude, не для человека. Станет слишком большим |
| B: В onboarding.md | Средне | Логичное место, но onboarding.md — про философию и "первые шаги", а не про полный workflow. Смешение |
| C: Отдельный файл | Хорошо | Чистое разделение. Ссылка из CLAUDE.md, onboarding.md, README.md. По паттерну progressive disclosure — основной документ Level 1 |
| D: В standard-process.md | Плохо | Нарушает принцип разделения аудиторий. standard-process.md — для Claude |

**Рекомендация: Вариант C** — отдельный файл `/.claude/user-guide.md` или `/docs/.system/developer-workflow.md`.

Аргументы:
- По аналогии с initialization.md, который живёт отдельно в `/.structure/` и является self-contained user-facing документом
- Ссылка размещается в CLAUDE.md (секция Quick Start), onboarding.md (секция "Первые шаги"), README.md (секция "Документация")
- Level 0 cheat sheet может быть секцией внутри этого же файла (в конце)
- Не нарушает SSOT — это не стандарт, а user-facing документ (исключение из SSOT, как CLAUDE.md и README.md)

### 10. Чек-лист для самопроверки User Guide

Перед финализацией документа проверить:

- [ ] Каждый шаг содержит: действие пользователя, действие Claude, точки решения, результат
- [ ] Ни один шаг не требует чтения standard-process.md для понимания
- [ ] Все технические термины (CONFLICT, WAITING, RUNNING) объяснены простым языком
- [ ] Есть секция "Быстрый старт — первые 5 минут"
- [ ] Есть хотя бы 1 сквозной пример диалога
- [ ] Есть секция "Если что-то пошло не так" (troubleshooting)
- [ ] Есть Level 0 cheat sheet (одностраничная памятка)
- [ ] Ссылки на SSOT — только как "подробнее", не как обязательное чтение
- [ ] Документ может быть прочитан за 10-15 минут
- [ ] Пользователь после прочтения знает: с чего начать, что будет происходить, когда его спросят, что он получит

## 1. Двойственность формулировок

### Проблема 1.1: Неопределённое условие "длительная разработка"

**Где:** Секция "2. Когда синхронизировать" → таблица "Обязательные триггеры" (строка 59)

**Цитата:**
> При длительной разработке (>2 дней) | Feature-ветку с main | Предотвращение накопления конфликтов

**Как может понять LLM:**
- Интерпретация A: Ровно через 2 дня 0 часов после создания ветки
- Интерпретация B: Если суммарное календарное время с момента создания ветки превысило 48 часов
- Интерпретация C: Если активная разработка (коммиты) длится более 2 рабочих дней
- Интерпретация D: Каждые 2 дня проверять и синхронизировать (повторяющееся действие)

**Рекомендация:**
> При работе в feature-ветке более 2 календарных дней с момента создания ветки — синхронизировать feature-ветку с main через rebase. Проверять необходимость синхронизации ежедневно после 2-дневного порога.

**Критичность:** P1 (Высокая) — блокирует корректное выполнение синхронизации

### Проблема 1.2: Неявное условие "при push feature-ветки"

**Где:** Секция "2. Когда синхронизировать" → таблица "Обязательные триггеры" (строка 60)

**Цитата:**
> Перед push feature-ветки | Feature-ветку с main | Проверка совместимости перед созданием PR

**Как может понять LLM:**
- Интерпретация A: Перед каждым `git push` (включая первый push ветки)
- Интерпретация B: Только перед первым push ветки в origin
- Интерпретация C: Только перед финальным push перед созданием PR
- Интерпретация D: Перед каждым push, если с момента последней синхронизации main изменился

**Рекомендация:**
> Перед финальным push feature-ветки (перед созданием PR) — синхронизировать feature-ветку с актуальной main. Проверить наличие изменений в main командой `git fetch origin && git log HEAD..origin/main`. Если изменений нет — синхронизация не требуется.

**Критичность:** P1 (Высокая) — влияет на понимание триггера

### Проблема 1.3: Двусмысленность термина "конфликтов не возникнет"

**Где:** Секция "3. Процесс синхронизации" → подсекция "Синхронизация main" (строка 85)

**Цитата:**
> `git pull` = `git fetch` + `git merge`. Для main это безопасно, т.к. прямые коммиты в main запрещены — конфликтов не возникнет.

**Как может понять LLM:**
- Интерпретация A: Конфликтов никогда не может возникнуть при pull main
- Интерпретация B: Конфликтов не возникнет, если локальная main не модифицирована (что гарантируется запретом коммитов)
- Интерпретация C: Конфликтов не возникнет при правильном workflow, но они технически возможны

**Рекомендация:**
> `git pull` = `git fetch` + `git merge`. Для main это безопасно: прямые коммиты в main запрещены, поэтому локальная main всегда идентична предыдущему состоянию origin/main. При pull происходит fast-forward merge без конфликтов.

**Критичность:** P2 (Средняя) — влияет на понимание механизма

### Проблема 1.4: Неоднозначность критерия "личная ветка"

**Где:** Секция "5. Запреты и ограничения" (строка 167)

**Цитата:**
> Force push после rebase — допустим ТОЛЬКО для personal feature-ветки

**Как может понять LLM:**
- Интерпретация A: Ветка, над которой работает только один разработчик (технически)
- Интерпретация B: Ветка, которую не просматривает никто другой в данный момент
- Интерпретация C: Ветка до создания PR (после PR она становится "публичной")

**Рекомендация:**
> Force push после rebase — допустим ТОЛЬКО для feature-ветки ДО создания Pull Request. После создания PR ветка становится общедоступной — force push запрещён (может сломать review-процесс).

**Критичность:** P2 (Средняя) — влияет на понимание ограничений

## 2. Риск неправильной интерпретации LLM

### Проблема 2.1: Отсутствие алгоритма определения "актуальности" main

**Где:** Секция "2. Когда синхронизировать" → таблица "Обязательные триггеры" (строка 57)

**Цитата:**
> Перед созданием ветки | `main` с remote | Ветка должна начинаться от актуальной main

**Как LLM может понять неправильно:**
- LLM может выполнить `git checkout main` без проверки, предполагая что main уже актуальна
- LLM может не понять, что "актуальная main" = "main после git pull origin main"

**Рекомендация:**
Добавить явный алгоритм проверки актуальности:

```markdown
Перед созданием ветки | `main` с remote | Ветка должна начинаться от актуальной main

**Проверка актуальности:**
1. `git fetch origin`
2. Сравнить: `git rev-parse main` vs `git rev-parse origin/main`
3. Если SHA различаются → выполнить `git pull origin main`
4. Только после синхронизации создавать ветку
```

**Критичность:** P1 (Высокая) — риск создания ветки от устаревшей main

### Проблема 2.2: Неявная зависимость от состояния репозитория

**Где:** Секция "3. Процесс синхронизации" → подсекция "Синхронизация main" (строки 76-82)

**Цитата:**
```bash
# 1. Переключиться на main и обновить
git checkout main
git pull origin main

# 2. Проверить статус
git status
```

**Как LLM может понять неправильно:**
- LLM может не проверить, есть ли незакоммиченные изменения перед checkout
- LLM может не обработать ошибку "error: Your local changes would be overwritten by checkout"
- LLM может не понять, что делать, если `git checkout main` провалился

**Рекомендация:**
Добавить предварительную проверку:

```markdown
### Синхронизация main

**Предусловие:** Убедиться, что нет незакоммиченных изменений:
```bash
git status --porcelain
```
Если вывод не пустой — закоммитить изменения или использовать `git stash`.

**Шаги:**
```bash
# 1. Переключиться на main и обновить
git checkout main
git pull origin main

# 2. Проверить статус
git status
# Вывод: On branch main, Your branch is up to date with 'origin/main'
```
```

**Критичность:** P1 (Высокая) — риск потери незакоммиченных изменений

### Проблема 2.3: Неполное описание процесса rebase

**Где:** Секция "3. Процесс синхронизации" → подсекция "Синхронизация feature-ветки с main" (строки 91-99)

**Цитата:**
```bash
# 1. Обновить main
git checkout main
git pull origin main

# 2. Вернуться в feature-ветку и выполнить rebase
git checkout feature/auth-42-43
git rebase main
```

**Как LLM может понять неправильно:**
- LLM может не понять, что делать, если rebase остановился с конфликтом
- LLM может не заметить, что после rebase нужен force push
- LLM может пропустить уведомление о том, что команда выполняется на нужной ветке

**Рекомендация:**
Дополнить описание полного цикла:

```markdown
### Синхронизация feature-ветки с main

**Шаги:**
```bash
# 1. Обновить main
git checkout main
git pull origin main

# 2. Вернуться в feature-ветку
git checkout feature/auth-42-43

# 3. Выполнить rebase
git rebase main

# Если возникли конфликты → см. секцию "4. Разрешение конфликтов"
# Если rebase прошёл успешно:

# 4. Обновить remote-ветку (требуется force push)
git push --force-with-lease origin feature/auth-42-43
```

**Важно:** После rebase история feature-ветки изменяется — требуется force push.
```

**Критичность:** P1 (Высокая) — риск незавершённого rebase или потери изменений

### Проблема 2.4: Противоречие между секциями о конфликтах

**Где:** Секция "3. Процесс синхронизации" → подсекция "Синхронизация main" (строка 85) vs Секция "4. Разрешение конфликтов" (строка 123)

**Цитата 1:**
> Для main это безопасно, т.к. прямые коммиты в main запрещены — конфликтов не возникнет.

**Цитата 2:**
> Конфликты возможны ТОЛЬКО при rebase feature-ветки на обновлённую main. При синхронизации самой main конфликтов не бывает (прямые коммиты запрещены).

**Как LLM может понять неправильно:**
- LLM может подумать, что информация дублируется, но не понять связь между утверждениями
- LLM может не заметить, что оба утверждения говорят об одном (но второе — более явно)

**Рекомендация:**
В секции 3 оставить краткую формулировку, в секции 4 добавить отсылку:

Секция 3:
> `git pull` на main — это fast-forward merge (подробнее о конфликтах → § 4).

Секция 4:
> Конфликты возможны ТОЛЬКО при rebase feature-ветки на обновлённую main. При синхронизации main конфликтов не возникает (прямые коммиты запрещены — см. § 3).

**Критичность:** P2 (Средняя) — снижает ясность через дублирование

### Проблема 2.5: Неявное ограничение на количество попыток разрешения конфликтов

**Где:** Секция "4. Разрешение конфликтов" → подсекция "Процесс разрешения" (строки 127-146)

**Цитата:**
```bash
# 4. Отметить как разрешённый и продолжить
git add src/auth/handler.js
git rebase --continue

# 5. Если конфликт неразрешим — отменить rebase
git rebase --abort
```

**Как LLM может понять неправильно:**
- LLM может не понять, что `git rebase --continue` может снова остановиться на следующем конфликте
- LLM может подумать, что после `git rebase --continue` процесс всегда завершается
- LLM может не понять, нужно ли повторять шаги 3-4 для каждого файла с конфликтом

**Рекомендация:**
Уточнить цикличность процесса:

```markdown
# 4. Отметить как разрешённый и продолжить
git add src/auth/handler.js
git rebase --continue

# Если rebase остановился на следующем конфликте — повторить шаги 2-4
# Rebase завершён, когда команда `git rebase --continue` не выдаёт новых конфликтов

# 5. Если конфликт неразрешим — отменить весь rebase
git rebase --abort
```

**Критичность:** P2 (Средняя) — риск неполного разрешения конфликтов

```markdown
**Обработка ошибок:**

Если `git pull origin main` провалился:

| Ошибка | Причина | Решение |
|--------|---------|---------|

Метрики помогут выявить проблемные практики команды.

**Рекомендация:** Не нужно (over-engineering)

### Зона 5.3: Синхронизация при работе с fork'ами (open-source проекты)

**Суть:**
Документ предполагает работу с одним origin. В open-source проектах разработчики работают с fork'ами:
- upstream — основной репозиторий
- origin — личный fork

Процесс синхронизации отличается:
```bash
git remote add upstream https://github.com/original/repo.git
git fetch upstream
git checkout main
git merge upstream/main
git push origin main
```

**Почему важно:**
Если проект станет open-source, стандарт синхронизации устареет.

**Расширение качества:** Умеренное (если проект планируется как open-source)

**Рекомендация:** Добавить сейчас (если проект будет публичным), иначе отложить

**Пример добавления:**
```markdown
## 6. Работа с fork'ами (open-source)

Если вы работаете с fork'ом основного репозитория:

1. Добавить upstream:
   ```bash
   git remote add upstream https://github.com/original/repo.git
   ```
2. Синхронизировать main fork'а с upstream:
   ```bash
   git fetch upstream
   git checkout main
   git merge upstream/main
   git push origin main
   ```
3. Создавать feature-ветки от актуального main fork'а
```

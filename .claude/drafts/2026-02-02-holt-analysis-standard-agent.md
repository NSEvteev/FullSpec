## 2. Риск неправильной интерпретации LLM

### Проблема 2.1: Границы "Области работы" vs "Контекст"

**Где:** Секция "5. Контекст для агента" → Что логировать (строка 575)

**Цитата:**
> **Не логировать:** технические чтения (standard-*.md, standard-state.md и т.п.) — это внутренний контекст агента.

**Как может понять LLM неправильно:**
LLM может посчитать, что все файлы вне "области работы" — технический контекст, и не логировать чтение важных файлов проекта (например, существующих агентов при создании нового).

**Что приведёт к ошибке:**
- Агент создаёт дубликат существующего агента, но не залогировал чтение списка агентов
- В логе отсутствует информация о прочитанных SSOT-ссылках, но включены случайные файлы

**Рекомендация:**
> **Не логировать в operations[]:**
> - Стандарты и инструкции (standard-*.md, create-*.md, validation-*.md)
> - State-файлы координации (locks.json, agents-status.json)
> - Правила проекта (/.claude/rules/*.md)
> - Собственный лог (agent-{name}-operation.json)
>
> **Логировать:**
> - Файлы в области работы агента (указаны в запросе)
> - Файлы, которые агент анализирует для принятия решений (список существующих агентов, README для обновления)

**Критичность:** Высокая (неполный лог приводит к невозможности отладки)

### Проблема 2.2: "Критичная операция" без явного списка

**Где:** Секция "1. Назначение агентов" (строка 70)

### Проблема 2.3: "Атомарность операций" без границ

**Где:** Секция "5. Контекст для агента" → Координация (строка 546)

**Цитата:**
> **Атомарность операций:** В рамках одного агента операция "read-modify-write" для state-файлов считается атомарной (Claude Code гарантирует последовательное выполнение).

**Как может понять LLM неправильно:**
LLM может посчитать, что "атомарность" означает отсутствие необходимости блокировки для ЛЮБЫХ файлов (не только state), если они изменяются одним агентом.

**Что приведёт к ошибке:**
Агент модифицирует файл проекта без блокировки, полагаясь на "атомарность", в то время как другой агент делает то же самое.

**Рекомендация:**
> **Атомарность операций для state-файлов:** В рамках одного агента операция "read-modify-write" для `agents-status.json` и `locks.json` считается атомарной (Claude Code гарантирует последовательное выполнение). Для файлов проекта блокировка обязательна.

**Критичность:** Высокая (риск race condition)

### Проблема 2.4: Непонятно, когда обновлять finished_at

**Где:** Секция "5. Контекст для агента" → Обновление finished_at (строка 579-581)

**Цитата:**
> - Обновить `finished_at` когда агент **завершает работу** (возвращает управление основному LLM с отчётом о выполнении)
> - Если пользователь запросил дополнительные правки В ТОЙ ЖЕ сессии агента → НЕ обновлять `finished_at`, продолжить работу

**Как может понять LLM неправильно:**
LLM может не понять, что такое "ТА ЖЕ сессия". Если агент вернул отчёт, пользователь попросил правки через час — это та же сессия или новая?

**Что приведёт к ошибке:**
- Агент обновляет `finished_at` после каждого возврата управления → в логе множество "завершений"
- Агент не обновляет `finished_at` вообще, считая всё одной сессией → finished_at = null навсегда

**Рекомендация:**
> - Обновить `finished_at` когда агент возвращает финальный отчёт И в промпте нет цикла "Ожидать правок"
> - Если агент работает в цикле правок (AskUserQuestion → правки → AskUserQuestion) — НЕ обновлять до финального "Задача завершена"
> - Если агент завершил, а через час пользователь попросил новое изменение — это НОВЫЙ запуск (новая запись в operations[])

**Критичность:** Средняя

### Проблема 2.5: "Хуки автоматизированы" без явного разделения ответственности

**Где:** Секция "5. Контекст для агента" → Автоматизация через хуки (строка 488)

**Цитата:**
> **Что остаётся на агенте:**
> - Проверка и создание блокировок в `locks.json`
> - Снятие блокировок **сразу после операции** (не в конце работы!)
> - Ведение лога в `agent-{name}-operation.json`

**Как может понять LLM неправильно:**
LLM может посчитать, что хук `SubagentStop` снимет все блокировки автоматически, и не снимать их после каждой операции (полагаясь на cleanup).

**Что приведёт к ошибке:**
Агент блокирует файл, обрабатывает его, переходит к следующему файлу, не сняв блокировку. Файл остаётся заблокированным до завершения работы агента, блокируя других агентов.

**Рекомендация:**
> **Что остаётся на агенте (хук НЕ выполнит):**
> - Проверка и создание блокировок в `locks.json` перед каждой операцией
> - Снятие блокировок **сразу после каждой операции** (не полагаться на хук SubagentStop!)
> - Ведение лога в `agent-{name}-operation.json`
>
> **Хук SubagentStop** выполняет только fallback cleanup (удаляет "забытые" блокировки при сбое агента).

**Критичность:** Высокая (блокировки не снимаются, deadlock)

### Проблема 2.6: "Предохранитель от зацикливания" без поведения при достижении

**Где:** Секция "3. Формат конфигурации" → Поле max_turns (строка 267)

**Цитата:**
> Ограничивает количество итераций агента (вызовов API). Предохранитель от зацикливания.

**Как может понять LLM неправильно:**
LLM (работающий как агент) может не понять, что делать при достижении max_turns: остановиться молча, вернуть ошибку, сохранить checkpoint?

**Что приведёт к ошибке:**
Агент достигает max_turns=30, обрывается без отчёта, пользователь не знает, выполнена задача или нет.

**Рекомендация:**
> Ограничивает количество итераций агента (вызовов API). При достижении лимита агент должен выполнить graceful shutdown: сохранить прогресс в `state/{agent-name}-checkpoint.json` и вернуть частичный результат с пометкой "Превышен max_turns, задача выполнена частично".

**Критичность:** Средняя

### Проблема 2.7: "Таймаут ожидания 30 сек" без учёта длинных операций

**Как может понять LLM неправильно:**

>    - Если после 6 проверок файл всё ещё заблокирован И блокирующий агент работает > 30 сек → вернуть ошибку "Файл {путь} заблокирован агентом {имя}, который выполняется более 30 сек. Попробуйте позже."

## 5. Слепые зоны автора

### Зона 5.1: Rollback агента при ошибке

**Суть:**
Документ описывает graceful shutdown и сохранение checkpoint, но не описывает механизм отката изменений, если агент выполнил задачу частично и сломал что-то.

**Почему важно:**
Агент с Edit/Write может изменить 10 файлов, сломать 11-й, и остановиться. Пользователю нужен способ откатить ВСЕ изменения агента.

**Расширение качества:** Значительное

**Рекомендация:** Добавить сейчас

**Пример добавления:**
> ### Rollback изменений
>
> Если агент внёс изменения и задача провалилась (ошибка или неверный результат):
> 1. Основной LLM читает `agent-{name}-operation.json`
> 2. Находит все операции `edit`, `create`, `write`
> 3. Для каждого изменённого файла:
>    - Если это `edit` — откатить через git (если включен version control)
>    - Если это `create` — удалить созданный файл (или переименовать в `_rollback_*`)
> 4. Обновить лог: добавить запись "rollback: reverted {N} operations"
>
> **Инструмент:** `/agent-rollback` (если создан) или вручную через git.

**Почему важно:**
Ускорение выполнения задач. Например, валидация 100 файлов — запустить 4 агента по 25 файлов параллельно.

**Расширение качества:** Умеренное

**Рекомендация:** Отложить (это оптимизация, не базовый функционал)

**Пример добавления (если добавлять):**

> 1. Основной LLM разделяет задачу на N частей
> 2. Запускает N агентов с разными областями работы
> 3. Агенты работают независимо (блокировки не пересекаются)

## 1. Двойственность формулировок

### Проблема 1.1: Неопределённое условие "когда создавать скилл"

**Где:** Секция "1. Назначение" (строки 56-59)

**Цитата:**
> **Когда создавать скилл:**
> - Есть SSOT-инструкция с воркфлоу
> - Нужен удобный триггер для вызова
> - Действие повторяется

**Как может понять LLM:**
- Интерпретация A: Все три условия должны выполняться одновременно (AND)
- Интерпретация B: Достаточно выполнения любого условия (OR)
- Интерпретация C: Первое обязательно, остальные опциональны

**Рекомендация:**
> **Когда создавать скилл (ВСЕ условия обязательны):**
> - ✅ Есть SSOT-инструкция с воркфлоу
> - ✅ Действие повторяется (используется > 2 раз)
> - ✅ Нужен триггер для быстрого вызова

**Критичность:** Высокая

**Где:** Секция "1. Назначение" (строка 58)

**Цитата:**
> - Нужен удобный триггер для вызова

**Как может понять LLM:**
- Интерпретация A: Субъективно решает LLM ("мне кажется, это удобно")
- Интерпретация B: Триггер нужен всегда (любой скилл = триггер)
- Интерпретация C: Пользователь запросил триггер явно

**Рекомендация:**

**Критичность:** Средняя

- Интерпретация C: Может повторяться в будущем

**Как может понять LLM:**

- Интерпретация B: Инструкция существует, но не валидирована

## 2. Риск неправильной интерпретации LLM

### Проблема 2.1: Неявная последовательность создания

**Где:** Секция "1. Назначение" (строки 66-70)

**Цитата:**
> **Принцип:**
> ```
> 1. Инструкция (SSOT)     ← Сначала
> 2. Скилл (триггер)       ← Потом
> ```

**Как может понять LLM:**
- Риск A: Создать скилл и инструкцию параллельно (нарушение SSOT-первого)
- Риск B: Создать скилл с ссылкой на несуществующую инструкцию (SSOT в планах)
- Риск C: Игнорировать порядок при рефакторинге (инструкция уже есть, скилл можно всегда)

**Рекомендация:**
> **Порядок создания (ОБЯЗАТЕЛЬНЫЙ):**
> 1. Проверить: существует ли файл SSOT-инструкции? Если нет → вызвать `/instruction-create`
> 2. Проверить: есть ли секции "Шаги" и "Чек-лист" в инструкции? Если нет → завершить создание инструкции
> 3. Только после выполнения п.1-2 → создавать скилл

**Критичность:** Высокая

---

### Проблема 2.2: Неопределённый масштаб правил для create/modify скиллов

**Где:** Секция "6. Правила для create/modify скиллов" (строка 212)

**Цитата:**
> Инструкции create-skill.md и modify-skill.md следуют дополнительным правилам.

**Как может понять LLM:**
- Риск A: Правила применяются только к файлам `create-skill.md` и `modify-skill.md` (буквально 2 файла)
- Риск B: Правила применяются ко всем `create-*.md` и `modify-*.md` инструкциям в папке skills
- Риск C: Правила применяются к инструкциям для ЛЮБЫХ скиллов (например, `create-structure.md` тоже)

**Рекомендация:**
> **Правила для create/modify скиллов** относятся к **инструкциям типа create и modify в папке `/.claude/.instructions/skills/`**: `create-skill.md`, `modify-skill.md` и любым будущим инструкциям для работы со скиллами.
>
> Эти правила НЕ применяются к инструкциям других объектов (например, `create-structure.md`).

**Критичность:** Высокая

### Проблема 2.3: Неявное ожидание при циклах подтверждения

**Где:** Секция "6. Правила для create/modify скиллов" → "Циклы подтверждения" (строка 230-234)

**Цитата:**
> Воркфлоу МОЖЕТ содержать цикл подтверждения. При выборе "Дорабатываем" — изучить связанные документы и внести изменения.

**Как может понять LLM:**
- Риск A: "Дорабатываем" → LLM самостоятельно изменяет файлы без повторного подтверждения
- Риск B: "Дорабатываем" → LLM предлагает изменения, но не применяет без явного "Применить"
- Риск C: "Дорабатываем" → LLM спрашивает ЧТО изменить, затем изменяет и повторяет подтверждение

**Рекомендация:**
> **Цикл подтверждения:**
> 1. AskUserQuestion: "Подтверждаю / Дорабатываем / Другое"
> 2. При выборе "Дорабатываем":
>    - Спросить: "Что именно изменить?"
>    - Изучить связанные документы (стандарт, create, modify, validation)
>    - Внести изменения
>    - **Повторить Шаг 1** (цикл до "Подтверждаю")

**Критичность:** Средняя

### Проблема 2.4: Неоднозначность "Связь со скиллом" ДОЛЖЕН

**Где:** Секция "6. Правила для create/modify скиллов" → "Связь со скиллом" (строка 279)

**Цитата:**
> Воркфлоу create/modify/validation скиллов ДОЛЖЕН иметь связанный скилл.

**Как может понять LLM:**
- Риск A: Воркфлоу = инструкция (файл должен иметь скилл)
- Риск B: Воркфлоу = процесс выполнения (в момент выполнения должен быть скилл)
- Риск C: ДОЛЖЕН = может не иметь, но рекомендуется

**Рекомендация:**
> **Каждая инструкция create/modify/validation скиллов ОБЯЗАНА иметь связанный скилл:**
> - Секция "Скиллы" должна быть непустой
> - Скилл должен существовать как файл `/.claude/skills/{name}/SKILL.md`
> - Скилл должен ссылаться на эту инструкцию как SSOT
>
> **Исключение:** standard-инструкции не требуют скилла (нет воркфлоу).

**Критичность:** Средняя

## 4. Полнота описания объекта

### 4.1. Не хватает: Версионирование скиллов

**Почему важно:**
При изменении SSOT-инструкции скилл может устареть (ссылки, параметры). Нет механизма отслеживания версий скилла относительно версии инструкции.

**Стоит ли добавлять:** Да

**Что даст:** Автоматическую проверку актуальности скиллов после обновления инструкций (CI/CD).

**Предложение:**
Добавить в frontmatter скилла:
```yaml
ssot-version: v1.2  # версия SSOT-инструкции
```

Скрипт `check-skill-version-drift.py` проверяет соответствие версий.

**Приоритет:** P2

## 7. Жизненный цикл скилла

## 5. Слепые зоны автора

### Зона 5.1: Миграция скиллов при рефакторинге инструкций

**Суть:** Документ не рассматривает сценарий массового изменения инструкций (например, переименование папки `.instructions/` → `.docs/`). Как обновить все SSOT-ссылки в скиллах?

**Почему важно:** Рефакторинг структуры проекта может сломать все скиллы, если нет механизма массовых изменений.

**Расширение качества:** Значительное

**Рекомендация:** Добавить сейчас

**Пример добавления:**
```markdown
## 8. Массовое изменение скиллов

Если меняется структура проекта (переименование папок инструкций):

1. Использовать скрипт `bulk-update-ssot.py`:
   ```bash
   python .claude/.instructions/skills/.scripts/bulk-update-ssot.py \
     --old-path ".instructions/" \
     --new-path ".docs/"
   ```

2. Валидация всех скиллов:
   ```bash
   python .claude/.instructions/skills/.scripts/validate-skill.py --all
   ```
```

**Приоритет:** P2

### Зона 5.2: Зависимости между скиллами

**Суть:** Скилл может вызывать другой скилл (например, `/skill-create` вызывает `/skill-validate`). Нет описания правил и ограничений таких зависимостей.

**Почему важно:** Циклические зависимости или глубокая вложенность могут создать неуправляемую систему.

**Расширение качества:** Умеренное

**Рекомендация:** Отложить (добавить при появлении первого кейса сложной зависимости)

**Пример добавления:**
```markdown
## 9. Зависимости между скиллами

Скилл МОЖЕТ вызывать другой скилл через команду `/{skill-name}`.

**Правила:**
- Максимальная глубина вызовов: 2 уровня
- Запрещены циклические зависимости (A → B → A)
- Зависимость должна быть явно указана в секции "Скиллы"
```

**Приоритет:** P3

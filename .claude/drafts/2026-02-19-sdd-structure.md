## Содержание

### Проблемы текущей модели

Текущая цепочка: `Discussion → Impact → Design → ADR → Plan Test → Plan Dev`

**1. Impact и Design перекрываются.**
Impact говорит "затронуты сервисы A, B, C". Design говорит "ответственности распределены так: A делает X, B делает Y". Это одно решение в два шага — Impact "предлагает", Design "решает". Два документа для одного мыслительного процесса.

**2. Design принимает решения, которые ADR может отменить.**
Design распределяет ответственности, рисует контракты, создаёт ~19 файлов — а потом ADR для конкретного сервиса может сказать "это технически невозможно". Обратная связь ADR → Design не описана как нормальный путь.

**3. ADR — это детализация Design, а не отдельный уровень.**
Design говорит "notification отвечает за X", ADR говорит "notification реализует X через Y". Это один процесс мышления, разорванный на два документа.

**4. Plan Test и Plan Dev — можно объединить.**
"Что тестировать" и "что реализовать" — один план для сервиса.

**5. 6 уровней для КАЖДОГО изменения.**
Даже для добавления одного эндпоинта нужно пройти все 6 уровней.

**6. Discussion/Impact создают 0 артефактов, Design создаёт ~19.**
Резкий скачок — "big bang" на Design. При откате — ~19 файлов впустую.

---

### Принцип: документы для разработчика, а не для процесса

Документы цепочки (Discussion, Impact, Design, ADR) — это **процесс принятия решений**. Они нужны, чтобы пользователь и LLM вместе пришли к решению.

Но при разработке LLM-разработчик не читает эти документы. Ему нужен **результат** цепочки — живая документация, описывающая текущее состояние системы + конкретный план работы.

**Вывод:** Цепочка — это процесс. Документы 1-8 (ниже) — это рабочий контекст. Цепочка должна быть спроектирована так, чтобы эффективно **порождать** рабочий контекст, а не наоборот.

---

### Рабочий контекст LLM-разработчика

Пример задачи: *"Реализовать WebSocket push notifications в notification-сервисе"*.

Что нужно LLM, чтобы сесть и писать код:

| # | Документ | Зачем | Пример |
|---|----------|-------|--------|
| 1 | **Задача** | Что именно сделать, scope, definition of done | "Добавить WS endpoint, подписку на события, доставку" |
| 2 | **Тесты** | Критерии приёмки — что должно пройти | "WS подключение с JWT → уведомление → mark as read" |
| 3 | **Мой сервис** | Как устроен сейчас — API, data model, код, зависимости | `services/notification.md` |
| 4 | **Контракты с соседями** | Как мой сервис общается с другими | "auth: POST /validate → 200 {valid, user_id}" |
| 5 | **Стандарты технологий** | Как писать код — naming, паттерны | `standard-postgresql.md`, `standard-redis.md` |
| 6 | **Обзор системы** | Кто вообще есть и как связан | Карта сервисов, потоки данных |
| 7 | **Домен** | Агрегаты, инварианты, события | "Уведомление: один пользователь, TTL 90 дней" |
| 8 | **Инфраструктура** | Куда деплоить, ограничения среды | "Redis для WS, PostgreSQL для хранения" |

**Что НЕ нужно при написании кода:**
- Текст дискуссии (проблема переварена в задачу)
- Импакт-анализ (промежуточный артефакт планирования)
- Полный Design-документ (релевантные части уже в описании сервиса и контрактах)

### Два контура документации

Ключевой инсайт: существуют два принципиально разных типа документации.

**Контур 1 — Аналитическая документация (процесс).**
Документы, в которых пользователь и LLM вместе принимают решения: дискуссии, анализ, проектирование. Это "строительные леса" — нужны при принятии решений, но не при разработке.

**Контур 2 — Документация для поставки (результат).**
Документы, которые LLM-разработчик реально читает при написании кода: описание сервиса, контракты, стандарты технологий, обзор системы, инфраструктура. Это "рабочий контекст".

Цепочка аналитических документов **порождает** документацию для поставки. Разработчик читает только контур 2.

### Структура specs/ (v2 — текущая)

**Ключевые решения:**
- Тесты и задачи — часть analysis/ (привязаны к конкретному изменению, не к сервису)
- Per-service = один файл `docs/{svc}.md`, не папка (нет подпапок → не нужна папка)
- Обзор системы = `docs/README.md` (индекс + таблица сервисов, без отдельного overview.md)
- Точка-префикс для служебных папок: `.system/`, `.technologies/`

```

│   ├── 0001-realtime-notifications/   # Всё по одному изменению — в одной папке
│   │   ├── ...                        # Аналитические документы (состав TBD)
│   │   └── ...                        # Тесты и задачи — тоже здесь
│   └── 0002-.../
│
├── docs/                              # Контур 2: для поставки (результат)
│   ├── notification.md                # Per-service: один файл — всё про notification
│   ├── auth.md                        # Per-service: один файл — всё про auth
│   ├── gateway.md                     # ...
│   ├── frontend.md                    # ...
│   ├── README.md                      # Индекс сервисов + ссылки на .system/, .technologies/

│   │   ├── infrastructure.md          # Платформа: deployment, networking, мониторинг
│   │   ├── overview.md                # Архитектура: связи сервисов, data flows, контекстная карта
│   │   └── testing.md                 # Тестирование: типы, структура, мокирование, команды
```

**Что содержит `docs/{svc}.md`:**

| Секция | Содержание |
|--------|-----------|
| Назначение | Что делает сервис, зона ответственности |
| API контракты | Endpoints, форматы, протоколы (провайдер описывает полностью) |
| Data Model | Таблицы, схемы, хранилища |
| Code Map | Структура кода, ключевые пакеты/модули |
| Зависимости | От кого зависит (абзац + ссылка на API провайдера) |
| Доменная модель | Агрегаты, инварианты, события |
| Границы автономии LLM | Что можно менять без согласования |

**Что содержит `docs/README.md`:** Чистый индекс — таблица сервисов, ссылки на .system/ и .technologies/. Архитектура — в overview.md.

**Что содержит `docs/.system/overview.md`:** Вся системная архитектура — карта сервисов, связи, сквозные потоки, контекстная карта доменов, shared-код. 6 секций.

**Что содержит `docs/.system/conventions.md`:** Кросс-сервисные конвенции API (формат ошибок, пагинация, auth-заголовки) + интерфейсы shared-пакетов (сигнатуры, примеры вызова).

**Что содержит `docs/.system/testing.md`:** Типы тестов (unit/integration/e2e/smoke/load), структура файлов, стратегия мокирования по уровням, межсервисные тесты, тестовые данные, команды запуска. Per-tech паттерны (pytest, fixtures) — в standard-{tech}.md.

**Что содержит `docs/.technologies/standard-{tech}.md`:** Конвенции кодирования для разработчика + ссылка на скрипт валидации. Цепочка автоматизации: standard → validate-{tech}.py → pre-commit → .claude/rules/{tech}.md.

**Ключевые отличия от текущей модели:**

| Аспект | Текущая модель | Новая модель |
|--------|----------------|--------------|
| Группировка аналитики | По типу (discussion/, impact/, design/) | **По изменению** (analysis/0001-.../) |
| Группировка docs | По типу (architecture/system/, services/) | **По сервису** (docs/{svc}.md) |
| Тесты и задачи | В docs (per-service папки) | **В analysis/** (привязаны к изменению) |
| Обзор системы | Отдельный overview.md | **docs/.system/overview.md** (архитектура) + **docs/README.md** (индекс) |
| Навигация для разработчика | Собирать из 5+ папок | **Один файл сервиса** + README |

### Межсервисная информация и контракты

**Проблема:** Некоторая информация межсервисная по природе — контракты, топология, инфраструктура. Она не принадлежит одному сервису.

**Выявленные категории:**

| Информация | Чья это ответственность? |
|------------|-------------------------|
| API-контракт (endpoint, формат, протокол) | Провайдера — кто предоставляет API |
| Зависимость от чужого API | Потребителя — кто вызывает |
| Топология системы (какие сервисы, как связаны) | Ничья / общая |
| Инфраструктура (deployment, networking, мониторинг) | Ничья / платформенная |
| Доменная модель (агрегаты, события) | Per-domain |

**Предварительное решение (не финальное):** Контракт живёт у провайдера.

- **Провайдер** (кто предоставляет API) описывает контракт полностью: endpoint, формат запроса/ответа, протокол, ограничения.
- **Потребитель** (кто вызывает) описывает зависимость человеко-читаемым абзацем + ссылкой на раздел в документе провайдера.

Пример для notification → auth:

В `docs/auth.md`:
```markdown
## API контракты

### POST /api/v1/auth/validate
Валидация JWT-токена. Принимает Bearer token в заголовке.
Возвращает: `{ valid: boolean, user_id: string }`
```

В `docs/notification.md`:
```markdown
## Зависимости

### auth — валидация JWT
Notification использует auth-сервис для валидации JWT-токенов при WebSocket-подключении
и REST-запросах. См. [POST /api/v1/auth/validate](auth.md#post-apiv1authvalidate).
```

**Решённые вопросы по межсервисной информации:**

1. **Топология + data flows + контекстная карта:** `docs/.system/overview.md` — один файл, вся системная архитектура.
2. **Инфраструктура:** `docs/.system/infrastructure.md` — платформенный документ, отдельно от overview.
3. **Домены:** Встраиваются в per-service документ как секция "Доменная модель". Глобальная контекстная карта — в overview.md.
4. **docs/README.md:** Чистый индекс (таблица сервисов, ссылки на .system/, .technologies/). Архитектурный слой — в overview.md.

### Отвергнутые направления

**"Три уровня: Discussion → Design → Plan"**
Сжатие 6 уровней в 3 путём поглощения Impact в Design, ADR в Design, Plan Test + Plan Dev в Plan. Проблема: Design становится монстром. Предложение "два прохода" внутри Design — это просто переименование Design + ADR.

**"Design сразу per-service"**
Discussion определяет scope, дальше параллельно создаются per-service Design-ы, Reconciler сверяет контракты. Проблема: без глобального взгляда per-service дизайны противоречат друг другу, Reconciliation бесконечен. **Хуже текущей модели.**

## Решённые вопросы

- **Два контура?** — Да: analysis/ (процесс) и docs/ (результат).
- **Группировка аналитики?** — По изменению (analysis/0001-...), не по типу документа.
- **Группировка docs?** — Per-service: один файл `docs/{svc}.md`, не папка.
- **Контракты?** — Контракт живёт у провайдера. Потребитель пишет абзац + ссылку.
- **Тесты и задачи?** — В analysis/ (привязаны к изменению). Docs/ содержит только постоянное описание системы.
- **docs/README.md?** — Чистый индекс (таблица сервисов, ссылки на .system/ и .technologies/). Архитектурный слой отдельно.
- **docs/.system/overview.md?** — Один файл, вся системная архитектура: карта сервисов, связи, сквозные потоки, контекстная карта доменов, shared-код. 6 секций.
- **Служебные папки?** — Точка-префикс: `.system/`, `.technologies/`.
- **Доменная модель?** — Встраивается в per-service документ как секция "Доменная модель".
- **Инфраструктура?** — `docs/.system/infrastructure.md` — платформенный документ, не привязан к сервису.
- **Секции docs/{svc}.md?** — 10 секций. Три уровня детализации: имплементационный (API контракты с request/response, Data Model с типами/индексами), архитектурный (Потоки — runtime-сценарии, Доменная модель), навигационный (остальное).
- **Стандарты технологий?** — В `docs/.technologies/`. Только `standard-{tech}.md` (конвенции для разработчика). Отдельный `validation-{tech}.md` не нужен — валидация автоматизирована скриптом. Цепочка: standard-{tech}.md (содержит ссылку на скрипт) → validate-{tech}.py (скрипт) → pre-commit hook → .claude/rules/{tech}.md (автозагрузка).
- **Кросс-сервисные конвенции и shared-интерфейсы?** — Отдельный файл `docs/.system/conventions.md`. Разделение: overview.md = архитектура (что система делает), conventions.md = конвенции (как пишем код одинаково). Аналогично `.technologies/standard-{tech}.md` для per-tech конвенций. 6 секций: формат ответов, формат ошибок, пагинация, аутентификация, версионирование, shared-пакеты (с интерфейсами и примерами).
- **Тестирование?** — Два уровня: `docs/.system/testing.md` (кросс-сервисная стратегия: типы тестов, структура файлов, мокирование по уровням, межсервисные тесты, тестовые данные, команды запуска) + секция "Тестирование" в каждом `standard-{tech}.md` (per-tech паттерны: фреймворк, фикстуры, мокирование конкретной технологии, типовые тесты). Отдельный документ с процедурой тестирования не нужен — LLM-разработчик ВСЕГДА работает через analysis/, задача указывает что тестировать.
- **Логирование?** — Два уровня: секция "Логирование" в `conventions.md` (кросс-сервисная конвенция: библиотека structlog, уровни, обязательные поля, именование событий, запрет PII) + секция "Логирование" в каждом `standard-{tech}.md` (per-tech настройка: какие события на каком уровне, настройка логгера технологии).
- **Developer commands?** — Отдельный документ не нужен. Makefile = SSOT для всех команд (`make help` — полный список). Docs/ ссылается на Makefile, не дублирует. Прямые вызовы инструментов (pytest, alembic) документируются в domain-документах как технологические детали.
- **Запуск миграций?** — Per-tech тема (в нашем проекте только PostgreSQL). Команды alembic живут в `standard-postgresql.md` рядом с паттерном кода миграции. Make-таргеты — в Makefile. Отдельный документ не нужен.

## Открытые вопросы

**Контур analysis/ — структура процесса:**

1. **Какие документы внутри `analysis/0001-.../`?** Дискуссия — точно. Что ещё? Design? Тесты? Задачи? Один файл или несколько?
2. **Аналитика — хранить или удалять?** `analysis/0001-...` после завершения — архив или мусор?
3. **Минимальная единица:** Какого размера изменение проходит через цепочку? Один эндпоинт? Фича? Эпик?

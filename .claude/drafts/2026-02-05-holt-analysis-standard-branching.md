## 1. Двойственность формулировок

### Проблема 1.1: Неопределённый критерий "слишком разные"

**Где:** Секция "5. Граничные случаи" → "Issues слишком разные для одной ветки" (строка 170-172)

**Цитата:**
> Если Issues из группы затрагивают разные области кода без пересечений или имеют разные TYPE-метки — разделить на отдельные ветки и мержить последовательно

**Как может понять LLM:**
- Интерпретация A: Разделять ТОЛЬКО если оба условия выполнены (разные области И разные TYPE)
- Интерпретация B: Разделять если хотя бы одно условие выполнено (разные области ИЛИ разные TYPE)
- Интерпретация C: "Без пересечений" означает 0% общих файлов (субъективно)

**Рекомендация:**
> Разделить на отдельные ветки если выполнено ЛЮБОЕ из условий:
> - Issues затрагивают файлы без пересечений (менее 20% общих файлов)
> - Issues имеют разные TYPE-метки (bug, feature, docs)

**Критичность:** Высокая (P1) — блокирует правильное принятие решения о разделении веток

**Критичность:** Средняя (P2) — не блокирует работу, но может привести к несвоевременной синхронизации

## 2. Риск неправильной интерпретации LLM

### Проблема 2.1: Отсутствие инструкции для парсинга группы Issues

**Где:** Секция "2. Naming Convention" → таблица элементов (строка 77)

**Цитата:**
> `{issue-numbers}` | Номера всех Issues группы через дефис | `42-43-44`

**Как может понять LLM:**
- LLM может попытаться распарсить строку "42-43-44" как диапазон (от 42 до 44), а не список
- LLM может перепутать порядок (43-42-44 или 42-44-43)
- LLM может не знать, нужна ли сортировка номеров

**Рекомендация:**
> `{issue-numbers}` | Номера всех Issues группы через дефис в порядке возрастания | `42-43-44` (сортировка: 42 < 43 < 44)
>
> Пример парсинга: `feature/auth-42-43-44` → Issues: #42, #43, #44

**Критичность:** Высокая (P1) — блокирует корректное извлечение номеров Issues из имени ветки

### Проблема 2.3: Отсутствие алгоритма для множественных Issues с разными TYPE

**Где:** Секция "2. Naming Convention" → "Правила выбора type" (строка 89-91)

**Цитата:**
> Если в группе Issues разные TYPE-метки — использовать тип первого Issue по номеру.

**Как может понять LLM:**
- Правило есть, но нет алгоритма проверки
- LLM может не знать, где брать список Issues (из аргументов команды? из контекста?)
- Неясно, что делать если Issues не имеют TYPE-меток

**Рекомендация:**
> Алгоритм выбора префикса:
> 1. Получить список номеров Issues: `{issue-numbers}` (например, 42-43-44)
> 2. Для каждого Issue извлечь TYPE-метку: `gh issue view {number} --json labels -q '.labels[] | select(.name | startswith("type:")) | .name'`
> 3. Если все TYPE одинаковы → использовать этот тип
> 4. Если TYPE разные → выбрать TYPE Issue с минимальным номером
> 5. Если хотя бы у одного Issue нет TYPE-метки → ОШИБКА, добавить метку перед созданием ветки

**Критичность:** Высокая (P1) — без алгоритма LLM не сможет реализовать правило

## 4. Полнота описания объекта

### 4.1. Не хватает: Процесс push ветки в remote

**Суть:**
- Документ описывает создание ветки локально (`git checkout -b`)
- Не описан процесс push ветки в remote (`git push -u origin {branch-name}`)
- Не указано, когда делать push (сразу после создания? после первого коммита?)

**Почему важно:**
- Без push нельзя создать PR
- LLM может забыть запушить ветку перед созданием PR

**Стоит ли добавлять:** Да (P1)

**Что даст:** Полный воркфлоу от создания ветки до PR

**Рекомендация:**
> Добавить в секцию "3. Жизненный цикл" → "Работа":
>
> ### Push ветки в remote
>
> После первого коммита запушить ветку в remote:
> ```bash
> git push -u origin {branch-name}
> ```
>
> Флаг `-u` (upstream) устанавливает связь между локальной и remote-веткой для последующих `git push` без аргументов.

### 4.4. Не хватает: Правила для Squash Merge

**Суть:**
- Документ упоминает "Squash Merge в main" (строка 104), но не объясняет:
  - Что это такое
  - Почему используется Squash вместо Regular Merge
  - Как это влияет на историю коммитов в ветке

**Почему важно:**
- LLM может не понять, зачем несколько коммитов в ветке, если они всё равно сквошатся
- Влияет на стратегию коммитов (можно делать много мелких коммитов)

**Стоит ли добавлять:** Да (P2)

**Что даст:** Понимание стратегии коммитов в ветке

**Рекомендация:**
> Добавить в секцию "3. Жизненный цикл" → "Завершение":
>
> **Squash Merge:** При merge PR в main все коммиты feature-ветки объединяются в один. Это позволяет:
> - Делать много мелких коммитов во время разработки (для удобства)
> - Сохранять чистую линейную историю в main (один коммит на PR)
>
> Подробнее: [standard-review.md § 4](../review/standard-review.md#4-merge-strategies)

### Зона 5.5: Обработка зависимых веток

**Суть:**
- Что делать, если фича A зависит от фичи B (которая ещё не смержена)?
- Нужно ли создавать ветку от feature-ветки B?
- Или ждать merge B → создать ветку A от обновлённой main?

**Почему важно:**
- Блокирующие зависимости замедляют разработку
- Создание ветки от ветки (вложенные ветки) нарушает модель

**Расширение качества:** Значительное (критично для параллельной работы)

**Рекомендация:** Добавить сейчас

**Пример добавления:**
> ### Зависимые задачи
>
> Если Issue A зависит от Issue B (который ещё не смержен):
>
> **Вариант 1 (рекомендуется):** Дождаться merge B → создать ветку A от обновлённой main
>
> **Вариант 2 (если B большой):** Создать draft PR для B → создать ветку A от main → синхронизировать A с веткой B вручную:
> ```bash
> git checkout feature/A-50
> git fetch origin feature/B-49
> git rebase origin/feature/B-49
> ```
>
> **ЗАПРЕЩЕНО:** Создание ветки A от ветки B через `git checkout -b` (вложенные ветки).

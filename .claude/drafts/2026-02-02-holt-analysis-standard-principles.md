## 1. Двойственность формулировок

### Проблема 1.1: Неопределённый критерий "максимально простым"

**Где:** Секция "1. KISS" → "Правило" (строка 47)

**Цитата:**
> Выбирать простое решение вместо сложного, даже если сложное кажется "элегантнее".

**Как может понять LLM:**
- Интерпретация A: Простота = меньше строк кода (всегда короче)
- Интерпретация B: Простота = меньше концепций (понятнее для читателя)
- Интерпретация C: Простота = меньше зависимостей или слоёв абстракции

**Рекомендация:**
> Выбирать решение, которое легче понять будущему читателю кода, даже если оно содержит больше строк. Простота = минимум концепций для понимания, а не минимум символов.

**Критичность:** Высокая (P1)

---

### Проблема 1.2: Субъективный термин "элегантнее"

**Где:** Секция "1. KISS" → "Правило" (строка 47)

**Цитата:**
> Выбирать простое решение вместо сложного, даже если сложное кажется "элегантнее".

**Как может понять LLM:**
- Интерпретация A: Элегантность = краткость записи
- Интерпретация B: Элегантность = использование продвинутых возможностей языка
- Интерпретация C: Элегантность = математическая красота решения

**Рекомендация:**
> Удалить термин "элегантнее" или заменить на конкретный пример: "даже если оно использует продвинутые возможности языка (декораторы, метаклассы, функторы)".

**Критичность:** Средняя (P2)

---

### Проблема 1.3: Размытое "без необходимости"

**Где:** Секция "1. KISS" → "Признаки нарушения" (строка 70)

**Цитата:**
> Цепочки map/filter/reduce без необходимости

**Как может понять LLM:**
- Интерпретация A: Без необходимости = всегда плохо (запрещено)
- Интерпретация B: Без необходимости = когда есть более простой способ (субъективно)
- Интерпретация C: Без необходимости = когда длина цепочки превышает N операций

**Рекомендация:**
> Цепочки map/filter/reduce из 3+ операций или с вложенными lambda

**Критичность:** Высокая (P1)

---

### Проблема 1.4: Неопределённое "похожие функции"

**Где:** Секция "2. DRY" → "Признаки нарушения" (строка 122)

**Цитата:**
> Похожие функции

**Как может понять LLM:**
- Интерпретация A: Похожие по имени
- Интерпретация B: Похожие по структуре (но разная логика)
- Интерпретация C: Похожие по сигнатуре (одинаковые аргументы)
- Интерпретация D: Похожие по блокам кода > 3 строк

**Рекомендация:**
> Функции с дублированием блоков кода (> 3 строк идентичного кода)

**Критичность:** Средняя (P2)

---

### Проблема 1.5: Размытое "неиспользуемые аргументы функций"

**Где:** Секция "3. YAGNI" → "Признаки нарушения" (строка 150)

**Цитата:**
> Неиспользуемые аргументы функций

**Как может понять LLM:**
- Интерпретация A: Аргументы, которые не упоминаются в теле функции
- Интерпретация B: Аргументы, которые упоминаются, но не влияют на результат
- Интерпретация C: Аргументы с дефолтными значениями, которые никогда не переопределяются

**Рекомендация:**
> Аргументы, которые не используются в теле функции (отсутствуют в коде после объявления)

**Критичность:** Средняя (P2)

---

### Проблема 1.6: Неоднозначность "конфигурационные параметры, которые никто не меняет"

**Где:** Секция "3. YAGNI" → "Признаки нарушения" (строка 153)

**Цитата:**
> Конфигурационные параметры, которые никто не меняет

**Как может понять LLM:**
- Интерпретация A: Параметры с дефолтными значениями, которые всегда дефолтные
- Интерпретация B: Параметры, которые не переопределяются в текущем коде (но могут в будущем)
- Интерпретация C: Параметры, которые не меняются через конфигурационные файлы

**Рекомендация:**
> Аргументы функции с дефолтными значениями, которые никогда не переопределяются при вызове (проверить все вызовы функции)

**Критичность:** Низкая (P3)

---

### Проблема 1.7: Двойственность "предсказуемое поведение"

**Где:** Секция "4. Принцип наименьшего удивления" → Таблица "Правила" (строка 190)

**Цитата:**
> Возвращаемые значения | Консистентные типы

**Как может понять LLM:**
- Интерпретация A: Всегда один и тот же тип (например, всегда `str`)
- Интерпретация B: Типы из одного семейства (например, `str | None`)
- Интерпретация C: Один тип на ветку выполнения (но разные типы в разных ветках)

**Рекомендация:**
> Возвращаемые значения | Один тип или тип + None. Недопустимо: `return "ok"` в одной ветке и `return 0` в другой

**Критичность:** Средняя (P2)

---

### Проблема 1.8: Размытое "расширение через аргументы"

**Где:** Секция "5. SOLID" → "O — Open/Closed" (строка 225)

**Цитата:**
> Расширение через аргументы, не изменение кода.

**Как может понять LLM:**
- Интерпретация A: Добавлять новые аргументы при расширении (изменение сигнатуры)
- Интерпретация B: Использовать аргументы для выбора поведения (if/else внутри)
- Интерпретация C: Использовать полиморфизм (подклассы или протоколы)

**Рекомендация:**
> Расширение через дополнительные аргументы с дефолтными значениями. Для добавления новой функциональности не переписывать тело функции, а добавить аргумент, который активирует новое поведение.

**Критичность:** Средняя (P2)

## 2. Риск неправильной интерпретации

### Проблема 2.1: Пример "Плохо" для DRY может быть неправильно понят

**Где:** Секция "2. DRY" → "Пример" (строки 87-98)

**Сейчас:**
```python
# ❌ Плохо: дублирование
def validate_instruction(path):
    if not path.exists():
        print(f"❌ Файл не найден: {path}")
        return False
    # ...

def validate_script(path):
    if not path.exists():
        print(f"❌ Файл не найден: {path}")
        return False
    # ...
```

**Риск:**
LLM может понять, что дублирование = полностью идентичные блоки. Если блоки отличаются хотя бы одной строкой (например, разные сообщения об ошибках), LLM может решить, что это не дублирование.

**Рекомендация:**
Добавить комментарий:
> Дублирование = повторение логики проверки, даже если сообщения различаются. Критерий: > 3 строк похожего кода.

**Критичность:** Высокая (P1)

---

### Проблема 2.2: Противоречие между KISS и SOLID (O)

**Где:** Секция "1. KISS" (строка 47) vs Секция "5. SOLID" → "O — Open/Closed" (строка 225)

**Противоречие:**
- KISS: "Выбирать простое решение вместо сложного"
- Open/Closed: "Расширение через аргументы" (пример с `format='text'`)

**Риск:**
LLM может увидеть конфликт: добавление аргументов для расширения усложняет функцию (нарушение KISS). Непонятно, когда применять KISS, а когда Open/Closed.

**Рекомендация:**
Добавить секцию "Приоритет принципов" или уточнить в Open/Closed:
> Расширение через аргументы допустимо, если это избегает дублирования кода. Приоритет: DRY > Open/Closed > KISS.

**Критичность:** Высокая (P1)

---

### Проблема 2.3: Неявное граничное условие для "дублирование блоков кода"

**Где:** Секция "2. DRY" → "Признаки нарушения" (строка 120)

**Цитата:**
> Копирование блоков кода между функциями

**Риск:**
LLM может не знать, с какой длины блок считается дублированием. 2 строки? 3 строки? 10 строк?

**Рекомендация:**
> Копирование блоков кода (> 3 строк) между функциями

**Критичность:** Высокая (P1)

---

### Проблема 2.4: Риск неправильного понимания "однобуквенные имена переменных"

**Где:** Секция "6. Читаемость и документация" → "Признаки нарушения" (строка 296)

**Цитата:**
> Однобуквенные имена переменных (кроме итераторов)

**Риск:**
LLM может понять "итераторы" слишком широко и разрешить `x`, `y`, `z` в любых контекстах, где есть цикл.

**Рекомендация:**
> Однобуквенные имена переменных (кроме `i`, `j`, `k` в циклах for и `f` для file handles)

**Критичность:** Средняя (P2)

---

### Проблема 2.5: Неоднозначность "ранний выход"

**Где:** Секция "7. Обработка ошибок" → Таблица "Правила" (строка 343)

**Цитата:**
> Ранний выход | Проверять условия в начале функции

**Риск:**
LLM может понять это как "всегда выходить из функции при первой проверке", что неприменимо к функциям с несколькими независимыми проверками.

**Рекомендация:**
> Ранний выход | Проверять критические условия (существование файла, валидность аргументов) в начале функции и выходить с ошибкой до выполнения основной логики

**Критичность:** Средняя (P2)

---

### Проблема 2.6: Противоречие "магические числа" и "YAGNI"

**Где:** Секция "6. Читаемость и документация" → "Признаки нарушения" (строка 298) vs Секция "3. YAGNI"

**Риск:**
LLM может создать константу для магического числа, используемого один раз (YAGNI — не нужно), или не создать константу для числа, используемого дважды (читаемость — нужно).

**Рекомендация:**
Добавить явное правило:
> Магические числа без объяснения | Выносить в константы, если число используется > 1 раза или смысл неочевиден из контекста

**Критичность:** Низкая (P3)

**Что отсутствует:**

**Почему важно:**

**Стоит ли добавлять:** Да

**Пример добавления:**
```markdown
### L — Liskov Substitution

**В Python:** Подклассы не должны нарушать контракт родительского класса.

```python
# ❌ Плохо: подкласс изменяет поведение
class File:
    def read(self): return "data"

class EmptyFile(File):
    def read(self): raise NotImplementedError  # Нарушение контракта

# ✅ Хорошо: подкласс расширяет поведение
class EmptyFile(File):
    def read(self): return ""  # Корректное расширение
```

### I — Interface Segregation

**В Python:** Использовать Protocol (PEP 544) для узких интерфейсов.

```python
# ❌ Плохо: один большой протокол
class Animal(Protocol):
    def fly(self): ...
    def swim(self): ...
    def walk(self): ...

# ✅ Хорошо: отдельные протоколы
class Flyable(Protocol):
    def fly(self): ...

class Swimmable(Protocol):
    def swim(self): ...
```
```

**Критичность:** Средняя (P2)

## 5. Слепые зоны автора

### Зона 5.1: Производительность vs Читаемость

**Суть:**
Документ не покрывает ситуации, когда KISS конфликтует с производительностью. Например, простой цикл может быть в 100 раз медленнее оптимизированного решения.

**Почему важно:**
LLM может переписать производительный код на "простой", сломав production-систему.

**Расширение качества:** Значительное

**Рекомендация:** Добавить сейчас

**Пример добавления:**
```markdown
## 12. Производительность

> **Правило:** Оптимизация допустима, если профилирование показало узкое место.

**Принцип:** Сначала простой код (KISS), затем профилирование, затем оптимизация только горячих участков.

**Пример:**
```python
# ✅ Хорошо: простой код для редких операций
def process_files(files):
    result = []
    for f in files:
        if f.suffix == '.md':
            result.append(f)
    return result

# ✅ Хорошо: оптимизация для частых операций
def process_million_files(files):
    return [f for f in files if f.suffix == '.md']  # Быстрее на больших данных
```

**Когда оптимизировать:**
- Операция выполняется > 1000 раз в секунду
- Профилирование показало > 10% времени на этом участке
```

**Критичность:** Высокая (P1)

---

### Зона 5.2: Тестируемость кода

**Суть:**
Принципы не упоминают связь с тестированием. Например, функции с побочными эффектами (нарушение "Принципа наименьшего удивления") сложнее тестировать.

**Почему важно:**
LLM может написать "правильный" по принципам код, который невозможно протестировать.

**Расширение качества:** Умеренное

**Рекомендация:** Добавить сейчас

**Пример добавления:**
```markdown
## 13. Тестируемость

> **Правило:** Код, соответствующий принципам, легко тестируется.

**Связь с принципами:**

| Принцип | Как помогает тестам |
|---------|---------------------|
| KISS | Меньше путей выполнения → меньше тестов |
| DRY | Изменение в одном месте → изменение теста в одном месте |
| Наименьшее удивление | Предсказуемое поведение → простые тесты |
| SOLID (S) | Один тест на одну функцию |
| SOLID (D) | Мокирование зависимостей |

**Пример:** Функция с глобальным состоянием (нарушение S) требует сложного setup/teardown в тестах.
```

**Критичность:** Средняя (P2)

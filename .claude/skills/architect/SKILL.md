# Управление архитектурой

Скилл для создания архитектурных документов на основе одобренных дискуссий.

## ВАЖНОЕ ПРАВИЛО

**При ЛЮБЫХ действиях с архитектурой — сначала прочитать этот скилл!**

Это включает:
- Создание нового архитектурного документа
- Изменение существующей архитектуры
- Изменение статуса архитектуры
- Работу с индексом `000_architecture.md`

**Почему:** Скрипты автоматически обновляют счётчики и индексы. Ручные операции ломают синхронизацию.

---

## Когда использовать

**Триггеры:**
- Команда `/architect`
- Перевод дискуссии в статус `🟢 approved` (автоматически)
- Запрос "создать архитектуру" или "создать architecture"

**Контекст:**
- Дискуссия находится в статусе `🟢 approved` (только что подтверждена)
- Секции "Принятое решение" и "Ревью решения" заполнены

---

## Workflow статусов

```
🟣 review (дискуссия) → решение подтверждено
      ↓
🟢 approved (дискуссия) → вызывается /summary-doc
      ↓
/summary-doc обновляет 000_SUMMARY.md → вызывается /architect
      ↓
/architect ЧИТАЕТ:
  1. 000_SUMMARY.md (контекст всех решений)
  2. Дискуссию (детали текущего решения)
      ↓
/architect создаёт архитектуру (🟡 draft)
      ↓
⚪ final (дискуссия) — /architect переводит статус после создания
```

**ВАЖНО:**
- Сначала вызывается `/summary-doc` для обновления агрегированного файла
- Затем `/architect` читает SUMMARY + дискуссию для полного контекста
- После создания архитектуры дискуссия переводится в `final`

## Инструкции

### Шаг 1: Собрать контекст

**1.1 Прочитать SUMMARY (контекст всех решений):**
```
general_docs/01_discuss/000_SUMMARY.md
```

Это даёт контекст о других принятых решениях в проекте.

**1.2 Прочитать дискуссию и убедиться:**
- Статус `🟢 approved`
- Секция "Принятое решение" заполнена
- Секция "Ревью решения" заполнена

**Если статус не approved:**
```
Дискуссия должна быть в статусе approved для создания архитектуры.
Текущий статус: [статус]
```

**ВАЖНО:** Архитектура должна учитывать контекст из SUMMARY — другие решения, которые могут влиять на текущую архитектуру.

### Шаг 2: Создать архитектуру через скрипт

**ОБЯЗАТЕЛЬНО использовать скрипт:**

```bash
python scripts/architecture_new.py -t "Название" -d "001" --desc "Описание"
```

**Параметры:**
- `-t` — название компонента/архитектуры
- `-d` — ID связанной дискуссии (например: 001)
- `--desc` — краткое описание

**Скрипт автоматически:**
- Генерирует следующий ID (001, 002, ...)
- Создаёт файл по шаблону
- Обновляет индекс `000_architecture.md`
- Обновляет счётчик `.doc_counter`
- Устанавливает статус `🟡 draft`

**ЗАПРЕЩЕНО:** Создавать файлы вручную через Write tool!

### Шаг 3: Заполнить архитектуру из дискуссии

После создания файла:

1. Прочитать созданный файл
2. Извлечь из дискуссии:
   - **Цель и контекст** — из "Анализ цели и контекст"
   - **Принятое решение** — технологии из "Принятое решение"
   - **Ограничения** — из "Неявные допущения"
3. Заполнить секции:
   - Компоненты
   - Внешние зависимости (выбранная библиотека/фреймворк)
   - Требования к качеству
4. Изменить статус: `🟡 draft` → `🔵 in_progress`

### Шаг 4: Обновить дискуссию

После создания архитектуры:

1. Изменить статус дискуссии: `🟢 approved` → `⚪ final`
2. Обновить секцию "Связанная архитектура":
   ```markdown
   ## Связанная архитектура

   - [x] Создан `02_architecture/[ID]_[название].md`
   ```
3. Добавить запись в "История обсуждения":
   ```markdown
   - [YYYY-MM-DD]: Создана архитектура [ID]_[название].md, статус → final
   ```

### Шаг 5: Запустить /doc-review

После заполнения архитектуры запустить `/doc-review` для критического анализа.

## Структура архитектурного документа

1. Метаданные (статус, версия, дата)
2. Оглавление
3. Цель и контекст
4. Связанные документы (дискуссия, ресурсы, план)
5. Принципы и ограничения
6. Компоненты
7. Диаграммы
8. Интерфейсы и контракты
9. Данные
10. Внешние зависимости
11. Требования к качеству
12. Риски и компромиссы
13. История изменений

## Workflow статусов архитектуры

```
🟡 draft        → Документ создан, структура не заполнена
      ↓
🔵 in_progress  → Активная работа над архитектурой
      ↓
🟣 review       → Ревью архитектуры
      ↓
🟢 approved     → /summary-doc-arch обновляет SUMMARY архитектуры
      ↓
/decision       → Читает ОБА SUMMARY, создаёт ADR
      ↓
⚪ final        → ADR создан, архитектура завершена
```

**ВАЖНО:**
- При переходе `review → approved` вызывается `/summary-doc-arch` для обновления SUMMARY архитектуры
- После `/summary-doc-arch` вызывается `/decision`, который читает **ОБА** SUMMARY файла:
  - `01_discuss/000_SUMMARY.md` — контекст принятых решений
  - `02_architecture/000_SUMMARY.md` — контекст архитектурных компонентов
- Архитектура НЕ создаёт ресурсы и планы напрямую — это делает ADR

**Переходы:**

| Из | В | Когда | Действия |
|----|---|-------|----------|
| `🟡 draft` | `🔵 in_progress` | После заполнения секций | Edit статус |
| `🔵 in_progress` | `🟣 review` | Архитектура готова | Запустить ревью |
| `🟣 review` | `🟢 approved` | Ревью пройдено | **Вызвать `/summary-doc-arch`**, затем `/decision` |
| `🟢 approved` | `⚪ final` | Создан ADR | `/decision` → ADR создан |

## Команды

### Скрипты

```bash
# Создать архитектуру
python scripts/architecture_new.py "Название"
python scripts/architecture_new.py -t "Название" -d "001" --desc "Описание"
python scripts/architecture_new.py -i  # интерактивно
```

### Makefile

```bash
make arch-new                              # интерактивно
make arch-new-topic TITLE="Название" DISCUSS="001"
make arch-index                            # показать индекс
```

## Пример использования

**Дискуссия:** 001_ui_staylgayd.md (статус: approved)
**Выбранное решение:** Mantine

**Шаг 1:** Проверить статус
```
Дискуссия 001_ui_staylgayd.md
Статус: 🟢 approved ✓
Принятое решение: Mantine ✓
```

**Шаг 2:** Создать архитектуру
```bash
python scripts/architecture_new.py -t "UI Styleguide" -d "001" --desc "Архитектура UI на базе Mantine"
```

**Шаг 3:** Заполнить из дискуссии
- Цель: "Единая система дизайна для консистентного UI"
- Компоненты: Mantine, @mantine/form, @mantine/hooks
- Зависимости: Mantine v7.x

**Шаг 4:** Обновить дискуссию
```markdown
**Статус:** ⚪ final

## Связанная архитектура

- [x] Создан `02_architecture/001_ui_styleguide.md`

## История обсуждения

- [2026-01-16]: Создана архитектура 001_ui_styleguide.md, статус → final
```

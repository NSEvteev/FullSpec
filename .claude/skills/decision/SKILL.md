# Управление ADR (Architecture Decision Records)

Скилл для создания архитектурных решений (ADR) на основе одобренных архитектурных документов.

## ВАЖНОЕ ПРАВИЛО

**При ЛЮБЫХ действиях с ADR — сначала прочитать этот скилл!**

Это включает:
- Создание нового ADR
- Изменение существующего ADR
- Изменение статуса ADR
- Работу с индексом `000_decisions.md`

**Почему:** Скрипты автоматически обновляют счётчики и индексы. Ручные операции ломают синхронизацию.

---

## Когда использовать

**Триггеры:**
- Команда `/decision`
- Перевод архитектуры в статус `🟢 approved`
- Запрос "создать ADR" или "создать решение"

**Контекст:**
- Архитектура находится в статусе `🟢 approved`
- Архитектура заполнена и прошла ревью

---

## Workflow в цепочке зависимостей

```
Дискуссия (approved) → /architect → Архитектура (draft → approved)
                                          ↓
                           /decision (этот скилл!)
                                          ↓
                                    ADR (draft → approved)
                                          ↓
                            /resource + /imp-plan
                                          ↓
                                Ресурсы + План реализации
```

**ВАЖНО:** ADR — это "разделитель" в цепочке:
- Вверх: Дискуссия ← Архитектура ← ADR
- Вниз: ADR → Ресурсы → План реализации

---

## Инструкции

### Шаг 1: Проверить архитектуру

1. Прочитать архитектуру и убедиться:
   - Статус `🟢 approved`
   - Все основные секции заполнены
   - Прошла ревью

**Если статус не approved:**
```
Архитектура должна быть в статусе approved для создания ADR.
Текущий статус: [статус]
```

### Шаг 2: Прочитать SUMMARY архитектуры

```
general_docs/02_architecture/000_SUMMARY.md
```

Это даёт контекст о других архитектурных решениях в проекте.

### Шаг 3: Создать ADR через скрипт

**ОБЯЗАТЕЛЬНО использовать скрипт:**

```bash
python scripts/decision_new.py -t "Название решения" -a "001" --domain "Backend/API"
```

**Параметры:**
- `-t` — название решения
- `-a` — ID связанной архитектуры (например: 001)
- `--domain` — область (UI/Frontend, Backend/API, Auth/Security, Database, Infrastructure, Architecture)
- `--desc` — краткое описание

**Скрипт автоматически:**
- Генерирует ID (DEC-001, DEC-002, ...)
- Создаёт файл по шаблону
- Обновляет индекс `000_decisions.md`
- Обновляет счётчик `.doc_counter`
- Устанавливает статус `🟡 draft`

**ЗАПРЕЩЕНО:** Создавать файлы вручную через Write tool!

### Шаг 4: Заполнить ADR из архитектуры

После создания файла:

1. Прочитать созданный ADR
2. Извлечь из архитектуры:
   - **Решение** — краткая суть принятого решения
   - **Контекст** — из "Цель и контекст"
   - **Обоснование** — почему выбрано это решение
   - **Альтернативы** — другие рассмотренные варианты (из дискуссии)
   - **Последствия** — что нужно реализовать
3. Заполнить секции:
   - Решение (1-2 предложения)
   - Контекст и требования
   - Обоснование (преимущества, недостатки)
   - Рассмотренные альтернативы
   - Последствия (задачи, изменения, риски)
4. Изменить статус: `🟡 draft` → `🟣 review`

### Шаг 5: Обновить архитектуру

После создания ADR:

1. Изменить статус архитектуры: `🟢 approved` → `⚪ final`
2. Обновить секцию "Связанные документы":
   ```markdown
   ## Связанные документы

   - **Дискуссия:** [001_*.md](../01_discuss/001_*.md)
   - **ADR:** [DEC-001_*.md](../04_decisions/DEC-001_*.md) ← ДОБАВИТЬ
   - **Ресурсы:** [Будут созданы из ADR]
   ```
3. Добавить в "История изменений"

### Шаг 6: Предложить следующие шаги

После заполнения ADR спросить пользователя:

```
ADR создан: DEC-XXX

Следующие шаги после одобрения ADR:
1. /resource — создать ресурсы (database, backend, frontend, infra)
2. /imp-plan — создать план реализации

Какие ресурсы нужно создать из этого ADR?
```

---

## Workflow статусов ADR

```
🟡 draft    → Документ создан, решение формируется
      ↓
🟣 review   → На ревью
      ↓
🟢 approved → Одобрено, можно создавать ресурсы и план
```

**Переходы:**

| Из | В | Когда | Действия |
|----|---|-------|----------|
| `🟡 draft` | `🟣 review` | ADR заполнен | Отправить на ревью |
| `🟣 review` | `🟢 approved` | Ревью пройдено | Можно создавать ресурсы |

## Команды

### Скрипты

```bash
# Создать ADR
python scripts/decision_new.py "Название"
python scripts/decision_new.py -t "Название" -a "001" --domain "Backend/API"
python scripts/decision_new.py -i  # интерактивно

# Удалить ADR
python scripts/decision_delete.py DEC-001
python scripts/decision_delete.py DEC-001 --force
```

### Makefile

```bash
make decision-new                              # интерактивно
make decision-new-topic TITLE="Название" ARCH="001"
make decision-index                            # показать индекс
make decision-delete ID="DEC-001"              # удалить
```

## Пример использования

**Архитектура:** 001_ui_styleguide.md (статус: approved)
**Решение:** Использовать Mantine для UI

**Шаг 1:** Проверить статус
```
Архитектура 001_ui_styleguide.md
Статус: 🟢 approved ✓
```

**Шаг 2:** Создать ADR
```bash
python scripts/decision_new.py -t "UI Framework - Mantine" -a "001" --domain "UI/Frontend"
```

**Шаг 3:** Заполнить из архитектуры
```markdown
## Решение

Использовать Mantine v7.x как основной UI-фреймворк для всех фронтенд-приложений.

## Контекст

Проекту требуется единая система компонентов для консистентного UI...

## Обоснование

1. Отличная кастомизация и темизация
2. Современный API с хуками
3. TypeScript first
```

**Шаг 4:** Обновить архитектуру
```markdown
**Статус:** ⚪ final

## Связанные документы

- **ADR:** [DEC-001_ui_framework_mantine.md](../04_decisions/DEC-001_ui_framework_mantine.md)
```

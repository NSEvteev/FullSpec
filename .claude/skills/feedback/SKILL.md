# Обратная связь по цепочке зависимостей

Скилл для отслеживания изменений и обновления связанных документов.

## Проблема

При изменении документа:
- Связанные документы могут стать неактуальными
- Нарушается консистентность цепочки
- Трудно отследить все зависимости

## Решение

`/feedback` автоматически:
1. Определяет тип изменённого документа
2. Находит связанные документы по цепочке
3. Создаёт задачи на обновление при необходимости

---

## Цепочка зависимостей

```
Дискуссия ← Архитектура ← ADR → Ресурсы → План → Документация
                          ↑
                    "разделитель"
```

**ADR — ключевая точка:**
- Изменения ВЫШЕ ADR (дискуссия, архитектура) → проверить ADR
- Изменения НИЖЕ ADR (ресурсы, план) → ADR обычно не меняется

---

## Когда использовать

**Триггеры:**
- Команда `/feedback`
- После существенного изменения документа
- Запрос "проверить связи" или "обновить цепочку"

**Контекст:**
- Документ был значительно изменён
- Нужно проверить консистентность связей

---

## Направления обратной связи

### Снизу вверх (от плана к дискуссии)

```
План изменился → проверить Ресурсы → проверить ADR
ADR существенно изменился → создать новую Дискуссию
```

**Когда использовать:**
- План не может быть реализован
- Обнаружены технические ограничения
- Требуется пересмотр решения

### Сверху вниз (от дискуссии к плану)

```
Дискуссия изменилась → обновить Архитектуру → обновить ADR → обновить Ресурсы → обновить План
```

**Когда использовать:**
- Изменились требования
- Новая информация
- Переосмысление решения

## Инструкции

### Шаг 1: Определить изменённый документ

```
/feedback general_docs/06_imp_plans/001_plan_auth.md
```

Или спросить:
```
Какой документ был изменён?
```

### Шаг 2: Определить тип и направление

**По типу документа:**

| Тип | Направление | Проверить |
|-----|-------------|-----------|
| План | Вверх | Ресурсы → ADR |
| Ресурс | Вверх | ADR |
| ADR | Вверх/Вниз | Архитектура, Ресурсы, План |
| Архитектура | Вверх/Вниз | Дискуссия, ADR |
| Дискуссия | Вниз | Архитектура |

### Шаг 3: Оценить существенность изменения

**Спросить пользователя:**

```
Какой тип изменения?

1. Мелкое (опечатки, форматирование) → не требует обновления
2. Среднее (уточнение деталей) → проверить связанные
3. Существенное (изменение решения) → обновить цепочку
```

### Шаг 4: Найти связанные документы

Прочитать секции "Связанные документы" в изменённом файле:

```markdown
## Связанные документы

- **ADR:** [DEC-001_*.md](../04_decisions/DEC-001_*.md)
- **Ресурсы:** [001_*.md](../05_resources/backend/001_*.md)
```

### Шаг 5: Создать задачи (если нужно)

**Для среднего изменения:**
```
Рекомендую проверить:
- [ ] ADR DEC-001 — убедиться в актуальности
- [ ] Ресурс 001_backend — проверить API
```

**Для существенного изменения:**
```bash
python scripts/task_new.py -t "Обновить цепочку: изменён план 001" -c docs -p high
```

### Шаг 6: Особый случай — существенное изменение ADR

Если ADR изменился существенно → создать новую дискуссию:

```bash
python scripts/discuss_new.py "Пересмотр решения DEC-001"
```

Содержимое дискуссии:
```markdown
## Исходный запрос пользователя

> Требуется пересмотр решения DEC-001 из-за [причина]

## Анализ цели и контекст

**Проблема:** Текущее решение [DEC-001] требует пересмотра.

**Причина:** [Описать причину пересмотра]

**Область обсуждения:**
- Альтернативы текущему решению
- Влияние на существующую реализацию
```

---

## Матрица влияния

| Изменился | Проверить | Действие |
|-----------|-----------|----------|
| Дискуссия | Архитектура | Если архитектура создана — проверить актуальность |
| Архитектура | ADR | Если ADR создан — проверить актуальность |
| ADR | Ресурсы, План | Проверить актуальность обоих |
| Ресурс | План, ADR | Если блокер — эскалировать в ADR |
| План | Ресурсы | Если не выполним — эскалировать |

## Пример использования

### Пример 1: Изменён план реализации

**Ситуация:** При реализации обнаружили, что выбранная библиотека не поддерживает нужную функциональность.

**Шаг 1:** Вызвать feedback
```
/feedback general_docs/06_imp_plans/001_plan_auth.md
```

**Шаг 2:** Оценить изменение
```
Тип изменения: Существенное (библиотека не подходит)
```

**Шаг 3:** Действия
1. Обновить ресурс backend — убрать неподходящую библиотеку
2. Проверить ADR — возможно нужно пересмотреть
3. Создать задачу на актуализацию

**Результат:**
```
Создана задача DOCS-00042: Пересмотреть ADR DEC-001 (библиотека не подходит)
Рекомендую создать дискуссию для выбора альтернативы.
```

### Пример 2: Уточнение в архитектуре

**Ситуация:** Добавили новый эндпоинт в архитектуру.

**Шаг 1:** Вызвать feedback
```
/feedback general_docs/02_architecture/001_api.md
```

**Шаг 2:** Оценить изменение
```
Тип изменения: Среднее (добавлен эндпоинт)
```

**Шаг 3:** Рекомендации
```
Рекомендую проверить:
- [ ] ADR — добавить эндпоинт в последствия
- [ ] Ресурс backend — добавить описание эндпоинта
- [ ] План — добавить задачу на реализацию
```

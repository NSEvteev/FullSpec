# Ревью документа на полноту

Скилл для проверки полноты и качества документа после завершения работы.

## Когда использовать

**Автоматическое применение:**
- После создания/редактирования файлов в `general_docs/`
- После создания/редактирования агентов в `.claude/agents/`
- После создания/редактирования скиллов в `.claude/skills/`
- После создания/редактирования инструкций в `llm_instructions/`
- После создания любых инструкций для LLM (файлы с "instruction", "llm", "claude" в пути)

**По запросу пользователя:**
- `/doc-review` — явный вызов
- "уделить внимание [документу]"
- "подумать над [документом]"
- "сделать ревью"
- "ультрасинк" / "ультрасинг"

## Инструкции

### Шаг 1: Определить тип документа

Прочитать документ и определить его тип:

| Тип | Расположение | Ключевые признаки |
|-----|--------------|-------------------|
| Дискуссия | `general_docs/01_discuss/` | Проблема → Варианты → Решение |
| Архитектура | `general_docs/02_architecture/` | Компоненты, зависимости, API |
| Decision (ADR) | `general_docs/04_decisions/` | Контекст, решение, последствия |
| Ресурс | `general_docs/05_resources/` | Схема, API, конфигурация |
| План | `general_docs/06_imp_plans/` | Подзадачи, зависимости, этапы |
| Скилл | `.claude/skills/` | Инструкции, примеры |
| Агент | `.claude/agents/` | Роль, задачи, скиллы |
| LLM-инструкция | `llm_instructions/` | Правила, шаблоны |

### Шаг 2: Проверить структурную полноту

**Для каждого типа документа проверить обязательные разделы:**

#### Дискуссия
- [ ] Метаданные (статус, участники, дата)
- [ ] Контекст/Проблема
- [ ] Варианты решения (минимум 2)
- [ ] Критерии оценки
- [ ] Принятое решение (если статус != draft)
- [ ] Связанные документы

#### Архитектура
- [ ] Обзор/Назначение
- [ ] Компоненты системы
- [ ] Взаимодействие компонентов
- [ ] API/Интерфейсы
- [ ] Зависимости
- [ ] Связанные документы

#### Decision (ADR)
- [ ] Статус (proposed/accepted/deprecated/superseded)
- [ ] Контекст
- [ ] Решение
- [ ] Последствия (плюсы и минусы)
- [ ] Связанные решения

#### Ресурс
- [ ] Назначение
- [ ] Структура/Схема
- [ ] Примеры использования
- [ ] Связанные документы

#### План реализации
- [ ] Цель
- [ ] Подзадачи с чекбоксами
- [ ] Зависимости между подзадачами
- [ ] Связанные документы (дискуссия, ADR)

#### Скилл
- [ ] Frontmatter (name, description, allowed-tools)
- [ ] Назначение
- [ ] Инструкции (пошаговые)
- [ ] Примеры использования (минимум 2)
- [ ] Связанные команды/файлы

#### Агент
- [ ] Frontmatter (name, description, skills)
- [ ] Роль и задачи
- [ ] Когда использовать
- [ ] Возможности
- [ ] Ограничения

### Шаг 3: Проверить смысловую полноту

Задать себе вопросы:

**Понятность:**
- [ ] Понятна ли цель документа без внешнего контекста?
- [ ] Определены ли все термины (или есть ссылки на глоссарий)?
- [ ] Есть ли примеры для сложных концепций?

**Полнота:**
- [ ] Рассмотрены ли все важные аспекты темы?
- [ ] Описаны ли edge cases и исключения?
- [ ] Указаны ли ограничения и допущения?

**Связность:**
- [ ] Есть ли ссылки на связанные документы?
- [ ] Соблюдается ли цепочка зависимостей?
- [ ] Обновлены ли индексы (000_*.md)?

**Актуальность:**
- [ ] Соответствует ли документ текущему состоянию кода/системы?
- [ ] Указана ли дата последнего обновления?

### Шаг 4: Сформировать отчёт

Вывести результат проверки:

```markdown
## Результат ревью: [название документа]

### Структура
✓ Все обязательные разделы присутствуют
— или —
✗ Отсутствуют разделы: [список]

### Полнота
✓ Документ полон
— или —
⚠ Рекомендации:
- [что добавить/уточнить]

### Связность
✓ Все связи в порядке
— или —
⚠ Требуется:
- Добавить ссылку на [документ]
- Обновить индекс [000_*.md]

### Итог
[Готов к использованию / Требуются доработки]
```

### Шаг 5: Предложить улучшения

Если найдены проблемы:

1. Показать отчёт пользователю
2. Предложить исправления:
   - Автоматические (добавить разделы, ссылки)
   - Требующие ввода (уточнить контент)
3. Спросить подтверждение перед изменениями

## Примеры использования

### Пример 1: Ревью новой дискуссии

**Сценарий:** Создана дискуссия `001_oauth_provider.md` о выборе OAuth-провайдера.

**Команда:**
```
/doc-review
```

**Вывод:**
```markdown
## Результат ревью: 001_oauth_provider.md

### Структура
✗ Отсутствуют разделы:
- Критерии оценки вариантов
- Связанные документы

### Полнота
⚠ Рекомендации:
- Добавить сравнительную таблицу провайдеров
- Указать требования к безопасности
- Описать fallback при недоступности провайдера

### Связность
⚠ Требуется:
- Добавить ссылку на архитектуру auth-сервиса
- Обновить индекс 000_discuss.md

### Итог
Требуются доработки. Исправить автоматически?
```

### Пример 2: Ревью нового скилла

**Сценарий:** Создан скилл `api-docs` для генерации API-документации.

**Команда:**
```
подумай над этим скиллом
```

**Вывод:**
```markdown
## Результат ревью: api-docs/SKILL.md

### Структура
✗ Отсутствуют разделы:
- Примеры использования (должно быть минимум 2)
- Связанные команды/файлы

### Полнота
⚠ Рекомендации:
- Добавить пример генерации для REST API
- Добавить пример для GraphQL
- Описать поддерживаемые форматы вывода

### Связность
⚠ Требуется:
- Добавить скилл в таблицу skills.md
- Назначить скилл агенту (если применимо)

### Итог
Требуются доработки.
```

### Пример 3: Ультрасинк после сессии работы

**Сценарий:** Пользователь завершил работу над несколькими документами и хочет убедиться, что ничего не упущено.

**Команда:**
```
ультрасинк
```

**Действие:**
1. Найти все изменённые .md файлы в текущей сессии
2. Провести ревью каждого файла
3. Проверить согласованность между документами
4. Вывести общий отчёт

**Вывод:**
```markdown
## Ультрасинк: проверка изменённых документов

### Проверено файлов: 3

| Файл | Статус | Проблемы |
|------|--------|----------|
| general_docs/01_discuss/002_caching.md | ⚠ | Нет связи с архитектурой |
| .claude/skills/cache-clear/SKILL.md | ✓ | — |
| llm_instructions/skills.md | ⚠ | Не добавлен cache-clear |

### Рекомендуемые действия:
1. Добавить ссылку на архитектуру в дискуссию
2. Добавить cache-clear в таблицу скиллов

Выполнить автоматически?
```

## Чек-лист для быстрой проверки

При нехватке времени можно использовать сокращённую проверку:

- [ ] Есть ли цель/назначение документа?
- [ ] Есть ли примеры?
- [ ] Есть ли ссылки на связанные документы?
- [ ] Обновлён ли индекс?
- [ ] Понятен ли документ без дополнительного контекста?

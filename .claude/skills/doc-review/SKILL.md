# Глубокое ревью документа

Скилл для **критического анализа** документа: не просто проверка структуры, а **осмысление содержания** и предложение улучшений.

## Философия

**Плохое ревью:** "Все секции на месте ✓"
**Хорошее ревью:** "Не рассмотрен вариант X, упущен критерий Y, нужно уточнить Z"

Задача — **думать** над документом, а не проверять галочки.

## Когда использовать

**Автоматическое применение:**
- После создания/редактирования файлов в `general_docs/`
- После создания/редактирования агентов в `.claude/agents/`
- После создания/редактирования скиллов в `.claude/skills/`
- После создания/редактирования инструкций в `llm_instructions/`

**По запросу пользователя:**
- `/doc-review` — явный вызов
- "уделить внимание [документу]"
- "подумать над [документом]"
- "сделать ревью"
- "ультрасинк" / "ультрасинг"

## Инструкции

### Шаг 1: Определить тип документа

| Тип | Расположение | Ключевые признаки |
|-----|--------------|-------------------|
| Дискуссия | `general_docs/01_discuss/` | Проблема → Варианты → Решение |
| Архитектура | `general_docs/02_architecture/` | Компоненты, зависимости, API |
| Decision (ADR) | `general_docs/04_decisions/` | Контекст, решение, последствия |
| Ресурс | `general_docs/05_resources/` | Схема, API, конфигурация |
| План | `general_docs/06_imp_plans/` | Подзадачи, зависимости, этапы |
| Скилл | `.claude/skills/` | Инструкции, примеры |
| Агент | `.claude/agents/` | Роль, задачи, скиллы |

### Шаг 2: Проверить структуру (быстро)

**Дискуссия:**
- [ ] Метаданные (статус, дата)
- [ ] Контекст/Проблема
- [ ] Варианты решения (минимум 2)
- [ ] Критерии оценки с весами
- [ ] Сравнительная таблица (⭐, итоги, места 🏆🥈🥉)
- [ ] Связанные документы

**Архитектура:**
- [ ] Обзор/Назначение
- [ ] Компоненты и взаимодействие
- [ ] API/Интерфейсы
- [ ] Зависимости

**Скилл/Агент:**
- [ ] Frontmatter
- [ ] Назначение и инструкции
- [ ] Примеры (минимум 2)

### Шаг 3: Критический анализ контента (ГЛАВНОЕ)

**Это ключевой шаг.** Нужно **думать** над содержанием, а не проверять формат.

#### Для дискуссий — задать вопросы:

**По вариантам:**
- Все ли релевантные альтернативы рассмотрены?
- Нет ли очевидного варианта, который упущен?
- Достаточно ли глубоко описаны плюсы/минусы?
- Есть ли "гибридный" вариант (комбинация)?

**По критериям:**
- Все ли важные критерии учтены?
- Релевантны ли веса критериев для данного проекта?
- Нет ли упущенных технических требований?
- Учтены ли нефункциональные требования (производительность, безопасность, масштабируемость)?

**По контексту:**
- Понятен ли контекст проекта? (стек, команда, сроки)
- Есть ли неявные допущения, которые нужно явно указать?
- Какие вопросы нужно задать пользователю для уточнения?

**Типичные упущения в дискуссиях:**
- Не рассмотрен вариант "сделать самим" / "не делать вообще"
- Не учтена кривая обучения для команды
- Не учтена поддержка и документация
- Не учтены лицензионные ограничения
- Не учтена совместимость со стеком проекта
- Не учтены edge cases и failure modes

#### Для архитектуры — задать вопросы:

- Понятно ли, как компоненты взаимодействуют?
- Описаны ли точки отказа и как их обрабатывать?
- Есть ли план масштабирования?
- Учтена ли безопасность?
- Понятно ли, где хранятся данные и как они защищены?

#### Для скиллов/агентов — задать вопросы:

- Понятно ли, когда использовать, а когда НЕ использовать?
- Покрывают ли примеры основные сценарии?
- Есть ли edge cases, которые не описаны?
- Понятны ли ограничения?

### Шаг 4: Сформировать отчёт

```markdown
## Ревью: [название документа]

### Структура
✓ Все секции на месте
— или —
✗ Отсутствует: [список]

### Критический анализ

**Упущенные варианты:**
- [Вариант, который стоит рассмотреть]
- [Ещё вариант]

**Упущенные критерии:**
- [Критерий, который важен для решения]
- [Ещё критерий]

**Вопросы для уточнения:**
- [Вопрос, на который нужен ответ для принятия решения]
- [Ещё вопрос]

**Неявные допущения:**
- [Допущение, которое стоит явно указать]

### Предложения по улучшению

1. **Добавить вариант:** [описание] — потому что [причина]
2. **Добавить критерий:** [название] — важен для [причина]
3. **Уточнить:** [что уточнить] — потому что [причина]

### Итог
[Готов / Требует доработки — что именно]
```

### Шаг 5: Внести улучшения

**Режим работы зависит от контекста:**

#### Автоматический режим (без подтверждения)
Применяется когда документ **только что создан** в этой же сессии:
- Документ создан через скрипт (discuss_new.py и т.д.)
- Документ создан через Write tool
- doc-review вызван сразу после создания

**В автоматическом режиме:**
1. Вывести отчёт с найденными улучшениями
2. **Сразу внести улучшения** без вопросов:
   - Добавить упущенные варианты в таблицу предложений
   - Добавить упущенные критерии
   - Дополнить сравнительную таблицу
   - Добавить вопросы в секцию "Дискуссия и анализ"
3. Показать итоговый результат

#### Интерактивный режим (с подтверждением)
Применяется когда:
- Явный вызов `/doc-review` пользователем
- Ревью существующего документа (не созданного в этой сессии)
- Команды "уделить внимание", "подумать", "ультрасинк"

**В интерактивном режиме:**
1. Вывести отчёт
2. Спросить пользователя, какие улучшения внести
3. Внести выбранные изменения

### Шаг 6: Предложить решение (только для дискуссий)

**После улучшений** — предложить пользователю выбрать решение.

**Алгоритм:**
1. Проанализировать сравнительную таблицу
2. Выбрать **топ-3 варианта** по итоговым баллам
3. Сформировать краткую аргументацию для каждого
4. Спросить пользователя через `AskUserQuestion`:

```
Какое решение принять?

Варианты:
1. [Лидер] — [краткая аргументация]
2. [2-е место] — [краткая аргументация]
3. [3-е место] — [краткая аргументация]
4. Другое (продолжить обсуждение / свой вариант)
```

**После выбора пользователя:**

1. **Если выбран вариант 1-3:**
   - Заполнить секцию "Принятое решение" выбранным вариантом
   - Заполнить секцию "Аргументация" обоснованием (по критериям)
   - Изменить статус: `🟡 draft` → `🟢 approved`
   - Обновить индекс дискуссий (000_discuss.md)

2. **Если выбран "Другое":**
   - Оставить статус `🟡 draft`
   - В секции "Принятое решение" оставить `[Ожидает обсуждения]`
   - Записать комментарий пользователя в "Дискуссия и анализ"

**Формат секций после принятия решения:**

```markdown
## Принятое решение

**Выбран вариант:** [Название варианта]

**Краткое обоснование:** [1-2 предложения почему этот вариант лучше]

## Аргументация

**Почему этот вариант:**
- [Критерий 1]: [почему хорошо]
- [Критерий 2]: [почему хорошо]
- [Критерий 3]: [почему хорошо]

**Почему не другие:**
- [Вариант 2]: [причина отказа]
- [Вариант 3]: [причина отказа]

**Риски и mitigation:**
- [Риск 1]: [как минимизировать]
```

### Формат статусов

**Использовать эмодзи для визуального различия:**

| Статус | Формат | Описание |
|--------|--------|----------|
| draft | `🟡 draft` | Новая дискуссия, решение не принято |
| in_progress | `🔵 in_progress` | Активное обсуждение |
| feedback | `🟠 feedback` | Ожидает обратной связи |
| review | `🟣 review` | На проверке |
| approved | `🟢 approved` | Решение принято |
| rejected | `🔴 rejected` | Отклонено |
| final | `⚪ final` | Завершено и реализовано |

**Workflow статусов для дискуссий:**
```
🟡 draft → 🔵 in_progress → 🟢 approved → ⚪ final
                ↓
           🟠 feedback (при вопросах)
                ↓
           🔵 in_progress
```

## Примеры

### Пример 1: Ревью дискуссии об UI-фреймворке

**Документ:** Дискуссия о выборе UI-фреймворка (shadcn/ui, MUI, Chakra, Ant Design)

**Плохое ревью:**
```
✓ Все секции на месте
✓ 4 варианта
✓ 6 критериев
✓ Сравнительная таблица есть
Готов к использованию.
```

**Хорошее ревью:**
```markdown
## Ревью: 001_ui_styleguide.md

### Структура
✓ Все секции на месте

### Критический анализ

**Упущенные варианты:**
- Headless UI + Tailwind — минималистичный подход без готовых стилей
- Radix UI — примитивы без стилей, максимальная гибкость
- Собственная дизайн-система — полный контроль, но больше работы

**Упущенные критерии:**
- Поддержка темизации (dark/light mode) — критично для современных приложений
- Совместимость со стеком (React/Next.js/Vue?) — не указан стек проекта
- Learning curve для команды — какой опыт у разработчиков?
- TypeScript поддержка — качество типов
- Мобильная адаптивность — responsive из коробки?

**Вопросы для уточнения:**
- Какой фронтенд-фреймворк используется? (React, Vue, Svelte?)
- Нужна ли поддержка SSR/SSG?
- Есть ли дизайнер в команде или нужны готовые стили?
- Какой опыт команды с Tailwind CSS?
- Нужна ли поддержка RTL (right-to-left)?

**Неявные допущения:**
- Предполагается React (shadcn/ui только для React)
- Предполагается, что bundle size важен (но насколько?)
- Не указаны требования к браузерной поддержке

### Предложения по улучшению

1. **Добавить вариант Headless UI** — если команда знает Tailwind, это легковесная альтернатива
2. **Добавить критерий "Темизация"** — dark mode сейчас стандарт
3. **Указать стек проекта** — от этого зависит выбор
4. **Добавить критерий "TypeScript"** — качество типов влияет на DX

### Итог
Требует доработки: уточнить стек проекта и добавить критерии темизации и TypeScript.
```

### Пример 2: Ревью скилла

**Документ:** Скилл для генерации API-документации

**Хорошее ревью:**
```markdown
## Ревью: api-docs/SKILL.md

### Структура
✗ Отсутствует: примеры использования (нужно минимум 2)

### Критический анализ

**Что не описано:**
- Какие форматы API поддерживаются? (REST, GraphQL, gRPC?)
- Куда сохраняется результат?
- Как обрабатываются ошибки парсинга?
- Нужен ли доступ к сети для получения схем?

**Упущенные сценарии:**
- Генерация для GraphQL схемы
- Генерация для OpenAPI/Swagger
- Обновление существующей документации (а не создание с нуля)

**Неочевидные edge cases:**
- Что делать с приватными эндпоинтами?
- Как обрабатывать deprecated методы?
- Как быть с эндпоинтами без документации в коде?

### Предложения по улучшению

1. **Добавить 2 примера:** REST API и GraphQL
2. **Указать поддерживаемые форматы** в описании
3. **Добавить секцию "Ограничения"** — что скилл НЕ делает

### Итог
Требует доработки: добавить примеры и описать поддерживаемые форматы.
```

### Пример 3: Ультрасинк

**Команда:** `ультрасинк`

**Действие:**
1. Найти все изменённые .md файлы в сессии
2. Провести критический анализ каждого
3. Проверить связность между документами
4. Выдать сводный отчёт с приоритизированными улучшениями

## Чек-лист критического мышления

При ревью всегда спрашивать себя:

- [ ] Что здесь **упущено**?
- [ ] Какие **альтернативы** не рассмотрены?
- [ ] Какие **вопросы** нужно задать для принятия решения?
- [ ] Какие **допущения** сделаны неявно?
- [ ] Что может **пойти не так**?
- [ ] Достаточно ли информации для **реализации**?

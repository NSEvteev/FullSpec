# Взятие Issue в работу

Команда для полного цикла работы с GitHub Issue: подготовка, выполнение задачи, закрытие.

**Связанные скиллы:**
- [issue-create](/.claude/skills/issue-create/SKILL.md) — создание Issue

- [issue-complete](/.claude/skills/issue-complete/SKILL.md) — закрытие Issue как выполненного
- [issue-delete](/.claude/skills/issue-delete/SKILL.md) — закрытие Issue как неактуального

## Формат вызова

```
/issue-execute <номер> [--no-branch] [--branch-name <имя>]
```

| Параметр | Описание | По умолчанию |
|----------|----------|--------------|
| `номер` | Номер Issue (обязательный) | — |
| `--no-branch` | Не создавать ветку | false |
| `--branch-name` | Кастомное имя ветки | Автогенерация |

### Состояние между запусками

Скилл сохраняет информацию о последнем взятом Issue в файл `.claude/state/last-issue-run.json`:

```json
{
  "issue": 123,
  "title": "[AUTH] Добавить OAuth авторизацию",
  "branch": "feature/123-oauth-auth",
  "started_at": "2026-01-20T10:30:00Z",
  "service": "auth",
  "type": "feature"
}
```

Это позволяет:
- `/issue-complete --last` — закрыть последний взятый Issue
- `/issue-review --last` — провести ревью последнего Issue

**Примеры:**
- `/issue-execute 123`
- `/issue-execute 123 --no-branch`
- `/issue-execute 123 --branch-name feature/oauth-login`

## Правила

### Что делает

**Правило:** Скилл выполняет полный цикл работы с Issue.

| Фаза | Действия | Шаги |
|------|----------|------|
| Подготовка | Назначить, пометить, создать ветку | 1-6 |
| Выполнение | Реализовать задачу (код/тесты/документация) | 7 |
| Завершение | Закрыть Issue через `/issue-complete` | 10 |

**Детальные действия подготовки:**

| Действие | Описание | Команда |
|----------|----------|---------|
| Назначить исполнителя | Назначить себя | `gh issue edit --add-assignee @me` |
| Добавить метку | Пометить как "в работе" | `gh issue edit --add-label "in-progress"` |
| Создать ветку | Для изоляции изменений | `git checkout -b feature/{num}-{slug}` |
| Добавить комментарий | Уведомить о начале работы | `gh issue comment --body "Начал работу"` |

**Связанная инструкция:** [issues.md](/.claude/instructions/git/issues.md)

### Формат ветки

**Правило:** Имя ветки формируется автоматически из номера и заголовка Issue.

```
{тип}/{номер}-{slug}
```

| Тип Issue | Префикс ветки |
|-----------|---------------|
| feature | `feature/` |
| bug | `fix/` |
| enhancement | `feature/` |

### Правила именования

**Slug** — первые 3-4 слова из заголовка в kebab-case:
- `[AUTH] Добавить OAuth авторизацию` → `oauth-auth`
- `[NOTIFY] Email не отправляется` → `email-not-sending`
- `[INFRA] Настроить CI pipeline` → `ci-pipeline`

**Примеры веток:**

| Issue | Ветка |
|-------|-------|
| #123 [AUTH] Добавить OAuth | `feature/123-oauth-auth` |
| #124 [NOTIFY] Баг с email | `fix/124-email-bug` |
| #125 [INFRA] CI pipeline | `feature/125-ci-pipeline` |

## Воркфлоу

### Шаг 1: Получить номер Issue

1. Из аргумента: `/issue-execute 123`
2. Проверить существование и статус: `gh issue view 123`

### Шаг 2: Показать Issue

```
📋 Issue #123

Заголовок: [AUTH] Добавить OAuth авторизацию
Статус: open
Метки: service:auth, feature
Назначен: —

Взять в работу? [Y/n]
```

### Шаг 3: Назначить исполнителя

```bash
gh issue edit 123 --add-assignee @me
```

### Шаг 4: Добавить метку in-progress

```bash
gh issue edit 123 --add-label "in-progress"
```

### Шаг 5: Создать ветку

**Если не указан `--no-branch`:**

1. Определить тип из меток (feature/bug)
2. Сформировать slug из заголовка
3. Создать ветку:

```bash
git checkout -b feature/123-oauth-auth
```

### Шаг 6: Добавить комментарий

```bash
gh issue comment 123 --body "🚀 Начал работу над задачей

Ветка: \`feature/123-oauth-auth\`"
```

### Шаг 6.5: Сохранить состояние

Сохранить информацию о текущем Issue в файл состояния:

```bash
mkdir -p .claude/state
cat > .claude/state/last-issue-run.json << 'EOF'
{
  "issue": 123,
  "title": "[AUTH] Добавить OAuth авторизацию",
  "branch": "feature/123-oauth-auth",
  "started_at": "2026-01-20T10:30:00Z",
  "service": "auth",
  "type": "feature"
}
EOF
```

**Поля:**
- `issue` — номер Issue
- `title` — заголовок Issue
- `branch` — имя созданной ветки (или `null` если `--no-branch`)
- `started_at` — время начала работы (ISO 8601)
- `service` — сервис из префикса заголовка
- `type` — тип задачи (feature/bug/enhancement)

### Шаг 7: Выполнить задачу

**Правило:** Непосредственное выполнение задачи, описанной в Issue.

1. Прочитать раздел "Описание" и "Критерии готовности" из body Issue
2. Определить тип задачи и необходимые действия:

| Тип задачи | Действия | Скиллы/инструменты |
|------------|----------|-------------------|
| Код | Написать/изменить код | Edit, Write, Bash |
| Тесты | Написать тесты, запустить | Edit, Write, Bash |
| Документация | Создать/обновить документы | /doc-create, /doc-update |
| Рефакторинг | Изменить структуру кода | Edit, Read, Grep |
| Инфраструктура | Настроить конфиги, CI/CD | Edit, Write, Bash |
| Issue/PR | Создать связанные задачи | /issue-create, gh pr |

3. Выполнить задачу:
   - Использовать соответствующие скиллы и инструменты
   - Следовать инструкциям из [/.claude/instructions/](/.claude/instructions/)
   - Коммитить изменения по правилам [commits.md](/.claude/instructions/git/commits.md)
   - Формат веток — см. [workflow.md](/.claude/instructions/git/workflow.md)

4. После выполнения — перейти к проверке (Шаг 8)

### Шаг 8: Проверка по чек-листу

Перед завершением проверить выполнение всех пунктов:

```
✅ Шаг 1: Получил номер Issue
✅ Шаг 2: Показал Issue и получил подтверждение
✅ Шаг 3: Назначил исполнителя
✅ Шаг 4: Добавил метку in-progress
✅ Шаг 5: Создал ветку
✅ Шаг 6: Добавил комментарий
✅ Шаг 7: Выполнил задачу
⏳ Шаг 10: Вызов /issue-review...
```

### Шаг 9: Результат

Вывести полный отчёт:

```
✅ Issue #123 взят в работу и выполнен

Выполнено:
- Назначен: @username
- Добавлена метка: in-progress
- Создана ветка: feature/123-oauth-auth
- Добавлен комментарий
- Задача выполнена

Текущая ветка: feature/123-oauth-auth
```

### Шаг 10: Ревью и завершение Issue

**Правило:** После выполнения задачи автоматически вызвать скилл ревью.

```
Задача выполнена. Запускаю /issue-review {номер}...
```

Вызвать скилл [/issue-review](/.claude/skills/issue-review/SKILL.md) с номером Issue:

```
/issue-review 123
```

Скилл issue-review:
1. Проверит качество кода (линтинг, типы)
2. Оценит реализацию (можно ли улучшить)
3. Проверит тесты
4. Проверит критерии готовности из Issue
5. Если всё ✅ — вызовет `/issue-complete` автоматически
6. Если есть проблемы — сообщит и не закроет Issue

## Обработка ошибок

| Ошибка | Действие |
|--------|----------|
| Номер не указан | Спросить: "Какой Issue взять в работу?" |
| Issue не существует | Сообщить: "Issue #{num} не найден" |
| Issue закрыт | Предупредить, предложить переоткрыть |
| Issue уже в работе | Показать информацию, предложить переназначить |
| Ветка уже существует | Предложить: переключиться / создать с другим именем |
| Есть несохранённые изменения | Предупредить: "Сохраните изменения перед переключением" |
| gh не авторизован | Сообщить: "Выполните: gh auth login" |

**Откат при ошибке:**

```bash
# Удалить созданную ветку (если была создана)
git branch -D feature/123-oauth-auth

# Удалить метку (если была добавлена)
gh issue edit 123 --remove-label "in-progress"

# Удалить назначение (если было добавлено)
gh issue edit 123 --remove-assignee @me
```

## FAQ / Troubleshooting

### Задача слишком большая — что делать?

**Проблема:** Issue содержит слишком много работы для одного PR.

**Решение:**
1. Разбить Issue на подзадачи через `/issue-create`
2. Связать подзадачи в описании родительского Issue
3. Закрыть родительский Issue после выполнения всех подзадач

### Как отменить взятие Issue в работу?

**Сценарий:** Нужно вернуть Issue в статус "открыт".

**Команды:**
```bash
# Удалить метку in-progress
gh issue edit {номер} --remove-label "in-progress"

# Снять назначение
gh issue edit {номер} --remove-assignee @me

# Удалить локальную ветку (если не нужна)
git branch -D feature/{номер}-{slug}
```

### Что делать если забыл номер Issue?

**Решение:**
```bash
# Список открытых Issue, назначенных на меня
gh issue list --assignee @me --state open

# Поиск по ключевому слову
gh issue list --search "OAuth"
```

### Ветка создана, но Issue не помечен — что делать?

**Причина:** Ошибка при выполнении gh команд.

**Решение:**
```
/issue-update {номер} --add-label "in-progress"
```

Или вручную:
```bash
gh issue edit {номер} --add-assignee @me --add-label "in-progress"
gh issue comment {номер} --body "🚀 Начал работу, ветка: feature/{номер}-{slug}"
```

### Как работать над несколькими Issue одновременно?

**Рекомендация:** Не рекомендуется, но если необходимо:

1. Каждый Issue — отдельная ветка
2. Переключение: `git checkout feature/{номер}-{slug}`
3. Stash для сохранения незакоммиченных изменений

### issue-review не вызывается автоматически — почему?

**Причина:** Скилл `issue-review` вызывается автоматически только после Шага 7 (выполнение задачи).

**Решение:** Вызвать вручную:
```
/issue-review {номер}
```

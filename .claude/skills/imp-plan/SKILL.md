# Управление планами реализации

Скилл для создания планов реализации на основе одобренных ADR (Architecture Decision Records).

## ВАЖНОЕ ПРАВИЛО

**При ЛЮБЫХ действиях с планами — сначала прочитать этот скилл!**

Это включает:
- Создание нового плана
- Изменение существующего плана
- Изменение статуса плана
- Работу с индексом `000_imp_plans.md`

**Почему:** Скрипты автоматически обновляют счётчики и индексы. Ручные операции ломают синхронизацию.

---

## Когда использовать

**Триггеры:**
- Команда `/imp-plan`
- После одобрения ADR (`🟢 approved`)
- После создания ресурсов (`/resource`)
- Запрос "создать план" или "план реализации"

**Контекст:**
- ADR находится в статусе `🟢 approved`
- Ресурсы созданы (опционально, но рекомендуется)

---

## Workflow в цепочке зависимостей

```
ADR (approved)
      ↓
/resource + /imp-plan (этот скилл!)
      ↓
Ресурсы (05_resources/) + План (06_imp_plans/)
      ↓
Реализация → Документация папок (README.md)
```

**План создаётся из ADR, а НЕ из архитектуры напрямую!**

---

## Инструкции

### Шаг 1: Проверить ADR

1. Прочитать ADR и убедиться:
   - Статус `🟢 approved`
   - Секция "Последствия" заполнена (что нужно реализовать)

**Если статус не approved:**
```
ADR должен быть в статусе approved для создания плана.
Текущий статус: [статус]
```

### Шаг 2: Определить фазы и задачи

Из секции "Последствия" ADR извлечь:
- Что нужно реализовать
- Какие зависимости между задачами
- Какие критерии готовности

**Типичные фазы:**
1. **Подготовка** — настройка окружения, зависимости
2. **Реализация** — основная работа
3. **Тестирование** — тесты, QA
4. **Документация** — обновление docs

### Шаг 3: Создать план через скрипт

**ОБЯЗАТЕЛЬНО использовать скрипт:**

```bash
python scripts/imp_plan_new.py -t "Название плана" -a "DEC-001" --domain "Backend/API"
```

**Параметры:**
- `-t` — название плана
- `-a` — ID связанного ADR (например: DEC-001)
- `--domain` — область (UI/Frontend, Backend/API, Auth/Security, Database, Infrastructure, Architecture)
- `--desc` — краткое описание

**Скрипт автоматически:**
- Генерирует ID (001, 002, ...)
- Создаёт файл по шаблону
- Обновляет индекс `000_imp_plans.md`
- Обновляет счётчик `.doc_counter`
- Устанавливает статус `🟡 draft`, прогресс 0%

**ЗАПРЕЩЕНО:** Создавать файлы вручную через Write tool!

### Шаг 4: Заполнить план

После создания файла:

1. Прочитать созданный план
2. Извлечь из ADR:
   - **Обзор** — что будет реализовано
   - **Задачи** — из секции "Последствия"
   - **Критерии готовности** — из требований
3. Разбить на фазы:
   - Подготовка
   - Реализация
   - Тестирование
   - Документация
4. Для каждой задачи указать:
   - Номер (1.1, 1.2, 2.1, ...)
   - Название
   - Описание
   - Зависимости
   - Статус: `task_planned`

### Шаг 5: Связать с ресурсами

Если ресурсы уже созданы:

```markdown
## Связанные документы

- **ADR:** [DEC-001_*.md](../04_decisions/DEC-001_*.md)
- **Ресурсы:**
  - [001_users_schema.md](../05_resources/database/001_users_schema.md)
  - [002_auth_api.md](../05_resources/backend/002_auth_api.md)
```

### Шаг 6: Обновить ADR

После создания плана:

1. Обновить секцию "Связанные документы" в ADR:
   ```markdown
   **Связанные документы:**
   - **Архитектура:** [001_*.md](../02_architecture/001_*.md)
   - **Ресурсы:** [ссылки на ресурсы]
   - **План реализации:** [001_plan_*.md](../06_imp_plans/001_plan_*.md) ← ДОБАВИТЬ
   ```

## Workflow статусов плана

```
🟡 draft        → План создан, задачи не определены
      ↓
🔵 in_progress  → Работа по плану идёт
      ↓
🟣 review       → План на проверке
      ↓
🧪 test         → Тестирование
      ↓
🟢 approved     → Одобрено
      ↓
⚪ final        → План завершён, всё реализовано
```

**Статусы задач:**
- `task_planned` — запланирована
- `task_in_progress` — в работе
- `task_testing` — на тестировании
- `task_done` — завершена
- `task_blocked` — заблокирована

## Команды

### Скрипты

```bash
# Создать план
python scripts/imp_plan_new.py -t "Название" -a "DEC-001" --domain "Backend/API"
python scripts/imp_plan_new.py -i  # интерактивно

# Удалить план
python scripts/imp_plan_delete.py 001
python scripts/imp_plan_delete.py 001 --force
```

### Makefile

```bash
make imp-plan-new                              # интерактивно
make imp-plan-new-topic TITLE="Auth System" ADR="DEC-001"
make imp-plan-index                            # показать индекс
make imp-plan-delete ID="001"                  # удалить
```

## Пример использования

**ADR:** DEC-001_auth_jwt.md (статус: approved)

**Последствия из ADR:**
- Настроить JWT библиотеку
- Создать таблицу users
- Реализовать эндпоинты auth
- Написать тесты
- Обновить документацию API

**Шаг 1:** Создать план
```bash
python scripts/imp_plan_new.py -t "Реализация JWT аутентификации" -a "DEC-001" --domain "Auth/Security"
```

**Шаг 2:** Заполнить задачи
```markdown
## Задачи

### Фаза 1: Подготовка

| # | Задача | Описание | Зависимости | Статус |
|---|--------|----------|-------------|--------|
| 1.1 | Установить jsonwebtoken | npm install jsonwebtoken | — | task_planned |
| 1.2 | Настроить переменные окружения | JWT_SECRET, JWT_EXPIRES | 1.1 | task_planned |

### Фаза 2: Реализация

| # | Задача | Описание | Зависимости | Статус |
|---|--------|----------|-------------|--------|
| 2.1 | Создать миграцию users | Таблица users с полями | 1.2 | task_planned |
| 2.2 | Реализовать /auth/login | POST эндпоинт | 2.1 | task_planned |
| 2.3 | Реализовать /auth/refresh | POST эндпоинт | 2.2 | task_planned |
| 2.4 | Реализовать /auth/logout | POST эндпоинт | 2.3 | task_planned |

### Фаза 3: Тестирование

| # | Задача | Описание | Зависимости | Статус |
|---|--------|----------|-------------|--------|
| 3.1 | Unit тесты auth service | Тесты для auth.service.ts | 2.4 | task_planned |
| 3.2 | Integration тесты | Тесты для API эндпоинтов | 3.1 | task_planned |

### Фаза 4: Документация

| # | Задача | Описание | Зависимости | Статус |
|---|--------|----------|-------------|--------|
| 4.1 | Обновить API docs | Swagger/OpenAPI | 3.2 | task_planned |
| 4.2 | Обновить README | Инструкции по использованию | 4.1 | task_planned |

## Критерии готовности

- [ ] Все эндпоинты работают
- [ ] Тесты написаны и проходят (coverage > 80%)
- [ ] Документация обновлена
- [ ] Code review пройдено
```

## Формат вызова

```
/skill-create [название] [--dry-run] [--auto]
```

| Параметр | Описание | По умолчанию |
|----------|----------|--------------|
| `название` | Имя скилла в формате `{объект}-{действие}` | Спросить |
| `--dry-run` | Показать план без создания | false |
| `--auto` | Автоматический режим (применить все улучшения, пропустить подтверждения) | false |

> 💡 **Совет:** Если запрос неоднозначный или требует уточнения, используйте [/prompt-update](/.claude/skills/prompt-update/SKILL.md) для улучшения промта перед созданием скилла:
> ```
> /prompt-update создай скилл для работы с тестами
> ```

### Режим --auto

В автоматическом режиме:
- Категория определяется автоматически (первая подходящая)
- Метаданные генерируются без подтверждения
- Ревью-предложения применяются автоматически (все рекомендованные)
- Связь с агентами пропускается
- Подтверждения не запрашиваются

```
/skill-create my-skill --auto

[Создание без интерактивных вопросов]

✅ Скилл создан: /.claude/skills/my-skill/SKILL.md
```

**Режим --dry-run:**
```
/skill-create my-skill --dry-run

📋 Предварительный просмотр (--dry-run)

Будет создано:
- /.claude/skills/my-skill/SKILL.md

Будет обновлено:
- /.claude/instructions/tools/skills.md (добавлена строка в таблицу)

Будет вызвано:
- /links-update /.claude/skills/my-skill/SKILL.md
- /skill-update my-skill

ℹ️ Изменения НЕ применены (--dry-run)
```

## Правила

### Правило одного действия

**Скилл = 1 действие над объектом.**

| Действие | Описание |
|----------|----------|
| `create` | Создание нового объекта |
| `update` | Обновление существующего |
| `delete` | Удаление объекта |

Если пользователь просит создать скилл "для управления X" (подразумевая create + update + delete) — **разделить на 3 скилла**, объяснив правило.

### Формат названия

```
{объект}-{действие}
```

- Только латиница, цифры, дефис
- Примеры: `skill-create`, `agent-update`, `doc-delete`

### Скрипты Python

Повторяемые операции (создание файлов, поиск, валидация) выносить в [scripts/](/.claude/scripts/).

**Требование:** Если скилл использует Python-скрипт, в разделе "Воркфлоу" скилла **обязательно** описать:
1. Когда вызывается скрипт (на каком шаге)
2. Какие аргументы передаются
3. Что возвращает скрипт
4. Как обрабатывается результат

**Пример описания в воркфлоу:**

> ### Шаг 4: Создать файлы
>
> Вызвать скрипт:
> ```bash
> python .claude/scripts/skill_create.py --name {название} --category {категория}
> ```
>
> Аргументы:
> - `--name` — название скилла
> - `--category` — категория
>
> Результат: создаёт папку и SKILL.md, возвращает путь к файлу.

### Ссылки на файлы и папки

**Правило:** Все файлы и папки репозитория, упоминаемые в тексте документа, должны быть оформлены как ссылки.

**Неправильно:**
```
Добавить скилл в /.claude/instructions/tools/skills.md
```

**Правильно:**
```
Добавить скилл в [skills.md](/.claude/instructions/tools/skills.md)
```

**Связанные скиллы:**
- [links-create](/.claude/skills/links-create/SKILL.md) — создание ссылок в документе
- [links-update](/.claude/skills/links-update/SKILL.md) — обновление ссылок в связанных документах

## Воркфлоу

### Шаг 0: Проверка пересечений

1. Прочитать индекс скиллов: [skills.md](/.claude/instructions/tools/skills.md)
2. Сравнить описание нового скилла с существующими
3. Если есть пересечение:
   - Изучить пересекающиеся скиллы
   - Вывести итог пользователю
   - Спросить: продолжить / отменить / изменить название

### Шаг 0b: Fail-fast проверки (БЛОКИРУЮЩИЕ)

> ⚠️ **КРИТИЧНО:** Эти проверки выполняются ДО начала создания скилла. При любой ошибке — немедленная остановка.

**Проверки:**

| Проверка | Команда | При ошибке |
|----------|---------|------------|
| Папка `.claude/skills/` существует | `ls .claude/skills/` | Создать или сообщить об ошибке |
| Git status чистый в `.claude/` | `git status .claude/ --porcelain` | Предупредить о несохранённых изменениях |
| Скилл с таким именем не существует | `ls .claude/skills/{название}/` | Спросить: перезаписать / отменить |
| Формат названия корректен | Regex: `^[a-z]+(-[a-z]+)+$` | Показать примеры правильного формата |

**Формат вывода при ошибке:**

```
❌ Fail-fast проверка не пройдена

Проверка: {название проверки}
Проблема: {описание}

Решение: {что сделать}

Продолжить невозможно. Исправьте проблему и попробуйте снова.
```

**Формат вывода при успехе:**

```
✅ Все предварительные проверки пройдены

- Папка .claude/skills/ существует
- Git status чистый
- Скилл {название} не существует
- Формат названия корректен

Продолжаю создание...
```

### Шаг 1: Получить название

1. Из аргумента: `/skill-create foo-create`
2. Или спросить: "Как назвать скилл?"
3. Валидация:
   - Формат `{объект}-{действие}`
   - Допустимые символы: `a-z`, `0-9`, `-`
   - Действие: `create`, `update`, `delete`

Если пользователь хочет несколько действий — разделить на отдельные скиллы.

### Шаг 2: Категория

1. Показать список существующих категорий из [skills.md](/.claude/instructions/tools/skills.md)
2. Предложить подходящую категорию
3. Если пользователь хочет новую — он должен явно сказать "новая категория: {название}"

**Категории:** см. [skills.md](/.claude/instructions/tools/skills.md) раздел "Категории"

### Шаг 3: Метаданные

1. Сгенерировать метаданные на основе названия и категории
2. Вывести на подтверждение пользователю
3. Если пользователь хочет изменить — скорректировать

| Поле | Описание |
|------|----------|
| `description` | Краткое описание (1 строка) |
| `allowed-tools` | Необходимые инструменты |
| `triggers.commands` | Команды для вызова |
| `triggers.phrases` | Фразы для активации (ru/en) |

**Пример вывода на подтверждение:**
```
Метаданные скилла agent-create:

description: Создание нового агента по шаблону
allowed-tools: Bash, Write, Read, Edit, Glob, Grep
triggers:
  commands: /agent-create
  phrases (ru): создай агента, новый агент
  phrases (en): create agent, new agent

Подтвердить? (да / изменить)
```

**Справочник allowed-tools:**
- `Bash` — выполнение команд
- `Read` — чтение файлов
- `Write` — создание файлов
- `Edit` — редактирование файлов
- `Glob` — поиск файлов по паттерну
- `Grep` — поиск в содержимом файлов
- `WebFetch` — запросы к URL
- `WebSearch` — поиск в интернете
- `Task` — запуск подзадач

### Шаг 4: Создать файлы

```bash
mkdir -p .claude/skills/{название}
```

Создать `SKILL.md` по шаблону (см. ниже).

Если нужны скрипты Python — создать в [scripts/](/.claude/scripts/) как `{название}.py`.

### Шаг 5: Наполнение содержимым

После создания скелета SKILL.md необходимо заполнить его реальным содержимым:

**1. Воркфлоу:**
- Определить последовательность шагов для выполнения скилла
- Для каждого шага описать конкретные действия
- Указать входные данные и ожидаемый результат
- Добавить примеры команд и вывода

**2. Правила:**
- Сформулировать специфичные правила для этого скилла
- Описать ограничения и граничные случаи
- Указать зависимости от других скиллов/инструментов

**3. Примеры использования:**
- Добавить реалистичные примеры: минимум 1
- Покрыть основной сценарий использования
- Покрыть граничные случаи (ошибки, особые ситуации)
- Для каждого примера: команда → диалог → результат

**4. Чек-лист (если нужен):**
- Создать список проверок для выполнения скилла
- Упорядочить по шагам воркфлоу

### Шаг 6: Ревью

Перед валидацией LLM проводит автоматическое ревью созданного скилла и предлагает улучшения.

**Что проверять:**

1. **Полнота воркфлоу:**
   - Все ли шаги описаны достаточно подробно?
   - Нет ли пропущенных шагов между действиями?
   - Понятно ли, что делать на каждом шаге?

2. **Качество правил:**
   - Достаточно ли правил для корректной работы?
   - Нет ли противоречий между правилами?
   - Покрыты ли граничные случаи?

3. **Примеры:**
   - Достаточно ли примеров для понимания?
   - Покрывают ли примеры типичные сценарии?
   - Есть ли примеры обработки ошибок?

4. **Интеграция:**
   - Корректно ли описаны связи с другими скиллами?
   - Нужны ли дополнительные зависимости?

**Формат вывода:**

```
📋 Ревью скилла {название}

✅ Сильные стороны:
- {что хорошо}

💡 Предложения по улучшению (по приоритету):

🔴 Критично (блокирует качество):
1. {проблема функциональности или безопасности}
2. {отсутствует обязательный раздел}

🟡 Рекомендуется (улучшает качество):
3. {улучшение UX}
4. {добавить пример}

🟢 По желанию (nice to have):
5. {косметическое улучшение}

❓ Вопросы для уточнения:
- {вопрос, если есть}

Применить предложения? (все / критичные / выборочно / пропустить)
```

**Приоритеты предложений:**

| Приоритет | Когда использовать | Пример |
|-----------|-------------------|--------|
| 🔴 Критично | Влияет на работоспособность, безопасность, нарушает стандарты | Отсутствует обработка ошибок |
| 🟡 Рекомендуется | Улучшает качество, UX, поддерживаемость | Добавить пример обработки ошибки |
| 🟢 По желанию | Косметические улучшения, стиль | Переформулировать описание |

**В режиме --auto:** применяются только 🔴 Критично и 🟡 Рекомендуется.

**Важно:** Ревью блокирует! создание скилла. Пользователь должен:
- Либо применить все предложения
- Либо выбрать отдельные улучшения
- Либо явно отказаться от предложений

### Шаг 7: Валидация

Проверить созданный SKILL.md по чек-листу:

- [ ] Есть описание назначения

**Вёрстка:**

- [ ] Нет вложенных блоков кода (``` внутри ```)

- [ ] Все блоки кода закрыты
- [ ] Все файлы и папки репозитория оформлены как ссылки

### Шаг 8: Связь с агентами

1. Прочитать [agents.md](/.claude/instructions/tools/agents.md)
2. Найти агентов с тегами, соответствующими категории скилла
3. Предложить: "Скилл `{название}` относится к `{категория}`. Назначить агенту `{агент}`?"
4. Если да — добавить скилл в `skills:` агента

### Шаг 9: Синхронизация ссылок

> **Параллелизм:** Шаги 9a и 9b могут выполняться параллельно, так как анализируют разные наборы файлов.

**Шаг 9a:** Вызвать скилл [/links-update](/.claude/skills/links-update/SKILL.md) для обновления связанных документов:

```
/links-update .claude/skills/{название}/SKILL.md
```

**Что обновляется автоматически:**
- [skills.md](/.claude/instructions/tools/skills.md) — добавить в таблицу категории
- [agents.md](/.claude/instructions/tools/agents.md) — если назначен агенту (Шаг 8)
- Связанные скиллы (`{объект}-update`, `{объект}-delete`) — добавить ссылку в "Связанные скиллы"
- **Связанные инструкции** — обновить инструкции, указанные в "Связанные инструкции" скилла

**Обновление связанных инструкций (ОБЯЗАТЕЛЬНО):**

1. Прочитать раздел "Связанные инструкции" в созданном SKILL.md
2. Для каждой связанной инструкции:
   - Прочитать инструкцию
   - Найти таблицы/разделы, где упоминаются скиллы той же категории
   - Добавить новый скилл в соответствующие места
3. Показать пользователю изменения и получить подтверждение

**Пример:**
```
📋 Обновление связанных инструкций

Скилл: issue-review
Связанные инструкции: git/issues.md

📄 git/issues.md
   Найдены разделы для обновления:
   - Таблица Workflow (строка 103) — добавить issue-review
   - Таблица Скиллы (строка 117) — добавить issue-review
   - Раздел описаний скиллов — добавить /issue-review

Применить? [Y/n]
```

**Ожидаемый вывод:**
```
✅ Ссылки обновлены для {название}

Обновлено документов: {N}
- skills.md — добавлен в категорию {категория}
- {объект}-update/SKILL.md — добавлена ссылка
- {объект}-delete/SKILL.md — добавлена ссылка

Обновлено инструкций: {M}
- {инструкция}.md — добавлен скилл в таблицы и разделы
```

**Шаг 9b:** Вызвать скилл [/skill-update](/.claude/skills/skill-update/SKILL.md) для интеграции нового скилла в существующие:

```
/skill-update {название}
```

**Что делает skill-update:**
1. Анализирует все существующие скиллы
2. Определяет, где новый скилл может быть полезен
3. Предлагает добавить вызов нового скилла в воркфлоу существующих
4. После подтверждения — вносит изменения

**Критерии применимости:**
- Любая категория скиллов
- Логическое дополнение воркфлоу

**Ожидаемый вывод:**
```
📋 Предложения по интеграции скилла {название}

Проанализировано скиллов: {N}
Найдено применимых: {M}

📄 {существующий-скилл} — Шаг {X}
   Польза: {описание}

Применить? [Y/n/выборочно]
```

**После применения:**
```
✅ Интеграция скилла {название} завершена

Обновлено скиллов: {K}
- {скилл-1} — добавлен в Шаг {X}
- {скилл-2} — добавлен в Шаг {Y}

## Обработка ошибок

> **SSOT:** [error-handling.md](/.claude/templates/error-handling.md) — паттерны обработки ошибок

Если на любом шаге произошла ошибка:

1. **Сообщить пользователю:**
   ```
   ❌ Ошибка создания скилла на шаге {N}: {описание ошибки}

   Процесс не завершён. Откатываю изменения...
   ```

2. **Откатить все изменения:**

   ```bash
   # Удалить папку скилла
   rm -rf .claude/skills/{название}/

   # Удалить скрипт (если был создан)
   rm -f .claude/scripts/{название}.py

   # Откатить изменения в skills.md и agents.md через git
   git checkout -- .claude/instructions/tools/skills.md
   git checkout -- .claude/instructions/tools/agents.md

   # Или если изменения уже закоммичены
   git revert HEAD --no-commit
   ```

3. **Подтвердить откат:**
   ```
   Изменения откачены. Можно попробовать снова.
   ```

**Типичные ошибки:**
- Скилл с таким названием уже существует
- Невалидный формат названия
- Не найден файл `skills.md` или `agents.md`
- Пользователь отменил операцию

## FAQ / Troubleshooting

### Когда использовать --auto?

| Ситуация | Режим |
|----------|-------|
| Первое создание, нужно понять процесс | Без флагов |
| Опытный пользователь, стандартный скилл | `--auto` |
| Проверка плана | `--dry-run` |

### Как создать несколько связанных скиллов?

Если нужны `foo-create`, `foo-update`, `foo-delete`:

1. Создайте первый: `/skill-create foo-create`
2. При создании второго `/skill-create foo-update` — он автоматически добавит ссылки на `foo-create` в "Связанные скиллы"
3. `/skill-update` будет вызван для обновления связей

### Что если скилл с таким именем уже существует?

Скилл предупредит на Шаге 0 и предложит:
- Использовать `/skill-update` для обновления
- Выбрать другое имя
- Отменить операцию

### Как добавить скилл к агенту?

На Шаге 8 скилл спросит, назначить ли агенту. Или позже:
1. Откройте `/.claude/agents/{agent}.md`
2. Добавьте скилл в `skills:` frontmatter

### Что делать, если валидация не прошла?

На Шаге 7 скилл проверяет:
- Корректность frontmatter
- Наличие обязательных разделов
- Синтаксис markdown

Если ошибка — исправьте и запустите `/skill-update` для повторной валидации.

### Как отменить созданный скилл?

```bash
rm -rf /.claude/skills/{skill-name}/
# Откатить изменения в skills.md
git checkout -- /.claude/instructions/tools/skills.md
```

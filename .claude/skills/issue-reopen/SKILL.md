## Формат вызова

```
/issue-reopen <#номер> ["причина"] [--assign] [--dry-run] [--json]
```

| Параметр | Описание | По умолчанию |
|----------|----------|--------------|
| `#номер` | Номер Issue | — (обязательный) |
| `"причина"` | Комментарий о причине переоткрытия | — |
| `--assign` | Назначить Issue на себя после переоткрытия | false |
| `--dry-run` | Показать план без изменений | false |
| `--json` | JSON формат вывода | false |
| `--priority` | Установить приоритет (high, medium, low) | — |

**Примеры:**
- `/issue-reopen #123` — простое переоткрытие
- `/issue-reopen #123 "Задача не была полностью выполнена"` — с причиной
- `/issue-reopen #123 --assign` — переоткрыть и назначить на себя
- `/issue-reopen #123 --dry-run` — предварительный просмотр
- `/issue-reopen #123 --priority high "Баг вернулся"` — с приоритетом

## Когда использовать

| Ситуация | Действие | Пример |
|----------|----------|--------|
| Задача закрыта преждевременно | `/issue-reopen` | Не все критерии выполнены |
| Найдены новые требования | `/issue-reopen` с комментарием | Заказчик добавил требования |
| Баг воспроизвёлся снова | `/issue-reopen --priority high` | Регрессия после деплоя |
| Решение оказалось неполным | `/issue-reopen --assign` | Нужна доработка |
| Ошибочное закрытие | `/issue-reopen` | Случайно закрыли |

**Не использовать когда:**
- Нужна совершенно новая задача → `/issue-create`
- Issue удалён (не закрыт) → невозможно переоткрыть
- Issue уже открыт → скилл сообщит об этом

## Правила

### Принцип сохранения контекста

**Правило:** При переоткрытии Issue всегда добавлять комментарий с причиной, если она не очевидна.

**Причина:** Это помогает:
- Понять, почему задача была переоткрыта
- Отследить историю изменений
- Принять решение о приоритете

### Рекомендуемые причины

| Тип | Шаблон комментария |
|-----|-------------------|
| Неполное решение | "Не все критерии готовности выполнены: {список}" |
| Новые требования | "Добавлены новые требования: {список}" |
| Регрессия | "Баг воспроизвёлся после {событие}: {описание}" |
| Ошибочное закрытие | "Закрыто по ошибке" |
| Отклонено ревью | "Решение отклонено на ревью: {причина}" |

### Автоматический приоритет

**Правило:** Если Issue переоткрывается из-за бага или регрессии — предлагать `priority:high`.

```
⚠️ Обнаружена регрессия

Рекомендуется установить высокий приоритет.

Установить priority:high? [Y/n]
```

### Шаг 0: Проверки

> **SSOT:** [environment-check](/.claude/skills/environment-check/SKILL.md), [input-validate](/.claude/skills/input-validate/SKILL.md)

**0a. Проверка окружения:**

```
/environment-check github --fix
```

**Что проверяется:**
- `gh` — GitHub CLI установлен
- `gh auth status` — авторизация выполнена

**При ошибке:**
```
❌ Окружение не готово

❌ gh: не авторизован

Выполните:
  gh auth login

Документация: https://cli.github.com/manual/gh_auth_login
```

**0b. Валидация ввода:**

```
/input-validate issue-number #123
```

**Что проверяется:**
- Номер Issue — число (без `#` или с `#`)
- Issue существует в репозитории

**При ошибке:**
```
❌ Ошибка: Issue #{номер} не найден

Возможные причины:
1. Issue не существует
2. Неправильный репозиторий
3. Нет доступа к приватному репозиторию

Проверьте номер и повторите попытку.
```

> ⚠️ **БЛОКИРУЮЩИЙ ШАГ:** При любой ошибке проверок — остановить выполнение.


| Поле | Описание |
|------|----------|

| `title` | Заголовок |
| `state` | Статус (OPEN / CLOSED) |
| `closedAt` | Дата закрытия |
| `stateReason` | Причина закрытия (completed / not_planned) |
| `labels` | Метки |
| `assignees` | Назначенные |
| `body` | Описание |

| Ошибка | Причина | Действие |
|--------|---------|----------|
| gh не установлен | GitHub CLI отсутствует | Показать инструкцию установки |

## FAQ / Troubleshooting

### Когда нужно переоткрывать Issue?

| Ситуация | Переоткрывать? |
|----------|----------------|
| Задача была закрыта преждевременно | ✅ Да |
| Найдены новые требования | ✅ Да |
| Баг воспроизвёлся снова | ✅ Да |
| Решение оказалось неполным | ✅ Да |
| Нужна совершенно новая задача | ❌ Нет → `/issue-create` |
| Issue устарел, но нужен похожий | ❌ Нет → `/issue-create` |

### Сохраняется ли история Issue?

**Да.** GitHub сохраняет всю историю:
- Закрытие и переоткрытие с датами
- Все комментарии
- Изменения меток и назначений
- Связанные PR и коммиты

### Что если нет прав на переоткрытие?

**Симптом:** Ошибка `Resource not accessible by integration`

**Причины:**
1. Нет доступа к репозиторию
2. Токен не имеет нужных прав
3. Репозиторий архивирован

**Решение:**
1. Проверить доступ: `gh repo view`
2. Проверить токен: `gh auth status`
3. Обратиться к владельцу репозитория

### Как переоткрыть несколько Issue сразу?

**Вариант 1 — последовательно:**
```
/issue-reopen #123
/issue-reopen #124
/issue-reopen #125
```

**Вариант 2 — через bash:**
```bash
for i in 123 124 125; do gh issue reopen $i; done
```

### Можно ли переоткрыть Issue, закрытый как "not planned"?

**Да.** GitHub позволяет переоткрывать Issue независимо от причины закрытия (completed или not_planned).

### Что делать, если Issue был удалён?

**Невозможно** переоткрыть удалённый Issue. Удаление Issue необратимо.

**Решение:** Создать новый Issue с помощью `/issue-create`.

### Как отменить переоткрытие?

Если Issue переоткрыт по ошибке:

```
/issue-complete #{номер}  # Закрыть как выполненный
# или
/issue-delete #{номер}    # Закрыть как неактуальный
```

### Почему рекомендуется добавлять комментарий?

**Причины:**
1. **Контекст** — понятно, почему задача вернулась
2. **История** — можно отследить паттерны (например, частые регрессии)
3. **Приоритизация** — легче оценить срочность
4. **Коммуникация** — команда видит изменения

### Влияет ли переоткрытие на связанные PR?

**Нет.** Связь между Issue и PR сохраняется. Если PR был merged, он останется merged.

**Но:** Если PR был закрыт вместе с Issue — PR нужно переоткрывать отдельно (если не merged).

## Следующие шаги

После переоткрытия Issue:

1. **Взять в работу:**
   ```
   /issue-execute #{номер}
   ```

2. **Обновить описание** (если нужно добавить детали):
   ```
   /issue-update #{номер}
   ```

3. **Изменить приоритет** (если срочно):
   ```
   /issue-update #{номер} --add-label "priority:high"
   ```

4. **Назначить на кого-то:**
   ```
   /issue-update #{номер} --assignee @username
   ```

**Типичный workflow после переоткрытия:**

```
/issue-reopen #123 "Баг вернулся после деплоя"
    ↓
/issue-execute #123
    ↓
[Работа над исправлением]
    ↓
/issue-review #123
    ↓
/issue-complete #123
```

## Формат вызова

```
/test-update [путь-к-тесту] [--reason sync|extend|fix] [--dry-run]
```

| Параметр | Описание | По умолчанию |
|----------|----------|--------------|
| `путь-к-тесту` | Путь к файлу теста или скиллу | Спросить |
| `--reason` | Причина обновления | Спросить |
| `--dry-run` | Показать план без применения | — |
| `--diff` | Показать подробный diff | — |

**Причины обновления:**

| Reason | Описание | Когда использовать |
|--------|----------|-------------------|
| `sync` | Синхронизация с изменённым кодом | Код изменился, тест устарел |
| `extend` | Расширение покрытия | Добавить сценарии |
| `fix` | Исправление теста | Тест некорректен |

## Автоопределение scope

> **SSOT:** Полная логика определения scope описана в [scope-detection.md](/.claude/templates/scope-detection.md).

**Краткая таблица:**

| Путь начинается с | Scope | Формат теста |
|-------------------|-------|--------------|
| `.claude/skills/*` | `claude` | Раздел "Тестирование" в SKILL.md |
| `.claude/instructions/*` | `claude` | `*.test.md` |
| `src/*` | `project` | `*.test.ts` рядом или в `tests/` |
| `tests/*` | `project` | `*.test.ts` |
| `shared/*` | `project` | `*.test.ts` |

**Правило:** Scope определяется по **пути**, а не по расширению файла.

## Правила

### Правило синхронизации

**Правило:** При обновлении теста по причине `sync` — сначала прочитать изменённый код, затем обновить тест.

**Алгоритм:**
1. Найти связанный с тестом код
2. Определить изменения в коде
3. Обновить ожидания теста под новое поведение

### Правило расширения

**Правило:** При расширении теста (`extend`) — не удалять существующие сценарии, только добавлять новые.

**Исключение:** Можно удалить дублирующиеся или устаревшие сценарии с подтверждением.

### Правило минимальных изменений

**Правило:** Изменять только то, что необходимо. Не рефакторить весь тест без явного запроса.

### Правило backup

**Правило:** Перед масштабными изменениями — показать diff и получить подтверждение.

## FAQ / Troubleshooting

### Тест не находит изменения в коде

**Проблема:** `/test-update --reason sync` не обнаруживает изменений.

**Причины:**
1. Код не изменился с момента последнего обновления теста
2. Изменения в несвязанных файлах
3. Неверно определена связь тест ↔ код

**Решение:**
```bash
# Проверить связь теста с кодом
/test-review путь-к-тесту

# Указать явно файл кода
/test-update путь-к-тесту --source путь-к-коду
```

### Как обновить несколько тестов одновременно?

**Способ 1:** Обновить папку
```bash
/test-update src/auth/ --reason sync
```

**Способ 2:** Использовать паттерн
```bash
/test-update "src/**/*.test.ts" --reason sync --dry-run
```

### Какую причину (reason) выбрать?

| Ситуация | Reason |
|----------|--------|
| Код изменился, тест устарел | `sync` |
| Нужно больше сценариев | `extend` |
| Тест работает неправильно | `fix` |

### Как откатить изменения?

Если `/test-update` внёс неверные изменения:
```bash
git checkout путь-к-тесту
```

### Scope определён неверно

> **SSOT:** См. [scope-detection.md](/.claude/templates/scope-detection.md)

Используйте явное указание scope:
```bash
/test-update путь --scope project
```

### Тест и код изменились одновременно

**Ситуация:** В одном PR изменён и скилл/код, и его тест.

**Workflow:**
```
1. Прочитать diff скилла → понять что изменилось
2. Прочитать diff теста → понять как обновился тест
3. Проверить соответствие:
   - Новый функционал покрыт тестом?
   - Старые тесты обновлены под изменения?
4. Выполнить тест
```

**Проверка через `/test-review`:**
```bash
/test-review путь-к-тесту --depth deep
```

**Правило:** Если изменился воркфлоу скилла — тест ДОЛЖЕН быть обновлён в том же коммите.

**Признаки проблемы:**
| Симптом | Возможная причина |
|---------|-------------------|
| Тест passed, но новый функционал не покрыт | Тест не обновлён |
| Тест failed на новом функционале | Ожидания устарели |
| Coverage снизился | Новый код без тестов |

### Как откатить тест к предыдущей версии?

**Из git (не закоммичено):**
```bash
git checkout -- путь-к-тесту
```

**Из git history (закоммичено):**
```bash
# Найти коммит
git log --oneline -- путь-к-тесту

# Откатить к конкретной версии
git checkout {commit-hash}^ -- путь-к-тесту
```

**Частичный откат (только часть изменений):**
```bash
# Посмотреть diff между версиями
git diff {old-commit}..{new-commit} -- путь-к-тесту

# Откатить вручную через /test-update
/test-update путь-к-тесту --reason fix
```

**После отката:**
```bash
# Проверить, что тест работает
/test-execute путь-к-тесту

# Если нужно — обновить под текущий код
/test-update путь-к-тесту --reason sync
```

```bash
# 1. Проверить, что тест работает

# 2. Отметить результат
/test-complete {путь} --status passed
```

**Типичная цепочка:**
```
/test-update → /test-execute → /test-complete
```

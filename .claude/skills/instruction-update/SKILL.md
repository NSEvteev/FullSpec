# Обновление инструкции

Команда для проверки файлов проекта на соответствие изменённой инструкции. Находит несоответствия и формирует список TODO.

**Связанные скиллы:**
- [instruction-create](/.claude/skills/instruction-create/SKILL.md) — создание инструкций
- [instruction-delete](/.claude/skills/instruction-delete/SKILL.md) — удаление инструкций
- [links-update](/.claude/skills/links-update/SKILL.md) — синхронизация ссылок
- [context-update](/.claude/skills/context-update/SKILL.md) — распространение контекста
- [test-update](/.claude/skills/test-update/SKILL.md) — обновление связанных тестов

**Связанные инструкции:**
- [README.md](/.claude/instructions/README.md) — индекс инструкций, статусы заполнения

## Формат вызова

```
/instruction-update <путь>
```

**Примеры:**
- `/instruction-update src/api/design.md`
- `/instruction-update git/commits.md`
- `/instruction-update platform/observability/metrics.md`

## Когда использовать

| Ситуация | Действие |
|----------|----------|
| Изменили правила в инструкции | `/instruction-update` — проверить, что нужно обновить в проекте |
| Добавили новое правило | `/instruction-update` — найти файлы, не соответствующие новому правилу |
| Хотите аудит соответствия | `/instruction-update` — получить список несоответствий |

## Правила

### Зона ответственности

Каждая инструкция отвечает за определённые файлы/папки проекта:

| Путь инструкции | Зона ответственности |
|-----------------|---------------------|
| `src/api/*.md` | `/src/**/api/**`, `/src/**/routes/**` |
| `src/data/*.md` | `/src/**/*.ts`, `/src/**/*.py` (форматы данных) |
| `src/runtime/*.md` | `/src/**/` (конфигурация runtime) |
| `src/security/*.md` | `/src/**/auth/**`, `/src/**/security/**` |
| `platform/*.md` | `/platform/**` |
| `config/*.md` | `/config/**` |
| `git/*.md` | `/.github/**`, `.git*` файлы |

### Формат раздела TODO

Раздел добавляется в **конец** файла инструкции:

```markdown
---

## TODO по инструкции

> Автоматически сгенерировано `/instruction-update` — {дата}

### Несоответствия

| Файл | Правило | Проблема |
|------|---------|----------|
| `/src/auth/login.ts` | Формат ошибок | Использует старый формат `{error: string}` |
| `/src/api/users.ts` | HTTP методы | DELETE возвращает 200 вместо 204 |

### Статистика

- Проверено файлов: {N}
- Найдено несоответствий: {M}
- Файлов без проблем: {K}
```

### Идемпотентность

При повторном вызове `/instruction-update`:
1. Удалить старый раздел "TODO по инструкции"
2. Создать новый с актуальными данными

## Воркфлоу

### Шаг 1: Получить путь к инструкции

1. Из аргумента: `/instruction-update src/api/design.md`
2. Или спросить: "Какую инструкцию проверить? (путь относительно /.claude/instructions/)"

**Валидация:**
- Файл существует в `/.claude/instructions/`
- Файл имеет расширение `.md`

### Шаг 2: Определить зону ответственности

1. Прочитать инструкцию
2. Найти поле `type` во frontmatter:
   - `standard` — применяется ко всем файлам в зоне
   - `project` — специфично для этого проекта
3. Определить зону по пути инструкции (см. [Маппинг инструкций на зоны](#маппинг-инструкций-на-зоны))

**Пример:**
```
Инструкция: src/api/design.md
Зона: /src/**/api/**, /src/**/routes/**
```

### Шаг 3: Найти файлы в зоне ответственности

```bash
# Пример для src/api/design.md
find /src -type f \( -path "*/api/*" -o -path "*/routes/*" \) -name "*.ts" -o -name "*.py"
```

Или через Glob:
```
Glob: src/**/api/**/*.{ts,py,js}
Glob: src/**/routes/**/*.{ts,py,js}
```

**Исключения:**
- `node_modules/`
- `dist/`, `build/`
- `*.test.ts`, `*.spec.ts`
- `__tests__/`

### Шаг 4: Извлечь правила из инструкции

1. Прочитать раздел "## Правила" из инструкции
2. Для каждого правила определить:
   - Что проверять (паттерн, структура, значение)
   - Как проверять (regex, AST, grep)
   - Критерий соответствия

**Пример правил из `src/api/design.md`:**

| Правило | Что проверять | Как проверять |
|---------|---------------|---------------|
| URL kebab-case | Пути в роутах | Regex: `/[a-z]+(-[a-z]+)*/` |
| HTTP методы | Handlers | Grep: `@Get`, `@Post`, etc. |
| Статус-коды | Return statements | Grep: `return.*status\(` |

### Шаг 5: Проверить файлы на соответствие

Для каждого файла из Шага 3:

1. Прочитать содержимое файла
2. Применить каждое правило из Шага 4
3. Записать несоответствия

**Алгоритм проверки:**

```
для каждого файла:
    для каждого правила:
        результат = применить_проверку(файл, правило)
        если результат.несоответствие:
            добавить_в_список(файл, правило, результат.описание)
```

### Шаг 6: Сформировать список несоответствий

Сгруппировать результаты:

1. **По файлам** — для исправления по одному файлу
2. **По правилам** — для понимания масштаба проблемы

**Формат:**

```markdown
### По файлам

**`/src/api/users.ts`** (3 несоответствия)
- [ ] Правило "HTTP методы": DELETE возвращает 200 вместо 204
- [ ] Правило "URL naming": путь `/getUsers` не в kebab-case
- [ ] Правило "Статус-коды": POST возвращает 200 вместо 201

**`/src/api/auth.ts`** (1 несоответствие)
- [ ] Правило "Формат ошибок": использует `{error: string}`

### По правилам

**Правило "HTTP методы"** — 5 файлов
**Правило "URL naming"** — 3 файла
**Правило "Формат ошибок"** — 2 файла
```

### Шаг 7: Добавить раздел TODO в инструкцию

1. Проверить, есть ли уже раздел "## TODO по инструкции"
2. Если есть — удалить его
3. Добавить новый раздел в конец файла

**Формат раздела:**

```markdown
---

## TODO по инструкции

> Автоматически сгенерировано `/instruction-update` — {YYYY-MM-DD}
>
> Повторный запуск: `/instruction-update {путь}`

### Несоответствия

{Таблица или список несоответствий}

### Статистика

- Проверено файлов: {N}
- Найдено несоответствий: {M}
- Файлов без проблем: {K}

### Следующие шаги

1. Исправить несоответствия в файлах
2. Запустить `/instruction-update {путь}` повторно
3. Убедиться, что список пуст
```

### Шаг 8: Обновить контекст связанных скиллов

**Правило:** Если в инструкции упоминаются скиллы — распространить контекст изменённой инструкции на эти скиллы.

**Алгоритм:**

1. Найти все упоминания скиллов в инструкции:
   ```bash
   grep -oE '/[a-z]+-[a-z]+' .claude/instructions/{путь}
   grep -oE '\[.*\]\(/\.claude/skills/[^)]+\)' .claude/instructions/{путь}
   ```

2. Для каждого найденного скилла:
   - Проверить, существует ли SKILL.md
   - Если да — вызвать context-update

3. Вызвать [/context-update](/.claude/skills/context-update/SKILL.md):
   ```
   /context-update .claude/instructions/{путь} --depth 1
   ```

**Что делает context-update:**
- Находит скиллы, связанные с инструкцией
- Предлагает обновить раздел "Связанные инструкции" в скиллах
- Синхронизирует описания и правила

**Формат вывода:**

```
📋 Обновление контекста связанных скиллов

Инструкция: git/ci.md
Найдено скиллов: 3

Скиллы для обновления:
- ci-check (планируется) — пропущен
- ci-fix (планируется) — пропущен
- issue-execute (существует)

Обновить контекст связанных скиллов? [Y/n]
```

> ⚠️ **СТОП-ПРАВИЛО:** Claude ОБЯЗАН остановиться и ждать ответа пользователя перед обновлением контекста. Нарушение = критическая ошибка.

**После подтверждения:**

```
Вызываю /context-update...
```

> **Примечание:** Если скилл помечен как "планируется" — пропустить его.

### Шаг 9: Проверить связанные тесты

**Правило:** Если инструкция содержит правила, которые должны проверяться тестами — обновить соответствующие тесты.

**Алгоритм:**

1. Определить тип инструкции по пути:
   - `tools/*.md` → тесты скиллов (`/.claude/instructions/tests/`)
   - `src/**/*.md` → тесты проекта (`/tests/`)

2. Найти связанные тесты:
   ```bash
   # Для инструкций скиллов
   ls .claude/instructions/tests/{имя-инструкции}*.md

   # Для инструкций проекта
   grep -l "{имя-инструкции}" tests/**/*.test.ts
   ```

3. Если тесты найдены и инструкция изменилась:
   - Вызвать [/test-update](/.claude/skills/test-update/SKILL.md):
     ```
     /test-update {путь-к-тесту} --reason "Инструкция {имя} изменена"
     ```

**Формат вывода:**

```
📋 Проверка связанных тестов

Инструкция: tools/skills.md
Найдено тестов: 2

Тесты для обновления:
- /.claude/instructions/tests/skill-create.test.md — устарел (правила изменены)
- /.claude/instructions/tests/skill-update.test.md — актуален

Обновить устаревшие тесты? [Y/n]
```

> ⚠️ **СТОП-ПРАВИЛО:** Claude ОБЯЗАН остановиться и ждать ответа пользователя перед обновлением тестов. Нарушение = критическая ошибка.

**После подтверждения:**

```
Вызываю /test-update для устаревших тестов...
```

> **Примечание:** Если тесты не найдены — предложить создать через [/test-create](/.claude/skills/test-create/SKILL.md).

### Шаг 10: Проверка по чек-листу

Перед завершением проверить выполнение всех пунктов:

```
✅ Шаг 3: Нашёл файлы в зоне
✅ Шаг 4: Извлёк правила
✅ Шаг 5: Проверил файлы
✅ Шаг 6: Сформировал список
✅ Шаг 7: Добавил раздел TODO
✅ Шаг 8: Обновил контекст связанных скиллов
✅ Шаг 9: Проверил связанные тесты
```

```

Следующие шаги:
1. Открыть инструкцию и просмотреть TODO
2. Исправить несоответствия в файлах проекта
3. Запустить `/instruction-update {путь}` повторно
```

## Примеры использования

### Пример 1: Проверка после изменения инструкции API

**Вызов:**
```
/instruction-update src/api/design.md
```

**Результат:**
```
✅ Проверка инструкции завершена

Инструкция: /.claude/instructions/src/api/design.md
Зона ответственности: /src/**/api/**, /src/**/routes/**

Статистика:
- Проверено файлов: 12
- Найдено несоответствий: 5
- Файлов без проблем: 7

Раздел "TODO по инструкции" добавлен в конец файла.
```

**Добавлено в инструкцию:**
```markdown
---

## TODO по инструкции

> Автоматически сгенерировано `/instruction-update` — 2024-01-15

### Несоответствия

| Файл | Правило | Проблема |
|------|---------|----------|
| `/src/api/users.ts:45` | URL naming | `/getUsers` → `/users` |
| `/src/api/users.ts:78` | HTTP методы | DELETE возвращает 200 → 204 |
| `/src/api/auth.ts:23` | Формат ошибок | `{error: string}` → `{error: {code, message}}` |

### Статистика

- Проверено файлов: 12
- Найдено несоответствий: 5
- Файлов без проблем: 7
```

### Пример 2: Инструкция без несоответствий

**Вызов:**
```
/instruction-update git/commits.md
```

**Результат:**
```
✅ Проверка инструкции завершена

Инструкция: /.claude/instructions/git/commits.md
Зона ответственности: Git history (последние 50 коммитов)

Статистика:
- Проверено коммитов: 50
- Найдено несоответствий: 0
- Все коммиты соответствуют Conventional Commits

Раздел TODO не добавлен — несоответствий не найдено.
```

### Пример 3: Файл инструкции не найден

**Вызов:**
```
/instruction-update src/api/unknown.md
```

**Ответ:**
```
❌ Инструкция не найдена: /.claude/instructions/src/api/unknown.md

Доступные инструкции в src/api/:
- design.md
- versioning.md
- deprecation.md
- swagger.md

Использование: /instruction-update src/api/design.md
```

### Пример 4: Повторная проверка (идемпотентность)

**Вызов (повторный):**
```
/instruction-update src/api/design.md
```

**Результат:**
```
ℹ️ Найден существующий раздел "TODO по инструкции" — будет заменён

✅ Проверка инструкции завершена

Статистика:
- Проверено файлов: 12
- Найдено несоответствий: 2 (было 5, исправлено 3)
- Файлов без проблем: 10

Раздел "TODO по инструкции" обновлён.
```

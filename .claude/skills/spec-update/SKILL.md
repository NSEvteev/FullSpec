# Работа с документом /specs/

Основной скилл для редактирования документов спецификаций. Вызывает `/spec-status` при переходе по workflow.

## Формат вызова

```
/spec-update <path> [--validate]
```

| Параметр | Описание |
|----------|----------|
| `path` | Путь к документу (сокращённый или полный) |
| `--validate` | Только проверить документ без редактирования |

**Способы начать работу:**

```
/spec-update discussions/001          # Явный вызов
/spec-update /specs/impact/002.md     # Полный путь
"поработаем с дискуссией 001"         # Естественный язык
"открой ADR auth/003"                 # Естественный язык
```

## Воркфлоу

### Шаг 1: Найти документ

По пути найти документ и определить его тип.

### Шаг 2: Прочитать документ

Загрузить содержимое и метаданные (статус, родитель, дата).

### Шаг 3: Показать контекст

```
📄 Discussion 001-auth-strategy

Статус: 🔍 REVIEW
Дата создания: 2025-01-20
Последнее изменение: 2025-01-21

Связанные документы:
- Impact: /specs/impact/001-auth-strategy.md (DRAFT)

Чек-лист текущего статуса:
- [x] Заполнена секция "Контекст"
- [x] Описаны варианты решения
- [ ] Выбран вариант решения

Что делаем?
```

### Шаг 4: Интерактивная работа

Пользователь может:
- Редактировать секции документа
- Задавать вопросы по контексту
- Запрашивать валидацию

### Шаг 5: Триггеры перехода

При определённых фразах — предложить изменение статуса.

### Шаг 6: Сохранение

При завершении работы — сохранить изменения.

## Триггеры перехода

| Фраза пользователя | Действие |
|--------------------|----------|
| "пошли дальше", "готово", "закончили" | Проверить чек-лист → предложить `/spec-status` |
| "отложим", "потом" | Сохранить прогресс, не менять статус |
| "отклонить", "не нужно" | Предложить перевод в REJECTED |
| "заменить", "новый вариант" | Предложить SUPERSEDED и создание нового |

**Пример:**

```
User: готово, можно двигаться дальше

Claude: 📋 Проверяю чек-лист для перехода REVIEW → APPROVED...

- [x] Выбран вариант решения
- [x] Заполнена секция "Решение"
- [x] Заполнена секция "Ревью решения"
- [x] Отвечены вопросы

✅ Чек-лист пройден!

Выполнить /spec-status discussions/001 approved? [Y/n]
```

## Режимы работы

### Режим редактирования (по умолчанию)

Интерактивная работа с документом.

### Режим валидации (--validate)

Только проверка документа:

```
/spec-update discussions/001 --validate

📋 Валидация Discussion 001-auth-strategy

Структура:
- [x] Метаданные корректны
- [x] Все обязательные секции присутствуют
- [x] Ссылки валидны

Чек-лист текущего статуса (REVIEW):
- [x] Заполнена секция "Контекст"
- [x] Описаны варианты решения
- [ ] Выбран вариант решения ← не заполнено

Чек-лист следующего статуса (APPROVED):
- [ ] Выбран вариант решения
- [ ] Заполнена секция "Решение"
- [ ] Заполнена секция "Ревью решения"

Готовность к переходу: ❌ НЕ ГОТОВ
Недостающие пункты: 4
```

## Примеры использования

### Пример 1: Работа с Discussion

```
/spec-update discussions/001

📄 Discussion 001-auth-strategy

Статус: 🔍 REVIEW
...

Что делаем?

User: добавь третий вариант — использование OAuth2

Claude: Добавляю вариант 3 в секцию "Варианты решения"...

✅ Добавлен вариант:
### Вариант 3: OAuth2
...

Обновить таблицу сравнения? [Y/n]
```

### Пример 2: Валидация ADR

```
/spec-update auth/adr/002 --validate

📋 Валидация ADR auth/002-jwt-tokens

Структура: ✅ OK
Ссылка на Impact: ✅ /specs/impact/001-auth-strategy.md
Breaking Changes: ⚠️ YES — секция заполнена

Чек-лист бизнес-логики:
| Правило | В Impact? |
|---------|-----------|
| JWT срок жизни 1 час | ✅ да |
| Refresh token 7 дней | ✅ да |
| Ротация при refresh | ❌ НЕТ |

⚠️ Найдена бизнес-логика не описанная в Impact!

Действия:
1. Обновить Impact (добавить правило ротации)
2. Удалить правило из ADR
3. Игнорировать (--force)
```

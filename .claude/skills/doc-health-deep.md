# Скилл: Глубокий смысловой аудит документации

## Назначение

Проводит комплексный смысловой анализ ВСЕЙ документации проекта за один проход с использованием предварительно собранного контекста.

**Ключевая особенность:** АВТОМАТИЧЕСКИ создаёт задачи AMY-XXXXX для исправления найденных проблем.

---

## Входные данные

Скилл работает с агрегированным контекстом из файла:
- `llm_tasks/temp/doc_context.json`

**Перед запуском убедись, что контекст собран:**
```bash
python scripts/check_doc_context.py
```

---

## Процесс работы

### Шаг 1: Прочитать агрегированный контекст

```bash
cat llm_tasks/temp/doc_context.json
```

Контекст содержит:
- **documents**: Массив всех документов с метаданными
- **glossary**: Термины глоссария и категории
- **dependency_graph**: Граф зависимостей между документами
  - nodes: Узлы графа (документы)
  - edges: Связи между документами
  - chains: Обнаруженные цепочки зависимостей
- **links**: Все ссылки (internal, external, broken)
- **statistics**: Статистика по типам, статусам, папкам

---

### Шаг 2: Провести смысловой анализ

Анализируй контекст по следующим критериям:

#### A. Цепочки зависимостей (КРИТИЧНО!)

**Проверить:**
- Все ли дискуссии (discuss) имеют связанную архитектуру (architecture)?
- Архитектура ссылается на Decision (ADR)?
- Decision имеет ресурсы (resource)?
- Ресурсы имеют планы реализации (imp_plan)?
- Обратная связь (FeedBack) соблюдена?

**Данные:** `dependency_graph.chains`

**Найти:**
- Разорванные цепочки (`complete: false`, `missing_links` не пусто)
- Висячие документы (нет входящих/исходящих связей в `edges`)
- Цепочки начинаются с discuss, но не доходят до imp_plan

**Приоритет проблем:** 🔴 Высокий

---

#### B. Актуальность содержания

**Проверить:**
- Даты обновления документов (`date` в documents)
- Статусы соответствуют реальному состоянию (`status`)
- Нет устаревших документов (>6 месяцев без обновлений)

**Найти:**
- Документы со статусом "draft" старше 1 месяца
- Документы "in_progress" старше 2 месяцев
- Документы без даты обновления
- Противоречия в датах (ресурс новее, чем архитектура в цепочке)

**Приоритет:** 🟡 Средний

---

#### C. Полнота описаний ресурсов

**Проверить:** Ресурсы (type: "resource") содержат необходимые разделы.

**Ожидаемые разделы по типам ресурсов:**

**database** (из headers, ищи заголовки):
- "Схема данных" или "Схема"
- "Миграции"
- "Запросы" или "Критические запросы"

**backend**:
- "API" или "Endpoints"
- "Бизнес-логика"
- "Интеграции" или "Внешние зависимости"

**frontend**:
- "Компоненты"
- "Состояние" или "State"
- "Маршрутизация"

**infra**:
- "Развертывание" или "Deployment"
- "Мониторинг"
- "Безопасность"

**Найти:**
- Ресурсы без обязательных секций
- Определи тип ресурса по пути (05_resources/database/, 05_resources/backend/, и т.д.)

**Приоритет:** 🟡 Средний

---

#### D. Использование глоссария

**Проверить:**
- Все ли термины из `glossary.terms` используются в документах?
- Нет ли терминов в документах, которых нет в глоссарии?

**Анализ:**
- Для каждого термина из glossary: найди упоминания в документах
  - Используй `grep` или анализ текста документов
  - Если термин не встречается нигде → кандидат на удаление
- Найди специфичные термины проекта в документах, которых нет в глоссарии → кандидаты на добавление

**Приоритет:** 🟢 Низкий

---

#### E. Битые ссылки (уже найдены)

**Данные:** `links.broken`

**Анализ:** Битые ссылки уже собраны агрегатором. Отметь их в отчёте, но НЕ создавай задачи для примеров в шаблонах.

**Игнорировать:**
- Ссылки в `templates/` (это placeholder-ы)
- Ссылки в примерах документации (например, `[Doc1](путь)`)

**Приоритет:** 🟡 Средний (только для реальных документов)

### Шаг 4: Создать задачи для Amy

**КРИТИЧНО:** Для КАЖДОЙ найденной критичной и средней проблемы создай задачу AMY-XXXXX.

**Как создавать задачи:**

```bash
# Используй скрипт task_new.py
python scripts/task_new.py \
  -t "Название задачи" \
  -p high \
  -c docs \
  -a amy-santiago \
  -d "Описание проблемы из аудита" \
  -f current
```

**Параметры:**
- `-t` (title): Краткое название задачи
- `-p` (priority): `high`, `medium`, `low`
- `-c` (category): `docs` (для задач по документации)
- `-a` (assignee): `amy-santiago` (исполнитель - Amy)
- `-d` (description): Описание проблемы
- `-f` (folder): `current` (текущие задачи)

**Примеры задач:**

#### Для разорванной цепочки:
```bash
python scripts/task_new.py \
  -t "Закрыть разорванную цепочку зависимостей для 001_auth" \
  -p high \
  -c docs \
  -a amy-santiago \
  -d "Дискуссия 001_auth.md → Архитектура 005_auth_flow.md, но нет Decision, ресурсов и плана. Создать недостающие документы и связать их." \
  -f current
```

#### Для устаревшего документа:
```bash
python scripts/task_new.py \
  -t "Обновить устаревший документ 002_api_gateway.md" \
  -p medium \
  -c docs \
  -a amy-santiago \
  -d "Документ 002_api_gateway.md не обновлялся 7 месяцев (последнее обновление: 2024-06-15). Проверить актуальность и обновить статус." \
  -f current
```

#### Для неполного ресурса:
```bash
python scripts/task_new.py \
  -t "Дополнить описание ресурса infra/kubernetes.md" \
  -p medium \
  -c docs \
  -a amy-santiago \
  -d "Ресурс infra/kubernetes.md содержит только 2 из 5 обязательных секций. Добавить недостающие: Мониторинг, Безопасность, Резервирование." \
  -f current
```

**После создания каждой задачи:**
1. Запомни её ID (AMY-00XXX)
2. Добавь в таблицу "Созданные задачи" в отчёте
3. Продолжай создавать следующие задачи

**Сколько задач создавать:**
- Критичные проблемы (🔴): ВСЕГДА создавать задачи
- Средние проблемы (🟡): Создавать задачи, если проблем не больше 10
- Низкие проблемы (🟢): НЕ создавать задачи, только упомянуть в отчёте

## Важные замечания

1. **НЕ создавай задачи для примеров в шаблонах** - битые ссылки в `templates/` это норма
2. **Создавай задачи только для реальных проблем** в документации проекта
3. **Группируй похожие проблемы** в одну задачу, если это логично
4. **Указывай конкретные пути к файлам** в описании задач
5. **Приоритизируй:** разорванные цепочки = high, остальное = medium/low

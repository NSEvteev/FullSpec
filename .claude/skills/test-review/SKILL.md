## Формат вызова

```
/test-review [путь-к-тесту] [--depth shallow|deep] [--fix] [--last-failed]
```

| Параметр | Описание | По умолчанию |
|----------|----------|--------------|
| `путь-к-тесту` | Путь к тесту или скиллу | Спросить |
| `--depth` | Глубина анализа | `shallow` |
| `--fix` | Автоматически исправить найденные проблемы | — |
| `--report` | Сохранить отчёт в файл | — |
| `--last-failed` | Проверить только failed тесты из последнего запуска `/test-execute` | — |

**Глубина анализа:**

| Depth | Что проверяется | Когда использовать |
|-------|-----------------|-------------------|
| `shallow` | Структура, базовое покрытие | Быстрая проверка |
| `deep` | Граничные случаи, мутационный анализ | Перед релизом |

## Правила

### Правило полноты

**Правило:** Тест должен покрывать все публичные входные точки объекта.

**Для скилла:**
- Все команды из triggers.commands
- Минимум 1 фраза из triggers.phrases
- Основной happy path

**Для кода:**
- Все экспортируемые функции
- Все публичные методы классов
- Основные use cases

### Правило негативных сценариев

**Правило:** Тест должен содержать минимум 1 негативный сценарий.

**Примеры негативных сценариев:**
- Невалидный ввод
- Пустые значения
- Превышение лимитов
- Ошибки сети/API

### Правило независимости

**Правило:** Каждый тест должен быть независим от других.

**Проверка:**
- Нет общего состояния между тестами
- Тесты можно запускать в любом порядке
- Нет зависимости от внешних сервисов (или они замокированы)

### Правило читаемости

**Правило:** Тест должен быть понятен без обращения к коду.

**Критерии:**
- Понятное название теста
- Чёткое разделение Arrange/Act/Assert
- Комментарии для сложной логики

## FAQ / Troubleshooting

### Какой coverage считается достаточным?

| Scope | Минимум | Рекомендуемый |
|-------|---------|---------------|
| `claude` | 80% сценариев | 100% critical path |
| `project` | 70% строк кода | 85% строк кода |

### Когда выбирать shallow vs deep анализ?

| Режим | Когда использовать |
|-------|-------------------|
| `--depth shallow` | Быстрая проверка, < 10 сценариев |
| `--depth deep` | Полный анализ, критичный функционал |

**Рекомендация:**
- `claude` scope → обычно достаточно `shallow`
- `project` scope → используйте `deep` для критичных модулей

### Как интерпретировать матрицу покрытия?

```
✅ Покрыто      — сценарий имеет тест
⚠️ Частично     — тест есть, но неполный
❌ Не покрыто   — тест отсутствует
```

### Review показывает 100%, но тесты падают

**Причина:** Coverage не гарантирует корректность тестов.

**Решение:**
1. Проверить качество assertions
2. Убедиться, что тесты проверяют правильные значения
3. Запустить: `/test-execute --verbose`

### Как исключить файлы из анализа?

```bash
/test-review src/auth/ --exclude "*.mock.ts,*.stub.ts"
```

### Scope определён неверно

> **SSOT:** См. [scope-detection.md](/.claude/templates/scope-detection.md)

Используйте явное указание:
```bash
/test-review путь --scope project
```

## Следующие шаги

После ревью теста:

**Если покрытие недостаточно:**
```bash
/test-update {путь} --reason extend
```

**Если тестов нет:**
```bash
/test-create {путь-к-коду}
```

**При низком coverage (< 50%):**
```bash
# Добавить функциональные тесты
/test-create {путь} --type functional
```

**Типичные цепочки:**
```
низкий coverage: /test-review → /test-create (functional) → /test-execute
расширение:      /test-review → /test-update (extend) → /test-execute
```

# Обработка кандидатов в глоссарий

Команда обрабатывает термины из раздела "Кандидаты" глоссария пакетами.

## Инструкции

### Шаг 1: Прочитать глоссарий

Прочитать `general_docs/glossary.md`:
- Раздел "## Термины" — существующие термины
- Раздел "## Кандидаты" — термины для обработки

### Шаг 2: Подготовить пакет определений

**Размер пакета:** до 7 терминов (ограничение на количество вопросов).

Для каждого кандидата подготовить определение, синонимы, антонимы.

### Шаг 3: Показать таблицу определений

Вывести таблицу со всеми подготовленными определениями:

```markdown
## Подготовленные определения (пакет 1/N)

| № | Термин | Определение | Синонимы |
|---|--------|-------------|----------|
| 1 | Термин1 | Определение1 | Син1 |
| 2 | Термин2 | Определение2 | — |
...
```

### Шаг 4: Спросить по каждому термину

Использовать AskUserQuestion с **отдельным вопросом для каждого термина**.

Максимум 4 вопроса в одном вызове AskUserQuestion (ограничение инструмента).

Формат вопроса:
```
Термин: [Название]
- Добавить (Рекомендовано)
- Отложить
- Удалить
```

### Шаг 5: Выполнить действия

**Для "Добавить":**
1. Добавить в раздел "## Термины" в алфавитном порядке
2. Удалить из раздела "## Кандидаты"

**Для "Отложить":**
- Обновить статус кандидата на "отложен"

**Для "Удалить":**
- Удалить из раздела "## Кандидаты"

### Шаг 6: Повторить для следующего пакета

Если остались необработанные кандидаты — перейти к шагу 2.

### Шаг 7: Вывести итог

```
Обработка завершена:
- Добавлено в глоссарий: X
- Отложено: Y
- Удалено: Z
```

## Формат термина в глоссарии

```markdown
### Термин

- **Определение:** Краткое, чёткое определение.
- **Контекст:** В каком модуле/архитектуре используется.
- **Синонимы:** Если есть.
- **Антонимы:** Если есть.
- **Связанные термины:** [Термин1](#термин1), [Термин2](#термин2)
- **Источник:** [название документа](путь/к/документу.md)
```

## Примеры использования

### Пример 1: Обработка 7 кандидатов (пакетный режим)

**Сценарий:** После запуска `/glossary-candidates` на архитектурном документе в глоссарий добавлено 7 кандидатов (JWT, OAuth2, RBAC, Refresh Token, Access Token, Token Rotation, Session). Нужно обработать все кандидаты за один раз.

**Команда:**
```bash
/glossary-review
```

**Ожидаемый результат:**
- Чтение 7 кандидатов из раздела "Кандидаты"
- Подготовка определений для всех терминов
- Показ таблицы с определениями пользователю
- Интерактивные вопросы по каждому термину (максимум 4 вопроса за раз)
- Добавление одобренных терминов в глоссарий
- Удаление отклонённых кандидатов

**Вывод (пример):**
```
Найдено кандидатов: 7

## Подготовленные определения (пакет 1/2)

| № | Термин | Определение | Синонимы |
|---|--------|-------------|----------|
| 1 | JWT | JSON Web Token — стандарт передачи данных между клиентом и сервером | Token |
| 2 | OAuth2 | Протокол авторизации для делегирования прав доступа | — |
| 3 | RBAC | Role-Based Access Control — система контроля доступа на основе ролей | — |
| 4 | Refresh Token | Долгоживущий токен для обновления Access Token | — |

[Интерактивные вопросы для терминов 1-4]

## Подготовленные определения (пакет 2/2)

| № | Термин | Определение | Синонимы |
|---|--------|-------------|----------|
| 5 | Access Token | Краткосрочный токен для доступа к защищённым ресурсам | Bearer Token |
| 6 | Token Rotation | Автоматическая смена токенов при каждом обновлении | — |
| 7 | Session | Данные пользовательской сессии на сервере | User Session |

[Интерактивные вопросы для терминов 5-7]

Обработка завершена:
- Добавлено в глоссарий: 6 (JWT, OAuth2, RBAC, Refresh Token, Access Token, Token Rotation)
- Отложено: 0
- Удалено: 1 (Session — уже есть в глоссарии как "Сессия")

✓ Глоссарий обновлён
```

### Пример 2: Обработка 2 кандидатов

**Сценарий:** После анализа нового документа добавлено 2 кандидата (Service Mesh, Circuit Breaker). Нужно быстро обработать их.

**Команда:**
```bash
/glossary-review
```

**Ожидаемый результат:**
- Чтение 2 кандидатов
- Подготовка определений
- Один пакет вопросов (меньше 4 терминов)

**Вывод (пример):**
```
Найдено кандидатов: 2

## Подготовленные определения (пакет 1/1)

| № | Термин | Определение | Синонимы |
|---|--------|-------------|----------|
| 1 | Service Mesh | Инфраструктурный слой для управления межсервисным взаимодействием | — |
| 2 | Circuit Breaker | Паттерн для предотвращения каскадных сбоев в распределённых системах | Предохранитель |

[Интерактивные вопросы]

Обработка завершена:
- Добавлено в глоссарий: 2 (Service Mesh, Circuit Breaker)
- Отложено: 0
- Удалено: 0

✓ Глоссарий обновлён

Следующий шаг: Запустить `/glossary-link` для добавления ссылок в документы
```

### Пример 3: Обработка с отложением термина

**Сценарий:** В кандидатах есть термин "Кэш", но пользователь хочет уточнить определение позже (недостаточно информации о том, какой именно кэш используется).

**Команда:**
```bash
/glossary-review
```

**Ожидаемый результат:**
- Подготовка определения для термина "Кэш"
- Пользователь выбирает "Отложить"
- Статус кандидата обновляется на "отложен"
- Термин остаётся в разделе "Кандидаты"

**Вывод (пример):**
```
Найдено кандидатов: 3

## Подготовленные определения (пакет 1/1)

| № | Термин | Определение | Синонимы |
|---|--------|-------------|----------|
| 1 | Кэш | Промежуточное хранилище для ускорения доступа к данным | Cache |
| 2 | Redis | In-memory база данных для кэширования и очередей | — |
| 3 | CDN | Content Delivery Network — сеть для быстрой доставки контента | — |

[Вопрос 1: Кэш]
Ответ пользователя: Отложить (нужно уточнить, какой именно кэш)

[Вопрос 2: Redis]
Ответ пользователя: Добавить

[Вопрос 3: CDN]
Ответ пользователя: Добавить

Обработка завершена:
- Добавлено в глоссарий: 2 (Redis, CDN)
- Отложено: 1 (Кэш — статус: "отложен, нужно уточнить тип")
- Удалено: 0

✓ Глоссарий обновлён
✓ Отложенные термины можно обработать позже через `/glossary-review`
```

## Формат вызова

```
/issue-review <номер> [--auto] [--strict]
```

| Параметр | Описание | По умолчанию |
|----------|----------|--------------|
| `номер` | Номер Issue (обязательный) | — |
| `--auto` | Автоматический режим (вызван из issue-execute) | false |
| `--strict` | Строгий режим (любое замечание = блокировка) | false |

**Примеры:**
- `/issue-review 123` — ревью Issue #123
- `/issue-review 123 --auto` — автоматический режим после issue-execute
- `/issue-review 123 --strict` — строгий режим

## Когда использовать

| Ситуация | Действие |
|----------|----------|
| После `/issue-execute` | Вызывается автоматически |
| Перед созданием PR | Проверить качество решения |
| После доработок | Повторное ревью |

## Правила

### Критерии ревью

**Правило:** Ревью проверяет 4 аспекта решения.

| Аспект | Что проверяется | Блокирует |
|--------|-----------------|:---------:|
| Качество кода | Линтинг, типизация, стиль | ✅ |
| Оптимальность | Можно ли реализовать лучше | ⚠️ |
| Тесты | Тесты проходят, покрытие | ✅ |
| Критерии готовности | Все пункты из Issue выполнены | ✅ |

**Легенда:**
- ✅ — блокирует закрытие при провале
- ⚠️ — предупреждение, решение за пользователем

### Оценка качества

**Правило:** При оценке "можно ли реализовать лучше" проверять:

| Критерий | Вопрос |
|----------|--------|
| Читаемость | Понятен ли код без комментариев? |
| Простота | Нет ли избыточной сложности? |
| Переиспользование | Используются ли существующие утилиты? |
| Производительность | Нет ли очевидных проблем? |
| Безопасность | Нет ли уязвимостей? |

## Воркфлоу

### Шаг 1: Получить контекст

1. Получить информацию об Issue:
   ```bash
   gh issue view {номер} --json title,body,labels
   ```

2. Найти изменённые файлы:
   ```bash
   git diff main --name-only
   ```

3. Определить scope ревью:
   ```
   📋 Ревью Issue #{номер}

   Заголовок: {title}
   Изменённые файлы: {count}

   Файлы:
   - src/auth/handlers.ts
   - src/auth/utils.ts
   - tests/auth/handlers.test.ts
   ```

### Шаг 2: Проверить качество кода

1. Запустить линтер (если настроен):
   ```bash
   npm run lint
   # или
   make lint
   ```

2. Проверить типы (если TypeScript):
   ```bash
   npm run typecheck
   # или
   npx tsc --noEmit
   ```

3. Результат:
   ```
   🔍 Качество кода

   Линтинг: ✅ Пройден (0 ошибок, 2 предупреждения)
   Типизация: ✅ Пройдена
   ```

   Или при ошибках:
   ```
   🔍 Качество кода

   Линтинг: ❌ 3 ошибки
   - src/auth/handlers.ts:45 — unused variable
   - src/auth/handlers.ts:67 — missing return type
   - src/auth/utils.ts:12 — any type

   Типизация: ❌ 1 ошибка
   - src/auth/handlers.ts:45 — Type 'string' is not assignable to 'number'
   ```

### Шаг 3: Оценить реализацию

**Правило:** Прочитать изменённый код, собрать ВСЕ предложения по улучшению, показать списком.

1. Прочитать каждый изменённый файл
2. Оценить по критериям из [Оценка качества](#оценка-качества)
3. **Собрать все предложения** (НЕ применять по одному):

   ```
   💡 Предложения по улучшению

   Найдено предложений: 3

   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

   [1] 📄 src/auth/handlers.ts:45-52

   Текущий код:
   ```typescript
   const result = [];
   for (const item of items) {
     if (item.active) {
       result.push(item.name);
     }
   }
   ```

   Предложение:
   ```typescript
   const result = items.filter(item => item.active).map(item => item.name);
   ```

   Причина: Более читаемый функциональный стиль

   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

   [2] 📄 src/auth/utils.ts:12

   Текущий код:
   ```typescript
   if (token === null || token === undefined) {
   ```

   Предложение:
   ```typescript
   if (!token) {
   ```

   Причина: Упрощение проверки

   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

   [3] 📄 src/auth/handlers.ts:78

   Текущий код:
   ```typescript
   console.log('Debug:', data);
   ```

   Предложение: Удалить отладочный вывод

   Причина: Не должно быть в продакшн-коде

   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

   Применить:
   [A] Все  [1,2,3] Выборочно  [N] Пропустить все
   ```

4. При согласии — применить выбранные через `Edit`

5. После применения:
   ```
   💡 Оценка реализации: ✅ Улучшено (3 изменения)
   ```
   или при отсутствии предложений:
   ```
   💡 Оценка реализации: ✅ Оптимально
   ```

### Шаг 4: Проверить тесты

1. Запустить тесты:
   ```bash
   npm test
   # или
   make test
   ```

2. Результат:
   ```
   🧪 Тесты

   Статус: ✅ Все тесты пройдены (24 passed)
   Покрытие: 85% (порог: 80%)
   ```

   Или при ошибках:
   ```
   🧪 Тесты

   Статус: ❌ 2 теста провалены

   FAIL src/auth/handlers.test.ts
     ✕ should validate token (45ms)
     ✕ should refresh expired token (23ms)

   Исправить и повторить? [Y/n]
   ```

3. При `Y` — попытаться исправить тесты, затем повторить

### Шаг 5: Проверить критерии готовности

1. Извлечь критерии из body Issue (чек-лист)
2. Проверить каждый пункт:

   ```
   📋 Критерии готовности

   - [x] Реализована авторизация через Google
   - [x] Реализована авторизация через GitHub
   - [x] Написаны тесты
   - [ ] Обновлена документация ⚠️

   Не выполнено: 1 пункт
   ```

3. Если есть невыполненные:
   ```
   ⚠️ Не все критерии выполнены

   Невыполненный пункт: "Обновлена документация"

   Варианты:
   [1] Выполнить сейчас
   [2] Пропустить (критерий неактуален)
   [3] Прервать ревью
   ```

### Шаг 7: Применить исправления (если есть)

**Правило:** Если на шагах 2-5 были найдены и согласованы исправления кода (НЕ документации) — применить их сразу и вернуться к Шагу 2.

**Алгоритм:**

1. Проверить, были ли согласованы изменения:
   - Шаг 2: ошибки линтинга/типов
   - Шаг 3: предложения по улучшению
   - Шаг 4: исправления тестов

2. Если изменения есть И это НЕ документация:
   ```
   🔄 Применяю исправления...

   Применено:
   - src/auth/handlers.ts — 2 изменения
   - src/auth/utils.ts — 1 изменение

   Возвращаюсь к проверке качества кода...
   ```

   **→ Перейти к Шагу 2** (повторная проверка)

3. Если изменения только в документации:
   ```
   📝 Изменения в документации требуют ручной проверки.

   Файлы:
   - doc/auth/README.md

   Применить? [Y/n]
   ```

4. Если изменений нет или все применены и проверка пройдена:
   ```
   ✅ Все проверки пройдены, изменений больше нет.
   ```

   **→ Перейти к Шагу 8**

### Шаг 8: Принять решение

**Если ревью пройдено (все проверки ✅):**

```
✅ Ревью пройдено. Вызываю /issue-complete {номер}...
```

Вызвать [/issue-complete](/.claude/skills/issue-complete/SKILL.md):

```
/issue-complete {номер}
```

**Если ревью не пройдено (есть ❌):**

```
❌ Ревью не пройдено

Issue #{номер} остаётся открытым.
Исправьте проблемы и запустите /issue-review {номер} повторно.
```

НЕ вызывать issue-complete.

```

✅ Шаг 5: Проверил критерии готовности
✅ Шаг 6: Сформировал отчёт
✅ Шаг 7: Применил исправления (цикл к Шагу 2 при необходимости)
✅ Шаг 8: Принял решение (complete / reject)
```

## Обработка ошибок

| Ошибка | Действие |
|--------|----------|
| Issue не найден | Сообщить: "Issue #{num} не найден" |
| Issue закрыт | Сообщить: "Issue #{num} уже закрыт" |
| Нет изменённых файлов | Предупредить: "Нет изменений для ревью" |
| Линтер не настроен | Пропустить проверку, предупредить |
| Тесты не настроены | Пропустить проверку, предупредить |
| Нет критериев в Issue | Спросить подтверждение выполнения |

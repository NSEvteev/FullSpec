# Создание инструкции

Команда для создания новой инструкции в `/.claude/instructions/`.

**Связанные скиллы:**
- [instruction-update](/.claude/skills/instruction-update/SKILL.md) — проверка файлов на соответствие
- [instruction-delete](/.claude/skills/instruction-delete/SKILL.md) — удаление инструкций
- [links-update](/.claude/skills/links-update/SKILL.md) — синхронизация ссылок
- [context-update](/.claude/skills/context-update/SKILL.md) — распространение контекста
- [prompt-update](/.claude/skills/prompt-update/SKILL.md) — улучшение промта перед созданием
- [skill-create](/.claude/skills/skill-create/SKILL.md) — создание скилла при необходимости (Шаг 11)

**Связанные инструкции:**

- [output-formats.md](/.claude/templates/output-formats.md) — форматы вывода (SSOT)

## Формат вызова

```
/instruction-create <путь> [--dry-run] [--auto]
```

| Параметр | Описание | По умолчанию |
|----------|----------|--------------|
| `путь` | Путь к инструкции (относительно `/.claude/instructions/`) | — (обязательный) |
| `--dry-run` | Показать план без создания | false |
| `--auto` | Автоматический режим (без подтверждений) | false |

**Примеры:**
- `/instruction-create src/api/design.md`
- `/instruction-create git/commits.md`
- `/instruction-create platform/observability/metrics.md --dry-run`
- `/instruction-create src/api/design.md --auto`

> 💡 **Совет:** Если запрос неоднозначный или требует уточнения, используйте [/prompt-update](/.claude/skills/prompt-update/SKILL.md) для улучшения промта перед созданием инструкции:
> ```
> /prompt-update создай инструкцию для API
> ```

### Режим --auto

В автоматическом режиме:
- Содержимое генерируется без подтверждения
- Ревью-предложения применяются автоматически
- Создание новых скиллов пропускается (только анализ)
- Подтверждения не запрашиваются

```
/instruction-create src/api/design.md --auto

[Создание без интерактивных вопросов]

✅ Инструкция создана: /.claude/instructions/src/api/design.md
```

**Режим --dry-run:**
```
/instruction-create src/api/design.md --dry-run

📋 Предварительный просмотр (--dry-run)

Будет создано:
- /.claude/instructions/src/api/design.md

Будет обновлено:
- /.claude/instructions/README.md (статус ✅)
- /refactoring.md (отметка ✓)

Будет вызвано:
- /links-update /.claude/instructions/src/api/design.md
- /context-update /.claude/instructions/src/api/design.md

ℹ️ Изменения НЕ применены (--dry-run)
```

## Правила

### Расположение инструкций

Инструкции находятся в `/.claude/instructions/` и зеркалируют структуру проекта:

| Папка проекта | Инструкции |
|---------------|------------|
| `/src/` | `/.claude/instructions/src/` |
| `/platform/` | `/.claude/instructions/platform/` |
| `/tests/` | `/.claude/instructions/tests/` |
| `/doc/` | `/.claude/instructions/doc/` |
| `/shared/` | `/.claude/instructions/shared/` |
| `/config/` | `/.claude/instructions/config/` |

**Внутренние инструкции:**
- `/.claude/instructions/git/` — Git workflow
- `/.claude/instructions/tools/` — агенты и скиллы

### Формат названия

- Только латиница, цифры, дефис
- Расширение `.md`
- Примеры: `design.md`, `error-handling.md`, `feature-flags.md`

### Обновление статусов

При создании инструкции **обязательно** отметить её как созданную:
- В [README.md](/.claude/instructions/README.md) — значок ✓
- В [refactoring.md](/refactoring.md) — значок ✓

## Воркфлоу

### Шаг 1: Получить путь

1. Из аргумента: `/instruction-create src/api/design.md`
2. Или спросить: "Какую инструкцию создать? (путь относительно /.claude/instructions/)"

**Валидация:**
- Путь начинается с одной из папок: `src/`, `platform/`, `tests/`, `doc/`, `shared/`, `config/`, `git/`, `tools/`
- Файл имеет расширение `.md`
- Путь соответствует структуре из [refactoring.md](/refactoring.md)

### Шаг 1b: Fail-fast проверки (БЛОКИРУЮЩИЕ)

> ⚠️ **КРИТИЧНО:** Эти проверки выполняются ДО начала создания инструкции. При любой ошибке — немедленная остановка.

**Проверки:**

| Проверка | Команда | При ошибке |
|----------|---------|------------|
| Папка `.claude/instructions/` существует | `ls .claude/instructions/` | Создать или сообщить об ошибке |
| Git status чистый в `.claude/` | `git status .claude/ --porcelain` | Предупредить о несохранённых изменениях |
| Путь соответствует допустимым папкам | Проверка regex | Показать список допустимых папок |
| Файл refactoring.md существует | `ls refactoring.md` | Сообщить о необходимости файла |
| Файл README.md инструкций существует | `ls .claude/instructions/README.md` | Создать базовый индекс |

**Формат вывода при ошибке:**

```
❌ Fail-fast проверка не пройдена

Проверка: {название проверки}
Проблема: {описание}

Решение: {что сделать}

Продолжить невозможно. Исправьте проблему и попробуйте снова.
```

**Формат вывода при успехе:**

```
✅ Все предварительные проверки пройдены

- Папка .claude/instructions/ существует
- Git status чистый
- Путь src/api/design.md корректен
- Файлы refactoring.md и README.md найдены

Продолжаю создание...
```

### Шаг 2: Проверка существования

```bash
ls .claude/instructions/{путь}
```

Если файл существует — сообщить и спросить:
- Перезаписать?
- Открыть для редактирования?
- Отменить?

### Шаг 3: Сгенерировать содержимое

**Краткий алгоритм:**
1. Найти описание в refactoring.md (после `←`)
2. Определить тип из README.md (`standard` / `project`)
3. Прочитать контекст папки (существующие инструкции)
4. Сгенерировать по шаблону

**Детали:**

**3.1. Найти описание в refactoring.md:**

1. Открыть [refactoring.md](/refactoring.md)
2. Найти строку с именем файла инструкции в разделе `/.claude/instructions/`
3. Извлечь описание из комментария после `←`

**Алгоритм поиска:**
1. Grep по refactoring.md: `{имя-файла}\s+←`
2. Извлечь текст после `←` до конца строки
3. Если не найдено — спросить описание у пользователя

**Пример поиска:**
```
/.claude/instructions/
  /src/
    /api/
      design.md                  ← URL, методы, статусы
```
Описание: "URL, методы, статусы"

**3.2. Определить тип инструкции:**

1. Открыть [README.md](/.claude/instructions/README.md) — единый индекс всех инструкций
2. Найти строку с инструкцией в таблице соответствующего раздела
3. Прочитать значение столбца "Тип"

**Типы инструкций:**

| Тип | Назначение | Когда использовать |
|-----|------------|-------------------|
| `standard` | Требования, КАК делать | Стандарты качества, применимы к любому проекту |
| `project` | Специфика, ЧТО есть | Описывает конкретный проект, структуру, конфигурации |

**Пример поиска типа:**
```
| [design.md](./src/api/design.md) | URL naming... | standard | ⬜ | ⬜ |
```
Тип: `standard`

**3.3. Прочитать контекст:**

1. В том же [README.md](/.claude/instructions/README.md) найти раздел папки назначения
2. Понять общий контекст и стиль инструкций в этой категории
3. Посмотреть существующие инструкции как примеры

**3.4. Сгенерировать содержимое:**

| Раздел | Источник |
|--------|----------|
| Заголовок | Имя файла → человекочитаемое название |
| Описание | Комментарий из refactoring.md |
| Правила | Анализ предметной области + существующие паттерны |
| Примеры | Конкретные примеры кода/конфигов из проекта |
| Связанные | Другие инструкции в той же папке |

**Структура инструкции:**
- Заголовок
- Краткое описание (из комментария в refactoring.md)
- Основные правила (2-5 правил)
- Примеры (минимум 1-2)
- Связанные инструкции

### Шаг 4: Создать файл

```bash
# Если нужна подпапка
mkdir -p .claude/instructions/{папка}
```

Записать содержимое в файл.

### Шаг 5: Ревью

Провести ревью созданной инструкции:

**Формат вывода:**
```
📋 Ревью инструкции {путь}

✅ Сильные стороны:
- {что хорошо в инструкции}

💡 Предложения по улучшению:
1. {улучшение 1}
2. {улучшение 2}

Применить улучшения? [Y/n/выборочно]
```

**Что проверять:**
- **Frontmatter:**
  - Поле `type` указано (`standard` или `project`)
  - Поле `description` заполнено
  - Поле `related` содержит связанные инструкции
- **Структура:**
  - Соответствие шаблону (заголовок, оглавление, разделы)
  - Наличие обязательных разделов (Правила, Примеры, Связанные инструкции)
- **Содержание:**
  - Полнота правил (достаточно ли для понимания?)
  - Качество примеров (конкретные, из проекта)
  - Связанные инструкции (все ли указаны?)

**Ревью блокирует!** Нельзя переходить к следующему шагу, пока не применены улучшения или пользователь явно отказался.

### Шаг 6: Обновить статусы

1. **Индекс инструкций** [README.md](/.claude/instructions/README.md):
   - Найти соответствующую таблицу (раздел папки)
   - Если инструкция уже в таблице → изменить статус с ⬜ на ✅
   - Если инструкции нет в таблице → добавить новую строку со ссылкой

   **Формат строки в таблице:**
   ```markdown
   | [design.md](./src/api/design.md) | URL naming, HTTP методы, статус-коды | ✅ |
   ```

2. **Файл рефакторинга** [refactoring.md](/refactoring.md):
   - Найти файл в структуре `/.claude/`
   - Добавить ✓ в конце строки

**Формат отметки в refactoring.md:**
```
design.md                         ← URL, методы, статусы ✓
```

> ⚠️ **Важно:** Ссылка в README.md обязательна! Формат: `[имя-файла.md](./путь/к/файлу.md)`

### Шаг 7: Синхронизация ссылок

Вызвать [/links-update](/.claude/skills/links-update/SKILL.md) для обновления ссылок в связанных документах:

```
/links-update .claude/instructions/{путь}
```

**Что делает links-update:**
1. Находит документы, которые должны ссылаться на новую инструкцию
2. Добавляет ссылки в разделы "Связанные инструкции"
3. Обновляет индексы папок

> **Примечание:** Скилл links-update может запросить подтверждение изменений. Следуйте его воркфлоу.

### Шаг 8: Обновить контекст

Вызвать [/context-update](/.claude/skills/context-update/SKILL.md) для распространения контекста:

```
/context-update .claude/instructions/{путь}
```

**Что делает context-update:**
1. Находит связанные документы по графу зависимостей
2. Обновляет их контекст с учётом новой инструкции
3. Синхронизирует описания и ссылки

> **Примечание:** Скилл context-update запрашивает подтверждение для каждого изменения. Следуйте его воркфлоу.

### Шаг 9: Проверить соответствие проекта

Вызвать [/instruction-update](/.claude/skills/instruction-update/SKILL.md) для проверки файлов проекта на соответствие новой инструкции:

```
/instruction-update {путь}
```

**Что делает instruction-update:**
1. Определяет зону ответственности инструкции
2. Находит файлы в этой зоне
3. Проверяет соответствие правилам
4. Добавляет раздел TODO в инструкцию (если есть несоответствия)

> **Примечание:** Этот шаг опционален для инструкций типа `project`, где проект ещё не содержит соответствующих файлов.

### Шаг 10: Обновление связанных скиллов (ОБЯЗАТЕЛЬНО)

**Правило:** Если инструкция относится к области, которую покрывают скиллы — эти скиллы ДОЛЖНЫ быть обновлены.

**Алгоритм поиска связанных скиллов:**

**Способ 1: По известным соответствиям (быстро)**

| Путь инструкции | Связанные скиллы | Категория скиллов |
|-----------------|------------------|-------------------|
| `git/issues.md` | issue-* | git |
| `git/workflow.md` | issue-execute | git |
| `git/commits.md` | issue-* | git |
| `git/review.md` | issue-review | git |
| `tools/documentation.md` | doc-* | documentation |
| `tools/skills.md` | skill-* | skill-management |
| `tools/agents.md` | agent-* (если появятся) | agent-management |

**Способ 2: Поиск по категории скиллов (рекомендуемый)**

1. Определить категорию из пути инструкции:
   - `git/*.md` → категория `git` (issue-* скиллы)
   - `tools/documentation.md` → категория `documentation` (doc-* скиллы)
   - `tools/skills.md` → категория `skill-management` (skill-* скиллы)
   - `tools/agents.md` → категория `agent-management` (agent-* скиллы)
   - `src/api/*.md`, `platform/*.md`, `config/*.md` → **нет прямых скиллов** (использовать Способ 3-4)

2. Найти скиллы этой категории в индексе [skills.md](/.claude/instructions/tools/skills.md):
   ```bash
   # Прочитать skills.md, найти секцию категории
   grep -A 20 "### git" .claude/instructions/tools/skills.md
   ```

**Способ 3: Полнотекстовый поиск (универсальный)**

```bash
# Поиск скиллов, которые упоминают путь или ключевые слова инструкции
grep -r "git/issues" .claude/skills/*/SKILL.md
grep -r "issues.md" .claude/skills/*/SKILL.md

# Поиск скиллов, которые работают с той же областью
grep -r "GitHub Issue" .claude/skills/*/SKILL.md
```

**Способ 4: Анализ содержимого (глубокий)**

1. Прочитать описание инструкции
2. Извлечь ключевые понятия: `Issues`, `commits`, `branches`, `PR`
3. Найти скиллы, которые работают с этими понятиями:
   ```bash
   for keyword in "Issue" "commit" "branch" "PR"; do
     grep -l "$keyword" .claude/skills/*/SKILL.md
   done
   ```

**Алгоритм обновления:**

1. Используй комбинацию способов выше для поиска связанных скиллов
2. **Для каждого найденного скилла:**
   - Прочитать SKILL.md
   - Проверить наличие раздела "Связанные инструкции"
   - Если раздела нет — добавить после "Связанные скиллы"
   - Если есть — добавить ссылку на новую инструкцию

3. **Формат добавления:**

   ```markdown
   **Связанные инструкции:**
   - [git/issues.md](/.claude/instructions/git/issues.md) — правила работы с Issues
   - [git/workflow.md](/.claude/instructions/git/workflow.md) — правила работы с ветками
   ```

**Пример 1: git/issues.md**

Создаётся инструкция `git/issues.md`:
1. Категория: `git` → скиллы категории `git`
2. Поиск в skills.md → issue-create, issue-update, issue-execute, issue-review, issue-complete, issue-delete
3. Обновить все 6 скиллов

**Пример 2: src/api/design.md**

Создаётся инструкция `src/api/design.md`:
1. Категория: `src` → скиллы работы с кодом
2. Поиск по ключевым словам "API", "design" → возможно doc-* скиллы
3. Если doc-* скиллы работают с `/src/` — добавить ссылку

**Пример 3: platform/observability/metrics.md**

Создаётся инструкция `platform/observability/metrics.md`:
1. Категория: `platform` → пока нет специальных скиллов
2. Полнотекстовый поиск по "metrics", "observability" → нет совпадений
3. Связанных скиллов нет — пропустить шаг

**Ожидаемый вывод:**

```
📋 Обновление связанных скиллов

Инструкция: git/issues.md

Поиск связанных скиллов...
- Категория: git
- Ключевые слова: Issue, Issues, задача

Найдено скиллов: 6

Скиллы для обновления:
- issue-create/SKILL.md
- issue-update/SKILL.md
- issue-execute/SKILL.md
- issue-complete/SKILL.md
- issue-delete/SKILL.md
- issue-review/SKILL.md

### Шаг 11: Анализ и предложение новых скиллов

**Цель:** Проанализировать инструкцию и предложить создание скиллов, которые формализуют описанные в ней операции.

**Когда актуален:**
- Инструкция описывает повторяющиеся операции (создание, обновление, удаление чего-либо)
- Операции имеют чёткий воркфлоу (шаги, правила, валидация)
- Автоматизация повысит качество и консистентность выполнения

**Алгоритм анализа:**

**1. Извлечь объекты и операции из инструкции:**

Найти в тексте:
- Существительные (объекты): Issue, commit, branch, PR, документ, конфигурация
- Глаголы (операции): создать, обновить, удалить, закрыть, объединить

## FAQ / Troubleshooting

### Какие типы инструкций существуют?

| Тип | Назначение | При инициализации |
|-----|------------|-------------------|
| `standard` | Стандарты качества (КАК делать) | Использовать as-is |
| `project` | Специфика проекта (ЧТО есть) | Заполнить под проект |

### Что если инструкция уже существует?

На Шаге 3 скилл предложит:
1. **Перезаписать** — создать заново
2. **Обновить** — вызвать `/instruction-update`
3. **Отменить**

### Как заполнить project-инструкцию?

1. Скилл создаёт шаблон с TODO-секциями
2. Откройте файл и заполните:
   - Удалите `{TODO: ...}` плейсхолдеры
   - Добавьте проектную специфику
3. Вызовите `/instruction-update` для валидации

### Почему нужно подтверждение?

Инструкции — критичные файлы проекта. Подтверждение `[Y/n]` на Шаге 7 защищает от случайных изменений.

### Как отменить созданную инструкцию?

```bash
rm /.claude/instructions/{path}.md
git checkout -- /.claude/instructions/README.md
```

### Какие скиллы вызываются после создания?

| Скилл | Зачем |
|-------|-------|
| `/links-update` | Синхронизация ссылок |
| `/context-update` | Распространение контекста |
| `/instruction-update` | Обновление существующих инструкций |

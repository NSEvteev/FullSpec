# Создание документации

Команда для автоматического создания документации при добавлении нового файла в `/src/`.

**Связанные скиллы:**
- [doc-update](/.claude/skills/doc-update/SKILL.md) — обновление документации
- [doc-delete](/.claude/skills/doc-delete/SKILL.md) — пометка документации при удалении
- [links-update](/.claude/skills/links-update/SKILL.md) — синхронизация ссылок

**Связанные инструкции:**
- [src/documentation.md](/.claude/instructions/src/documentation.md) — правила документирования кода
- [scope-detection.md](/.claude/templates/scope-detection.md) — SSOT для определения scope (раздел "Scope для документации")

## Формат вызова

```
/doc-create <путь-к-файлу-в-src> [--dry-run]
```

| Параметр | Описание | По умолчанию |
|----------|----------|--------------|
| `путь-к-файлу` | Файл в `/src/` для документирования | — (обязательный) |
| `--dry-run` | Показать план без создания | false |

**Примеры:**
- `/doc-create /src/auth/backend/handlers.ts`
- `/doc-create /src/notification/services/email.py`
- `/doc-create /src/auth/utils/jwt.ts --dry-run`

**Режим --dry-run:**
```
/doc-create /src/auth/backend/handlers.ts --dry-run

📋 Предварительный просмотр (--dry-run)

Будет создано:
- /doc/src/auth/backend/handlers.md

Будет обновлено:
- /src/auth/backend/handlers.ts (добавлена ссылка на документацию)

Будет вызвано:
- /links-update /doc/src/auth/backend/handlers.md

ℹ️ Изменения НЕ применены (--dry-run)
```

## Правила

### Маппинг путей

**Правило:** Документация зеркалирует структуру `/src/` в `/doc/src/`.

```
/src/{service}/{path}.{ext} → /doc/src/{service}/{path}.md
```

| Исходный файл | Документация |
|---------------|--------------|
| `/src/auth/backend/handlers.ts` | `/doc/src/auth/backend/handlers.md` |
| `/src/notification/services/email.py` | `/doc/src/notification/services/email.md` |
| `/src/payment/database/schema.sql` | `/doc/src/payment/database/schema.md` |

**Связанная инструкция:** [documentation.md](/.claude/instructions/src/documentation.md)

### Формат ссылки в коде

**Правило:** В начало исходного файла добавляется ссылка на документацию.

**Python:**
```python
"""
{Описание модуля}

Документация: /doc/src/{service}/{path}.md
"""
```

**TypeScript/JavaScript:**
```typescript
/**
 * {Описание модуля}
 *
 * Документация: /doc/src/{service}/{path}.md
 */
```

**SQL:**
```sql
-- {Описание}
-- Документация: /doc/src/{service}/{path}.md
```

### Ссылки на файлы

**Правило:** Все файлы и папки репозитория в документе должны быть оформлены как ссылки.

**Связанный скилл:** [links-create](/.claude/skills/links-create/SKILL.md)

## Воркфлоу

### Шаг 1: Получить путь к файлу

1. Из аргумента: `/doc-create /src/auth/backend/handlers.ts`
2. Или спросить: "Какой файл задокументировать?"
3. Валидация:
   - Файл существует
   - Путь начинается с `/src/`
   - Файл не является директорией

### Шаг 2: Определить путь документации

Преобразовать путь по правилу маппинга:

```
/src/auth/backend/handlers.ts → /doc/src/auth/backend/handlers.md
```

### Шаг 3: Проверить существование

1. Проверить, существует ли документация
2. Если существует — спросить:

```
⚠️ Документация уже существует: /doc/src/auth/backend/handlers.md

Варианты:
[1] Перезаписать
[2] Открыть для редактирования (/doc-update)
[3] Отменить

Что выбрать?
```

### Шаг 4: Прочитать исходный файл

1. Определить язык по расширению
2. Извлечь:
   - Название модуля (из имени файла или первого класса/функции)
   - Публичные функции/классы
   - Импорты (зависимости)

**Поддерживаемые языки:**

| Расширение | Язык | Как извлекать API |
|------------|------|-------------------|
| `.ts`, `.js` | TypeScript/JavaScript | `export function`, `export class` |
| `.py` | Python | `def`, `class` без `_` префикса |
| `.go` | Go | Функции с заглавной буквы |
| `.sql` | SQL | `CREATE TABLE`, `CREATE FUNCTION` |

### Шаг 5: Создать документацию

1. Создать папку если не существует:
   ```bash
   mkdir -p /doc/src/{service}/{path}/
   ```

2. Создать файл по шаблону (см. [Шаблон документации](#шаблон-документации))

3. Заполнить:
   - Название модуля
   - Ссылку на исходный код
   - Список API (функции/классы)
   - Зависимости

### Шаг 6: Добавить ссылку в исходный файл

1. Определить формат комментария по языку
2. Добавить в начало файла (после shebang/pragma если есть)
3. Использовать Edit tool

### Шаг 7: Синхронизация ссылок

Вызвать скилл [/links-update](/.claude/skills/links-update/SKILL.md) для обновления связанных документов:

```
/links-update /doc/src/{service}/{path}.md
```

**Что обновляется:**
- README.md сервиса — добавить ссылку на новую документацию
- Связанные документы — если новый файл упоминается

### Шаг 8: Проверка по чек-листу

Перед завершением проверить выполнение всех пунктов:

```
✅ Шаг 1: Получил путь к файлу
✅ Шаг 2: Определил путь документации
✅ Шаг 3: Проверил существование
✅ Шаг 4: Прочитал исходный файл
✅ Шаг 5: Создал документацию
✅ Шаг 6: Добавил ссылку в исходный файл
✅ Шаг 7: Вызвал /links-update
```

### Шаг 9: Результат

Вывести полный отчёт:

```
✅ Документация создана

Файлы:
- Документация: /doc/src/auth/backend/handlers.md
- Исходный файл: /src/auth/backend/handlers.ts (добавлена ссылка)

Извлечено:
- Функций: 3 (login, logout, validateToken)
- Зависимостей: 2

Синхронизация:
- /links-update выполнен

Следующие шаги:
- Заполнить описания функций в документации
- Добавить примеры использования
```

## Обработка ошибок

| Ошибка | Действие |
|--------|----------|
| Файл не существует | Сообщить: "Файл {путь} не найден" |
| Файл не в `/src/` | Сообщить: "Скилл работает только с файлами в /src/" |
| Документация существует | Спросить: перезаписать / редактировать / отменить |
| Не удалось извлечь API | Создать минимальный шаблон, предупредить |
| Ошибка записи | Сообщить, откатить изменения |

**Откат при ошибке:**

```bash
# Удалить созданную документацию
rm -f /doc/src/{service}/{path}.md

# Откатить изменения в исходном файле
git checkout -- /src/{service}/{path}.{ext}
```

## Шаблон документации

```markdown
# {Название модуля}

> Исходный код: [{filename}](/{path-to-source})

{Краткое описание — заполнить вручную}

## API

### {FunctionName}

{Описание — заполнить вручную}

**Сигнатура:**
\`\`\`{language}
{signature}
\`\`\`

---

## Зависимости

- [{dependency}](/{path}) — {description}
```

## Примеры использования

### Пример 1: TypeScript файл

**Вызов:**
```
/doc-create /src/auth/backend/handlers.ts
```

**Диалог:**
```
> Создаю документацию для /src/auth/backend/handlers.ts

Извлекаю API...
- login(request: LoginRequest): Promise<AuthResponse>
- logout(token: string): Promise<void>
- validateToken(token: string): boolean

Зависимости:
- /src/auth/utils/jwt.ts

Создать документацию? [Y/n]
> Y

✅ Документация создана

Файлы:
- /doc/src/auth/backend/handlers.md
- /src/auth/backend/handlers.ts (добавлена ссылка)

Извлечено функций: 3
```

**Создаётся `/doc/src/auth/backend/handlers.md`:**
```markdown
# Auth Handlers

> Исходный код: [handlers.ts](/src/auth/backend/handlers.ts)

Обработчики аутентификации.

## API

### login

Аутентификация пользователя.

**Сигнатура:**
\`\`\`typescript
async function login(request: LoginRequest): Promise<AuthResponse>
\`\`\`

### logout

Завершение сессии.

**Сигнатура:**
\`\`\`typescript
async function logout(token: string): Promise<void>
\`\`\`

### validateToken

Проверка токена.

**Сигнатура:**
\`\`\`typescript
function validateToken(token: string): boolean
\`\`\`

---

## Зависимости

- [jwt.ts](/src/auth/utils/jwt.ts) — работа с JWT токенами
```

# Валидация входных данных

Микро-скилл для валидации входных параметров. Предназначен для переиспользования в других скиллах.

**Связанные скиллы:**
- [environment-check](/.claude/skills/environment-check/SKILL.md) — проверка окружения (gh, git, python)

**Используется в:**
- [skill-create](/.claude/skills/skill-create/SKILL.md) — валидация названия скилла
- [instruction-create](/.claude/skills/instruction-create/SKILL.md) — валидация пути инструкции
- [doc-create](/.claude/skills/doc-create/SKILL.md) — валидация пути файла

**Шаблоны:**
- [output-formats.md](/.claude/templates/output-formats.md) — форматы вывода (SSOT)

## Формат вызова

```
/input-validate <тип> <значение> [--required] [--silent]
```

| Параметр | Описание | По умолчанию |
|----------|----------|--------------|
| `тип` | Тип валидации (см. таблицу ниже) | Обязательный |
| `значение` | Значение для проверки | Обязательный |
| `--required` | Ошибка если значение пустое | false |
| `--silent` | Не выводить сообщения, только код возврата | false |

---

## Типы валидации

| Тип | Паттерн | Примеры валидных значений |
|-----|---------|---------------------------|
| `skill-name` | `^[a-z]+(-[a-z]+)+$` | `skill-create`, `doc-update` |
| `file-path` | Существует в файловой системе | `/src/auth/login.ts` |
| `dir-path` | Существующая директория | `/.claude/skills/` |
| `md-file` | `.md` файл, существует | `README.md` |
| `issue-number` | `^#?\d+$` | `#123`, `456` |
| `branch-name` | `^[a-zA-Z0-9\-_/]+$` | `feature/auth`, `fix-123` |
| `url` | URL формат | `https://github.com/...` |
| `email` | Email формат | `user@example.com` |
| `semver` | Семантическая версия | `1.2.3`, `0.1.0-beta` |
| `json` | Валидный JSON | `{"key": "value"}` |
| `yaml-frontmatter` | YAML между `---` | Frontmatter в .md |
| `not-empty` | Непустая строка | Любой текст |

## Воркфлоу

### Шаг 1: Получить параметры

1. Тип валидации (обязательный)
2. Значение для проверки
3. Флаги (--required, --silent)

### Шаг 2: Выполнить валидацию

В зависимости от типа:

**skill-name:**
```bash
# Проверка формата
[[ "$value" =~ ^[a-z]+(-[a-z]+)+$ ]]
```

**file-path / md-file:**
```bash
# Проверка существования через Glob
Glob: pattern="$value"
```

**dir-path:**
```bash
# Проверка директории
ls "$value" 2>/dev/null
```

**issue-number:**
```bash
# Проверка формата и существования (опционально)
[[ "$value" =~ ^#?[0-9]+$ ]]
gh issue view "${value#\#}" --json number 2>/dev/null
```

**json:**
```bash
# Проверка валидности JSON
echo "$value" | python -m json.tool 2>/dev/null
```

### Шаг 3: Вернуть результат

**Успех:**
```
✅ Валидация пройдена

Тип: {тип}
Значение: {значение}
```

**Ошибка:**
```
❌ Валидация не пройдена

Тип: {тип}
Значение: {значение}
Ошибка: {описание ошибки}

Ожидаемый формат: {паттерн или описание}
```

**Следующие шаги:**
- При успехе — продолжить выполнение основного скилла
- При ошибке — запросить корректное значение у пользователя
- При batch-валидации — собрать все ошибки перед выводом

## Интеграция в скиллы

### Как использовать в других скиллах

В воркфлоу скилла добавить шаг валидации:

```markdown
### Шаг 1: Валидация входных данных

Вызвать [/input-validate](/.claude/skills/input-validate/SKILL.md):

1. `/input-validate skill-name {название} --required`
2. `/input-validate file-path {путь}` (если указан)

Если валидация не пройдена — остановить выполнение.
```

### Пример интеграции в skill-create

```markdown
### Шаг 1: Получить название

1. Из аргумента: `/skill-create foo-create`
2. Валидация: `/input-validate skill-name foo-create --required`
3. Если ошибка — показать формат и запросить снова
```

### Batch-валидация

Для проверки нескольких параметров:

```
/input-validate skill-name my-skill --required
/input-validate dir-path /.claude/skills/ --required
/input-validate md-file ./README.md
```

## Обработка ошибок

> **SSOT:** [error-handling.md](/.claude/templates/error-handling.md)

| Ошибка | Код | Действие |
|--------|-----|----------|
| Тип не указан | `ERR_NO_TYPE` | Показать список доступных типов |
| Неизвестный тип | `ERR_UNKNOWN_TYPE` | Показать список доступных типов |
| Значение не указано | `ERR_NO_VALUE` | Запросить значение |
| Значение пустое + --required | `ERR_REQUIRED` | Сообщить об обязательности |
| Формат не соответствует | `ERR_FORMAT` | Показать ожидаемый формат |
| Файл не существует | `ERR_NOT_FOUND` | Показать путь и предложить варианты |

## FAQ / Troubleshooting

### Когда использовать input-validate vs ручную проверку?

**input-validate** — когда:
- Проверка выполняется в нескольких скиллах (переиспользование)
- Нужен стандартизированный вывод ошибок
- Требуется batch-валидация нескольких параметров

**Ручная проверка** — когда:
- Уникальная бизнес-логика
- Проверка специфична для одного скилла
- Нужна кастомная обработка ошибок

### Какой тип использовать для проверки существования скилла?

Используйте `dir-path`:
```
/input-validate dir-path /.claude/skills/{skill-name}/
```

### Как проверить несколько условий для одного значения?

Вызывайте валидацию последовательно:
```
/input-validate not-empty $path --required
/input-validate file-path $path
/input-validate md-file $path
```

### Как добавить кастомную валидацию?

1. Добавить новый тип в таблицу "Типы валидации"
2. Описать паттерн или логику проверки
3. Добавить примеры валидных значений
4. Описать обработку в "Шаг 2: Выполнить валидацию"

Пример нового типа:
```markdown
| `docker-image` | `^[\w.-]+(/[\w.-]+)?(:\w+)?$` | `nginx:latest`, `myrepo/app` |
```

### Что делать если валидация не проходит?

1. Проверить формат значения по паттерну в таблице "Типы валидации"
2. Для `file-path`/`dir-path` — проверить существование через `ls`
3. Использовать `--silent` для отладки в batch-режиме
4. Проверить похожие файлы в выводе ошибки

### Как пропустить валидацию в особых случаях?

Валидацию можно пропустить, если:
- Значение опционально и не передано (без `--required`)
- Используется `--silent` и результат игнорируется

**Важно:** Не рекомендуется пропускать валидацию для обязательных параметров — это может привести к ошибкам на поздних этапах выполнения скилла.

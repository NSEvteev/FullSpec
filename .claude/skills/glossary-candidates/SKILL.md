# Поиск кандидатов в глоссарий

Команда анализирует указанный .md документ и добавляет специфичные термины в раздел "Кандидаты" глоссария.

## Инструкции

### Шаг 1: Получить путь к документу

Пользователь должен указать путь к .md файлу для анализа.

Если путь не указан — спросить у пользователя.

### Шаг 2: Прочитать документ

Использовать Read tool для чтения указанного файла.

### Шаг 3: Прочитать текущий глоссарий

Прочитать `general_docs/glossary.md`:
- Раздел "## Термины" — существующие термины
- Раздел "## Кандидаты" — уже добавленные кандидаты

### Шаг 4: Найти специфичные термины

**Искать:**
- Технические термины проекта
- Аббревиатуры (API, SDK, DTO и т.д.)
- Термины в кавычках или курсиве
- Слова с заглавной буквы в середине предложения
- Термины из предметной области (трейдинг, финансы)
- Названия компонентов, модулей, сервисов

**Исключить:**
- Общеизвестные технические термины (HTTP, JSON, SQL)
- Термины, уже существующие в глоссарии
- Термины, уже добавленные как кандидаты
- Имена файлов и пути
- Названия внешних библиотек

### Шаг 5: Добавить кандидатов в глоссарий

Добавить найденные термины в раздел "## Кандидаты" в формате:

```markdown
### [Термин]

- **Источник:** [путь к документу]
- **Контекст:** [цитата из документа, где встречается термин]
- **Статус:** ожидает рассмотрения
```

Использовать Edit tool для обновления `general_docs/glossary.md`.

### Шаг 6: Вывести результат

Показать:
1. Список найденных кандидатов
2. Количество добавленных терминов
3. Термины, которые были пропущены (уже существуют)

## Примеры использования

### Пример 1: Анализ новой дискуссии

**Сценарий:** Создан новый документ дискуссии `01_discuss/005_microservices_architecture.md` с описанием микросервисной архитектуры. Документ содержит специфичные термины проекта (Service Mesh, API Gateway, Event Bus). Нужно найти эти термины и добавить их как кандидатов в глоссарий.

**Команда:**
```bash
/glossary-candidates general_docs/01_discuss/005_microservices_architecture.md
```

**Ожидаемый результат:**
- Анализ документа на наличие технических терминов
- Исключение общеизвестных терминов (HTTP, JSON, Docker)
- Добавление специфичных терминов в раздел "Кандидаты" глоссария
- Сохранение источника и контекста для каждого термина

**Вывод (пример):**
```
Проанализирован: general_docs/01_discuss/005_microservices_architecture.md

Найдено кандидатов: 8
- Service Mesh (строка 23)
- API Gateway (строка 45)
- Event Bus (строка 67)
- Circuit Breaker (строка 89)
- Service Registry (строка 102)
- Load Balancer (строка 124)
- Health Check Endpoint (строка 156)
- Distributed Tracing (строка 178)

Пропущено (уже в глоссарии): 3
- Архитектура
- Ресурс
- Decision (ADR)

Пропущено (общеизвестные термины): 5
- HTTP
- JSON
- Docker
- Kubernetes
- REST

✓ Кандидаты добавлены в general_docs/glossary.md
```

### Пример 2: Анализ архитектурного документа

**Сценарий:** Создан архитектурный документ `02_architecture/004_auth_flow.md` с подробным описанием процесса аутентификации. Документ содержит аббревиатуры (JWT, OAuth2, RBAC) и специфичные термины. Нужно найти термины, которых нет в глоссарии.

**Команда:**
```bash
/glossary-candidates general_docs/02_architecture/004_auth_flow.md
```

**Ожидаемый результат:**
- Поиск аббревиатур и технических терминов
- Исключение терминов, уже добавленных как кандидаты
- Сохранение контекста (цитата из документа)

**Вывод (пример):**
```
Проанализирован: general_docs/02_architecture/004_auth_flow.md

Найдено кандидатов: 6
- JWT (строка 12) — контекст: "Используется JWT для передачи данных пользователя"
- OAuth2 (строка 34) — контекст: "Интеграция с OAuth2 провайдерами"
- RBAC (строка 56) — контекст: "Система прав на основе RBAC"
- Refresh Token (строка 78) — контекст: "Refresh Token хранится в httpOnly cookie"
- Access Token (строка 90) — контекст: "Access Token передаётся в заголовке Authorization"
- Token Rotation (строка 112) — контекст: "Token Rotation при каждом обновлении"

Пропущено (уже в глоссарии): 2
- Агент
- Скилл

Пропущено (уже кандидаты): 1
- API Gateway

✓ Кандидаты добавлены в general_docs/glossary.md

Следующий шаг: Запустить `/glossary-review` для обработки кандидатов
```

### Пример 3: Анализ документа без новых терминов

**Сценарий:** Обновлён документ `01_discuss/001_project_structure.md` — добавлено описание новой папки. Нужно проверить, появились ли новые термины для глоссария.

**Команда:**
```bash
/glossary-candidates general_docs/01_discuss/001_project_structure.md
```

**Ожидаемый результат:**
- Анализ документа
- Все найденные термины уже есть в глоссарии или кандидатах
- Сообщение, что новых терминов не найдено

**Вывод (пример):**
```

Найдено кандидатов: 0

Пропущено (уже в глоссарии): 12
- Дискуссия
- Архитектура
- Decision (ADR)
- Ресурс
- План реализации
- Документация папок
- Цепочка зависимостей
- Индекс
- Агент
- Скилл
- Команда
- Бэклог

✓ Новых терминов не найдено
✓ Все термины документа уже есть в глоссарии
```

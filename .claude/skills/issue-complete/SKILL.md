## Формат вызова

```
/issue-complete [номер | --last] [--pr <номер-pr>] [--comment <текст>]
```

| Параметр | Описание | По умолчанию |
|----------|----------|--------------|
| `номер` | Номер Issue | — |
| `--last` | Использовать последний взятый Issue | false |
| `--pr` | Номер связанного PR | Автоопределение |
| `--comment` | Дополнительный комментарий | — |

**Примеры:**
- `/issue-complete 123` — закрыть Issue #123
- `/issue-complete --last` — закрыть последний Issue из `.claude/state/last-issue-run.json`
- `/issue-complete 123 --pr 456` — закрыть с указанием PR
- `/issue-complete 123 --comment "Реализовано в v1.2.0"` — с комментарием

### Флаг --last

При использовании `--last` скилл читает номер Issue из файла состояния:

```bash
cat .claude/state/last-issue-run.json | jq '.issue'
```

Если файл состояния не существует или пуст:
```
⚠️ Файл состояния не найден: .claude/state/last-issue-run.json

Укажите номер Issue явно:
/issue-complete {номер}
```

## Когда использовать

| Ситуация | Действие |
|----------|----------|
| PR смержен | Закрыть связанный Issue как выполненный |
| Задача выполнена без PR | Закрыть с комментарием о выполнении |
| Задача выполнена частично | Закрыть текущий, создать новый для оставшегося |

## Правила

### Причина закрытия

**Правило:** Issue закрывается как `completed` (выполнено).

```bash
gh issue close 123 --reason completed
```

### Связь с PR

**Правило:** Если Issue закрывается через PR, указать номер PR в комментарии.

```
Выполнено в PR #456
```

### Проверка выполнения

**Правило:** Перед закрытием проверить:
1. Все критерии готовности (acceptance criteria) выполнены
2. Связанный PR смержен (если есть)
3. Метка `in-progress` снята

## Воркфлоу

### Шаг 1: Получить номер Issue

1. Из аргумента: `/issue-complete 123`
2. Или спросить: "Какой Issue закрыть?"

### Шаг 2: Проверить статус

1. Получить информацию об Issue:
   ```bash
   gh issue view 123 --json state,title,labels,assignees,body
   ```

2. Проверить:
   - Issue открыт (state: OPEN)
   - Есть метка `in-progress` (задача была в работе)

3. Если Issue уже закрыт:
   ```
   ⚠️ Issue #123 уже закрыт.

   Статус: {closed_reason}
   Закрыт: {closed_at}
   ```

### Шаг 3: Проверить выполнение задачи

**Правило:** Перед закрытием убедиться, что задача реально выполнена.

1. Прочитать раздел "Критерии готовности" из body Issue
2. Проверить каждый пункт:
   ```
   📋 Критерии готовности Issue #123:

   - [x] Генерация QR-кода
   - [x] Валидация TOTP
   - [ ] Backup codes ⚠️
   - [x] Тесты

   ⚠️ Не все критерии выполнены!

   Варианты:
   [1] Закрыть всё равно (критерий неактуален)
   [2] Отменить закрытие
   [3] Создать новый Issue для невыполненного
   ```

3. Если критериев нет в Issue — спросить:
   ```
   Критерии готовности не найдены в Issue.

   Подтвердите, что задача выполнена: [Y/n]
   ```

### Шаг 4: Проверить связанный PR

1. Поиск PR, связанного с Issue:
   ```bash
   gh pr list --search "closes #123 OR fixes #123 OR resolves #123" --state merged --json number,title
   ```

2. Если найден смерженный PR — использовать его номер
3. Если не найден — спросить:
   ```
   Связанный PR не найден автоматически.

   Варианты:
   [1] Указать номер PR вручную
   [2] Закрыть без PR (задача выполнена иначе)
   ```

4. **Проверить CI статус** (если есть PR):
   ```bash
   gh pr checks {pr-number}
   ```

   - Если все checks passed → продолжить
   - Если есть failed checks:
     ```
     ⚠️ CI checks не прошли для PR #456

     Failed:
     - ❌ test (exit code 1)
     - ❌ lint (exit code 1)

     Варианты:
     [1] Закрыть Issue всё равно (checks неактуальны)
     [2] Отменить закрытие и исправить CI
     ```

   > **Связь с CI:** См. [ci.md](/.claude/instructions/git/ci.md#faq--troubleshooting) для диагностики failed checks.

### Шаг 5: Добавить комментарий

Сформировать комментарий:

**Если есть PR:**
```
✅ Выполнено в PR #456
```

**Если без PR:**
```
✅ Задача выполнена

{дополнительный комментарий, если указан}
```

Добавить комментарий:
```bash
gh issue comment 123 --body "✅ Выполнено в PR #456"
```

### Шаг 6: Закрыть Issue

1. Снять метку `in-progress` (если есть):
   ```bash
   gh issue edit 123 --remove-label "in-progress"
   ```

2. Закрыть как выполненный:
   ```bash
   gh issue close 123 --reason completed
   ```

### Шаг 7: Обновить документацию

**Правило:** После закрытия Issue автоматически обновить документацию.

```
Обновляю документацию...
```

Вызвать скилл [/doc-update](/.claude/skills/doc-update/SKILL.md) для файлов, изменённых в рамках задачи:

```
/doc-update
```

Скилл doc-update:
1. Найдёт изменённые файлы в `/src/`
2. Обновит соответствующую документацию в `/doc/`
3. Добавит ссылки на новый функционал

### Шаг 8: Проверка по чек-листу

```
✅ Шаг 1: Получил номер Issue
✅ Шаг 2: Проверил статус (открыт, in-progress)
✅ Шаг 3: Проверил выполнение критериев готовности
✅ Шаг 4: Нашёл связанный PR + проверил CI статус
✅ Шаг 5: Добавил комментарий
✅ Шаг 6: Закрыл Issue
✅ Шаг 7: Обновил документацию
```

### Шаг 9: Результат

```
✅ Issue #123 закрыт как выполненный

Заголовок: [AUTH] Добавить OAuth авторизацию
Связанный PR: #456
Комментарий: ✅ Выполнено в PR #456

Следующие шаги:
- Удалить ветку (если не удалена автоматически)
- Проверить, что функционал работает в main
```

## FAQ / Troubleshooting

### Как закрыть Issue без PR?

**Сценарий:** Задача выполнена без кода (настройка, документация).

**Команда:**
```
/issue-complete {номер} --comment "Выполнено: {описание}"
```

**Что произойдёт:**
1. Вопрос "Связанный PR не найден. Закрыть без PR?"
2. При подтверждении — Issue закрывается с комментарием

### Критерии готовности не выполнены — что делать?

**Варианты:**
1. **Выполнить сейчас** — доделать задачу
2. **Пропустить** — если критерий неактуален
3. **Создать новый Issue** — для невыполненного критерия

**Команда для варианта 3:**
```
/issue-create --service {сервис} "Доделать: {критерий}"
```

### CI checks не прошли — можно ли закрыть?

**Ответ:** Можно, но не рекомендуется.

**При выборе "Закрыть всё равно":**
- Issue закрывается
- CI проблема остаётся
- Возможны проблемы в main ветке

**Рекомендация:** Исправить CI перед закрытием.

### Как переоткрыть закрытый Issue?

**Команда:**
```bash
gh issue reopen {номер}
```

После переоткрытия можно снова взять в работу:
```
/issue-execute {номер}
```

### doc-update не вызывается — почему?

**Причина:** Скилл `doc-update` вызывается только если есть изменения в `/src/`.

**Для ручного вызова:**
```
/doc-update
```

### Метка in-progress не снимается — что делать?

**Причина:** Ошибка при выполнении `gh issue edit`.

**Ручное исправление:**
```bash
gh issue edit {номер} --remove-label "in-progress"
```

# Распространение контекста

Команда для распространения изменений контекста по графу связанных документов после работы с .md файлом.

**Связанные скиллы:**
- [links-update](/.claude/skills/links-update/SKILL.md) — обновление ссылок (синтаксис)
- [links-create](/.claude/skills/links-create/SKILL.md) — создание ссылок в документе
- [context-delete](/.claude/skills/context-delete/SKILL.md) — очистка контекста при удалении

**Шаблоны:**
- [output-formats.md](/.claude/instructions/skills/output.md) — форматы вывода (SSOT)

## Формат вызова

```
/context-update <файл.md> [--depth N]
```

| Параметр | Описание | По умолчанию |
|----------|----------|--------------|
| `файл.md` | Файл, в котором произошли изменения | — |
| `--depth N` | Глубина обхода графа связей | 2 |

## Отличие от links-update

| Аспект | links-update | context-update |
|--------|--------------|----------------|
| Цель | Обновить **ссылки** (синтаксис) | Обновить **контекст** (семантика) |
| Что меняет | Добавляет/исправляет `[text](path)` | Добавляет/изменяет описания, связи |
| Глубина | Прямые ссылки | Транзитивные связи (A → B → C) |
| Решение | Автоматическое | Предложение пользователю |
| Когда | После создания файла | После изменения содержимого |

## Когда использовать

| Событие | Действие |
|---------|----------|
| Добавлен новый раздел в документ | Проверить, нужно ли обновить связанные документы |
| Изменена логика/архитектура | Распространить изменения по зависимостям |
| Добавлено новое правило | Обновить документы, которые должны следовать правилу |
| Удалён раздел | Проверить, не сломались ли ссылки на него |
| Завершена работа над документом | Синхронизировать контекст со связанными файлами |

## Воркфлоу

### Шаг 1: Определить инкремент контекста

1. Прочитать файл A
2. Если есть git history — сравнить с предыдущей версией:
   ```bash
   git diff HEAD~1 -- файл.md
   ```
3. Если нет — анализировать весь файл как новый контекст

**Определить:**
- Что добавлено (новые разделы, правила, описания)
- Что изменено (переформулировки, уточнения)
- Что удалено (убранные разделы, устаревшая информация)

**Результат:**
```
Инкремент контекста в {файл}:

Добавлено:
- Раздел "Правила ссылок на папки"
- Новый тип связи "Файл/папка → Документы"

Изменено:
- Расширена область применения (не только .md файлы)

Удалено:
- (ничего)
```

### Шаг 2: Найти прямые связи (уровень 1)

1. Найти все ссылки на .md файлы в документе A
2. Использовать скрипт [find_references.py](/.claude/scripts/find_references.py) для поиска обратных ссылок:
   ```bash
   python .claude/scripts/find_references.py файл.md --format json
   ```

**Fallback (если скрипт не существует):**
- Поиск исходящих ссылок: Grep по документу A для паттерна `\[.*\]\(.*\.md\)`
- Поиск входящих ссылок: Grep по всем .md файлам для имени файла A
  ```bash
  grep -r "имя-файла-A" --include="*.md" .
  ```
- Ручная проверка:
  - Индексы: `skills.md`, `agents.md`, `README.md` в папках
  - Связанные скиллы: `{объект}-create`, `{объект}-update`, `{объект}-delete`

**Результат:** Список файлов B1, B2, ... Bn, связанных с A

### Шаг 3: Анализ релевантности

Для каждого файла B из списка:

1. Прочитать B, определить его область ответственности
2. Оценить релевантность изменений A для документа B

**Критерии релевантности:**

| Релевантность | Критерий | Примеры |
|---------------|----------|---------|
| **Высокая** | B напрямую описывает или использует то, что изменилось в A | Индекс скиллов ← новый скилл; Связанный скилл ← изменённые правила |
| **Средняя** | B ссылается на A или на область, затронутую изменениями | CLAUDE.md ← изменения в skills.md |
| **Низкая** | B упоминает A, но изменения не влияют на содержание B | Общая документация ← мелкие правки |
| **Нет связи** | B не связан с изменениями в A | Пропустить, не идти глубже |

**Упрощённый алгоритм:**

```
1. Если B — индекс, а A — элемент индекса → высокая
2. Если B — связанный скилл (тот же объект) → высокая
3. Если B ссылается на A напрямую → средняя
4. Если B в той же папке/категории → средняя
5. Если B упоминает A только в примерах → низкая
6. Иначе → нет связи
```

**Результат:** Список {B, релевантность: высокая/средняя/низкая/нет}

### Шаг 4: Транзитивные связи (уровень 2+)

Для каждого B с релевантностью "высокая" или "средняя":

1. Найти ссылки на .md файлы в B
2. Для каждого файла C:
   - Проверить связь C с инкрементом A
   - Если есть связь — добавить в список
3. Повторить до достижения `--depth`

**Фильтрация:** На следующий уровень идём только если:
- Текущий файл релевантен (высокая/средняя)
- Не достигнута максимальная глубина
- Файл ещё не был обработан (избежание циклов)

### Шаг 5: Сформировать предложения

Для каждого релевантного файла сформировать:

1. **Текстовое описание:**
   ```
   Файл: skills.md
   Связь: A ⊂ B (skill-create входит в индекс скиллов)
   Предложение: Обновить описание skill-create в таблице категории
   ```

2. **Готовый diff:**
   ```diff
   - | skill-create | Создание скилла | /skill-create |
   + | skill-create | Создание скилла по шаблону с валидацией | /skill-create |
   ```

Вывести пользователю:

## Типы связей контекста

### Иерархические

| Тип | Описание | Пример |
|-----|----------|--------|
| Индекс → Элемент | Индекс содержит ссылку на элемент | skills.md → skill-create |
| Родитель → Потомок | Общий документ ссылается на частный | CLAUDE.md → skills.md |

### Семантические

| Тип | Описание | Пример |
|-----|----------|--------|
| Правило → Применение | Правило используется в другом месте | skill-create (правила) → links-create (использует) |
| Зависимость | Один документ зависит от другого | context-update → [find_references.py](/.claude/scripts/find_references.py) |

### По направлению

| Направление | Описание |
|-------------|----------|
| A → B | A ссылается на B (исходящая) |
| A ← B | B ссылается на A (входящая) |
| A ↔ B | Взаимные ссылки |

## FAQ / Troubleshooting

### Чем context-update отличается от links-update?

| Вопрос | links-update | context-update |
|--------|--------------|----------------|
| Что обновляет? | Синтаксис ссылок `[text](path)` | Семантический контекст (описания, связи) |
| Когда использовать? | После создания/перемещения файла | После изменения содержимого |
| Глубина? | Только прямые ссылки | Транзитивные связи (A → B → C) |
| Автоматизация? | Полностью автоматический | Предлагает изменения на одобрение |

**Правило:** Если поменялся путь — links-update. Если поменялось содержание — context-update.

### Как выбрать правильную глубину (--depth)?

| Глубина | Когда использовать |
|---------|-------------------|
| `--depth 1` | Мелкие правки, локальные изменения |
| `--depth 2` (по умолчанию) | Обычные изменения, новые разделы |
| `--depth 3+` | Архитектурные изменения, рефакторинг |

**Совет:** Начните с `--depth 2`. Если видите много "пропущено из-за глубины" — увеличьте.

### Что делать, если найдено слишком много связей?

При >20 связях скилл предложит фильтрацию:

```
Найдено связей: 45

Показать: [1] Все  [2] Только высокая релевантность  [3] По уровням
```

**Рекомендации:**
1. Выберите "Только высокая релевантность" для быстрой обработки
2. Используйте `--depth 1` для локальных изменений
3. Обрабатывайте по частям: сначала уровень 1, потом глубже

### Как избежать циклических зависимостей?

Скилл автоматически отслеживает обработанные файлы и не заходит в них повторно. Если видите предупреждение о цикле:

```
⚠️ Обнаружен цикл: A → B → C → A
Файл A пропущен (уже обработан)
```

Это нормально — просто означает, что граф связей имеет цикл. Изменения применятся корректно.

### Можно ли откатить изменения?

Да, через Git:

```bash
# Посмотреть что изменилось
git diff

# Откатить все
git checkout -- .

# Откатить конкретный файл
git checkout -- путь/к/файлу.md
```

**Совет:** Делайте коммит перед запуском context-update с большой глубиной.

### Что если скрипт find_references.py не найден?

Скилл использует fallback-методы:
1. Grep по паттерну ссылок
2. Поиск по имени файла
3. Проверка стандартных индексов

Функциональность сохраняется, но может быть менее точной. Для полного функционала создайте скрипт по шаблону в `/.claude/scripts/`.

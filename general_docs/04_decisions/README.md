## Назначение

Фиксация принятых архитектурных решений на основе одобренных дискуссий и архитектурных документов.

**Architecture Decision Record (ADR)** — это краткий документ, который:
- Фиксирует **что** решили (1-2 предложения)
- Объясняет **почему** выбрали это решение
- Перечисляет **какие альтернативы** рассмотрели и отклонили
- Описывает **последствия** этого решения

**Отличие от архитектурного документа:**
- **Архитектурный документ** (02_architecture/) — детальное описание системы, как она устроена
- **Decision (ADR)** (04_decisions/) — краткая запись решения, что выбрали и почему

## Правила создания

### Именование файлов
- **Формат:** `DEC-[XXX]_название_решения.md`
- **Примеры:**
  - `DEC-001_auth_jwt.md`
  - `DEC-002_microservices_architecture.md`
  - `DEC-003_database_postgres.md`
- **ID:** Последовательная нумерация (DEC-001, DEC-002, DEC-003...)
- **Уникальность:** ID должен быть уникальным в рамках папки 04_decisions/

### Обязательные разделы

1. **Метаданные** (в начале файла)
   - Статус документа (draft, proposed, approved, final, deprecated, superseded)
   - Дата создания
   - Автор
   - Связанные документы (дискуссии, архитектура)

2. **Решение** (краткая суть)
   - Что именно решили (1-2 предложения)
   - Максимально конкретно и понятно

3. **Контекст**
   - Проблема, которую решаем
   - Требования к решению
   - Ограничения

4. **Обоснование**
   - Почему выбрали это решение
   - Преимущества
   - Недостатки (которые принимаем осознанно)

5. **Рассмотренные альтернативы**
   - Минимум 2 альтернативы
   - Для каждой: описание + почему отклонено
   - Объективное сравнение

6. **Последствия**
   - Что нужно реализовать
   - Изменения в архитектуре
   - Влияние на другие решения
   - Риски и митигация

## Workflow статусов

### Статусы Decision (ADR)

| Статус | Описание | Когда использовать |
|--------|----------|-------------------|
| `draft` | Черновик решения | Первичная фиксация, обсуждение |
| `proposed` | Решение предложено | Готово к обсуждению с командой |
| `approved` | Решение одобрено | Команда согласна, можно реализовывать |
| `final` | Решение финализировано | Реализовано и используется |
| `deprecated` | Решение устарело | Больше не актуально, см. архив |
| `superseded` | Заменено новым | Есть новая версия решения |

### Переходы между статусами

```
draft → proposed → approved → final
                     ↓
                deprecated / superseded → archive/
```

**Правила:**
- `draft` → `proposed`: решение полностью описано, готово к обсуждению
- `proposed` → `approved`: команда одобрила решение
- `approved` → `final`: решение реализовано
- `final` → `deprecated`: решение устарело
- `final` → `superseded`: создана новая версия (DEC-XXX_v2.md)

## Процесс создания решения

### Шаг 1: Предпосылки

**Требования:**
- ✅ Завершена [📖 дискуссия](../glossary.md#дискуссия) (статус `approved` или `final`)
- ✅ Создан архитектурный документ (статус `approved` или `final`)

**Не создавать Decision (ADR) если:**
- ❌ Дискуссия ещё в статусе `draft`
- ❌ Нет архитектурного обоснования
- ❌ Решение тривиально (выбор библиотеки, naming convention)

### Шаг 2: Создание документа

1. **Прочитать шаблон:**
   ```
   llm_instructions/templates/decision_adr.md
   ```

2. **Прочитать индекс для определения ID:**
   ```
   general_docs/04_decisions/000_decisions.md
   ```

3. **Создать файл:**
   - Имя: `DEC-XXX_название.md`
   - Скопировать шаблон
   - Статус: `draft`

4. **Заполнить разделы:**
   - Решение (что решили)
   - Контекст (проблема, требования)
   - Обоснование (почему)
   - Альтернативы (что рассмотрели и отклонили)
   - Последствия (что влечёт)

5. **Добавить ссылки:**
   - На связанную дискуссию
   - На архитектурный документ
   - На ресурсы (если есть)
   - На планы реализации (если есть)

6. **Обновить индекс:**
   - Добавить строку в таблицу `000_decisions.md`
   - Обновить статистику

### Шаг 3: Согласование

1. Изменить статус на `proposed`
2. Обсудить с командой
3. При одобрении → `approved`
4. При реализации → `final`

## Связи с другими документами

### Входящие ссылки (откуда):
- **Дискуссия** → Decision (ADR): "На основе этой дискуссии принято решение DEC-XXX"
- **Архитектура** → Decision (ADR): "Архитектурное обоснование в A-XXX"

### Исходящие ссылки (куда):
- Decision (ADR) → **Ресурсы**: "Для реализации нужны ресурсы R-XXX, R-YYY"
- Decision (ADR) → **План**: "План реализации P-XXX"

## Архив устаревших решений

Папка `archive/` хранит:
- Устаревшие решения (статус `deprecated`)
- Отклонённые варианты
- Замещённые версии (статус `superseded`)

### Когда переносить в архив:

1. **Решение устарело:**
   - Изменить статус на `deprecated`
   - Переместить файл в `archive/`
   - Указать причину устаревания в документе
   - Обновить индекс

2. **Создана новая версия:**
   - Создать `DEC-XXX_название_v2.md` (новая версия)
   - Переименовать старую: `DEC-XXX_название_v1.md`
   - Переместить v1 в `archive/`
   - В v2 указать: "Заменяет DEC-XXX_v1"
   - В v1 указать: "Заменено на DEC-XXX_v2"
   - Обновить индекс

## Примеры

### ✅ Хороший Decision (ADR)

```markdown
# DEC-001: Использование JWT для аутентификации

**Статус:** final
**Дата:** 2026-01-15

## Решение
Использовать JWT токены + refresh token для аутентификации пользователей.

## Контекст
Нужна stateless аутентификация для микросервисной архитектуры.
Требования: масштабируемость, простота интеграции, поддержка OAuth.

## Обоснование
- Stateless — легко масштабируется
- Стандартное решение в микросервисах
- Поддержка OAuth 2.0 из коробки

## Рассмотренные альтернативы
1. **Session-based** (отклонено): проблемы с масштабированием, нужен shared storage
2. **OAuth только** (отклонено): избыточно для внутренней системы

## Последствия
- Нужен refresh token mechanism (R-001)
- Политика истечения токенов (15 мин access, 7 дней refresh)
- Усложняется logout (blacklist токенов) (P-001)
```

### ❌ Плохой Decision (ADR)

```markdown
# DEC-001: Аутентификация

Решили использовать JWT.

Причины:
- Это хорошо
- Все так делают
```

**Проблемы:**
- ❌ Нет контекста (какая проблема?)
- ❌ Нет альтернатив (что ещё рассматривали?)
- ❌ Нет последствий (что это влечёт?)
- ❌ Поверхностное обоснование

---

## Частые ошибки

1. **Слишком детально:**
   - ❌ Описание реализации (это для ресурсов/планов)
   - ✅ Краткое решение + обоснование

2. **Нет альтернатив:**
   - ❌ "Выбрали PostgreSQL" (а что ещё рассматривали?)
   - ✅ "Выбрали PostgreSQL, рассматривали MySQL (отклонено из-за...) и MongoDB (отклонено из-за...)"

3. **Субъективное обоснование:**
   - ❌ "JWT — это круто"
   - ✅ "JWT обеспечивает stateless аутентификацию, что критично для масштабирования микросервисов"

4. **Изменение решения после статуса `final`:**
   - ❌ Менять существующий документ
   - ✅ Создать новую версию, старую в archive/

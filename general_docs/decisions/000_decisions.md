## Быстрый поиск

### По статусу
- **draft** (0) — черновики решений
- **proposed** (0) — предложенные решения (ждут обсуждения)
- **approved** (0) — одобренные решения (готовы к реализации)
- **final** (0) — финализированные и используемые решения
- **deprecated** (0) — устаревшие решения (см. archive/)
- **superseded** (0) — заменённые новыми версиями (см. archive/)

### По дате
- **Последние обновления:** Нет решений
- **Недавно созданные:** Нет решений

## Правила обновления индекса

**КРИТИЧЕСКИ ВАЖНО для LLM:**

При создании/изменении/удалении архитектурных решений:

1. **После создания нового решения:**
   - Добавить строку в таблицу "Индекс решений"
   - Обновить статистику (увеличить счетчик для соответствующего статуса)
   - Обновить дату последнего обновления

2. **При изменении статуса:**
   - Обновить статус в таблице
   - Пересчитать статистику (уменьшить старый статус, увеличить новый)
   - Обновить дату последнего обновления

3. **При устаревании решения:**
   - Изменить статус на `deprecated` или `superseded`
   - Переместить файл в `archive/`
   - Обновить статистику
   - В новом решении указать ссылку на старое

**Автоматизация:** Использовать `/doc-index decisions` для автоматического обновления этого файла.

## Автоматическое обновление

Этот файл автоматически обновляется скиллом `/doc-index`.

**Команды:**
```bash
# Обновить индекс решений
/doc-index decisions

# Проверить актуальность индекса
/doc-index decisions --check

# Обновить все индексы в general_docs
/doc-index
```

**Проверка здоровья:**
```bash
# Проверить документацию и индексы
make docs-health

# Проверить только индексы
make docs-check-indexes
```

## Шаблон для нового архитектурного решения

**Имя файла:** `DEC-[XXX]_краткое_описание.md`

**Пример:** `DEC-001_auth_jwt.md`

**Шаблон:** См. [templates/decision_adr.md](../../llm_instructions/templates/decision_adr.md)

**Workflow:**
1. Создать файл на основе одобренной дискуссии и архитектурного документа
2. Статус `draft` — планирование решения
3. Добавить в этот индекс
4. Заполнить все обязательные разделы (Решение, Обоснование, Альтернативы, Последствия)
5. Следовать [📖 workflow статусов](../glossary.md#workflow-статусов)
6. При переходе в `final` → создать ресурсы и план реализации

## Назначение

Фиксация обсуждений "пользователь ↔ LLM" о функциональности проекта.

**[📖 Дискуссия](../glossary.md#дискуссия)** — это структурированный документ, который:
- Сохраняет исходный запрос пользователя
- Фиксирует предложения LLM по реализации
- Документирует принятые решения с аргументацией
- Служит основой для создания архитектурных документов

## Правила создания

### Именование файлов
- **Формат:** `[ID]_краткое_описание.md`
- **Примеры:**
  - `001_user_auth_and_roles.md`
  - `002_data_export_feature.md`
  - `003_notification_system.md`
- **ID:** Последовательная нумерация (001, 002, 003...)
- **Уникальность:** ID должен быть уникальным в рамках папки 01_discuss/

### Обязательные разделы

1. **Метаданные** (в начале файла)
   - Статус документа
   - Дата создания
   - Дата последнего обновления
   - Автор (пользователь/LLM)

2. **Исходный запрос пользователя**
   - Дословная фиксация запроса
   - Контекст обсуждения

3. **Анализ цели и контекст**
   - Какую проблему/возможность обсуждаем?
   - Почему это важно?
   - Какие есть ограничения?

4. **Предложения LLM/архитектора**
   - Варианты решений
   - Плюсы и минусы каждого варианта
   - Таблица со статусами ([ПРИНЯТ], [ОТКЛОНЕН])
   - Для отклонённых — краткая причина отказа

5. **Дискуссия и анализ**
   - Критика вариантов
   - Вопросы и уточнения
   - Дополнительные соображения

6. **Принятое решение**
   - Чёткая формулировка итогового выбора
   - Аргументация выбора

7. **Принятые архитектурные решения**
   - Список действий
   - Ссылки на создаваемые файлы в `02_architecture/`

8. **Добавления к глоссарию**
   - Новые термины, введённые в дискуссии
   - Их определения

### Метаданные
- **Статус:** Обязательно указывать статус документа (см. раздел "Workflow статусов")
- **Дата создания:** Дата начала дискуссии
- **Дата обновления:** Обновлять при каждом изменении
- **Связанные документы:** Ссылки на архитектуру, планы, другие дискуссии

## Workflow статусов

```
draft → in_progress → review → approved → final
           ↓              ↑
        feedback ─────────┘
```

**Возможные откаты:**
- `final → feedback` — при уточнении документации из архитектуры
- `review → in_progress` — при необходимости доработки

**Подробности:** См. [📖 Workflow статусов](../glossary.md#workflow-статусов) в глоссарии

### Описание статусов

- **draft:** Фиксация идеи, начальная формулировка запроса
- **in_progress:** Активное обсуждение, анализ вариантов
- **feedback:** Обратная связь от архитектуры, уточнение дискуссии
- **review:** Фиксация решения, проверка полноты
- **approved:** Решение одобрено, готово к созданию архитектуры
- **final:** Дискуссия завершена, архитектура создана

## Связь с другими документами

### Исходящие связи
- **Дискуссия → Архитектура** (обязательно при статусе `approved`)
  - После принятия решения создаются архитектурные документы
  - В дискуссии указываются ссылки на создаваемые файлы архитектуры

### Входящие связи
- **Архитектура → Дискуссия** (обратная связь)
  - При изменении архитектуры обновляется связанная дискуссия
  - Статус дискуссии меняется на `feedback`
  - Добавляется запись о том, что и почему изменилось

**[📖 Цепочка зависимостей](../glossary.md#цепочка-зависимостей):**
```
[Дискуссия] → Архитектура → Ресурсы → План реализации → Документация папок
```

**Позиция в цепочке:** Начало (источник требований и решений)

## Правила обратной связи ([📖 Feedback](../glossary.md#feedback))

**При изменении связанной архитектуры:**

1. **Обновить дискуссию:**
   - Изменить статус на `feedback`
   - Добавить секцию "История обратной связи"
   - Зафиксировать, какие части изменены и почему

2. **Дополнить документ:**
   - Указать дату обновления
   - Описать причины изменений
   - Обновить ссылки на архитектуру

3. **Чек-лист проверки изменений:**
   - [ ] Обновлен индекс ([000_discuss.md](000_discuss.md))
   - [ ] Проверены и обновлены все ссылки на архитектуру
   - [ ] Обновлен статус документа
   - [ ] Добавлена история изменений
   - [ ] Обновлена дата последнего обновления

## Автоматизация

### Обновление индекса
```bash
# Автоматическое обновление индекса дискуссий
/doc-index discuss

# Проверка актуальности индекса
/doc-index discuss --check
```

### Проверка документации
```bash
# Полная проверка здоровья документации
make docs-health

# Проверка только дискуссий
make docs-check-discuss
```

### Создание новой дискуссии

**Рекомендуемый процесс:**

1. **Начало дискуссии:**
   - Пользователь задаёт вопрос или описывает требование
   - LLM создаёт файл дискуссии со статусом `draft`
   - Обновляется индекс [000_discuss.md](000_discuss.md)

2. **Обсуждение:**
   - LLM предлагает варианты решений
   - Статус меняется на `in_progress`
   - Заполняются секции анализа и предложений

3. **Принятие решения:**
   - Выбирается оптимальный вариант
   - Статус меняется на `review`
   - Фиксируется решение с аргументацией

4. **Одобрение:**
   - Решение проверяется на полноту
   - Статус меняется на `approved`
   - Планируется создание архитектуры

5. **Завершение:**
   - Создаётся архитектурный документ
   - Статус дискуссии меняется на `final`
   - Добавляются ссылки на архитектуру

## Частые ошибки

**1. Неполная фиксация контекста**
- ❌ "Нужна аутентификация"
- ✅ "Требуется реализовать аутентификацию пользователей с поддержкой JWT токенов, OAuth2, восстановления пароля и двухфакторной авторизации"

**2. Отсутствие анализа вариантов**
- ❌ "Используем PostgreSQL"
- ✅ "Рассмотрены варианты: PostgreSQL (плюсы: ACID, JSON, плюсы), MongoDB (плюсы: гибкая схема, минусы: нет ACID). Выбран PostgreSQL, т.к. приоритет — консистентность данных"

**3. Нет связи с архитектурой**
- ❌ Дискуссия завершена, но архитектура не создана
- ✅ Явные ссылки на создаваемые файлы архитектуры

**4. Устаревшие дискуссии не обновляются**
- ❌ Архитектура изменилась, но дискуссия не обновлена
- ✅ При изменении архитектуры запускается процесс feedback

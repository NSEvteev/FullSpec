## Исходный запрос пользователя

> Любому проекту нужна страница входа (UI). Мы сейчас не обсуждаем хранение пользователей и методы АПИ для аутентификации. Давай обсудим UI и что нам нужно, чтобы у нас был UI стайлгайд, выбраны стили и т.д.

## Анализ цели и контекст

**Проблема:** Проекту нужна визуальная система — единый стайлгайд для UI-компонентов, начиная со страницы входа.

**Scope дискуссии:**
- UI-фреймворк / библиотека компонентов
- Дизайн-система (цвета, типографика, spacing)
- Подход к стилям (CSS-in-JS, Tailwind, SCSS и т.д.)
- Страница входа как первый UI-компонент

**Вне scope:**
- Backend аутентификации
- API endpoints
- Хранение пользователей

## Детальный анализ вариантов

### Вариант 1: Tailwind CSS + Headless UI

**Что это:**
- Tailwind CSS — utility-first CSS фреймворк
- Headless UI — unstyled, accessible компоненты (dropdown, dialog, tabs)

**Стек:**
```
Tailwind CSS + Headless UI + custom components
```

**Плюсы:**
- Максимальная гибкость в дизайне
- Маленький bundle (только используемые стили)
- Отличная документация
- Большое community

**Минусы:**
- Нет готовых стилизованных компонентов
- Нужно писать все стили самим
- Verbose классы в JSX

---

### Вариант 2: shadcn/ui

**Что это:**
- Коллекция переиспользуемых компонентов на базе Radix UI + Tailwind
- Компоненты копируются в проект (не npm-зависимость)
- Полная кастомизация

**Стек:**
```
shadcn/ui (Radix + Tailwind) → копируется в packages/ui/
```

**Плюсы:**
- Готовые красивые компоненты
- Полный контроль над кодом
- Accessible (Radix)
- Легко кастомизировать

**Минусы:**
- Только React
- Нужно поддерживать скопированный код

---

### Вариант 3: MUI (Material UI)

**Что это:**
- Полноценная библиотека компонентов
- Material Design из коробки

**Плюсы:**
- Всё из коробки
- Хорошая документация
- Theming system

**Минусы:**
- Тяжёлый (большой bundle)
- Material Design не всем нравится
- Сложно уйти от Material-стиля

---

### Вариант 4: Ant Design

**Что это:**
- Enterprise UI библиотека от Alibaba
- Много компонентов для бизнес-приложений

**Плюсы:**
- Очень много компонентов
- Хорошие формы, таблицы
- Enterprise-ready

**Минусы:**
- Специфичный "китайский" стиль
- Большой bundle
- Сложная кастомизация

---

### Вариант 5: Собственная дизайн-система

**Что это:**
- Полностью свои компоненты с нуля
- Свой дизайн-язык

**Плюсы:**
- Уникальный стиль
- Полный контроль

**Минусы:**
- Очень долго
- Нужен дизайнер
- Нужно писать accessibility

---

## Рекомендация

**Предварительно рекомендую: Вариант 2 (shadcn/ui)**

**Причины:**
1. Баланс между готовыми компонентами и гибкостью
2. Компоненты становятся частью проекта — полный контроль
3. Tailwind под капотом — легко кастомизировать
4. Хорошая accessibility из Radix
5. Активно развивается, популярен в 2025-2026

## Назначение

Планирование архитектуры на основе одобренных дискуссий.

**Архитектурный документ** — это детальное описание компонента системы, которое:
- Определяет цель и контекст компонента
- Описывает структуру, интерфейсы и взаимодействия
- Планирует необходимые ресурсы (БД, API, сервисы)
- Служит основой для создания ресурсов и планов реализации

## Правила создания

### Именование файлов
- **Формат:** `[ID]_название_компонента.md`
- **Примеры:**
  - `001_system_overview.md`
  - `002_auth_module.md`
  - `003_data_pipeline.md`
- **ID:** Последовательная нумерация (001, 002, 003...)
- **Уникальность:** ID должен быть уникальным в рамках папки 02_architecture/

### Обязательные разделы

1. **Метаданные** (в начале файла)
   - Статус документа
   - Дата создания
   - Дата последнего обновления
   - Версия (следовать правилам версионирования)

2. **Цель и контекст компонента**
   - Зачем нужен этот компонент?
   - Какую проблему решает?
   - Место в общей системе

3. **Ссылки на связанные документы**
   - Обязательные ссылки на ВСЕ связанные дискуссии
   - Ссылки на ресурсы
   - Ссылки на планы реализации

4. **Оглавление документа**
   - Ссылки на пункты документа

5. **Принципы и ограничения**
   - Архитектурные принципы
   - Технические ограничения
   - Требования compliance

6. **Диаграммы**
   - C4-уровни (Context, Container, Component)
   - UML-последовательности
   - Ссылки на файлы в 03_diagrams/

7. **Интерфейсы и контракты**
   - API (OpenAPI/Swagger)
   - Схемы сообщений
   - Форматы событий

8. **Данные**
   - Модели данных
   - Стратегии доступа
   - Миграции

9. **Внешние зависимости**
   - Сервисы
   - Библиотеки
   - API

10. **Требования к качеству**
    - Доступность
    - Производительность
    - Безопасность

11. **Риски и компромиссы**

12. **Выделение состава ресурсов**
    - Перечень необходимых ресурсов
    - Классификация по типам

13. **История изменений** (Changelog)
    - Таблица версий с описанием изменений

### Метаданные
- **Статус:** Обязательно указывать статус документа (см. раздел "Workflow статусов")
- **Дата создания:** Дата начала работы над архитектурой
- **Дата обновления:** Обновлять при каждом изменении
- **Версия:** Формат MAJOR.MINOR (1.0, 1.1, 2.0)
- **Связанные документы:** Ссылки на дискуссии, ресурсы, планы

## Workflow статусов

```
draft → in_progress → review → approved → final
           ↓              ↑
        feedback ─────────┘
```

**Возможные откаты:**
- `final → feedback` — при уточнении из ресурсов
- `review → in_progress` — при необходимости доработки

**Подробности:** См. [📖 Workflow статусов](../glossary.md#workflow-статусов) в глоссарии

### Описание статусов

- **draft:** Планирование работы над архитектурой
- **in_progress:** Активная работа над архитектурой
- **feedback:** Обратная связь от ресурсов, уточнение архитектуры
- **review:** Ревью архитектуры, проверка полноты
- **approved:** Одобрено, готово к созданию ресурсов и плана
- **final:** Архитектура завершена, ресурсы и план созданы

## Связь с другими документами

### Входящие связи
- **Дискуссия → Архитектура** (обязательно)
  - Архитектура создается на основе одобренных дискуссий
  - Должны быть ссылки на ВСЕ связанные дискуссии

### Исходящие связи
- **Архитектура → Ресурсы** (обязательно при статусе `approved`)
  - После одобрения архитектуры создаются описания ресурсов
  - В архитектуре указываются ссылки на создаваемые ресурсы

- **Архитектура → План реализации** (обязательно при статусе `approved`)
  - На основе архитектуры создается план реализации
  - В архитектуре указывается ссылка на план

- **Архитектура → Диаграммы** (опционально)
  - Если требуются диаграммы .drawio, они создаются в 03_diagrams/
  - В архитектуре указываются ссылки на диаграммы

**[📖 Цепочка зависимостей](../glossary.md#цепочка-зависимостей):**
```
Дискуссия → [Архитектура] → Ресурсы → План реализации → Документация папок
```

**Позиция в цепочке:** Второй этап (после дискуссий)

## Правила обратной связи ([📖 Feedback](../glossary.md#feedback))

**При создании/изменении ресурсов на основе этой архитектуры:**

1. **Обновить архитектуру:**
   - Добавить ссылки на новые ресурсы
   - Обновить дату последнего изменения
   - При необходимости изменить статус на `feedback`

2. **Запустить Feedback для дискуссий:**
   - Обновить все связанные дискуссии
   - Добавить записи о том, что и почему изменилось

3. **Чек-лист проверки изменений:**
   - [ ] Обновлен индекс ([000_architecture.md](000_architecture.md))
   - [ ] Проверены и обновлены все ссылки на дискуссии, ресурсы, планы
   - [ ] Обновлена история изменений (Changelog)
   - [ ] Запущен процесс Feedback для связанных дискуссий
   - [ ] Обновлен статус документа

## Автоматизация

### Обновление индекса
```bash
# Автоматическое обновление индекса архитектуры
/doc-index architecture

# Проверка актуальности индекса
/doc-index architecture --check
```

### Проверка документации
```bash
# Полная проверка здоровья документации
make docs-health

# Проверка только архитектуры
make docs-check-architecture
```

### Создание нового архитектурного документа

**Рекомендуемый процесс:**

1. **На основе одобренной дискуссии:**
   - Дискуссия должна иметь статус `approved`
   - Создать файл архитектуры со статусом `draft`
   - Обновить индекс [000_architecture.md](000_architecture.md)

2. **Планирование:**
   - Определить компоненты и их взаимодействия
   - Спланировать необходимые ресурсы
   - Статус меняется на `in_progress`

3. **Заполнение:**
   - Заполнить все обязательные секции
   - Создать диаграммы (если нужно)
   - Добавить ссылки на дискуссии

4. **Ревью:**
   - Проверить полноту описания
   - Статус меняется на `review`

5. **Одобрение:**
   - Архитектура одобрена
   - Статус меняется на `approved`
   - Создать описания ресурсов
   - Создать план реализации

6. **Завершение:**
   - Все ресурсы и план созданы
   - Статус меняется на `final`
   - Обновить дискуссии (feedback)

## Частые ошибки

**1. Отсутствие ссылок на дискуссии**
- ❌ Архитектура без ссылок на дискуссии
- ✅ Явные ссылки на ВСЕ связанные дискуссии

**2. Неполное описание ресурсов**
- ❌ "Нужна база данных"
- ✅ "Требуется PostgreSQL 14+ с таблицами: users, sessions, permissions. Индексы на email, session_id. Размер: ~10M записей."

**3. Нет истории изменений**
- ❌ Архитектура изменилась, но changelog не обновлен
- ✅ Каждое изменение фиксируется в таблице Changelog

**4. Архитектура не синхронизирована с ресурсами**
- ❌ Ресурсы изменились, но архитектура не обновлена
- ✅ При изменении ресурсов запускается процесс feedback для архитектуры

# Индекс архитектурных документов

## О файлах в этой папке

**Этот файл (000_architecture.md) — ЧТО:**
- **Динамический индекс** всех архитектурных документов в папке
- Содержит таблицу с метаданными всех документов (ID, название, статус, даты)
- **Для LLM:** Быстрый обзор всей архитектуры и её состояния

**Файл [000_SUMMARY.md](000_SUMMARY.md) — ЗАЧЕМ:**
- **Агрегация решений** для передачи контекста на следующий этап (ADR)
- Содержит краткое описание всех одобренных архитектурных документов по областям
- **Для LLM:** Контекст для создания архитектурных решений (ADR)
- Обновляется скиллом `/summary-arch` при переходе архитектуры в `approved`

**Файл README.md — КАК:**
- **Статичные правила** работы с архитектурными документами
- Описывает, как создавать новую архитектуру
- Объясняет workflow статусов
- Содержит требования к структуре документов
- **Для LLM:** Инструкции по созданию и поддержке архитектуры

## Правила обновления индекса

При создании/изменении/удалении архитектурных документов:

1. **После создания** — добавить строку в таблицу
2. **При изменении статуса** — обновить статус в таблице
3. **При удалении** — удалить строку из таблицы

## Автоматическое обновление

Этот файл автоматически обновляется скиллом `/doc-index`.

**Команды:**
```bash
# Обновить индекс архитектуры
/doc-index architecture

# Проверить актуальность индекса
/doc-index architecture --check

# Обновить все индексы в general_docs
/doc-index
```

**Проверка здоровья:**
```bash
# Проверить документацию и индексы
make docs-health

# Проверить только индексы
make docs-check-indexes
```

## Шаблон для нового архитектурного документа

**Имя файла:** `[ID]_краткое_описание.md`

**Пример:** `001_system_overview.md`

**Шаблон:** См. [templates/architecture.md](../../llm_instructions/templates/architecture.md)

**Workflow:**
1. Создать файл на основе одобренной дискуссии
2. Статус `draft` — планирование архитектуры
3. Добавить в этот индекс
4. Заполнить все обязательные разделы
5. Следовать [📖 workflow статусов](../glossary.md#workflow-статусов)
6. При переходе в `approved` → создать ресурсы и план реализации

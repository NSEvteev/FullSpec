## Назначение

Хранение диаграмм для иллюстрации архитектурных документов.

**Диаграмма** — это визуальное представление архитектуры, которое:
- Иллюстрирует структуру компонентов
- Показывает взаимодействия между элементами
- Облегчает понимание сложных систем
- Дополняет текстовые описания в архитектуре

---

## Структура хранения

### Организация по архитектурным документам

Каждый архитектурный документ может иметь свою папку с диаграммами:

```
03_diagrams/
├── [ID]_[название_архитектуры]/
│   ├── [ID]-[описание].drawio
│   ├── [ID]-[описание].drawio
│   └── ...
│
└── [ID]_[название_архитектуры]/
    └── ...
```

**Пример:**
```
03_diagrams/
├── 001_system_overview/
│   ├── 001-context.drawio       # C4 Context диаграмма
│   ├── 001-container.drawio     # C4 Container диаграмма
│   └── 001-component.drawio     # C4 Component диаграмма
│
└── 002_auth_module/
    ├── 002-auth-flow.drawio     # Диаграмма потока аутентификации
    └── 002-data-model.drawio    # Модель данных
```

### Правила именования

**Папки:**
- Формат: `[ID]_[название_архитектурного_документа]/`
- ID должен совпадать с ID в `02_architecture/`
- Пример: `001_system_overview/`

**Файлы диаграмм:**
- Формат: `[ID]-[краткое_описание].drawio`
- ID совпадает с ID папки/архитектуры
- Пример: `001-context.drawio`, `002-auth-flow.drawio`

## Типы диаграмм

### 1. C4 диаграммы (рекомендуется)

**C4 Context (Контекст):**
- Показывает систему в контексте внешнего мира
- Пользователи и внешние системы
- Высокоуровневое представление

**C4 Container (Контейнер):**
- Показывает основные технические блоки
- Приложения, сервисы, БД
- Технологический стек

**C4 Component (Компонент):**
- Детализация внутри контейнера
- Компоненты и их взаимодействия
- Архитектурные паттерны

### 2. UML диаграммы

**Sequence (Последовательность):**
- Взаимодействие компонентов во времени
- API вызовы
- Потоки данных

**Class (Классы):**
- Структура данных
- Модели и их связи

### 3. Прочие диаграммы

**Data Flow:**
- Потоки данных в системе
- ETL процессы

**Deployment:**
- Инфраструктура и развертывание
- Сетевая топология

## Требования к диаграммам

### Обязательные элементы

1. **Легенда:**
   - Объяснение цветов, форм, стрелок
   - Если используются нестандартные обозначения

2. **Подписи:**
   - Все элементы должны быть подписаны
   - Понятные, описательные названия

3. **Самодостаточность:**
   - Диаграмма должна быть понятна без дополнительного контекста
   - Ключевая информация должна быть на диаграмме

### Рекомендации

1. **Простота:**
   - Не перегружать диаграмму
   - Одна диаграмма — одна идея
   - При необходимости создать несколько диаграмм

2. **Консистентность:**
   - Использовать единый стиль в проекте
   - Одинаковые обозначения для одних и тех же элементов

3. **Актуальность:**
   - Обновлять диаграммы при изменении архитектуры
   - Не допускать расхождений с кодом

## Workflow статусов

```
in_progress → final
      ↓         ↑
      └─────────┘
```

**Статусы:**
- **in_progress:** Работа над диаграммой
- **final:** Диаграмма завершена и добавлена в архитектуру

**Возможные откаты:**
- `final → in_progress` — при уточнении архитектуры

## Инструменты

### Рекомендуемые

**Draw.io (.drawio):**
- Бесплатный, open-source
- Работает в браузере и десктопе
- Интеграция с VS Code
- Поддержка C4, UML, и других нотаций

### Альтернативы

**Mermaid:**
- Текстовый формат (в Markdown)
- Для простых диаграмм
- Встроенная поддержка в GitHub, GitLab

**PlantUML:**
- Текстовый формат
- Хорошо для UML диаграмм

## Связь с другими документами

### Связь с архитектурой

**Архитектура ← Диаграммы:**
- Диаграммы создаются для иллюстрации архитектуры
- В архитектурном документе должны быть ссылки на диаграммы
- При изменении архитектуры — обновить диаграммы

**Пример ссылки в архитектуре:**
```markdown
## Диаграммы

### Контекстная диаграмма
![Context Diagram](../03_diagrams/001_system_overview/001-context.drawio)

См. [03_diagrams/001_system_overview/001-context.drawio](../03_diagrams/001_system_overview/001-context.drawio)
```

---

## Правила обратной связи

**При изменении архитектуры:**

1. **Проверить диаграммы:**
   - Соответствуют ли диаграммы новой архитектуре?
   - Нужно ли обновить существующие диаграммы?
   - Нужны ли новые диаграммы?

2. **Обновить диаграммы:**
   - Изменить статус на `in_progress`
   - Внести изменения
   - Вернуть статус в `final`
   - Обновить индекс [000_diagrams.md](000_diagrams.md)

3. **Чек-лист:**
   - [ ] Диаграммы соответствуют архитектуре
   - [ ] Все ссылки в архитектуре рабочие
   - [ ] Обновлен индекс диаграмм
   - [ ] Статус корректный

### Создание новой диаграммы

**Рекомендуемый процесс:**

1. **Определить архитектурный документ:**
   - Найти ID в `02_architecture/`
   - Проверить, есть ли папка `03_diagrams/[ID]_название/`

2. **Создать папку (если нет):**
   ```bash
   mkdir 03_diagrams/001_system_overview
   ```

3. **Создать диаграмму:**
   - Открыть Draw.io
   - Создать диаграмму
   - Сохранить как `03_diagrams/001_system_overview/001-context.drawio`

4. **Добавить метаданные:**
   - Статус: `in_progress`
   - Тип: C4 Context, UML Sequence и т.д.

5. **Обновить индекс:**
   - Добавить запись в [000_diagrams.md](000_diagrams.md)
   - Обновить статистику

6. **Добавить ссылку в архитектуру:**
   - Открыть `02_architecture/001_system_overview.md`
   - Добавить секцию "Диаграммы"
   - Добавить ссылку на диаграмму

7. **Завершить:**
   - Статус диаграммы: `final`
   - Обновить индекс

## Частые ошибки

**1. Неправильное именование**
- ❌ `diagram1.drawio`, `my_diagram.drawio`
- ✅ `001-context.drawio`, `002-auth-flow.drawio`

**2. Отсутствие связи с архитектурой**
- ❌ Диаграмма создана, но нет ссылки в архитектурном документе
- ✅ В архитектуре есть секция "Диаграммы" со ссылками

**3. Перегруженные диаграммы**
- ❌ Все компоненты системы на одной диаграмме
- ✅ Несколько диаграмм с разным уровнем детализации

**4. Устаревшие диаграммы**
- ❌ Архитектура изменилась, но диаграммы не обновлены
- ✅ При изменении архитектуры обновляются связанные диаграммы

**Фреймворк spec-driven разработки для Claude Code.**
71 скилл · 23 AI-агента · 80+ скриптов валидации · полный цикл от идеи до релиза

---

Большинство AI-воркфлоу выглядят так: описал фичу в чате, получил код, вставил в проект, надеешься что работает. Контекст теряется между сессиями. Архитектурные решения живут в чьей-то голове. Тесты пишутся после — или никогда. Документация устаревает в момент написания.

**FullSpec переворачивает этот подход.** Ты определяешь *что* строишь и *зачем* до написания первой строки кода. Спецификации становятся источником истины. Код пишется по спекам, тестируется по спекам, ревьюится по спекам. Когда реальность конфликтует с планом, спеки обновляются автоматически.

Результат: каждое решение прослеживается от требования до задеплоенного кода. Документация остаётся актуальной. Ничего не теряется.

> Внутренняя документация проекта (инструкции, стандарты, спецификации) написана на русском языке. Claude Code понимает русский нативно — форкните, настройте со своим Claude, и всё работает как есть.

## Как это работает: процесс /chain

Одна команда — `/chain` — оркестрирует весь цикл разработки. 8 фаз, от сбора требований до production-релиза:

```
/chain "Добавить аутентификацию с OAuth2"
```

### Фаза 1 — Аналитическая цепочка (4 документа, каждый валидируется перед следующим)

```
  Discussion      Что мы строим и зачем?
                  → Требования (REQ-N), критерии успеха, scope, риски
                  → Результат: specs/analysis/0001-auth-oauth2/discussion.md

  Design          Как изменится система?
                  → Сервисы (SVC-N) с API-контрактами, моделями данных, безопасностью
                  → Межсервисные интеграции (INT-N), системные тесты (STS-N)
                  → Результат: specs/analysis/0001-auth-oauth2/design.md

  Plan Tests      Как проверяем что работает?
                  → Acceptance-сценарии (TC-N) пишутся ДО кода
                  → Тестовые данные, матрица покрытия
                  → Результат: specs/analysis/0001-auth-oauth2/plan-test.md

  Plan Dev        Какие задачи, в каком порядке?
                  → Задачи (TASK-N) с зависимостями, блоки выполнения (BLOCK-N)
                  → Подзадачи, кросс-сервисная координация
                  → Результат: specs/analysis/0001-auth-oauth2/plan-dev.md
```

### Фаза 2 — Docs Sync (параллельные агенты)

```
  service-agent × N      Per-service документация генерируется (10 секций каждая)
  technology-agent × M   Per-tech стандарты кодирования создаются/обновляются
  system-agent           System overview, конвенции, документация инфраструктуры обновлены
```

### Фаза 3 — Запуск

```
  GitHub Issues созданы из TASK-N (с метками, описаниями, ссылками на спеки)
  Milestone создан для релиза
  Feature-ветка создана: 0001-auth-oauth2
  Статус всей цепочки: RUNNING
```

### Фаза 4 — Реализация

```
  dev-agent работает по задачам блок за блоком (волны BLOCK-N)
  Каждая задача: код → тесты → lint → коммит (Conventional Commits)
  CONFLICT-CHECK после каждого коммита — если код противоречит спекам,
  процесс приостанавливается и спеки обновляются автоматически (каскад)
```

### Фаза 5 — Финальная валидация

```
  /docker-up    Запуск dev-окружения, healthcheck всех сервисов
  /test         Sync main → unit/integration/e2e тесты → lint → build
  /test-ui      Playwright smoke-тесты со скриншотами
  Результат: отчёт READY или NOT READY
```

### Фаза 6 — Доставка в main

```
  /review       Локальное code review (code-reviewer агент, 7 критериев, P1/P2/P3)
  /pr-create    PR с автосбором Issues, метками, описанием
  /merge        Squash merge, Issues закрыты, ветка удалена
```

### Фаза 7 — Завершение

```
  /chain-done   Каскад REVIEW → DONE
                Planned Changes становятся AS IS в документации
                system-agent mode=done: кросс-цепочечная валидация
```

### Фаза 8 — Релиз

```
  /release-create   Тег, GitHub Release, CHANGELOG, release notes
  /post-release     Валидация: артефакты релиза, статус деплоя
```

## Что происходит, когда план не выдерживает столкновения с реальностью

Большинство spec-driven инструментов считают спецификации статичными — написал, реализовал, готово. Но реальная разработка так не работает.

**FullSpec имеет CONFLICT-детекцию.** Когда dev-agent обнаруживает, что код не соответствует спеке — API endpoint требует другую форму, модель данных не подходит, требование по производительности недостижимо — система:

1. **Обнаруживает** конфликт автоматически (после каждого коммита)
2. **Приостанавливает** реализацию
3. **Определяет** какой уровень спеки затронут (Plan Dev? Design? Discussion?)
4. **Каскадно обновляет** спеки сверху вниз
5. **Возобновляет** реализацию с обновлёнными спеками

Это не обходной путь и не крайний случай — это основная часть процесса. Спеки развиваются вместе с кодом, а не вопреки ему.

### 80+ скриптов валидации и 30 pre-commit хуков

Каждый коммит валидируется автоматически:

- Консистентность структуры (README ↔ structure.json)
- Формат документов (frontmatter, секции, именование)

## Drafts: сохранение мышления

Разговоры с Claude Code исчезают после завершения сессии. Drafts — нет.

Перед реализацией сложной задачи Claude сохраняет план в `.claude/drafts/` — структурированный документ с контекстом, решениями и пошаговым Tasklist. Это значит:

- **Ничего не теряется между сессиями.** Откройте проект завтра — план на месте.
- **Любой агент может подхватить работу.** Каждая задача в Tasklist ссылается на конкретную секцию драфта.
- **Решения прослеживаемы.** Почему выбран вариант A, а не B? Это в драфте.

```
.claude/drafts/
├── 2026-03-16-auth-oauth2.md       # План с 8 задачами
├── 2026-03-10-api-performance.md   # Исследование: сравнение 3 альтернатив
└── examples/                       # Эталонные шаблоны
```

Drafts коммитятся в git. Это рабочие файлы, не финальная документация — после завершения работы можно оставить для истории или удалить.

## Структура проекта

Каждая папка содержит `.instructions/` — Claude читает их автоматически и применяет соответствующие стандарты.

```
src/           → Исходный код сервисов (backend, database, tests)
shared/        → Контракты API (OpenAPI, Protobuf, AsyncAPI), события, общие библиотеки
platform/      → Docker, Gateway (Traefik/Nginx), Kubernetes, мониторинг (Prometheus/Grafana)
config/        → Конфигурации окружений (dev / staging / prod), feature flags
tests/         → Системные тесты (e2e, integration, load, smoke)
specs/         → Спецификации: аналитические цепочки + живая документация
  analysis/    →   Discussion → Design → Plan Tests → Plan Dev per feature
  docs/        →   Per-service документация, system overview, per-tech стандарты
.claude/       → 71 скилл, 23 агента, 16 контекстных правил, drafts
.github/       → CI/CD workflows, шаблоны Issues, шаблон PR, метки
.instructions/ → Мета-стандарты: как писать инструкции, скрипты, скиллы
```

Полное дерево: [.structure/README.md](.structure/README.md)

## Как FullSpec соотносится с альтернативами

| | FullSpec | [Spec Kit](https://github.com/github/spec-kit) (GitHub) | [Kiro](https://kiro.dev/) (AWS) | [OpenSpec](https://github.com/Fission-AI/OpenSpec) |
|---|---|---|---|---|
| Планирование тестов до кода | Да (фаза Plan Tests) | Нет | Нет | Нет |
| Обнаружение конфликтов (код ↔ спеки) | Автоматическое + каскадное обновление | Нет | Нет | Нет |
| Параллельные AI-агенты | 23 агента | 0 | 1 | 0 |
| Живая документация | Автосинхронизация из спек | Нет | Нет | Нет |
| GitHub-автоматизация (Issues, PR, Review, Release) | Полная | Нет | Нет | Нет |
| Pre-commit валидация | 30 хуков | 0 | 0 | 0 |
| AI-агностичность | Только Claude Code | 20+ агентов | Только Claude/Bedrock | 20+ агентов |
| Цена | Бесплатно (MIT) | Бесплатно (MIT) | $0–200/мес | Бесплатно (MIT) |
| Зрелость | Новый | 77k stars | GA (enterprise) | 31k stars |

FullSpec — самый полный, но и самый категоричный. Если нужны AI-агностичные лёгкие спеки — Spec Kit или OpenSpec. Если нужна полноценная IDE с AWS-интеграцией — Kiro. Если нужен полный цикл, автоматизированный от начала до конца — это FullSpec.

## Сообщение об уязвимостях

Если вы обнаружили уязвимость в безопасности, пожалуйста, **НЕ создавайте публичный issue**.

### Как сообщить

1. **Email:** Отправьте подробное описание на [n.s.evteev@ya.ru](mailto:n.s.evteev@ya.ru)
2. **Тема письма:** "Проблема безопасности проекта [название репозитория]: [краткое описание]"
3. **Содержание:**
   - Описание уязвимости
   - Шаги для воспроизведения
   - Потенциальное влияние
   - Предлагаемое решение (если есть)

### Что ожидать

- **Подтверждение:** В течение 48 часов
- **Оценка:** В течение 7 дней
- **Исправление:** Зависит от серьезности (критические — в течение 48 часов)
- **Публичное раскрытие:** После выпуска исправления

### Ответственное раскрытие

Мы просим:
- Дать нам разумное время для исправления до публичного раскрытия
- Не использовать уязвимость в злонамеренных целях
- Не получать доступ к данным других пользователей

В благодарность за ответственное раскрытие:
- Ваше имя будет указано в CHANGELOG (если вы хотите)
- Мы публично поблагодарим вас после исправления

## Рекомендации по безопасности

### Для разработчиков

**1. Зависимости:**
```bash
# Регулярно проверяйте уязвимости
npm audit
pip-audit

# Обновляйте зависимости
npm update
pip install --upgrade -r requirements.txt
```

**2. Секреты:**
- НИКОГДА не коммитить `.env` файлы
- Использовать `.env.example` для шаблонов
- Проверять коммиты на наличие секретов перед push

**3. Аутентификация:**
- Использовать bcrypt/argon2 для хэширования паролей
- Минимум 10 раундов bcrypt
- JWT токены с коротким временем жизни (15 мин для access, 7 дней для refresh)
- Использовать httpOnly cookies для токенов

**4. Валидация:**
- Валидировать ВСЕ входные данные на бэкенде
- Использовать библиотеки валидации (Zod, Joi, Pydantic)
- Санитизировать SQL запросы (использовать ORM/prepared statements)

**5. HTTPS:**
- ВСЕГДА использовать HTTPS в продакшене
- Настроить HSTS заголовки
- Использовать secure cookies

**6. CORS:**
- Настроить строгие CORS политики
- Разрешать только доверенные origins

**7. Rate Limiting:**
- Включить rate limiting на API Gateway
- Особенно для эндпоинтов аутентификации

### Для пользователей

**1. Пароли:**
- Минимум 10 символов
- Включать буквы, цифры, спецсимволы
- Не использовать один пароль для разных сервисов

**2. Двухфакторная аутентификация:**
- Включите 2FA когда доступно

**3. Обновления:**
- Регулярно обновляйте приложение
- Следите за уведомлениями о безопасности

## Безопасность инфраструктуры

### Docker

```dockerfile
# Использовать specific версии (не latest)
FROM node:18.19-alpine

# Запускать от непривилегированного пользователя
USER node

# Сканировать образы на уязвимости
# docker scan image:tag
```

### Kubernetes

```yaml
# Security Context
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  allowPrivilegeEscalation: false
  capabilities:
    drop:
      - ALL
```

### Environment Variables

```bash
# Production
export NODE_ENV=production
export JWT_SECRET=$(openssl rand -base64 32)
export DB_PASSWORD=$(openssl rand -base64 24)

# НИКОГДА не использовать значения по умолчанию
```

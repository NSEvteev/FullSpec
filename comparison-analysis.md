## Сравнение по возможностям

### 1. Процесс от идеи до кода

| Возможность | FullSpec | Spec Kit | Kiro | OpenSpec | Без структуры |
|---|---|---|---|---|---|
| Формализация требований | Discussion (REQ-N, критерии успеха) | Constitution + Specification | Requirements (EARS-нотация) | Proposal | Нет |
| Архитектурное проектирование | Design (SVC-N с 9 подсекциями, API-контракты, Data Model) | Plan | Design (markdown) | Design | Нет |
| Планирование тестов ДО кода | Plan Tests (TC-N acceptance-сценарии) | Нет | Нет | Нет | Нет |
| Планирование задач | Plan Dev (TASK-N с зависимостями, BLOCK-N волны) | Tasks | Tasks (чеклист) | Tasks (YAML) | Нет |
| Полная цепочка одной командой | `/chain` (оркестратор) | `specify` CLI | Нет (ручной запуск каждой фазы) | `/opsx:propose` → `/opsx:apply` | Нет |

**Вывод:** FullSpec — единственный, кто включает **планирование тестов** как отдельный этап перед кодом. Spec Kit и OpenSpec более лёгкие, но и менее структурированные. Kiro имеет хорошую формализацию требований (EARS), но ограничен IDE.

### 2. Обратная связь код → спецификации (CONFLICT)

| | FullSpec | Spec Kit | Kiro | OpenSpec | Без структуры |
|---|---|---|---|---|---|
| Обнаружение конфликтов | Автоматическое (dev-agent CONFLICT-CHECK после каждого коммита) | Нет | Нет | Нет | Нет |
| Каскадное обновление спек | Да (снизу вверх: определение уровня, сверху вниз: обновление) | Нет | Нет | Известный недостаток (#705) | Нет |
| Статус-машина | 7 статусов (DRAFT→WAITING→RUNNING→CONFLICT→REVIEW→DONE + REJECTED) | Нет | Нет | Нет | Нет |

**Вывод:** Это **уникальная возможность** FullSpec. В Spec Kit, Kiro и OpenSpec спецификации статичны — если при реализации выясняется, что дизайн невозможен, разработчик должен вручную вернуться и исправить спеки. FullSpec обнаруживает это автоматически и запускает контролируемый процесс обновления.

## Честные плюсы и минусы

### FullSpec

**Плюсы:**
- **Самый полный цикл** — от идеи до production release, 8 фаз, 71 скилл, 23 агента
- **Планирование тестов ДО кода** — уникально среди всех конкурентов
- **CONFLICT-детекция** — автоматическое обнаружение расхождений код↔спеки, каскадное обновление
- **Living documentation** — docs синхронизируются из спецификаций автоматически
- **Per-tech стандарты** — 8 технологий с конвенциями, генерируемыми из Design
- **GitHub-интеграция** — полная автоматизация Issues, PR, Review, Release
- **Валидация на 3 уровнях** — pre-commit (30 хуков), CI, ручные скрипты
- **Open source (MIT)** — бесплатно, форкни и настрой
- **Структура template** — клонировал и начал, вся инфраструктура на месте
- **Параллельные агенты** — docs-sync, code-review, issue-создание ускоряются параллельной работой

**Минусы:**
- **Высокий порог входа** — 71 скилл, 23 агента, 8 фаз, 7 статусов, нужно время на изучение
- **Vendor lock-in на Claude Code** — не работает с Cursor, Copilot, другими AI-агентами
- **Overhead для мелких задач** — полная цепочка Discussion→Design→Plan Tests→Plan Dev для багфикса в 2 строки — избыточна (есть `--doc-only`, но сам процесс тяжёлый)
- **Сложность системы** — 80+ скриптов валидации, 30 pre-commit хуков создают высокую когнитивную нагрузку
- **Документация на русском** — барьер для нерусскоязычных разработчиков (хотя Claude понимает русский, чтение инструкций сложнее)
- **Новый проект** — нет community, нет production-примеров, нет звёзд GitHub
- **Жёсткий формат** — 9 подсекций SVC-N, 5 полей TASK-N, строгие frontmatter — мало гибкости
- **Требует Docker + Python + gh CLI** — зависимости для валидации, не все проекты их используют
- **Нет поддержки multi-repo** — один template = один проект
- **Нет визуального дашборда** — вся навигация через CLI и файлы

### Spec Kit (GitHub)

**Плюсы:**
- **Огромное community** — 77.4k stars, поддержка GitHub/Microsoft
- **AI-агностичность** — работает с 20+ AI-агентами (Copilot, Claude, Cursor, Gemini, Windsurf...)
- **Простота** — 6 фаз, несколько команд, низкий порог входа
- **Creative exploration** — параллельные реализации на разных стеках
- **Brownfield-совместимость** — можно применить к существующему проекту
- **Активное развитие** — 620+ коммитов, регулярные обновления
- **Хорошая документация** — блог-пост, deep dive, примеры

**Минусы:**
- **Нет обратной связи** — спеки статичны, конфликты не обнаруживаются автоматически
- **Позиционируется как эксперимент** — авторы прямо говорят "не production-ready"
- **Нет тестового планирования** — тесты не часть процесса
- **Нет CI/CD** — только генерация спецификаций, дальше вручную
- **Нет валидации результата** — документированные случаи, когда AI считает спеку выполненной, но функциональность отсутствует
- **Нет living docs** — спеки не синхронизируются с кодом
- **Нет code review** — нет встроенного ревью против спецификаций
- **Нет GitHub-интеграции** — Issues, PR, Release — всё вручную

### Kiro (AWS)

**Плюсы:**
- **Полноценная IDE** — не нужен отдельный инструмент, всё в одном окне
- **EARS-нотация** — формальный подход к требованиям (из авиа/автоиндустрии)
- **Agent Hooks** — автоматизация на событиях (файл сохранён → тесты, API изменён → docs)
- **Steering Files** — persistent контекст (product.md, tech.md, structure.md)
- **Enterprise backing** — AWS/Amazon, стабильное финансирование, GA
- **Хороший UX** — визуальный интерфейс для спецификаций, не только файлы

**Минусы:**
- **Дорого** — от $20 до $200/мес, 50 free credits расходуются за одну сессию
- **Vendor lock-in** — только Claude Sonnet через Amazon Bedrock, нельзя сменить модель
- **Проприетарный** — не open source, нет контроля над кодом
- **Статичные спеки** — не эволюционируют по мере прогресса агента
- **Жёсткий пайплайн** — Requirements→Design→Tasks "убивает momentum" при итерациях
- **UX-баги** — файлы не появляются в контексте до рестарта, деградация на длинном контексте
- **Нет тестового планирования** — тесты не встроены в спецификационный процесс
- **Нет обратной связи код→спеки** — конфликты не обнаруживаются
- **Ограниченная автоматизация** — нет code review, нет release automation

### OpenSpec (Fission AI)

**Плюсы:**
- **Лёгкий и гибкий** — 4 команды, fluid iteration, нет rigid phase gates
- **AI-агностичность** — 20+ AI-ассистентов
- **Brownfield-фокус** — сделан для существующих проектов
- **Хорошее community** — 31.1k stars, активная разработка
- **Профили** — разные настройки для разных проектов
- **MIT лицензия** — бесплатно
- **npm-установка** — простой старт

**Минусы:**
- **Нет обратной связи между артефактами** — при генерации issues из tasks не пропагируются в specs (#783)
- **Нет регенерации downstream** — изменил proposal → design и tasks не обновляются (#705)
- **Только структурная валидация** — ловит пропущенные секции, но не семантические противоречия
- **Слабая поддержка микросервисов** — multi-repo проблема (#725)
- **Требует мощную модель** — Opus 4.5 или GPT 5.2 для качественной работы
- **Нет тестового планирования** — тесты не часть процесса
- **Нет CI/CD, code review, release automation** — только генерация спецификаций
- **Телеметрия по умолчанию** — нужно явно отключать
- **Нет living docs** — спеки не синхронизируются с рабочей документацией

### Без структуры ("vibe coding")

**Плюсы:**
- **Нулевой порог входа** — открыл чат, написал промпт, получил код
- **Максимальная скорость для простых задач** — прототипы, скрипты, эксперименты
- **Нет зависимостей** — не нужны Docker, Python, CLI
- **Максимальная гибкость** — любой AI, любой язык, любой формат
- **Подходит для обучения** — идеально для изучения нового языка/фреймворка

**Минусы:**
- **Контекст теряется** — каждая сессия "с чистого листа"
- **Нет консистентности** — тот же промпт → разный результат
- **Невозможно масштабировать** — на команду, на длительный проект
- **Архитектура ad-hoc** — AI "галлюцинирует" решения без контекста
- **Нет трассируемости** — невозможно понять, почему решение было принято
- **Technical debt** — накапливается экспоненциально
- **Тесты не генерируются** — или генерируются бессистемно
- **Документация отсутствует** — или устаревает мгновенно
- **Не подходит для compliance** — нет аудиторского следа

## Когда что выбрать

| Ситуация | Лучший выбор | Почему |
|---|---|---|
| Новый microservice проект с Claude Code | **FullSpec** | Полная инфраструктура от старта, 8 фаз, living docs |
| Быстрый прототип за день | **Без структуры** | Overhead любого фреймворка не окупится |
| Проект с Cursor/Copilot/Windsurf | **Spec Kit** или **OpenSpec** | FullSpec и Kiro привязаны к своим AI |
| Enterprise с AWS-стеком | **Kiro** | Интеграция с AWS, формальные требования (EARS), платная поддержка |
| Существующий проект + нужна структура | **OpenSpec** | Brownfield-фокус, лёгкий, 4 команды |
| Большая команда, нужен контроль качества | **FullSpec** | Code review агент, 30 pre-commit хуков, CI, CONFLICT-detection |
| Один разработчик, маленький проект | **Spec Kit** или **OpenSpec** | Легче, не нужна вся инфраструктура FullSpec |
| Нужна AI-агностичность | **Spec Kit** или **OpenSpec** | 20+ AI-агентов, нет vendor lock-in |
| Нужна трассируемость решений | **FullSpec** | Единственный с полным аудиторским следом от требования до релиза |
| Хакатон / учебный проект | **Без структуры** | Скорость важнее качества |

## Уникальные черты FullSpec (чего нет у других)

1. **Plan Tests как отдельная фаза** — TC-N acceptance-сценарии определяются ДО кода. Ни один конкурент не включает тестовое планирование в спецификационный процесс.

2. **CONFLICT-детекция и каскадное обновление** — автоматическое обнаружение расхождений между кодом и спеками. Каскадное обновление снизу вверх (обнаружение) и сверху вниз (исправление). У Spec Kit, Kiro и OpenSpec спеки статичны.

3. **23 параллельных AI-агента** — docs-sync запускает N service-agent + M technology-agent параллельно. Code review — по N сервисов параллельно. Конкуренты используют один AI-агент последовательно.

4. **Living documentation** — per-service docs (10 секций), system overview, per-tech стандарты автоматически синхронизируются из спецификаций. Другие генерируют спеки, но не рабочую документацию.

5. **30 pre-commit хуков + 80+ скриптов** — валидация на каждом коммите: структура, формат, ссылки, frontmatter, security (gitleaks). Конкуренты не валидируют.

6. **Полная GitHub автоматизация** — Issues из TASK-N, Milestone, Branch, PR с автосбором, Code Review агент (7 критериев), Release с CHANGELOG. У конкурентов — только спеки.

7. **Контекстные правила** — 16 правил автоматически загружаются по типу файла (.ts → TypeScript standard, .py → FastAPI standard). Ближайший аналог — Kiro Steering Files, но с 3 файлами vs 16.

## Что FullSpec может позаимствовать у конкурентов

| У кого | Что | Зачем |
|---|---|---|
| **Spec Kit** | AI-агностичность | Снять vendor lock-in с Claude Code |
| **Spec Kit** | Creative exploration (параллельные реализации) | Помощь в выборе стека |
| **Kiro** | EARS-нотация для требований | Более формальные и однозначные требования |
| **Kiro** | Визуальный интерфейс для спеков | Снизить когнитивную нагрузку |
| **Kiro** | Agent Hooks (событийные автоматизации) | FullSpec уже имеет pre-commit, но event-driven hooks шире |
| **OpenSpec** | Лёгкий режим (4 команды, fluid iteration) | Для мелких задач, где полная цепочка — overhead |
| **OpenSpec** | Профили для разных проектов | Кастомизация процесса под тип проекта |
| **Все** | Визуальный дашборд | Навигация по 71 скиллу и 23 агентам |

## Заключение

**FullSpec** — самый полный и глубокий подход к spec-driven development. Он закрывает весь цикл от идеи до release, включая уникальные возможности (Plan Tests, CONFLICT-detection, living docs, 23 агента, полная GitHub-автоматизация).

Но за полноту приходится платить сложностью: 71 скилл, 23 агента, 80+ скриптов, 30 хуков — это enterprise-grade система, и порог входа выше, чем у конкурентов.

**Spec Kit** и **OpenSpec** — более лёгкие альтернативы, идеальные для тех, кто хочет структуру без полного enterprise-процесса. Их преимущество — AI-агностичность.

**Kiro** — сильная IDE, но дорогая и с vendor lock-in. Подходит для AWS-экосистемы.

**Без структуры** — подходит для прототипов и обучения, но не для проектов длительнее недели.

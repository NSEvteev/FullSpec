## Кодекс поведения

Этот проект придерживается профессиональных стандартов общения:
- Уважительное отношение ко всем участникам
- Конструктивная критика кода, а не личности
- Фокус на решении проблем
- Открытость к обсуждению и обратной связи

## Как помочь проекту

### Сообщить о баге

1. Проверьте, не сообщали ли об этом баге ранее в [Issues](https://github.com/your-org/project_template/issues)
2. Создайте новый issue с подробным описанием:
   - Версия проекта
   - Шаги для воспроизведения
   - Ожидаемое поведение vs фактическое
   - Логи и скриншоты (если применимо)
   - Окружение (ОС, браузер, версии зависимостей)

### Предложить новую функциональность

1. Создайте issue с описанием:
   - Проблема, которую решает функциональность
   - Предлагаемое решение
   - Альтернативные варианты (если есть)
2. Дождитесь обсуждения и одобрения перед началом разработки

### Исправить баг или добавить функцию

1. Форкните репозиторий
2. Создайте ветку от `main`: `git checkout -b feature/my-feature`
3. Внесите изменения
4. Создайте Pull Request

## Процесс разработки

### 1. Подготовка окружения

```bash
# Клонировать репозиторий
git clone <repository-url>
cd project_template

# Инициализация
make init

# Запуск в dev режиме
make dev
```

### 2. Создание ветки

Используйте префиксы для веток:
- `feature/` — новая функциональность
- `fix/` — исправление бага
- `docs/` — изменения в документации
- `refactor/` — рефакторинг кода
- `test/` — добавление тестов

```bash
git checkout -b feature/user-profile
```

### 3. Разработка

```bash
# Запустить сервисы
make dev

# Просмотр логов
make logs

# Запуск тестов
make test

# Линтинг и форматирование
make lint
make format
```

### 4. Перед коммитом

```bash
# Проверка кода
make check  # lint + tests

# Проверка документации
make docs-health
make gloss-health
```

## Стандарты кода

### Общие правила

- **DRY (Don't Repeat Yourself):** Избегайте дублирования кода
- **KISS (Keep It Simple, Stupid):** Простота важнее преждевременной оптимизации
- **YAGNI (You Aren't Gonna Need It):** Не добавляйте функциональность "на будущее"
- **Single Responsibility:** Одна функция/класс — одна ответственность

### Именование

- **Переменные и функции:** camelCase (JS/TS), snake_case (Python)
- **Классы:** PascalCase
- **Константы:** UPPER_SNAKE_CASE
- **Файлы:** kebab-case для компонентов, snake_case для модулей Python

### Комментарии

- Комментировать **"почему"**, а не **"что"**
- Код должен быть самодокументируемым
- Сложная логика требует пояснения
- Использовать JSDoc/docstrings для публичных API

```javascript
// ❌ Плохо
// Увеличиваем счётчик на 1
counter += 1;

// ✅ Хорошо
// Пропускаем первую строку CSV (заголовки)
counter += 1;
```

### Форматирование

- Проект использует EditorConfig — настройки применятся автоматически
- Перед коммитом: `make format`
- Линтинг: `make lint`

## Работа с документацией

### Структура документации

Проект использует многоуровневую систему документации:

```
general_docs/
├── glossary.md             # Глоссарий терминов
├── 00_services/            # IT-сервисы
├── 01_discuss/             # Дискуссии (идея → решение)
├── 02_architecture/        # Архитектурные документы
├── 03_diagrams/            # Диаграммы (Mermaid)
├── 04_decisions/           # Архитектурные решения (ADR)
├── 05_resources/           # Описания ресурсов
└── 06_imp_plans/           # Планы реализации
```

### Обязательно документировать

1. **Новые термины** — добавить в `general_docs/glossary.md`
2. **Новые сервисы/пакеты** — создать `README.md` в корне
3. **Архитектурные решения** — создать документ в `general_docs/02_architecture/`
4. **API изменения** — обновить документацию ресурса

### Проверка документации

```bash
# Полная проверка
make docs-check

# Только ссылки
make docs-links

# Глоссарий
make gloss-health
```

### Правила оформления

- Используйте **относительные ссылки** (не абсолютные)
- Термины из глоссария: `[📖 Термин](../general_docs/glossary.md#термин)`
- Заголовки с пробелом: `# Заголовок` (не `#Заголовок`)
- Списки с пробелом: `- Элемент` (не `-Элемент`)

Подробнее: [general_docs.md](llm_instructions/general_docs.md)

## Коммиты и Pull Requests

### Формат сообщений коммитов

Используйте [Conventional Commits](https://www.conventionalcommits.org/ru/v1.0.0/):

```
<тип>(<область>): краткое описание

[опционально: тело коммита]

[опционально: footer]
```

**Типы:**
- `feat` — новая функциональность
- `fix` — исправление бага
- `docs` — изменения в документации
- `style` — форматирование (не влияет на код)
- `refactor` — рефакторинг
- `test` — добавление/исправление тестов
- `chore` — рутинные задачи (обновление зависимостей, и т.д.)

**Примеры:**

```bash
feat(auth): добавлена поддержка OAuth Google

fix(users): исправлена валидация email при регистрации

docs(glossary): добавлены термины RBAC и JWT

refactor(api-gateway): упрощена логика rate limiting
```

### Pull Request

1. **Заголовок:** Краткое описание изменений
2. **Описание:** Что изменено и почему
3. **Связанные issues:** Укажите `Closes #123` или `Fixes #456`
4. **Тестирование:** Как проверить изменения
5. **Чеклист:**
   - [ ] Тесты пройдены (`make test`)
   - [ ] Линтинг пройден (`make lint`)
   - [ ] Документация обновлена
   - [ ] Проверка документации пройдена (`make docs-check`)
   - [ ] Changelog обновлён (если требуется)

**Шаблон PR:**

```markdown
## Описание
Краткое описание изменений.

## Тип изменения
- [ ] Исправление бага (fix)
- [ ] Новая функциональность (feature)
- [ ] Breaking change (существующая функциональность изменена)
- [ ] Документация

## Как протестировать
1. Запустить `make dev`
2. Перейти на http://localhost:####
3. ...

## Чеклист
- [ ] Код следует стандартам проекта
- [ ] Тесты пройдены
- [ ] Документация обновлена
- [ ] Changelog обновлён

Closes #123
```

## Тестирование

### Запуск тестов

```bash
# Все тесты
make test

# Unit тесты
make test-unit

# E2E тесты
make test-e2e

# Integration тесты
make test-integration
```

### Требования к покрытию

- Unit тесты: минимум 80% покрытие
- Критические пути: 100% покрытие
- Новая функциональность должна включать тесты

### Типы тестов

1. **Unit тесты** — изолированные функции/классы
2. **Integration тесты** — взаимодействие между модулями
3. **E2E тесты** — пользовательские сценарии

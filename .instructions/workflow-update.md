## Обзор

**Скилл:** [/instruction-update](/.claude/skills/instruction-update/SKILL.md)

**Назначение:** Проверить файлы проекта на соответствие правилам инструкции и обновить инструкцию при необходимости.

**Всего шагов:** 12

---

## Шаги

### Шаг 1: Получить путь

Получить путь к инструкции из параметров вызова.

**Формат:** `{папка}/{название}.md` относительно `/.claude/.instructions/`

---

### Шаг 2: Определить зону ответственности

Каждая инструкция отвечает за определённую область проекта.

**Алгоритм:**
1. Прочитать путь инструкции
2. Определить зону по маппингу (см. [Маппинг зон](#маппинг-зон))
3. Если зона не определена — запросить у пользователя

**Пример:**
- `src/api/versioning.md` → зона `/src/**/api/**`
- `git/commits.md` → зона `**/*.md` + git операции

---

### Шаг 3: Найти файлы в зоне

```bash
# Найти все файлы в зоне ответственности
Glob → {паттерн зоны}
```

**Фильтры:**
- Исключить `node_modules/`, `.git/`, `dist/`
- Учитывать расширения из инструкции

---

### Шаг 4: Извлечь правила

Прочитать инструкцию и извлечь проверяемые правила.

**Типы правил:**
- **Структурные:** наличие файлов, папок
- **Форматные:** именование, frontmatter
- **Содержательные:** наличие секций, паттернов
- **Ссылочные:** валидность ссылок

---

### Шаг 5: Проверить файлы

Для каждого файла в зоне:

```
1. Прочитать файл
2. Применить правила
3. Собрать несоответствия
```

**Результат:** список `[файл, правило, описание]`

---

### Шаг 6: Сформировать список несоответствий

**Формат вывода:**

```markdown
## Несоответствия

| Файл | Правило | Описание |
|------|---------|----------|
| /src/api/users.ts | Версионирование | Отсутствует версия в пути |
| /src/api/orders.ts | Документация | Нет JSDoc комментариев |
```

**Если несоответствий нет:** сообщить "✅ Все файлы соответствуют инструкции"

---

### Шаг 7: Внести изменения в инструкцию

**Если найдены несоответствия:**

1. Спросить пользователя: "Обновить инструкцию или исправить файлы?"
2. При выборе "обновить инструкцию":
   - Добавить раздел TODO или изменить правила
   - Документировать причину изменения

**Если несоответствий нет:** пропустить шаг.

---

### Шаг 8: Обновить README папки

**Путь:** `/.claude/.instructions/{папка}/README.md`

**Что обновить:**
- Дату последней проверки
- Статус инструкции (если изменился)

---

### Шаг 9: Обновить контекст скиллов

Вызвать: `/context-update`

**Что делает:**
- Обновляет скиллы, связанные с инструкцией
- Синхронизирует изменения по графу зависимостей

---

### Шаг 10: Проверить тесты

**Если есть связанные тесты:**
1. Найти тесты в `/.claude/skills/*/tests.md`
2. Проверить актуальность тест-кейсов
3. При необходимости предложить `/test-update`

---

### Шаг 11: Проверка по чек-листу

Пройти по чек-листу (см. ниже). Все пункты должны быть выполнены.

---

### Шаг 12: Результат

```
✅ Проверка завершена

Инструкция: /.claude/.instructions/{путь}
Зона: {паттерн}
Проверено файлов: {N}
Несоответствий: {M}

{Если M > 0:}
Действие: {обновлена инструкция | файлы помечены для исправления}
```

## Маппинг зон

| Путь инструкции | Зона ответственности |
|-----------------|---------------------|
| `src/**/*.md` | `/src/**` |
| `src/api/**/*.md` | `/src/**/api/**` |
| `src/data/**/*.md` | `/src/**` (работа с данными) |
| `platform/**/*.md` | `/platform/**` |
| `shared/**/*.md` | `/shared/**` |
| `tests/**/*.md` | `/tests/**`, `**/*.test.*` |
| `git/**/*.md` | Git операции, `**/*.md` |
| `doc/**/*.md` | `/doc/**` |
